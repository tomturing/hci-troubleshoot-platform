# 需求：desired_schema.sql 质量修复（Batch 1 + 2）

## 变更历史

| 日期 | 版本 | 变更内容 |
|------|------|---------|
| 2026-04-09 | v1.0 | 初版，基于两份 Schema 审核报告提炼需求 |

## 背景与问题

- **业务背景**: 数据库 Schema (`database/desired_schema.sql`) 在引入 Atlas 迁移工具时进行了大规模重构（v6.2），此后经历了多次快速迭代，遗留了若干设计缺陷和代码质量问题。
- **核心痛点**:
  1. `generate_case_id()` 函数在并发场景下存在竞态条件，可能产生重复工单号
  2. 3 个向量检索字段（`embedding`）没有创建向量索引，语义检索是全表扫描
  3. 消息全文检索索引使用 `english` 分词器，对中文内容完全无效
  4. `diagnostic_item` 表缺少防止重复条目的唯一约束
  5. `session.user_id` 字段无外键约束，可产生孤儿记录
  6. 多个数值型字段（评分、风险等级、阶段）缺少 CHECK 约束，应用层 bug 可写入非法值
  7. `kbd_entry`、`sop_document` 表有 `updated_at` 字段但无自动维护触发器
  8. 存在冗余索引（`idx_user_client_id`）和低价值索引（3 个无 WHERE 条件的 `trace_id` 裸索引）
- **影响范围**:
  - D-001（竞态）: 影响工单创建，高并发时工单号重复
  - D-002（向量索引）: 影响 S0 意图识别和知识库检索性能，数据量增长后全表扫描不可接受
  - D-003（分词）: 影响消息内容搜索，中文内容无法被 `english` 分词器正确索引
  - D-004（唯一约束）: 影响 AI 诊断条目一致性，重复条目导致 Prompt 构建混乱
  - D-005（外键）: 数据完整性风险
  - D-006（CHECK 约束）: 数据质量风险，非法值可被写入数据库

## 需求描述

- **功能概述**: 通过修改 `database/desired_schema.sql` 并生成 Atlas 迁移文件，系统性修复经 Schema 审核发现的缺陷
- **用户场景**: 数据库管理员运行 `atlas migrate apply` 后，所有约束和索引自动生效，无需手动干预
- **预期收益**:
  - 消除工单号并发重复风险
  - 向量检索建立索引，查询性能从 O(n) 提升到 O(log n)
  - 数据库层防止非法数据写入，降低排查线上 bug 的成本
  - 代码库减少冗余维护负担

## 功能需求

1. 修复 `generate_case_id()` 并发竞态（D-001）
2. 为 3 个 `embedding vector(1536)` 字段创建 IVFFlat 向量索引（D-002）
3. 修复消息全文检索索引分词配置（D-003）
4. `diagnostic_item` 补复合唯一约束（D-004）
5. `session.user_id` 补外键约束（D-005）
6. 补充 7 个 CHECK 约束（D-006）
7. `kbd_entry` / `sop_document` 补全 `updated_at` 触发器（P2-2）
8. 清理冗余索引 `idx_user_client_id`（P1-1）
9. 清理低价值裸 trace_id 索引（user/environment/eval）（O-001）

## 非功能需求

- **幂等性**: 所有 DDL 变更必须使用 `IF NOT EXISTS` / `IF EXISTS`，可重复执行
- **零停机**: 添加约束前已有数据必须符合约束，或约束为nullable（CHECK允许NULL）
- **文档同步**: `desired_schema.sql` 修改须同步更新 `docs/solution/数据库设计.md`

## 验收标准

- [ ] `desired_schema.sql` 中 `generate_case_id()` 使用 `pg_advisory_xact_lock` + `MAX`
- [ ] `kb_category` / `kbd_entry` / `sop_chunk` 各有一个 `ivfflat` 向量索引
- [ ] `idx_message_content_search` 使用 `'simple'` 而非 `'english'`
- [ ] `diagnostic_item` 有 `UNIQUE(conversation_id, type, seq)` 约束
- [ ] `session` 有 `fk_session_user_id REFERENCES "user"`
- [ ] 7 个 CHECK 约束全部存在（grep 验证）
- [ ] `kbd_entry` / `sop_document` 各有 `updated_at` 触发器
- [ ] `idx_user_client_id` 不存在（已移除）
- [ ] `grep "idx_user_trace_id\|idx_environment_trace_id\|idx_eval_trace_id" desired_schema.sql` 返回空（仅注释行）

## 约束条件

- **技术约束**:
  - IVFFlat 索引需要数据量 > `lists * 10`（当前 lists=100，即 >1000 条）才能有效，入库前可创建索引但效果有限
  - CHECK 约束为 NULL-safe（`IS NULL OR (条件)`），兼容历史 NULL 数据
- **依赖约束**: 本需求只修改 `desired_schema.sql`，Atlas 迁移文件须在有 PG 连接的环境生成

## 风险与假设

- **风险**: 添加 `session.user_id` 外键时，若已有孤儿记录（user 已删但 session 残留），迁移会失败
  - 缓解：迁移前执行 `DELETE FROM session WHERE user_id NOT IN (SELECT user_id FROM "user")`
- **假设**: `check_diagnostic_item_stage` CHECK 约束值域 `('S2','S3','S4','S5')` 与当前业务逻辑一致
