## 背景与问题

通过代码走查与运行时观测，发现以下 4 个彼此独立但共同影响诊断体验的缺陷：

### 缺陷 1：工单状态流转被前端短路（设计 vs 实现偏差）

- **设计意图**：`case.status` 的 `created → confirmed` 转换由 conversation-service 在 S0 意图识别完成后触发（写入 `category_id` 时同步推进）
- **实际行为**：前端在工单创建成功后立即调用 `PUT /api/cases/{id}/confirm`，工单还未开始任何 AI 诊断就已经变为 `confirmed`
- **表现**：用户看到"工单已创建，正在自动确认…" → "工单已确认，正在连接 AI 助手…"，两条消息间隔约 200ms，实际上整个 S0 识别过程从未被等待
- **影响**：`case.status` 失去了业务语义——`confirmed` 理应代表"故障分类已确认"，现在只代表"工单被前端点击了一下"

### 缺陷 2：S0 高置信度路径绕过了用户确认

- **设计意图**：S0 意图识别阶段，无论置信度高低，都应展示候选选项让用户确认，最少 2 个（命中项 + "以上都不是"），最多 3 个
- **实际行为**：Prompt 的"情况一：高置信度"分支允许 AI 直接在回复末尾输出 `已确认故障分类：虚拟机-003 虚拟机开机失败`，无需客户确认，AI 自行决策
- **影响**：客户没有确认机会，分类可能错误，导致后续 S1-S6 都跑偏方向；违背"让客户参与确认"的产品设计原则

### 缺陷 3：S0 完成后 AI 诊断卡死，永远不推进到 S1（最严重）

本缺陷由三处实现错误叠加造成，任意一处都会导致 S1 永远无法启动：

**错误 3-a：STAGE_TRIGGERS 触发词与 AI 实际输出格式不匹配**
- `conversation_manager.py` 的 `STAGE_TRIGGERS[("S0","S1")]` 包含的正则是旧格式（"确认您的故障情况是…"）
- 新 S0 Prompt 要求 AI 输出 `已确认故障分类：虚拟机-003 虚拟机开机失败`
- 两者完全不匹配，`detect_stage_transition()` 永远返回 `None`，`diagnostic_stage` 从不更新出 S0

**错误 3-b：category_info 写入被锁死在 `if new_stage:` 内**
- `conversation_service.py` 中，`_update_conversation_category()` 调用被包在 `if new_stage:` 块里
- 由于错误 3-a 导致 `new_stage = None`，`category_info` 即使被成功提取也永远不会写入数据库
- `conversation.category_id` 始终为 `None`，S1 Prompt 拿不到分类上下文

**错误 3-c：S0 候选确认方法是死代码**
- `ConversationManager` 实现了 `parse_candidate_selection()`、`resolve_candidate_category()`、`should_trigger_s0_failure()` 等完整的候选确认链路
- 但 `conversation_service.py` 的 `send_message_stream_only()` 从未调用这些方法
- 用户发送"①"或"1"时，字符直接发给 AI 重跑 S0 Prompt，陷入死循环

### 缺陷 4：终端历史记录无法写入数据库

- **现象**：终端历史回放界面"数据库一条数据都没有"
- **根因**：`api-gateway/app/main.py` 初始化 `TerminalService` 时只传了 `redis_manager`，`db_manager` 参数缺失（为 `None`）
  ```python
  terminal_service = TerminalService(redis_manager)  # db_manager 默认 None
  ```
- **表现**：`create_operation()` 执行 `async with self.db.async_session_factory()` 时 `self.db is None`，抛 `AttributeError`，被路由层 `except Exception` 吞掉返回 HTTP 500；前端的 `flushOutputBuffer()` 只打了 `console.error`，用户无感知

## 功能需求

### FR-1：工单状态语义对齐
- `created → confirmed` 转换权回归 conversation-service
- 前端不得在工单创建后立即调用 confirm 接口
- conversation-service 在 S0 分类写入 `category_id` 的同时调用 `PUT /api/cases/{id}/confirm`
- 前端通过 SSE 事件 `case_status_updated` 感知状态变化并更新 UI

### FR-2：S0 必须展示候选选项
- 删除 Prompt 中的"高置信度直接确认"分支
- 统一为"候选确认"模式：最多 2 个候选（①②）+ 固定的 ③"以上都不是"
- 高置信度时 ① 直接给出最高匹配分类，② 给出次高分类（或相近分类），③ 兜底
- 若只有 1 个候选：① 命中项，② 以上都不是（请补充描述）

### FR-3：S0 → S1 阶段切换正确执行
- 更新 `STAGE_TRIGGERS[("S0","S1")]` 触发词，匹配候选确认完成后的 AI 输出格式
- 解耦 `_update_conversation_category` 调用：用户确认分类后直接写入，不依赖 `new_stage` 是否非空
- 接入 `parse_candidate_selection()` / `resolve_candidate_category()` 处理用户选 ①②③ 的场景

### FR-4：终端操作记录正常写库
- `TerminalService` 必须注入有效的 `DatabaseManager` 实例
- `POST /api/terminal/operations` 响应码应为 `201`，数据落库
- 终端历史回放界面能查到操作记录

## 非功能需求

- **可观测性**：每个修复点新增结构化日志（`event=`、`case_id=`、`stage=`），方便 Grafana 追踪
- **向后兼容**：所有 API 路由签名不变，只修改内部行为
- **测试覆盖**：新增或更新对应的单元测试

## 验收标准

- [ ] AC-1：工单创建后，前端界面显示状态为 `created`（待分类），直到 S0 完成后状态变为 `confirmed`（已分类）
- [ ] AC-2：所有 S0 AI 回复末尾均以候选选项形式呈现（① / ② / ③），不出现"已确认故障分类：…"直接写入的情况
- [ ] AC-3：用户选 ① 后，对话进入 S1 阶段，AI 开始故障定位；`conversation.diagnostic_stage` 从 S0 变为 S1
- [ ] AC-4：`GET /api/terminal/operations?case_id=X` 在 SSH 会话产生输入/输出后，返回非空列表；`POST /api/terminal/operations` 响应为 201
- [ ] AC-5：Ruff lint 零错误

## 约束条件

- **技术约束**：Python 环境用 `uv`；前端用 `pnpm`；不破坏现有 API 合约
- **文档约束**：改动涉及 `backend/`、`frontend/` 时，必须同步更新 `docs/solution/` 至少一个现行全量文档
- **优先级**：缺陷 3（S0 卡死）> 缺陷 4（终端无记录）> 缺陷 1（状态语义）> 缺陷 2（强制确认）
