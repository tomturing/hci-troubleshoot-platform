## 背景与问题

### 业务背景

hci-troubleshoot-platform（htp）是生产级多租户运维排障平台，ops-agent 是独立研发的 SOP 驱动型诊断智能体。当前已存在一个 `feature/ops-agent-integration` 分支，将 ops-agent 接入 htp 作为 AI 助手选项之一。

### 核心痛点

当前集成方案存在根本性的语义层次错配，导致三个不可接受的问题：

1. **对话上下文每轮重置**：ops-agent 被当作无状态 LLM 客户端调用，每次 HTTP 请求都触发新的 `agent.run()`，从 intake 重新开始，完全丢弃对话历史。用户每轮都在重新开始诊断。

2. **实时数据被丢弃**：htp 在调用前已预取了告警日志、失败任务等实时数据并注入 system prompt，但 ops-agent 只取最后一条 user 消息，实时上下文全部丢弃。ops-agent 只能靠 `get_info_from_user` 工具问用户"告警是什么"。

3. **两套状态机并行干扰**：htp 的 S0-S6 阶段状态机与 ops-agent 的 intake→closed 状态机同时运行，互相不知道对方的存在，互相干扰。

### 根本原因

集成分叉点在 `AIAssistantRegistry`（LLM 客户端层），而 ops-agent 需要在"会话管理 + 大脑执行"的分界线上切入，即处于 `send_message_stream_only()` 这一整体方法的更上层。错误的插入层次导致以上所有问题无法在当前架构下修复。

### 影响范围

- **用户**：使用 ops-agent 模式的客服人员，排障流程每轮断裂，体验极差
- **开发团队**：无法通过 A/B 测试比较两种大脑的实际效果
- **产品**：ops-agent 的 SOP 驱动能力、精准章节导航能力无法在 htp 平台中体现

## 需求描述

### 功能概述

重新设计 ops-agent 合入 htp 的架构方案，实现"**大脑可选，骨架手脚共用**"：

- `骨架手脚`（不可替换）：acli 工具执行、SCP API 工具调用、工单管理、消息持久化、高危操作确认、审计追踪、实时告警预取
- `大脑`（可热切换）：htp 原生大脑（S0-S6 + kb-service）或 ops-agent 大脑（intake→closed + SOP JSONL）

### 用户场景

1. **工程师 A/B 测试**：在创建对话时选择"htp 大脑"或"ops-agent 大脑"，对同一故障场景进行平行测试，通过 Grafana 对比效果指标
2. **ops-agent 多轮排障**：使用 ops-agent 模式的客服，可以连续多轮对话，ops-agent 保持诊断上下文（intake → routing → validation → solution），不会每轮重置
3. **实时数据感知**：ops-agent 启动时能看到 htp 预取的实时告警数据，不需要用户重复粘贴告警信息
4. **渐进接通工具执行**：长期目标，ops-agent 引导的检查步骤可以通过 htp 的 acli 工具自动执行，而不是等用户手动输入结果

### 预期收益

- 两套推理策略可以平行运行并量化对比，找到最优诊断大脑
- ops-agent 的 500 步深度推理、SOP 强制遵循、精准章节导航能力得到完整体现
- htp 的所有现有能力（acli、确认机制、审计、持久化）不受任何影响

## 功能需求

### 必须实现（Phase 1 - 修复语义错配）

1. **会话连续性**：同一对话中多轮消息，ops-agent 大脑必须能识别并接续上下文，不从 intake 重新开始
2. **隔离执行**：ops-agent 大脑路径不触发 htp 的 S0-S6 system prompt 构建，不触发阶段转换检测，不触发 scheduler pod 分配
3. **真流式输出**：ops-agent 必须在 ReAct 每步 LLM 推理时实时 yield token，不能等待整个链路完成后伪造 SSE 流（TTFT 不超过 3 秒）

### 应该实现（Phase 2 - 接通手脚）

4. **实时环境数据注入**：htp 调用 ops-agent 前自动预取告警/任务数据，ops-agent 能感知并利用这些数据，避免问用户已知信息
5. **会话持久化**：ops-agent 的多轮上下文存储于 Redis（TTL 4 小时），服务重启后可恢复

### 可以实现（Phase 3 - A/B 测试能力）

6. **前端大脑选择器**：创建对话时展示"选择诊断助手"，标注"htp 大脑"或"ops-agent 大脑"，对话标题栏显示当前模式
7. **可观测性**：Grafana 增加 `assistant_type` 维度的 TTFT、成功率、轮次数等对比指标

### 选做（Phase 4 - 工具 Bridge）

8. **工具执行代理**：ops-agent 发出的 acli 命令请求，由 htp 的 ToolRouter 代为执行，ops-agent 收到执行结果而非等待用户输入

---

## 非功能需求

- **性能**：首 token 到达时间（TTFT）≤ 3 秒
- **可用性**：ops-agent 服务不可达时，自动降级到 htp 大脑，不中断用户操作
- **可维护性**：两个大脑的代码互不耦合，修改一个不影响另一个；跨仓库接口变更通过 OpenAPI 契约管理
- **可观测性**：ops-agent 模式下的每次请求均有完整链路追踪（trace_id 贯穿）

## 验收标准

### Phase 1 验收

- [ ] 选择 ops-agent 模式后，进行 5 轮对话，第 5 轮中 ops-agent 仍能引用第 1 轮的故障描述（上下文保持）
- [ ] 同一故障描述，选 htp 大脑和 ops-agent 大脑，htp 大脑进入 S0-S6 流程，ops-agent 大脑进入 intake→routing 流程，互不干扰
- [ ] ops-agent 模式首 token 延迟 ≤ 3 秒（在测试环境 LLM 正常响应时）
- [ ] 选择 openclaw 模式的现有对话，全部功能不受任何影响

### Phase 2 验收

- [ ] ops-agent 模式启动时，system message 中包含本次预取的实时告警数据（可通过 debug 日志验证）
- [ ] ops-agent 服务重启后，仍能接续 4 小时内的对话上下文

### Phase 3 验收

- [ ] 创建对话界面提供助手类型选择，选择记录到 `conversation.assistant_type` 字段
- [ ] Grafana 面板中可以按 `assistant_type` 对比 TTFT 和请求成功率

## 约束条件

- **技术约束**：htp 使用 FastAPI + SQLAlchemy + PostgreSQL + Redis + K3s；ops-agent 使用 Python + uv，独立仓库
- **兼容性约束**：现有 openclaw 路径必须零破坏；ops-agent 的 CLI/ACP 模式必须零破坏
- **仓库约束**：两个项目是独立仓库，不能引入共享 Python 包依赖；跨仓库集成通过 HTTP API 完成
- **优先级约束**：Phase 1 最优先，确保 ops-agent 在 htp 中能正常工作后，再推进 Phase 2-4

## 风险与假设

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|---------|
| base_agent.py 真流式改造破坏 CLI 模式 | 高 | 中 | 严格单元测试，流式改造走独立代码路径 |
| Redis session pool 引入新依赖 | 低 | 低 | Phase 1 先用 in-memory dict，Phase 2 再升级 Redis |
| 跨仓库接口不同步导致兼容性问题 | 高 | 中 | 强制执行 OpenAPI 契约，两个仓库 PR 同期合并 |
| ops-agent 服务不稳定导致 htp 主流程受影响 | 高 | 低 | 实现降级机制，ops-agent 不可达时自动切换到 htp 大脑 |
