# 需求：Atlas 声明式数据库发布改造

## 背景与动机

当前数据库发布流程存在两类系统性缺陷，从截图（`tool_audit_log does not exist at position 112:6`）可见，SQL 错误直到生产部署后才暴露，说明质量卡控完全缺失。

### 问题一：自动化程度不足

现有流程在以下环节需要**人工干预**：

| 步骤 | 手动动作 | 后果 |
|------|---------|------|
| 新增迁移文件后 | 必须手动执行 `scripts/ci/sync-db-migrations.sh` | 忘记执行 → CI `check-db-migrations-sync` 失败，阻断 PR |
| ConfigMap 体积限制 | 所有 SQL 静态嵌入 YAML，随迁移增多趋向 1MB 上限 | 迁移文件超过限制时整个 Chart 无法部署 |
| 迁移版本跟踪 | `schema_migrations` 是 dbmate 私有格式 | 与 Atlas 等主流工具不兼容 |

**根因**：SQL 迁移文件通过静态 ConfigMap 嵌入 Helm Chart，是 Kubernetes 对文件的原始妥协方案，没有文件变化即 ConfigMap 变化的自动感知机制。

### 问题二：发布前无验证

| 缺失的检验点 | 后果（真实案例） |
|-------------|--------------|
| SQL 语法正确性 | `20260402002_migrate_audit_data.sql` 引用了不存在的表 `tool_audit_log`，在 dev 才发现 |
| 迁移依赖顺序 | 多个 `20260402` 前缀迁移文件互相依赖，顺序错误导致链断裂 |
| 危险操作检测 | DROP TABLE、无默认值的 NOT NULL 列等破坏性操作无法被 CI 拦截 |
| 幂等性验证 | 迁移文件重复执行时是否安全，无法在 PR 阶段验证 |

---

## 需求规格

### REQ-DB-001：消除手动同步步骤

**目标**：新增或修改迁移文件后，**零手动操作**即可完成 CI/CD 流程。

**验收标准**：
- `sync-db-migrations.sh` 脚本从仓库中删除
- `check-db-migrations-sync` CI job 从 `ci.yml` 中删除  
- 开发者不再需要将 SQL 同步到 ConfigMap

### REQ-DB-002：PR 阶段 SQL 全量验证

**目标**：在 PR 合并前，通过真实 PostgreSQL 实例完整执行所有迁移文件，任何 SQL 错误在 PR 阶段阻断。

**验收标准**：
- PR CI 中存在 `db-migration-test` job，使用 postgres service container
- 全量迁移执行成功为 PR 合并前置条件
- 危险操作（DROP TABLE、无默认值 NOT NULL 列等）通过 `atlas migrate lint` 检测并阻断

### REQ-DB-003：声明式 Schema 管理

**目标**：维护单一 `database/desired_schema.sql` 作为期望状态，新增字段/表时修改该文件，通过工具自动生成迁移文件，减少手写迁移错误。

**验收标准**：
- `database/desired_schema.sql` 存在且为当前 17 张业务表的完整定义
- 开发者通过 `atlas migrate diff --env local <名称>` 生成迁移文件
- 迁移文件经 `atlas migrate lint` 质量检查

### REQ-DB-004：迁移镜像化，彻底废除 ConfigMap

**目标**：迁移文件打包进 Docker 镜像，与应用服务镜像同等管理，不再依赖 ConfigMap volume。

**验收标准**：
- `Dockerfile.migrations` 存在且可构建
- CI 在 main push 时构建 `db-migrate` 镜像，标签格式同其他服务
- `db-migrations-configmap.yaml` 从 Helm Chart 中删除
- `db-migrate-job.yaml` 使用 Atlas 镜像而非 dbmate + ConfigMap volume

## 现有方案废弃清单

以下文件/机制在改造完成后**全部废弃删除**：

| 废弃项 | 类型 | 替代方案 |
|--------|------|---------|
| `deploy/helm/hci-platform/templates/hooks/db-migrations-configmap.yaml` | Helm 模板 | 迁移镜像化，ConfigMap 不再需要 |
| `deploy/helm/hci-platform/templates/hooks/db-migrations-configmap.yaml.bak` | 备份文件 | 删除 |
| `scripts/ci/sync-db-migrations.sh` | CI 脚本 | 不再需要手动同步 |
| `scripts/ci/check-db-migrations-sync.sh` | CI 脚本 | 替换为 `atlas migrate validate` |
| `ci.yml` 中的 `check-db-migrations-sync` job | CI 流程 | 替换为 `db-migration-test` job |
| `database/migrations/` 中的旧 dbmate 文件（执行层） | 迁移文件 | 历史文件保留存档，Atlas 使用 `atlas-migrations/` 目录 |
| `schema_migrations` 表（dbmate 版本跟踪） | DB 表 | 由 `atlas_schema_revisions` 表替代（Atlas 自动管理） |

---

## 约束条件

1. **不破坏现有 dev/staging/prod 数据**：已执行的迁移不重新执行
2. **零停机迁移**：使用 Atlas baseline 机制，已有 DB 首次切换时跳过 baseline
3. **向后兼容**：Helm Job 保持 ArgoCD PreSync Hook 模式不变，仅替换工具
4. **CI 不依赖本地安装**：所有工具通过容器执行，不要求开发者本地安装 Atlas
