# ops-agent 集成需求

## 变更历史
| 日期 | 版本 | 变更内容 | 关联事件文档 |
|------|------|---------|------------|
| 2026-04-27 | v1.0 | 初版 | [2026-04-27-ops-agent集成.md](2026-04-27-ops-agent集成.md) |

## 背景与问题
- **业务背景**：现有hci-troubleshoot-platform是一个企业级排障平台，需要引入一个新的SOP引导的工具型agent（ops-agent）来增强排障能力
- **核心痛点**：两个项目设计理念不同，直接合并会导致架构冲突
- **影响范围**：conversation-service（对话管理）、AI助手模块

## 需求描述
- **功能概述**：将ops-agent作为可选择的AI助手集成到hci-troubleshoot-platform中
- **用户场景**：用户在排障时可以选择使用ops-agent进行SOP引导的排障
- **预期收益**：保留两个项目的独立性，同时提供统一的用户体验

## 功能需求
1. ops-agent提供OpenAI-compatible API端点
2. hci-troubleshoot-platform新增OpsAgentAssistant适配器
3. 用户可以在对话中切换使用不同的AI助手
4. 支持完整的流式对话体验

## 非功能需求
- **性能要求**：集成后响应时间不超过现有AI助手的120%
- **可用性要求**：ops-agent故障不影响其他AI助手的使用
- **安全要求**：遵循现有的认证和授权机制
- **可维护性**：提供完整的日志和监控指标

## 验收标准
- [ ] ops-agent的OpenAI-compatible API端点可以正常响应
- [ ] hci-troubleshoot-platform可以通过适配器调用ops-agent
- [ ] 用户可以在对话中选择使用ops-agent
- [ ] 完整的流式对话体验正常工作
- [ ] 监控和日志正常输出

## 约束条件
- **技术约束**：使用现有的AIAssistantClient协议
- **资源约束**：两个项目保持独立部署
- **依赖约束**：ops-agent需要暴露HTTP API

## 风险与假设
- **已知风险**：SOP格式可能需要适配
- **假设条件**：ops-agent可以暴露OpenAI-compatible API
