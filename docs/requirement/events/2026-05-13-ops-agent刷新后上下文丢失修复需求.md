# ops-agent 刷新页面后历史上下文丢失修复需求

## 变更历史

| 日期 | 版本 | 变更内容 |
|------|------|---------|
| 2026-05-13 | v1.0 | 初版 |

---

## 背景与问题

### 业务背景

用户使用 ops-agent 进行故障排查时，整个诊断流程涉及多轮 AI 交互（问题收集 → 信息确认 → 初步诊断 → 故障假设 → 验证假设 → 方案确认 → 执行修复），通常需要 10–30 分钟完成。

### 核心痛点

**刷新页面后 ops-agent 从头开始**：用户在诊断进行到中途（例如步骤 3 初步诊断）时，
偶然刷新了页面，或网络中断后重新进入。此时：

- 前端 **消息列表正常恢复**（hci DB 里有完整记录）
- 但 ops-agent 大脑 **"失忆"**，重新从步骤 1 问「您遇到了什么问题？」
- 已经完成的诊断步骤全部作废，用户必须重新回答所有问题

### 影响范围

- 所有使用 ops-agent 作为大脑的对话会话
- ops-agent 进程重启（pod 重调度、服务更新）同样触发此问题
- 在 CI/CD 发布期间尤为严重（每次更新 ops-agent 都会中断所有正在进行的诊断）

---

## 需求描述

### 功能概述

在 ops-agent 进程重启后，能够从磁盘恢复该会话的完整 LLM 消息历史，
使 AI 恢复诊断上下文，从断点处继续而不是从头开始。

### 用户场景

1. 用户开始故障排查，进行到步骤 4（故障假设阶段）
2. ops-agent pod 因更新被重调度，进程内存清空
3. 用户刷新页面，前端消息列表正常加载
4. **修复后**：ops-agent 自动从 trajectory 文件恢复历史，从步骤 4 继续诊断
5. **修复前**：ops-agent 从步骤 1 重新开始

### 预期收益

- 消除因进程重启导致的诊断中断问题
- 提升用户在 ops-agent 排障过程中的连续性体验
- 与 ops-web（Streamlit 界面）的体验对齐

---

## 功能需求

1. **持久化 LLM 消息历史**：ops-agent 每次完成一轮对话后，将 LLM message history 写入 per-session trajectory 文件
2. **恢复 LLM 消息历史**：ops-agent 在创建新 session（`session_new`）时，若指定 `trajectory_dir`，则从该目录读取最新 trajectory 文件并注入历史
3. **per-session 隔离**：每个 hci conversation_id 对应独立的 trajectory 目录，互不干扰
4. **幂等性**：session 已存在于内存时，`session_new` 直接返回，不重新加载（不影响进行中的对话）

## 非功能需求

- **性能**：trajectory 文件读取不影响首条消息响应时间（读取在 session_new 时进行，不在消息发送路径上）
- **存储**：trajectory 文件大小与对话轮次正相关，单个会话通常 10–200KB
- **持久化**：生产环境需挂载 PVC，确保 pod 重调度后文件不丢失

---

## 验收标准

- [ ] V1：ops-agent 进程重启后，已有会话重新发起对话时 AI 能体现对前序上下文的理解
- [ ] V2：trajectory 文件在 `$OPS_AGENT_TRAJECTORY_DIR/{conversation_id}/` 目录下可见
- [ ] V3：hci DB 中的消息列表显示完整（不因本次改动丢失任何消息）
- [ ] V4：ops-web / CLI 路径不受影响（不改变现有行为）

## 约束条件

- **技术约束**：改动不能破坏 ops-web 的 `ACPServer(trajectory_file=...)` 调用路径
- **部署约束**：生产环境需为 ops-agent pod 挂载 PVC 到 `/data/ops-agent-trajectories`
