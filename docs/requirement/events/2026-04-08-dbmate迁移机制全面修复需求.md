# dbmate 迁移机制全面修复需求

## 背景

自 PR #84 引入 dbmate 以来，数据库迁移链多次出现故障，累计产生 9 个修复 PR（#84/#85/#91/#108/#113/#116/#121 等）。根因分析显示存在三个层面的系统性问题，需要一次性彻底修复。

## 问题描述

### P0：sop_document / sop_chunk 表从未在 K8s 环境创建

`20260402001_sop_tables.sql` 与 `20260402001_kb_category_sop_hit_tracking.sql` 使用相同 version 号，dbmate 静默跳过前者，导致 `sop_document` / `sop_chunk` 两张被后端代码（kb-service、conversation-service）实际引用的表从未在 K8s 环境中被创建。

### P1：Alembic 迁移链与 dbmate 并行，产生状态矛盾

项目中同时存在两套迁移系统（dbmate + Alembic），管理同一个 PostgreSQL 数据库，各自有独立的追踪表（`schema_migrations` vs `alembic_version`），没有协调机制。`0005_schema_repair.py` 中包含创建 7 张废弃表的代码，如果被执行会与 dbmate 路径产生冲突。

### P2：没有重复 version 号的 CI 检查

dbmate 重复 version 是静默 bug，CI 中没有检测机制，导致 P0 问题在 4 个 PR 之后才被发现。

### P3：开发者缺少本地快捷命令

`make db-sync` / `make db-check` 不存在，开发者需要记住完整脚本路径，容易遗忘同步 ConfigMap 步骤。

## 需求要求

### F1：修复 sop_document / sop_chunk 缺失问题
- 删除重复 version 的旧文件 `20260402001_sop_tables.sql`
- 以新 version（`20260408001`）重新纳入 sop_document / sop_chunk 建表 DDL
- 所有 DDL 使用 `IF NOT EXISTS` 保证幂等
- 更新 `00_baseline.sql`，补充历史 version 记录

### F2：彻底废弃 Alembic 迁移链
- 停用 hci-platform-data 的 Alembic K8s Job（`dbMigrate.enabled=false` 永久锁定）
- 修改 Alembic Job 模板为空操作占位，防止未来被误启用
- 修改 `0005_schema_repair.py`，删除废弃表创建代码（§5-§8）
- 在代码中明确注释说明 Alembic 已废弃

### F3：CI 增加 version 唯一性检查
- `check-db-migrations-sync.sh` 增加重复 version 检测逻辑
- 发现重复时输出具体文件名并以非零状态码退出
- CI `check-db-migrations-sync` job 自动覆盖此检查

### F4：本地开发快捷命令
- `make db-sync`：调用 `sync-db-migrations.sh` 同步 ConfigMap
- `make db-check`：调用 `check-db-migrations-sync.sh` 做本地验证

### F5：更新 database/README.md
- 明确命名规范（禁止日期和序号之间加下划线）
- 补充迁移脚本幂等性规范（强制要求，含错误和正确示例）
- 更新工作流，包含 `make db-sync` 步骤
- 注明 Alembic 已废弃

## 验收标准

- [ ] `ls database/migrations/ | cut -d_ -f1 | sort | uniq -d` 无输出（无重复 version）
- [ ] `make db-check` 通过，无报错
- [ ] `sop_document` / `sop_chunk` 表在新迁移执行后存在
- [ ] Alembic K8s Job 已停用（`enabled: false` + 空操作容器）
- [ ] `0005_schema_repair.py` 不再包含 kb_document / kb_chunk 等废弃表创建代码
- [ ] `database/README.md` 包含幂等性规范和 Alembic 废弃声明
