# SOP 多叉决策树——解析策略与校验模式设计需求

## 背景与问题

SOP 文档由各业务方提供，格式自由度高：
- 案例多的 SOP：100+ 叶节点，多层嵌套（H1→H2→H3→H4→H5+）
- 案例少的 SOP：1 个叶节点，H1 直接是叶（仅两层）
- 话术不统一：同一语义的标题可能写为"解决方案"、"处理方法"、"解决方法"、"处理步骤"等
- 标题层级有时写错（H5 本该是"判断方法"却放在 H4，或更深）

原有设计中，`sop_template.py` 里 `SOPNode.model_validator` 对不完整叶节点直接抛出 `ValueError`，
导致任何一个叶节点结构缺失就导致整棵树构建失败，无法实现容错入库与差异审计。

## 核心需求

### N1：叶优先解析策略（"找头尾"）

解析 SOP 文档不应从标题层级推导树角色，而应：
1. **找尾（叶节点）**：识别拥有「判断方法」+「解决方案」配对段落的节点作为叶节点锚点——识别依据是**关键词语义匹配**，与标题绝对层级无关
2. **找头（根节点）**：文档的最高层标题 = 根节点（场景名称）
3. **推导中间**：根节点与叶节点之间的所有标题节点，角色由**相对层级差**决定，不依赖绝对 H 数字

### N2：关键词等效识别

段落类型不由标题层级决定，由关键词语义匹配决定：

| 目标字段 | 等效关键词（任一匹配即可）|
|---------|------------------------|
| `diagnosis` | 判断方法 / 判断依据 / 排查方法 / 排查步骤 / 识别方法 / 确认方法 / 诊断方法 |
| `solution`  | 解决方案 / 解决方法 / 处理方法 / 处理步骤 / 修复方法 / 修复步骤 / 解决步骤 |

配对约束：同一叶节点下 diagnosis 和 solution 必须成对出现。

### N3：源标题溯源字段（话术归一审计）

`DiagnosisDetail` 和 `SolutionDetail` 各自新增 `source_heading: str | None` 字段，
记录 docx 中原始标题文本（如"处理方法"），供：
- 上传后展示话术差异报告（"发现 N 处话术不规范：处理方法→解决方案"）
- 跨文档聚合查询，找出系统性话术问题

### N4：宽松校验模式

Pydantic 模型层**不抛出异常**——不完整叶节点（缺 diagnosis 或 solution）可以正常构建 `SOPNode`。
校验结果由 `SOPValidationResult` 收集，分三级处理：
- `error`：阻断入库，上传接口返回 422
- `warning`：允许入库，前端展示提示
- 两者均写入 `sop_tree.validation_issues` 持久化存储

### N5：validation_issues 三层作用

1. **即时反馈**：上传接口响应中直接返回，告知用户具体缺失
2. **持久化审计**：存入 `sop_tree.validation_issues`（JSONB），事后查询
3. **异类检测原材料**：跨文档聚合，找出话术不规范文档，触发批量修正流程

## 验收标准

- [ ] `SOPNode` 无 `model_validator`，残缺叶节点可正常构建
- [ ] `DiagnosisDetail` 和 `SolutionDetail` 各有 `source_heading` 字段
- [ ] `sop_template.py` 模块 docstring 包含完整关键词等效表
- [ ] `sop_tree.validation_issues` JSONB 列正常写入、可查询
- [ ] 上传接口对 `warning` 级别问题允许入库并在响应中返回
