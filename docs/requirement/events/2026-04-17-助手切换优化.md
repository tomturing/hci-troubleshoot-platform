# AI 助手选择器交互优化 — 支持对话中切换

## 变更历史
| 日期 | 版本 | 变更内容 | 关联事件文档 |
|------|------|---------|------------|
| 2026-04-17 | v1.0 | 初版 | 本文档 |

## 背景与问题

### 业务背景
PR #171 实现了 AI 助手选择器的智能显示机制（后端驱动），但 UI 设计存在局限性：助手选择器只在「创建工单」对话框中显示，一旦 conversation 创建，助手类型就固定，用户无法在对话过程中切换。

### 核心痛点
1. **切换场景被忽略**：用户可能在多种场景下需要切换助手：
   - 回答不满意时：想换一个助手试试
   - 问题类型变化：代码问题 vs 系统问题
   - 分阶段策略：先用快速模型收集信息，再用强大模型深度分析
2. **被迫新建工单**：当前切换助手只能关闭工单重新创建，打断对话流程
3. **后端能力浪费**：后端已支持动态切换（`assistant_type` 可在每条消息中透传），但前端未利用

### 影响范围
- 影响用户：所有使用 HCI 排障平台的工程师
- 影响场景：任何需要多助手协同排查的复杂问题

## 需求描述

### 功能概述
将助手选择器从「工单创建对话框」移至「对话界面」，支持用户在任意对话轮次切换助手，切换后保留对话历史，新助手继承上下文继续对话。

### 用户场景
```
场景 1：回答不满意
用户提问 → OpenClaw 回答 → 用户觉得不够详细 → 切换到 Nabobot → Nabobot 基于历史上下文继续分析

场景 2：问题类型变化
用户描述系统问题 → OpenClaw 分析系统状态 → 用户提到代码问题 → 切换到 CodeAssistant → CodeAssistant 分析代码日志

场景 3：分阶段策略
用户描述故障 → 快速模型（GLM）快速收集信息 → 切换到强大模型（Claude）→ Claude 深度分析
```

### 预期收益
- 提升对话效率：无需关闭工单重新创建
- 增强灵活性：支持多助手协同排查
- 利好后端能力：充分利用已有的动态切换机制

## 功能需求

### 必须实现
1. **助手选择器位置调整**：从工单创建对话框移至对话界面（消息区域顶部或输入区域）
2. **动态切换支持**：发送消息时可携带当前选中的 `assistant_type`
3. **历史保留**：切换助手后，对话历史不变，新助手继承上下文
4. **当前助手显示**：对话界面明确显示当前使用的助手名称

### 可选增强（不在本次迭代范围）
- 切换时提示用户「已切换到 XXX 助手」
- 记录每次切换的时间点到对话历史
- 支持设置工单级别的默认助手

## 非功能需求

### 性能要求
- 切换助手不应增加额外 API 调用（复用现有参数透传）
- UI 切换响应时间 < 100ms

### 可用性要求
- 助手选择器始终可见（不隐藏）
- 切换交互简单直观（下拉选择）
- 当前助手状态明确展示

### 可观测性
- 前端日志记录助手切换事件（用于后续分析用户行为）

## 验收标准

- [ ] 标准 1：对话界面中助手选择器始终可见（有多个可用助手时）
- [ ] 标准 2：用户可在任意对话轮次切换助手
- [ ] 标准 3：切换助手后，发送新消息时新助手响应
- [ ] 标准 4：对话历史完整保留，不因切换助手丢失
- [ ] 标准 5：工单创建对话框中的助手选择器移除（避免重复）

## 约束条件

### 技术约束
- 后端 API 不需要新增端点，只需打通现有参数传递链路
- 前端需修改 Schema 和消息发送逻辑

### 依赖约束
- 依赖 PR #171 已实现的助手列表 API（`/api/assistants`）
- 依赖后端 `send_message_stream_only` 已支持 `assistant_type` 参数

## 风险与假设

### 已知风险
| 飭险 | 影响 | 缓解措施 |
|------|------|---------|
| 不同助手上下文理解能力不同 | 中 | 切换后提示「已切换到 XXX，将基于历史继续分析」 |
| 用户频繁切换导致混乱 | 低 | 记录切换事件，后续分析优化 |

### 假设条件
- 用户理解切换助手是「换一个模型继续对话」，而非「重新开始」
- 后端已支持动态切换（已验证：`conversation_service.py` 第 225 行）
