# DB Schema 漂移修复需求

## 背景

2026-04-07 对 dev 环境（hci-dev namespace）进行例行检查时，发现 case-service 创建工单返回 HTTP 500，排查后定位为 **数据库 schema 漂移**：

- `schema_migrations` 表记录了 8 次迁移已执行（版本 `20260305001` ~ `20260401001`）
- 但对应的 DDL 实际未生效：10 张表缺失、15 个列缺失
- 推测原因：数据库曾从早期备份恢复，恢复时携带了 `schema_migrations` 记录，但 DDL 并非该备份时的最新状态

## 问题现象

| 现象 | 说明 |
|------|------|
| case-service HTTP 500 | 缺少 `customer` 等表，ORM 查询失败 |
| conversation-service 异常 | `conversation` 表缺少 `repeat_question_count`、`hypothesis` 等列 |
| 知识库不可用 | `kbd_entry`、`sop_document`、`sop_chunk` 表缺失 |
| 数据管道无法入库 | `raw_cases`、`knowledge_atoms`、`error_code_index` 表缺失 |
| schema_migrations 与实际 DDL 不一致 | 记录显示"已迁移"但 DDL 未执行 |

## 影响范围

- **dev 环境**：所有依赖 schema v6.2 的服务不可用
- **staging/prod 环境**：`dbMigrate.enabled: false`，未执行迁移，可能存在相同漂移

## 需求

1. **修复 dev 环境 schema**：补齐所有缺失的表和列，使 DB 达到 v6.2 规范
2. **幂等迁移脚本**：所有 DDL 使用 `IF NOT EXISTS`，任何环境可安全重复执行
3. **双迁移系统同步**：dbmate（K8s/Helm）和 Alembic（本地开发）同步新增修复版本
4. **多环境推进策略**：提供 staging/prod 分阶段操作检查清单
5. **自动化脚本**：一键修复脚本，降低人工操作风险

## 验收标准

- [x] dev 环境 `information_schema.tables` 返回 23 张 public 表（含 17 业务表 + 6 系统表）
- [x] case-service 创建工单返回 HTTP 201
- [x] dbmate 修复迁移 `20260407001_schema_repair.sql` 可幂等执行
- [x] Alembic 版本 `0005_schema_repair.py` 与 dbmate 迁移内容对齐
- [x] `scripts/ops/db-repair-env.sh` 可对任意 namespace 执行修复
- [ ] staging 环境修复并验证（Phase 2，待执行）
- [ ] prod 环境修复并验证（Phase 3，待执行）
