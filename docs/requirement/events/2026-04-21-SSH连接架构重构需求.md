# SSH 连接架构重构需求

## 变更历史
| 日期 | 版本 | 变更内容 | 关联事件文档 |
|------|------|---------|------------|
| 2026-04-21 | v1.0 | 初版 | 本文档 |

## 背景与问题

### 业务背景
现场工程师使用 HCI 排障平台时，通过 SSH 连接客户环境获取诊断信息。当前 SSH 连接由各组件自行管理，存在以下问题：

### 核心痛点

1. **SSH 连接分散管理**：
   - TerminalPanel.vue 自己创建 WebSocket 连接
   - CaseCreateDialog.vue（通过 chatStore）创建另一个 WebSocket 连接
   - 两个组件各自管理连接状态，无法共享

2. **环境数据采集时机错误**：
   - 当前设计：只在「创建工单时通过 SSH 连接」才自动采集
   - 实际场景：工单可能先创建（远程沟通），后到现场连接 SSH
   - 用户看到的提示「连接 SSH 后采集」暗示会自动采集，但实际不会

3. **Bridge 检测弹框无刷新按钮**：
   - Bridge 未运行时弹框提示下载启动
   - 用户启动后需关闭弹框重新填写信息
   - 缺少「重新检测」按钮，体验不流畅

4. **连接过程卡住**：
   - 输入正确 SSH 信息点击「连接并创建工单」会卡住
   - 无明确成功/失败反馈
   - 缺少日志记录，难以排查

5. **连接状态显示体验差**：
   - 连接中显示纯文本，层次不分
   - 用户无法快速识别当前状态和目标

### 影响范围
- 所有使用 SSH 连接功能的用户
- 工单创建流程、终端面板使用

## 需求描述

### 功能概述
将 SSH 连接从组件私有改为全局统一管理，修复连接卡住和 Bridge 检测弹框交互问题。

### 用户场景

**场景 1：创建工单时连接 SSH**
> 用户填写工单信息 → 点击「连接并创建工单」→ 系统建立 SSH 连接 → 自动采集环境数据 → 工单创建成功

**场景 2：工单已存在，连接 SSH**
> 用户打开终端面板 → 填写 SSH 信息 → 连接成功 → 自动检测环境数据状态 → 智能采集（无数据或过期时）→ 用户继续操作

**场景 3：Bridge 未运行**
> 弹框提示 Bridge 未运行 → 用户启动 Bridge → 点击「重新检测」→ 状态刷新 → 继续填写 SSH 信息

### 预期收益
- SSH 连接统一管理，减少资源浪费和重复认证
- 环境数据采集时机正确，用户体验符合预期
- 连接过程可视化，用户清楚当前状态

## 功能需求

### F1: SSH 连接全局管理
- chatStore 持有唯一的 SSH WebSocket 连接
- 所有组件通过 chatStore 发送命令、获取状态
- 连接状态全局可观测

### F2: 智能环境数据采集
- SSH 连接成功后自动检查环境数据状态
- 无数据 → 自动采集
- 有数据但过期（>5min）→ 自动采集
- 有新鲜数据 → 不动作，用户可手动刷新

### F3: Bridge 检测弹框优化
- 显示 Bridge 未运行提示
- 提供「重新检测」按钮
- 点击按钮刷新 Bridge 状态，无需关闭弹框

### F4: 连接流程日志与反馈
- WebSocket 连接各阶段添加日志
- 连接成功/失败明确反馈用户
- 超时自动失败并提示

### F5: 连接状态 UX 优化
- 连接中显示分层状态：
  - "正在连接 SSH..." - 蓝色、字体较大（主状态）
  - "目标: root@host:port" - 色、字体较小（辅助信息）

## 非功能需求

- 性能要求：SSH 连接建立 < 15秒，采集 < 30秒
- 可观测性：连接状态实时更新，日志可追溯
- 安全要求：SSH 密码不持久化，仅 session 内使用

## 验收标准

- [ ] **AS1**：TerminalPanel 和 CaseCreateDialog 共用同一个 SSH 连接（chatStore 持有）
- [ ] **AS2**：SSH 连接成功后，环境数据无或过期时自动采集
- [ ] **AS3**：Bridge 弹框有「重新检测」按钮，点击可刷新状态
- [ ] **AS4**：连接过程不卡住，有明确成功/失败反馈
- [ ] **AS5**：连接中状态显示分层（蓝色主状态 + 灰色辅助信息）

## 约束条件

- 技术约束：WebSocket 连接需兼容现有 terminal_bridge.exe
- 资源约束：不修改 backend，仅前端重构
- 依赖约束：terminal_bridge.exe 提供 WebSocket 服务

## 风险与假设

### 已知风险
- WebSocket 连接状态管理复杂，需处理断开重连
- 多组件共享连接，需明确命令输出分发

### 假设条件
- terminal_bridge.exe 正常运行时 WebSocket 可用
- 用户理解 SSH 连接是全局资源
