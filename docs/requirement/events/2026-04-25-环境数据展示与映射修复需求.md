# 环境数据展示与映射修复需求

## 变更历史
| 日期 | 版本 | 变更内容 | 关联事件文档 |
|------|------|---------|------------|
| 2026-04-25 | v1.0 | 初版，描述环境数据全链路映射错误 | 本文档 |

---

## 背景与问题

SSH 连接成功后，系统通过 `acli` 命令采集 HCI 环境数据（集群信息、告警、任务），
存入数据库并在前端"环境数据"卡片展示，同时注入到 Agent S0 Prompt 作为上下文。

现场反馈以下问题：

1. **集群信息**显示"未知"，数据库存储的是乱序键值（含命令回显、时间戳等垃圾字段）
2. **告警**全部显示为 INFO，内容和时间全为空
3. **任务**显示数字状态（3）而非中文（失败），名称为空
4. **Agent 上下文**使用以上错误数据，导致 AI 推理依据错误

---

## 核心痛点

- 用户提交工单、SSH 采集数据后，环境卡片提供错误信息，无法帮助 Agent 准确推理
- Agent 告警分析失效（全 INFO）、任务分析失效（数字状态无语义）
- 集群基本信息完全丢失（版本、节点数全部"未知"）

---

## 影响范围

| 用户路径 | 影响 |
|---------|------|
| 创建工单后查看环境数据卡片 | 显示错误/空数据 |
| Agent 分析问题时参考环境上下文 | 推理依据错误 |
| "发送到助手"功能发送的数据 | 错误数据传递给人工 |

---

## 功能需求

### F1：修复前端 cluster 解析器
- `acli platform info get` 输出格式为 `key=value` + `[section]` 的 ini 风格
- 当前按 `:` 分割导致解析完全错误
- 必须剥离命令自身的回显行（`__HCI_DONE_*` 标记行及命令本身）
- 解析后需提取：HCI 版本、集群组播地址、节点信息、authkey 等有效字段

### F2：修复后端 alert 字段映射
原始 acli alert 数据字段：
- `urgent_type`: `"紧急"` / `"普通"` → 映射为 `level: "CRITICAL"` / `"WARNING"`
- `description`: 告警详细描述 → 映射为 `content`
- `start`: 告警时间 → 映射为 `time`
- `target`: 告警目标 → 映射为 `source`

### F3：修复后端 task 字段映射
原始 acli task 数据字段：
- `process`: `"失败"` / `"完成"` / `"执行中"` → 映射为 `status`（取代数字 `status` 字段）
- `type`: 任务类型名称（"系统备份"等） → 映射为 `name`
- `start`: 开始时间 → 映射为 `time`
- `description`: 错误详情 → 映射为 `error`（截断至 200 字符）

---

## 非功能需求

- **兼容性**：已入库的错误数据重新采集后可覆盖（upsert 幂等）
- **容错**：字段缺失时返回合理默认值，不抛异常
- **可观测**：解析失败时记录 warning 日志，包含原始输出片段

---

## 验收标准

- [ ] V1：前端采集集群信息后，DB `env_data` 中含 `ip`、`name`、`zta_enable` 等来自 `acli platform info get` 的有效字段
- [ ] V2：前端展示集群信息时，不显示命令回显或时间戳等垃圾 key
- [ ] V3：告警卡片显示"紧急"映射的 CRITICAL / "普通"映射的 WARNING，不显示 INFO
- [ ] V4：告警卡片显示完整描述内容（非空）
- [ ] V5：任务卡片显示中文状态（失败/完成/执行中），不显示数字 3
- [ ] V6：任务卡片显示任务类型名称（如"系统备份"），不显示空
- [ ] V7：Agent S0 Prompt 中的告警和任务内容与页面展示一致，均为有意义的信息

---

## 约束条件

- 技术约束：后端 Python，使用 uv；前端 TypeScript/Vue3，使用 pnpm
- 不引入新的数据库字段或表结构变更（在现有 `env_data` JSONB 中存储）
- 前端解析器必须与后端提取逻辑解耦（各自只做自己职责范围内的工作）
