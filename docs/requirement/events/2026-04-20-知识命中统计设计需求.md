# 知识命中统计（hit_count）设计需求

|------|------|---------|------------|

## 背景与问题

### 业务背景

HCI 智能排障平台的知识库由三类资产组成：

- **`kb_category`**：故障分类树（235 节点，5 个域），是意图识别的锚点
- **`sop_document`**：SOP 排障手册（一棵故障子树的结构化排障流程）
- **`kbd_entry`**：KBD 知识条目（一个真实的客户历史案例，对应故障树的叶节点）

### 故障树模型（核心领域概念，指导整个设计）

故障诊断的本质是在一棵多叉树上的导航：

```
故障树（多叉树）:
  L1: 虚拟机
    L2: 开机失败
      L3: 内存不足    ← leaf → 对应一条 KBD（历史案例）
      L3: 磁盘故障    ← leaf → 对应一条 KBD（历史案例）
      L3: 新问题      ← 待添加叶子的父节点出口（零 KBD）
    L2: 网络中断
      ...

SOP = 覆盖一棵故障子树的结构化排障流程
KBD = 树上某一片叶子的历史实例（真实案例）
case = 一次排障 = 从根进入，找到唯一叶子（或父节点出口）
```

因此，正常情况下：
- `case : kb_category` = 1 : 1（S0 意图识别确认唯一分类）
- `case : sop_document` = 1 : 0~1（有 SOP 则命中，无则未收录）
- `case : kbd_entry` = 1 : 0~1（S4 根因确认后对应唯一叶节点，未收录则 NULL）

### 核心痛点

| 问题 | 现状 | 影响 |
|------|------|------|
| `kb_category.hit_count` 统计有缺陷 | 按 conversation 计数，断线重连会虚增（1 个 case 多次 conversation 各自 +1）| 热门分类数据失真 |
| `sop_document.hit_count` 完全未实现 | 前端 `hit_count \|\| 0` 容错，全部显示 0 | 无法知道哪些 SOP 被实际使用 |
| `kbd_entry.hit_count` 完全未实现 | 同上 | 无法评估 KBD 质量和使用价值 |
| SOP 使用记录从未落库 | knowledge_retriever 注入后 sop_document_id 直接丢失 | 无法追溯某次排障用了哪个 SOP |
| 根因与 KBD 无关联 | diagnostic_item.root_cause 只存文字，不存 KBD 引用 | 知识闭环缺失，新问题无法沉淀为 KBD |

## 需求描述

### 功能概述

以「有多少个唯一 case 命中了该知识资产」为统一语义，正确实现三类知识资产的命中统计，并在 admin-UI 的分类管理页面正确展示。同时为知识质量持续改进奠定数据基础。

### 用户场景

1. **运营人员**查看某分类的命中次数，判断该故障类型的出现频率 → 决定是否优先补充对应 SOP/KBD
2. **运营人员**在分类详情页看到某条 KBD 的命中次数为 0 → 判断该条目质量差或领域冷门 → 考虑归档
3. **运营人员**在分类详情页看到某 SOP 命中次数高但 composite_score 低 → 该 SOP 需要优化
4. **管理员**在工单详情页，发现 AI 自动关联的 KBD 不准确 → 手动修正 `resolved_kbd_entry_id`

### 预期收益

- 知识资产使用价值可量化，指导运营投入方向
- 修复现有 `kb_category.hit_count` 虚增 Bug，数据可信
- 为知识质量评估（hit_count × composite_score 关联分析）提供基础

## 功能需求

### FR-01：统一 hit_count 语义

所有三类知识资产的 hit_count 语义统一为：**有多少个唯一 case_id 命中了该知识资产**。

物化列 `hit_count` 可通过以下 SQL 随时校验：
```sql
SELECT COUNT(DISTINCT case_id) FROM conversation WHERE category_id = 'vm-003';
SELECT COUNT(DISTINCT case_id) FROM conversation WHERE sop_document_id = 5;
SELECT COUNT(DISTINCT case_id) FROM conversation WHERE resolved_kbd_entry_id = 123;
```

### FR-02：修复 kb_category.hit_count

在 S0→S1 转换时，增加 case 级去重检查：同一 case 下若已有其他 conversation 写入了相同 `category_id`（即断线重连场景），跳过计数，不重复 +1。

### FR-03：实现 sop_document.hit_count

1. `conversation` 表增加 `sop_document_id` 字段（nullable FK）
2. S0→S1 转换后，按 `category_id` 查已发布 SOP，命中则写入 `conversation.sop_document_id`
3. 同 case 级去重后 `sop_document.hit_count + 1`

### FR-04：实现 kbd_entry.hit_count

1. `conversation` 表增加 `resolved_kbd_entry_id` 字段（nullable FK）
2. S4 根因确认时，AI 结构化输出携带 `kbd_entry_id`（从 S2/S3 检索候选中选一条，或 null）
3. 写入 `conversation.resolved_kbd_entry_id`，去重后 `kbd_entry.hit_count + 1`

### FR-05：admin-UI 可编辑 resolved_kbd_entry_id

管理员可在工单详情（或分类管理页）手动修正 AI 自动推断的 `resolved_kbd_entry_id`，同时重新计算相关 hit_count（加新减旧）。

## 非功能需求

- **原子性**：hit_count 增量使用 `UPDATE ... SET hit_count = hit_count + 1`，禁止 SELECT+UPDATE 模式
- **去重正确性**：通过 `conversation.case_id + category_id/sop_document_id/resolved_kbd_entry_id` 在数据库层面保证，无需应用层维护 set
- **可追溯性**：所有命中记录可通过 `conversation` 表反查原始 case，hit_count 可随时校验
- **可观测性**：hit_count 写入操作必须有结构化日志
- **幂等性**：同一 conversation 多次触发相同命中，不重复计数

---

## 验收标准

- [ ] `kb_category.hit_count`：同一 case 断线重连后只算一次
- [ ] `kb_category.hit_count`：可通过 `COUNT(DISTINCT case_id)` SQL 校验物化列准确性
- [ ] `sop_document.hit_count`：SOP 命中时正确 +1，前端不再显示全 0
- [ ] `kbd_entry.hit_count`：S4 根因确认后正确关联并计数
- [ ] `conversation.sop_document_id`：S1 命中 SOP 时正确写入
- [ ] `conversation.resolved_kbd_entry_id`：S4 确认后正确写入，admin-UI 可编辑
- [ ] 所有 hit_count 物化列与统计 SQL 结果一致（允许秒级延迟）

## 约束条件

- 技术约束：Python 3.12 / FastAPI / SQLAlchemy async / PostgreSQL 15 / Vue 3 + Element Plus
- Schema 变更通过 Atlas migrate diff 管理（`desired_schema.sql` 为权威来源）
- 禁止在 SSE 流程关键路径上增加同步 DB 调用（使用 `asyncio.create_task` 异步执行）

## 风险与假设

| 风险 | 影响 | 缓解 |
|------|------|------|
| AI 结构化输出 kbd_entry_id 不准确 | 中 | admin-UI 提供手动修正入口（FR-05）|
| S4 阶段未总是触发（对话中途关闭）| 中 | `resolved_kbd_entry_id` 允许 NULL，不影响其他计数 |
| hit_count 物化列与 SQL 统计短暂不一致 | 低 | 提供校验 SQL，允许秒级延迟 |
