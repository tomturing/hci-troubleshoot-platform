# 项目交付标准化

> 本文档基于业界最佳实践，对当前项目的发布交付体系进行完整分析，包含：参考标准、差距分析、整改计划。
>
> **项目背景**：开发环境（多台）WSL Ubuntu 24.04；云上生产（2台）Ubuntu 24.04 16C32G4T；代码仓库 GitHub Private；编排 K3s + Helm Chart（hci-platform）

---

## 一、业界最佳实践参考标准

### 1.1 全生命周期流程

```
开发者本机
  ├─ feature/* 分支开发
  ├─ pre-commit hooks：format/lint/type-check（<5秒本地阻断）
  └─▶ PR 提交 → GitHub
         │
         ▼
【CI 流水线（GitHub Actions）】
  P0 质量门禁（并行，全通过才放行）
   ├─ lint + format check（ruff / eslint/prettier）
   ├─ 单元测试 + 覆盖率 ≥ 70%（带缓存，<3分钟）
   ├─ Helm lint + template 渲染验证
   └─ 文档/变更同步检查
  P1 集成测试（Docker Compose 临时环境，可选）
   └─ E2E 核心链路冒烟（<10分钟）
         │
         │ PR 通过审核 → 合并 main
         ▼
【CD 构建阶段（main push 触发）】
  镜像构建 + 推送（ghcr.io，多服务矩阵）
   ├─ 多阶段 Dockerfile（builder + runtime 分层）
   ├─ 镜像标签：<yyyyMMdd>-<HHmm>-<sha7>（不可变）
   ├─ 镜像安全扫描（Trivy，高危阻断）
   └─ 输出 release-metadata.json
         │
         ▼
【GitOps 环境晋级（双仓模型）】
  应用仓库（代码）→ 自动 PR 到环境仓库
  environments/
   dev/values.yaml    ← 自动合入
   staging/values.yaml ← 手动审批晋级
   prod/values.yaml    ← 发布窗口 + 审批
  ArgoCD 监听环境仓库 → 同步集群状态
         │
         ▼
【发布后验证（自动）】
  健康检查轮询
  冒烟测试（关键接口）
  30分钟观察窗口（自动告警）
  告警：错误率↑ / P95延迟↑ / Pod非预期重启
         │
  正常 → 完成    /    异常 → 自动触发回滚
```

### 1.2 关键质量控制节点

| 节点 | 机制 | 阻断条件 |
|------|------|---------|
| 本地提交前 | pre-commit hooks | lint/format 失败 |
| PR 合入前 | CI P0 门禁 | 任一失败则 PR 不可合入 |
| 镜像构建后 | Trivy 安全扫描 | HIGH/CRITICAL 漏洞 |
| 发布前 | preflight checklist | 手动签字确认 |
| 发布后 | 自动冒烟 + 观察窗口 | 阈值触发自动回滚 |

### 1.3 分支策略（主干开发）

```
main（保护）─────────────────────────────────────►
  │          │
  ◄─ feature/xxx    ◄─ hotfix/xxx
```

- `feature/*`：从 main 切，完成后 PR 回 main
- `hotfix/*`：紧急修复，从 main 切，合并后直接打 tag
- 无 develop 分支（小团队，主干开发更简洁）
- `release/vX.Y`：里程碑归档标签，不作为发布来源

### 1.4 版本与标签规范

```
镜像标签：20260319-1430-a3f5c2d（日期-时间-sha7，不可变唯一）
Git tag：v2.1.0（里程碑/外部发布时打）
Helm Chart version 与 appVersion 独立递增
```

### 1.5 密钥管理标准

| 环境 | 方式 | 禁止 |
|------|------|------|
| 开发 | `.env.local`（gitignore） | 写入代码 |
| CI | GitHub Secrets（组织级，分环境） | 写入 workflow 文件 |
| 生产 | K8s Sealed Secrets 或 Vault | 写入 values.yaml |

## 三、整改计划

### Sprint 1：堵住高风险缺口（优先级：必做）

#### S1-T1：接入 ghcr.io + CI 镜像构建 Job（GAP-03）

在 `.github/workflows/ci.yml` 中增加 `build-and-push` job：

```yaml
build-and-push:
  needs: [unit-tests, integration-tests]
  runs-on: ubuntu-latest
  permissions:
    contents: read
    packages: write
  strategy:
    matrix:
      service: [api-gateway, case-service, conversation-service, scheduler-service, kb-service]
  steps:
    - uses: actions/checkout@v4
    - uses: docker/login-action@v3
      with:
        registry: ghcr.io
        username: ${{ github.actor }}
        password: ${{ secrets.GITHUB_TOKEN }}   # 内置，无需额外配置
    - name: 生成镜像标签
      id: tag
      run: echo "tag=$(date +%Y%m%d-%H%M)-${GITHUB_SHA::7}" >> $GITHUB_OUTPUT
    - uses: docker/build-push-action@v5
      with:
        context: backend/${{ matrix.service }}
        push: true
        tags: |
          ghcr.io/${{ github.repository }}/${{ matrix.service }}:${{ steps.tag.outputs.tag }}
          ghcr.io/${{ github.repository }}/${{ matrix.service }}:latest
```

> **使用 `ghcr.io`（GitHub Container Registry）：私有仓库零成本，内置 `GITHUB_TOKEN` 鉴权。**

#### S1-T2：Trivy 安全扫描（GAP-02）

在构建 Job 后追加：

```yaml
    - uses: aquasecurity/trivy-action@master
      with:
        image-ref: ghcr.io/${{ github.repository }}/${{ matrix.service }}:${{ steps.tag.outputs.tag }}
        exit-code: '1'           # HIGH/CRITICAL 阻断构建
        severity: HIGH,CRITICAL
        format: sarif
        output: trivy-results.sarif
    - uses: github/codeql-action/upload-sarif@v3
      if: always()
      with:
        sarif_file: trivy-results.sarif
```

#### S1-T3：测试覆盖率基线（GAP-10）

```diff
- uv run pytest tests/unit/ ... --cov=backend
+ uv run pytest tests/unit/ ... --cov=backend --cov-fail-under=60
```

#### S1-T4：数据库 Alembic 版本化迁移（GAP-08）

```bash
uv add alembic
uv run alembic init backend/shared/migrations
# 将现有 init_schema.sql 转为初始 migration
uv run alembic revision --autogenerate -m "initial_schema"
```

### Sprint 3：工程成熟度（优先级：中）

#### S3-T1：pre-commit hooks（GAP-01）

```yaml
# .pre-commit-config.yaml
repos:
  - repo: https://github.com/astral-sh/ruff-pre-commit
    rev: v0.4.0
    hooks:
      - id: ruff
      - id: ruff-format
  - repo: https://github.com/pre-commit/mirrors-prettier
    rev: v4.0.0
    hooks:
      - id: prettier
        types_or: [ts, vue, css]
```

#### S3-T2：AlertManager 告警规则（GAP-09）

在 `deploy/observability/` 下添加 PrometheusRule，通知渠道接飞书/企业微信 Webhook。

#### S3-T3：staging namespace 软隔离（GAP-06）

利用第二台云主机创建 `hci-staging` namespace，或在同一集群中用不同 namespace 实现软隔离。

#### S3-T4：GitHub 分支保护规则（WIP-06）

在 GitHub → Settings → Branches 配置 main 分支：require PR + status checks + no force push。
