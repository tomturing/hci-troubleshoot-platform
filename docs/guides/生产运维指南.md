# 生产环境部署手册（Kubernetes / Ubuntu 22.04.5 LTS）

更新时间：2026-03-02  
适用环境：云端 Ubuntu 22.04.5 LTS，16C/32G，单机 K8s（推荐 K3s）

## 1. 部署原则（生产基线）

- 编排统一使用 Kubernetes（不再以 Docker Compose 作为生产入口）。
- 最小权限：宿主机与容器都尽量非 root。
- 配置分层：`values.yaml`（默认）+ `values-prod.yaml`（生产基线）+ 私有 override（密钥与域名）。
- 镜像不可变：使用版本 tag（如 `2026.03.02-1`），避免 `latest`。
- 可回滚：所有发布通过 `helm upgrade --install`，保留 revision，支持 `helm rollback`。

## 6. 镜像策略（生产推荐 vs 单机快速）

## 6.1 推荐：私有镜像仓库（生产最佳实践）

1. 构建并推送镜像（CI/CD 或手工）：

- `hci-api-gateway`
- `hci-case-service`
- `hci-conversation-service`
- `hci-scheduler-service`
- `hci-customer-ui`
- `hci-admin-ui`
- `hci-openclaw`（若启用 chart 内 openclaw）

2. 使用固定 tag（示例 `2026.03.02-1`），不要用 `latest`。

3. 在 Helm override 设置：

- `global.imageRegistry: "registry.example.com/"`
- 各服务 `image.tag: "2026.03.02-1"`
- `global.imagePullPolicy: IfNotPresent`（已缓存）或 `Always`（强制拉取）

## 6.2 单机 K3s 快速方式（无仓库）

可直接用脚本本地构建并导入 K3s containerd：

```bash
cd /srv/hci/repos/AIBot/hci-troubleshoot-platform
bash scripts/k3s-build.sh
```

说明：该方式适合单机环境与快速迭代，不适合多节点生产集群。

## 6.3 两种方式的区别（你当前问题的直接答案）

| 维度 | 私有镜像仓库（推荐） | 单机本地导入 K3s |
|---|---|---|
| 适用规模 | 单机/多节点/跨环境统一 | 仅单机或临时测试 |
| 可重复部署 | 高（节点按 tag 拉取同一镜像） | 低（每台机器都要手工导入） |
| 运维复杂度 | 前期需搭建仓库，后期最省事 | 前期简单，节点变多后维护成本陡增 |
| 灰度/回滚 | 标准化（tag + Helm revision） | 弱（镜像分发不可追踪） |
| CI/CD 集成 | 强（构建后直接推仓库） | 弱（需额外导出/导入步骤） |
| 安全审计 | 强（可做镜像扫描、签名） | 弱（离散导入难审计） |

结论：

- 你当前是单 node，也建议直接走“私有仓库”主路径，后续扩展多 nodes 不需要重做镜像分发体系。
- 本地导入保留给离线测试或临时排障。

## 12. 单 node 到多 nodes 的平滑过渡方案（重点）

你当前线上是单 node，建议按以下阶段迁移，做到“业务不停或最小中断”：

### 阶段 A：先在单 node 上完成“多节点前置改造”

1. 镜像统一切到私有仓库（固定 tag），停止依赖本地导入。
2. 所有发布走 `helm upgrade --install`，并沉淀 `values-prod.override.yaml`。
3. 将业务服务副本策略与资源限制固化（已在 chart 有基础值）。
4. 对 Stateful 组件评估存储方案：
   - 若继续自建 PostgreSQL/Redis：准备可跨节点存储（如 Longhorn/Ceph）。
   - 更推荐：迁移到托管 PostgreSQL/Redis（把状态外置，最平滑）。

### 阶段 B：扩容集群为多节点（先加节点，不改业务拓扑）

1. 新增 worker 节点加入 K3s 集群。
2. 保持当前 Helm release 不变，先观察调度与网络。
3. 验证：
   - `kubectl get nodes/pods -o wide`
   - 核心 API 健康检查与联调脚本通过。

### 阶段 C：提升高可用（渐进式）

1. 无状态服务逐步扩副本（如 `apiGateway/case/conversation` 从 1->2）。
2. 加 `PodDisruptionBudget`，避免维护期间全部实例同时中断。
3. 增加 `topologySpreadConstraints` / `podAntiAffinity`，让副本分散到不同节点。
4. 再引入 HPA（基于 CPU/内存或自定义指标）。

### 阶段 D：状态服务高可用

两条路线二选一：

- 路线 1（推荐）：PostgreSQL/Redis 托管化（RDS/云 Redis）。
- 路线 2：K8s 内自建高可用（需要成熟 Operator + 跨节点存储）。

注意：当前默认 `local-path` 存储适合单节点，不是多节点高可用最终形态。

### 迁移期间保持“平滑”的关键控制点

1. 不改 release 名称（持续使用 `hci-platform`）。
2. 每次只改一类变量（先镜像，再副本，再调度策略，再状态层）。
3. 每次变更后执行：
   - `helm upgrade --install ... --wait`
   - `scripts/ops/k3s-verify.sh`
   - 关键联调（`scripts/test_multi_assistant.sh`）
4. 任一步异常，立即 `helm rollback` 回上一个 revision。

## 14. 故障排查手册

> 本节记录从首次部署到服务完全可用过程中遇到的典型问题，按类别分类。
> **环境问题** 和 **配置问题** 提供人工操作注意事项；**项目 Bug** 已在代码中修复。

### 一、环境问题（需人工处理）

> 这类问题与运行环境强相关，无法通过代码彻底解决，每次在新环境部署时必须人工核对。

### E-01：容器 Runtime 类型影响 Promtail 日志采集

**现象：** Grafana Explore 查询 Loki 无任何数据，`promtail_files_active_total = 0`。

**根因：** k3s 使用 Docker runtime 时，Pod 日志文件是软链接，实际内容在 `/var/lib/docker/containers/`。  
Promtail 挂载了 `/var/log/pods` 但未挂载 `/var/lib/docker/containers`，软链接解析失败。

**与 containerd 的区别：**

| 环境 | 日志路径 | Pipeline Stage | 额外挂载 |
|---|---|---|---|
| k3s 默认 (containerd) | `/var/log/pods` 直接 | `cri: {}` | 无需额外挂载 |
| Docker runtime (裸机/旧 k8s) | `/var/log/pods` → 软链接 `/var/lib/docker/containers/` | `docker: {}` | 需挂载 `/var/lib/docker/containers` |

**注意事项：**
1. 部署前确认 runtime：`kubectl get nodes -o jsonpath='{.items[*].status.nodeInfo.containerRuntimeVersion}'`
2. 在 `deploy/env/platform.env` 中设置 `HCI_K8S_CONTAINER_RUNTIME=docker` 或 `containerd`
3. 本项目 Helm chart 已根据此变量自动选择正确配置（v2.1.0 修复）

---

### E-02：WSL2 下 hosts 文件需手动配置

**现象：** 浏览器无法访问 `http://app.hci.local`、`http://admin.hci.local` 等域名。

**根因：** WSL2 的 IP 不固定，且 Windows hosts 文件和 WSL hosts 文件需要分别配置。

**注意事项：**
1. 获取 Traefik 的 LoadBalancer IP：
   ```bash
   kubectl get svc -n kube-system traefik -o jsonpath='{.status.loadBalancer.ingress[0].ip}'
   ```
2. 在 **Windows** `C:\Windows\System32\drivers\etc\hosts` 中添加：
   ```
   <TRAEFIK_IP>  api.hci.local
   <TRAEFIK_IP>  app.hci.local
   <TRAEFIK_IP>  admin.hci.local
   <TRAEFIK_IP>  grafana.hci.local
   ```
3. 在 **WSL** `/etc/hosts` 中添加相同内容
4. WSL 重启后 IP 可能变化，需重新执行上述步骤
5. 使用 nip.io 可避免此问题：设置 `HCI_DOMAIN=<TRAEFIK_IP>.nip.io`

---

### E-03：Postgres 密码首次设置后不可更改

**现象：** 重新部署后服务无法连接数据库，错误 `password authentication failed`。

**根因：** PostgreSQL 密码存储在 PVC（持久化存储）的 pg_auth 文件中。Helm upgrade 只更新 K8s Secret，不会修改数据库内部的密码。

**注意事项：**
1. **首次部署前确定密码，之后不要修改**
2. 如必须修改密码，需要先进入 Pod 手动修改：
   ```bash
   kubectl exec -n hci-troubleshoot postgres-0 -- \
     psql -U hci_admin -c "ALTER USER hci_admin PASSWORD 'new-password';"
   # 然后更新 Helm values 和 Secret
   ```
3. 如重建 PVC，所有数据丢失，相当于全新部署

---

### E-04：Traefik Ingress 入口点与 TLS 配置必须匹配

**现象：** 所有服务返回 404，Traefik 日志显示无路由匹配。

**根因：** `ingress.annotations.traefik.ingress.kubernetes.io/router.entrypoints: websecure` 要求 HTTPS，但未配置 TLS 证书时 Traefik 无法路由。

**注意事项：**
1. **本地/开发环境**：使用 `web`（HTTP）
   ```yaml
   # .local/values-prod.override.yaml
   ingress:
     annotations:
       traefik.ingress.kubernetes.io/router.entrypoints: web
   ```
2. **生产环境**：使用 `websecure`（HTTPS），需同时配置：
   - Let's Encrypt / 自签名证书
   - Traefik TLS resolver
   - Ingress tls 字段指定 Secret
3. 在 `deploy/env/platform.env` 中通过 `HCI_INGRESS_ENTRYPOINT` 统一管理

---

### E-05：OpenClaw 宿主机配置文件 token 优先级高于 K8s Secret

**现象：** conversation-service 调用 openclaw 返回 HTTP 401 Unauthorized。

**根因：** openclaw 加载 config 的优先级：`~/.openclaw/openclaw.json` > 环境变量。  
宿主机 `openclaw.json` 中 `gateway.auth.token` 与 K8s Secret 中 `OPENCLAW_GATEWAY_TOKEN` 不一致时，conversation-service 使用错误的 token。

**注意事项：**
1. 部署时检查宿主机 openclaw.json：
   ```bash
   cat ~/.openclaw/openclaw.json | python3 -c "import sys,json; d=json.load(sys.stdin); print(d['gateway']['auth']['token'])"
   ```
2. **确保 `secrets.env` 中的 `HCI_SECRET_OPENCLAW_TOKEN` 与上面的值完全一致**
3. openclaw 重新配置后需要 `kubectl rollout restart deploy/conversation-service`
4. 详见 `deploy/env/platform.env` 中 `HCI_OPENCLAW_TOKEN_NOTE` 字段说明

### 二、配置问题（需人工处理）

> 这类问题在不同部署环境中配置值不同，需要在 `deploy/env/` 目录下设置正确的值，然后通过 Helm 统一下发。

### C-01：Grafana 登录后重定向到 localhost

**现象：** 访问 `http://grafana.hci.local` 后，登录成功，但页面重定向到 `http://localhost:3000/...`

**根因：** Grafana 未配置 `GF_SERVER_DOMAIN` 和 `GF_SERVER_ROOT_URL`，默认使用 localhost 作为重定向 URL。

**配置：** 已在 Helm chart 中修复（通过 `global.domain` 自动推导）。  
如部署新环境域名不同，在 `deploy/env/platform.env` 中修改：
```env
HCI_GRAFANA_DOMAIN="grafana.your-domain.com"
HCI_GRAFANA_ROOT_URL="https://grafana.your-domain.com/"
```

**操作快捷修复（不重新部署时）：**

```bash
kubectl set env deploy/grafana -n hci-observability \
  GF_SERVER_DOMAIN=grafana.hci.local \
  GF_SERVER_ROOT_URL=http://grafana.hci.local/
```

### C-02：Grafana admin 密码首次部署后重置

**现象：** 首次登录 Grafana 时提示用户名或密码错误。

**根因：** Grafana 将 admin 密码存储在其数据库 (SQLite/PVC) 中。Helm upgrade 不会更新已存储的密码。

**操作：**
```bash
# 查找 Grafana Pod
GRAFANA_POD=$(kubectl get pods -n hci-observability -l app.kubernetes.io/name=grafana -o jsonpath='{.items[0].metadata.name}')
# 重置密码
kubectl exec -n hci-observability $GRAFANA_POD -- \
  grafana cli admin reset-admin-password 'your-new-password'
```

注意：每次修改 `secrets.grafanaAdminPassword`，如果 Grafana PVC 中已有数据，必须执行上述命令手动同步。

### C-03：Openclaw 密钥与模型 API Key 设置

**现象：** 对话返回 `GATEWAY_ERROR: 上游服务暂时不可用 status 422`。

**根因：** `secrets.zaiApiKey` 为空，openclaw 无有效 API Key 无法调用上游 ZAI 模型。

**配置：**
```env
# deploy/env/secrets.env
HCI_SECRET_ZAI_API_KEY="your-zai-api-key"  # 从 https://open.bigmodel.cn/ 获取，注入为 Secret 中的 OPENCLAW_API_KEY
```

### C-04：多环境 imagePullPolicy 设置不一致

**现象：** CI/CD 部署后仍使用旧镜像；本地部署每次都尝试从网络拉取。

**规则：**

| 环境 | imagePullPolicy | 原因 |
|---|---|---|
| 本地开发 | `IfNotPresent` | 镜像由本地 docker build 导入，不走仓库 |
| CI/CD | `Always` | 确保拉取最新镜像 |
| 生产 | `Always` | 同上 |

**配置：** 在 `deploy/env/platform.env` 中设置 `HCI_IMAGE_PULL_POLICY`  
本地 override 已设为 `IfNotPresent`（见 `.local/values-prod.override.yaml`）

### C-05：Tempo 内存资源下限

**现象：** Tempo Pod 反复 OOMKilled，状态一直是 0/1 Running，链路追踪数据无法写入。

**根因：** 默认配置 `memory limit: 256Mi` 不够 Tempo 正常运行。

**配置：** `deploy/env/platform.env` 中 `HCI_TEMPO_MEMORY_LIMIT="512Mi"`（已修复）

**生产环境建议：** >= 1Gi，高流量时 >= 2Gi。

### 四、统一配置管理方案

### 配置文件层级

```
deploy/env/
  platform.env          ← 所有非敏感变量（提交 git）
  secrets.env.example   ← 敏感变量模板（提交 git）
  secrets.env           ← 本地真实敏感值（不提交 git）
  README.md             ← 使用说明

deploy/helm/hci-platform/
  values.yaml           ← Helm 默认值（提交 git）

.local/
  values-prod.override.yaml  ← 个人本地覆盖（不提交 git）
```

### 变量修改流程

```
修改 deploy/env/platform.env 或 secrets.env
         ↓
helm upgrade --install 重新部署（引用 secrets.env）
         ↓
kubectl rollout status 验证
```

### 快速检查命令

```bash
# 检查 openclaw token 一致性

# 检查 Grafana 环境变量
kubectl get deploy grafana -n hci-observability -o jsonpath='{.spec.template.spec.containers[0].env}' | python3 -m json.tool | grep -E "DOMAIN|ROOT_URL"

# 检查所有 Pod 就绪状态
kubectl get pods -n hci-troubleshoot && kubectl get pods -n hci-observability
```
