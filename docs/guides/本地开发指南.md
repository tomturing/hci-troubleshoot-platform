## 1. 开发环境搭建

### 1.1 系统要求

- **操作系统**: Linux, macOS, Windows (with WSL2)
- **Python**: 3.12+
- **Node.js**: 18+
- **go**: 1.24+
- **Docker**: 24+
- **Docker Compose**: 2.20+
- **k3s**: 1.34+ (宿主机需额外安装 `socat` 包以支持端口转发: `sudo apt-get install -y socat`)
- **helm**: 3.14+
- **PostgreSQL**: 15
- **Redis**: 7

> **🚨 关键网络警告 (针对 WSL / 本地测试)**:
> 
> 如果您在本地开发测试期间遇到 `nip.io` 等外网域名解析为 `198.18.x.x` 或其他非预期内网 IP 导致 404 错误，
> **这是因为您当前的主机启用了 Clash / V2ray 等网络代理软件，且其默认开启了 Fake-IP (伪装 IP) 接管了全局或局域网的 DNS 解析。**
> 测试 Kubernetes Ingress（如 `*.nip.io`）时，**必须临时关闭代理软件**，或为其配置绕过相关的局域网段。

### 4.4 WebSocket 集成

```typescript
// src/api/websocket.ts
export class WebSocketClient {
  private ws: WebSocket | null = null
  private clientId: string
  private caseId: string

  constructor(clientId: string, caseId: string) {
    this.clientId = clientId
    this.caseId = caseId
  }

  connect() {
    const wsUrl = `ws://localhost:8000/ws/${this.clientId}?case_id=${this.caseId}`
    this.ws = new WebSocket(wsUrl)

    this.ws.onopen = () => {
      console.log('WebSocket connected')
    }

    this.ws.onmessage = (event) => {
      const data = JSON.parse(event.data)
      this.handleMessage(data)
    }

    this.ws.onerror = (error) => {
      console.error('WebSocket error:', error)
    }

    this.ws.onclose = () => {
      console.log('WebSocket closed')
    }
  }

  sendMessage(content: string) {
    if (this.ws && this.ws.readyState === WebSocket.OPEN) {
      this.ws.send(JSON.stringify({
        type: 'user_message',
        case_id: this.caseId,
        content
      }))
    }
  }

  private handleMessage(data: any) {
    // Handle different message types
    switch (data.type) {
      case 'system':
        console.log('System message:', data)
        break
      case 'assistant_message':
        console.log('AI response:', data.content)
        break
      case 'command':
        console.log('Command:', data.command)
        break
    }
  }

  disconnect() {
    if (this.ws) {
      this.ws.close()
      this.ws = null
    }
  }
}
```

## 6A. 部署环境切换

> **Docker Compose 和 K3s 不能同时运行**（端口和数据库冲突），需先停一个再起另一个。两套环境的数据（PostgreSQL 卷）相互独立。

### 从 K3s 切回 Docker Compose

```bash
# 1. 停止 K3s
sudo systemctl stop k3s

# 2. 启动业务栈
docker compose -f deploy/docker/docker-compose.yml up -d

# 3. 启动可观测性栈
docker compose -f deploy/observability/docker-compose-obs.yml up -d

# 访问地址：
#  Customer UI: http://localhost:3001
#  Admin UI:    http://localhost:3002
#  Grafana:     http://localhost:3000 (admin/admin)
```

### 从 Docker Compose 切回 K3s

```bash
# 1. 停止 Docker Compose
docker compose -f deploy/docker/docker-compose.yml down
docker compose -f deploy/observability/docker-compose-obs.yml down

# 2. 启动 K3s
sudo systemctl start k3s
sudo chmod 644 /etc/rancher/k3s/k3s.yaml

# 访问地址（通过 Traefik Ingress）：
#  Customer UI: http://<WSL2_IP>/
#  Admin UI:    http://<WSL2_IP>/admin/
```

### WSL 重启后恢复 K3s

WSL `shutdown` 后所有进程终止，K3s 需要手动重启：

```bash
# 每次 WSL 重启后执行
sudo systemctl start k3s
# kubeconfig 权限已通过 --write-kubeconfig-mode 644 自动处理

# 等待 30 秒后所有 Pod 自动恢复
kubectl get pods -n hci-troubleshoot    # 9 pods Running
kubectl get pods -n hci-observability   # 4 pods Running
```

> **提示**：如需 K3s 随 WSL 启动自动运行，执行 `sudo systemctl enable k3s`（需确认 `/etc/wsl.conf` 中 `[boot] systemd=true`）。

### 访问地址速查表

### 8.3 应用优化

#### 使用连接池

```python
# database.py
from sqlalchemy import create_engine
from sqlalchemy.pool import QueuePool

engine = create_engine(
    DATABASE_URL,
    poolclass=QueuePool,
    pool_size=20,
    max_overflow=0,
```

#### 使用缓存

```python
import redis
from functools import wraps

redis_client = redis.Redis(host='localhost', port=6379, decode_responses=True)

def cache(ttl=300):
    def decorator(func):
        @wraps(func)
        async def wrapper(*args, **kwargs):
            cache_key = f"{func.__name__}:{args}:{kwargs}"
            cached = redis_client.get(cache_key)
            if cached:
                return json.loads(cached)
            
            result = await func(*args, **kwargs)
            redis_client.setex(cache_key, ttl, json.dumps(result))
            return result
        return wrapper
    return decorator
```

### 9.2 WebSocket 连接失败

### 9.4 K3s 本地部署：Clash TUN / nip.io 无法访问

**问题**: 浏览器访问 `http://<WSL_IP>.nip.io/` 返回 `ERR_EMPTY_RESPONSE`，nslookup 显示解析到 `198.18.x.x`

**根因**: Clash Verge / ClashX 开启了 **TUN 模式 + Fake-IP**，将 `nip.io` 域名的 DNS 查询劫持并返回代理内部 IP（`198.18.0.0/15` 段），浏览器连接到此假 IP 没有响应。

**快速诊断**:
```bash
python3 -c "import socket; print(socket.gethostbyname('1.2.3.4.nip.io'))"
# 正常: 1.2.3.4
# 异常: 198.18.x.x（Clash Fake-IP）
```

**修复方案**（三选一）:

| 方案 | 步骤 | 持久性 |
|------|------|--------|
| A: Clash 直连规则 | 在 Clash 订阅的 `prepend-rules` 添加 `DOMAIN-SUFFIX,nip.io,DIRECT` | 持久 ✅ |
| B: Windows hosts | `C:\Windows\System32\drivers\etc\hosts` 添加 `<WSL_IP>  <WSL_IP>.nip.io`，然后 `ipconfig /flushdns` | 持久 ✅ |
| C: 绕过 nip.io | 部署时加 `--no-nipio` 参数，直接通过裸 IP 访问 | 无需 DNS |

**参考**: `docs/21_本地K3s部署日志.md` DUAL-005

### 9.5 K3s 本地部署：helm upgrade 意外删除 postgres

**问题**: 手动执行 `helm upgrade` 后 postgres-0 Pod 消失，数据库连接全部断开

**根因**: `hci-platform` Chart 默认 `dataLayer.manage: false`，若历史部署使用了 `manage: true`，再次 upgrade 时不传该参数会导致 Helm 将已有 postgres StatefulSet 视为孤立资源并删除。

**⚠️ 高风险操作规则**:
```bash
# ❌ 危险：默认值可能删除数据库
helm upgrade hci-platform ...

# ✅ 安全：始终显式传入当前状态
helm upgrade hci-platform ... --set dataLayer.manage=true

# ✅ 最安全：使用部署脚本（自动检测当前状态）
bash scripts/ops/k3s-deploy-dualrepo.sh
```

**恢复方法**（数据不丢失，PVC 有 Retain 策略）:
```bash
bash scripts/ops/k3s-deploy-dualrepo.sh --datalayer-manage true
```

**参考**: `docs/21_本地K3s部署日志.md` DUAL-009

### 9.6 K3s 本地部署：创建工单 500 / 数据库字段不存在

**问题**: 页面操作报 500，后台日志 `UndefinedColumnError: column "xxx" does not exist`

**根因**: 本地 Postgres 从旧 `init_schema.sql` 初始化（或 PVC 复用了旧版 schema），而代码已累积多个版本的迁移字段。

**修复**: 部署脚本已集成自动迁移（步骤 8.5），每次 `k3s-deploy-dualrepo.sh` 都会按序执行所有迁移脚本，无需手动操作。

手动执行方式:
```bash
for sql in database/migrate_evaluation_v1.sql database/migrate_kb_v3.sql \
           database/migrate_p4_v1.sql database/migrate_tool_audit_log.sql \
           database/migrate_conversation_p4_v1.sql; do
  echo "--- $sql ---"
  kubectl exec -i -n hci-troubleshoot postgres-0 -- \
    psql -U hci_admin -d hci_troubleshoot < "$sql"
done
```

### 9.7 K3s 本地部署：AI 响应出现错误（openclaw 未部署）

**问题**: 用户发送消息后提示 "AI 响应出现错误，请稍后重试"

**诊断**:
```bash
kubectl logs -n hci-troubleshoot deployment/conversation-service --since=5m | grep -i "error\|disconnect"
```

**常见根因**:

| 错误日志 | 根因 | 解决方案 |
|---------|------|---------|
| `Server disconnected without sending a response` | openclaw Pod 未部署，`OPENCLAW_BASE_URL` 指向不存在的 Service | 配置直连 ZhipuAI（见下方） |
| `404 Not Found` 且 URL 含 `/v1/chat/completions` | ZhipuAI 路径不含 `/v1/`，URL 多了前缀 | 设置 `aiCompletionsPath: /chat/completions` |

**配置直连 ZhipuAI**（在 `hci-platform-env/environments/dev/values.yaml` 中）:
```yaml
conversationService:
  externalDns: true   # conversation-service Pod 使用外部 DNS 解析 ZhipuAI 域名

config:
  openclawBaseUrl: "https://open.bigmodel.cn/api/paas/v4"
  openclawDefaultModel: "glm-4-flash"
  aiCompletionsPath: "/chat/completions"   # ZhipuAI v4 路径（无 /v1/ 前缀）

secrets:
  openclawToken: "<ZhipuAI API Key>"
```

## 11. 附录：K3s 网络架构填坑指南 (Retrospective)

在将项目落地到 WSL2 下的 K3s 时，我们在网络暴露和路由问题上曾走过以下弯路，特此记录以防后人踩坑：

### 11.1 端口转发 (`port-forward`) 并非长久之计
- **现象**: 最初尝试使用 `kubectl port-forward svc/grafana 3002:3000` 暴露 Grafana 时，终端报错 `unable to do port forwarding: socat not found`。
- **原因**: Kubelet 在宿主机物理转发网络包时，必须依赖 Linux 基础网络工具包 `socat`。
- **教训**: 我们虽然通过 `sudo apt-get install -y socat` 修复了报错，但这只是“治标”。`port-forward` 工具设计初衷仅用于**临时调试**，长时间保持容易断开且性能低下。**真正的生产实践应当使用 Ingress 七层负载均衡暴露服务**。

### 11.2 Path-based 路由 (`/admin`) 的前端地狱
- **现象**: 我们曾试图效仿 Docker Compose 中通过不同的端口来区分前后端，在 K3s Traefik 中使用 `/admin/` 这一路径前缀强行将流量路由至 Admin UI。
- **原因与痛点**: `http://<IP>/admin/` 这样的设计会导致极高的前后端耦合成本：
   1. 修改所有前端 Vite 配置的 `base: '/admin/'`。
   2. 修改 Vue Router 使用 `/admin` 为基础历史路由。
   3. 修改 Nginx 配置处理别名和重定向。
   4. 当请求到达真正的容器内 Nginx 时，极易产生 404 (找不到匹配真实路径的文件)。
- **教训**: **坚决避免将一个独立的 SPA 应用包裹在子路径下**。应该在 Ingress 层采用云原生的 **泛域名 (Subdomain) 路由**（例如配置 `admin.xxx.com`），不仅彻底免去了前端任何侵入式的代码修改（维持系统根路径 `/` 即可），也更贴合真实的云服务生产环境。

### 11.3 Fake-IP 与泛域名开发调试的死敌
- **现象**: 在放弃 `/admin` 并转型泛域名体系后，为了团队开发免修 `hosts` 文件，我们采纳了 `nip.io` 动态泛解析机制（如 `admin.127.0.0.1.nip.io` 直接映射回本地）。但部分开发人员报告无法访问，发现 curl 回调 IP 变成了神秘的 `198.18.0.x`。
- **原因**: 开发机上运行的 VPN 代理软件（如 Clash）开启了针对一切 `TUN / 虚拟网卡` 流量的 **Fake-IP 模式**。它拦截了 `nip.io` 的 DNS 查询，抢先投喂了一个假内网 IP 用以建立代理通道，导致请求根本没有发往你的本地 K3s Traefik。
- **教训**:
   1. 在进行任何包含公网 DNS 解析的 K8s 本地开发时（尤其是 `nip.io`, `sslip.io`），测试验证前**必须暂时关闭网络代理**。
   2. 如果代理实在无法关闭，只能**妥协且极度违反云原生最佳实践地**去修改 K3s `service-node-port-range`，并将 K8s Service 回退为原始粗暴的 `NodePort` 模式（用端口而非域名来映射），从物理层杜绝 DNS 劫持。但请记住，**这不应出现在标准实践中**。
