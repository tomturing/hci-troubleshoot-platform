# 多助手接入评估（OpenClaw / NanoBot / NanoClaw / PicoClaw / ZeroClaw）

更新时间: 2026-02-27

## 1. 结论（先看这个）

- 当前平台已经具备“统一接入层”雏形：以 `assistant_type + ASSISTANT_REGISTRY_JSON` 做多助手路由。
- 当前平台的实际调用协议是 **OpenAI 兼容 `POST /v1/chat/completions` + Bearer Token**。
- 在你本地已 clone 的 5 个助手中，能直接走该协议的只有：
  - `openclaw`（需开启 chatCompletions endpoint）
  - `zeroclaw`（内置兼容层，支持 `/v1/chat/completions`）
- `nanobot` / `nanoclaw` / `picoclaw` 目前都不是“面向外部的 OpenAI 风格推理 API 服务”，不能无改造直接接入当前链路。

一句话：**可先打通 OpenClaw + ZeroClaw 真实链路；其余 3 个助手需要额外 Adapter。**

## 2. 平台当前统一接入能力（hci-troubleshoot-platform）

### 2.1 已实现的统一点

- Conversation 侧统一通过 `AIAssistantRegistry` 管理助手客户端。
- Scheduler 侧统一通过 `ASSISTANT_REGISTRY_JSON` 管理多类型池。
- API Gateway 已有 `/api/assistants` 统一入口代理。

### 2.2 当前统一协议（非常关键）

Conversation 客户端固定调用：
- `POST {base_url}/v1/chat/completions`
- Header: `Authorization: Bearer <gateway_token>`
- SSE 流式解析 `data: ...`

这意味着：**不提供该接口的助手，当前无法直接接入。**

## 3. 五个助手接入方式核验

## 3.1 OpenClaw

- 网关端口：默认 `18789`
- 接口：支持 `POST /v1/chat/completions`
- 鉴权：`Authorization: Bearer <token>`
- 注意：该端点在官方文档中说明 **默认关闭**，需显式开启
  - `gateway.http.endpoints.chatCompletions.enabled=true`

参考：
- [openclaw/docs/gateway/openai-http-api.md](D:\AIBot\openclaw\docs\gateway\openai-http-api.md)
- [openclaw/docker-compose.yml](D:\AIBot\openclaw\docker-compose.yml)

接入结论：**可直接接入当前 HCI 链路**。

## 3.2 ZeroClaw

- 网关端口：默认 `42617`
- 原生入口：`POST /api/chat`（推荐）
- 兼容入口：`POST /v1/chat/completions`（兼容 OpenClaw 调用方）
- 鉴权：默认需要先 `POST /pair` 交换 token，后续 `Authorization: Bearer <token>`

参考：
- [zeroclaw/README.md](D:\AIBot\zeroclaw\README.md)
- [zeroclaw/docs/migration/openclaw-migration-guide.md](D:\AIBot\zeroclaw\docs\migration\openclaw-migration-guide.md)
- [zeroclaw/src/gateway/mod.rs](D:\AIBot\zeroclaw\src\gateway\mod.rs)

接入结论：**可直接接入当前 HCI 链路**（建议先用 `/v1/chat/completions` 兼容模式）。

## 3.3 NanoBot

- 启动方式：`nanobot gateway`
- 默认配置里可见 `gateway.port=18790`，但代码侧主要是渠道总线 + 桥接模式
- WhatsApp 通过单独 Node bridge（默认 `ws://localhost:3001`）与 Python 进程通信
- 未发现稳定的对外 `POST /v1/chat/completions` 或 `/api/chat` 服务接口

参考：
- [nanobot/README.md](D:\AIBot\nanobot\README.md)

- [nanobot/bridge/src/index.ts](D:\AIBot\nanobot\bridge\src\index.ts)

接入结论：**不能直接接入当前 HCI 链路**，需 Adapter。

## 3.4 PicoClaw

- 启动方式：`picoclaw gateway`
- 默认网关：`127.0.0.1:18790`
- 当前明确提供健康接口：`/health`、`/ready`
- 渠道回调以 webhook 路径为主（如 `/webhook/line`、`/webhook/wecom`）
- README 提到“支持 OpenAI-compatible protocol”是作为上游模型 provider 兼容，不是作为对外统一推理 API
- 未发现稳定的对外 `POST /v1/chat/completions` 或 `/api/chat` 服务接口

参考：

- 未发现稳定的对外 `POST /v1/chat/completions` 或 `/api/chat` 接口

## 4. 是否存在“统一接入方式”

## 4.1 现状答案

- **有，但只覆盖 OpenClaw / ZeroClaw：**
  - 统一协议：`/v1/chat/completions` + Bearer Token + SSE
  - 统一配置：`ASSISTANT_REGISTRY_JSON`

## 4.2 对 5 个助手统一的现实答案

- **不存在现成单协议可直连全部 5 个助手。**
- 要做到“5 合 1”统一接入，必须新增一层 `assistant-adapter`（协议转换层）。

---

## 5. 建议落地路径（按优先级）

1. 先打通可直接接入的真实链路：
   - OpenClaw
   - ZeroClaw
2. 在 `ASSISTANT_REGISTRY_JSON` 中配置这两个助手，完成前后端可选 + 真实路由验证。
3. 为 NanoBot / PicoClaw / NanoClaw 新增 Adapter（统一暴露 `/v1/chat/completions`）：
   - Adapter 内部再转它们各自 runtime/CLI/channel 协议。
4. 将 Conversation 的 `AIAssistantClient` 从“单一 OpenAI 兼容实现”扩展为多 Adapter 实现（按 `assistant_type` 选择）。

---

## 6. 当前风险与注意事项

- OpenClaw 的 `/v1/chat/completions` 默认关闭，需确认配置已开启。
- ZeroClaw 默认要求 pairing，未完成 `/pair` 则会 401。
- 当前 scheduler 设计偏 K8s Pod 池，若本地非 K8s 环境，Conversation 会回退到静态 `base_url`。
