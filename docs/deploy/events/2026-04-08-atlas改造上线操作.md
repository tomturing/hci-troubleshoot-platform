## 二、上线前提条件

1. PR CI 全绿（含新的 `db-migration-test` workflow）
2. PR 已创建，等待合并
3. dev 环境确认无活跃迁移 Job 正在运行

---

## 三、分环境上线操作

### Step 1：PR 合并（应用仓库）

```bash
# 确认 CI 全绿后合并 PR feature/db-migration-atlas-upgrade
# CI 自动构建 db-migrate 镜像，标签格式：ghcr.io/sangfor-hci/hci-platform/db-migrate:<sha>
```

### Step 2：CI 构建 db-migrate 镜像

CI 在 main push 时自动执行（在 `ci.yml` 的 build-and-push 矩阵中添加 `db-migrate` 服务后）：

```
CI 构建 → ghcr.io/sangfor-hci/hci-platform/db-migrate:20260408-XXXX-xxxxxxx
```

**注意**：需要在 `ci.yml` 中确认 `db-migrate` 已加入构建矩阵（本 PR 包含此变更）。

### Step 3：dev 环境首次切换（环境仓库）

```bash
# 在 hci-platform-env 仓库，修改 dev/values.yaml：
cat >> hci-platform-env/environments/dev/values.yaml << 'EOF'
dbMigrate:
  image: "ghcr.io/sangfor-hci/hci-platform/db-migrate:20260408-XXXX-xxxxxxx"  # 实际镜像 tag
  atlasBaseline: "20260408000000"  # 已有 DB，跳过 baseline 全量建表
EOF
```

**预期行为**：
- ArgoCD Sync 触发 PreSync Hook
- Atlas 检测到 `atlas_schema_revisions` 表不存在（首次运行）
- 创建 `atlas_schema_revisions` 表
- 因设置 `--baseline 20260408000000`，跳过 baseline 迁移
- 检查无新迁移文件（PR 变更后首次无新业务迁移），输出 "No migration files to execute"
- Job 成功，ArgoCD 继续同步

### Step 4：验证 dev 环境

```bash
# 检查 Job 状态
kubectl get job -n hci-dev | grep db-migrate

# 查看 Job 日志
kubectl logs job/<job-name> -n hci-dev

# 验证 atlas_schema_revisions 表创建
kubectl exec -n hci-dev <postgres-pod> -- psql -U hci_admin hci_troubleshoot \
  -c "SELECT version, applied_at FROM atlas_schema_revisions ORDER BY applied_at;"

# 验证业务表完整性（旧数据不受影响）
kubectl exec -n hci-dev <postgres-pod> -- psql -U hci_admin hci_troubleshoot \
  -c "\dt" | grep -E "user|customer|case|conversation"
```

**完成标准**：
- `db-migrate-xxx` Job 状态为 Completed
- `atlas_schema_revisions` 表存在，包含 baseline 记录（因 baseline 被记为已跳过）
- 所有业务表存在，数据完整
- 不存在 `schema_migrations` 表（可选：保留历史记录，不影响 Atlas）

### Step 5：staging 环境切换

与 dev 操作相同，修改 staging/values.yaml 中的 `dbMigrate.image` 和 `atlasBaseline`。

### Step 6：prod 环境切换（需审批）

1. 创建 prod 变更 PR，添加 `dbMigrate.image` 和 `atlasBaseline`
2. 获得技术负责人审批
3. 在发布窗口执行，过程与 dev 完全一致

## 四、回滚方案

**场景**：Atlas Job 失败，需要回滚。

```bash
# 1. 检查失败日志
kubectl logs job/db-migrate-<revision> -n <namespace>

# 2. 若是 atlas_schema_revisions 问题，手动删除并重试
# kubectl exec -n <namespace> <postgres-pod> -- psql -U hci_admin hci_troubleshoot \
#   -c "DROP TABLE IF EXISTS atlas_schema_revisions;"

# 3. 若需完全回退到 dbmate，恢复旧版本镜像 tag 即可
# （旧 db-migrate-job.yaml 已删除，需手动恢复 git 版本）
```

**注意**：Atlas 改造不修改任何业务表数据，回滚风险极低。

---

## 五、切换后清理（所有环境完成后）

当 dev/staging/prod 均完成 Atlas 切换且稳定运行 1 周后，可从 values.yaml 移除 `atlasBaseline` 字段：

```yaml
# 移除此字段（atlas_schema_revisions 已记录 baseline 已处理）
# atlasBaseline: "20260408000000"
```

同时可选清理旧 `schema_migrations` 表（保留无害，删除更干净）：

```sql
-- 可选：清理 dbmate 遗留的版本跟踪表
-- 执行前确认 Atlas 已完全接管（atlas_schema_revisions 存在且记录完整）
DROP TABLE IF EXISTS schema_migrations;
```
