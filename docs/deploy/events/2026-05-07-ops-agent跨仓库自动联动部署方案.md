# ops-agent 跨仓库自动联动部署方案

> **状态**：待实现（TODO）
> **优先级**：P2（ops-agent 更新频率较低时，手动更新成本可接受）

## 背景

### 当前问题

ops-agent 是独立仓库 (`ghcr.io/p3n9w31/ops-agent`)，不在本项目 CI 构建范围内。每次 ops-agent 发布新版本时，需要手动更新 `hci-platform-env` 仓库的 `opsAgent.image.tag`，触发 ArgoCD 同步部署。

### 现状

| 项目 | 镜像仓库 | CI 构建 | 自动部署 |
|------|---------|---------|---------|
| hci-troubleshoot-platform | `ghcr.io/tomturing/hci-troubleshoot-platform/` | 本项目 workflow | ✅ 自动 |
| ops-agent | `ghcr.io/p3n9w31/ops-agent` | ops-agent 仓库 workflow | ❌ 手动 |

### 触发事件

2026-05-07：ArgoCD `hci-platform-dev` 应用长时间处于 Processing 状态，根因是 `opsAgent.image.tag` 被错误设置为本项目 commit hash，导致镜像拉取失败。

## 方案设计

### 架构图

```
[ops-agent 仓库]                    [hci-platform-env 仓库]              [ArgoCD]
       │                                   │                               │
       │ 1. CI 构建完成                     │                               │
       │ 2. 推送镜像到 GHCR                 │                               │
       │    ghcr.io/p3n9w31/ops-agent       │                               │
       │                                    │                               │
       │ 3. repository_dispatch webhook ────►                               │
       │    event-type: ops-agent-released  │                               │
       │    payload: {tag, sha}             │                               │
       │                                    │                               │
       │                                    │ 4. 接收 webhook                │
       │                                    │ 5. 更新 values.yaml            │
       │                                    │    opsAgent.image.tag          │
       │                                    │ 6. git commit + push           │
       │                                    │    [skip ci]                   │
       │                                    │                               │
       │                                    │                               │ 7. 自动检测 git 变更
       │                                    │                               │ 8. Sync → Healthy
       │                                    │                               │
       ▼                                    ▼                               ▼
```

### 组件清单

| 组件 | 位置 | 职责 |
|------|------|------|
| ops-agent CI | `P3n9W31/ops-agent/.github/workflows/release.yml` | 构建镜像 + 触发 webhook |
| webhook 接收 | `tomturing/hci-platform-env/.github/workflows/ops-agent-sync.yml` | 更新 values.yaml |
| GitHub PAT | `P3n9W31/ops-agent` → `HCI_ENV_REPO_PAT` secret | 跨仓库 webhook 权限 |
| ArgoCD | K8s 集群 | 自动同步部署 |

## 实现步骤

### Step 1：ops-agent 仓库配置 webhook 触发

**文件**：`P3n9W31/ops-agent/.github/workflows/release.yml`

```yaml
name: Release ops-agent

on:
  push:
    branches: [main, feature-hci]
  workflow_dispatch:

jobs:
  build-and-push:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write
    steps:
      - uses: actions/checkout@v4
      
      - name: 登录 GHCR
        uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}
      
      - name: 构建并推送镜像
        uses: docker/build-push-action@v5
        with:
          push: true
          tags: |
            ghcr.io/p3n9w31/ops-agent:latest
            ghcr.io/p3n9w31/ops-agent:${{ github.sha }}
      
      # 新增：触发跨仓库联动
      - name: 触发 hci-platform-env 更新
        if: success()
        uses: peter-evans/repository-dispatch@v3
        with:
          token: ${{ secrets.HCI_ENV_REPO_PAT }}
          repository: tomturing/hci-platform-env
          event-type: ops-agent-released
          client-payload: |
            {
              "tag": "latest",
              "sha": "${{ github.sha }}",
              "branch": "${{ github.ref_name }}"
            }
```

### Step 2：hci-platform-env 仓库配置 webhook 接收

**文件**：`tomturing/hci-platform-env/.github/workflows/ops-agent-sync.yml`

```yaml
name: Sync ops-agent tag

on:
  repository_dispatch:
    types: [ops-agent-released]

jobs:
  update-values:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          token: ${{ secrets.HCI_ENV_REPO_PAT }}
      
      - name: 提取 webhook payload
        id: payload
        run: |
          echo "tag=${{ github.event.client_payload.tag }}" >> $GITHUB_OUTPUT
          echo "sha=${{ github.event.client_payload.sha }}" >> $GITHUB_OUTPUT
          echo "branch=${{ github.event.client_payload.branch }}" >> $GITHUB_OUTPUT
      
      - name: 更新 opsAgent.image.tag
        run: |
          TAG="${{ steps.payload.outputs.tag }}"
          
          # 更新所有环境的 values.yaml
          for env in dev staging prod; do
            VALUES_FILE="environments/${env}/values.yaml"
            if [[ -f "$VALUES_FILE" ]]; then
              # 使用 sed 更新 opsAgent.image.tag
              sed -i 's/tag: ".*"/tag: "${TAG}/' "$VALUES_FILE" \
                --expression='/^opsAgent:/,/^[A-Za-z0-9_]+:$/{
                  s/^\([[:space:]]*tag: ".*"\)/\1/
                }'
              
              echo "Updated $VALUES_FILE: opsAgent.image.tag -> $TAG"
            fi
          done
      
      - name: 提交更新
        run: |
          git config user.name "ops-agent-bot"
          git config user.email "bot@ops-agent.local"
          git add environments/*/values.yaml
          git commit -m "chore: ops-agent tag -> ${TAG} [skip ci]" || echo "No changes"
          git push origin main
```

### Step 3：配置 GitHub PAT

**位置**：`P3n9W31/ops-agent` 仓库 Settings → Secrets → `HCI_ENV_REPO_PAT`

**权限要求**：
- `repo`（读写 `tomturing/hci-platform-env` 仓库）
- `workflow`（触发 workflow）

**创建步骤**：
1. GitHub → Settings → Developer settings → Personal access tokens → Fine-grained tokens
2. Repository access: `tomturing/hci-platform-env`
3. Permissions: `Contents: Read and write`, `Workflows: Read and write`
4. 复制 token，添加到 `P3n9W31/ops-agent` 的 Secrets

## 冻结条件

以下情况下**不建议实现自动联动**：

1. **ops-agent 更新频率 ≤ 每月 1 次**：手动更新成本很低（每次 5 分钟），不值得投入开发时间
2. **ops-agent 仓库权限受限**：无法配置 `HCI_ENV_REPO_PAT` secret
3. **多环境差异化需求**：不同环境需要不同版本的 ops-agent（自动联动会统一更新）

## 替代方案

### 方案 A：合并 ops-agent 到本项目

将 ops-agent 源码迁移到 `backend/ops-agent-service/`，统一 CI 构建。

**优点**：
- 统一版本管理
- 无需跨仓库联动

**缺点**：
- ops-agent 无法独立迭代
- 需要大量代码迁移工作

### 方案 C：手动更新 Skill

已实现 `/ops-agent-update` skill，通过 CLI 快速更新。

**适用场景**：ops-agent 更新频率较低时（≤ 每月 1 次）
