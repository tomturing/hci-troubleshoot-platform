## 1. 发布流程总览

### 1.1 核心原则

1. **声明式优先**：环境状态以 Git 为唯一事实来源，禁止直接在集群手工修改配置。
2. **不可变发布**：镜像标签一旦推送不可覆盖，只能新增，格式：`YYYYMMDD-HHMM-<sha7>`。
3. **先验证再晋级**：dev → staging → prod，不可跳层。
4. **回滚优先**：任何发布步骤都必须存在可执行的回滚路径。
5. **无审批不入生产**：prod 变更必须经审批确认后方可执行同步。

### 1.2 双仓模型

| 仓库 | 职责 | 变更方式 |
|------|------|---------|
| **应用仓库**（hci-troubleshoot-platform）| 业务代码、Dockerfile、Helm Chart 模板 | 开发 PR |
| **环境仓库**（hci-platform-env）| 各环境 values.yaml、发布版本声明 | CI 自动 PR / 人工审批 |

### 1.3 发布路径（推荐：GitOps）

```
开发者 PR → main 合并
    ↓
CI 构建镜像 → 推送 ghcr.io → 生成 release-metadata.json
    ↓
CI 在环境仓库自动创建 PR（更新 image.tag）
    ↓
dev：PR 自动合入 → ArgoCD 自动同步
staging：人工审批合入 → ArgoCD 自动同步
prod：发布窗口 + 审批 → 人工触发 ArgoCD 同步
    ↓
发布后验证（30分钟观察窗口）
    ↓
正常 → 完成  /  异常 → 回滚
```

### 1.4 应急兜底路径（仅极端情况）

```bash
# ⚠️ 此路径绕过GitOps，使用后必须立即将变更补录到环境仓库
bash scripts/k3s-release.sh --services all
```

## 2. GitOps 接入配置

> 如 ArgoCD 已正常运行，可跳至 [§3 发布前检查](#3-发布前检查preflight)。

### 2.1 前置条件

- 集群可访问：`kubectl cluster-info` 正常
- ArgoCD 命名空间存在：`kubectl get ns argocd`
- 已创建环境仓库 `hci-platform-env`（建议本机路径：`/mnt/d/aihci/hci-platform-env`）
- GitHub Secrets 已配置：`ENV_REPO_PAT`

### 2.2 PAT 权限要求

| PAT 用途 | 权限要求 |
|---------|---------|
| ArgoCD 仓库读取（argocd-repo-creds）| `Contents: Read-only`、`Metadata: Read-only` |
| Actions 环境仓库写入（ENV_REPO_PAT）| `Contents: Read/Write`、`Pull requests: Read/Write`、`Metadata: Read-only` |

**推荐使用 Fine-grained PAT，仅授权所需仓库，不使用 Classic PAT。**

### 2.3 环境仓库初始化

```bash
# 1. 新建私有仓库 hci-platform-env，克隆到本机
git clone git@github.com:<org>/hci-platform-env.git /mnt/d/aihci/hci-platform-env

# 2. 将骨架模板复制到环境仓库
cp -r deploy/gitops/env-repo-template/environments /mnt/d/aihci/hci-platform-env/
cd /mnt/d/aihci/hci-platform-env && git add . && git commit -m "初始化环境仓库结构" && git push
```

### 2.4 创建 ArgoCD Application

#### 本地 dev 集群（WSL k3s）

| 访问端点 | 地址 |
|---------|------|
| ArgoCD UI | `http://172.26.101.255:30808`（NodePort 30808） |

```bash
# 登录本地 ArgoCD
argocd login 172.26.101.255:30808 --username admin --password <password> --insecure
```

#### 云端 Hub-Spoke 架构（staging + prod）

ArgoCD 以 **Hub-Spoke** 模式运行：ArgoCD 控制面安装在 staging 集群（`192.168.0.4`），同时管理 staging 和 prod 两个集群。prod 集群（`192.168.0.3`）通过 `argocd cluster add` 注册后即可被统一调度。

```
ArgoCD（staging: 192.168.0.4:30808）
  ├── staging 集群（本机，namespace: hci-staging）
  │     ├── hci-platform-infra-staging← StorageClass / RBAC
  │     ├── hci-platform-data-staging← PostgreSQL / Redis
  │     ├── hci-platform-obs-staging  ← Prometheus / Loki / Tempo / Grafana
  │     └── hci-platform-staging     ← 业务微服务
  └── prod 集群（192.168.0.3，namespace: hci-prod）
        ├── hci-platform-infra-prod  ← StorageClass / RBAC
        ├── hci-platform-data-prod   ← PostgreSQL / Redis
        ├── hci-platform-obs-prod    ← 可观测性栈（hci-observability）
        └── hci-platform-prod        ← 业务微服务
```

| 访问端点 | 地址 |
|---------|------|
| ArgoCD UI | `http://192.168.0.4:30808`（NodePort 30808） |
| Staging 业务 | `http://acli.sangfor.com.cn`（端口见 staging values） |
| Prod 业务 | `http://acli.sangfor.com.cn`（端口见 prod values） |
| Staging Grafana | `http://acli.sangfor.com.cn:4888/grafana/` |
| Prod Grafana | `http://acli.sangfor.com.cn:3888/grafana/` |

#### 注册 prod 集群（首次）

```bash
# 在可访问 ArgoCD API 的节点上执行（需先 argocd login）
argocd login 192.168.0.4:30808 --username admin --password <password> --insecure
argocd cluster add <prod-context-name> --name prod-cluster --insecure
```

或直接创建 Secret（适合没有 argocd CLI 的环境）：

```bash
# 参考 deploy/gitops/local/cluster-prod-secret.example.yaml
kubectl apply -f deploy/gitops/local/cluster-prod-secret.yaml -n argocd
```

#### 应用所有 Application

```bash
# 本地 dev 集群（local-only ArgoCD）
kubectl label ns argocd hci.env.role=dev --overwrite
bash scripts/ops/argocd-apply-apps.sh --scope local --yes

# 云端 ArgoCD（管理 staging + prod）
kubectl label ns argocd hci.env.role=staging --overwrite
bash scripts/ops/argocd-apply-apps.sh --scope cloud --yes
```

> **目录说明**：`local/` 对应本地 WSL dev 集群，`cloud/` 对应云端 ArgoCD（hub-spoke 管理 staging + prod）
>
> **防误操作说明**：统一使用 `scripts/ops/argocd-apply-apps.sh`，脚本会校验设备角色；`local` 仅允许 dev，`cloud` 仅允许 staging。

```bash
# 验证状态
kubectl -n argocd get applications
# 期望：所有 Application 状态为 Synced & Healthy
# prod Application 默认不自动同步，需手动触发或配置 syncPolicy
```

### 2.5 私有仓库认证

若出现 `authentication required` 错误：

```bash
# 复制模板，填写真实 PAT（此文件不提交 Git）
cp deploy/gitops/argocd-repo-creds.example.yaml deploy/gitops/local/argocd-repo-creds.yaml
# 编辑文件，填写 username 和 password（PAT）
kubectl apply -f deploy/gitops/local/argocd-repo-creds.yaml
# 等待 10-30 秒后重新检查
kubectl -n argocd get applications
```

### 2.6 首次联调验收标准

- [ ] 完成 1 次 dev 端到端同步（工作流 → 环境 PR → ArgoCD 同步 → 集群生效）
- [ ] 完成 1 次 staging 端到端同步
- [ ] 完成 1 次回退演练并恢复成功
- [ ] dev/staging Application 状态为 `Synced` & `Healthy`

### 2.7 ArgoCD Server NodePort 配置

ArgoCD 官方 manifests 默认将 `argocd-server` Service 类型设为 `ClusterIP`，无法直接通过节点 IP 访问 WebUI。

本项目通过 `deploy/gitops/argocd-ops/argocd-server-nodeport.yaml` 覆盖为 NodePort，统一端口为 **30808**：

```bash
# 验证 Service 类型
kubectl get svc argocd-server -n argocd
# TYPE: NodePort, PORT(S): 80:30808/TCP
```

> **注意**：ArgoCD 升级脚本（`scripts/ops/argocd-upgrade.sh`）会先应用官方 manifests，再由 `argocd-ops` Application 自动同步 NodePort 覆盖配置。

### 2.8 ArgoCD 补丁升级与 repo-server 自愈

当 ArgoCD 仅需同 minor 下 patch 升级时，统一使用仓库脚本执行，当前默认版本为 `v3.3.6`：

```bash
# 默认升级到 v3.3.6，并自动应用 watchdog 清单
bash scripts/ops/argocd-upgrade.sh

# 如需显式指定 kubectl（例如 k3s）
KUBECTL='sudo -n k3s kubectl' bash scripts/ops/argocd-upgrade.sh
```

该脚本会：

1. 确保 `argocd` namespace 存在。
2. 应用 `https://raw.githubusercontent.com/argoproj/argo-cd/v3.3.6/manifests/install.yaml`。
3. 等待 `argocd-server`、`argocd-repo-server`、`argocd-application-controller` 就绪。
4. 应用 `deploy/gitops/argocd-ops/argocd-repo-server-copyutil-watchdog.yaml`。

`argocd-repo-server-copyutil-watchdog` 为每 5 分钟执行一次的 CronJob：

1. 扫描 `argocd-repo-server` Pod 的 `copyutil` 容器/初始化容器状态。
2. 若发现 `CrashLoopBackOff`，自动删除异常 Pod，触发重建。
3. 输出带 `trace_id` 的结构化日志，便于回溯。

可手动验证：

```bash
kubectl -n argocd get cronjob argocd-repo-server-copyutil-watchdog
kubectl -n argocd get jobs --sort-by=.metadata.creationTimestamp | tail -n 5
kubectl -n argocd logs job/<latest-watchdog-job>
```

## 3. 发布前检查（Preflight）

### 3.1 变更准备

- [ ] 变更已通过 PR 审核并合并到 main
- [ ] CI 全部通过（Lint / 测试 / Helm 校验 / 镜像构建）
- [ ] 已确认目标镜像标签存在且不可变
- [ ] 已确认影响范围与风险等级
- [ ] 受影响文档已同步更新

### 3.2 配置与依赖检查

- [ ] 环境仓库（values.yaml）已更新到目标 image.tag
- [ ] Secret/ConfigMap 变更已审核，无敏感信息暴露
- [ ] 数据库变更有迁移脚本与回退说明
- [ ] 若本次包含会话/评分字段变更，已确认 staging/prod 已执行补充迁移（至少覆盖 `database/migrate_evaluation_v1.sql`、`database/migrate_conversation_p4_v1.sql`）
- [ ] 关键依赖服务状态正常（DB、Redis、Ingress、可观测栈）
- [ ] 若 `conversationService.externalDns=true`，已确认 `conversation-service` 的 DNS 搜索域与目标 namespace 一致，且 `/health` 中 `database`、`kb_service`、`ai_assistants` 全部为 `ok/true`

### 3.3 发布审批（仅 prod）

- [ ] 发布窗口已确认
- [ ] 发布责任人与观察责任人已明确
- [ ] 已获得生产发布审批

## 4. 发布后验证（Postflight）

### 4.1 即时检查（发布完成后立即执行）

```bash
# 命名空间工作负载状态
kubectl -n hci get pods
kubectl -n hci get deploy

# 最近发布历史
helm -n hci history hci-platform

# 项目内验证脚本
bash scripts/k3s-verify.sh
```

### 4.2 健康检查要点

- [ ] 所有目标工作负载 Pod 全部 Ready
- [ ] 关键接口健康检查通过（`/health`）
- [ ] 前端关键页面可访问，核心交互正常

建议额外执行以下检查，覆盖本次 staging 事故涉及的两类高频回归：

```bash
# 1. conversation-service 依赖健康必须全部为 green
kubectl -n <namespace> exec deploy/conversation-service -- \
    python -c "import requests, json; r=requests.get('http://127.0.0.1:8002/health', timeout=15); print(json.dumps(r.json(), ensure_ascii=False))"

# 2. 会话表关键列必须存在，避免 create conversation 触发 UndefinedColumnError
kubectl -n <namespace> exec postgres-0 -- \
    psql "postgresql://<user>:<password>@localhost:5432/<db>" \
    -c "SELECT column_name FROM information_schema.columns WHERE table_name='conversation' AND column_name IN ('diagnostic_stage','category_l1','category_l2','category_id','hypothesis','react_state','pending_confirm') ORDER BY column_name;"
```

### 4.3 30 分钟观察窗口

```bash
# 自动采样观察脚本（默认每 60 秒采样一次）
NAMESPACE=hci RELEASE_NAME=hci-platform WINDOW_MINUTES=30 \
  bash scripts/release-observe.sh
```

### 4.4 异常 Pod 快速感知与治理

生产常见异常（`ImagePullBackOff`、`CrashLoopBackOff`、`ErrImagePull`）建议采用“自动告警 + 快速巡检”双轨：

1. 可观测性命名空间部署 `kube-state-metrics` 与 `alertmanager`（清单：`deploy/gitops/argocd-ops/pod-anomaly-observability.yaml`）。
2. Prometheus 加载 Pod 异常告警规则，按 namespace/pod 聚合。
3. `hci-platform-obs` Chart 建议使用 `scrapeNamespaces`（列表）覆盖抓取范围，至少包含：`hci`、`hci-observability`、`argocd`。
   - **kube-state-metrics 专用 scrape job（重要）**：`hci-services` job 的 namespace relabel 规则会把 kube-state-metrics 指标的 `namespace` 标签覆盖为其所在 ns（`hci-observability`），导致业务 namespace 数据不可见。Prometheus 配置使用独立的 `kube-state-metrics` job（`honor_labels: true`），并在 `hci-services` job 中加 `drop` 规则排除该 Pod，避免两个 job 重复采集。**如须修改 Helm 配置，务必保留此双 job 结构**。
4. 飞书 Webhook 通过双仓模式管理（**禁止**将真实地址提交到 app repo）：
   - 实际 URL 存放在 `hci-platform-env` 环境仓库 `secrets.feishuWebhookUrl` 字段
   - `hci-platform-obs/templates/secrets.yaml` 引用 `.Values.secrets.feishuWebhookUrl` 渲染 `alertmanager-config` Secret
   - 与 `grafanaAdminPassword` 处理模式完全一致
5. **Alertmanager DNS 注意事项（PIT-034）**：K3s 环境下 Clash fake-IP 会导致 Pod 无法连通外部 API。Alertmanager Deployment 已设置 `dnsPolicy: None` + 真实 DNS（`114.114.114.114`），如重建 alertmanager 需保留此配置。
6. 值班同学使用一键脚本快速巡检：

```bash
bash scripts/ops/pod-anomaly-scan.sh
```

WebUI 说明：

1. Grafana 可作为 Pod 状态与告警可视化入口（推荐）。
   - 仪表板（`HCI Pod 异常观测看板`，UID `hci-pod-anomaly`）的等待原因类面板同时覆盖主容器和 Init 容器指标（`kube_pod_container_status_waiting_reason` + `kube_pod_init_container_status_waiting_reason`），可捕获 `Init:ImagePullBackOff` 等初始化阶段故障。
2. Kubernetes 原生 Dashboard 不是当前环境默认组件，若需要可另行安装并按最小权限开放。

观察期间关注指标：

| 指标 | 正常阈值 |
|------|---------|
| HTTP 5xx 错误率 | < 0.5%（基线） |
| P95 响应延迟 | 无持续上升趋势 |
| Pod 重启次数 | 观察窗口内 0 次异常重启 |
| CrashLoopBackOff | 0 |

## 5. 回滚SOP

### 5.1 回滚触发条件（满足任一即触发）

- 核心接口持续不可用（超过 5 分钟）
- 错误率超过基线阈值且持续上升
- 用户核心路径不可用（创建工单 / 对话 / AI 回复）
- 出现不可接受的数据一致性风险

### 5.2 GitOps 回滚（**首选**）

```bash
# 1. 在环境仓库回退到上一稳定 commit
cd /mnt/d/aihci/hci-platform-env
git log --oneline -5   # 确认目标 commit
git revert HEAD        # 生成回退 commit（保留历史）
git push               # 推送后 ArgoCD 自动同步回退版本

# 2. 确认 ArgoCD 同步状态
kubectl -n argocd get applications

# 3. 验证服务恢复
kubectl -n hci get pods
bash scripts/k3s-verify.sh
```

### 5.3 Helm 直接回滚（**应急备用**）

```bash
# 查看修订历史
helm -n hci history hci-platform

# 回滚到上一个稳定 revision
helm -n hci rollback hci-platform <REVISION>

# 验证
kubectl -n hci get pods
bash scripts/k3s-verify.sh
```

> **注意**：Helm 直接回滚后，环境仓库状态与集群状态不同步（配置漂移）。必须在 24 小时内补录环境仓库，恢复声明式一致性。

### 5.4 演练模式（不执行真实回滚）

```bash
NAMESPACE=hci RELEASE_NAME=hci-platform \
  bash scripts/rollback-drill.sh
```

### 5.5 回滚后动作

1. 标记本次发布失败，冻结继续发布
2. 输出故障摘要：影响范围、根因假设、恢复耗时
3. 若使用 Helm 应急回滚：补录环境仓库至一致状态
4. 创建修复任务，安排复盘
