## 1. 发布流程总览

### 1.1 核心原则

1. **声明式优先**：环境状态以 Git 为唯一事实来源，禁止直接在集群手工修改配置。
2. **不可变发布**：镜像标签一旦推送不可覆盖，只能新增，格式：`YYYYMMDD-HHMM-<sha7>`。
3. **先验证再晋级**：dev → staging → prod，不可跳层。
4. **回滚优先**：任何发布步骤都必须存在可执行的回滚路径。
5. **无审批不入生产**：prod 变更必须经审批确认后方可执行同步。

### 1.2 双仓模型

| 仓库 | 职责 | 变更方式 |
|------|------|---------|
| **应用仓库**（hci-troubleshoot-platform）| 业务代码、Dockerfile、Helm Chart 模板 | 开发 PR |
| **环境仓库**（hci-platform-env）| 各环境 values.yaml、发布版本声明 | CI 自动 PR / 人工审批 |

### 1.3 发布路径（推荐：GitOps）

```
开发者 PR → CI 检查（10+ 项，详见 §3）
    ↓
CI 全绿 → 合并到 main
    ↓
main push 触发 → CI 构建
    ├── prepare: 生成镜像标签（YYYYMMDD-HHMM-sha7）
    ├── build-and-push: 矩阵构建 7 个服务镜像
    │     ├── api-gateway
    │     ├── case-service
    │     ├── conversation-service
    │     ├── scheduler-service
    │     ├── kb-service
    │     ├── customer-ui
    │     └── admin-ui
    └── Trivy 安全扫描（上报 Security Tab，不阻断）
    ↓
CI 串行触发环境同步（dev 优先，避免并发竞争）
    ├── dev：env-repo-sync.yml → 环境仓库 PR → 合并 → ArgoCD 自动同步
    │     └── PreSync Hook：db-migrate-job 执行数据库迁移
    └── staging：env-repo-sync.yml → 环境仓库 PR → 合并 → ArgoCD 自动同步
    │     └── PreSync Hook：db-migrate-job 执行数据库迁移
prod：发布窗口 + 审批 → 人工触发 env-repo-sync.yml → ArgoCD 同步
    └── PreSync Hook：db-migrate-job 执行数据库迁移
    ↓
发布后验证（30分钟观察窗口）
    ├── Pod 状态巡检：kubectl get pods / scripts/ops/pod-anomaly-scan.sh
    ├── 健康检查：/health 接口 + 关键表字段校验
    ├── 观察脚本：scripts/release-observe.sh 自动采样
    └── 异常告警：Prometheus + Alertmanager + 飞书 Webhook
    ↓
正常 → 完成  /  异常 → 回滚（§5）
```

#### 1.3.1 容器发布逻辑

**触发时机**：PR 合并到 main 后，由 `ci.yml` workflow 自动触发。

**构建流程**：

| 阶段 | Job | 说明 |
|------|-----|------|
| 1. 准备 | `prepare` | 生成唯一镜像标签 `YYYYMMDD-HHMM-sha7` |
| 2. 构建 | `build-and-push` | 矩阵构建 7 个服务，推送 `ghcr.io` |
| 3. 安全 | Trivy 扫描 | 扫描镜像漏洞，上报 GitHub Security，不阻断发布 |
| 4. 晋级 | `auto-deploy-non-prod` | 串行触发 dev → staging 环境同步 |

**镜像命名规则**：

```
ghcr.io/<owner>/hci-troubleshoot-platform/<service>:YYYYMMDD-HHMM-sha7
ghcr.io/<owner>/hci-troubleshoot-platform/<service>:latest
```

- **不可变性**：镜像标签一旦推送不可覆盖，只能新增
- **latest 标签**：仅用于开发调试，生产环境必须使用时间戳标签

**可观测点**：

| 观测维度 | 检查方式 | 异常阈值 |
|---------|---------|---------|
| 镜像推送成功 | GitHub Packages 页面 | 构建失败 |
| 镜像拉取成功 | `kubectl describe pod` | `ImagePullBackOff` |
| 容器启动成功 | `kubectl get pods` | `CrashLoopBackOff` |
| 安全漏洞 | GitHub Security Tab | HIGH/CRITICAL 漏洞 |
| ArgoCD 同步状态 | ArgoCD UI / `argocd app get` | `OutOfSync` / `Degraded` |

#### 1.3.2 数据库发布逻辑

**触发时机**：ArgoCD Application 同步前，由 `PreSync Hook` 自动执行。

**数据库管理模式**：**Atlas 声明式管理**（自 2026-04-08 起）

- 真实来源（Source of Truth）：`database/desired_schema.sql`（表结构/枚举/索引）+ `database/desired_extras.sql`（函数/触发器）
- 禁止手写逐步迁移脚本，禁止使用 dbmate/Alembic
- Atlas 通过对比期望 Schema 与实际 Schema 自动生成并执行最小化 DDL 变更
- `desired_extras.sql` 中所有函数/触发器均使用 `CREATE OR REPLACE`，天然幂等

**执行流程**：

```
ArgoCD Sync 触发
    ↓
PreSync Hook: db-migrate-job.yaml（hci-platform-data Chart）
    ├── Step 1: psql 执行 desired_extras.sql（函数/触发器 upsert）
    ├── Step 2: atlas schema apply --env <env> --auto-approve
    │     ├── 对比 desired_schema.sql 与实际 DB Schema
    │     ├── 自动生成最小化 DDL（ADD COLUMN / CREATE TABLE / 等）
    │     └── 执行变更并验证
    ├── Step 3: psql 再次执行 desired_extras.sql（确保函数依赖新列）
    └── Job 完成后 Application 继续同步
```

**关键配置**：

| 文件 | 职责 |
|------|------|
| `database/desired_schema.sql` | **唯一 Schema 真实来源**，定义所有表/枚举/索引 |
| `database/desired_extras.sql` | 函数/触发器，幂等 `CREATE OR REPLACE` |
| `scripts/db-migrate.sh` | 3 步执行脚本：extras → atlas apply → extras |
| `Dockerfile.migrations` | 打包 atlas CLI + psql 的独立迁移镜像 |
| `atlas.hcl` | Atlas 环境配置（local / ci / prod），读取 `DATABASE_URL` |
| `deploy/helm/hci-platform-data/templates/hooks/db-migrate-job.yaml` | ArgoCD PreSync Hook，拉起迁移 Job |

**Schema 变更操作规范**：

1. 直接修改 `database/desired_schema.sql`（**不要创建逐步迁移文件**）
2. 如有函数/触发器变更，同时修改 `database/desired_extras.sql`
3. 提交 PR → CI 自动通过 `db-migration-test` workflow 做 dry-run 验证
4. PR 合并 → CI 构建新 `db-migrate` 镜像（仅当 Schema 文件变更时触发）
5. 环境仓库 values.yaml 更新 `dbMigrate.image.tag` → ArgoCD Sync → PreSync Hook 执行

**CI 验证**：

| Job | Workflow | 说明 |
|-----|----------|------|
| `migration-test` | `db-migration-test.yml` | 启动真实 PostgreSQL，执行完整 `scripts/db-migrate.sh`，验证 Atlas apply 成功 |
| `build-db-migrate` | `ci.yml` | 仅当 `desired_schema.sql`/`desired_extras.sql`/`Dockerfile.migrations`/`db-migrate.sh` 变更时触发构建 |

**可观测点**：

| 观测维度 | 检查方式 | 异常阈值 |
|---------|---------|---------|
| 迁移 Job 成功 | `kubectl get job db-migrate-<rev> -n <ns>` | Job 失败 |
| 迁移日志 | `kubectl logs job/db-migrate-<rev> -n <ns>` | SQL 错误 / Atlas apply 失败 |
| 表结构正确性 | `\dt` 或 `\d <table>` | 表缺失 / 列缺失 |
| 数据库表数量 | `SELECT COUNT(*) FROM pg_tables WHERE schemaname='public';` | 与预期不符 |

**常见问题**：

| 问题 | 根因 | 解决方案 |
|------|------|---------|
| `relation "xxx" does not exist` | `desired_extras.sql` 引用了尚未创建的表/列 | 先在 `desired_schema.sql` 添加对应定义再提交 |
| Atlas apply 报 `irreversible change` | 检测到数据破坏性操作（如列重命名） | 改用两步提交：先加新列，再删旧列 |
| Job 超时失败 | `atlas schema apply` 执行时间过长或 DB 连接失败 | 检查 DB 连接串、检查 Atlas 版本兼容性 |
| 函数编译失败 | `desired_extras.sql` 语法错误 | 本地 `psql < database/desired_extras.sql` 验证 |

> **环境同步串行执行**：为避免并发推送竞争，dev 和 staging 环境同步采用串行执行，dev 优先。详见 [环境同步串行化方案](../solution/events/2026-04-07-环境同步串行化方案.md)。

## 2. GitOps 接入配置

> 如 ArgoCD 已正常运行，可跳至 [§3 发布前检查](#3-发布前检查preflight)。

### 2.1 前置条件

- 集群可访问：`kubectl cluster-info` 正常
- ArgoCD 命名空间存在：`kubectl get ns argocd`
- 已创建环境仓库 `hci-platform-env`（建议本机路径：`/mnt/d/aihci/hci-platform-env`）
- GitHub Secrets 已配置：`ENV_REPO_PAT`

### 2.2 PAT 权限要求

| PAT 用途 | 权限要求 |
|---------|---------|
| ArgoCD 仓库读取（argocd-repo-creds）| `Contents: Read-only`、`Metadata: Read-only` |
| Actions 环境仓库写入（ENV_REPO_PAT）| `Contents: Read/Write`、`Pull requests: Read/Write`、`Metadata: Read-only` |

**推荐使用 Fine-grained PAT，仅授权所需仓库，不使用 Classic PAT。**

### 2.3 环境仓库初始化

```bash
# 1. 新建私有仓库 hci-platform-env，克隆到本机
git clone git@github.com:<org>/hci-platform-env.git /mnt/d/aihci/hci-platform-env

# 2. 将骨架模板复制到环境仓库
cp -r deploy/gitops/env-repo-template/environments /mnt/d/aihci/hci-platform-env/
cd /mnt/d/aihci/hci-platform-env && git add . && git commit -m "初始化环境仓库结构" && git push
```

### 2.4 创建 ArgoCD Application

#### 本地 dev 集群（WSL k3s）

| 访问端点 | 地址 |
|---------|------|
| ArgoCD UI | `http://172.26.101.255:30808`（NodePort 30808） |

```bash
# 登录本地 ArgoCD
argocd login 172.26.101.255:30808 --username admin --password <password> --insecure
```

#### 云端 Hub-Spoke 架构（staging + prod）

ArgoCD 以 **Hub-Spoke** 模式运行：ArgoCD 控制面安装在 staging 集群（`192.168.0.4`），同时管理 staging 和 prod 两个集群。prod 集群（`192.168.0.3`）通过 `argocd cluster add` 注册后即可被统一调度。

```
ArgoCD（staging: 192.168.0.4:30808）
  ├── staging 集群（本机，namespace: hci-staging）
  │     ├── hci-platform-infra-staging← StorageClass / RBAC
  │     ├── hci-platform-data-staging← PostgreSQL / Redis
  │     ├── hci-platform-obs-staging  ← Prometheus / Loki / Tempo / Grafana
  │     └── hci-platform-staging     ← 业务微服务
  └── prod 集群（192.168.0.3，namespace: hci-prod）
        ├── hci-platform-infra-prod  ← StorageClass / RBAC
        ├── hci-platform-data-prod   ← PostgreSQL / Redis
        ├── hci-platform-obs-prod    ← 可观测性栈（hci-observability）
        └── hci-platform-prod        ← 业务微服务
```

| 访问端点 | 地址 |
|---------|------|
| ArgoCD UI | `http://192.168.0.4:30808`（NodePort 30808） |
| Staging 业务 | `http://acli.sangfor.com.cn`（端口见 staging values） |
| Prod 业务 | `http://acli.sangfor.com.cn`（端口见 prod values） |
| Staging Grafana | `http://acli.sangfor.com.cn:4888/grafana/` |
| Prod Grafana | `http://acli.sangfor.com.cn:3888/grafana/` |

#### 注册 prod 集群（首次）

```bash
# 在可访问 ArgoCD API 的节点上执行（需先 argocd login）
argocd login 192.168.0.4:30808 --username admin --password <password> --insecure
argocd cluster add <prod-context-name> --name prod-cluster --insecure
```

或直接创建 Secret（适合没有 argocd CLI 的环境）：

```bash
# 参考 deploy/gitops/local/cluster-prod-secret.example.yaml
kubectl apply -f deploy/gitops/local/cluster-prod-secret.yaml -n argocd
```

#### 应用所有 Application

**App of Apps 分层架构**：两个集群各有一个 `argocd-ops` Application 负责 bootstrap，互不干涉。

```
dev 集群
└── argocd-ops  (argo-apps/local/argocd-ops.yaml)
    ├── 管理 deploy/gitops/argocd-ops/         — watchdog 运维配件
    └── 管理 deploy/gitops/argo-apps/local/    — dev 所有 Application 定义

staging 集群
└── argocd-ops  (argo-apps/cloud/argocd-ops.yaml)
    ├── 管理 deploy/gitops/argocd-ops/         — watchdog 运维配件
    └── 管理 deploy/gitops/argo-apps/cloud/    — staging/prod 所有 Application 定义
```

> **重要**：`cloud/` 里的 Application 只能由 staging 侧 ArgoCD 管理，dev 侧不可 apply，否则会因缺少凭据导致永久 sync 失败（详见 [pitfalls/k8s.md D-001](pitfalls/k8s.md)）。

```bash
# 首次 bootstrap（每个集群只需执行一次）
# dev 机器
kubectl label ns argocd hci.env.role=dev --overwrite
kubectl apply -f deploy/gitops/argo-apps/local/argocd-ops.yaml

# staging 机器
kubectl label ns argocd hci.env.role=staging --overwrite
kubectl apply -f deploy/gitops/argo-apps/cloud/argocd-ops.yaml
```

> **bootstrap 后**：ArgoCD 会自动 reconcile 对应目录下所有 Application，无需手动 apply 其他文件。

```bash
# 或使用脚本（含环境校验）
bash scripts/ops/argocd-apply-apps.sh --scope local --yes   # dev 机器
bash scripts/ops/argocd-apply-apps.sh --scope cloud --yes   # staging 机器
```

> **目录说明**：`local/` 对应本地 WSL dev 集群，`cloud/` 对应云端 ArgoCD（hub-spoke 管理 staging + prod）
>
> **防误操作说明**：统一使用 `scripts/ops/argocd-apply-apps.sh`，脚本会校验设备角色；`local` 仅允许 dev，`cloud` 仅允许 staging。

```bash
# 验证状态
kubectl -n argocd get applications
# 期望：所有 Application 状态为 Synced & Healthy
# prod Application 默认不自动同步，需手动触发或配置 syncPolicy
```

### 2.5 私有仓库认证

若出现 `authentication required` 错误：

```bash
# 复制模板，填写真实 PAT（此文件不提交 Git）
cp deploy/gitops/argocd-repo-creds.example.yaml deploy/gitops/local/argocd-repo-creds.yaml
# 编辑文件，填写 username 和 password（PAT）
kubectl apply -f deploy/gitops/local/argocd-repo-creds.yaml
# 等待 10-30 秒后重新检查
kubectl -n argocd get applications
```

### 2.6 首次联调验收标准

- [ ] 完成 1 次 dev 端到端同步（工作流 → 环境 PR → ArgoCD 同步 → 集群生效）
- [ ] 完成 1 次 staging 端到端同步
- [ ] 完成 1 次回退演练并恢复成功
- [ ] dev/staging Application 状态为 `Synced` & `Healthy`

### 2.7 ArgoCD Server NodePort 配置

ArgoCD 官方 manifests 默认将 `argocd-server` Service 类型设为 `ClusterIP`，无法直接通过节点 IP 访问 WebUI。

本项目通过 `deploy/gitops/argocd-ops/argocd-server-nodeport.yaml` 覆盖为 NodePort，统一端口为 **30808**：

```bash
# 验证 Service 类型
kubectl get svc argocd-server -n argocd
# TYPE: NodePort, PORT(S): 80:30808/TCP
```

> **注意**：ArgoCD 升级脚本（`scripts/ops/argocd-upgrade.sh`）会先应用官方 manifests，再由 `argocd-ops` Application 自动同步 NodePort 覆盖配置。

### 2.8 ArgoCD 补丁升级与 repo-server 自愈

当 ArgoCD 仅需同 minor 下 patch 升级时，统一使用仓库脚本执行，当前默认版本为 `v3.3.6`：

```bash
# 默认升级到 v3.3.6，并自动应用 watchdog 清单
bash scripts/ops/argocd-upgrade.sh

# 如需显式指定 kubectl（例如 k3s）
KUBECTL='sudo -n k3s kubectl' bash scripts/ops/argocd-upgrade.sh
```

该脚本会：

1. 确保 `argocd` namespace 存在。
2. 应用 `https://raw.githubusercontent.com/argoproj/argo-cd/v3.3.6/manifests/install.yaml`。
3. 等待 `argocd-server`、`argocd-repo-server`、`argocd-application-controller` 就绪。
4. 应用 `deploy/gitops/argocd-ops/argocd-repo-server-copyutil-watchdog.yaml`。

`argocd-repo-server-copyutil-watchdog` 为每 5 分钟执行一次的 CronJob：

1. 扫描 `argocd-repo-server` Pod 的 `copyutil` 容器/初始化容器状态。
2. 若发现 `CrashLoopBackOff`，自动删除异常 Pod，触发重建。
3. 输出带 `trace_id` 的结构化日志，便于回溯。

可手动验证：

```bash
kubectl -n argocd get cronjob argocd-repo-server-copyutil-watchdog
kubectl -n argocd get jobs --sort-by=.metadata.creationTimestamp | tail -n 5
kubectl -n argocd logs job/<latest-watchdog-job>
```

## 3. PR 检查项详解

> 本章节列出所有 PR 必须通过的 CI 检查项，按类别分组，标注是否阻断合并。

### 3.1 阻断性检查（必须通过）

| 检查项 | Workflow | Job | 说明 | 失败影响 |
|--------|----------|-----|------|---------|
| **docs-governance** | ci.yml | docs-governance | 文档门禁：改动 `backend/`、`frontend/`、`deploy/`、`scripts/`、`database/` 必须同步更新 `docs/` | PR 无法合并 |
| **lint** | ci.yml | lint | Ruff 代码检查 + 格式验证 | PR 无法合并 |
| **frontend-build** | ci.yml | frontend-build | 前端 `pnpm build` 必须成功 | PR 无法合并 |
| **helm-validate** | ci.yml | helm-validate | Helm lint + template 渲染 + unittest | PR 无法合并 |

#### 3.1.1 docs-governance 详细说明

**触发条件**：PR 改动 `backend/`、`frontend/`、`deploy/`、`scripts/`、`database/` 目录（不含 `.github/workflows/`）

**通过条件**：同一 PR 中至少更新以下之一：
- `docs/` 目录下的文档
- `README.md`
- `AGENTS.md`
- `CLAUDE.md`

**旁路方式**：无（文档门禁不可旁路）

**常见失败场景**：
- 新增 API 接口但未更新 API 文档
- 修改 Helm 配置但未更新部署文档
- 数据库表变更但未更新数据库设计文档

#### 3.1.2 db-migration-test 详细说明

**触发条件**：PR 改动 `database/`、`scripts/db-migrate.sh`、`Dockerfile.migrations`、`atlas.hcl`

**通过条件**：`db-migration-test.yml` 的 `migration-test` job 成功执行完整的 `scripts/db-migrate.sh`，包含：

1. 启动真实 PostgreSQL 服务容器
2. 执行 `psql < database/desired_extras.sql`（函数/触发器）
3. 执行 `atlas schema apply --env ci --auto-approve`（对比 `desired_schema.sql` 并执行 DDL）
4. 再次执行 `psql < database/desired_extras.sql`（确保函数依赖新列）
5. 验证最终 Schema 状态

**Schema 变更规范**：

- 直接修改 `database/desired_schema.sql`，不创建逐步迁移文件
- 函数/触发器变更时同步修改 `database/desired_extras.sql`
- **禁止使用 `database/migrations/` 目录**（已废弃，目录已删除）

**常见失败场景**：

- `desired_schema.sql` 有语法错误
- `desired_extras.sql` 引用了不存在的列/表（顺序问题）
- Atlas 检测到破坏性变更（如直接重命名列）

### 5.4 异常 Pod 快速感知与治理

生产常见异常（`ImagePullBackOff`、`CrashLoopBackOff`、`ErrImagePull`）建议采用“自动告警 + 快速巡检”双轨：

1. 可观测性命名空间部署 `kube-state-metrics` 与 `alertmanager`（清单：`deploy/gitops/argocd-ops/pod-anomaly-observability.yaml`）。
2. Prometheus 加载 Pod 异常告警规则，按 namespace/pod 聚合。
3. `hci-platform-obs` Chart 建议使用 `scrapeNamespaces`（列表）覆盖抓取范围，至少包含：`hci`、`hci-observability`、`argocd`。
   - **kube-state-metrics 专用 scrape job（重要）**：`hci-services` job 的 namespace relabel 规则会把 kube-state-metrics 指标的 `namespace` 标签覆盖为其所在 ns（`hci-observability`），导致业务 namespace 数据不可见。Prometheus 配置使用独立的 `kube-state-metrics` job（`honor_labels: true`），并在 `hci-services` job 中加 `drop` 规则排除该 Pod，避免两个 job 重复采集。**如须修改 Helm 配置，务必保留此双 job 结构**。
4. 飞书 Webhook 通过双仓模式管理（**禁止**将真实地址提交到 app repo）：
   - 实际 URL 存放在 `hci-platform-env` 环境仓库 `secrets.feishuWebhookUrl` 字段
   - `hci-platform-obs/templates/secrets.yaml` 引用 `.Values.secrets.feishuWebhookUrl` 渲染 `alertmanager-config` Secret
   - 与 `grafanaAdminPassword` 处理模式完全一致
5. **Alertmanager DNS 注意事项（PIT-034）**：K3s 环境下 Clash fake-IP 会导致 Pod 无法连通外部 API。Alertmanager Deployment 已设置 `dnsPolicy: None` + 真实 DNS（`114.114.114.114`），如重建 alertmanager 需保留此配置。
6. 值班同学使用一键脚本快速巡检：

```bash
bash scripts/ops/pod-anomaly-scan.sh
```

WebUI 说明：

1. Grafana 可作为 Pod 状态与告警可视化入口（推荐）。
   - 仪表板（`HCI Pod 异常观测看板`，UID `hci-pod-anomaly`）的等待原因类面板同时覆盖主容器和 Init 容器指标（`kube_pod_container_status_waiting_reason` + `kube_pod_init_container_status_waiting_reason`），可捕获 `Init:ImagePullBackOff` 等初始化阶段故障。

## 6. 回滚SOP

### 6.1 回滚触发条件（满足任一即触发）

- 核心接口持续不可用（超过 5 分钟）
- 错误率超过基线阈值且持续上升
- 用户核心路径不可用（创建工单 / 对话 / AI 回复）
- 出现不可接受的数据一致性风险

### 6.2 GitOps 回滚（**首选**）

```bash
# 1. 在环境仓库回退到上一稳定 commit
cd /mnt/d/aihci/hci-platform-env
git log --oneline -5   # 确认目标 commit
git revert HEAD        # 生成回退 commit（保留历史）
git push               # 推送后 ArgoCD 自动同步回退版本

# 2. 确认 ArgoCD 同步状态
kubectl -n argocd get applications

# 3. 验证服务恢复
kubectl -n hci get pods
bash scripts/k3s-verify.sh
```

### 6.3 Helm 直接回滚（**应急备用**）

```bash
# 查看修订历史
helm -n hci history hci-platform

# 回滚到上一个稳定 revision
helm -n hci rollback hci-platform <REVISION>

# 验证
kubectl -n hci get pods
bash scripts/k3s-verify.sh
```

> **注意**：Helm 直接回滚后，环境仓库状态与集群状态不同步（配置漂移）。必须在 24 小时内补录环境仓库，恢复声明式一致性。

### 6.4 演练模式（不执行真实回滚）

```bash
NAMESPACE=hci RELEASE_NAME=hci-platform \
  bash scripts/rollback-drill.sh
```

### 6.5 回滚后动作

1. 标记本次发布失败，冻结继续发布
2. 输出故障摘要：影响范围、根因假设、恢复耗时
3. 若使用 Helm 应急回滚：补录环境仓库至一致状态
4. 创建修复任务，安排复盘
