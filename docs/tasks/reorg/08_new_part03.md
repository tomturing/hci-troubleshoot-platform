## 五、目标架构：专业HCI智能排障Claude

> **本节记录一个关键认知修正：**
> 前四节的重构方案方向正确，但仍然不够彻底。
> 最终目标"再造一个专业HCI智能排障Claude"，要求的是 **Agent 架构**，而不是"更好的问答系统"。

### 5.1 对"再造专业HCI智能排障Claude"的深刻理解

Claude 之所以强大，不是因为它"存了很多答案"，而是因为它的架构是这样的：

```
LLM 是大脑，工具是它的手和眼
LLM 主动决定"下一步做什么" → 调用工具 → 解读结果 → 再决定 → 再调用
直到问题解决，才给用户最终答案
```

**当前系统（包括 Phase 0-2 重构）本质上仍是：**

```
用户发消息 → 平台预先组装 Prompt → LLM 生成一段话 → 等待下一条用户消息
```

这两个架构有根本区别：

| 维度 | 当前系统（含 Phase 0-2 重构） | 目标系统（专业HCI排障Claude）|
|------|------------------------------|----------------------------|
| **控制权** | 平台决定注入什么知识，LLM 被动接受 | LLM 主动决定调用什么工具 |
| **信息获取** | 一次性 RAG 注入，LLM 无法追加获取 | LLM 随时按需调用工具获取信息 |
| **推理模式** | 单轮生成（一条消息对应一次生成） | 多步推理（一条消息 → N次工具调用 → 最终回答）|
| **诊断能力** | 被动回答用户提供的信息 | 主动侦察系统状态 |
| **知识角色** | RAG 是"地基"，决定能不能回答 | RAG 是"工具之一"，与其他工具平等 |

> **与"当前重构方案（Phase 0-2）"的关系**：
> Phase 0-2 是 **过渡阶段**（改善体验，快速上线），目标架构是 **Phase 3 的核心任务**。
> 两者不冲突，但必须认清：Phase 0-2 的终点不是目标终点。

```
                │

```

### 5.3 工具目录（Tool Catalog）

工具是 Agent 的能力边界，分三类：

#### 类型1：知识工具（Level 0，无需授权，静默执行）

| 工具 | 功能 |
|------|------|
| `search_knowledge(query, category?)` | RAG 语义检索 |
| `match_sop(query)` | SOP 精确匹配 |
| `get_hci_mechanism(topic)` | 查询 HCI 机制知识（待建）|

#### 类型2：诊断工具（Level 1，只读，提示后自动执行）

通过 SCP REST API 和 acli 安全容器执行，底层物理隔离，平台保障安全底线：

| 工具 | 功能 |
|------|------|
| `get_active_alerts()` | 获取当前活跃告警列表 |
| `get_failed_tasks()` | 获取失败任务列表 |
| `get_vm_status(vm_id)` | 查询虚拟机状态 |
| `get_cluster_health()` | 集群整体健康状态 |
| `acli_vm_list()` | 列出所有VM及状态 |
| `acli_host_list()` | 列出宿主机及资源使用 |
| `acli_vg_list()` | 列出存储卷组 |
| `search_logs(keyword, time_range)` | 日志关键字搜索 |

#### 类型3：操作工具（Level 2/3，有风险，强制用户授权）

| 工具 | 风险等级 |
|------|---------|
| `acli_vm_on(vm_id)` | Level 2（确认）|
| `acli_vm_off(vm_id)` | Level 2（确认）|
| `acli_vm_migrate(vm_id, host)` | Level 2（确认）|
| `modify_config(target, key, value)` | Level 3（强确认）|

#### 永久禁止工具（安全红线）

```
❌ 删除数据 / 格式化磁盘 / 删除 VM
❌ 修改认证配置（密码/证书/密钥）
❌ 关闭集群或宿主机
❌ 任何未经分析的批量操作
```

### 5.4 授权机制设计（风险分级策略）

```
┌──────────────────────────────────────────────────────────────┐
│                      授权分级策略                             │
│                                                              │
│  Level 0: 知识查询     → 静默执行，无需提示                   │
│           （search_knowledge / match_sop）                   │
│                                                              │
│  Level 1: 只读诊断     → 提示执行意图，3秒无操作自动执行       │
│           （get_vm_status / get_cluster_health ...）         │
│                                                              │
│  Level 2: 低风险操作   → 明确提示 + 确认按钮，不自动执行       │
│           （restart_vm / migrate_vm ...）                    │
│                                                              │
│  Level 3: 高风险操作   → 强确认 + 展示变更内容 + 输入确认码   │
│           （modify_config / restart_service ...）            │
│                                                              │
│  Level X: 禁止操作     → 无论如何都不执行，解释原因            │
└──────────────────────────────────────────────────────────────┘
```

会话开始时，用户可设置会话级授权策略（如"本次排障全程授权 Level 1 自动执行"），减少重复确认摩擦。

### 5.6 SCP 工具接入架构

**核心发现**：80% 以上的客户问题来源于两类——**告警确认**和**失败任务排查**。

**工具接入层：SCP（Service Control Platform）**

| 工具类型 | 接入方式 |
|---------|---------|
| 管理 REST API | 通过 SCP 提供（告警列表、任务列表、VM状态、集群健康等）|
| acli 命令 | 通过 SCP 封装后对外暴露（安全容器隔离，与 HCI host 彻底隔离）|
| 监控数据 | ❌ 暂无接口（Prometheus/Grafana 暂无直接调用方式）|

**acli 安全模型（关键）：**
```
acli 类比 VMware 的 esxcli，但有本质区别：
  esxcli: 直接在 ESXi host 上执行，权限高，风险高
  acli（本项目）: 封装在安全容器中，与 HCI host 彻底物理隔离
               只提供经过审计的安全命令子集
               → 工具调用安全底线由平台保证，AI 层无需重复防御
```

**两个诊断入口（基于 SCP 告警/任务的关键洞察）：**

```
入口A（用户文字描述）：走完整诊断状态机 S0→S6（问题澄清 → 假设形成 → 验证收敛）
入口B（告警/任务结构化数据）：直接跳到 S2/S3，效率大幅提升

示例：
  传统：用户"虚拟机起不来" → AI追问分类→ 多轮澄清
  SCP驱动：告警{type:"VM_BOOT_FAIL", error:"ECPU_OVERCOMMIT"} → 分类已知，直接验证
```

**新增架构组件：SCP Adapter**

```
AI（ReAct 循环）
    ↓ 工具调用请求
SCP Adapter（独立微服务 / MCP Server）
    ├── REST API 调用 → SCP
    └── acli 命令执行 → acli 安全容器 → HCI 集群（物理隔离）
```

### 5.7 关键技术决策

#### 一、授权机制：工具定义静态声明风险级别

**背景**：Claude 的隐式风险判断依赖顶尖模型能力，GLM 无法照搬。

**本项目方案：风险级别在工具定义时静态声明，框架强制执行，LLM 不需要判断风险。**

```python
TOOL_REGISTRY = [
    {
        "name": "get_active_alerts",
        "risk_level": 1,        # 静态声明，框架控制
        "policy": "auto",
        "description": "获取当前活跃告警列表",
    },
    {
        "name": "vm_migrate",
        "risk_level": 2,
        "policy": "confirm",    # 框架强制要求用户确认
        "description": "迁移虚拟机到指定宿主机",
    },
]
```

| 对比维度 | Claude 隐式方式 | 本项目显式方式 |
|---------|---------------|--------------|
| 谁判断风险 | LLM 在推理时 | 工具定义里静态声明 |
| 对模型要求 | 极高（顶尖模型） | 低（只需会选工具）|
| 可靠性 | 依赖模型，可能漂移 | 确定性，框架强制 |
| 可审计性 | 难以追溯 | 每次调用都有策略记录 |

> **GLM-4 的 function calling 能力完全足够**：它只需要"选工具、填参数"，不需要判断风险。这比 Claude 的方式**更可靠**，也**更适合生产环境审计**。

---

#### 二、框架选型：自实现轻量 ReAct，不用 LangChain / LangGraph

**结论：自己实现，约 100 行核心代码。**

**理由：**

1. **GLM 格式差异**：两个框架以 OpenAI `tool_use` 格式为基础设计，GLM 在并行工具调用、降级行为、历史剪裁等细节上存在差异，会触发难以调试的隐式失败。

2. **状态机已确定，图结构是负担**：LangGraph 适合"分支由 LLM 动态决定"的场景。本项目的 7 阶段诊断状态机是领域约束下的确定性转换，不需要 `StateGraph`。

3. **与现有架构集成代价高**：`conversation-service` 已有 FastAPI + SSE + SQLAlchemy，引入 LangChain 每一层都是阻抗失配。

4. **ReAct 核心逻辑约 30 行**，简单透明，可完全掌控：

```python
async def run_react_loop(session, user_input):
    messages = session.history + [{"role": "user", "content": user_input}]

    for _ in range(MAX_STEPS):
        response = await llm.chat(messages, tools=TOOL_REGISTRY)

        if response.finish_reason == "stop":
            return response.content                  # 最终答案，退出

        if response.finish_reason == "tool_calls":
            for tool_call in response.tool_calls:
                policy = get_tool_policy(tool_call.name)   # 风险由框架检查
                if policy == "confirm":
                    await sse_push_confirmation_request(tool_call)
                    result = await wait_for_user_confirmation()
                else:
                    result = await execute_tool(tool_call)

                messages.append(tool_result_message(tool_call, result))
```

5. **排障场景不能有黑盒**：LangChain 的隐式重试可能导致操作日志不完整，影响事后审计。

**各组件实现方案汇总：**

| 组件 | 方案 |
|------|------|
| ReAct 执行器 | 自实现 ~100 行，内嵌 conversation-service |
| 工具注册表 | Python 列表 + Pydantic 模型（含 risk_level、policy 元数据）|
| 对话状态 | 扩展现有 ConversationSession，加 diagnostic_stage 字段 |
| SCP Adapter | 新增独立微服务或 client 模块，封装 SCP REST API + acli |
| 授权层 | 框架硬编码规则：risk_level → policy |
| 人工确认 | 现有 SSE 通道，推确认请求，等前端响应 |
