# HCI 智能排障平台 — 核心设计理念与差距分析

> **文档定位**：WHAT — 为什么当前系统不够好，目标系统应该是什么样的（理论框架与设计原则）
> **版本**：3.0 | **更新日期**：2026-03-25
> **关联文档**：[05 架构决策](./05_AI助手层设计.md) | [11 实施规范](./11_完整技术方案.md)

## ⚡ 一、核心架构原则（最高优先级，所有设计决策的基础）

> **此原则是在 2026-03-23 通过对比 Claude Code Skill 技术文档与项目 SOP Skill 实现后得出的关键洞察，
> 必须在所有涉及 SOP / KB / 案例库 / 知识注入的设计中优先遵守。**

### 所有知识注入的本质是「条件触发的上下文注入」

无论是 SOP 技能文件、RAG 案例 chunks、工具文档，还是 Claude Code 的 Skill——
**在 LLM 眼中，它们具有完全相同的技术本质**：

```
触发条件满足
    ↓
将预先编写/检索到的结构化内容注入 LLM 的上下文窗口（Context Window）
    ↓
LLM 依据注入内容进行推理和生成
```

它们的差异**不是本质差异，而是实现质量的差异**：

| 维度 | 劣质实现 | 优质实现 |
|---|---|---|
| **触发机制** | 精确关键字字符串扫描（当前 SopMatcher） | Embedding 语义相似度匹配（Claude Code Skill 的 description 路由） |
| **注入粒度** | 整篇SOP文档 / 512-token 段落切块 | 知识原子（有类型标签 + 触发条件的最小语义单元） |
| **触发时机** | 用户第一句话到来时（S0，信息最模糊时） | 问题域界定后（S1），或 LLM 推理后自主决定（方案A） |
| **动态性** | 纯静态文本注入 | 支持运行时数据填充（`!``command``预处理） |

### 因果链：路由质量 → 注入质量 → LLM推理质量

```
触发机制质量
    └→ 决定"注入了什么"的精准度
            └→ 决定 LLM 在 context window 中"看到什么"
                    └→ 直接决定推理质量和回复质量

关键字匹配（最弱）< 精确分类过滤 < Embedding语义路由（最强）
```

### 直接推论（影响所有下游设计）

1. **SopMatcher 的关键字路由是已知最差的触发机制**，"主机离线了" 无法命中 "节点降级" SOP——用户语言与SOP关键字之间的gap是永久性的，无法通过增加关键字解决
2. **SOP 与 RAG 不应该是两套系统**，它们是同一件事（条件触发的上下文注入）的两种粒度，应统一为语义路由 + 知识原子
3. **检索时机必须在 S1（范围界定后）而不是 S0（症状模糊期）**，S0 用模糊 query 检索相当于用最差质量的输入触发整个注入链
4. **注入内容的类型标签是必须的**：diagnostic_step / fix_action / background 注入后 LLM 的使用方式完全不同，混注 = 降低所有内容的可用性

## ⚡ 二、知识源深度对比分析（真实数据验证，2026-03-23）

> 以真实原始数据为证据，验证并延伸核心架构原则，确立「双轨知识架构」的设计基础。
> 所有后续 SOP / KB 存储设计、检索策略、Prompt 组装逻辑必须以本节结论为准。

### SOP（程序性知识）vs KB案例（陈述性知识）：真实样本对比

**样本 A — SOP 原始数据**（来源：`data-pipeline/sop_skills/vm_power_failure/chapters/sub_内存不足.md`）

```markdown
## 有启动虚拟机失败任务 → 内存不足

### 前置检查
获取【启动虚拟机】失败任务的【描述】信息：
  acli task get -v ${vmid} -t ${YYYY-mm-dd} -k '启动虚拟机' -s 'failed'
关键字 → 章节：CPU不足 / 内存不足 / 内部异常 / 序列号过期 / ...（共 20+ 分支）

### 内存不足
#### 判断方法
  acli system memory info | grep MemAvailable
  - MemAvailable < 2G → 内存耗尽
#### 解决方案
  关闭部分业务虚拟机 / 在线迁移虚拟机到其他主机
```

**样本 B — KB案例原始数据**（来源：`data-pipeline/cases_raw/cases.jsonl`，案例ID: 33780）

```
标题：虚拟机启动失败，没有主机能够启动这台虚拟机，错误码：0x0C000005
【根因】因为编辑虚拟机重要性导致虚拟机内存分组切换，cgroup bug 导致内存计算错误
【解决方案】
  1. 临时：后台使用 qm start vmid 命令将虚拟机先以非大页拉起来
  2. 重启主机释放空间
  3. 升级 6.12.0 解决（补丁待出）TD2025072800477
```

### 七维对比分析

| 维度 | SOP（程序性知识） | KB案例（陈述性知识） | 对 AI Agent 的意义 |
|------|-----------------|---------------------|-------------------|
| **1. 结构化程度** | 高：树状决策结构，前置检查→20+分支→判断条件→执行步骤 | 中：固定字段，内容是叙事性自然语言 | SOP 结构可被 Agent 直接"遍历"；案例需要 Agent 提炼 |
| **2. 触发条件精确性** | 极高：精确报错字符串 + 数值阈值（Percpu>5G, VmallocUsed>10G） | 中：有错误码（0x0C000005），但缺乏判断阈值 | SOP 提供"何时做决策"；案例提供"曾经发生了什么" |
| **3. 命令可执行性** | 高：完整 acli 命令 + 读哪个字段 + 判断阈值 | 低：描述了"查看"而非命令，无完整命令语法 | SOP 可直接驱动 tool_call；案例只能作参考 |
| **4. 根因明确性** | 低：给解决方案，不解释底层 why | **极高：明确记录了 cgroup bug 是根因** | 案例的根因是假设生成的黄金来源；SOP 不记录此类归因 |
| **5. 覆盖范围** | 宽：一个章节覆盖多种失败路径 | 窄：每个 case 是唯一的失败场景 | SOP 是泛化知识；案例是特化实例 |
| **6. 时效性** | 相对稳定：排查方法不频繁过期 | 强时效：版本号、补丁号、TD跟踪号都是时点信息 | 案例必须有版本元数据才能安全使用 |
| **7. AI 执行友好性** | **最高**：Agent 可按树结构执行、对阈值判断、调用命令 | 低：叙事步骤，Agent 看到"查看"后不知执行什么 | SOP 是程序，案例是证言 |

### 核心结论：两类知识承担不同的认知职责

```
SOP（程序性知识）= "如何做"
  → 适合放进 Agent 的「动作执行上下文」
  → 类比：机械师手册 — 拿到故障代码直接按步骤走

KB案例（陈述性知识）= "曾经发生过什么"
  → 适合放进 Agent 的「相似案例记忆库」
  → 类比：医院病历 — 参考历史病例辅助诊断，不能直接复制治疗方案
```

**它们不是竞争关系，而是两个不同的认知层次：**

```
用户报故障
    ├─[SOP路由] 语义路由 → 命中SOP → 注入SOP知识原子 → Agent 走程序
    │                       ↓ 未命中
    └─[KB检索] 语义相似度 → 找历史相似案例 → 注入案例根因 → Agent 形成假设
                             ↓ 未命中
                    [机制推理兜底] → 标注【机制推理】
```

### 双轨知识架构的设计推论

**SOP 轨道设计原则：**
1. **存储单元**：知识原子（knowledge_atom），不做 512-token 滑块，按决策树节点分割
2. **每个原子包含**：序列路径、知识类型（diagnostic_step / decision_gate / threshold / fix_action）、可执行命令、判断条件
3. **路由机制**：语义匹配（embedding 相似度），而非关键字字符串扫描
4. **格式标签**：注入 Prompt 时标注 `【SOP 排障流程】`，LLM 知道此内容应被严格执行

**KB 轨道设计原则：**
1. **存储单元**：完整案例文档（不拆分），向量化 `problem_desc + root_cause` 字段
2. **路由机制**：BM25 + 向量混合检索，主要匹配 `problem_desc` 字段
3. **Agent 行为**：从 `root_cause` 提取假设 → 用 `solution` 作参考（非配方）→ 结合当前环境验证假设
4. **格式标签**：注入 Prompt 时标注 `【历史案例参考】`，LLM 知道此内容是证据而非程序

### 三级 Fallback 触发规则

```
Priority 1: SOP 命中
  → system_prompt 包含 [SOP排障流程] 段
  → AI 走 SOP 程序执行路径

Priority 2: SOP 未命中，KB 案例命中（≥1 条相似度阈值以上）
  → system_prompt 包含 [历史案例参考] 段
  → AI 用案例 root_cause 生成假设，方向性排查

Priority 3: 双轨均未命中
  → system_prompt 包含 [机制推理模式] 段
  → AI 基于领域训练知识推理，所有结论标注【机制推理】
```
