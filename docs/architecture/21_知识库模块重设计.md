## 一、现状分析与问题清单

### 1.1 数据现状（实测）

| 维度 | 数据 |
|------|------|
| **KBD 目标条数** | **5,557 条**（来自 `案例生产详细数据24-26.xlsx` 第一列，已实测） |
| KBD 来源 API | `GET https://support.sangfor.com.cn/spt/openapi/case/es/getDetailById/{id}` |
| KBD 认证方式 | Cookie（PHPSESSID + HMACCOUNT 等），需从浏览器定期刷新 |
| **图片关键性** | **⚠️ 重大约束：大量 KBD 的关键排查信息在截图里**；没有图片语义化，数据不可用 |
| cases.jsonl 可用性 | 低——仅有文字字段，无图片内容，**不可直接入库** |
| KBD 三个核心字段 | 问题描述、有效排查步骤、解决方案（其余字段非必须）|
| KBD 旧分类字段 | `main_module` — **废弃，不再使用** |
| SOP | 新建生命周期管理。旧有 `sop_skills/` 为 legacy，不纳入改造范围 |
| category_baseline 节点 | 198 个叶节点，5 个领域（虚拟机54 / 存储46 / 硬件45 / 平台30 / 网络23） |
| kb_category 表 | 当前为**空**，无 `code` 字段，category_baseline.yaml 从未导入 DB |
| knowledge_atoms 表 | category_id 为无外键约束的 VARCHAR（与 category_baseline 完全脱节） |

### 1.2 当前架构存在的核心问题

| 编号 | 问题 | 影响 |
|------|------|------|
| P1 | KBD `main_module` 旧分类字段废弃，入库时必须通过 LLM 分类到 `category_id` | 无分类则无法按 category 路由检索 |
| P2 | `knowledge_atoms` 通过 atom_writer.py 直接写 DB，绕过 kb-service API | 服务边界破坏，难以审计 |
| P3 | enricher.py 输出 `fault_category` 为自由字符串，与 `kb_category` 无任何关联 | 分类数据孤岛，无法聚合分析 |
| P4 | `kb_category` 表为空且无 `code` 字段，无法与 category_baseline.yaml 对齐 | category_id 无法作为跨系统的稳定标识 |
| P5 | 没有审核工作流：KBD 入库后直接可用，没有 draft→review→publish 状态流转 | 低质量数据直接污染检索结果 |
| **P6** | **图片信息完全缺失**：cases.jsonl 仅有文字字段，大量关键排查信息在截图里，导致现有数据不可用 | 必须重新从 sangfor.com API 抓取 + Vision LLM 处理图片 |

## 二、业务需求确认

### 2.1 核心需求
- **分类锚点**：所有知识必须锚定到 category_baseline.yaml 的某个叶节点（198 个，唯一 id 如 `虚拟机-003`）
- **两种知识形式**：
  - **KBD**（知识库文档）：~1,000 字，1,894 条起步，目标 6,000 条，来源 sangfor.com
  - **SOP**（标准操作规程）：~20,000 字，当前 2 个，持续增加，来源 docx 或从 KBD 聚合
- **完整生命周期**：生产 → 审核 → 消费 → 归档，**统一由 kb-service 实现，不允许 bypass**
  - kb-service 是唯一有权写 `kbd_entry` / `sop_document` / `sop_chunk` 的服务
  - 外部脚本（fetcher / SOP 导入工具）只做数据采集，最终必须调用 kb-service API
- **审核入口**：admin-ui 的分类管理页 + 知识审核页，审核后才可消费
- **消费方式**：category_baseline 用于意图识别（输出确定的 category_id），并为对话提供检索上下文

### 2.2 架构边界（明确）

```
┌─────────────────────────────────────────────────────────────┐
│ 采集层（外部脚本，只读外部数据源）                            │
│   scripts/fetch_kbd.py  ← sangfor.com 结构化 API            │
│   scripts/import_sop.py ← 本地 docx 文件                    │
└─────────────────┬───────────────────────────────────────────┘
                  │ HTTP POST（调用 kb-service API）
                  ▼
┌─────────────────────────────────────────────────────────────┐
│ kb-service（统一生命周期，唯一写入权）                        │
│   生产： POST /api/kbd/import  POST /api/sop/import          │
│   分类： POST /api/kb/classify（LLM→category_id）           │
│   审核： POST /api/admin/kb/{id}/approve|reject              │
│   消费： GET  /api/kb/route  GET /api/kb/categories          │
│   归档： POST /api/admin/kb/{id}/archive                     │
└─────────────────────────────────────────────────────────────┘
```

### 2.3 生产路径（修正）

**KBD 生产路径**（图片是关键信息，必须下载并进行 Vision LLM 语义化后才可入库）：
```
Excel（案例生产详细数据24-26.xlsx）— 5,557 个案例 ID
  │
  ▼ [Stage 1: fetch  — scripts/kbd/run.py fetch]
sangfor.com API（Cookie认证）
  ├── GET /spt/openapi/case/es/getDetailById/{id}
  ├── 案例详情 JSON → cache/{case_id}/raw.json（flock幂等，存在跳过）
  └── 提取 img src → 下载图片 → cache/{case_id}/img_{N}.png
  │
  ▼ [Stage 2: vision — scripts/kbd/run.py vision]
Vision LLM（gpt-4o，base64图片输入）
  └── 每张截图 → 中文语义描述 → cache/{case_id}/img_{N}.desc.txt
  │
  ▼ [Stage 3: import — scripts/kbd/run.py import]
HTML + 图片描述 → 三段结构化 Markdown（问题描述/排查步骤/解决方案）
  └── 写 kbd_entry（status=draft）
  │
  ▼ [Stage 4: classify — scripts/kbd/run.py classify]
LLM 分类（gpt-4o-mini，198个分类节点供选）
  └── title + problem_desc → category_id → 更新 kbd_entry.ai_category_id + confidence
  │
  ▼ [Admin Review — admin-ui 知识审核页]
审核人查看 rendered Markdown + AI 分类建议
  ├── 可编辑 content_md（修订截图描述/措辞）
  ├── 可修改分类（从198节点中选择）
  └── approve → published（触发 embedding 生成）
         └── reject → rejected（审核人填写拒绝原因）
```

**SOP 生产路径1**（docx 源文件）：
```
本地 .docx 文件  →  scripts/import_sop.py（docx→章节MD）
                 →  POST /api/sop/import
                 →  admin-ui 审核
                 →  status=published，触发 sop_chunk embedding
```

**SOP 生产路径2**（从 KBD 聚合）：
```
admin-ui 选择同一 category 下的已审核 KBD
         →  POST /api/admin/sop/aggregate（LLM 聚合生成 draft SOP）
         →  admin-ui 审核编辑
         →  status=published
```

## 三、三套设计方案

---

### 方案A：统一文档层 + 分类锚点（推荐）

**核心思想**：以 `kb_document` 为统一存储层（KBD 和 SOP 都在此），用**确定的 category_id**（对应 category_baseline.yaml 叶节点）做锚点，chunk 层负责语义检索，knowledge_atoms 作为从 SOP 提炼的步骤缓存。

#### A.1 数据模型变更

```
kb_category（category_baseline 镜像）
  ┣━ id: VARCHAR(32) PRIMARY KEY       # 直接用 YAML 的 id（如"虚拟机-003"）
  ┣━ domain: VARCHAR(50)               # 领域（虚拟机/存储/硬件/平台/网络）
  ┣━ label: VARCHAR(100)               # 可读标签
  ┣━ path: TEXT[]                      # 完整路径数组
  ┗━ embedding: vector(1536)           # 用于意图识别的向量（text-embedding大模型）

kb_document（统一知识文档，KBD 和 SOP 都在这里）
  ┣━ id: SERIAL PK
  ┣━ source_id: VARCHAR(50) UNIQUE     # 原始 ID（KBD: cases.id; SOP: sop-xxx）
  ┣━ doc_type: VARCHAR(10)             # 'kbd' | 'sop'（新增，区分两种形式）
  ┣━ category_id: VARCHAR(32) NOT NULL # FK → kb_category.id（强制锚定，为空不允许）
  ┣━ title: VARCHAR(500)
  ┣━ content_md: TEXT                  # KBD 用模板 MD；SOP 用完整 MD
  ┣━ summary: TEXT                     # LLM 生成摘要（500字以内）
  ┣━ judgment_logic: TEXT              # 排查逻辑（LLM生成）
  ┣━ tags: TEXT[]
  ┣━ status: VARCHAR(20)               # draft→under_review→published→archived
  ┣━ reviewer, reviewed_at, review_note
  ┗━ source_type: VARCHAR(20)          # 'sangfor_web'|'docx'|'aggregated_sop'

kb_chunk（语义检索层，KBD 和 SOP chunk 都在这里，按 doc_type 分工）
  ┣━ id: SERIAL PK
  ┣━ document_id: FK → kb_document.id
  ┣━ chunk_type: VARCHAR(20)           # 新增：'full_doc'|'section'|'step'
  ┣━ content: TEXT
  ┣━ embedding: vector(1536)           # 统一用1536维（文档量小，精度优先）
  ┣━ tsv: tsvector                     # BM25
  ┗━ metadata: JSONB                   # {section_title, step_index, ...}

knowledge_atoms（SOP 步骤缓存，仅 SOP 来源）
  ┣━ id: VARCHAR(32) PK
  ┣━ document_id: INTEGER FK → kb_document.id  # 新增：必须关联到 SOP 文档
  ┣━ category_id: VARCHAR(32) FK → kb_category.id  # 改为强制外键
  ┣━ atom_type: VARCHAR(32)
  ┣━ trigger_json: JSONB
  ┣━ content_json: JSONB
  ┗━ verified: BOOLEAN

raw_cases（ETL 暂存，不变）
```

#### A.2 服务边界

```
data-pipeline（只写 raw_cases + 触发 kb-service API）
  fetcher → raw_cases（暂存）
  enricher → 调用 POST /api/kb/classify（LLM 分类到 category_id）
  ingestor → 调用 POST /api/kb/ingest（写 kb_document + kb_chunk）

kb-service（统一 KB 操作，不允许外部直接写 kb_document/kb_chunk/knowledge_atoms）
  POST /api/kb/ingest          → 写 kb_document + kb_chunk（状态=draft）
  POST /api/kb/classify        → LLM 分类，输出 category_id
  POST /api/kb/search          → BM25+向量混合检索
  POST /api/kb/atoms/search    → 知识原子检索
  GET  /api/kb/categories      → 返回 category_baseline 树（给意图识别用）

admin-service（审核工作流）
  GET  /api/admin/kb/pending   → 待审核列表
  POST /api/admin/kb/{id}/approve → 审核通过（status→published）
  POST /api/admin/kb/{id}/reject  → 驳回（status→rejected + review_note）
  POST /api/admin/kb/categories/sync → 同步 category_baseline.yaml 到 kb_category 表
```

#### A.3 检索流程（三轨串行路由）

```
第0轨：意图识别
  用户输入 → embedding → cos_sim(kb_category.embedding) → top3 候选 → LLM 确认 → category_id

第1轨：SOP 优先
  kb_document WHERE category_id=X AND doc_type='sop' AND status='published'
    → 有结果: 加载 SOP chunks → knowledge_atoms 步骤引导 → 解决 → 结案
    → 无结果或未解决: 进入第2轨

第2轨：KBD 覆盖
  kb_chunk WHERE document_id IN (SELECT id FROM kb_document WHERE category_id=X AND doc_type='kbd')
    → BM25(tsv) + 向量 → RRF 融合 → top10 chunks → LLM RAG context
    → 解决 → 结案
    → 未解决: 进入第3轨

第3轨：人工兜底
  生成升级工单，转人工处理
```

> 三轨之间是**串行路由**，不是并行合并。KBD 和 SOP 结果永远不会在同一次响应中混排。

#### A.4 优缺点

| 优点 | 缺点 |
|------|------|
| KBD/SOP 统一存储，生命周期管理简单 | `knowledge_atoms.document_id` 新增约束，需要迁移现有数据 |
| `category_id` 强外键，分类数据一致性有保障 | SOP 入库后 chunk 爆炸：1个75KB SOP → ~150个 chunk |
| 服务边界清晰（所有写操作走 kb-service API） | 意图识别需要给 198 个节点生成 embedding（一次性启动任务） |
| 检索统一（BM25+向量 RRF，KBD和SOP共用） | KBD 和 SOP 在同表但形态差异大，chunk_type 字段需仔细维护 |
| 审核状态机统一（所有知识 draft→published） | `judgment_logic` 等 LLM 生成字段质量依赖 prompt 工程 |

### 方案B：双轨分离（KBD / SOP 独立表）

**核心思想**：KBD 和 SOP 有本质形态差异（字数、结构、检索粒度不同），用独立表保持 schema 语义清晰。通过统一的 `category_id` 做路由键，检索逻辑采用**三轨串行路由**（SOP优先 → KBD覆盖 → 人工兜底），两张表**串行独立查询，无需合并结果**。

#### B.1 数据模型

```
kb_category（同方案A：category_baseline 镜像）

kbd_entry（KBD 专表）
  ┣━ id: SERIAL PK
  ┣━ source_id: VARCHAR(50) UNIQUE    # cases.id
  ┣━ category_id: VARCHAR(32) NOT NULL FK → kb_category.id
  ┣━ title, problem_desc, troubleshoot_steps, root_cause, solution（4个原始字段保留）
  ┣━ summary: TEXT                    # LLM 摘要（用于 LLM context 注入，约500字）
  ┣━ tags: TEXT[]
  ┣━ embedding: vector(1536)          # **单条 KBD 直接 embedding**（不需要 chunk，体量小）
  ┣━ tsv: tsvector
  ┗━ status: VARCHAR(20)              # draft→published→archived

sop_document（SOP 专表，保留原 kb_document 字段集）
  ┣━ id: SERIAL PK
  ┣━ source_id: VARCHAR(50) UNIQUE    # sop-vm-start-failure
  ┣━ category_id: VARCHAR(32) NOT NULL FK → kb_category.id
  ┣━ title: VARCHAR(500)
  ┣━ content_md: TEXT                 # 完整 MD
  ┗━ status: VARCHAR(20)

sop_chunk（SOP 拆块，schema 同原 kb_chunk）
  ┣━ 复用原 kb_chunk 表，增加 chunk_type VARCHAR(20)

knowledge_atoms（同方案A，增加 sop_document_id FK）
```

#### B.2 检索路由（三轨串行）

```python
# kb-service 内部三轨串行路由（无合并，顺序尝试）
async def route(category_id: str, query: str) -> SearchResult:

    # 第1轨：SOP 优先
    sop = await sop_document_repo.get_published(category_id)
    if sop:
        chunks = await sop_chunk_repo.search(sop.id, query)   # chunk RRF
        return SearchResult(track="sop", sop_id=sop.id, chunks=chunks)

    # 第2轨：KBD 覆盖
    kbd_docs = await kbd_entry_repo.search(category_id, query)  # 单条 embedding + BM25
    if kbd_docs:
        return SearchResult(track="kbd", docs=kbd_docs)

    # 第3轨：人工兜底
    return SearchResult(track="human_escalation", category_id=category_id)
```

> **没有结果合并（merge）**。`sop_document` 和 `kbd_entry` 从不在同一次请求中同时返回结果。

#### B.3 优缺点

| 优点 | 缺点 |
|------|------|
| KBD 整条 embedding（无分块），对 ~600 字短文档天然适配，检索精度更高 | 两张主表各自维护，新增公共字段（如 tag 管理）需改两张表 |
| SOP schema 独立，可存 SOP 专属元数据（chapter_count / docx_hash / version）| admin-ui 审核页需区分两种知识类型（可通过 tab 区分，影响有限） |
| KBD 原始字段（problem_desc / root_cause）直接存储，运营人员审核直观 | 报表统计 SQL 需 UNION 两表（如"各 category 下知识总条数"）|
| 三轨路由逻辑最简洁：先查 sop_document，有则走 SOP；无则查 kbd_entry | category_id FK 需要在两张表各自维护 |
| KBD 入库无需分块，pipeline 最简单 | — |

### 方案C：知识原子优先（Atom-First）

**核心思想**：所有知识最终都被提炼为 `knowledge_atoms`（知识原子），`kb_document/kbd_entry` 只作为 source archive 存在。系统直接检索和引用知识原子，通过原子组合实现 SOP 引导。这是 RAG → Agent 演进路径上的终态架构，但当前执行成本极高。

#### C.1 数据模型

```
kb_category（同 A/B）

knowledge_source（source archive，KBD/SOP 原文归档）
  ┣━ id: SERIAL PK
  ┣━ source_type: 'kbd'|'sop'
  ┣━ content_md: TEXT
  ┣━ status: VARCHAR(20)
  ┗━ category_id: VARCHAR(32) FK

knowledge_atoms（核心检索表，大幅扩展）
  ┣━ id: VARCHAR(32) PK
  ┣━ atom_type: VARCHAR(32)            # diagnostic_step|fix_action|decision_gate|threshold|error_code_ref|background|symptom（新增symptom）
  ┣━ source_id: INTEGER FK → knowledge_source.id
  ┣━ category_id: VARCHAR(32) FK → kb_category.id
  ┣━ title: VARCHAR(200)              # 原子标题（LLM生成，可读）
  ┣━ content_md: TEXT                 # 原子内容
  ┣━ embedding: vector(1536)          # 直接对原子做 embedding
  ┣━ tsv: tsvector
  ┣━ prerequisites: VARCHAR(32)[]     # 前置原子 ID 列表（有向图）
  ┣━ trigger_json: JSONB              # 触发条件
  ┣━ verified: BOOLEAN
  ┗━ confidence: NUMERIC(3,2)

-- 无需 kb_chunk 表，atom 本身就是检索单元
```

#### C.2 LLM 提炼管道（从 KBD/SOP → atoms）

```
KBD (1条 ~600字) → LLM → 3-8个 atoms 
  (symptom 1个 + root_cause 1个 + fix_action 1-3个 + background 0-2个)

SOP (1个 ~20000字) → LLM 分段提炼 → 50-200个 atoms
  (决策树结构：decision_gate + diagnostic_step + fix_action + threshold)

总 atom 数量估算：
  1,894 KBD × 5原子/条 ≈ 9,470 atoms
  2 SOP × 100原子/个 ≈ 200 atoms
  合计 ≈ 9,670 atoms（规模可控）
```

#### C.3 优缺点

| 优点 | 缺点 |
|------|------|
| 检索粒度最细，一个 atom ~200字，精确度最高 | LLM 提炼管道复杂，1,894 条 KBD 需 ~200 次 LLM 调用（按批8条/次），耗时约1小时 |
| 支持有向图式推理（prerequisites 链路） | KBD 原始字段（problem_desc等）被 LLM 提炼后信息可能丢失，难以溯源 |
| 从 RAG 向 Agent 自然演进（atoms可直接用于工具调用） | 提炼质量强依赖 LLM prompt，人工审核 9,000+ atoms 工作量极大 |
| 排障引导可以是 atom 执行序列，而不是大段文字 | 当前团队 LLM 成本和运维能力是否支持？ |
| embedding 直接在 atom 级別（精确），无 RRF 噪音 | 提炼过程是黑盒，LLM 幻觉风险高，需要额外的验证机制 |

|------|------|------|------|

## 五、推荐与理由

### 方案A vs 方案B 重新评估（修正后）

「Federated Search 复杂度高」是误判——三轨串行路由下两表永远不合并结果。修正后**两个方案的核心差异只剩一点**：

| 差异点 | 方案A | 方案B |
|--------|-------|-------|
| KBD 存储形态 | 4个字段合并成 Markdown 全文，再分块 | 4个字段独立列存储，直接可审 |
| KBD 检索粒度 | chunk（~512 token，对 600 字文档意义边际） | 整条 embedding（短文档天然最优）|
| 三轨路由 SQL | `WHERE doc_type='sop'` / `'kbd'` | 直接查 `sop_document` / `kbd_entry`（语义更清晰）|
| schema 维护 | 一张主表 | 两张主表（SOP极少，实际维护量很小）|

### 推荐：**方案B（双轨分离）**

**理由**：
1. **KBD 字段结构天然适配方案B**：4个字段（problem_desc/troubleshoot_steps/root_cause/solution）直接列存储，运营审核时一目了然；方案A要拼 Markdown 再分块，600字文档二次分块增加检索噪音且没有精度收益
2. **三轨路由更简洁直观**：「查 sop_document → 有则走 SOP，无则查 kbd_entry」，SQL 语义比 `WHERE doc_type=?` 更清晰，减少类型判断错误风险
3. **SOP量极少，双表维护成本可接受**：当前2个SOP，预期不超过20个，sop_document 几乎不变更；kbd_entry 高频写入。两表访问模式完全不同，分离反而更清晰
4. **「Federated Search 复杂度高」是误判**，该缺点不成立

### 方案A 仍适合的情况
- 如果未来 KBD 来源会扩展为非结构化长文档（失去4字段结构），统一表更灵活
- 如果运营团队对"一张表管理所有知识"有强烈偏好

### 7.3 流水线四个 Stage

```
scripts/kbd/run.py — CLI 入口
  ├── pipeline  全量运行（或 --stages 指定）
  ├── fetch     Stage 1
  ├── vision    Stage 2
  ├── import    Stage 3
  └── classify  Stage 4

scripts/kbd/
  ├── config.py       配置（pydantic-settings，从 .env 读取）
  ├── fetcher.py      HTTP 抓取 + 图片下载（写文件到 cache/{case_id}/）
  ├── image_proc.py   Vision LLM 图片语义化（写 img_{N}.desc.txt）
  ├── converter.py    HTML + 图片描述 → 三段 Markdown
  ├── classifier.py   LLM 分类（198 节点，输出 category_id）
  ├── importer.py     写 kbd_entry（幂等，已审核的不覆盖）
  ├── pipeline.py     Stage 编排（断点续传、跳过已完成）
  └── .env.example    配置模板
```

#### Stage 1: fetch（`fetcher.py`）

```python
# HTTP 请求规格
GET {SANGFOR_API_BASE}/spt/openapi/case/es/getDetailById/{case_id}
Headers:
  Accept: application/vnd.edusoho.v2+json
  Cookie: {SANGFOR_COOKIE}         # 从 .env 注入
  Http_x_requested_with: xmlhttprequest
  Referer: https://support.sangfor.com.cn/cases/list?product_id=33&type=1
```

- 幂等：`cache/{case_id}/raw.json` 存在则跳过（可 `--force` 删除后重跑）
- 请求间隔：`SANGFOR_REQUEST_DELAY=0.8s`（避免触发限流）
- 每案例图片下载：异步并发（`VISION_CONCURRENCY=3`）
- Cookie 过期（401）时：**立即停止整个批次**，提示用户刷新 Cookie

#### Stage 2: vision（`image_proc.py`）

- 读本地图片文件 → Base64 编码 → `data:image/{mime};base64,{b64}`
- Prompt 专为 HCI 故障排查截图设计：强调错误码、状态值、命令输出
- 失败（API 超时/错误）：写 `img_{N}.failed`，不中断批次，可重跑
- 图片过小（`< MIN_IMAGE_SIZE=2048 字节`）：跳过（icon/占位图）

#### Stage 3: import（`converter.py` + `importer.py`）

**HTML → Markdown 转换策略**：

1. 尝试从 API 响应 JSON 中解析出三个分段（按标题关键字匹配）
2. 在每个 HTML 段中，将 `<img>` 标签替换为：
   ```markdown
   > **【截图说明】**：{vision_desc}
   ```
3. 使用 `markdownify` 将 HTML 转为 Markdown
4. 三段拼合为 `content_md`：
   ```markdown
   # {title}

   ## 问题描述
   {problem_desc_md}

   ## 排查步骤
   {troubleshoot_md}

   ## 解决方案
   {solution_md}
   ```
5. 写入 `kbd_entry`（`status='draft'`，幂等：已进入 pending_review/published/archived 的不覆盖）

#### Stage 4: classify（`classifier.py`）

```
Prompt = 198个分类节点（code | 路径 | 标签）
输入 = 案例标题 + 问题描述（截断至800字）
输出 = {"category_id": "虚拟机-001", "confidence": 0.85, "reason": "..."}
```

- 校验 `category_id` 是否在 198 个合法 code 中（LLM 有时会错误创造新 ID）
- 低置信度（`< 0.5`）：写入但标记，审核页提示「需人工重新分类」
- 更新 `kbd_entry.ai_category_id`，**不修改 status**

### 9.2 数据库表设计（最终确认版）

> 迁移脚本：`database/migrate_kbd_pipeline_v1.sql`（已创建）

#### 知识库相关表全览（共 4 张）

```
kb_category          — 分类树（198节点，category_baseline镜像）
kbd_entry            — KBD知识条目（完整生命周期）
sop_document         — SOP文档（待实现）
sop_chunk            — SOP分块检索（待实现）
```

> **设计决策（已确认）**：KBD 生产阶段的中间产物（原始 JSON、图片文件、Vision 描述）**使用文件存储**，不入库。
> 理由：①断点续传用文件存在性判断即可；②并发幂等用 `flock` 文件锁；③Vision token 成本不需要聚合统计；④省去维护两张过渡表（kbd_raw/kbd_image）的成本。
>
> **文件存储结构（work_dir 由 `config.py` 的 `KBD_CACHE_DIR` 配置）**：
> ```
> scripts/kbd/cache/
> ├── {support_id}/
> │   ├── raw.json           # API 完整响应（存在 = fetch done，跳过）
> │   ├── fetch.failed       # 失败记录（JSON，含错误信息）
> │   ├── abnormal.json      # 必填字段缺失记录（写入异常队列，不入 kbd_entry）
> │   ├── img_0.png          # 原始图片（按序号）
> │   ├── img_0.desc.txt     # Vision 描述（存在 = vision done，跳过）
> │   ├── img_0.failed       # Vision 失败标记（可独立重跑）
> │   └── img_1.png ...
> └── {support_id}.lock      # flock 文件锁（防并发重复抓取）
> ```

```sql
CREATE TABLE kbd_entry (
    id                 BIGSERIAL    PRIMARY KEY,
    support_id         VARCHAR(20)  UNIQUE NOT NULL,             -- 深信服案例ID，对应 API rows.id
    support_url        TEXT,                                    -- 原始案例页面URL，供审核人溯源
    title              TEXT         NOT NULL,                   -- 案例标题，对应 API rows.name
    content_md         TEXT,                                    -- 全段结构化Markdown（含图片语义块）
    metadata           JSONB        NOT NULL DEFAULT '{}',      -- API补充字段（sangfor分类/版本/工程师ID等）
    -- 分类（双轨：AI建议 + 人工确认）
    category_id        VARCHAR(32)  REFERENCES kb_category(code), -- 人工确认分类
    ai_category_id     VARCHAR(32),                             -- AI分类建议
    ai_category_conf   FLOAT,                                   -- 置信度（<0.5提示人工核对）
    ai_category_reason TEXT,                                    -- AI分类理由（供审核参考）
    -- 检索字段（published时生成）
    embedding          vector(1536),                            -- 全文语义向量（1536维）
    tsv                tsvector,                                -- BM25全文检索
    -- 状态机
    status             VARCHAR(20)  NOT NULL DEFAULT 'draft',   -- draft→published→archived/rejected
    reviewer_id        INTEGER,                                 -- 审核人ID
    reviewed_at        TIMESTAMPTZ,                             -- 审核时间
    review_note        TEXT,                                    -- 审核备注（拒绝原因等）
    published_at       TIMESTAMPTZ,                             -- 发布时间（触发embedding生成）
    archived_at        TIMESTAMPTZ,                             -- 归档时间（手动触发）
    created_at         TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
    updated_at         TIMESTAMPTZ  NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_kbd_entry_status         ON kbd_entry(status);
CREATE INDEX idx_kbd_entry_category       ON kbd_entry(category_id) WHERE status = 'published';
CREATE INDEX idx_kbd_entry_ai_category    ON kbd_entry(ai_category_id);
CREATE INDEX idx_kbd_entry_published      ON kbd_entry(published_at DESC) WHERE status = 'published';
CREATE INDEX idx_kbd_entry_tsv            ON kbd_entry USING GIN(tsv);
CREATE INDEX idx_kbd_entry_metadata       ON kbd_entry USING GIN(metadata);
```
