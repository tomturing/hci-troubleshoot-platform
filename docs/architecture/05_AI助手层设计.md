## 原始讨论记录

### 用户提问

我先说明一下场景，然后再谈谈我的疑问，我们看看怎么处理更好。

**场景：**

1. 我们的 AI Assistant Layer，作为一个全能 AI 排障助手团队：需要共享所有的知识、技能。
2. AI Assistant Layer 在同一天，甚至同一时刻会同时处理多个工单：每个工单的上下文信息需要独立，不能混乱。
3. 现阶段是一个 openclaw 容器，多个 session 响应多个不同的工单。

**疑问：**

1. 采用什么方案让整个 AI Assistant Layer 共享知识和经验。AI 层可以是多个 claw、bot。
2. 我们现在积累了 7000 篇 web 形式案例库，一个具体的实例可以通过：https://support.sangfor.com.cn/cases/list?product_id=33&type=1&category_id=36402&isOpen=true 查看，这些案例是转换成知识更好呢，还是技能更好。（说明一下，我理解的知识就是存在本地文件如上面你提到的存在 AGENTS 的 memory，技能就是类似 https://skillsmp.com/zh 发布的 skill，加载跟 openclaw）
3. 如何设计 memory、skill 共享，session 的上下文又是隔离的机制。业界针对这块的最佳实践是什么，给出 3 种方案，及其优劣势让我来选择。
4. 现阶段是一个 openclaw 多个 session 来处理不同工单，做好隔离与共享也可以满足业务需求。但架构设计是一个 case 拉起一个 openclaw pod，各 pod 之间做好 memory 和 skill 的共享，一个 pod 只响应一个 case，pod 的生命周期与 case 的生命周期保持一致。需要确认业界的最佳实践是什么？那种方式更好。

---

### 解答

#### 问题 1：AI 层跨实例知识共享

**结论：共享层必须在 AI 层之外。**

知识通过**外部存储**共享，AI 实例本身无状态。无论是一个 openclaw 还是 N 个 pod，都从同一个知识源拉取：

```
┌──────────────────────────────────────────────────────┐
│              AI Assistant Layer (无状态)              │
│  openclaw-pod-1  openclaw-pod-2  nabobot-pod-1       │
└──────────┬───────────────┬──────────────┬────────────┘
           │               │              │
           └───────────────┴──────────────┘
                           │
           ┌───────────────▼──────────────┐
           │       共享知识层              │
           │  PostgreSQL + pgvector (RAG) │  ← 7000案例 + SOP
           │  共享 Skill 文件 (PVC/NFS)   │  ← 技能包
           │  AGENTS.md (角色+准则)       │  ← 每次启动注入
           └──────────────────────────────┘
```

---

#### 问题 2：7000 案例 → 知识还是技能？

**结论：案例 → 知识（RAG），SOP → 技能（Skill）**

| 内容类型 | 推荐方案 | 理由 |
|---|---|---|
| 7000 个故障案例 | **知识（RAG 检索）** | 案例是事实实例，数量大，不能全量注入，按相似度召回"最像当前故障的案例"，是经典 RAG 场景 |
| 10 篇 SOP 文档 | **技能（Skill 文件）** | SOP 是操作流程，内容结构化，数量少，适合作为 agent 工具加载 |
| 领域常识/命令手册 | **知识（RAG）** | 同案例 |

**概念澄清：**

- `AGENTS.md` 是**角色定义**（每次启动强制注入），不是动态知识
- **知识 ≠ 存在文件里**，知识 = 可被检索的语义内容（文件只是载体，pgvector 才是访问层）
- **技能 ≠ SkillsMP 的远程 skill**，技能 = 给 agent 的行为指令包（SKILL.md），可以是本地文件

---

#### 问题 3：共享记忆 + 技能 & Session 隔离——3 种方案

##### 方案 A：RAG 中心化 + Agent 完全无状态

```
┌─────────────────────────────────────────────────────────────────┐
│  每次对话请求时动态组装 context：                                 │
│                                                                   │
│  messages = [                                                     │
│    system: AGENTS.md (角色+准则，所有 pod 相同)                  │  ← 共享
│    system: 匹配到的 SKILL.md (SOP 技能包)                        │  ← 共享
│    system: RAG 召回的 top-k 案例片段                             │  ← 共享
│    user/assistant: 本 case 的对话历史 (来自 PostgreSQL)          │  ← 隔离
│    user: 当前消息                                                 │
│  ]                                                               │
└─────────────────────────────────────────────────────────────────┘
```

- Agent 本身不保存任何状态，session 上下文全在 PostgreSQL
- 知识共享靠 RAG 服务，Pod 重启不丢失任何东西
- **优势**：架构最简洁，完美隔离，水平扩展无限制，session-memory 污染问题永远不存在
- **劣势**：没有"跨 case 经验积累"能力——AI 不会从已解决工单中自动学习
- **适合**：当前阶段，知识库由人工维护和更新

---

##### 方案 B：两层记忆（全局记忆 + Session 记忆严格分离）

```
┌──────────────────┐          ┌──────────────────────┐
│  全局记忆层       │          │  Session 记忆层       │
│  (所有 pod 共享) │          │  (每 case 独立)       │
│                  │          │                       │
│  • 已解决案例摘要 │          │  • 本次对话历史        │
│  • 积累的诊断经验 │          │  • 本次收集的环境信息  │
│  • 常见错误模式   │          │  • 本次执行的命令记录  │
│                  │          │                       │
│  写入：case 关闭  │          │  写入：对话过程实时     │
│  读取：每次对话  │          │  销毁：case 关闭后     │
└──────────────────┘          └──────────────────────┘
         │                              │
         └──────────────┬───────────────┘
                        │
                   Conversation Service
                   组装 messages[] 时分别读取
```

- 全局记忆只在 case 成功关闭时写入（避免噪声）
- Session 记忆严格隔离，互不可见
- **优势**：AI 会越来越"有经验"，解决过的问题下次更快定位
- **劣势**：全局记忆写入质量难以保证（谁来提炼？自动提炼有噪声）；全局记忆随时间膨胀需要治理；实现复杂度高
- **适合**：有大量重复性故障、希望 AI 自我迭代的中长期场景

---

##### 方案 C：MCP Server 统一知识和技能访问

```
┌─────────────────────────────────────────────────────┐
│  AI Agent (openclaw)                                │
│                                                      │
│  Tools:                                             │
│  • search_knowledge(query) → MCP KB Server         │
│  • get_skill(name)         → MCP Skill Server      │
│  • get_case_history()      → MCP Context Server    │
│  • execute_command()       → MCP Tool Server       │
└─────────────────────────────────────────────────────┘
         │              │              │
         ▼              ▼              ▼
   [KB MCP Server] [Skill MCP]  [Context MCP]
   共享给所有 pod    共享技能包    session 隔离数据
```

- 知识和技能都以 MCP 工具形式暴露，agent 主动调用
- Session 隔离通过 Context MCP Server 的 case_id 参数实现
- **优势**：agent 可以精准控制"何时查什么"，避免无效 context 占用 token；最符合 agentic 趋势；多类型 AI 都能接同一套 MCP
- **劣势**：openclaw 的 MCP 支持需要验证；需要额外维护 MCP server；对 AI 模型的工具调用能力有要求（GLM-4.7 足够）
- **适合**：长期架构目标，尤其是多种 AI 助手共存时统一接口

---

##### 三方案横向对比

| 维度 | 方案 A（RAG 无状态）| 方案 B（两层记忆）| 方案 C（MCP）|
|---|---|---|---|
| 实现复杂度 | ⭐ 低 | ⭐⭐⭐ 高 | ⭐⭐⭐⭐ 最高 |
| Session 隔离 | ✅ 完美 | ✅ 完美 | ✅ 完美 |
| 知识共享 | ✅ RAG 召回 | ✅ RAG + 经验积累 | ✅ 按需精准检索 |
| AI 自我进化 | ❌ 靠人工维护 | ✅ 自动积累 | ⚠️ 可实现但要设计 |
| 多助手兼容 | ✅ 都从 pgvector 取 | ✅ 统一存储 | ✅ 最佳（统一工具接口）|
| 当前可落地 | ✅ 马上可做 | ⚠️ 需评估 GLM 质量 | ⚠️ 需验证 openclaw MCP |
| token 效率 | ⚠️ 固定注入 top-k | ⚠️ 同上 | ✅ 按需调用 |

#### 问题 4：单 Pod 多 Session vs 一 Pod 一 Case

##### 两种模式本质差异

```
模式 A：单 Pod 多 Session（当前）
┌─────────────────────────────────┐
│  openclaw Pod                   │
│  session: case-001 ──┐          │
│  session: case-002 ──┤─ 今天的问题 │
│  session: case-003 ──┘          │
└─────────────────────────────────┘
  内存共享、进程共享、session-memory 污染

模式 B：一 Pod 一 Case（架构设计目标）
┌────────────┐  ┌────────────┐  ┌────────────┐
│ openclaw-1 │  │ openclaw-2 │  │ openclaw-3 │
│ case-001   │  │ case-002   │  │ case-003   │
│ 完全隔离   │  │ 完全隔离   │  │ 完全隔离   │
└────────────┘  └────────────┘  └────────────┘
  知识共享靠外部 PVC + pgvector
```

##### 业界最佳实践（参考 LLM Agent 产品）

| 产品/场景 | 模式 | 原因 |
|---|---|---|
| ChatGPT 对话 | **共享推理服务，独立 thread** | 无状态推理，上下文在外部 |
| GitHub Copilot Workspace | **一任务一实例** | 任务有生命周期，需要独立沙箱 |
| Devin/SWE-agent | **一任务一容器** | 执行环境完全隔离 |
| Customer Support Bot | **共享模型，独立会话** | 短会话，无执行环境 |
| **HCI 排障（本项目场景）** | **→ 一 Case 一 Pod** | 理由如下 |

##### 本项目应该选一 Pod 一 Case，理由

1. **排障有执行动作**：AI 会建议执行命令，未来可能直连 HCI 环境，必须隔离
2. **Case 生命周期明确**：从 created 到 closed，可以精确控制 Pod 生命周期
3. **隔离是系统性需求**：session-memory、进程内共享状态，任何一个维度都可能污染，根治方案是物理隔离
4. **知识共享在外部**：采用方案 A/B/C 任一种，多 Pod 共享知识完全可行

##### 一 Pod 一 Case 的实现要点

```
Case 创建 → Scheduler 分配 Pod
                │
                ▼
         Pod 启动时注入：
         • AGENTS.md（角色）
         • Skill 文件（能力）
         • case_id（身份）
                │
                ▼
         Case 处理中：
         • 对话历史存 PostgreSQL（不靠 Pod 内文件）
         • RAG 调用外部 KB Service
         • Pod 本身完全无状态
                │
                ▼
         Case 关闭 → Pod 销毁（或归还热备池）
         • 若方案 B：关闭前提炼经验写入全局记忆
```

## 综合建议与决策

针对**当前阶段**（单 openclaw，7000 案例待建设）：

| 决策点 | 推荐 | 理由 |
|---|---|---|
| 知识共享方案 | **方案 A**（RAG 无状态） | 当前阶段最可落地，知识库建设比 memory 机制更优先 |
| 案例库处理 | **RAG 知识** | 7000 案例做向量化检索，按相似度召回 |
| SOP 文档 | **Skill 文件** | 10 篇，结构化，适合 agent 加载 |
| Pod 模式 | **一 Pod 一 Case**（逐步迁移） | 先禁用 session-memory（已做），再推进 Pod 池调度 |
| 近期行动 | **建 KB Service + 7000 案例摄入** | 这是 AI 质量最大提升点，比架构精雕细琢回报更高 |

**长期目标**：方案 C（MCP）+ 一 Pod 一 Case，这是 agentic 场景的行业方向。

## 补充讨论：openclaw 自主学习能力的利用

### 用户提问

上面我们讨论的方案都是我们作为知识和技能的生成者，AI 层作为知识的使用者，这些方案我们正在实现中。但 openclaw 本身也有知识积累能力，现成的可以直接使用。如果我让 openclaw 自己学习 SOP，自己通过 web MCP 查看 7000 篇案例然后自己整理成知识。你对此有什么想法和建议，或者能想出一个两全其美的方案。

随便再问一句：session-memory 被禁用后会有哪些影响，让 openclaw 自主学习有没有更好的解决方案。

---

### 解答

#### 一、session-memory 禁用的真实影响

先把这件事说清楚，影响分两个维度：

**对话连续性（几乎无损）**

session-memory 保存的是"跨 session 的学习摘要"，而不是对话历史。对话历史存在 PostgreSQL，每次 case 打开时 Conversation Service 会把历史消息注入 context，这部分完全不受影响。

**跨 case 经验积累（损失的是这个）**

| 功能 | 禁用前 | 禁用后 |
|---|---|---|
| 同一 case 内对话连续 | ✅ | ✅（来自 PostgreSQL，不依赖 session-memory）|
| case 关闭后重新打开记忆还在 | ✅ | ✅（同上）|
| 解决了 A case 的经验，自动用于 B case | ✅ | ❌（这个没了）|
| 学到的知识（如 VM 手册全文）跨 session 共享 | ✅（但有污染风险）| ❌ |

**禁用的根本原因是架构缺陷，不是功能本身坏**

session-memory 的问题不是"不该学"，而是"学到的东西没有边界地写入所有 session 共享的同一个日期文件"：

```
/home/node/.openclaw/workspace/memory/2026-03-05.md
　　↑ 所有 session 都往这一个文件写，所有 session 都读这一个文件
　　↑ 导致 VM 学习内容污染到排障工单 session
```

如果 memory 是 per-case 隔离的，session-memory 本身是非常有价值的功能。

---

#### 二、openclaw 自主学习的可行性分析

openclaw 内置的知识积累能力是真实存在的，让它自己学 SOP 和 7000 案例的思路方向对，但有几个现实约束需要正视：

| 约束 | 说明 |
|---|---|
| Web MCP 一次能处理的页面有限 | 7000 篇案例需要分批学习，不能一次完成 |
| session-memory 是平铺 Markdown | 适合经验摘要，不适合结构化索引（没有向量检索能力）|
| 学习质量取决于 AI 自身 | AI 提炼的质量不可审计，可能引入噪声或偏见 |
| 重复运行幂等性 | 同一篇案例学两次，memory 文件会重复追加 |

**结论**：openclaw 自主学习 **不是 pgvector RAG 的替代品**，而是 **一种互补的知识生产方式**。

---

#### 三、两全其美方案：AI 驱动知识提炼 + 外部 RAG 存储

核心思想：**让 openclaw 做知识生产者，pgvector 做知识仓库，生产 pod 做知识消费者**——三者分离，互不污染。

```
                    ┌──────────────────────────────────────┐
                    │         离线知识生产（一次性 + 增量）   │
                    │                                      │
  7000 案例网页  ──→│  Learning Pod (openclaw)              │
  10 篇 SOP      ──→│  • Web MCP：浏览、理解案例            │
  新增案例        ──→│  • 整理成结构化摘要                   │──→  KB Service
                    │  • 通过 KB Ingestion MCP 写入 pgvector│      (pgvector)
                    │                                      │
                    │  ⚑ 这个 pod 专门学习，不处理用户请求  │
                    └──────────────────────────────────────┘

                    ┌──────────────────────────────────────┐
                    │         在线排障（实时，并发）          │
                    │                                      │
  用户工单 case-001 │  Production Pod-1 (openclaw)          │
  用户工单 case-002 │  Production Pod-2 (openclaw)          │──→  KB Service
  用户工单 case-003 │  Production Pod-3 (openclaw)          │      (只读 RAG)
                    │  • session-memory 禁用（隔离保证）    │
                    │  • 对话历史来自 PostgreSQL            │
                    │  • 知识来自 KB Search MCP             │
                    │                                      │
                    └──────────────────────────────────────┘
```

**Learning Pod 工作方式**

```
1. 定时任务启动 Learning Pod（或手动触发）
2. 注入专用 system prompt：
   "你是知识整理助手，任务是系统性地阅读案例，
    提炼故障模式、解决步骤、关键命令，
    通过 KB Ingestion MCP 存入知识库，
    不需要与用户对话。"
3. 使用 Web MCP 分批访问案例：
   https://support.sangfor.com.cn/cases/list?...
4. 每处理完一批，调用 KB MCP 写入：
   kb_ingest(title, category, content, doc_type="ai_extracted")
5. 学习完成后 pod 销毁
```

**这个方案的优势**

| 维度 | 效果 |
|---|---|
| AI 自主学习 | ✅ openclaw 真正参与知识生产，不是人工整理 |
| 知识结构化 | ✅ 存入 pgvector，支持语义检索，不是平铺 Markdown |
| 隔离保证 | ✅ 生产 pod 不开 session-memory，永远无污染 |
| 知识可审计 | ✅ pgvector 里的内容可以查看、修改、删除 |
| 增量更新 | ✅ 新案例入库触发 Learning Pod 增量学习 |
| 质量控制 | ✅ 可以在写入前加人工审核环节 |

---

#### 四、session-memory 的更好解决方案

禁用 session-memory 是"断电"处理，治标不治本。如果想恢复这个能力同时保证隔离，有两个方向：

**方向 A：一 Pod 一 Case（根治方案）**

一 Pod 一 Case 架构下，每个 pod 的 memory 目录天然隔离：

```
openclaw-pod-case-001/home/node/.openclaw/workspace/memory/
openclaw-pod-case-002/home/node/.openclaw/workspace/memory/
openclaw-pod-case-003/home/node/.openclaw/workspace/memory/
```

每个 pod 只服务一个 case，memory 写什么都不会影响其他 case。case 关闭 pod 销毁，memory 也自然消失。这是**一 Pod 一 Case 架构最重要的附带收益之一**。

此方案下可以**重新启用 session-memory**，让 AI 在 case 处理过程中自由学习，且完全隔离。

**方向 B：共享只读全局记忆 + 禁用写入（折中方案）**

```
全局记忆文件（只读，人工策划或 Learning Pod 生成）
  ↓ 启动时作为 system prompt 注入（不依赖 session-memory 机制）
Production Pod
  session-memory 禁用（不写，不读 memory 文件）
  对话历史来自 PostgreSQL
```

好处：AI 有"背景知识"，但不会自动污染。代价：学到的新东西需要经过 Learning Pod 人工/自动整理后才能进入知识库。

---

#### 五、完整推荐路线

```
阶段 0（现在，已完成）
  ├─ session-memory 禁用 ✅
  └─ 工单 session 完全隔离 ✅

阶段 1（近期，Q2）
  ├─ KB Service + pgvector 上线
  ├─ Learning Pod：使用 web MCP 学习 7000 案例 → 写入 KB
  ├─ SOP 文档 → Skill 文件，启动时注入
  └─ Production Pod 通过 RAG 使用知识（方案 A 无状态）

阶段 2（中期，Q3）
  ├─ 一 Pod 一 Case 调度架构落地
  ├─ 重新启用 session-memory（此时已天然隔离）
  └─ Learning Pod 增量学习新案例 → 自动入库

阶段 3（长期）
  ├─ MCP Server 统一知识/技能接口
  ├─ Learning Pod → KB MCP → pgvector 全链路自动化
  └─ Production Pod 通过 MCP 精准按需调用知识
```

**一句话总结**：openclaw 的自主学习能力非常有价值，应该利用——但让它把学到的东西写进我们的 pgvector 知识库（通过 MCP），而不是写进共享的 session-memory 文件。Learning Pod 负责生产知识，Production Pod 负责使用知识，职责分离才是两全其美。

## 十、上下文管理设计分析与 AI 层选型对比

> 本节记录两个关联设计决策：
> 1. 为什么在 AI 层自身有上下文能力的情况下，我们仍然选择从 DB 重建历史？
> 2. 直接调用智谱 API（纯 LLM）与通过 OpenClaw（agent runtime）的详细方案对比。

---

### 10.1 AI 上下文管理：DB 重建 vs AI 自持

#### 背景疑问

我们每轮对话都从 PostgreSQL 读取历史消息，重新组装：

```python
messages = [
    {"role": "system",    "content": system_prompt},   # 含 RAG 结果
    {"role": "user",      "content": "虚拟机不能启动"},  # 第1轮
    {"role": "assistant", "content": "我来帮你分析..."},
    {"role": "user",      "content": "报错是 OOM"},      # 第2轮
    {"role": "assistant", "content": "内存不足通常是..."},
    {"role": "user",      "content": "推荐哪个命令？"},   # 本轮（刚 INSERT）
]
```

这引出一个问题：OpenClaw 或 LLM 本身是否已具备上下文管理能力，我们这样做是否多此一举？

#### 各 AI 层的上下文能力说明

| AI 层 | 上下文能力 | 本质 | 掉电后 |
|-------|-----------|------|--------|
| 纯 LLM（智谱 GLM API）| 完全无状态 | 每次 HTTP 请求独立推理，请求结束即遗忘 | 无需讨论 |
| OpenClaw Pi runtime  | Session 级内存 | `session.memory` 存储本轮 agent loop 中间推理和工具调用结果 | Pod 重启即丢 |
| OpenClaw（完整 agent loop）| 工具调用记录 | 在一次 agent loop 内多步工具调用共享 scratchpad | Pod 重启即丢 |

**关键区分**：

- OpenClaw 的 `session.memory` 是**工作暂存区**（scratchpad），用于在一次 agent loop 内记录"我刚用什么工具、得到了什么结果"
- 我们从 DB 读取历史是为了**持久化对话记录**——跨 Pod 重启、跨工单转移、供人类审计

这两者是**互补层**，而不是竞争关系：

```
DB message 表
└── 持久化层：对话的完整可审计记录，随工单生命周期存在

OpenClaw session.memory
└── 暂存层：当前 agent loop 的推理中间状态，随对话轮次归零
```

#### 为什么必须从 DB 重建，而不是依赖 AI 侧"记忆"？

| 原因 | 说明 |
|------|------|
| **可靠性** | Pod 崩溃、重启、缩容后历史不丢。DB 是唯一可靠来源。 |
| **可移植性** | 工单可以被重新分配给不同 Pod（当 Pod 池扩缩容时），历史跟着 DB 走，不依赖特定 Pod |
| **人工接管** | case 进入 `in_progress` 状态转人工时，人工客服需要读完整历史 |
| **Token 预算由我们控制** | 截断到最近 20 条是我们的业务决策，AI 无法替我们做这个权衡 |
| **可审计性** | 每条消息带有 `trace_id`，监管/排查时可重放完整过程 |
| **RAG 结果注入** | `system_prompt` 在每轮根据当前问题动态检索构建，必须从外部注入，AI 侧无感知 |

#### Stage 2 的上下文张力（需特别注意）

进入阶段二（Agentic RAG）后，ProductionClaw 会在 agent loop 内自主积累工具调用结果到 session.memory。此时存在潜在的上下文重复问题：

```
我们传入的 messages[] 包含：
  ...历史N条对话（来自 DB）...

OpenClaw session.memory 可能包含：
  ...之前轮次的工具调用摘要（如果跨轮次保留）...
```

**规避方案**：在 `productionclaw/AGENTS.md` 中明确 `session.memory` 仅用于**单轮 agent loop 内**的工具调用暂存，每轮对话开始时不携带上一轮的 session.memory 内容。这与当前 AGENTS.md 的 `session-memory` 配置一致。

### 10.2 三阶段上下文管理全景

三个阶段对"进入 AI 层之前如何组装上下文"的处理方式**不同**，需分开说明。

#### 概览表

| 维度 | Stage 1（外挂 RAG） | Stage 2（Agentic RAG）| Stage 3（混合架构）|
|------|--------------------|-----------------------|--------------------|
| **谁组装 messages[]** | conversation-service | conversation-service（简化）| conversation-service（最简）|
| **System Prompt 内容** | Tier1 身份 + Tier2 SOP + Tier3 KB chunks + Tier4 工单 | AGENTS.md 身份 + 工单上下文 | AGENTS.md 身份 + 工单上下文 |
| **SOP 知识来源** | HTTP 查 KB Service → 注入文本 | OpenClaw 自主工具调用 | Pod 本地 ConfigMap（零延迟）|
| **案例知识来源** | HTTP 查 KB Service → 注入文本 | OpenClaw 自主工具调用 | OpenClaw 自主工具调用 |
| **DB 历史消息** | 最近 20 条，全量传入 | 最近 20 条，全量传入 | 可适当缩短（session.memory 补充）|
| **session.memory** | 禁用（多 Session 共享 Pod，有污染风险）| 禁用（同上，或重新启用视 Pod 架构）| **启用，跨轮持久**（一 Pod 一 Case，天然隔离）|
| **LLM 对照组处理方式** | 与 OpenClaw 完全相同 | **分叉**：仍需外挂 RAG，无自主工具调用 | **分叉加剧**：LLM 无本地 skills，无自主多跳 |

#### Stage 1 — conversation-service 是 RAG 编排者

```
每轮 POST /conversations/{id}/message 时：

conversation-service 并发：
    ├── POST /api/kb/sop/match(query)      → SOP节点文本
    └── POST /api/kb/search(query, top_k)  → KB chunks

组装 messages[] = [
  { role:"system", content: "[Tier1 HCI 排障助手角色]
                              [Tier2 SOP 精确命中内容]
                              [Tier3 KB语义检索chunks]
                              [Tier4 工单 ID 上下文]" },
  { role:"user",      content: "历史消息 1" },   ← 来自 DB（最近20条）
  { role:"assistant", content: "历史消息 2" },
  ...
  { role:"user",      content: "本轮用户问题" }, ← 刚 INSERT 的
]

→ POST /v1/chat/completions (stream=true)
  → OpenClaw 或 GLM API（两者路径完全一致）
```

**关键约束**：conversation-service 是"盲目的"——它不理解对话语义，每轮都固定检索一次，不会根据上下文判断"此轮需不需要检索"。

#### Stage 2 — conversation-service 降级为消息路由器

```
每轮 POST /conversations/{id}/message 时：

conversation-service 只做：
  ① INSERT user message（独立 session）
  ② SELECT DB history（最近20条）
  ③ 组装简化 messages[]
  ④ 路由到 ProductionClaw Pod

messages[] = [
  { role:"system", content: "[AGENTS.md 排障规范]
                              [工单 ID + 当前 Case 上下文]" },
  { role:"user",      content: "历史消息 1" },   ← DB
  { role:"assistant", content: "历史消息 2" },
  ...
  { role:"user",      content: "本轮用户问题" },
]

→ POST /v1/chat/completions (stream=true)
  → ProductionClaw（Pi agent runtime）
      ↓ 进入 AI 层内部，自主 tool-calling
      自主 POST /api/kb/search(...)
      自主 POST /api/kb/sop/match(...)
```

**LLM 对照组在 Stage 2 分叉**：GLM 直接调用没有工具，只能继续用 Stage 1 的外挂方式，无法参与公平的 Stage 2 对比。

#### Stage 3 — session.memory 成为工单级持久工作记忆

一 Pod 一 Case 架构下，session.memory 可以安全启用，并且**跨轮持续积累**：

```
Case 开始（Pod 启动）
    → BOOTSTRAP.md 执行，KB 预加载，初始化 session-memory 档案

第 1 轮对话：
  messages[] = [AGENTS.md] + [DB history 0条] + [本轮用户消息]
     → OpenClaw 推理
     → 自主查 KB（本地 SOP skills + 工具调用案例库）
     → 回复
  对话结束后，OpenClaw 更新 session.memory：
  "当前假设：VM 内存不足 [高置信度]；已排除：网络问题"

第 2 轮对话：
  messages[] = [AGENTS.md] + [DB history 2条] + [本轮用户消息]
     → OpenClaw 看到 DB 历史 + session.memory 里的推理摘要
     → 已有假设框架，无需重复检索基础 SOP
     → 回复

...（session.memory 随每轮对话持续丰富）...

Case 关闭（Pod 销毁）
    → session.memory 随 emptyDir 消失（But 精华已通过 LearningClaw 提炼进 KB）
```

**session.memory 与 DB messages[] 的互补关系**：

| 数据层 | 存储什么 | 谁写 | 谁读 | 生命周期 |
|--------|---------|------|------|---------|
| DB messages[] | 逐字对话记录（verbatim） | conversation-service | conversation-service 每轮 SELECT | 永久，随工单存档 |
| session.memory | AI 的推理笔记（假设、已排除方向、命令结果摘要） | OpenClaw 自主更新 | OpenClaw（context 的一部分） | 随 Pod 生命周期，Case 关闭后消失 |

Stage 3 可以适度**缩短 DB history 传入窗口**（如从 20 条缩至 10 条），因为 session.memory 已经把核心推理进展压缩记录。这有助于节省 token。

### 10.3 进入 AI 层后：prompt 可控性与内部处理差异

> **结论先行**：prompt 进入 AI 层后，对我们来说存在一个"可控边界"——边界之外的推理我们看不到，但边界本身的宽窄因路径不同而差异显著。

#### LLM 直接调用（智谱 GLM API）— 单步推理黑盒

```
我们发出的 messages[]
      ↓
┌──────────────────────────────────────────────────────┐
│             GLM Transformer 前向推理                  │
│                                                      │
│  ① 解析全部 messages[]（约 4K~16K tokens）           │
│  ② 一次前向传播：token-by-token 贪婪/采样生成         │
│  ③ 直到 <EOS> 或 max_tokens                          │
│                                                      │
│  对我们完全不透明：                                   │
│    · 为什么选这个词？                                 │
│    · 它"想查"某资料但我们没传进来？没机会知道         │
│    · 中间推理链（CoT）不可见（除非显式要求输出）       │
└──────────────────────────────────────────────────────┘
      ↓
  token stream → 前端直接展示
```

**我们能控制的**：messages[] 的内容和顺序、temperature/top_p、model 版本、max_tokens。  
**我们不能控制的**：推理路径、注意力分布、为什么某个 RAG chunk 被忽略了。

#### OpenClaw Pi Agent Runtime — 多步 agent loop 半透明

```
我们发出的 messages[]
      ↓
┌──────────────────────────────────────────────────────────────────────┐
│                     Pi Agent Runtime（openclaw）                      │
│                                                                      │
│  ① 解析配置层（每次 session 启动时读入，一次性）                       │
│      SOUL.md   → 性格约束："简洁、务实、不夸大"                        │
│      IDENTITY.md → 身份："HCI AI 排障助手"                            │
│      AGENTS.md → 行为规范：排障框架、session.memory 更新、命令格式     │
│      TOOLS.md  → 工具声明：KB Search、SOP Match、...                  │
│                                                                      │
│  ② 进入 agent loop（每轮对话一次循环）                                 │
│                                                                      │
│      Loop 迭代 1：                                                    │
│        LLM 接收 messages[] + 工具定义                                 │
│        LLM 推理："我需要先查一下 KB"                                   │
│        LLM 输出：tool_call { name:"kb_search", query:"VM OOM" }      │
│              ↓                                                        │
│        Pi runtime 拦截 tool_call，执行实际 HTTP 请求                   │
│              ↓                                                        │
│        HTTP → KB Service → 返回 chunks                               │
│              ↓                                                        │
│      Loop 迭代 2：                                                    │
│        messages[] 追加 tool_result（chunks 内容）                     │
│        LLM 再次推理："我还需要 SOP 吗？不需要了，直接回答。"            │
│        LLM 输出：最终文本 token stream                                │
│              ↓                                                        │
│      loop 结束（LLM 不再调用工具）                                     │
│                                                                      │
│  可观测的部分（openclaw 日志）：                                       │
│    · 每次工具调用的参数和结果                                          │
│    · loop 迭代次数                                                    │
│    · 最终触发 token stream 输出的时机                                  │
│                                                                      │
│  仍不可观测：                                                          │
│    · LLM 为什么决定调这个工具而不是那个                                │
│    · LLM 内部的推理步骤（除非 AGENTS.md 里要求 CoT 输出）              │
└──────────────────────────────────────────────────────────────────────┘
      ↓
  token stream → conversation-service → SSE → 前端
```

**我们多出的可控维度**（相比 LLM 直接调用）：

| 可控点 | 如何控制 | 文件位置 |
|--------|--------|---------|
| 角色人格 | SOUL.md 中定义价值观和行为约束 | `deploy/claw-configs/productionclaw/SOUL.md` |
| 排障行为框架 | AGENTS.md 中的对话规范、session.memory 更新规范、命令格式 | `deploy/claw-configs/productionclaw/AGENTS.md` |
| 哪些工具可调 | TOOLS.md 声明工具及调用条件 | `deploy/claw-configs/productionclaw/TOOLS.md` |
| 启动时预加载 | BOOTSTRAP.md 定义初始化步骤 | `deploy/claw-configs/productionclaw/BOOTSTRAP.md` |
| 工具调用日志 | openclaw 自身日志，可接入 Loki | K8s logs |

**核心结论**：LLM 的黑盒边界就是 prompt 入口；OpenClaw 的黑盒边界被 AGENTS.md/SOUL.md/TOOLS.md 切分成了若干个可配置的"隔间"，我们通过配置文件塑造行为，推理本身仍不可见，但行为约束是白盒可编辑的。

### 10.4 排障问答效果调优全景

> 当整体问答效果不佳时，按以下层次从外到内排查和调优。  
> 每个调优点都注明了生效阶段（Stage 1 / 2 / 3）和可操作的具体入口。

#### 层次一：知识库质量（KB Service + pgvector）

这是最高杠杆的调优点——AI 回答差，80% 是因为它"没有拿到正确的资料"。

| 调优点 | 问题表现 | 调优方法 | 生效阶段 |
|--------|---------|--------|---------|
| **Chunk 大小** | 检索到的片段太碎（缺上下文）或太长（噪声多）| 调整 `KB_CHUNK_SIZE`（当前 512 tokens），对比 256/1024 | S1-S3 |
| **Chunk 重叠** | 跨块边界的关键信息被截断 | 调整 `KB_CHUNK_OVERLAP`（当前 50），增大至 100-150 | S1-S3 |
| **Embedding 模型** | 语义检索召回质量差 | 从 `bge-small-zh` 升级到 `bge-large-zh` 或 `text-embedding-3-small` | S1-S3 |
| **top_k 数量** | 上下文不够（召回太少）/ token 浪费（召回太多）| 调整 `KB_SEARCH_TOP_N`（当前 3），试 5-8 | S1-S3 |
| **相似度阈值** | 检索到不相关内容 | pgvector 查询加 `WHERE 1 - (embedding <=> $1) > 0.75` 过滤 | S1-S3 |
| **SOP 关键字覆盖** | SOP 精确匹配未命中 | 补充 `kb_sop_node.keywords` 字段的关键词 | S1-S3 |
| **文档摄入质量** | 原始 MD 文档结构混乱 | 重新处理：去除无效格式，拆分成更小语义单元 | S1-S3 |
| **混合检索权重** | 只有向量检索，缺乏关键词精确匹配 | 启用 `kb_chunk.tsv` 的 BM25 全文检索，做 RRF 融合 | S1-S3 |

**快速验证命令**：
```sql
-- 查看某次查询的检索结果质量
SELECT kc.content, 1 - (kc.embedding <=> '[0.1, 0.2, ...]') AS score
FROM kb_chunk kc
ORDER BY score DESC LIMIT 10;
```

#### 层次二：System Prompt 工程（Stage 1 核心调优点）

| 调优点 | 问题表现 | 调优方法 | 文件位置 |
|--------|---------|--------|---------|
| **Tier1 基础身份** | AI 回复不符合 HCI 领域专业要求 | 修改 `_SYSTEM_BASE` 常量，增加限制条件 | `conversation_service.py` |
| **"无命中"兜底指令** | KB 无结果时 AI 乱编答案 | 当前兜底："不得自行推测"，可再加"列出可能需要工程师补充的信息" | `conversation_service.py` |
| **KB 上下文截断上限** | 优质 chunk 被截断 | 调整 `KB_CONTEXT_MAX_CHARS`（当前配置值），适当增大 | `settings.py` |
| **Tier 优先级** | SOP 和 KB chunks 让 AI 混淆 | 在 sections 之间加更清晰的角色说明（"以下是权威 SOP，优先参考"）| `conversation_service.py` |

#### 层次三：AGENTS.md / SOUL.md / TOOLS.md（Stage 2+ 核心调优点）

| 调优点 | 问题表现 | 调优方法 | 文件位置 |
|--------|---------|--------|---------|
| **排障逻辑框架** | AI 回复发散，没有假设-验证结构 | 在 AGENTS.md "排障逻辑框架" 章节补充更强的约束："未经验证不得给出结论" | `claw-configs/productionclaw/AGENTS.md` |
| **命令建议格式** | 命令建议缺少风险提示 | AGENTS.md 中"高风险命令必须添加警告"规则已存在，可强化细节 | 同上 |
| **session.memory 结构** | AI 每轮重复提问已知信息 | 强化 session.memory 更新规范，要求每轮标记"已确认事项" | 同上 |
| **SOUL.md 简洁约束** | 回复过长，主次不清 | SOUL.md 加 "每次回复不超过 300 字，重点先说" | `claw-configs/productionclaw/SOUL.md` |
| **工具调用时机** | AI 每轮都检索，浪费 token 且慢 | AGENTS.md 加 "仅在收到新症状时触发检索，同一轮不重复调用" | 同上 |
| **BOOTSTRAP 预加载** | 第一轮回复缺乏针对性 | BOOTSTRAP.md 增加：KB 无命中时要求 AI 主动向用户索要具体报错截图 | `claw-configs/productionclaw/BOOTSTRAP.md` |

#### 层次四：对话历史窗口管理

| 调优点 | 问题表现 | 调优方法 | 代码位置 |
|--------|---------|--------|---------|
| **窗口大小（20条）** | 长工单前期上下文丢失 | 增大到 30-40 条，或改为按 token 数截断 | `conversation_service.py:selected_messages = all_messages[-20:]` |
| **消息选择策略** | 早期"故障描述"轮丢失，AI 忘记根本问题 | 保留第 1 条 user 消息（故障描述），动态截断中间历史 | 同上 |
| **Stage 3 token 优化** | session.memory 已记录推理摘要，DB 历史冗余 | 窗口缩至 10 条，节省 context window | 同上 |

#### 层次五：模型参数调整

| 调优点 | 问题表现 | 调优方法 | 配置位置 |
|--------|---------|--------|---------|
| **模型版本** | GLM-4-Flash 回答较浅 | 切换到 GLM-4-Plus（token 成本 ~3×，质量提升显著）| `values.yaml: config.glmModelId` |
| **Temperature** | 回答发散或重复 | 降低 temperature（当前未显式传，使用模型默认 ~0.7，可改为 0.3）| `ai_client.py: payload` |
| **max_tokens** | 回复被截断 | 增加 `max_tokens` 限制（当前未传，依赖模型默认）| 同上 |

#### 层次六：数据采集与评估闭环

以上调优的前提是有客观指标。当前已有基础设施，需要启用：

| 指标 | 采集方式 | 用途 |
|------|--------|------|
| 用户评分（1-5）| `assistant_evaluation` 表的 `user_rating` 字段 | 整体满意度趋势 |
| 工单解决率 | `case.status = 'closed'` 且用户主动关闭 vs 超时关闭 | 有效解决 vs 放弃 |
| KB 命中率 | conversation-service 日志 `kb_chunks_used` | RAG 有无生效 |
| SOP 命中率 | 日志 `kb_sop_hit` / `kb_no_hit` | SOP 关键词覆盖度 |
| TTFT（首 token 延迟）| `AI_TTFT_SECONDS` Prometheus histogram | AI 响应速度 |
| 轮次数 | `conversation.message_count / 2` | 越少说明越精准 |
| assistant_type 分组 | `assistant_evaluation.assistant_type` | A/B 对比 OpenClaw vs GLM |

**推荐调优工作流**：
```
1. 建立基线：开启用户评分，采集 2 周数据
2. 定位短板：KB 命中率低 → 先调层次一；
             AI 格式差  → 调层次三；
             回答太浅   → 调层次五
3. 逐层调整：每次只改一个变量，观察评分变化 1 周
4. A/B 验证：切换 assistant_type 让一半工单用新配置
5. 沉淀最佳实践：优质配置提炼回 AGENTS.md/SOUL.md
```

### 10.5 VS Code Copilot 工作流分析与对比借鉴

> 本节以 VS Code Copilot Agent Mode（当前使用 Claude Sonnet 4.6 模型）为参照，分析其完整工作流，  
> 与 HCI 排障助手的设计做对比，提炼可借鉴的优点和关键差异。

#### VS Code Copilot Agent Mode 完整流程

```
用户在 Chat 面板输入问题
        │
        ▼
① 上下文自动收集（VS Code 侧，进入 prompt 之前）
  ├── 编辑器状态：当前文件内容 + 光标位置 + 选中代码
  ├── 工作区：文件树结构、最近打开的标签页内容
  ├── 诊断信息：LSP 报告的所有 Error / Warning（自动注入）
  ├── 终端输出：最近一次命令的 stdout/stderr（需用户允许）
  ├── 当前对话历史：本 session 全部轮次（滑动窗口 + 自动摘要）
  └── 附件：用户手动 #file: #selection #terminal 引用的内容
        │
        ▼
② 组装 messages[]（VS Code 内部，对用户不可见）
  messages = [
    { role:"system", content: "<VS Code 内部 system prompt，定义助手角色和工具使用规范>" },
    { role:"user",   content: "<历史轮次 1>" },
    { role:"assistant", content: "<历史轮次 1 回复>" },
    ...
    { role:"user",   content: "<当前问题> + <当前文件/诊断等自动注入上下文>" },
  ]
  加上 tools 声明（所有可用工具的 OpenAI function 格式定义）
        │
        ▼
③ POST https://api.anthropic.com/v1/messages   ← 调用 Claude Sonnet 4.6
  payload = { model, messages, tools, stream:true, max_tokens }
        │
        ▼
④ Claude 返回（可能是工具调用，也可能是最终文本）
  情况 A：返回 tool_use
    → VS Code 在本地执行工具（读文件/运行命令/搜索）
    → 工具结果追加到 messages[] 作为 tool_result
    → 回到步骤③，继续对话（多轮 tool-calling 循环）
  情况 B：返回 text stream
    → VS Code 渲染 token 到 Chat 面板
    → 同时解析 code blocks 提取可执行的文件编辑操作
        │
        ▼
⑤ 工具循环结束，用户看到最终回复
```

#### 工具（Tools）详细说明

Copilot Agent 模式中"工具"是声明给 Claude 的本地操作 API，Claude 决定何时调用：

| 工具 | 功能 | 类比 HCI 系统 |
|------|------|-------------|
| `read_file` | 读取工作区中任意文件内容 | 无直接对应（HCI 没有文件系统访问）|
| `run_in_terminal` | 在 VS Code 终端执行 shell 命令 | 无（HCI 只建议命令，不执行）|
| `create_file` / `replace_string_in_file` | 修改代码文件 | 无 |
| `grep_search` | 正则搜索文件内容 | → KB 关键字搜索 |
| `semantic_search` | 语义搜索工作区代码 | → KB 向量检索 `POST /api/kb/search` |
| `get_errors` | 获取 LSP 诊断信息 | → 无对应（HCI 可类比为"获取系统告警"）|
| `list_dir` | 列出目录结构 | 无 |

**工具调用可观测性**：Copilot 在 UI 中会显示 "Reading file xxx" / "Running command..." 等中间状态，用户全程可见。

#### 对话历史管理

| 维度 | VS Code Copilot | HCI 排障助手 |
|------|----------------|------------|
| 历史存储 | VS Code 内存（session 级）| PostgreSQL `message` 表（持久化）|
| 历史获取 | 内存直接读取 | 每轮 `SELECT` 独立 session |
| 长对话处理 | 滑动窗口 + **自动摘要注入**（不丢早期信息）| 简单截断最近 20 条（早期信息可能丢失）|
| 跨 session 记忆 | ❌ 每次打开新 session 历史清空 | ✅ DB 持久化，随工单生命周期存在 |
| 历史可审计 | ❌ 关闭 IDE 即消失 | ✅ 永久存档，带 trace_id |

#### 上下文管理对比

| 维度 | VS Code Copilot | HCI Stage 1 | HCI Stage 2/3 |
|------|----------------|-------------|---------------|
| 上下文自动收集 | ✅ 工作区文件、诊断、终端 | ⚠️ 仅 KB 注入（无环境自动采集）| ⚠️ 同 S1 |
| 知识检索 | ✅ Agent 自主 semantic_search | ❌ conversation-service 外挂注入 | ✅ Agent 自主工具调用 |
| 多跳推理 | ✅ read_file → 发现引用 → 再 read | ❌ 单次注入 | ✅ 多轮工具调用 |
| 中间推理可见性 | ✅ 展示 "Reading xxx..." | ❌ 用户只看到最终流 | ⚠️ 日志可见，用户侧无展示 |
| 诊断/告警自动注入 | ✅ LSP 错误自动进入 prompt | ❌ 未实现 | ❌ 未实现 |
| Token 效率 | ✅ 按需工具调用，不预注入 | ⚠️ 固定 top_k 注入，有冗余 | ✅ 按需 |

#### 可借鉴的核心优点

**① 诊断/告警自动注入**（高价值，已有实现路径）

Copilot 会把 LSP 错误自动注入上下文，HCI 通过**客户端主动采集**实现同等效果。

HCI 实际方案（见 [17_客户端设计.md §2](17_客户端设计.md)）：
- 工程师通过 SSH 连接客户后台环境时，**自动并发调用 3 个接口**（非 Zabbix/SNMP）：
  - a. 客户环境基本信息（集群版本、节点状态、存储池）
  - b. 客户环境告警信息（活跃告警列表、级别、受影响组件）  
  - c. 客户环境任务信息（最近任务状态、失败原因）
- 采集结果自动注入 system_prompt **Tier4** 的环境上下文区块
- AI 在工程师第一句话发出之前就已看到告警和任务状态

> **数据支撑**：任务失败类和告警类问题占所有 case 的 **80% 以上**，这 3 个接口覆盖了绝大多数首轮诊断所需的背景信息。

此外，HCI 还提供 **acli 被动式命令体系**：AI 建议工程师执行 acli 命令，工程师授权后执行，结果回填至对话上下文（类似 Copilot 的 `run_in_terminal` + 用户确认机制）。详见 [17_客户端设计.md §3](17_客户端设计.md)。

**② 长对话摘要替代截断**（中等价值）

当前 `all_messages[-20:]` 简单截断会丢失第 1 轮"故障描述"中的关键信息。  
Copilot 的做法是对早期轮次做 LLM 摘要后注入：
```python
# 伪代码：当消息超过 20 条时
summary = await llm.summarize(messages[:len-10])  # 早期历史压缩
recent = messages[-10:]                             # 保留最近 10 条
messages_to_send = [summary_msg] + recent           # 摘要 + 近期
```
这应进入 Stage 3 的工作列表。

**③ 工具调用中间状态展示**（用户体验）

Copilot 会在 UI 显示 "Checking knowledge base..." 等状态，让用户知道 AI 正在工作。  
HCI 可在 SSE 流中插入 `event: status` 类型事件，前端渲染进度提示。

**④ 工具调用后继续决策**（Stage 2 核心能力）

Copilot read_file 结果出来后，Claude 可以判断"还需要再读另一个文件"。  
HCI Stage 2 的 ProductionClaw 工具调用也有这个能力，这是与 GLM 直接调用最本质的区别。

#### HCI 独有的优势（Copilot 没有的）

| 优势 | 说明 |
|------|------|
| **KB 领域知识** | 7000 篇 HCI 产线案例 + SOP，Copilot 没有这些 |
| **Case 生命周期管理** | 工单状态机、关闭流程、评分、审计 |
| **多工程师隔离** | 每个 case 独立 Pod + DB session，完全隔离 |
| **持久化历史** | 工单可跨天、跨设备、人工接管，Copilot session 关闭即失 |
| **可审计 trace_id** | 每条消息追踪到具体请求链路 |
| **LearningClaw 自进化** | AI 从历史工单中自主提炼经验，Copilot 没有这套闭环 |

### 10.6 AI 层入口 Prompt 审计镜像设计

> **关于评分与质量评价系统的完整设计**，见 [16_评分机制与评价系统.md](16_评分机制与评价系统.md)。本节聚焦 prompt 审计镜像的技术实现，评分维度、综合质量分、数据库扩展、前端评分卡等内容已移至专项文档。

#### 镜像的是什么？

"在进入 AI 层的总入口镜像一份数据"，镜像的就是**完整的 `messages[]` payload**——即 `POST /v1/chat/completions` 请求体中的全部内容：

```json
{
  "model": "productionclaw",
  "stream": true,
  "messages": [
    { "role": "system",    "content": "【Tier1 HCI排障助手角色定义】\n\n【Tier2 SOP: VM宕机处理...】\n\n【Tier3 KB chunks: [1]来源:案例123...】\n\n【Tier4 工单ID: Q20260311001】" },
    { "role": "user",      "content": "虚拟机不能启动" },
    { "role": "assistant", "content": "我来帮你分析..." },
    { "role": "user",      "content": "报错是 OOM" },     ← 刚 INSERT 的
  ]
}
```

这个 payload 包含了：**4-Tier system prompt（含真实 KB 召回结果）+ 完整历史（截断后）+ 本轮用户消息**。这是模型实际"看到"的全部信息，是调优分析最直接的依据。

**镜像采集点**：`conversation_service.py` 中，`_build_system_prompt()` 和 `SELECT history` 都完成之后、`ai_client.chat_completion_stream()` 调用之前。

```python
# 现有代码流程（伪代码）
await save_user_message(...)                     # ① DB INSERT user
system_prompt = await _build_system_prompt(...)  # ② 4-Tier RAG 构建
all_messages = await get_messages(...)           # ③ DB SELECT history
history_messages = build_history(...)            # ④ 组装

# ↓↓↓ 镜像采集点：此处 history_messages 就是完整 prompt ↓↓↓
await _audit_prompt(history_messages, ...)       # ⑤ 新增：镜像

async for chunk in ai_client.chat_completion_stream(history_messages): # ⑥ 发给 AI
```

#### 为什么要镜像

| 用途 | 说明 |
|------|------|
| **调优诊断** | AI 回答差时，精确看到它拿到了哪些 KB chunks、SOP 有没有命中 |
| **黄金数据集** | `{完整 prompt, AI 回复, 用户评分}` 三元组，可用于 fine-tuning 或 few-shot 优化 |
| **Prompt 工程验证** | 验证每次修改 system_prompt 后实际注入效果 |
| **合规审计** | 记录 AI 说过什么话、基于什么上下文，责任追溯 |
| **A/B 分析** | 对比 OpenClaw 和 GLM 收到的是否真的相同 prompt |

#### 三种实现方案

**方案 A：Structlog → Loki（轻量，已有基础设施）**

```python
logger.info(
    event="prompt_audit",
    case_id=case_id,
    conversation_id=str(conversation_id),
    assistant_type=resolved_assistant_type,
    model=model,
    # 元数据（每次全量记录，小字段）
    message_count=len(history_messages),
    system_prompt_chars=len(history_messages[0]["content"]),
    has_sop=has_sop,
    kb_chunks_count=kb_chunks_count,
    # 完整 payload（按策略采样，大字段）
    full_messages=history_messages if should_audit() else None,
)
```

**优点**：零新增基础设施，Grafana 可查  
**缺点**：Loki 不适合存大 JSON（超出行大小可能截断）；全量 payload 日志量大

---

**方案 B：PostgreSQL `prompt_audit` 表（结构化，可关联评分）**

```sql
CREATE TABLE prompt_audit (
    audit_id     UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    case_id      VARCHAR(20) REFERENCES "case"(case_id),
    conversation_id UUID REFERENCES conversation(conversation_id),
    assistant_type  VARCHAR(50),
    model        VARCHAR(100),
    -- 元数据（始终记录）
    message_count   INT,
    has_sop      BOOLEAN,
    kb_chunks_count INT,
    system_prompt_chars INT,
    -- 完整 payload（按策略记录）
    messages     JSONB,   -- 完整 messages[] 数组
    -- 关联回路
    user_rating  SMALLINT,  -- 后续 PATCH 更新（用户评分后回填）
    created_at   TIMESTAMPTZ DEFAULT now(),
    trace_id     VARCHAR(64)
);
CREATE INDEX idx_prompt_audit_case ON prompt_audit(case_id);
CREATE INDEX idx_prompt_audit_rating ON prompt_audit(user_rating) WHERE user_rating IS NOT NULL;
```

**优点**：完全结构化，可 JOIN `assistant_evaluation` 表，精确关联"这条 prompt → 这个评分"；支持 `WHERE user_rating < 3 AND has_sop = false` 类型分析  
**缺点**：JSONB 大字段占存储；需要新建表和迁移

---

**方案 C：元数据入 PostgreSQL + 完整 payload 入对象存储**

```
prompt_audit 表（轻量元数据）
  audit_id, case_id, conversation_id, has_sop, kb_count, user_rating ...
  payload_ref: "minio://hci-audit/2026/03/11/{audit_id}.json"  ← 文件指针

  2026/03/11/{audit_id}.json  ← 完整 messages[] JSON 文件（压缩后 ~10KB）
```

**优点**：存储分层，DB 轻量，大文件廉价存储；规模化后最优  
**缺点**：需要 MinIO（项目已有）但多一个依赖；查询时需两次 IO

#### 推荐方案（当前规模）

| 阶段 | 方案 | 理由 |
|------|------|------|
| **立即可做** | 方案 A（Loki，元数据）| 零改动，先看元数据分布（KB 命中率、system prompt 长度分布）|
| **近期（1-2周）** | 方案 B（PostgreSQL）| 加 `prompt_audit` 表，按策略采样全量 payload |
| **中长期** | 方案 C | 当每日工单数 > 100，payload 存 MinIO |

#### 采集策略（避免无脑全量）

> **核心原则**：`should_capture_full_payload()` 在**消息发送时**（stream 触发前）决定是否采集，此时用户还没有评分。因此采集判断不能依赖 `user_rating`，改用 `close_reason` 等系统可自动推断的信号。

```python
def should_capture_full_payload(
    close_reason: str | None,   # "user_command" | "timeout" | "abandon" | None（工单尚未关闭）
    kb_hit: bool,
    case_id: str,
) -> bool:
    """决定是否采集完整 messages[] payload（在 stream 调用前判断）"""
    # 规则 1：放弃型关闭，高风险 case，必须采集
    if close_reason == "abandon":
        return True
    # 规则 2：KB 无命中，边界案例，价值高
    if not kb_hit:
        return True
    # 规则 3：随机 10% 采样（基线数据）
    if random.random() < 0.10:
        return True
    return False
```

**用户评分补充触发**（异步，在用户评分后执行）：
```python
async def on_user_rating(audit_id: UUID, user_rating: int):
    """用户评分后，如评分 ≤ 3 且之前未采集完整 payload，则补充采集"""
    if user_rating <= 3:
        audit = await get_prompt_audit(audit_id)
        if audit.messages is None:        # 之前未采集完整 payload
            await backfill_payload(audit_id)  # 从对话记录重建并写入
    await update_prompt_audit_rating(audit_id, user_rating)
```

### 10.7 直接调用智谱 API vs 调用 OpenClaw 详细方案对比

#### 表面相似，内部不同

从我们的 `conversation-service` 视角，两者外观完全一致：

```python
# ai_client.py — 两种 AI 都走同一个 OpenClawAssistant 类
POST /v1/chat/completions
Body: {"model": ..., "messages": [...], "stream": true, "user": "case-xxx"}

← SSE token stream
```

区别在于**那个 URL 背后发生了什么**。

#### 详细对比矩阵

| 对比维度 | 直接调用智谱 GLM API | 调用 OpenClaw（Pi agent runtime）|
|---------|---------------------|--------------------------------|
| **本质** | 纯 LLM：单次推理 → 返回文本 | Agent runtime：可能多轮 tool-calling → 最终返回文本 |
| **上下文管理** | 无状态；messages[] 是你给的全部 | Session 级暂存（scratchpad）+ 你传的 messages[] |
| **当 messages[] 到达时** | 模型做一次前向推理，生成 token | Pi runtime 解析，进入 agent loop，可能先调工具再推理 |
| **工具调用** | 支持 function definitions，但需外部编排（`conversation-service` 决定何时调什么）| 模型自主决定调哪个工具、何时停止、结果如何影响下一步 |
| **KB 集成（阶段一）** | conversation-service 外挂注入（我们预先查询 KB，拼入 system_prompt）| 同左（Stage 1 下 OpenClaw 也是被动接收，TOOLS.md 暂时形同虚设）|
| **KB 集成（阶段二）** | 仍需外挂（GLM 本身没有访问 KB 的主动能力）| ProductionClaw 通过 TOOLS.md 定义的工具**自主**查询 KB |
| **多跳推理** | ❌ 每次 API 调用只做一次推理 | ✅ agent loop 内可迭代：检索→分析→再检索→缩小范围 |
| **确定性** | 高（相同 prompt + 低 temperature ≈ 相同输出）| 低（agent 每次可能选择不同工具调用路径）|
| **可调试性** | 高（完全看到发进去什么，出来什么）| 中（还需查看 agent 内部工具调用 trace）|
| **TTFT（首 token 延迟）** | ~1-3s（远程 API + 网络开销）| ~1-5s（本地 Pod，取决于 GPU 和是否触发工具调用）|
| **响应超时风险** | 低（API 通常在 60s 内响应）| 中（agent loop 复杂时可能较长，当前 `AI_CLIENT_READ_TIMEOUT_SEC=120`）|
| **行为配置** | system_prompt 是唯一约束 | `AGENTS.md`（角色+准则）+ `SOUL.md`（性格）+ `TOOLS.md`（工具定义）多层约束 |
| **基础设施** | API Key，零 K8s 资源 | Pod 部署（Deployment/StatefulSet）+ 热备 Pod 池 |
| **计费方式** | 按 token（GLM-4 ~¥0.1/1K tokens）| 按算力（GPU Server 月租）|
| **`user` 字段作用** | 用于速率限制/审计，无业务路由 | 映射为 OpenClaw session key，用于 Agent-to-Agent 路由 |
| **当前在项目中的角色** | 评估对照组（AssistantRegistry 添加一条配置即可启用）| 主要生产路径（`productionclaw` assistant_type）|

#### 两者在代码层面的接入差异

```python
# 接入智谱（values.yaml）
llm-zhipu:
  base_url: "https://open.bigmodel.cn/api/paas/v4"
  model: "glm-4-flash"          # 或 glm-4-plus
  gateway_token: "${ZHIPU_API_KEY}"
  enabled: true
  warm_pool_size: 0              # 无状态，无需 K8s Pod

# 接入 OpenClaw（values.yaml）
productionclaw:
  base_url: "http://openclaw-service:8080"
  model: "productionclaw"
  gateway_token: "${OPENCLAW_GATEWAY_TOKEN}"
  enabled: true
  warm_pool_size: 3              # 有状态 Pod，需热备
```

两者在 `ai_client.py` 中使用完全相同的 `OpenClawAssistant` 类——因为 `/v1/chat/completions` 接口规范相同。

#### 选型结论

| 使用场景 | 推荐 |
|---------|------|
| 快速启动、成本优先、确定性排障 | 直接调用智谱 GLM API |
| 阶段一 RAG 注入排障（当前阶段）| 两者等效，OpenClaw 略复杂但可积累 agent 能力 |
| 阶段二 Agentic RAG（KB 自主检索）| **必须选 OpenClaw**，GLM API 无法实现自主工具调用闭环 |
| A/B 评估对照 | 启用两者，通过 `assistant_type` 字段分组采集数据 |
| 长期生产路径 | OpenClaw（能力演进路径更完整）|

> **核心结论**：当前阶段（Stage 1）两者能力接近，OpenClaw 作为 agent runtime 的真正优势要到 Stage 2 才能发挥。建议保留 OpenClaw 作为主路径，接入智谱作为对照组，通过 `assistant_evaluation` 表采集数据做质量对比。

### 11.2 为什么要有"无聊的" openclaw 守护进程？能不能干掉？

**短答案：不建议现在干掉，理由是技术债，但路线图上有明确的替代路径。**

#### 为什么它存在

```
conversation-service
  └── ai_client.check_health()  每 10s 发一次探针
        └── POST http://openclaw:18789/v1/chat/completions  body={}
              └── openclaw 返回 400 Bad Request
                    └── ai_client 判定：400 = healthy ✅

  └── ai_client.chat_completion_stream()  真实业务请求
        └── 先走 pod_endpoint（ProductionClaw 地址）
              └── 若 pod_endpoint 为空（未绑定 Pod）→ fallback 到 base_url
                      └── base_url = http://openclaw:18789  ← 就是守护进程
```

**openclaw 守护进程承担两个职责：**
1. **健康检查靶标**：conversation-service 的 `check_health()` 循环探针目标，确认整个 AI 层存活
2. **Fallback 兜底**：当 Scheduler 还未为某个 Case 分配 ProductionClaw Pod 时（或 Pod 分配超时），请求 fallback 到守护进程处理

#### 它的问题

| 问题 | 说明 |
|------|------|
| **session-memory 无隔离** | 守护进程无 `session-memory` hook，多个并发对话 session 共享 openclaw 内部状态，可能产生跨 Case 上下文污染 |
| **无 Case 绑定语义** | 守护进程不知道当前 session 服务哪个 Case，BOOTSTRAP.md 中的 `CASE_ID` 注入机制不生效 |
| **compaction.mode=safeguard** | 守护进程配置了 safeguard 压缩，与 ProductionClaw 的默认行为不同，Fallback 使用时行为不一致 |
| **controlUi 开放** | `dangerouslyAllowHostHeaderOriginFallback=true` 是安全降级开关，长期不应保留 |

#### 能干掉的前提条件

要安全移除守护进程，需要同时满足：

```
✅ health check 改为探针某个 ProductionClaw 暖池 Pod
✅ Scheduler 保证在对话开始前完成 Pod 绑定（消除 Fallback 场景）
✅ conversation-service base_url 改为空或某个明确的暖池地址
✅ 暖池 Pod 数量充足（>= 高峰并发 Case 数 + 1 个备用）
```

**建议路线**：Phase 3（ReAct 引擎）完成后，Scheduler 的 Pod 管理足够稳定，届时评估移除守护进程，将健康检查改为 `GET /health`（Scheduler Service 代理），消除这个架构冗余。

### 11.4 KB 层详解：双轨知识检索

#### 架构职责

KB Service（`kb-service:8004`）是平台唯一的知识检索入口，为 conversation-service 的 System Prompt 构建提供两轨知识：

```
conversation-service
  └── _build_system_prompt(query="虚拟机开机", case_id, stage)
        ├── kb_client.sop_match()  → SOP 精确匹配（程序性知识）
        └── kb_client.search()    → 语义向量检索（陈述性知识）
```

#### 两轨对比

| 维度 | SOP 精确匹配 | 语义向量检索 |
|------|------------|------------|
| **实现** | 内存 Trie 关键词匹配（`SopMatcher`）| bge-small-zh-v1.5（512维）+ BM25 → RRF 融合 |
| **延迟** | < 1ms（内存操作）| 50-200ms（Embedding + pgvector ANN 查询）|
| **精准度** | 高（关键词完全命中才返回）| 中（语义相似，可能有噪声）|
| **召回率** | 低（用户语言 ≠ SOP 关键词时 miss）| 高（语义相近自动召回）|
| **知识来源** | `data-pipeline/sop_skills/*.yaml` | PostgreSQL pgvector（`knowledge_atoms` 表）|
| **返回结构** | 单个 SOP 节点（含步骤 + 命令）| N 个 chunk（含 rrf_score + case_id）|
| **注入方式** | Segment4 = SOP 步骤（指令性）| Segment4 = 案例列表（参考性）|

#### 三级 Fallback 策略

```python
# conversation_service.py _build_system_prompt()
sop_node, kb_chunks = await asyncio.gather(sop_task, search_task, ...)

if sop_node:
    # 一级：SOP 命中 → 按步骤执行路径
    Segment4 = _SEGMENT_SOP_REFERENCE.format(sop_content)
    fallback_level = "sop"

elif kb_chunks:
    # 二级：KB 案例命中 → 假设生成路径
    Segment4 = _SEGMENT_CASE_REFERENCE.format(chunks[:KB_CONTEXT_MAX_CHARS])
    fallback_level = "kb_case"

else:
    # 三级：均未命中 → 机制推理兜底（GLM 用自身训练知识）
    Segment4 = _SEGMENT_NO_REFERENCE  # 含【机制推理】标注
    fallback_level = "mechanism"
```

**已知缺陷（架构文档 08 中的"SopMatcher 最差触发机制"问题）**：
- 用户说"开机失败" → SopMatcher 可能无法命中 "**启动虚拟机失败**" 关键词
- 根本原因：用户语言与 SOP 关键词之间存在永久性语义 gap，无法靠堆关键词解决
- 规划修复：SOP 触发机制迁移至 Embedding 语义路由（见文档 08 核心架构原则）

#### EmbeddingService 调用链

```
kb-service 内部
  └── SearchEngine.search(query)
        └── EmbeddingService.embed(query)
              └── POST http://宿主机IP:18100/embed
                    └── hci-embedding Docker 容器（bge-small-zh-v1.5）
                          → 返回 512 维向量
        └── pgvector ANN 查询（cosine similarity）
        └── BM25 全文检索（psycopg2 + tsvector）
        └── RRF(vector_rank, bm25_rank) → 最终排序
```

**注意**：`EmbeddingService` 依赖宿主机上的 Docker 容器（`hci-embedding`，独立于 K8s 集群），若该容器停止则语义检索降级为纯 BM25。

### 11.5 AI 层（OpenClaw + ProductionClaw）调用详解

#### conversation-service → ProductionClaw 调用路径

```python
# ai_client.py OpenClawAssistant.chat_completion_stream()

# 端点选择策略（双端点 + fallback）
endpoints_to_try = [pod_endpoint, base_url]  # pod_endpoint 优先
#   pod_endpoint = http://productionclaw-pool-aaba948f:18789  （Scheduler 分配）
#   base_url     = http://openclaw:18789                       （守护进程兜底）

for idx, endpoint in enumerate(endpoints_to_try):
    # 发请求 → 逐行读 SSE
    # 若首 Token 前出现可恢复错误（连接断、超时）→ 切换到下一个端点重试
    # 若已收到首 Token → 不重试（避免重复内容）
```

**可重试错误签名**：`incomplete chunked read` / `peer closed connection` / `read timeout` / `remoteprotocolerror` / `connection reset`

#### ProductionClaw 内部对话流程

```
POST /v1/chat/completions 到达 ProductionClaw Pod
    │
    ▼
openclaw gateway（node openclaw.mjs）
    ├── 验证 Authorization: Bearer  <OPENCLAW_GATEWAY_TOKEN>
    │     不匹配 → HTTP 401（PIT-010 已知问题）
    │
    ├── 解析 messages[]（来自 conversation-service 的 5段式 system prompt + history）
    │
    ├── 进入 Agent Loop
    │     ├── 模型推理（GLM-5，POST https://open.bigmodel.cn/...）
    │     ├── 若模型决定调工具 → 解析 tool_call → 执行 → 结果注入下轮推理
    │     └── 若模型生成最终回答 → 流式输出 token
    │
    ├── session-memory hook：每轮推理后将关键信息写入本 Pod session 记忆
    │     （Pod 销毁后这部分记忆消失，需 LearningClaw 提炼后写入 KB 才能持久化）
    │
    └── SSE stream 回传
```

#### ProductionClaw 的"工具"现状与规划

| 工具来源 | 当前状态 | 说明 |
|---------|---------|------|
| `TOOLS.md`（init ConfigMap 注入）| **仅文档，未连接实际 MCP Server** | 描述了 KB Search、SCP API 等工具接口，但 Phase 1-2 阶段 ProductionClaw 作为纯对话模型使用，工具调用由上层 conversation-service 代劳 |
| KB 检索 | conversation-service 代替执行 | 5段式 Prompt 中已注入检索结果，ProductionClaw 无需主动调用 |
| SCP API（Phase 3）| 规划中 | ReAct 模式下 ProductionClaw 将自主调用 SCP 查询告警、执行诊断命令 |

**设计准则**：Phase 1-2 的 ProductionClaw 是"被动接收 prompt 的高质量 LLM"；Phase 3 后才真正成为"主动调用工具的 Agent"。

### 11.6 健康检查与 400 误判澄清

**现象**：conversation-service 日志中持续出现 `POST /v1/chat/completions → 400 Bad Request`

**这是正常行为，不是异常**：

```python
# ai_client.py check_health()
async def check_health(self) -> bool:
    response = await self.client.post(url, json={}, ...)  # 故意发空 body
    if response.status_code in (200, 400, 422):
        return True   # 400 = openclaw 收到了请求（说明服务存活）= healthy
    return False
```

每 10 秒探测一次，`400` 代表 openclaw 存活并拒绝了格式错误的请求——这是预期的行为，不是业务报错。

**真正的 AI 报错信号**：
- 前端显示"AI 响应出现错误，请稍后重试" → SSE 收到 `event: error`
- conversation-service 日志出现 `event="ai_error"` 或 `event="ai_exception"`
- openclaw 日志出现 `401 Unauthorized`（Token 不匹配，见 PIT-010）
