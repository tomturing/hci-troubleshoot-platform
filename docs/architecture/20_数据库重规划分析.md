# HCI 智能排障平台 — 数据库重规划分析

> **文档目的**：对当前 17 张数据库表进行全面深入审核，识别设计冗余与架构问题，按项目模块重新规划表结构，输出目标态表格清单供后期重构参考。
> **版本**：1.1
> **创建日期**：2026-03-31
> **更新日期**：2026-04-01
> **状态**：已落地实施 — `database/init_schema.sql` 已按此规划完成以下变更：
> - ✅ 删除 `session` 表（Redis 为主态，PG 层不再保存连接状态）
> - ✅ 删除 `kb_synonym` 表（ORM 模型存在但无任何 DAO 查询，确认为死代码）
> - ✅ 删除 `kb_sop_node` 表（关键字路由废弃，SOP 内容迁移至 `knowledge_atoms` 语义路由）
> - ✅ 删除 `prompt_audit` + `tool_audit_log` 表，新建 `audit_log` 统一审计表（按 `audit_type` 区分）
>
> **init_schema.sql 实际表数：12 张**（v3.1）

---

## 一、审核摘要

### 1.1 结论

| 项目 | 数量 |
|------|------|
| 当前表总数 | **17 张** |
| 建议保留（不变） | 7 张 |
| 建议优化/改造 | 2 张 |
| 建议合并（消除） | 4 张 |
| 建议废弃（迁移后删除） | 4 张 |
| 重构后目标表总数 | **12 张** |

**减少 5 张的来源**：
- `session` → 直接 `DROP TABLE`（从未写入 PG，Redis 是 Source of Truth；原始设计意图已被 Redis + `conversation.pod_id` 完全替代）
- `kb_sop_node` → 内容迁移至 `knowledge_atoms` 后废弃（架构明确替换方向）
- `kb_synonym` → 合并入 `kb_category.metadata`（数量 < 500，无独立管理接口需求）
- `error_code_index` → 合并入 `knowledge_atoms`（原子类型扩展）
- `prompt_audit` + `tool_audit_log` → 合并为统一 `audit_log` 表

### 1.2 核心问题诊断

| # | 问题类型 | 影响表 | 说明 |
|---|---------|--------|------|
| P1 | **双轨审计拆散** | `prompt_audit`、`tool_audit_log` | 两张表的职责同属"AI 行为审计"，字段结构不同但查询场景高度重叠，分离造成JOIN复杂度和运维负担 |
| P2 | **SOP 存储即将被淘汰** | `kb_sop_node` | 架构文档（08_重构方案.md）明确指出关键字路由是"已知最差触发机制"，`knowledge_atoms` 将成为 SOP 的唯一存储形式，`kb_sop_node` 处于架构死路 |
| P3 | **错误码孤岛** | `error_code_index` | 这张表在知识体系中孤立存在，无 `kb_document` 外键，无 `knowledge_atoms` 归属，查询路径不清晰 |
| P4 | **Session 角色模糊** | `session` | Redis 是活跃 Session 的 Source of Truth，PostgreSQL `session` 表的角色未明确定义：是主存储还是历史日志？ |
| P5 | **environment 使用率极低** | `environment` | 功能在需求文档中标注为"未来功能"，MVP 阶段几乎无写入，但表已建立占用维护成本 |
| P6 | **prompt_audit 储量过大** | `prompt_audit` | `messages` 字段存储每轮完整的 LLM message 历史（JSONB），一次对话可能产生几十 KB 数据，无 TTL/分区策略 |
| P7 | **分类体系架构孤岛** | `kb_category`、`knowledge_atoms`、`conversation` | `category_baseline.yaml`（198 条叶节点）是排障系统的**全局分类枢纽**，串联意图识别 → SOP/KB 检索 → ETL 标注全链路。但 `kb_category` 表缺少 `code` 业务键（与 YAML 的"虚拟机-003"格式不对齐）、无 `embedding` 向量、是空表；`knowledge_atoms.category_id` 为无外键约束的孤立 VARCHAR；`conversation.category_id` 为单值，无法存储意图识别输出的 1-3 个确认分类 |

## 二、当前 17 张表逐表审核

### 模块一：用户工单（case-service，4 张）

#### T01 `user` — 用户表

| 项 | 内容 |
|----|------|
| **评估结论** | ✅ 保留（不变） |
| **当前作用** | 存储临时用户与认证用户身份信息，以 `client_id`（前端 localStorage UUID）作为业务幂等键 |
| **设计质量** | 优秀：`client_id UNIQUE`、`user_type` 预留认证升级、触发器维护 `updated_at`、三个精准索引 |
| **改进建议** | 无。认证用户 password_hash/refresh_token 等字段待实现认证系统时通过 migration 添加，不提前建列 |

#### T02 `case` — 工单表

| 项 | 内容 |
|----|------|
| **评估结论** | ✅ 保留（不变） |
| **当前作用** | 排障工单核心实体，承载从问题提交到关闭的全生命周期，含状态机（6 态）、来源信息、分类、AI 助手类型 |
| **设计质量** | 优秀：`case_id` 业务可读格式（`Q20260331NNNNN`）、`close_reason` CHECK 约束、冗余 `client_id` 加速查询、v3.2 评分体系基础 |
| **改进建议** | `resolved_at`/`in_progress` 状态当前未使用，未来若引入人工处理升级流程可启用 |

#### T03 `environment` — 环境信息表

| 项 | 内容 |
|----|------|
| **评估结论** | ⚠️ 保留，明确角色定位 |
| **当前作用** | 存储前端（或客户端）采集的 HCI 现场环境数据，JSONB 全量存储，按 `env_type` 分类 |
| **设计质量** | 基本合理：GIN 索引支持 JSON 路径查询，但索引偏多（量少时无意义）|
| **设计问题** | 需求文档标注为"未来功能"，MVP 阶段几乎未写入（客户端数据采集尚未实现）；与 `case.metadata` JSONB 边界不清晰 |
| **改进建议** | 明确规则：若单次采集数据量 < 10KB 且无多次快照需求，数据直接写入 `case.metadata`；若需多时点快照（如中途重采）才写 `environment` 表。当前可保留表但暂停自动索引创建 |

#### T04 `session` — WebSocket 会话表

| 项 | 内容 |
|----|------|
| **评估结论** | ❌ 废弃（直接 `DROP TABLE`，零风险）|
| **当前作用** | 原设计意图：记录用户 + 工单 + AI POD 三者绑定关系。从未实际写入 PG——`SessionManager` 全部操作 Redis，PG session 表无任何写入路径 |
| **设计问题** | **孤儿表**：连接状态已由 Redis（TTL 管理）承接，POD 绑定已由 `conversation.pod_id` 承接，PG session 表成为没有任何写入方的死表 |
| **处置方案** | 直接 `DROP TABLE session`。无需迁移，无历史数据，无代码依赖 |

### 模块二：对话（conversation-service，5 张 → 重构后 4 张）

#### T05 `conversation` — 对话会话表

| 项 | 内容 |
|----|------|
| **评估结论** | ✅ 保留（不变） |
| **当前作用** | 核心对话容器，一个 case 可有多个 conversation；承载 P4 诊断状态机（S0-S6）、分类结果、假设列表、ReAct 状态快照、pending 确认 |
| **设计质量** | 优秀：JSONB 弹性字段 + 结构化状态字段并存；`independent session` 模式避免 SSE 长事务死锁（已在文档说明）|
| **改进建议** | `hypothesis`、`react_state`、`pending_confirm` 三个 JSONB 字段随对话深入体积增长，建议评估是否需要 JSONB 压缩或分离到单独快照表（未来优化项）|

#### T06 `message` — 消息表

| 项 | 内容 |
|----|------|
| **评估结论** | ✅ 保留（不变） |
| **当前作用** | 存储 LLM Context Window 中的 **Messages 部分**（对话历史，约占 Token 22%）：四种角色（user/assistant/system/command）的消息记录。每次 LLM 调用时取出同一 `case_id` 的全量记录直接组装为 `messages[]` 数组送入模型。**不包含**：System Instructions（运行时动态构建）、Tool Definitions（代码静态定义的 acli schema）、Tool Results（acli 执行返回值和 RAG 检索结果）|
| **设计质量** | 优秀：冗余 `case_id` 避免 JOIN、GIN tsvector 全文索引、触发器维护 `conversation.message_count` |
| **改进建议** | 高频写入表，未来规模较大时可按月分区（`created_at`）|

#### T07 `assistant_evaluation` — AI 助手评估表

| 项 | 内容 |
|----|------|
| **评估结论** | ✅ 保留（不变） |
| **当前作用** | 以**工单（`case_id`）为粒度**的双轨评分：用户主动评分（`score` 1-5，~30% 覆盖率）+ 综合质量分（`composite_score` 0-100，100% 覆盖）；`score_breakdown` 记录各维度分解。一个 case 对应一条评估记录 |
| **设计质量** | 优秀：冗余 `close_reason` 避免 JOIN；`composite_score` partial index 加速低分预警查询 |
| **⚠️ 归属错位** | 主键是 `case_id`，语义上属于**工单模块（case-service）**，不属于对话模块。当前住在 conversation-service 是工程实现的错位——`quality_score.py` 需要读 `prompt_audit`（同一服务），导致评估表跟着住过来了。严格领域划分应归 case-service，由 case 关闭事件触发计算 |
| **改进建议** | 将 `quality_score.py` 迁移到 case-service，或通过事件/消息队列跨服务触发计算；评估记录的所有权应与工单生命周期对齐 |

#### T08 `prompt_audit` — AI Prompt 审计表 → **拟合并**

| 项 | 内容 |
|----|------|
| **评估结论** | 🔀 合并入 `audit_log` |
| **当前作用** | 记录每轮 LLM 调用时 **System Instructions 的构建过程**：注入了哪些 KB 案例（`kb_chunks_count/kb_top_score`）、是否加载了 SOP 手册（`has_sop`）、各部分 Token 占比（`context_breakdown`：专家身份+方法论 / 分类基准 / SOP / 对话历史）；附录整轮 messages 完整快照（`messages` JSONB，体积大）|
| **设计质量** | 功能完整但存储压力大：`messages` JSONB 字段存储整轮对话历史，单条可达数十 KB |
| **设计问题** | 1) 无数据保留策略（无 TTL、无分区）；2) 与 `tool_audit_log` 同属"AI 行为审计"但分散在两张表，运维成本翻倍；3) 查询时需跨表 JOIN 才能关联工具调用 |
| **合并建议** | 与 `tool_audit_log` 合并为 `audit_log(audit_type, payload JSONB)`，见 §三 |

#### T09 `tool_audit_log` — 工具调用审计表 → **拟合并**

| 项 | 内容 |
|----|------|
| **评估结论** | 🔀 合并入 `audit_log` |
| **当前作用** | 记录 ReAct 执行器每次工具调用（当前只有 `acli` 一个工具，类似 esxcli 的 HCI 运维命令行接口）：tool_name、risk_level、policy、authorized_by、result、error、duration_ms |
| **设计质量** | 字段精准，risk_level 分级设计优秀；但作为安全审计表，"record 不可删除"要求需通过权限而非表结构保障 |
| **设计问题** | 与 `prompt_audit` 同属 AI 审计域，分离导致"某次 LLM 调用触发了哪些工具"这类问题需跨表 JOIN |
| **合并建议** | 与 `prompt_audit` 合并为 `audit_log`，见 §三 |

### 模块三：知识库（kb-service，5 张 → 重构后 3-4 张）

#### T10 `kb_document` — 知识库文档表

| 项 | 内容 |
|----|------|
| **评估结论** | ✅ 保留（不变） |
| **当前作用** | 知识文档 Source of Truth：全文 MD、YAML 元数据、分类、状态机（draft→published）、审核流程、content_hash 去重 |
| **设计质量** | 优秀：状态机完整（6 态）、content_hash 幂等去重、`source_type` 区分来源（kb/sop/realtime）|
| **改进建议** | 当文档量 > 10,000 后考虑按 `status`/`source_type` 做表分区；`content_md` + `yaml_meta` 存储体积较大，冷归档文档可将 `content_md` 移至对象存储（S3）|

#### T11 `kb_chunk` — 文档分块 + 向量表

| 项 | 内容 |
|----|------|
| **评估结论** | ✅ 保留（不变） |
| **当前作用** | 双路检索核心：`embedding vector(384)` 支持 IVFFlat 余弦相似度检索，`tsv tsvector` 支持 BM25 全文检索，两路通过 RRF 融合重排 |
| **设计质量** | 优秀：分离 document_id 外键确保 cascade delete 正确；GIN 索引覆盖 BM25；IVFFlat 需延迟创建（数据量 ≥ 1000）|
| **改进建议** | 目前 embedding 维度硬编码为 384（bge-small-zh），切换到更高维度 embedding API 时需重建所有向量，届时需要 migration 计划 |

#### T12 `kb_sop_node` — SOP 决策树节点表 → **拟废弃**

| 项 | 内容 |
|----|------|
| **评估结论** | ❌ 废弃（内容迁移后删除） |
| **当前作用** | 存储 SOP 排障手册决策树节点，通过 `keywords TEXT[]` 数组做关键字精确路由 |
| **架构冲突** | 架构文档 08 明确指出：**"SopMatcher 的关键字路由是已知最差的触发机制"**——用户语言与 SOP 关键字之间的 gap 是永久性的，无法通过增加关键字解决。"SOP 与 RAG 不应该是两套系统，它们应统一为语义路由 + 知识原子" |
| **替代方案** | `knowledge_atoms` 表，`atom_type = 'diagnostic_step'/'decision_gate'/'fix_action'`，配合 embedding 语义匹配路由 |
| **迁移路径** | 1) 将 `kb_sop_node` 全量内容转换为 `knowledge_atoms` 记录；2) 为每条记录生成 `trigger_json.embedding`；3) 切换路由层到 `knowledge_atoms`；4) 验证后删除 `kb_sop_node` |

#### T13 `kb_category` — 知识分类树（**全局分类枢纽**）

| 项 | 内容 |
|----|------|
| **评估结论** | 🔧 必须改造：补充 `code` 业务键 + `embedding` 向量列，完成 YAML 数据初始化入库 |
| **战略地位** | `category_baseline.yaml`（198 条叶节点）是排障流程的**中枢**。完整流程如下：<br>① **意图识别（核心流程）**：系统结合用户描述 + 告警日志 + 任务日志，从 198 个叶节点中选出 1-3 个最匹配的故障分类，展示给客户确认——**这是整个排障流程意图识别阶段的输出**；<br>② **动态检索路由**：客户确认分类后，后续所有 SOP 检索和 KB 检索均以确认的分类节点为路由依据，实现针对性知识召回；<br>③ **ETL 分类基准**：`knowledge_atoms.category_id`（如"虚拟机-003"）来自此 YAML，确保知识原子的适用范围与在线分类体系一致。<br><br>**工单（case）→ 对话（conversation.category_ids）→ 知识库（knowledge_atoms.category_id）→ 数据管道（ETL 标注）** 四个模块均通过分类 ID 串联，`kb_category` 是整个系统的横向枢纽 |
| **当前设计缺陷** | 1. **ID 体系断层**：`kb_category` 主键是 `SERIAL`（整型），但 `category_baseline.yaml` 的 ID 是字符串格式（如"虚拟机-003"）；`knowledge_atoms.category_id VARCHAR(32)` 存储 YAML ID 但**无外键约束**，两套 ID 体系完全脱节<br>2. **缺少业务键字段**：无 `code VARCHAR(32) UNIQUE`，无法从 YAML ID 反查 DB 记录，代码层只能用字符串自行维护一致性<br>3. **无 embedding 向量**：意图识别阶段需要"用户描述的故障" → 语义匹配 → 最相关的分类节点；`keywords[]` 关键字匹配无法覆盖用户自然语言的表达差异（如"虚拟机起不来"≠关键字"虚拟机开关机失败"）<br>4. **是空表**：当前 `kb_category` 无任何业务代码查询，`category_baseline.yaml` 数据从未导入 DB，表存在但实际未投入使用<br>5. **单值存储**：`conversation.category_id` 是单值字段，但意图识别输出 1-3 个分类节点，存在信息丢失 |
| **改造方案** | 见下方 DDL 变更与初始化方案 |

**`kb_category` 改造 DDL 变更**：

```sql
-- 新增业务键（对应 category_baseline.yaml 的 id 字段格式）
ALTER TABLE kb_category
    ADD COLUMN code VARCHAR(32) UNIQUE;  -- 如 "虚拟机-003", "存储-012"

-- 新增语义向量（意图识别阶段对用户描述做 embedding 相似度匹配）
ALTER TABLE kb_category
    ADD COLUMN embedding vector(1536);   -- 对 "label + path" 拼接文本生成的向量

-- 新增索引
CREATE INDEX idx_kb_category_code ON kb_category(code);
-- 向量索引（叶节点数量 198 条，用 HNSW 优于 IVFFlat）：
-- CREATE INDEX idx_kb_category_embed ON kb_category
--     USING hnsw (embedding vector_cosine_ops);
```

**`conversation` 表改造（单值 → 数组）**：

```sql
-- 意图识别输出 1-3 个叶节点，单值 category_id 改为数组
ALTER TABLE conversation
    ADD COLUMN category_ids INTEGER[] DEFAULT '{}';  -- 引用 kb_category.id 列表
-- 渐进式迁移：先加新列，完成代码切换后废弃旧 category_id 列
```

**`category_baseline.yaml` 初始化方案**（`scripts/seed_categories.py`，一次性执行）：

```
执行步骤：
1. 解析 YAML path 数组，从根到叶递归 INSERT 中间层节点
   - ON CONFLICT (name, level) DO NOTHING（根、域等节点去重）
2. 叶节点写入 code = yaml.id（如 "虚拟机-003"）、name = yaml.label
3. 批量调用 embedding API
   - 输入文本："label > path[0] > path[1] > ..."（面包屑拼接）
   - 写入 kb_category.embedding（1536 维）
4. 验证：SELECT count(*) FROM kb_category WHERE code IS NOT NULL → 应为 198

分工边界：
- category_baseline.yaml = 种子文件（版本控制，YAML 可读，人工可维护）
- kb_category            = 运行时权威数据源（DB 状态机，支持向量检索和关联查询）
- 后续新增分类：通过管理接口写 kb_category，同步更新 YAML（或放弃 YAML 同步，以 DB 为准）
```

### 模块四：数据管道（data-pipeline，3 张 → 重构后 2 张）

#### T15 `raw_cases` — 原始工单数据表

| 项 | 内容 |
|----|------|
| **评估结论** | ✅ 保留（不变） |
| **当前作用** | ETL 管道输入层：存储从外部知识库爬取的、已脱敏的历史工单原始 Markdown，`quality_score` 过滤低质量工单 |
| **设计质量** | 合理：幂等键 `case_id UNIQUE`、质量评分索引倒序查询高质工单 |
| **改进建议** | 此表数据量可能达到 7000+ 条，`content_text TEXT` 字段存储 Markdown 全文，建议评估压缩存储（`content_text BYTEA` + gzip）|

#### T16 `knowledge_atoms` — 知识原子表

| 项 | 内容 |
|----|------|
| **评估结论** | ✅ 保留，扩展承载 SOP 和错误码 |
| **当前作用** | AI 自动提炼的知识原子候选，`verified` 字段控制人工审核门控；`atom_type` 区分类型（diagnostic_step/fix_action/decision_gate）|
| **设计质量** | 优秀：source_type 追溯来源、confidence 置信度、verified 审核门控 |
| **扩展方向** | 重构后此表还需承载：① `kb_sop_node` 迁移过来的 SOP 节点（atom_type 扩展）；② `error_code_index` 的错误码数据（atom_type = 'error_code_ref'）；需在 `trigger_json` 字段中增加 `error_code_patterns` 子字段 |

#### T17 `error_code_index` — 错误码索引表 → **拟合并**

| 项 | 内容 |
|----|------|
| **评估结论** | 🔀 合并入 `knowledge_atoms` |
| **当前作用** | 错误码到故障原因的快速索引，供检索时精确匹配加权 |
| **设计问题** | 1) 无 `kb_document` 外键，数据孤岛；2) 与 `knowledge_atoms` 存在语义重叠（知识原子本就应覆盖错误码触发场景）；3) 查询时需要 UNION 两个数据源增加复杂度 |
| **合并方案** | 扩展 `knowledge_atoms.trigger_json` 结构：新增 `trigger_json.error_code_patterns: string[]` 字段，`atom_type = 'error_code_ref'` 的原子就是当前 `error_code_index` 的完整替代 |

```sql
CREATE TABLE audit_log (
    id           VARCHAR(36)  PRIMARY KEY,           -- UUID
    audit_type   VARCHAR(20)  NOT NULL,               -- 'prompt' | 'tool_call' | 'system'
    conversation_id  UUID     NOT NULL,               -- 关联 conversation
    turn_index   SMALLINT,                            -- prompt 类型：第几轮对话
    tool_name    VARCHAR(100),                        -- tool_call 类型：工具名称
    risk_level   SMALLINT,                            -- tool_call 类型：1=只读 2=写 3=高危
    policy       VARCHAR(20),                         -- tool_call 类型：auto|notify|confirm|block
    authorized_by VARCHAR(100),                       -- tool_call 类型：高危操作确认用户
    payload      JSONB        NOT NULL DEFAULT '{}',  -- 类型专属字段（prompt 上下文 / 工具结果）
    error        TEXT,                                -- 错误信息（失败时）
    duration_ms  INTEGER,                             -- 执行耗时
    started_at   TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
    completed_at TIMESTAMPTZ,
    trace_id     VARCHAR(64)
);
-- 索引
CREATE INDEX idx_audit_log_conversation  ON audit_log(conversation_id, started_at DESC);
CREATE INDEX idx_audit_log_type          ON audit_log(audit_type, started_at DESC);
CREATE INDEX idx_audit_log_tool_name     ON audit_log(tool_name) WHERE tool_name IS NOT NULL;
CREATE INDEX idx_audit_log_risk_level    ON audit_log(risk_level) WHERE risk_level >= 2;
CREATE INDEX idx_audit_log_trace_id      ON audit_log(trace_id) WHERE trace_id IS NOT NULL;
```

## 五、迁移路径（参考顺序）

### 阶段 R1：审计表合并（低风险，不影响业务）

**前置评估**：`prompt_audit` 现有数据量；`tool_audit_log` 数据是否需要历史保留

1. 创建 `audit_log` 表（新建，不删旧表）
2. 编写迁移脚本：将 `prompt_audit` 历史数据转换为 `audit_type='prompt'` 记录写入 `audit_log`
3. 将 `tool_audit_log` 历史数据转换为 `audit_type='tool_call'` 记录写入 `audit_log`
4. 更新 conversation-service 代码：写入目标改为 `audit_log`
5. 验证新表数据完整性后，重命名/归档旧两张表
6. 删除旧表

### 阶段 R2：SOP 节点迁移（中等风险，影响 KB 路由层）

**前置评估**：`kb_sop_node` 当前节点数量；生产环境 SOP 路由命中率基线

1. 扩展 `knowledge_atoms` atom_type 枚举，添加 `diagnostic_step/decision_gate/threshold/fix_action`
2. 编写迁移脚本：`kb_sop_node` → `knowledge_atoms`（保留 `skill_id`、`keywords`→`trigger_json.keywords`、`content`→`content_json.full_text`）
3. 为迁入的原子批量生成 embedding（调用 z.ai API）写入 `trigger_json.embedding`
4. KB Service 路由层新增基于 embedding 相似度的语义路由，与关键字路由并行运行（A/B 对比）
5. 验证语义路由命中率 ≥ 关键字路由后，停用关键字路由
6. 删除 `kb_sop_node` 表

### 阶段 R3：错误码迁移（低风险）

1. 扩展 `knowledge_atoms` `atom_type` 添加 `error_code_ref`
2. `trigger_json` 新增 `error_code_patterns: string[]` 字段规范
3. 迁移 `error_code_index` 数据
4. 删除 `error_code_index` 表

### 阶段 R4：Session 表直接删除（零风险）

1. 确认代码中无任何服务对 PG `session` 表的写入路径（已验证：`SessionManager` 全部操作 Redis，零 PG 写入）
2. 确认无 ORM Model 文件引用 `session` 表
3. 执行 `DROP TABLE IF EXISTS session;`
4. 从 `init_schema.sql` 中删除 `session` 建表 DDL

### 阶段 R5：分类体系初始化（零风险，建议最优先执行）

**前置条件**：无（独立，不依赖其他阶段）

**为什么不能用 YAML 文件直接驱动运行时**：意图识别需要 embedding 语义检索（YAML 不支持）；跨模块关联查询需要 DB 索引；运营人员需要管理接口动态增减分类节点。YAML 保留为种子文件（版本控制），DB 作为运行时权威数据源。

1. 为 `kb_category` 添加列：
   ```sql
   ALTER TABLE kb_category ADD COLUMN code VARCHAR(32) UNIQUE;
   ALTER TABLE kb_category ADD COLUMN embedding vector(1536);
   CREATE INDEX idx_kb_category_code ON kb_category(code);
   ```
2. 编写 `scripts/seed_categories.py`，执行以下操作：
   - 解析 `data-pipeline/config/category_baseline.yaml`
   - 对每条记录按 `path` 数组递归 INSERT 中间层节点（`ON CONFLICT DO NOTHING`）
   - 叶节点写入 `code = yaml_id`（如"虚拟机-003"）
   - 调用 embedding API，对 `label + " > " + " > ".join(path)` 生成 1536 维向量，写入 `embedding`
3. 执行种子脚本，验证：`SELECT count(*) FROM kb_category WHERE code IS NOT NULL` → 应为 198
4. 为 `conversation` 表添加 `category_ids INTEGER[] DEFAULT '{}'` 列（保留旧 `category_id` 渐进迁移）：
   ```sql
   ALTER TABLE conversation ADD COLUMN category_ids INTEGER[] DEFAULT '{}';
   ```
5. 更新意图识别模块：输出由字符串 YAML-ID 改为对 `kb_category.embedding` 做向量检索，返回 top-3 叶节点的 `kb_category.id` 列表；写入 `conversation.category_ids`
6. `knowledge_atoms.category_id` 改为与 `kb_category.code` 对齐（可加 FK 约束或软关联，视 ORM 灵活性决定）

## 六、附录：FAQ

### Q1：为什么是 12 张而不是更少？

合并更激进（如把 `message` 并入 `conversation`、把 `environment` 并入 `case.metadata`）会打破关注点分离，增加单表体积和锁竞争。当前 12 张表在"精简"与"清晰"之间取到了较好平衡点。

### Q2：`kb_synonym` 是否建议合并？

建议条件：同义词数量长期 < 500 且无独立 CRUD 接口需求时合并到 `kb_category.metadata = {"synonyms": [...]}`；若已有管理后台专门管理同义词，保持独立表。

### Q3：`audit_log` 合并后如何处理 retention 策略差异？

`prompt_audit` 数据量大（每条可达数十 KB），`tool_audit_log` 数据量小但安全合规要求高。合并后通过两个机制解决：
1. `audit_log.audit_type` 字段 + 分区表（PostgreSQL range partition by started_at），`prompt` 类型三个月后可 detach partition 归档；`tool_call` 类型长期保留
2. `prompt` 类型的完整 messages 不写入 payload，改写外部存储引用（`payload.payload_ref`），减少行体积

### Q4：迁移期间如何保证双写一致性？

各阶段均采用"新旧并存→流量切换→旧表归档"策略：
1. 先建新表
2. 代码层新增双写（同时写旧表和新表），验证数据一致性
3. 读取层切换到新表
4. 去掉旧表写入
5. 等待一个 retention 周期后删除旧表

### Q5：`category_baseline.yaml` 的数据为何必须入库？用 YAML 直接驱动运行时不行吗？

**不行**，原因有三：

1. **意图识别性能**：从 198 个分类节点中，对用户描述（自然语言）做语义匹配，找出最相关的 1-3 个叶节点，需要 `embedding vector` + pgvector 索引。YAML 是静态文件，无向量检索能力，关键字匹配覆盖不了用户表达的语言差异。

2. **跨模块关联查询**：`conversation.category_ids → kb_category.id → knowledge_atoms.category_id`，这条关联链路需要 DB 外键约束和索引来保证一致性和查询性能。YAML 无法参与 SQL JOIN。

3. **分类树可维护性**：业务演进中，运营团队需要新增/修改/废弃分类节点（`source = 'auto_suggested'` → 审核 → `manual`），需要 DB 的 CRUD 接口和状态机。YAML 只能通过 git PR 维护，没有在线管理能力。

**分工建议**：
- `category_baseline.yaml` = **种子文件**（版本控制，人工可读，用于环境重建/对齐）
- `kb_category` = **运行时权威数据源**（DB 状态机，embedding 向量，支持关联查询）
- 两者通过 `scripts/seed_categories.py` 保持对齐；后续新增分类以 DB 为准，YAML 可异步同步或归档
