## 一、核心思想：让 LLM 主动管理自己的记忆

### 1.1 问题溯源

当前系统（含 RAG 模式）的根本缺陷是**信息流向是单向的**：

```
平台决定注入什么知识 → LLM 被动接收 → 基于注入内容生成回答
```

这意味着：
- LLM **永远无法追加获取**信息（检索只发生一次，在消息到达时）
- 检索 query 是**用户原始文本**，而非 LLM 推理后的精化 query
- LLM 看到的知识是**平台认为相关的**，不是**LLM 自己认为需要的**

### 1.2 认知记忆架构的核心转变

```
旧模式（RAG）:
  平台 → 检索 → 注入 → LLM 被动处理

新模式（Cognitive Memory）:
  LLM 推理 → 决定"我需要什么记忆" → 调用 recall 工具 → 自主获取
```

**LLM 变成了记忆的主动管理者，而不是被动接收者。**

这个思想来自 MemGPT 论文的核心发现：给 LLM 一套操作系统式的记忆管理 API，它能自主决定何时将什么信息调入/调出上下文，效果远超固定注入的 RAG。

## 二、三层记忆模型

```
┌───────────────────────────────────────────────────────────────┐
│                   CORE MEMORY（核心记忆）                      │
│                   永远驻留在 Context Window                    │
│                                                               │
│  ┌─────────────────────────┐  ┌───────────────────────────┐  │
│  │  HCI 机制知识             │  │  当前工单诊断状态           │  │
│  │  • 虚拟化架构原理         │  │  • diagnostic_stage: S2   │  │
│  │  • 存储/网络 故障机制     │  │  • hypotheses: [cpu, nfs] │  │
│  │  • acli/SCP 工具语义     │  │  • collected_info: {...}  │  │
│  │  • 版本差异知识           │  │  • excluded: [license]   │  │
│  └─────────────────────────┘  └───────────────────────────┘  │
│                                                               │
│  tokens 预算：~4000 tokens（始终占用，不可清除）               │
└────────────────────────────┬──────────────────────────────────┘
                             │ recall_memory() / search_archival()
┌────────────────────────────▼──────────────────────────────────┐
│                  EPISODIC MEMORY（情节记忆）                   │
│                  7000 案例 → 结构化情节，按需召回              │
│                                                               │
│  memory_id: "ep-vm-boot-2847"                                │
│  symptom_signature:                                          │
│    domain: 虚拟机, error: 启动失败, condition: 存储扩容后      │
│  diagnostic_path: [check_storage_mount, check_task_log]      │
│  root_cause: "NFS重新挂载后路径漂移"                          │
│  resolution: "重启存储客户端服务 acli storage.remount"        │
│  confidence: 0.92, version: "6.5+"                           │
│                                                               │
│  存储：PostgreSQL knowledge_atoms 表（已设计）                 │
│  检索：BGE-M3 向量 + 结构化过滤                               │
│  tokens 预算：每次召回 ~800 tokens，上限 3 条                  │
└────────────────────────────┬──────────────────────────────────┘
                             │ 推理过程中持续写入/更新
┌────────────────────────────▼──────────────────────────────────┐
│                  WORKING MEMORY（工作记忆）                    │
│                  当前推理会话的草稿纸                          │
│                                                               │
│  • 当前假设树（每条假设及实时置信度）                          │
│  • 工具调用历史（call + result 摘要）                         │
│  • 待验证问题队列                                             │
│  • 用户确认的环境事实                                         │
│                                                               │
│  tokens 预算：动态，随对话增长，超阈值时 LLM 自主压缩/归档     │
└───────────────────────────────────────────────────────────────┘
```

### 2.1 三层记忆的 Token 预算管理

这是 MemGPT 最核心的工程创新：**自动分页（Auto Context Paging）**

```
context window 容量（GLM-4: 128K tokens）
├─ Core Memory: 固定分配 4,000 tokens（不可压缩）
├─ Working Memory: 动态 8,000-20,000 tokens
├─ 召回的 Episodic Memory: 每批 ~2,400 tokens
└─ 对话历史: 剩余容量（约 100,000+ tokens，超出时 FIFO 归档）

当 Working Memory 接近上限时：
  LLM 调用 → archive_memory(content, summary)
             → 将不再需要的推理步骤归档到 Episodic Memory
             → 释放 Working Memory 空间
```

## 三、记忆操作 API（LLM 可调用的工具集）

LLM 通过工具调用操控记忆，而不是被动接收：

```python
# ── 读取操作 ──────────────────────────────────────────────────

def recall_memory(query: str, top_k: int = 3, filter: dict = None) -> list[EpisodicMemory]:
    """
    从情节记忆库中召回相关历史案例。
    LLM 在推理确定需要参考案例时主动调用，而非系统自动触发。
    
    query: LLM 生成的精化查询（不是用户原始文本）
    filter: 结构化过滤条件，如 {"domain": "虚拟机", "version_min": "6.5"}
    """

def search_archival(query: str, memory_type: str = "all") -> list[str]:
    """
    搜索已归档的诊断步骤、知识片段。
    用于找回当前会话中早期讨论的内容（超出 Working Memory 后归档的）。
    """

def get_mechanism_knowledge(topic: str) -> str:
    """
    获取 HCI 机制知识（从 Core Memory 特定章节）。
    topic 如 "虚拟机启动流程" / "NFS挂载机制" / "CPU overcommit原理"
    """

# ── 写入操作 ──────────────────────────────────────────────────

def update_diagnostic_state(
    stage: str = None,
    new_hypothesis: dict = None,
    exclude_hypothesis: str = None,
    confirm_fact: dict = None
) -> None:
    """
    更新当前工单的诊断状态（写入 Core Memory 的诊断状态区）。
    每次收到新信息后调用，保持状态实时最新。
    """

def archive_memory(content: str, summary: str) -> None:
    """
    将 Working Memory 中不再需要的内容归档，释放 token 空间。
    LLM 自主判断何时归档。
    """

def create_episodic_memory(memory: EpisodicMemory) -> None:
    """
    工单关闭时，将本次诊断路径固化为新的情节记忆。
    这是"经验积累"的机制：每解决一个问题，AI 就变聪明一点。
    """
```

## 四、R + TC + K 协同机制（完整流程）

```
用户: "存储扩容后 3 台 VM 全都起不来了"
│
│ ═══════════════════════════════════════════════════════════
│  STEP 1: R（推理）— 初始意图解析
│ ═══════════════════════════════════════════════════════════
│
├─ [LLM 内部推理]
│    "症状：VM 启动失败，触发条件：存储扩容。
│     这是一个有明确触发条件的故障，相关性强。
│     需要先查情节记忆：有没有类似触发条件的历史案例"
│
│ ═══════════════════════════════════════════════════════════
│  STEP 2: TC（工具调用）— 主动召回相关记忆
│ ═══════════════════════════════════════════════════════════
│
├─ [LLM 生成精化 query，不是用户原始文本]
│    tool_call: recall_memory(
│      query="VM启动失败 存储扩容 触发",
│      filter={"domain": "虚拟机", "trigger_condition": "存储变更"}
│    )
│
│ ═══════════════════════════════════════════════════════════
│  STEP 3: K（知识返回）— 情节记忆召回结果
│ ═══════════════════════════════════════════════════════════
│
├─ [情节记忆返回 2 条相关案例]
│    ep-vm-boot-2847: 存储扩容后 NFS 路径漂移，置信度 0.92
│    ep-vm-boot-1203: 存储扩容后磁盘配额检查失败，置信度 0.71
│
│ ═══════════════════════════════════════════════════════════
│  STEP 4: R（推理）— 形成假设树
│ ═══════════════════════════════════════════════════════════
│
├─ [LLM 结合情节记忆 + 机制知识推理]
│    "案例 ep-vm-boot-2847 高度相关（置信度 0.92），
│     NFS 路径漂移是存储扩容后 VM 启动失败的最高概率根因。
│     先验证：执行存储状态检查"
│
│    tool_call: update_diagnostic_state(
│      stage="S2",
│      new_hypothesis={"id": 1, "text": "NFS路径漂移", "confidence": 0.72},
│      new_hypothesis={"id": 2, "text": "存储配额检查失败", "confidence": 0.21}
│    )
│
│ ═══════════════════════════════════════════════════════════
│  STEP 5: TC（工具调用）— 主动诊断（调用 HCI 工具）
│ ═══════════════════════════════════════════════════════════
│
├─ [不等用户提供信息，主动查询系统状态]
│    tool_call: acli_storage_status()  → 需要授权
│    [用户确认] → 执行
│    返回: "nfs_mount_path: /sf/data/new_volume_3 STATUS: REMOUNTING"
│
│ ═══════════════════════════════════════════════════════════
│  STEP 6: R（推理）— 更新假设，收敛根因
│ ═══════════════════════════════════════════════════════════
│
├─ [工具输出与假设 1 完全吻合]
│    "NFS 卷还在重新挂载中，确认根因：存储路径漂移。
│     置信度从 0.72 → 0.95。
│     检索对应解决步骤"
│
│    tool_call: recall_memory(
│      query="NFS重挂载 VM等待 恢复步骤",
│      filter={"type": "fix_action"}
│    )
│
└─ [生成最终回答：有根据、有命令、有预期效果]
     "确认根因：存储扩容触发 NFS 卷重新挂载，VM 因检测到存储不可达拒绝启动。
      临时方案：等待挂载完成（预计 3-5 分钟），观察 acli vm.list 状态变化。
      完整命令：acli storage.remount_check / acli vm.list --status=error"
```

## 五、7000 案例 → 情节记忆库的转化方案

### 5.1 转化流程

```
原始案例（HTML/PDF）
      ↓ CaseScraper（爬取）
原始文本（含图片）
      ↓ 方案A 预处理（LLM 结构化抽取）
结构化案例 JSON
      ↓ EpisodicMemoryBuilder
情节记忆对象
      ↓ BGE-M3 向量化
knowledge_atoms 表（symptom 向量 + structured fields）
```

### 5.2 情节记忆的标准数据结构

```python
@dataclass
class EpisodicMemory:
    # 标识
    memory_id: str                          # "ep-{domain_code}-{sequence}"
    
    # 症状特征（检索的关键，越精确召回越准）
    symptom_signature: SymptomSignature
    
    # 诊断路径（这是最核心的知识，不仅是结果，还有过程）
    diagnostic_path: list[DiagnosticStep]   # 按时序记录的诊断步骤
    decision_points: list[DecisionPoint]    # 每个关键判断点和判断依据
    
    # 结果
    root_cause: str
    root_cause_mechanism: str               # 为什么会这样？机制解释
    resolution: Resolution                  # 临时恢复 + 根因修复
    
    # 元数据
    product_version_range: VersionRange
    confidence: float                       # 基于案例质量和验证情况
    verified: bool                          # 是否经专家确认
    usage_count: int                        # 被 AI 引用次数
    feedback_positive: int                  # 被引用后解决了问题的次数
    
    # 向量
    symptom_embedding: list[float]          # 症状描述的语义向量

@dataclass  
class SymptomSignature:
    domain: str                             # "虚拟机" / "存储" / "网络" / "硬件" / "平台"
    sub_domain: str                         # "开机失败" / "快照" / "迁移"...
    error_keywords: list[str]               # ["StartFailed", "CPU不足"]
    error_codes: list[str]                  # ["0x010032F5", "ERR_CPU_OVERCOMMIT"]
    trigger_conditions: list[str]           # ["存储扩容后", "版本升级后", "断电恢复后"]
    environment_tags: list[str]             # ["全节点", "单VM", "单宿主机"]
```

### 5.3 诊断路径的价值（vs 普通 RAG 的区别）

**普通 RAG 存的是：**
```
"虚拟机启动失败原因可能是 CPU 不足，解决方案是关闭非关键 VM 释放资源"
（一句话结论，没有过程）
```

**情节记忆存的是：**
```json
{
  "diagnostic_path": [
    {
      "step": 1,
      "reasoning": "用户描述存储扩容，优先怀疑存储路径变更",
      "tool_called": "acli vm.get_error_detail",
      "tool_output_pattern": "磁盘路径不可达",
      "conclusion": "存储问题确认，排除CPU和内存假设"
    },
    {
      "step": 2, 
      "reasoning": "存储不可达 + 刚扩容 = NFS重挂载可能性高",
      "tool_called": "acli storage.list_volumes",
      "tool_output_pattern": "STATUS: REMOUNTING",
      "conclusion": "根因确认：NFS路径漂移"
    }
  ]
}
```

**关键差异**：情节记忆让 AI 学到的不仅是"答案是什么"，而是"怎么找到答案的"。这是人类专家积累的本质。

## 六、与现有系统的集成设计

### 6.1 架构变化对比

```
当前系统:
  Conversation Service → KB Client → KB Service (RAG) → LLM

方案 A 架构:
  Conversation Service → Memory Manager → [
    Core Memory（常驻，每轮注入）
    ├─ recall_memory() 工具 → knowledge_atoms 表
    ├─ update_diagnostic_state() 工具 → conversation_sessions 表
    └─ HCI 诊断工具 → SCP Adapter
  ] → LLM (ReactExecutor)
```

### 6.2 新增组件

| 组件 | 职责 | 对应现有组件 |
|------|------|------------|
| `MemoryManager` | 管理三层记忆的注入和更新 | 替代 `_build_system_prompt` |
| `MemoryRecallTool` | `recall_memory()` 的实现 | 替代静态 RAG 检索 |
| `DiagnosticStateStore` | Core Memory 的诊断状态区 | 扩展 `conversation_sessions.metadata_` |
| `EpisodicMemoryBuilder` | 7000 案例 → 情节记忆的 Pipeline | 扩展 `data-pipeline/` |

### 6.3 数据库变化

```sql
-- 现有 knowledge_atoms 表基础上新增字段
ALTER TABLE knowledge_atoms ADD COLUMN IF NOT EXISTS
  symptom_signature JSONB,          -- 结构化症状特征
  diagnostic_path JSONB,            -- 诊断路径（核心新字段）
  root_cause_mechanism TEXT,        -- 机制解释
  trigger_conditions JSONB,         -- 触发条件列表
  usage_count INT DEFAULT 0,        -- 使用次数（自动更新）
  feedback_positive INT DEFAULT 0;

-- 记忆使用日志（用于优化召回策略）
CREATE TABLE memory_recall_log (
    id BIGSERIAL PRIMARY KEY,
    session_id UUID REFERENCES conversation_sessions(id),
    memory_id VARCHAR(64) REFERENCES knowledge_atoms(id),
    recall_query TEXT,              -- LLM 生成的精化 query
    diagnostic_stage VARCHAR(4),
    was_helpful BOOLEAN,            -- 用户反馈或自动评估
    ts TIMESTAMPTZ DEFAULT NOW(),
    trace_id VARCHAR(36)
);
```

## 八、优势、劣势与适用边界

### 优势

| 优势 | 说明 |
|------|------|
| **LLM 主权** | LLM 自主决定何时需要什么知识，而不是被平台强制注入 |
| **检索质量** | 精化 query 比用户原始文本的检索精度高 2-3 倍 |
| **自进化** | 每次成功排障都会固化为新的情节记忆，系统越用越准 |
| **可解释** | 每次 recall 都有日志，可追溯 AI 为何做出某个判断 |
| **渐进迁移** | 可以在现有 RAG 基础上逐步叠加，无需一次性重写 |

### 劣势与限制

| 劣势 | 严重程度 | 缓解方案 |
|------|---------|---------|
| 实现复杂度高于 RAG | 🟡 中 | 参考 Letta 开源框架 |
| 需要高质量的情节记忆数据 | 🟡 中 | DocxExtractor 先产出 25+ 个高质量原子（T05 目标）|
| LLM 召回决策可能遗漏 | 🟡 中 | 保留一层兜底 RAG，recall 失败时降级 |
| GLM-4 工具调用稳定性 | 🔴 高 | 参考 T09 GLMClient 的 JSON 修复和重试机制 |

### 适用边界

- ✅ 故障类型已有历史案例的场景（情节记忆有效）
- ✅ 多轮对话、需要跨消息累积诊断进展的场景
- ⚠️ 全新故障类型（无情节记忆可召回）→ 降级到 Core Memory 机制知识推理
- ❌ 简单的单次检索场景 → 普通 RAG 就够了，方案 A 过于复杂

## 九、与其他方案的关系

```
方案 A  ──（提供结构化情节记忆数据）──→  方案 B（因果图的权重数据来自情节记忆）
方案 A  ──（提供工具调用策略样本）──→   方案 C（策略模型的训练数据）
方案 A  ←──（图遍历推理更新假设）────   方案 B（方案B的推理结果写入 Working Memory）

建议演进路线：
  Phase 3：方案 A（认知记忆，解决"知识主权"问题）
  Phase 4：方案 A + B（叠加因果图，解决"诊断路径最优化"问题）
  Phase 5：方案 A + B + C（叠加策略蒸馏，解决"速度和精度"问题）
```
