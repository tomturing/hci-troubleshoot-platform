## 一、核心洞察：知识的本质是因果结构，不是文本片段

### 1.1 RAG 的根本性缺陷

RAG 存储的是文本的**语义相似度**关系：
```
"VM启动失败" ─── [0.89 相似] ─── chunk: "CPU不足会导致VM无法启动..."
```

这只是**相关性**（A 和 B 经常一起出现），不是**因果性**（A 导致了 B）。

当 AI 用相关性做推理时，会得出这样的错误逻辑：
```
用户: "VM起不来"
AI 检索: top-3 都是 CPU 相关案例（因为 CPU 案例最多）
AI 输出: "建议检查 CPU 使用率"
实际: 该用户的问题是 NFS 路径漂移（低频）
结果: 错误引导，浪费排障时间
```

### 1.2 因果图的根本性优势

因果图存储的是**机制关系**：
```
症状节点: "VM启动失败"
    ├──[触发条件: 存储扩容后, 权重 0.31]──→ 工具: acli storage.check_mount()
    │       ↓ 输出模式: REMOUNTING
    │       ──[支持, 权重 0.89]──→ 根因: "NFS路径漂移"
    │
    ├──[触发条件: 无特殊操作, 权重 0.42]──→ 工具: acli cluster.cpu_usage()
    │       ↓ 输出模式: >90%
    │       ──[支持, 权重 0.91]──→ 根因: "CPU overcommit"
    │
    └──[触发条件: 升级后, 权重 0.18]──────→ 工具: acli license.check()
            ↓ 输出模式: EXPIRED
            ──[支持, 权重 0.95]──→ 根因: "许可证过期"
```

**核心差别**：因果图让 AI 知道的不是"什么文档相似"，而是"触发条件 → 工具 → 输出 → 根因"的完整因果链。

## 二、因果图数据模型

### 2.1 图的节点类型

```
NodeType:
  SYMPTOM（症状）    — 用户可观察的现象，如 "VM启动失败"
  TRIGGER（触发条件）— 症状出现前的操作，如 "存储扩容后"
  TOOL（工具）       — 可执行的诊断动作，如 "acli storage.check_mount()"
  OBSERVATION（观测模式）— 工具输出中的关键模式，如 "STATUS: REMOUNTING"
  HYPOTHESIS（假设） — 可能的根因，如 "NFS路径漂移"
  RESOLUTION（解决方案）— 修复动作，如 "重启存储客户端"
  BACKGROUND（背景知识）— 机制解释，如 "NFS客户端在卷重挂载期间会断开现有连接"
```

### 2.2 图的边类型

```
EdgeType:
  SUGGESTS（暗示）   — 症状 → 工具   : "VM启动失败 暗示 应检查存储状态"
  TRIGGERS（触发）   — 触发条件 → 假设: "存储扩容 触发 路径漂移风险"
  SUPPORTS（支持）   — 观测模式 → 假设: "REMOUNTING 支持 NFS路径漂移假设"
  REFUTES（反驳）    — 观测模式 → 假设: "存储状态正常 反驳 存储根因假设"
  RESOLVES（解决）   — 解决方案 → 假设: "重启存储客户端 解决 NFS路径漂移"
  EXPLAINS（解释）   — 背景知识 → 假设: "NFS机制 解释 路径漂移如何发生"
  NEXT_STEP（下一步）— 工具 → 工具  : "存储检查正常后 → 下一步检查CPU"
```

### 2.3 图的边权重来源

```
每条边的权重 = f(案例频率, 解决成功率, 知识源权威度)

案例频率（7000案例统计）:
  SYMPTOM → TRIGGER 边：在 VM启动失败 案例中，多少比例有"存储扩容"触发条件？
  答: 31/200 = 0.155 → 初始边权重 0.155

解决成功率（反馈更新）:
  该路径被 AI 使用后，解决了问题的比例？
  答: 28/31 = 0.90 → 成功率权重乘数 ×0.90

知识源权威度:
  来源于 docx 权威手册 → 权重乘数 ×1.5
  来源于人工确认案例   → 权重乘数 ×1.2
  来源于未验证案例     → 权重乘数 ×0.8

最终边权重 = 0.155 × 0.90 × 1.2（假设人工确认）= 0.167
```

## 三、因果图的构建流程

### 3.1 从 7000 案例到因果图（三步提取）

```
原始案例
  ↓
Step 1: 实体识别（LLM + NER）
─────────────────────────────
输入: 一个案例的文本
输出: {症状节点, 触发条件, 操作步骤, 工具名, 根因, 解决方案}

示例:
  "客户做存储扩容后，3台VM无法启动，报错'磁盘读取失败'。
   执行 acli storage.list_volumes 发现卷状态为 REMOUNTING。
   执行 acli storage.remount_check 等待重挂载完成后，VM 恢复正常。"

提取结果:
  症状: VM启动失败, 错误模式: "磁盘读取失败"
  触发条件: 存储扩容
  工具1: acli.storage.list_volumes → 观测: STATUS=REMOUNTING
  工具2: acli.storage.remount_check → 观测: COMPLETED
  根因: NFS路径漂移（重挂载期间）
  解决方案: 等待重挂载 + remount_check 确认

  ↓
Step 2: 关系抽取（LLM 结构化）
─────────────────────────────
将提取的实体组装为图的边:
  (VM启动失败) -[SUGGESTS, weight += 1]→ (acli.storage.list_volumes)
  (存储扩容) -[TRIGGERS, weight += 1]→ (NFS路径漂移)
  (STATUS=REMOUNTING) -[SUPPORTS, weight += 1]→ (NFS路径漂移)
  (acli.storage.remount_check) -[RESOLVES, weight += 1]→ (NFS路径漂移)

  ↓
Step 3: 图权重归一化 + 冲突解决
─────────────────────────────
对 7000 案例重复以上过程，边权重累积计数后归一化。
冲突规则（docx 权威源 > 案例，高质量案例 > 低质量案例）
输出: 完整因果图（~5000 节点，~20000 边，估算）
```

### 3.2 因果图的存储设计

```sql
-- 图节点表
CREATE TABLE causal_nodes (
    id          VARCHAR(64) PRIMARY KEY,    -- "node-symptom-vm-boot-fail"
    type        VARCHAR(20) NOT NULL,       -- SYMPTOM / TOOL / HYPOTHESIS / ...
    label       TEXT NOT NULL,              -- 可读标签
    domain      VARCHAR(16),               -- 所属故障域
    metadata    JSONB,                      -- 额外属性（工具参数、错误码等）
    embedding   VECTOR(1536),              -- 节点语义向量（用于模糊匹配）
    created_at  TIMESTAMPTZ DEFAULT NOW()
);

-- 图边表
CREATE TABLE causal_edges (
    id          BIGSERIAL PRIMARY KEY,
    from_node   VARCHAR(64) REFERENCES causal_nodes(id),
    to_node     VARCHAR(64) REFERENCES causal_nodes(id),
    edge_type   VARCHAR(20) NOT NULL,      -- SUGGESTS / SUPPORTS / REFUTES / ...
    weight      FLOAT NOT NULL DEFAULT 0.0,
    source_type VARCHAR(16),               -- docx / case / inferred
    case_count  INT DEFAULT 0,             -- 有多少案例支持这条边
    success_rate FLOAT,                    -- 使用这条诊断路径的成功率
    metadata    JSONB,
    -- 触发条件（可选，只有部分边有）
    trigger_condition TEXT,               -- "存储扩容后" / "版本升级后"
    version_range JSONB,                  -- 版本适用范围
    updated_at  TIMESTAMPTZ DEFAULT NOW()
);

-- 索引: 按 from_node + edge_type 快速遍历
CREATE INDEX idx_causal_edges_from ON causal_edges (from_node, edge_type);
-- pgvector: 症状节点的模糊匹配
CREATE INDEX idx_causal_nodes_embedding ON causal_nodes USING hnsw (embedding vector_cosine_ops);
```

## 四、图遍历推理算法（MCTS 变体）

### 4.1 最大信息增益工具选择

诊断的核心决策问题是：**在当前状态下，调用哪个工具能最快缩小根因假设空间？**

这等价于信息论中的**最大信息增益（Maximum Information Gain）**问题，也是 MCTS 在诊断领域的直接应用。

```python
def select_next_diagnostic_tool(
    current_state: DiagnosticState,
    causal_graph: CausalGraph
) -> DiagnosticTool:
    """
    在因果图中，从当前确认的症状节点出发，
    选择能最大化缩小假设空间的下一个工具节点。
    
    算法：最大信息增益树搜索
    """
    # 当前活跃假设（尚未被确认或排除）
    active_hypotheses = current_state.active_hypotheses
    
    # 找出所有可执行的诊断工具（当前状态可到达的工具节点）
    candidate_tools = causal_graph.get_reachable_tools(
        from_nodes=current_state.confirmed_symptoms,
        exclude_called=current_state.called_tools
    )
    
    best_tool = None
    best_ig = -1.0
    
    for tool in candidate_tools:
        # 计算调用此工具的信息增益
        ig = information_gain(tool, active_hypotheses, causal_graph)
        # ig = Σ P(output_pattern) × Σ [变化的假设置信度]
        
        # 加权：考虑触发条件匹配度
        trigger_bonus = trigger_condition_match(tool, current_state.context)
        
        # 考虑风险：Level 1 工具优先，Level 2+ 需要用户授权
        risk_penalty = tool.risk_level * 0.1
        
        score = ig * trigger_bonus - risk_penalty
        if score > best_ig:
            best_ig = score
            best_tool = tool
    
    return best_tool


def information_gain(
    tool: DiagnosticTool,
    hypotheses: list[Hypothesis],
    graph: CausalGraph
) -> float:
    """
    调用某工具可能的信息增益：
    看该工具的各种输出模式分别支持/反驳哪些假设，
    加权平均计算假设集合的熵减少量。
    """
    output_patterns = graph.get_tool_output_patterns(tool)
    total_ig = 0.0
    
    for pattern in output_patterns:
        p_pattern = pattern.historical_frequency  # 历史上出现此输出的概率
        delta_entropy = compute_entropy_change(
            hypotheses=hypotheses,
            supports=graph.get_supported_hypotheses(pattern),
            refutes=graph.get_refuted_hypotheses(pattern)
        )
        total_ig += p_pattern * delta_entropy
    
    return total_ig
```

### 4.2 图遍历推理完整流程

```
初始状态:
  confirmed_symptoms: ["VM启动失败"]
  context: {"trigger": "存储扩容后", "affected_scope": "全部VM"}
  active_hypotheses: 所有 VM启动失败 的可能根因（来自图的邻居）
        ├─ H1: NFS路径漂移 (prior: 0.31, trigger_bonus: +0.3 → 0.61)
        ├─ H2: CPU overcommit (prior: 0.42, trigger_bonus: -0.2 → 0.22)
        ├─ H3: 许可证过期 (prior: 0.18, trigger_bonus: -0.1 → 0.08)
        └─ H4: 存储配额 (prior: 0.09, trigger_bonus: +0.2 → 0.29)

Step 1: select_next_diagnostic_tool()
  候选工具:
    T1: acli storage.list_volumes  — 可区分 H1/H4，IG = 0.52
    T2: acli cluster.cpu_usage    — 可区分 H2，IG = 0.21
    T3: acli license.check        — 可区分 H3，IG = 0.07
  
  选择: T1（最高 IG）
  → "请允许我查看存储卷状态（只读操作）"（Level 1，提示后自动）

Step 2: 执行工具，获取输出
  acli storage.list_volumes → 输出包含 "STATUS: REMOUNTING"
  
  图匹配: OBSERVATION("STATUS: REMOUNTING") 
          -[SUPPORTS weight 0.89]→ H1: NFS路径漂移
          -[REFUTES  weight 0.85]→ H4: 存储配额

Step 3: 更新假设置信度
  H1: 0.61 × 0.89 → 正则化后: 0.87 ✅（高度确认）
  H4: 0.29 × (1-0.85) → 正则化后: 0.07 ❌（基本排除）
  H2: 0.22 → 不变 → 0.08（因 H1 概率升高而相对下降）
  H3: 0.08 → 不变 → 0.05

Step 4: 置信度 > 0.80 → 根因确认
  根因: NFS路径漂移（置信度 0.87）
  
  图查询: RESOLVES 边从 NFS路径漂移 出发
  → 解决方案节点: acli storage.remount_check + 等待挂载
  → 背景解释节点: "NFS客户端在新卷挂载期间临时断开已有连接，属正常行为，无需担心"

输出结果（有根据 + 有命令 + 有解释 + 有预期）:
  "确认根因：存储扩容触发NFS卷重新挂载，VM等待存储恢复后会自动恢复。
   临时方案：等待约5分钟，监控挂载进度：
             acli storage.remount_check（无风险，只读）
   如10分钟未恢复，执行强制重挂载：acli storage.force_remount（需确认）"
```

## 五、R + TC + K 协同机制（与方案 A 的对比）

| 维度 | 方案 A（认知记忆） | 方案 B（因果图） |
|------|-------------------|----------------|
| **R 推理层** | LLM 自由推理，参考召回的案例 | LLM + 图遍历算法共同决策 |
| **TC 工具调用** | LLM 自主决定调哪个工具 | 图算法计算最大信息增益工具 |
| **K 知识层** | 情节记忆（案例的完整场景） | 因果图（案例的结构性关系） |
| **推理可解释性** | 中（LLM 可能无法说清为何选此工具） | 高（每步选择都有 IG 分数可追溯） |
| **新场景适应** | 好（LLM 可用机制知识推理） | 差（图中无节点时无法遍历）|
| **最优诊断路径** | 不保证（LLM 可能走弯路）| 保证（算法选最大 IG 工具）|

**结论**：方案 B 的推理是**确定性算法辅助 LLM**，方案 A 的推理是**纯 LLM**。两者互补：方案 A 处理新场景，方案 B 优化已知场景的诊断效率。

## 六、从 7000 案例构建因果图的质量保证

### 6.1 自动验证机制

```python
class CausalGraphValidator:
    """构建完成后对图进行一致性验证"""
    
    def check_reflexive_loops(self, graph):
        """检测：A → B → A 的循环（不合逻辑的因果环）"""
    
    def check_docx_consistency(self, graph, docx_atoms):
        """验证：图中的高权重边是否与 docx 手册一致"""
    
    def check_orphan_nodes(self, graph):
        """检测：没有入边或出边的孤立节点（可能是提取错误）"""
    
    def cross_validate_with_cases(self, graph, held_out_cases):
        """
        留出验证：用 500 案例构图，100 案例验证。
        验证指标：图遍历正确诊断率
        目标：>70%（Phase 3）→ >85%（Phase 4）
        """
```

### 6.2 冲突处理规则

```
边冲突（两条属性相同的边权重矛盾）:
  规则1: docx 手册来源的边 > 案例来源的边
  规则2: 高质量案例（quality_score > 70）来源的边 > 低质量
  规则3: 新版本案例 > 旧版本案例（HCI 版本演进）
  规则4: 成功率 0% 的边标记为 DEPRECATED，不用于推理

节点歧义（同一概念有多种表述）:
  使用 BGE-M3 语义相似度 > 0.95 → 合并为同一节点
  保留所有别名（用于模糊匹配入口）
```

## 九、优势、劣势与适用边界

### 优势

| 优势 | 说明 |
|------|------|
| **最优诊断路径** | 算法保证每步选效率最高的工具，人类专家也难以超越 |
| **高可解释性** | 每条诊断路径都对应图中的边，决策过程完全透明 |
| **触发条件利用** | 用户提供的"存储扩容后"等触发条件直接参与图遍历，精准定位 |
| **持续优化** | success_rate 自动更新，能持续改进诊断效率 |
| **错误码精确匹配** | 错误码节点连接到根因，精确匹配优先于语义检索 |

### 劣势与限制

| 劣势 | 严重程度 | 缓解方案 |
|------|---------|---------|
| 构建成本高（需结构化抽取） | 🟡 中 | 方案A的EpisodicMemoryBuilder已包含此步骤 |
| 新故障类型图中无节点 | 🔴 高 | 降级到方案A（LLM+机制知识推理）|
| 图维护成本（版本更新时） | 🟡 中 | 版本标记 + 自动 deprecate 旧边 |
| MCTS 在深层复杂图中计算量大 | 🟡 中 | 搜索深度限制（max_depth=5），加剪枝 |
