# 知识工程方案 C：工具合成与策略蒸馏架构（Tool Synthesis + Policy Distillation）

**版本**: v1.0  
**日期**: 2026-03-23  
**定位**: 前沿方案 C，推荐作为 Phase 5 性能优化层  
**学术根基**: 知识蒸馏（Hinton 2015）+ AlphaCode2 专家混合 + Toolformer（Meta 2023）  
**适读对象**: 架构师、ML 工程师

## 一、核心思想：把知识"编译"进工具，而不是检索

### 1.1 一个关键洞察

方案 A 和 B 都在解答同一个问题：**LLM 如何更好地使用知识**。

方案 C 提出了一个完全不同的问题：**能否把知识从 LLM 的工作负担中卸载，让工具自己"携带"知识？**

类比：
```
方案 A/B 的 LLM: 医学院毕业的全科医生，看病前需要翻阅病例和教材
方案 C 的系统:   全科医生 + 一批专科护士（每个护士专门处理特定疾病）
                 护士看到症状直接报告结论，医生只做最终综合判断
```

### 1.2 两个技术支柱

**支柱1：工具合成（Tool Synthesis）**  
不是"LLM 调用工具"，而是"为特定诊断场景自动生成专用工具"。  
从 7000 案例中提取高频诊断模式，将每个模式固化为一个可执行的"诊断工具"。

**支柱2：策略蒸馏（Policy Distillation）**  
把 LLM（大模型）的诊断决策知识，蒸馏到轻量专家模型（小模型）中。  
小模型处理 80% 的标准场景，LLM 只处理 20% 的复杂/新型场景。

## 三、工具合成（Tool Synthesis）

### 3.1 什么是"合成工具"

合成工具不是手写的函数，而是**从案例中自动生成的诊断脚本**。

```
传统工具（手写）:
  def acli_vm_status(vm_id: str) -> dict:
      return scp_adapter.call("acli vm.get " + vm_id)

合成工具（自动生成）:
  DiagToolKit "vm-boot-storage-check"（从 200 个案例自动生成）:
    步骤1: 获取失败VM的错误任务   → acli vm.get_task_error(vm_id)
    步骤2: 检查存储挂载状态       → acli storage.list_volumes()
    步骤3: 如果有 REMOUNTING → 等待状态
           如果有 ERROR      → 上升告警
           如果全部 ONLINE   → 排除存储假设，转向 CPU 检查
    步骤4: 检查宿主机磁盘空间     → acli system.df()
    输出: {root_cause_candidate, confidence, next_action}
```

**关键点**：合成工具封装了整个**诊断子流程**，而不是单个命令。LLM 调用的是"诊断VM启动失败（存储相关）"，而不是"查存储状态"。

### 3.2 工具合成算法

```python
class ToolSynthesizer:
    """
    从案例聚类中自动生成诊断工具包（DiagToolKit）。
    每个 DiagToolKit 对应因果图中的一条高频诊断路径。
    """
    
    def synthesize_from_cases(
        self,
        case_cluster: list[StructuredCase],    # 症状相似的案例集合
        min_frequency: int = 20               # 最少 20 个案例才合成工具
    ) -> DiagToolKit:
        
        # Step 1: 提取高频工具调用序列（序列模式挖掘，类似 GSP 算法）
        frequent_sequences = mine_tool_sequences(
            cases=case_cluster,
            min_support=0.6    # 60% 以上案例使用相同工具序列
        )
        
        # Step 2: 提取工具输出 → 诊断结论的映射
        output_rules = extract_output_pattern_rules(
            cases=case_cluster,
            tool_sequences=frequent_sequences
        )
        
        # Step 3: 生成 DiagToolKit（Python 代码 + 元数据）
        # 注意：可以用 LLM 生成代码框架，然后人工 review
        toolkit_code = self.generate_toolkit_code(
            sequences=frequent_sequences,
            rules=output_rules,
            case_count=len(case_cluster)
        )
        
        return DiagToolKit(
            name=f"diag-{derive_toolkit_name(case_cluster)}",
            code=toolkit_code,
            trigger_signatures=extract_trigger_signatures(case_cluster),
            applicable_domains=extract_domains(case_cluster),
            confidence_threshold=0.75,
            source_case_count=len(case_cluster)
        )
```

### 3.3 工具合成的触发规则

```
触发条件（满足任一即合成）：
  1. 同一症状 + 同一工具序列 出现 ≥ 20 次
  2. 某条因果图路径的 cumulative_weight ≥ 10.0
  3. 专家手工标注（高优先级场景无论频次都合成）

不合成的情况：
  1. 频次 < 20（样本不足，合成的工具不可靠）
  2. 前置条件太复杂（需要 ≥ 5 个信息才能触发，不如用 LLM 推理）
```

## 四、策略蒸馏（Policy Distillation）

### 4.1 什么是诊断策略，为何要蒸馏

**大模型（GLM-4）做诊断**：
- 推理质量高，能处理复杂/新型场景
- 延迟高（1-3 秒每次推理）
- 成本高（每次调用 API 计费）
- 对于 80% 的标准场景：大材小用

**策略蒸馏的目标**：提炼 GLM-4 在标准场景中的诊断决策模式，训练成小模型，用小模型处理标准场景，GLM-4 处理剩余复杂场景。

```
训练过程:
  教师（GLM-4）← 7000 案例 + 手册 → 学生（BGE-M3 微调）
  
  教师动作：对每个案例输入，GLM-4 输出 {first_tool, diagnosis, confidence}
  学生目标：用 3-5 倍小的模型，在 80% 场景中复现教师的决策
```

### 4.2 蒸馏数据集构建

```python
class DistillationDatasetBuilder:
    """
    用 GLM-4 对历史案例打标，生成蒸馏训练集
    """
    
    def build_dataset(self, cases: list[StructuredCase]) -> list[TrainingSample]:
        dataset = []
        for case in cases:
            # Step 1: 构造输入（只用前 200 字 + 关键字段）
            input_text = self.extract_routing_features(case)
            
            # Step 2: 用 GLM-4 推理：如果我是专家，第一步该做什么？
            teacher_output = glm4_client.infer(
                prompt=DISTILLATION_TEACHER_PROMPT.format(case=input_text),
                schema=DiagnosisOutput
            )
            
            # Step 3: 质量过滤（剔除 GLM-4 幻觉样本）
            if self.is_high_quality(teacher_output, case):
                dataset.append(TrainingSample(
                    input=input_text,
                    label=teacher_output
                ))
        
        return dataset

# 标签结构（学生模型要预测的）
@dataclass
class DiagnosisOutput:
    domain: str              # 故障域
    sub_domain: str          # 子类别
    first_tool: str          # 第一步应该调用的工具
    tool_rationale: str      # 为什么选这个工具（用于蒸馏损失计算）
    est_root_cause: str      # 估计的根因（供快速确认，不强制采纳）
    confidence: float        # 整体置信度
```

### 4.3 小模型架构

```
推荐方案: BGE-M3 微调（中文优化，支持 GPU/CPU 推理，开源）
模型大小: 570M parameters（vs GLM-4 的 ~130B）
推理延迟: ~50ms（vs GLM-4 的 ~1500ms）

微调任务类型: 多任务分类
  任务1: domain 分类（5类）— 准确率目标 >95%
  任务2: sub_domain 分类（198类）— 准确率目标 >85%
  任务3: first_tool 分类（~30个工具）— 准确率目标 >80%
  任务4: est_root_cause 分类（~500个根因）— 准确率目标 >70%

训练数据量:
  7000 案例 × 质量过滤（保留 60%）≈ 4200 条训练样本
  Train/Val/Test = 3360 / 420 / 420
```

## 五、R + TC + K 协同机制

```
K（知识）= 编译进小模型的诊断策略 + DiagToolKit 内嵌的执行规则
TC（工具调用）= 小模型路由 → DiagToolKit → SCP 实际执行
R（推理）= GLM-4 做最终综合判断，处理异常和例外

协同流程:
                用户输入
                    │
                    ▼
    ┌───────────────────────────────┐
    │  K+TC: 小模型 (<100ms)        │
    │  路由 + 预判根因 + 推荐工具   │
    └───────────┬───────────────────┘
    置信度 ≥ 0.8│          置信度 < 0.8
                ▼                 ▼
    ┌───────────────────┐    ┌──────────────────────┐
    │  TC: DiagToolKit  │    │  R+TC+K（方案A+B）    │
    │  快速执行诊断步骤  │    │  GLM-4 深度推理       │
    └────────┬──────────┘    └──────────────────────┘
             │
             ▼
    ┌────────────────────────────┐
    │  R: GLM-4 综合判断（轻量）  │
    │  - 验证工具输出与预期一致   │
    │  - 生成用户解释             │
    │  - 处理例外情况             │
    └────────────────────────────┘
```

## 六、三方案的最优组合策略

方案 C 不是取代 A 和 B，而是在 A+B 基础上的**性能优化层**：

```
当 confidence ≥ 0.8（标准场景，约 70% 的工单）:
  使用方案 C（小模型路由 + DiagToolKit）
  典型场景：常见硬件告警、常见 VM 启动失败、常见存储 IO 慢
  延迟：~500ms 端到端（包含工具执行）
  成本：无 GLM-4 API 调用

当 0.6 ≤ confidence < 0.8（半标准场景，约 20% 的工单）:
  使用方案 B（因果图遍历 + GLM-4 轻量调用）
  典型场景：触发条件组合复杂，图中路径非单一
  延迟：~2s

当 confidence < 0.6（新型/复杂场景，约 10% 的工单）:
  使用方案 A（认知记忆 + GLM-4 深度推理）
  典型场景：新版本特有问题、多组件联合故障
  延迟：~5-10s（多轮工具调用）

自动降级机制:
  方案C失败（工具执行异常）→ 自动切换方案B
  方案B失败（图中无路径）  → 自动切换方案A
  方案A失败（LLM推理无结果）→ 转人工支持，并记录为新知识
```

## 九、优势、劣势与适用边界

### 优势

| 优势 | 指标 |
|------|------|
| **速度最快** | 标准场景端到端 <500ms，含工具执行 |
| **成本最低** | 70% 工单无需调用 GLM-4 API |
| **确定性强** | DiagToolKit 的执行规则是代码，不是 LLM 生成 |
| **可持续优化** | 小模型可随新案例积累定期重新训练 |

### 劣势与限制

| 劣势 | 严重程度 | 缓解方案 |
|------|---------|---------|
| 需要足够案例才能训练（冷启动） | 🔴 高 | Phase 5 依赖 Phase 3/4 先完成 |
| 小模型覆盖不到新型故障 | 🟡 中 | 自动降级到 LLM（方案A） |
| 蒸馏过程引入 GLM-4 的错误 | 🟡 中 | 人工抽检 + 测试集验证 |
| 工具版本管理（HCI升级后需更新） | 🟡 中 | DiagToolKit 版本标记 + 测试套件 |

### 适用边界

- ✅ 高频、标准化故障场景（常见告警、常见资源不足类问题）
- ✅ 对响应速度要求高的场景（SLA 要求 <1s 响应）
- ✅ 需要降低 API 成本的场景（并发量大时）
- ❌ 新型故障（需方案A+B处理）
- ❌ 跨多组件联合故障（需GLM-4复杂推理）
