## ⚡ 一、核心架构原则（最高优先级，所有设计决策的基础）

> **此原则是在 2026-03-23 通过对比 Claude Code Skill 技术文档与项目 SOP Skill 实现后得出的关键洞察，
> 必须在所有涉及 SOP / KB / 案例库 / 知识注入的设计中优先遵守。**

### 所有知识注入的本质是「条件触发的上下文注入」

无论是 SOP 技能文件、RAG 案例 chunks、工具文档，还是 Claude Code 的 Skill——
**在 LLM 眼中，它们具有完全相同的技术本质**：

```
触发条件满足
    ↓
将预先编写/检索到的结构化内容注入 LLM 的上下文窗口（Context Window）
    ↓
LLM 依据注入内容进行推理和生成
```

它们的差异**不是本质差异，而是实现质量的差异**：

| 维度 | 劣质实现 | 优质实现 |
|---|---|---|
| **触发机制** | 精确关键字字符串扫描（当前 SopMatcher） | Embedding 语义相似度匹配（Claude Code Skill 的 description 路由） |
| **注入粒度** | 整篇SOP文档 / 512-token 段落切块 | 知识原子（有类型标签 + 触发条件的最小语义单元） |
| **触发时机** | 用户第一句话到来时（S0，信息最模糊时） | 问题域界定后（S1），或 LLM 推理后自主决定（方案A） |
| **动态性** | 纯静态文本注入 | 支持运行时数据填充（`!``command``预处理） |

### 因果链：路由质量 → 注入质量 → LLM推理质量

```
触发机制质量
    └→ 决定"注入了什么"的精准度
            └→ 决定 LLM 在 context window 中"看到什么"
                    └→ 直接决定推理质量和回复质量

关键字匹配（最弱）< 精确分类过滤 < Embedding语义路由（最强）
```

### 直接推论（影响所有下游设计）

1. **SopMatcher 的关键字路由是已知最差的触发机制**，"主机离线了" 无法命中 "节点降级" SOP——用户语言与SOP关键字之间的gap是永久性的，无法通过增加关键字解决
2. **SOP 与 RAG 不应该是两套系统**，它们是同一件事（条件触发的上下文注入）的两种粒度，应统一为语义路由 + 知识原子
3. **检索时机必须在 S1（范围界定后）而不是 S0（症状模糊期）**，S0 用模糊 query 检索相当于用最差质量的输入触发整个注入链
4. **注入内容的类型标签是必须的**：diagnostic_step / fix_action / background 注入后 LLM 的使用方式完全不同，混注 = 降低所有内容的可用性

## ⚡ 二、知识源深度对比分析（真实数据验证，2026-03-23）

> 以真实原始数据为证据，验证并延伸核心架构原则，确立「双轨知识架构」的设计基础。
> 所有后续 SOP / KB 存储设计、检索策略、Prompt 组装逻辑必须以本节结论为准。

### SOP（程序性知识）vs KB案例（陈述性知识）：真实样本对比

**样本 A — SOP 原始数据**（来源：`data-pipeline/sop_skills/vm_power_failure/chapters/sub_内存不足.md`）

```markdown
## 有启动虚拟机失败任务 → 内存不足

### 前置检查
获取【启动虚拟机】失败任务的【描述】信息：
  acli task get -v ${vmid} -t ${YYYY-mm-dd} -k '启动虚拟机' -s 'failed'
关键字 → 章节：CPU不足 / 内存不足 / 内部异常 / 序列号过期 / ...（共 20+ 分支）

### 内存不足
#### 判断方法
  acli system memory info | grep MemAvailable
  - MemAvailable < 2G → 内存耗尽
#### 解决方案
  关闭部分业务虚拟机 / 在线迁移虚拟机到其他主机
```

**样本 B — KB案例原始数据**（来源：`data-pipeline/cases_raw/cases.jsonl`，案例ID: 33780）

```
标题：虚拟机启动失败，没有主机能够启动这台虚拟机，错误码：0x0C000005
【根因】因为编辑虚拟机重要性导致虚拟机内存分组切换，cgroup bug 导致内存计算错误
【解决方案】
  1. 临时：后台使用 qm start vmid 命令将虚拟机先以非大页拉起来
  2. 重启主机释放空间
  3. 升级 6.12.0 解决（补丁待出）TD2025072800477
```

### 七维对比分析

| 维度 | SOP（程序性知识） | KB案例（陈述性知识） | 对 AI Agent 的意义 |
|------|-----------------|---------------------|-------------------|
| **1. 结构化程度** | 高：树状决策结构，前置检查→20+分支→判断条件→执行步骤 | 中：固定字段，内容是叙事性自然语言 | SOP 结构可被 Agent 直接"遍历"；案例需要 Agent 提炼 |
| **2. 触发条件精确性** | 极高：精确报错字符串 + 数值阈值（Percpu>5G, VmallocUsed>10G） | 中：有错误码（0x0C000005），但缺乏判断阈值 | SOP 提供"何时做决策"；案例提供"曾经发生了什么" |
| **3. 命令可执行性** | 高：完整 acli 命令 + 读哪个字段 + 判断阈值 | 低：描述了"查看"而非命令，无完整命令语法 | SOP 可直接驱动 tool_call；案例只能作参考 |
| **4. 根因明确性** | 低：给解决方案，不解释底层 why | **极高：明确记录了 cgroup bug 是根因** | 案例的根因是假设生成的黄金来源；SOP 不记录此类归因 |
| **5. 覆盖范围** | 宽：一个章节覆盖多种失败路径 | 窄：每个 case 是唯一的失败场景 | SOP 是泛化知识；案例是特化实例 |
| **6. 时效性** | 相对稳定：排查方法不频繁过期 | 强时效：版本号、补丁号、TD跟踪号都是时点信息 | 案例必须有版本元数据才能安全使用 |
| **7. AI 执行友好性** | **最高**：Agent 可按树结构执行、对阈值判断、调用命令 | 低：叙事步骤，Agent 看到"查看"后不知执行什么 | SOP 是程序，案例是证言 |

### 核心结论：两类知识承担不同的认知职责

```
SOP（程序性知识）= "如何做"
  → 适合放进 Agent 的「动作执行上下文」
  → 类比：机械师手册 — 拿到故障代码直接按步骤走

KB案例（陈述性知识）= "曾经发生过什么"
  → 适合放进 Agent 的「相似案例记忆库」
  → 类比：医院病历 — 参考历史病例辅助诊断，不能直接复制治疗方案
```

**它们不是竞争关系，而是两个不同的认知层次：**

```
用户报故障
    ├─[SOP路由] 语义路由 → 命中SOP → 注入SOP知识原子 → Agent 走程序
    │                       ↓ 未命中
    └─[KB检索] 语义相似度 → 找历史相似案例 → 注入案例根因 → Agent 形成假设
                             ↓ 未命中
                    [机制推理兜底] → 标注【机制推理】
```

### 双轨知识架构的设计推论

**SOP 轨道设计原则：**
1. **存储单元**：知识原子（knowledge_atom），不做 512-token 滑块，按决策树节点分割
2. **每个原子包含**：序列路径、知识类型（diagnostic_step / decision_gate / threshold / fix_action）、可执行命令、判断条件
3. **路由机制**：语义匹配（embedding 相似度），而非关键字字符串扫描
4. **格式标签**：注入 Prompt 时标注 `【SOP 排障流程】`，LLM 知道此内容应被严格执行

**KB 轨道设计原则：**
1. **存储单元**：完整案例文档（不拆分），向量化 `problem_desc + root_cause` 字段
2. **路由机制**：BM25 + 向量混合检索，主要匹配 `problem_desc` 字段
3. **Agent 行为**：从 `root_cause` 提取假设 → 用 `solution` 作参考（非配方）→ 结合当前环境验证假设
4. **格式标签**：注入 Prompt 时标注 `【历史案例参考】`，LLM 知道此内容是证据而非程序

### 三级 Fallback 触发规则

```
Priority 1: SOP 命中
  → system_prompt 包含 [SOP排障流程] 段
  → AI 走 SOP 程序执行路径

Priority 2: SOP 未命中，KB 案例命中（≥1 条相似度阈值以上）
  → system_prompt 包含 [历史案例参考] 段
  → AI 用案例 root_cause 生成假设，方向性排查

Priority 3: 双轨均未命中
  → system_prompt 包含 [机制推理模式] 段
  → AI 基于领域训练知识推理，所有结论标注【机制推理】
```

## 三、Claude Agent 工作原理（认知基础）

> 理解 Claude 是如何工作的，才能理解本项目与它之间的差距在哪里。

### 3.1 Claude Agent 的核心循环：Think → Act → Observe

Claude 不是"问一句、答一句"的问答机器，而是一个**持续运转的 Agent 循环**：

```
用户输入
   ↓
[思考] 理解意图 + 制定策略
   ↓
[行动] 调用工具（搜代码、读文件、跑命令…）
   ↓
[观察] 分析工具返回结果
   ↓
[判断] 任务完成了吗？
   ├── 否 → 继续思考→行动→观察
   └── 是 → 向用户报告结果
```

### 3.2 Claude 的全程工作方式（排障场景）

**第一步：接收模糊输入，主动澄清**

用户说："用户保存工单时，有时候会报 500 错误"

Claude 面临信息不足时会：
1. **识别关键缺失**：什么条件下触发？哪个服务？有没有报错日志？
2. **主动提问**（不猜测）
> 原则：不盲目行动，先对齐目标。

**第二步：自主侦察（并行工具调用）**

用户回答后，Claude 立刻并行展开调查：
```
并行搜索：
├── grep_search("create_case" 的路由和 handler)
├── grep_search(日志中的 trace_id 或 500 错误关键词)
└── read_file(case-service 的 case_router.py)
```
**Claude 自己去找证据，不需要用户一条一条粘贴代码。**

**第三步：建立假设，深挖验证**

发现可疑点后**继续验证**，而不是直接下结论。

**第四步：与用户对话，确认修复方向**

Claude 发现问题后**不会直接改代码**，而是先汇报推理过程，确认方向后再执行修复并跑测试验证。

### 3.3 关键能力总结

| 阶段 | Claude 在做什么 |
|------|----------------|
| **理解意图** | 从模糊描述提取关键信息，识别缺失 |
| **主动提问** | 只问真正影响决策的问题，不问废话 |
| **自主侦察** | 并行调用工具，快速收敛证据 |
| **构建假设** | 从证据推理，而非猜测 |
| **交叉验证** | 对假设继续深挖，避免误判 |
| **透明报告** | 把推理过程和证据一起展示给用户 |
| **执行+验证** | 修复后自动跑测试确认 |

### 3.4 Claude 的知识来源：三种机制

Claude 本身不做 RAG——它的知识烘焙在模型权重里。但在 Agent 模式下，通过**工具调用**实现了等效甚至更强的效果：

| 来源 | 机制 | 例子 |
|------|------|------|
| **参数化知识** | 训练时压缩进权重 | Python 语法、框架 API、通用 debug 模式 |
| **上下文知识** | 工具读取后注入 context window | 读代码文件、日志、文档 |
| **外部 RAG** | 需要专门接入才有 | 企业私有知识库、文档库 |

> **Claude 之所以强大，不是因为存了答案，而是因为它有推理能力 + 工具调用 + 知识三者的协同。本项目要复刻的，正是这三者的组合。**

### 3.5 本项目的 RAG 定位

本项目的 RAG（KB Service）是给**最终用户**提供的领域知识增强：
- 故障描述 → 向量检索相似 SOP → 召回相关排障步骤 → AI 结合上下文给出建议
- **核心目标**：让 AI 具备"HCI 领域专家知识"，弥补通用 LLM 在 HCI 特定故障处理上的经验空白

## 四、Claude 式能力 vs 当前系统的本质差距

### 4.1 Claude 排障的本质：侦探式推理

Claude 处理故障排障时，本质上是一个**推理引擎**，知识是燃料，不是答案本身：

```
症状输入
  ↓
[形成假设]  "可能是 A、B、C 三种原因"
  ↓
[精准提问]  "请问…的状态是什么？"   ← 每个问题都在验证/排除一个假设
  ↓
[更新假设]  排除 C，剩余 A 和 B
  ↓
[再次提问]  "执行 xxx 命令看一下输出…"
  ↓
[收敛定位]  根因 = A
  ↓
[方案生成]  基于根因 + 环境 → 具体可执行步骤
```

> **知识服务于推理，而不是代替推理。**

### 4.2 当前系统的本质：智能检索 + 摘要系统

```
用户第一句话
  ↓
[检索触发]  关键字匹配 SOP + 向量检索 topK
  ↓
[注入 Prompt]  "参考资料：[检索结果]，请基于此回答"
  ↓
[LLM 生成]  将参考资料转述给用户
```

> **当前系统是一个文档检索 + 摘要系统，不是排障推理系统。**

### 4.3 六个本质差距

**差距 1：问题理解的深度**

| 维度 | Claude | 当前系统 |
|------|--------|---------|
| 对模糊输入的处理 | 主动拆解歧义，识别缺失信息，精准追问 | 拿模糊输入直接检索，召回噪声高 |
| 对技术术语的理解 | 语义理解（"虚拟机起不来" → 推导多种可能原因） | 字符串匹配 |
| 上下文积累 | 每轮对话都在缩小问题空间 | 每轮对话独立检索，不累积诊断进展 |

**差距 2：知识的角色定位**

```
Claude:      知识 → 提供 context → 辅助推理 → 生成方案（推理是核心）
当前系统:    知识 → 直接呈现给用户 → 用户自己理解（知识是答案本身）
```

**差距 3：不确定性的处理方式**

| 情形 | Claude | 当前系统 |
|------|--------|---------|
| 知识库有相关内容 | 结合环境信息推断，给出置信度 | 原文引用，不推断 |
| 知识库无相关内容 | "目前信息不足，我有几个可能方向，需要你提供…" | "知识库暂未收录，建议联系技术支持" |
| 症状有多种可能 | 主动列出假设树，按概率排序，逐一排查 | 召回多个案例并列展示，让用户自判 |

> **当前系统把"我不知道"和"你的问题无解"混为一谈——这是认知层面的根本错误。**

**差距 4：诊断过程 vs 答案检索**

真正的排障专家遵循固定的诊断流程：
```
① 确认故障现象（复现条件、影响范围、发生时间）
② 收集环境信息（版本、配置、日志）
③ 形成假设（Top 3 可能原因）
④ 设计验证步骤（最小代价先验证高概率假设）
⑤ 执行验证、收集结果
⑥ 更新假设、收敛根因
⑦ 方案设计（临时恢复 vs 根因修复）
```

> **当前系统跳过了 ①-⑥，直接到 ⑦——而且 ⑦ 是基于检索到的历史案例，不是基于用户当前环境。**

**差距 5：Prompt 哲学的根本分歧**

| | 哲学定位 | 行为表现 |
|---|---|---|
| **当前系统** | "知识库的代言人，只说库里有的" | 检索到什么展示什么，无内容则拒绝回答 |
| **Claude** | "推理者，知识库是参考，推理能力服务具体问题" | 以知识为燃料，用推理能力灵活应对 |

**差距 6：知识的粒度与形态**

| 知识形态 | Claude | 当前系统（设计） |
|---------|--------|----------------|
| **机制知识** | 知道"为什么"，能迁移到新情况 | 不存在，chunks 是案例片段 |
| **程序性知识** | 知道"排查顺序"，能动态调整 | 部分（SOP），但静态 |
| **元认知** | 知道自己不知道什么，知道何时追问 | 不存在，只有二值判断（有/无）|

### 4.4 一个关键认知误区

项目设计文档中有一句话揭示了核心误区：

> "精准优于召回：有就是有，没有就是没有，减少幻觉，杜绝胡说"

这个原则**保护了数据质量，但牺牲了推理能力**。

**"有就是有，没有就是没有"是图书馆的原则，不是专家的原则。**
专家在没有先例的情况下，依然能凭机制推理给出建议。

### 4.5 结论：问题不只是"数据没入库"

即使今天把 7000 条案例全部入库、KB Service 上线，**当前架构仍然无法达到"Claude 式体验"**，因为：

1. **没有诊断推理链** — Prompt 没有引导 LLM 进行假设→验证→收敛的过程
2. **检索时机错误** — 第一句话就检索，而不是"先问清楚问题再检索"
3. **知识形态缺失** — 案例 chunks 不等于机制知识，LLM 需要机制知识才能推理
4. **交互模式错误** — 系统设定自己是"回答者"，应该是"排障引导者"

## 五、目标架构：专业HCI智能排障Claude

> **本节记录一个关键认知修正：**
> 前四节的重构方案方向正确，但仍然不够彻底。
> 最终目标"再造一个专业HCI智能排障Claude"，要求的是 **Agent 架构**，而不是"更好的问答系统"。

### 5.1 对"再造专业HCI智能排障Claude"的深刻理解

Claude 之所以强大，不是因为它"存了很多答案"，而是因为它的架构是这样的：

```
LLM 是大脑，工具是它的手和眼
LLM 主动决定"下一步做什么" → 调用工具 → 解读结果 → 再决定 → 再调用
直到问题解决，才给用户最终答案
```

**当前系统（包括 Phase 0-2 重构）本质上仍是：**

```
用户发消息 → 平台预先组装 Prompt → LLM 生成一段话 → 等待下一条用户消息
```

这两个架构有根本区别：

| 维度 | 当前系统（含 Phase 0-2 重构） | 目标系统（专业HCI排障Claude）|
|------|------------------------------|----------------------------|
| **控制权** | 平台决定注入什么知识，LLM 被动接受 | LLM 主动决定调用什么工具 |
| **信息获取** | 一次性 RAG 注入，LLM 无法追加获取 | LLM 随时按需调用工具获取信息 |
| **推理模式** | 单轮生成（一条消息对应一次生成） | 多步推理（一条消息 → N次工具调用 → 最终回答）|
| **诊断能力** | 被动回答用户提供的信息 | 主动侦察系统状态 |
| **知识角色** | RAG 是"地基"，决定能不能回答 | RAG 是"工具之一"，与其他工具平等 |

> **与"当前重构方案（Phase 0-2）"的关系**：
> Phase 0-2 是 **过渡阶段**（改善体验，快速上线），目标架构是 **Phase 3 的核心任务**。
> 两者不冲突，但必须认清：Phase 0-2 的终点不是目标终点。

### 5.2 目标架构：ReAct 模式（Reasoning + Acting）

```
用户: "3台虚拟机突然起不来了"
                │
                ▼
    ┌───────────────────────┐
    │  LLM 推理（内部）      │
    │  "需要先了解VM当前状态"│
    └───────────┬───────────┘
                │ 工具调用
                ▼
    ┌───────────────────────┐     需要授权？
    │  get_vm_status(vm_id) │ ──→ [授权提示] ──→ 用户确认
    └───────────┬───────────┘
                │ 返回结果：VM ERROR, CPU overcommit
                ▼
    ┌───────────────────────┐
    │  LLM 推理（内部）      │
    │  "CPU问题，查集群资源" │
    └───────────┬───────────┘
                │ 工具调用（只读，自动执行）
                ▼
    ┌───────────────────────────┐
    │  get_cluster_cpu_usage()  │
    └───────────┬───────────────┘
                │ 返回结果：集群 CPU 95%
                ▼
    ┌───────────────────────────────────────┐
    │  LLM 推理（内部）                      │
    │  "根因确认：CPU资源耗尽，有3台竞争资源"│
    │  "检索相关案例和 SOP 获取操作建议"     │
    └───────────┬───────────────────────────┘
                │ 工具调用
                ▼
    ┌───────────────────────────┐
    │  search_knowledge(query)  │  ← RAG 是工具，不是地基
    └───────────┬───────────────┘
                │ 返回：相关 SOP 和案例
                ▼
    ┌───────────────────────────────────────┐
    │  LLM 生成最终回答（有根据、有步骤）    │
    │  临时恢复：迁移1台VM释放CPU            │
    │  根因修复：扩容集群或重新规划资源      │
    └───────────────────────────────────────┘
                │
                ▼
    用户收到：有根因、有方案、有具体命令的专家级回答
```

### 5.3 工具目录（Tool Catalog）

工具是 Agent 的能力边界，分三类：

#### 类型1：知识工具（Level 0，无需授权，静默执行）

| 工具 | 功能 |
|------|------|
| `search_knowledge(query, category?)` | RAG 语义检索 |
| `match_sop(query)` | SOP 精确匹配 |
| `get_hci_mechanism(topic)` | 查询 HCI 机制知识（待建）|

#### 类型2：诊断工具（Level 1，只读，提示后自动执行）

通过 SCP REST API 和 acli 安全容器执行，底层物理隔离，平台保障安全底线：

| 工具 | 功能 |
|------|------|
| `get_active_alerts()` | 获取当前活跃告警列表 |
| `get_failed_tasks()` | 获取失败任务列表 |
| `get_vm_status(vm_id)` | 查询虚拟机状态 |
| `get_cluster_health()` | 集群整体健康状态 |
| `acli_vm_list()` | 列出所有VM及状态 |
| `acli_host_list()` | 列出宿主机及资源使用 |
| `acli_vg_list()` | 列出存储卷组 |
| `search_logs(keyword, time_range)` | 日志关键字搜索 |

#### 类型3：操作工具（Level 2/3，有风险，强制用户授权）

| 工具 | 风险等级 |
|------|---------|
| `acli_vm_on(vm_id)` | Level 2（确认）|
| `acli_vm_off(vm_id)` | Level 2（确认）|
| `acli_vm_migrate(vm_id, host)` | Level 2（确认）|
| `modify_config(target, key, value)` | Level 3（强确认）|

#### 永久禁止工具（安全红线）

```
❌ 删除数据 / 格式化磁盘 / 删除 VM
❌ 修改认证配置（密码/证书/密钥）
❌ 关闭集群或宿主机
❌ 任何未经分析的批量操作
```

### 5.4 授权机制设计（风险分级策略）

```
┌──────────────────────────────────────────────────────────────┐
│                      授权分级策略                             │
│                                                              │
│  Level 0: 知识查询     → 静默执行，无需提示                   │
│           （search_knowledge / match_sop）                   │
│                                                              │
│  Level 1: 只读诊断     → 提示执行意图，3秒无操作自动执行       │
│           （get_vm_status / get_cluster_health ...）         │
│                                                              │
│  Level 2: 低风险操作   → 明确提示 + 确认按钮，不自动执行       │
│           （restart_vm / migrate_vm ...）                    │
│                                                              │
│  Level 3: 高风险操作   → 强确认 + 展示变更内容 + 输入确认码   │
│           （modify_config / restart_service ...）            │
│                                                              │
│  Level X: 禁止操作     → 无论如何都不执行，解释原因            │
└──────────────────────────────────────────────────────────────┘
```

会话开始时，用户可设置会话级授权策略（如"本次排障全程授权 Level 1 自动执行"），减少重复确认摩擦。

### 5.6 SCP 工具接入架构

**核心发现**：80% 以上的客户问题来源于两类——**告警确认**和**失败任务排查**。

**工具接入层：SCP（Service Control Platform）**

| 工具类型 | 接入方式 |
|---------|---------|
| 管理 REST API | 通过 SCP 提供（告警列表、任务列表、VM状态、集群健康等）|
| acli 命令 | 通过 SCP 封装后对外暴露（安全容器隔离，与 HCI host 彻底隔离）|
| 监控数据 | ❌ 暂无接口（Prometheus/Grafana 暂无直接调用方式）|

**acli 安全模型（关键）：**
```
acli 类比 VMware 的 esxcli，但有本质区别：
  esxcli: 直接在 ESXi host 上执行，权限高，风险高
  acli（本项目）: 封装在安全容器中，与 HCI host 彻底物理隔离
               只提供经过审计的安全命令子集
               → 工具调用安全底线由平台保证，AI 层无需重复防御
```

**两个诊断入口（基于 SCP 告警/任务的关键洞察）：**

```
入口A（用户文字描述）：走完整诊断状态机 S0→S6（问题澄清 → 假设形成 → 验证收敛）
入口B（告警/任务结构化数据）：直接跳到 S2/S3，效率大幅提升

示例：
  传统：用户"虚拟机起不来" → AI追问分类→ 多轮澄清
  SCP驱动：告警{type:"VM_BOOT_FAIL", error:"ECPU_OVERCOMMIT"} → 分类已知，直接验证
```

**新增架构组件：SCP Adapter**

```
AI（ReAct 循环）
    ↓ 工具调用请求
SCP Adapter（独立微服务 / MCP Server）
    ├── REST API 调用 → SCP
    └── acli 命令执行 → acli 安全容器 → HCI 集群（物理隔离）
```

### 5.7 关键技术决策

#### 一、授权机制：工具定义静态声明风险级别

**背景**：Claude 的隐式风险判断依赖顶尖模型能力，GLM 无法照搬。

**本项目方案：风险级别在工具定义时静态声明，框架强制执行，LLM 不需要判断风险。**

```python
TOOL_REGISTRY = [
    {
        "name": "get_active_alerts",
        "risk_level": 1,        # 静态声明，框架控制
        "policy": "auto",
        "description": "获取当前活跃告警列表",
    },
    {
        "name": "vm_migrate",
        "risk_level": 2,
        "policy": "confirm",    # 框架强制要求用户确认
        "description": "迁移虚拟机到指定宿主机",
    },
]
```

| 对比维度 | Claude 隐式方式 | 本项目显式方式 |
|---------|---------------|--------------|
| 谁判断风险 | LLM 在推理时 | 工具定义里静态声明 |
| 对模型要求 | 极高（顶尖模型） | 低（只需会选工具）|
| 可靠性 | 依赖模型，可能漂移 | 确定性，框架强制 |
| 可审计性 | 难以追溯 | 每次调用都有策略记录 |

> **GLM-4 的 function calling 能力完全足够**：它只需要"选工具、填参数"，不需要判断风险。这比 Claude 的方式**更可靠**，也**更适合生产环境审计**。

---

#### 二、框架选型：自实现轻量 ReAct，不用 LangChain / LangGraph

**结论：自己实现，约 100 行核心代码。**

**理由：**

1. **GLM 格式差异**：两个框架以 OpenAI `tool_use` 格式为基础设计，GLM 在并行工具调用、降级行为、历史剪裁等细节上存在差异，会触发难以调试的隐式失败。

2. **状态机已确定，图结构是负担**：LangGraph 适合"分支由 LLM 动态决定"的场景。本项目的 7 阶段诊断状态机是领域约束下的确定性转换，不需要 `StateGraph`。

3. **与现有架构集成代价高**：`conversation-service` 已有 FastAPI + SSE + SQLAlchemy，引入 LangChain 每一层都是阻抗失配。

4. **ReAct 核心逻辑约 30 行**，简单透明，可完全掌控：

```python
async def run_react_loop(session, user_input):
    messages = session.history + [{"role": "user", "content": user_input}]

    for _ in range(MAX_STEPS):
        response = await llm.chat(messages, tools=TOOL_REGISTRY)

        if response.finish_reason == "stop":
            return response.content                  # 最终答案，退出

        if response.finish_reason == "tool_calls":
            for tool_call in response.tool_calls:
                policy = get_tool_policy(tool_call.name)   # 风险由框架检查
                if policy == "confirm":
                    await sse_push_confirmation_request(tool_call)
                    result = await wait_for_user_confirmation()
                else:
                    result = await execute_tool(tool_call)

                messages.append(tool_result_message(tool_call, result))
```

5. **排障场景不能有黑盒**：LangChain 的隐式重试可能导致操作日志不完整，影响事后审计。

**各组件实现方案汇总：**

| 组件 | 方案 |
|------|------|
| ReAct 执行器 | 自实现 ~100 行，内嵌 conversation-service |
| 工具注册表 | Python 列表 + Pydantic 模型（含 risk_level、policy 元数据）|
| 对话状态 | 扩展现有 ConversationSession，加 diagnostic_stage 字段 |
| SCP Adapter | 新增独立微服务或 client 模块，封装 SCP REST API + acli |
| 授权层 | 框架硬编码规则：risk_level → policy |
| 人工确认 | 现有 SSE 通道，推确认请求，等前端响应 |
