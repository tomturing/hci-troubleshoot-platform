## 零、背景认知 — Claude Agent 工作原理

> 本节是分析的认知基础：理解 Claude 是如何工作的，才能理解本项目与它之间的差距在哪里。

### 0.1 Claude Agent 的核心循环：Think → Act → Observe

Claude 不是"问一句、答一句"的问答机器，而是一个**持续运转的 Agent 循环**：

```
用户输入
   ↓
[思考] 理解意图 + 制定策略
   ↓
[行动] 调用工具（搜代码、读文件、跑命令…）
   ↓
[观察] 分析工具返回结果
   ↓
[判断] 任务完成了吗？
   ├── 否 → 继续思考→行动→观察
   └── 是 → 向用户报告结果
```

### 0.2 以查 Bug 为例：Claude 的全程工作方式

#### 第一步：接收模糊输入，主动澄清

用户说："用户保存工单时，有时候会报 500 错误"

Claude 面临信息不足时会：
1. **识别关键缺失**：什么条件下触发？哪个服务？有没有报错日志？
2. **主动提问**（不猜测）：
   - 500 是偶发还是必现？
   - 有没有具体的 trace_id 或时间戳？
   - 是最近上线新代码后出现的，还是一直存在？

> **原则：不盲目行动，先对齐目标。**

#### 第二步：自主侦察（并行工具调用）

用户回答后，Claude 立刻并行展开调查：

```
并行搜索：
├── grep_search("保存工单" / "create_case" 的路由和 handler)
├── grep_search(日志中的 trace_id 或 500 错误关键词)
├── file_search(昨天修改了哪些文件)
└── read_file(case-service 的 case_router.py)
```

**Claude 自己去找证据，不需要用户一条一条粘贴代码。**

#### 第三步：建立假设，深挖验证

假设 Claude 发现昨天 `case_service.py` 修改了保存逻辑，它会：
1. 读取修改前后的差异
2. 追踪调用链：`router → service → repository → SQL`
3. 发现可疑点后**继续验证**，而不是直接下结论

#### 第四步：与用户对话，确认修复方向

Claude 发现问题后**不会直接改代码**，而是先汇报推理过程，确认方向后再执行修复并跑测试验证。

### 0.3 整个过程的关键能力

| 阶段 | Claude 在做什么 |
|------|----------------|
| **理解意图** | 从模糊描述提取关键信息，识别缺失 |
| **主动提问** | 只问真正影响决策的问题，不问废话 |
| **自主侦察** | 并行调用工具，快速收敛证据 |
| **构建假设** | 从证据推理，而非猜测 |
| **交叉验证** | 对假设继续深挖，避免误判 |
| **透明报告** | 把推理过程和证据一起展示给用户 |
| **执行+验证** | 修复后自动跑测试确认 |

### 0.4 Claude 的知识来源：三种机制

Claude 本身不做 RAG——它的知识烘焙在模型权重里。但在 Agent 模式下，通过**工具调用**实现了等效甚至更强的效果：

| 来源 | 机制 | 例子 |
|------|------|------|
| **参数化知识** | 训练时压缩进权重 | Python 语法、框架 API、通用 debug 模式 |
| **上下文知识** | 工具读取后注入 context window | 读代码文件、日志、文档 |
| **外部 RAG** | 需要专门接入才有 | 企业私有知识库、文档库 |

```
传统 RAG：
  问题 → 向量检索文档片段 → 注入 prompt → 生成回答

Claude Agent（代码库场景）：
  问题 → grep/file_search/read_file 检索代码 → 注入 context → 推理回答
```

两者信息流完全一致，区别只是**检索引擎不同**。

### 0.5 本项目的 RAG 定位

本项目的 RAG（KB Service）是给**最终用户**提供的领域知识增强：

- 故障描述 → 向量检索相似 SOP → 召回相关排障步骤 → AI 结合上下文给出建议
- **核心目标**：让 AI 具备"HCI 领域专家知识"，弥补通用 LLM 在 HCI 特定故障处理上的经验空白

> **Claude 之所以强大，不是因为存了答案，而是因为它有推理能力 + 工具调用 + 知识三者的协同。本项目要复刻的，正是这三者的组合。**

### 2.2 三层职责分工（设计意图 vs 实际）

| 角色 | 设计职责 | 当前实际状态 |
|------|---------|------------|
| **LearningClaw** | 读 7000 案例 → 提炼 → 写入 KB | ❌ 未部署，知识库是空的 |
| **ProductionClaw** | 读 KB → 组装 Prompt → 排障对话 | ⚠️ 代码逻辑已有，但消费不到知识 |
| **KB Service** | 存储 + 检索知识 / SOP | ❌ 未实现，是最大缺口 |

### 2.3 Prompt 设计的结构性缺陷（最核心问题）

当前 `_SYSTEM_BASE` 的关键约束：

```
"只能根据下方'参考资料'部分的内容回答问题，不得使用训练数据中的其他知识"
"如果参考资料中没有相关内容，必须直接回复：当前知识库暂未收录..."
```

**致命逻辑链**：

```
KB Service 未上线
    → kb_client 返回空
    → 4-Tier Prompt Tier2/Tier3 全空
    → 触发"未找到相关知识"提示
    → LLM 被强制约束"不得自行推测"
    → 用户任何问题 → "知识库暂未收录，建议联系技术支持"
```

**这直接导致测试效果与预期相差甚远。**

### 2.4 RAG 设计与 AI 层的对接断层

| 环节 | RAG 设计描述 | AI 层实际调用 | 断层 |
|------|------------|------------|------|
| SOP 匹配 | 关键字精确路由，0 LLM 开销 | `kb_client.sop_match()` HTTP 调用 | KB Service 未上线，永远返回 None |
| 语义检索 | BM25+Vector+RRF 融合 | `kb_client.search()` | 同上，返回 `[]` |
| LLM 分类器 | 先分类再缩小检索范围 | **未实现** | 直接全库搜，无分类前置 |
| Rerank | LLM 对 top-5 评分过滤 | **未实现** | 搜索结果直接注入，无质量把关 |
| 知识数据 | 7000 案例 + 10 本 SOP | 600 条 MD 未入库，SOP 格式损坏 | 数据层完全缺失 |

## 三、Claude 式能力 vs 当前系统的本质差距

### 3.1 Claude 排障的本质：侦探式推理

Claude 处理故障排障时，本质上是一个**推理引擎**，知识是燃料，不是答案本身：

```
症状输入
  ↓
[形成假设]  "可能是 A、B、C 三种原因"
  ↓
[精准提问]  "请问…的状态是什么？"   ← 每个问题都在验证/排除一个假设
  ↓
[更新假设]  排除 C，剩余 A 和 B
  ↓
[再次提问]  "执行 xxx 命令看一下输出…"
  ↓
[收敛定位]  根因 = A
  ↓
[方案生成]  基于根因 + 环境 → 具体可执行步骤
```

> **知识服务于推理，而不是代替推理。**

### 3.2 当前系统的本质：智能检索 + 摘要系统

```
用户第一句话
  ↓
[检索触发]  关键字匹配 SOP + 向量检索 topK
  ↓
[注入 Prompt]  "参考资料：[检索结果]，请基于此回答"
  ↓
[LLM 生成]  将参考资料转述给用户
```

> **当前系统是一个文档检索 + 摘要系统，不是排障推理系统。**

### 3.3 六个本质差距

#### 差距 1：问题理解的深度

| 维度 | Claude | 当前系统 |
|------|--------|---------|
| 对模糊输入的处理 | 主动拆解歧义，识别缺失信息，精准追问 | 拿模糊输入直接检索，召回噪声高 |
| 对技术术语的理解 | 语义理解（"虚拟机起不来" → 推导多种可能原因） | 字符串匹配 |
| 上下文积累 | 每轮对话都在缩小问题空间 | 每轮对话独立检索，不累积诊断进展 |

#### 差距 2：知识的角色定位

```
Claude:      知识 → 提供 context → 辅助推理 → 生成方案
                                       ↑
                                 推理是核心

当前系统:    知识 → 直接呈现给用户 → 用户自己理解
                    ↑
              知识是答案本身
```

Claude 看到一个案例，会提炼出"这种症状背后的机制是什么"，然后用机制来推理用户的具体情况。当前系统直接把案例原文贴给用户。

#### 差距 3：不确定性的处理方式

| 情形 | Claude | 当前系统 |
|------|--------|---------|
| 知识库有相关内容 | 结合环境信息推断，给出置信度 | 原文引用，不推断 |
| 知识库无相关内容 | "目前信息不足，我有几个可能方向，需要你提供…" | "知识库暂未收录，建议联系技术支持" |
| 症状有多种可能 | 主动列出假设树，按概率排序，逐一排查 | 召回多个案例并列展示，让用户自判 |

> **当前系统把"我不知道"和"你的问题无解"混为一谈——这是认知层面的根本错误。**

#### 差距 4：诊断过程 vs 答案检索

真正的排障专家遵循固定的诊断流程：

```
① 确认故障现象（复现条件、影响范围、发生时间）
② 收集环境信息（版本、配置、日志）
③ 形成假设（Top 3 可能原因）
④ 设计验证步骤（最小代价先验证高概率假设）
⑤ 执行验证、收集结果
⑥ 更新假设、收敛根因
⑦ 方案设计（临时恢复 vs 根因修复）
```

> **当前系统跳过了 ①-⑥，直接到 ⑦——而且 ⑦ 是基于检索到的历史案例，不是基于用户当前环境。**

#### 差距 5：Prompt 哲学的根本分歧

| | 哲学定位 | 行为表现 |
|---|---|---|
| **当前系统** | "知识库的代言人，只说库里有的" | 检索到什么展示什么，无内容则拒绝回答 |
| **Claude** | "推理者，知识库是参考，推理能力服务具体问题" | 以知识为燃料，用推理能力灵活应对 |

```
用户：虚拟机开不起来

[知识库代言人]:
  检索到 3 个相关案例，以下是参考资料...
  [贴出案例原文]

[推理者]:
  虚拟机无法启动有几种常见原因，我需要先了解几个信息：
  1. 报错信息是什么？
  2. 是所有 VM 还是特定 VM？
  3. 最近有没有做过什么操作？
  ——根据你的回答我来帮你定位具体原因。
```

#### 差距 6：知识的粒度与形态

| 知识形态 | Claude | 当前系统（设计） |
|---------|--------|----------------|
| **机制知识** | 知道"为什么"，能迁移到新情况 | 不存在，chunks 是案例片段 |
| **程序性知识** | 知道"排查顺序"，能动态调整 | 部分（SOP），但静态 |
| **陈述性知识** | 知道"是什么"，HCI 领域常识 | 案例库（7000 条），但未入库 |
| **元认知** | 知道自己不知道什么，知道何时追问 | 不存在，只有二值判断（有/无）|

### 3.4 一个关键认知误区

项目设计文档中有一句话揭示了核心误区：

> "精准优于召回：有就是有，没有就是没有，减少幻觉，杜绝胡说"

这个原则**保护了数据质量，但牺牲了推理能力**。

Claude 之所以是"HCI 专家"，不是因为它存了 7000 个答案，而是：
1. 它理解 HCI 系统的**运行机制**
2. 它有**诊断推理**的方法论
3. 它**知道该问什么问题**来收集信息

> **"有就是有，没有就是没有"是图书馆的原则，不是专家的原则。专家在没有先例的情况下，依然能凭机制推理给出建议。**

### 3.5 差距图谱

```
目标：HCI 领域专家 Claude
┌────────────────────────────────────────┐
│  推理层  诊断推理 + 假设树 + 动态问询   │ ← 当前完全缺失
│  知识层  机制理解 + 程序性知识 + RAG   │ ← RAG 未上线，机制知识无
│  交互层  多轮收敛 + 主动引导 + 不确定性 │ ← Prompt 限制了这层
└────────────────────────────────────────┘

当前系统
┌────────────────────────────────────────┐
│  推理层  ✗ 不存在                      │
│  知识层  ✗ 数据库空 + KB Service 未上线 │
│  交互层  ⚠️ 被 Prompt 限制为"文档转述员" │
└────────────────────────────────────────┘
```

### 3.6 结论：问题不只是"数据没入库"

即使今天把 7000 条案例全部入库、KB Service 上线，**当前架构仍然无法达到"Claude 式体验"**，因为：

1. **没有诊断推理链** — Prompt 没有引导 LLM 进行假设→验证→收敛的过程
2. **检索时机错误** — 第一句话就检索，而不是"先问清楚问题再检索"
3. **知识形态缺失** — 案例 chunks 不等于机制知识，LLM 需要机制知识才能推理
4. **交互模式错误** — 系统设定自己是"回答者"，应该是"排障引导者"

## 四、重构方案

> **核心转变**：从"检索增强问答系统"→"推理优先专家系统（知识辅助）"

### 4.0 重构总纲

三节分析揭示了一个清晰的结论：**当前系统的问题不是单点的，而是系统性的**。  
修一个 Bug 解决不了，必须在四个正交维度上同时重构：

```
维度A: Prompt 哲学     ← 当前是"图书馆员"，目标是"侦探医生"
维度B: 对话状态机      ← 当前无状态，目标是7阶段诊断流程
维度C: 知识基础设施    ← 当前是空库，目标是三层知识体系
维度D: 架构执行路径    ← 当前设计未落地，目标是设计与实现对齐
```

四个维度并行推进，但有依赖优先级——**维度A 可以立即执行，且独立于其他三个，是成本最低、收益最快的改动**。

---

### 4.1 维度A：Prompt 哲学重写

#### 4.1.1 核心问题诊断

当前 `_SYSTEM_BASE` 只有一段话，本质是一个**约束声明**，没有赋予 AI 任何能力：

```python
# 当前的 Prompt 哲学：约束 > 能力
_SYSTEM_BASE = """
你是 HCI 智能排障助手...
严格答复规则：
1. 只能根据"参考资料"部分的内容回答
2. 如果参考资料中没有相关内容，必须直接回复：「知识库暂未收录...」
3. 严禁自由发挥、推测或补充
"""
```

这段 Prompt 犯了三个错误：
1. **角色定义缺失**：只说了不能做什么，没说具体是谁、有什么能力
2. **方法论缺失**：没有"如何排障"的指引，AI 不知道该问什么问题
3. **知识使用策略倒置**：知识应该是推理的燃料，而不是唯一答案来源

#### 4.1.2 重构后的 Prompt 架构（5段式）

新的 System Prompt 重构为五个功能段，每段有明确职责：

```
┌─────────────────────────────────────────────────────────┐
│ Segment 1: 专家身份（Who am I）                          │
│   → 定义专业背景、能力范围、思维方式                      │
├─────────────────────────────────────────────────────────┤
│ Segment 2: 诊断方法论（How I work）                      │
│   → 7步诊断流程，对话的行为准则                           │
├─────────────────────────────────────────────────────────┤
│ Segment 3: 机制知识锚（Domain knowledge）                │
│   → HCI 系统核心机制、常见故障模式（静态注入）            │
```

#### 4.1.4 Segment 2 重写：诊断方法论

```
## 工作方法

每次排障，你遵循结构化诊断流程：

1. 【症状采集】确认故障现象、影响范围、发生时间点、复现条件
2. 【范围界定】定位问题域（虚拟机/存储/网络/平台/硬件），不要跨域乱猜
3. 【假设形成】基于症状列出 Top 2-3 个可能根因，按概率从高到低排序
4. 【最小验证】设计代价最小的验证步骤，先验证高概率假设
5. 【证据收集】引导用户执行验证命令，解读输出
6. 【根因收敛】排除假设，直到置信度 > 80%
7. 【方案给出】分级方案：临时恢复（分钟级）+ 根因修复（彻底解决）

## 关键原则

- **先收信息，后给结论**：不要在第一轮就给出确定性答案
- **一次只问一个问题**：不要同时抛出 5 个问题让用户应接不暇
- **命令必须说明用途**：建议执行任何命令前，说明"这个命令看什么，为什么要看"
- **不确定时说不确定**：用"可能是…，需要确认"而非"就是…"
- **没有先例也能推理**：历史案例是参考，不是限制；知识库没有的问题，
  用 HCI 系统机制来推理，明确标注"基于机制推断，置信度中等"
```

```
### 虚拟机故障常见根因树
虚拟机无法启动 → 可能原因（按概率）：
  ① 资源不足（CPU/内存/存储空间）→ 查 acli vm.list / df -h

### 网络故障诊断路径

  物理层（网线/光模块/交换机端口）→ ethtool peth0
  → 虚拟层（vSwitch/vNIC 配置）→ ovs-vsctl show
  → IP 层（路由/VLAN）→ ip route / ping

[此处为精简示例；完整实现应从 SOP 手册和案例库中自动提炼后填充]
```

### 4.2 维度B：诊断推理链 — 对话状态机

#### 4.2.1 核心问题诊断

当前对话流程是**无状态的扁平结构**：

```
消息1 → 检索 → 生成回复
消息2 → 检索 → 生成回复   ← 不知道上一轮问了什么，诊断没有进展
消息3 → 检索 → 生成回复   ← 每轮都是全新的"问答"
```

结果是：用户说了三轮，AI 还在原地踏步，没有向根因收敛。

#### 4.2.2 重构后：7阶段诊断状态机

```
┌──────────────────────────────────────────────────────────────┐
│                    诊断状态机                                 │
│                                                              │
│  S0:INTAKE → S1:SCOPING → S2:HYPOTHESIS → S3:VERIFICATION   │
│                                │               │            │
│                          S4:ROOT_CAUSE ←───────┘            │
│                                │                            │
│                          S5:RESOLUTION                      │
│                                │                            │
│                          S6:CLOSURE                         │
└──────────────────────────────────────────────────────────────┘
```

| 阶段 | 名称 | AI 的任务 | 完成条件 | RAG 触发？ |
|------|------|---------|---------|-----------|
| S0 | **症状采集** | 收集：现象/范围/时间/复现条件 | 信息足够形成假设 | ❌ 不触发 |
| S1 | **范围界定** | 确定故障域（L1/L2分类） | 问题域明确 | ✅ **此时触发** |
| S2 | **假设形成** | 给出 Top 2-3 假设，按概率排序 | 用户确认方向 | ✅ 补充检索 |
| S3 | **逐一验证** | 设计最小验证步骤，引导执行 | 根因置信度 > 80% | ⚠️ 按需 |
| S4 | **根因确认** | 明确宣布根因，得到用户确认 | 用户确认 | ❌ |
| S5 | **方案设计** | 临时恢复 + 根因修复两套方案 | 用户选择执行 | ✅ 精准检索 |
| S6 | **排障小结** | 生成结构化记录，供 LearningClaw 使用 | 工单关闭 | ❌ |

#### 4.2.3 关键设计：检索时机延迟

**当前系统的问题**：在 S0（症状模糊期）就触发 RAG 检索，相当于用"虚拟机开不起来"这种模糊 query 去搜库，召回结果必然噪声高。

**重构后**：在 S1（范围清晰后）触发检索，此时 query 变为"虚拟机开机失败，宿主机显示 CPU 不足，集群大小4节点"，检索质量大幅提升。

```python
# 重构后的检索触发逻辑（伪代码）
async def should_trigger_rag(conversation_stage: str, message: str) -> bool:
    if conversation_stage in ["S0"]:
        return False  # 信息不足，不检索
    if conversation_stage in ["S1", "S2"]:
        return True   # 范围明确，精准检索
    if conversation_stage in ["S3", "S4"]:
        return False  # 验证阶段，依赖AI推理
    if conversation_stage == "S5":
        return True   # 方案阶段，检索最佳实践
    return False
```

> 前四节的重构方案方向正确，但仍然不够彻底。
> 最终目标"再造一个专业HCI智能排障Claude"，要求的是 **Agent 架构**，而不是"更好的问答系统"。

### 5.1 已确认的关键信息

| 信息项 | 结论 | 对架构的影响 |
|--------|------|------------|
| **AI 能否直接操作 HCI 系统** | ✅ **可以，需要客户授权** | 整个系统从 RAG 增强问答 → 真正的 Agentic 系统 |
| 授权模式 | 待细化（命令级/会话级/风险分级） | 决定 Tool Use 执行层的设计 |
| HCI 工具/API 面 | 待补充 | 决定 Agent 能力上限 |
| 故障类型分布 | 待补充 | 决定 SOP 和机制知识的优先级 |
| 人工排障思维模型 | 待补充 | 填充方法论和假设树 |

### 5.2 对"再造专业HCI智能排障Claude"的深刻理解

Claude 之所以强大，不是因为它"存了很多答案"，而是因为它的架构是这样的：

```
LLM 是大脑，工具是它的手和眼
LLM 主动决定"下一步做什么" → 调用工具 → 解读结果 → 再决定 → 再调用
直到问题解决，才给用户最终答案
```

**当前系统（包括前四节的重构方案）本质上仍是：**

```
用户发消息 → 平台预先组装 Prompt → LLM 生成一段话 → 等待下一条用户消息
```

这两个架构有根本区别：

| 维度 | 当前系统（含 Phase 0-2 重构） | 目标系统（专业HCI排障Claude）|
|------|------------------------------|----------------------------|
| **控制权** | 平台决定注入什么知识，LLM 被动接受 | LLM 主动决定调用什么工具 |
| **信息获取** | 一次性 RAG 注入，LLM 无法追加获取 | LLM 随时按需调用工具获取信息 |
| **推理模式** | 单轮生成（一条消息对应一次生成） | 多步推理（一条消息 → N次工具调用 → 最终回答）|
| **诊断能力** | 被动回答用户提供的信息 | 主动侦察系统状态 |
| **知识角色** | RAG 是"地基"，决定能不能回答 | RAG 是"工具之一"，与其他工具平等 |

### 5.3 目标架构：ReAct 模式（Reasoning + Acting）

这是 Claude 实际使用的架构模式，也是本项目的最终目标：

```
用户: "3台虚拟机突然起不来了"
                │
                ▼
    ┌───────────────────────┐
    │  LLM 推理（内部）      │
    │  "需要先了解VM当前状态"│
    └───────────┬───────────┘
                │ 工具调用
                ▼
    ┌───────────────────────┐     需要授权？
    │  get_vm_status(vm_id) │ ──→ [授权提示] ──→ 用户确认
    └───────────┬───────────┘
                │ 返回结果：VM ERROR, CPU overcommit
                ▼
    ┌───────────────────────┐
    │  LLM 推理（内部）      │
    │  "CPU问题，查集群资源" │
    └───────────┬───────────┘
                │ 工具调用（只读，自动执行）
                ▼
    ┌───────────────────────────┐
    │  get_cluster_cpu_usage()  │
    └───────────┬───────────────┘
                │ 返回结果：集群 CPU 95%
                ▼
    ┌───────────────────────────────────────┐
    │  LLM 推理（内部）                      │
    │  "根因确认：CPU资源耗尽，有3台竞争资源"│
    │  "检索相关案例和 SOP 获取操作建议"     │
    └───────────┬───────────────────────────┘
                │ 工具调用
                ▼
    ┌───────────────────────────┐
    │  search_knowledge(query)  │  ← RAG 是工具，不是地基
    └───────────┬───────────────┘
                │ 返回：相关 SOP 和案例
                ▼
    ┌───────────────────────────────────────┐
    │  LLM 生成最终回答（有根据、有步骤）    │
    │  临时恢复：迁移1台VM释放CPU            │
    │  根因修复：扩容集群或重新规划资源      │
    └───────────────────────────────────────┘
                │
                ▼
    用户收到：有根因、有方案、有具体命令的专家级回答
```

> **与"当前重构方案（Phase 0-2）"的关系**：
> Phase 0-2 是 **过渡阶段**（改善体验，快速上线），目标架构是 **Phase 3 的核心任务**。
> 两者不冲突，但必须认清：Phase 0-2 的终点不是目标终点。

### 5.4 目标系统的工具目录（Tool Catalog）

工具是 Agent 的能力边界。"专业HCI排障Claude"的工具目录分三类：

#### 类型1：知识工具（无需授权，自动执行）

| 工具 | 功能 | 对应现有能力 |
|------|------|------------|
| `search_knowledge(query, category?)` | RAG 语义检索 | KB Service |
| `match_sop(query)` | SOP 精确匹配 | sop_matcher |
| `get_hci_mechanism(topic)` | 查询 HCI 机制知识 | 机制知识库（待建）|

#### 类型2：诊断工具（只读，低风险，提示后自动执行）

| 工具 | 功能 | 典型命令 |
|------|------|---------|
| `get_vm_status(vm_id)` | 查询 VM 当前状态 | acli vm.get |
| `get_cluster_health()` | 集群整体健康状态 | acli cluster.status |
| `get_resource_usage()` | CPU/内存/存储用量 | acli host.list |
| `get_storage_status()` | 存储卷/磁盘状态 | acli vg.list |
| `get_network_status()` | 网络连通性和配置 | ovs-vsctl show |
| `search_logs(keyword, time_range)` | 日志关键字搜索 | journal/syslog |
| `get_alerts(time_range)` | 告警历史 | 告警 API |

#### 类型3：操作工具（有风险，强制用户授权）

| 工具 | 风险等级 | 授权要求 |
|------|---------|---------|
| `restart_vm(vm_id)` | 中 | 确认 + 说明影响 |
| `migrate_vm(vm_id, target_host)` | 中 | 确认 + 检查前置条件 |
| `restart_service(service_name)` | 中高 | 确认 + 说明影响范围 |
| `modify_config(target, key, value)` | 高 | 强确认 + 变更记录 |
| `execute_command(cmd)` | 动态评估 | 按命令内容动态分级 |

#### 永久禁止工具（安全红线）

```
❌ 删除数据 / 格式化磁盘 / 删除 VM
❌ 修改认证配置（密码/证书/密钥）
❌ 关闭集群或宿主机
❌ 任何未经分析的批量操作
```

### 5.5 授权机制设计（初稿）

基于安全性和用户体验的平衡，建议采用**风险分级授权**（模式C）：

```
┌──────────────────────────────────────────────────────────────┐
│                      授权分级策略                             │
│                                                              │
│  Level 0: 知识查询     → 静默执行，无需提示                   │
│           （search_knowledge / match_sop）                   │
│                                                              │
│  Level 1: 只读诊断     → 提示执行意图，3秒无操作自动执行       │
│           （get_vm_status / get_cluster_health ...）         │
│                                                              │
│  Level 2: 低风险操作   → 明确提示 + 确认按钮，不自动执行       │
│           （restart_vm / migrate_vm ...）                    │
│                                                              │
│  Level 3: 高风险操作   → 强确认 + 展示变更内容 + 输入确认码   │
│           （modify_config / restart_service ...）            │
│                                                              │
│  Level X: 禁止操作     → 无论如何都不执行，解释原因            │
└──────────────────────────────────────────────────────────────┘
```

工单开始时，用户可以设置会话级授权策略（如"本次排障全程授权 Level 1 自动执行"），减少重复确认的摩擦。

### 5.8 关键信息补充（2026-03-20 确认）

#### 一、关于授权机制：Claude 的参考模型

Claude 自身的工具授权机制如下，可作为本项目设计参考：

- **无感执行**：只读、无副作用的工具（文件读取、搜索）→ 静默执行，不打断用户
- **提示执行**：有潜在影响的操作（写文件、运行命令）→ 明确说明意图，执行前展示将要做什么
- **强制确认**：高风险或不可逆操作（删除、外部调用）→ 等待用户明确确认
- **原则**：可逆操作 > 不可逆操作；本地影响 > 共享系统影响；操作前说明而非事后解释

**对本项目的启示**：Claude 的授权粒度不是预先配置的，而是**由 LLM 在推理时自主判断操作风险等级**，然后决定是否需要确认。这比固定规则表更灵活，也更符合"再造Claude"的目标。

#### 二、HCI 平台工具面（已确认）

这是目前最重要的架构信息，直接决定 Agent 的能力边界。

**核心发现：80% 以上的客户问题来源于两类：**
```
a. 平台告警的确认（Alert Confirmation）
b. 失败任务的排查（Failed Task Investigation）
```

**工具接入层：SCP（Service Control Platform）**

| 工具类型 | 接入方式 | 说明 |
|---------|---------|------|
| 管理 REST API | 通过 SCP 提供 | 告警列表、任务列表、VM状态、集群健康等 |
| acli 命令 | 通过 SCP 封装后对外暴露 | 安全容器隔离，与 HCI host 彻底隔离 |
| 监控数据 | ❌ 暂无接口 | Prometheus/Grafana 暂无直接调用方式 |

**acli 安全模型（关键）：**
```
acli 类比 VMware 的 esxcli，但有本质区别：
  esxcli: 直接在 ESXi host 上执行，权限高，风险高
  acli（本项目）: 封装在安全容器中，与 HCI host 彻底物理隔离
               只提供经过审计的安全命令子集
               → 这意味着工具调用的安全底线由平台保证，AI 层无需重复防御
```

**对工具目录的影响：**
- 工具调用入口统一走 **SCP Adapter**（不直接连 HCI host）
- acli 的安全容器隔离 = Level 1（只读诊断）工具可以更放心地设置为"提示后自动执行"
- 操作类工具（Level 2/3）仍需用户确认，但底层安全由 acli 安全容器兜底

**修订后的工具目录（基于 SCP 接入）：**

```
通过 SCP REST API 调用（告警/任务/状态查询）：
  get_active_alerts()          → 获取当前活跃告警列表          Level 1
  get_failed_tasks()           → 获取失败任务列表              Level 1
  get_vm_status(vm_id)         → 查询虚拟机状态               Level 1
  get_cluster_health()         → 集群整体健康状态              Level 1
  get_task_detail(task_id)     → 获取任务详情和错误信息         Level 1

通过 SCP + acli 安全容器调用（深度诊断）：
  acli_vm_list()               → 列出所有VM及状态             Level 1
  acli_host_list()             → 列出宿主机及资源使用          Level 1
  acli_vg_list()               → 列出存储卷组                 Level 1
  acli_vm_get(vm_id)           → 获取VM详细配置               Level 1
  acli_net_list()              → 列出网络配置                  Level 1
  acli_vm_on(vm_id)            → 启动虚拟机                   Level 2（需确认）
  acli_vm_off(vm_id)           → 关闭虚拟机                   Level 2（需确认）
  acli_vm_migrate(vm_id, host) → 迁移虚拟机                   Level 2（需确认）
```

**关键架构决策：新增 SCP Adapter 服务**
```
AI（ReAct 循环）
    ↓ 工具调用请求
SCP Adapter（新增微服务 / MCP Server）
    ├── REST API 调用 → SCP
    └── acli 命令执行 → acli 安全容器 → HCI 集群（物理隔离）
```

#### 三、故障类型分类基准（已整理）

**文件位置**：`data-pipeline/config/category_baseline.yaml`（从项目根目录迁移）

**覆盖范围**：198 个分类节点，5 大域

| 域 | 分类数 | 备注 |
|----|--------|------|
| 虚拟机 | 54 | 最多，含生命周期全流程 |
| 存储 | 46 | 含虚拟存储、FC/iSCSI/NFS |
| 硬件 | 45 | 含风扇/电源/主板/RAID/IPMI |
| 平台 | 30 | 含物理主机/系统管理/授权 |
| 网络 | 23 | 含物理/虚拟网络 |

**价值评估：非常有用，是整个知识体系的骨架。**

具体用途：
1. **ETL Pipeline 分类标签**：案例入库时的 `category_l1/l2` 来源（已在 `知识库RAG设计.md` 中规划）
2. **LLM 分类器输入**：检索前先分类，缩小检索范围（即 `taxonomy_context`）
3. **SOP 路由索引**：告警/任务 → 故障分类 → SOP 路由的中间映射层
4. **机制知识锚组织结构**：按 5 大域分别编写 HCI 机制知识，198 个节点覆盖率可量化

**与 SCP 告警/任务对接的关键洞察：**

80% 的问题来自告警确认和失败任务排查，意味着诊断流程可以被大幅简化：

```
传统模式（模糊描述 → 分类 → 假设 → 验证）：
  用户："虚拟机起不来"
  AI：需要多轮追问确认分类...

SCP 告警驱动模式（结构化输入 → 直接定位）：
  告警/任务：{ type: "VM_BOOT_FAIL", vm_id: "vm-123", error_code: "E_CPU_OVERCOMMIT" }
  AI：分类已知（虚拟机开机失败），根因方向已知（CPU 不足），
      直接进入 S3（验证）而非 S0（症状采集）
```

**这个发现让诊断状态机可以有两个入口**：
- `入口A`：用户文字描述（走完整 S0→S6）
- `入口B`：告警/任务结构化数据（直接跳到 S1/S2，效率大幅提升）

### 5.10 关键技术决策（2026-03-20 确认）

#### 一、授权机制：不依赖 LLM 判断风险，改为工具定义静态声明

**背景**：Claude 的隐式风险判断是其顶尖模型能力的体现，GLM 无法照搬。

**本项目方案：风险级别在工具定义时静态声明，框架强制执行，LLM 无需判断。**

```python
# 工具注册表 —— 风险级别由工具定义，不由 LLM 运行时判断
TOOL_REGISTRY = [
    {
        "name": "get_active_alerts",
        "risk_level": 1,        # 静态声明
        "policy": "auto",       # 框架根据此字段决定执行方式
        "description": "获取当前活跃告警列表",
    },
    {
        "name": "vm_migrate",
        "risk_level": 2,
        "policy": "confirm",    # 框架强制要求用户确认，不询问 LLM
        "description": "迁移虚拟机到指定宿主机",
    },
]
```

| 对比维度 | Claude 隐式方式 | 本项目显式方式 |
|---------|---------------|--------------|
| 谁判断风险 | LLM 在推理时 | 工具定义里静态声明 |
| 对模型要求 | 极高（顶尖模型） | 低（只需会选工具） |
| 可靠性 | 依赖模型能力，可能漂移 | 确定性，框架强制 |
| 可审计性 | 难以追溯 LLM 为何判断 | 每次调用都有策略记录 |

**GLM-4 的 function calling 能力完全足够**：它只需要"选工具、填参数"，不需要判断风险。这比 Claude 的方式**更可靠**，也**更适合生产环境审计**。

---

#### 二、框架选型：自实现轻量 ReAct 执行器，不用 LangChain / LangGraph

**结论：自己实现，约 100 行核心代码。**

**不用 LangChain / LangGraph 的原因：**

1. **GLM 格式差异**：两个框架以 OpenAI `tool_use` 格式为基础设计，GLM 在并行工具调用、降级行为、历史剪裁等细节上存在差异，会触发难以调试的隐式失败。

2. **状态机已确定，图结构是负担**：LangGraph 适合"分支由 LLM 动态决定"的场景。我们的 7 阶段诊断状态机是领域约束下的确定性转换，`StateGraph` 反而需要额外胶水代码。

3. **与现有架构集成代价高**：`conversation-service` 已有 FastAPI + SSE + SQLAlchemy。引入 LangChain 意味着对话状态、SSE 流式输出、数据库操作都要绕过框架的同等抽象——每一层都是阻抗失配。

4. **ReAct 循环本身极其简单**，核心逻辑约 30 行：

```python
async def run_react_loop(session, user_input):
    messages = session.history + [{"role": "user", "content": user_input}]

    for _ in range(MAX_STEPS):
        response = await llm.chat(messages, tools=TOOL_REGISTRY)

        if response.finish_reason == "stop":
            return response.content                  # 最终答案，退出

        if response.finish_reason == "tool_calls":
            for tool_call in response.tool_calls:
                # 风险由框架检查，不问 LLM
                policy = get_tool_policy(tool_call.name)
                if policy == "confirm":
                    await sse_push_confirmation_request(tool_call)  # 推给前端
                    result = await wait_for_user_confirmation()
                else:
                    result = await execute_tool(tool_call)          # 直接执行

                messages.append(tool_result_message(tool_call, result))
```

5. **排障场景不能有黑盒**：LangChain 在工具链路上有大量隐式行为（重试、格式转换、错误吞没）。一次被框架静默重试的操作可能导致故障扩大或操作日志不完整，影响事后审计。

**各组件实现方案汇总：**

| 组件 | 方案 | 说明 |
|------|------|------|
| ReAct 执行器 | 自实现 ~100 行 | 内嵌 conversation-service |
| 工具注册表 | Python 列表 + Pydantic 模型 | 含 risk_level、policy 元数据 |
| 对话状态 | 扩展现有 ConversationSession | 加 diagnostic_stage 字段 |
| SCP Adapter | 新增独立微服务或 client 模块 | 封装 SCP REST API + acli |
| 授权层 | 框架硬编码规则 | risk_level → policy |
| 人工确认 | 现有 SSE 通道 | 推确认请求，等前端响应 |

> 如果将来需要跨会话持久化长时排障任务，可以考虑引入 LangGraph 作为持久化层，但执行逻辑仍然自控。现阶段不需要。
