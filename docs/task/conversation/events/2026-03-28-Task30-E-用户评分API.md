# Task 30-E：用户评分 API（conversation-service）

```
你是一名负责 hci-troubleshoot-platform conversation-service API 开发的 agent。

【仓库】
git clone https://github.com/tomturing/hci-troubleshoot-platform.git
cd hci-troubleshoot-platform

【前置条件】
Task 30-A（数据库迁移）已执行，assistant_evaluation 表存在，含 user_rating、composite_score 列。

【任务目标】
文件：backend/conversation-service/app/routes/conversations.py（或新建 evaluate.py）

新增路由：POST /conversations/{conversation_id}/evaluate

请求体：
{ "score": 4 }   // 1-5 星整数

处理逻辑：
1. 验证 score ∈ {1,2,3,4,5}
2. UPSERT assistant_evaluation：设置 user_rating = score
3. 调用 QualityScoreService（或直接通过 HTTP 调用 case-service）重新计算 composite_score，将新 score 加入用户满意度维度
4. 返回 {"ok": true, "composite_score": <新值>}
5. 幂等：同一 conversation_id 重复调用应返回 200 而非 409

另外新增管理统计接口（admin 角色 token 鉴权）：
- GET /admin/quality/stats  → 返回近 7 天平均 composite_score、各 close_reason 分布
- GET /admin/quality/cases  → 返回低分工单列表（composite_score < 40），分页

接口设计详见 docs/16_评分机制与评价系统.md §6。

【约束】
- 保持与现有路由文件的风格一致（FastAPI，依赖注入，SQLAlchemy）
- 使用 JWT 或现有 token 机制鉴权，admin 接口限制普通用户
- 代码注释使用中文

【验收标准】
- POST /conversations/{id}/evaluate 入参 {"score": 4} → 200，assistant_evaluation.user_rating = 4
- 重复调用同一 conversation → 200（幂等），不报 409
- composite_score 在评分后重新计算，值比无用户评分时变化（验证用户满意度维度插入）
- GET /admin/quality/stats 返回合法 JSON

完成后提交 PR，等待 Claude 审核。
```
