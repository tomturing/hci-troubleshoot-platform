# ops-agent 大脑可选：集成任务分解

> 关联方案：[2026-04-28-ops-agent大脑可选集成重设计方案.md](../../../solution/agent/events/2026-04-28-ops-agent大脑可选集成重设计方案.md)

## Phase 1：修复语义错配（核心，优先级最高）

> **目标**：让 ops-agent 在 htp 里能多轮对话，不每轮重置，不互相干扰  
> **预计工作量**：3-5 天  
> **仓库**：htp + ops-agent

### T1-1：ops-agent 真流式输出改造

- **仓库**：ops-agent
- **描述**：改造 `base_agent.py` 的 ReAct 主循环，使其在每步 LLM 推理时实时 yield LLMTokenDelta，而不是等整个链路完成后在 `openai_compat.py` 里伪造 SSE 流。
- **文件变更**：
  - `ops_agent/agent/base_agent.py`：ReAct 循环改为 `async def run_streaming()` 生成器方法，保留原有 `run()`/`execute_task()` 走原有路径（CLI/ACP 零影响）
  - `ops_agent/server/openai_compat.py`：流式模式调用 `run_streaming()`，非流式模式保持原有调用
- **验收标准**：
  - [ ] ops-agent 模式首 token 延迟 ≤ 3 秒（LLM 正常响应时）
  - [ ] CLI 模式 (`ops-agent chat`) 无任何变化
  - [ ] ACP 模式无任何变化
- **依赖**：无
- **预计耗时**：1.5 天

---

### T1-2：ops-agent 支持 session_id（in-memory 实现）

- **仓库**：ops-agent
- **描述**：`/v1/chat/completions` 接受 `session_id` 参数，用 LRU 内存字典（容量 1000）维护 session_id → messages 历史的映射。同一 session_id 的连续请求会在上一轮 messages 基础上继续推理。
- **文件变更**：
  - `ops_agent/server/openai_compat.py`：ChatCompletionRequest 增加 `session_id` 和 `hci_context` 字段；实现 session 恢复逻辑
  - `ops_agent/server/session_store.py`（新增）：LRU(1000) 内存会话存储，支持 get/set/clear
- **验收标准**：
  - [ ] 5 轮连续对话（相同 session_id），第 5 轮回复中 ops-agent 仍能引用第 1 轮的故障关键词
  - [ ] 不传 session_id 时行为与当前一致（新建会话，向后兼容）
  - [ ] session 数超过 1000 时，最旧的 session 被 LRU 淘汰（内存不泄漏）
- **依赖**：T1-1
- **预计耗时**：0.5 天

---

### T1-3：htp 新增 BrainPort Protocol 和 BrainEvent 类型

- **仓库**：htp（conversation-service）
- **描述**：创建 `app/core/brain_port.py`，定义 `BrainPort` Protocol、`BrainTextChunk`、`BrainStageUpdate`、`BrainEscalation`、`BrainUnavailableError`。这是整个重构的类型基础。
- **文件变更**：
  - `backend/conversation-service/app/core/brain_port.py`（新增，约 50 行）
- **验收标准**：
  - [ ] `isinstance(HTPBrainAdapter(), BrainPort)` 和 `isinstance(OpsAgentBrainAdapter(), BrainPort)` 均返回 True（Protocol 静态检查）
  - [ ] `from app.core.brain_port import BrainPort, BrainEvent` 无导入错误
- **依赖**：无
- **预计耗时**：0.5 天

---

### T1-4：创建 HTPBrainAdapter（逻辑搬家）

- **仓库**：htp（conversation-service）
- **描述**：创建 `app/adapters/htp_brain_adapter.py`，将 `conversation_service.py` 中 `send_message_stream_only()` 的大脑逻辑**搬家**到此。不修改任何业务逻辑，只是换了容身之所，然后实现 `BrainPort` 接口。
- **文件变更**：
  - `backend/conversation-service/app/adapters/htp_brain_adapter.py`（新增，逻辑从 conversation_service.py 迁入）
  - `backend/conversation-service/app/services/conversation_service.py`：原有逻辑替换为对 `HTPBrainAdapter.process()` 的委托调用
- **验收标准**：
  - [ ] 搬家前后，所有现有 E2E 测试和集成测试通过，零回归
  - [ ] openclaw 模式、glm 模式的行为与搬家前完全一致（对用户不可见）
  - [ ] `conversation_service.py` 的 `send_message_stream_only` 行数减少 ≥ 60%
- **依赖**：T1-3
- **预计耗时**：1 天

---

### T1-5：创建 OpsAgentBrainAdapter（反腐层）

- **仓库**：htp（conversation-service）
- **描述**：创建 `app/adapters/ops_agent_brain_adapter.py`，替换现有的 `OpsAgentAssistant` 类。新实现：传递 `session_id` 和 `hci_context`；接收真流式 SSE；解析 `x_stage_update` 扩展字段；ops-agent 不可达时 raise `BrainUnavailableError`。
- **文件变更**：
  - `backend/conversation-service/app/adapters/ops_agent_brain_adapter.py`（新增，约 80 行）
  - `backend/conversation-service/app/services/ai_client.py`：`OpsAgentAssistant` 标记废弃（不删除，观察期 1 周后删除）
- **验收标准**：
  - [ ] ops-agent 服务正常时，`OpsAgentBrainAdapter.process()` 正常 yield BrainTextChunk
  - [ ] ops-agent 服务不可达时，raise `BrainUnavailableError`（不抛原始 httpx 异常）
  - [ ] `session_id` 和 `hci_context` 正确出现在发出的 HTTP 请求 body 中
- **依赖**：T1-3, T1-2（ops-agent 侧先完成）
- **预计耗时**：0.5 天

---

### T1-6：创建 BrainRouter（含降级逻辑）

- **仓库**：htp（conversation-service）
- **描述**：创建 `app/adapters/brain_router.py`，实现 `BrainRouter` 类：根据 `assistant_type` 路由；ops-agent 不可达时自动降级到 htp 大脑；记录降级事件 metric。
- **文件变更**：
  - `backend/conversation-service/app/adapters/brain_router.py`（新增，约 50 行）
  - `backend/conversation-service/app/dependencies.py`：注入 `BrainRouter` 实例到 `ConversationService`
- **验收标准**：
  - [ ] `assistant_type=openclaw` 路由到 HTPBrainAdapter
  - [ ] `assistant_type=ops-agent` 路由到 OpsAgentBrainAdapter
  - [ ] `assistant_type=unknown` 降级到 HTPBrainAdapter（不报错）
  - [ ] ops-agent 不可达时，BrainRouter 自动降级，用户侧无报错（仅日志警告）
- **依赖**：T1-4, T1-5
- **预计耗时**：0.5 天

---

### T1-7：scheduler 跳过 ops-agent

- **仓库**：htp（conversation-service）
- **描述**：`_resolve_pod_endpoint()` 对 `assistant_type=ops-agent` 直接返回配置文件中的静态地址，跳过 scheduler 调用，消除不必要的调度延迟。
- **文件变更**：
  - `backend/conversation-service/app/services/conversation_service.py`：3 行条件判断
- **验收标准**：
  - [ ] ops-agent 模式下，日志中无 `scheduler_client.allocate_pod` 调用记录
- **依赖**：T1-6
- **预计耗时**：0.5 天

---

### T1-8：Phase 1 集成验证

- **描述**：端到端验证 Phase 1 完整功能
- **验收场景**：
  - [ ] 场景 A（多轮连续性）：新建 ops-agent 对话 → 第 1 轮描述 "VPN 隧道断了" → 第 3 轮回复 ops-agent 仍处于同一诊断线程，没有重新进入 intake
  - [ ] 场景 B（大脑隔离）：新建 openclaw 对话并完成 S0 选择 → 不影响同时进行的 ops-agent 对话
  - [ ] 场景 C（降级）：停掉 ops-agent 服务 → ops-agent 对话自动切换到 htp 大脑，用户收到"临时使用备用助手"提示
  - [ ] 场景 D（流式）：ops-agent 回复在 3 秒内出现首 token
- **依赖**：T1-1 到 T1-7

## Phase 2：接通手脚（实时数据 + 会话持久化）

> **目标**：ops-agent 能看到实时告警；服务重启后会话不丢失  
> **前置条件**：Phase 1 完成并验证稳定  
> **预计工作量**：2-3 天

### T2-1：htp 侧 env_context 预取与注入

- **仓库**：htp（conversation-service）
- **描述**：`send_message_stream_only` 在调用 `brain_router` 前，对 `assistant_type=ops-agent` 的请求预取 `active_alerts`（SCP）和 `failed_tasks`（SCP），构造 `HCIContext` dict，传给 `OpsAgentBrainAdapter.process(env_context=...)`。
- **文件变更**：
  - `backend/conversation-service/app/services/conversation_service.py`：`_prefetch_env_context()` 方法，约 30 行
  - `backend/conversation-service/app/adapters/ops_agent_brain_adapter.py`：`hci_context` 字段序列化
- **验收标准**：
  - [ ] ops-agent 模式下，debug 日志包含预取的告警数量
  - [ ] ops-agent 返回内容中能引用预取的告警（不再问用户"告警是什么"）
- **预计耗时**：0.5 天

---

### T2-2：ops-agent Redis 会话持久化

- **仓库**：ops-agent
- **描述**：将 T1-2 中的 LRU 内存存储升级为 Redis（TTL 4 小时）。支持降级：Redis 不可达时自动回退到内存存储。
- **文件变更**：
  - `ops_agent/server/session_store.py`：`RedisSessionStore`（主）+ `InMemorySessionStore`（降级）
  - `ops_agent/server/openai_compat.py`：session_store 通过依赖注入配置
  - `ops_agent/utils/config.py`：增加 `redis_url` 配置项（可选）
- **验收标准**：
  - [ ] ops-agent 服务重启后，使用相同 session_id 的对话能恢复之前的上下文
  - [ ] Redis 不可用时，自动降级到内存存储，日志告警，不中断服务
- **预计耗时**：1 天

- **文件变更**：

- **验收标准**：
  - [ ] 创建对话时可选择"htp 大脑"或"ops-agent 大脑"
  - [ ] 选择结果通过 `assistant_type` 字段传给 API，存入 `conversation.assistant_type`
  - [ ] 对话界面能清晰区分当前使用的大脑类型

## Phase 4：工具 Bridge（选做）

> **目标**：ops-agent 引导的 acli 命令由 htp 自动执行  
> **前置条件**：Phase 1-3 完成，且 A/B 测试结论显示 ops-agent 大脑更优  
> **预计工作量**：3-5 天  
> **注意**：此阶段对架构改动较大，需在 Phase 1-3 稳定后单独立项评估

### T4-1：ToolBridgeRequest SSE 事件（ops-agent 侧）

- **仓库**：ops-agent
- **描述**：ops-agent 调用 `get_info_from_user` 且问题是"执行某命令并告诉我结果"时，改为 yield 一个结构化的 `ToolBridgeRequest` SSE 事件，而不是直接问用户。

### T4-2：ToolBridge 执行器（htp 侧）

- **仓库**：htp（conversation-service）
- **描述**：`OpsAgentBrainAdapter` 识别 `x_tool_bridge_request` 扩展 SSE 事件，调用 `ToolRouter` 执行对应的 acli 命令，将结果通过续传请求发回 ops-agent。
