### T1：Schema 迁移

**目标**：修改 `database/desired_schema.sql`，Atlas 生成迁移文件

**变更内容**：
- `conversation` 表增加字段：
  - `sop_document_id integer REFERENCES sop_document(id) ON DELETE SET NULL`

- `kbd_entry` 表增加字段：
  - `hit_count integer NOT NULL DEFAULT 0`
- `sop_document` 表增加字段：
  - `hit_count integer NOT NULL DEFAULT 0`

**执行步骤**：
```bash
# 1. 修改 desired_schema.sql
# 2. Atlas diff 生成迁移文件
atlas migrate diff kb_hit_count_redesign \
  --dir "file://database/migrations" \
  --to "file://database/desired_schema.sql" \
  --dev-url "docker://postgres/15/dev"
# 3. 检查生成的迁移文件，确认内容正确
```

**完成标准**：`atlas migrate validate` 通过，迁移 SQL 内容与设计一致

### T3：kb-service 新增 SOP hit 接口

**目标**：实现 `POST /api/kb/sop/{document_id}/hit`

**文件位置**：`backend/kb-service/app/routes/`（建议新建 `hits.py` 路由文件）

**接口规范**：
```
POST /api/kb/sop/{document_id}/hit
Header: Authorization: Bearer <internal-token>
Response 200: { "success": true, "document_id": 5, "hit_count": 12 }
Response 404: { "error": "sop_document not found" }
实现：UPDATE sop_document SET hit_count = hit_count + 1 WHERE id = :id RETURNING hit_count
日志：structlog.info("sop_hit_count_incremented", document_id=..., new_hit_count=...)
```

**完成标准**：`pytest tests/unit/kb-service/test_sop_hit.py` 通过

### T4：kb-service 新增 KBD hit / decrement 接口

**目标**：实现 `POST /api/kb/kbd/{kbd_id}/hit` 和 `POST /api/kb/kbd/{kbd_id}/hit/decrement`

**文件位置**：`backend/kb-service/app/routes/hits.py`（同 T3）

**接口规范**：
```
POST /api/kb/kbd/{kbd_id}/hit
Response 200: { "success": true, "kbd_id": 123, "hit_count": 8 }
实现：UPDATE kbd_entry SET hit_count = hit_count + 1 WHERE id = :id RETURNING hit_count

POST /api/kb/kbd/{kbd_id}/hit/decrement
Response 200: { "success": true, "kbd_id": 123, "hit_count": 7 }
实现：UPDATE kbd_entry SET hit_count = GREATEST(0, hit_count - 1) WHERE id = :id RETURNING hit_count
日志：structlog.info("kbd_hit_count_decremented", kbd_id=..., new_hit_count=...)
```

**完成标准**：`pytest tests/unit/kb-service/test_kbd_hit.py` 通过

### T5：修复 kb_category.hit_count 去重逻辑

**目标**：`kb-service POST /api/kb/categories/{code}/hit` 接口的调用逻辑加 case 级去重

**注意**：接口本身不变，去重逻辑在 conversation-service 调用侧实现（T6 处理），但如需在接口侧做幂等（可选的防御层），可增加可选的 `case_id` 参数：

```
POST /api/kb/categories/{code}/hit
Body（可选）: { "case_id": "C202504200001" }
```

若携带 `case_id`，则接口检查 `conversation` 表中是否已存在该 `case_id` + `category_code` 的组合命中记录，若存在则直接返回当前 hit_count 不 +1（幂等接口）。

**注意**：此幂等逻辑可先不实现，依赖 T6 的调用侧去重即可（Phase 1 仅做 T6）。

**完成标准**：同一 case 重复调用该接口时 hit_count 不重复增加

### T8：S4 根因确认写 resolved_kbd_entry_id

**目标**：在 S4 根因确认阶段，将 AI 结构化输出中的 kbd_entry_id 写入 `conversation.resolved_kbd_entry_id`

**所在文件**：`backend/conversation-service/app/services/conversation_service.py`

**触发时机**：`diagnostic_item` type=`root_cause` 插入时

**实现思路**：
```python
async def _update_resolved_kbd(self, conversation_id, case_id, kbd_entry_id: int | None):
    if kbd_entry_id is None:
        # 新问题，未收录，记录到日志但不触发 hit
        logger.info("resolved_kbd_not_found", case_id=case_id)
        return
    await self.repo.update_conversation(conversation_id, {"resolved_kbd_entry_id": kbd_entry_id})
    await self.kb_client.post(f"/api/kb/kbd/{kbd_entry_id}/hit")
    logger.info("resolved_kbd_hit_incremented", case_id=case_id, kbd_entry_id=kbd_entry_id)
```

**AI 结构化输出解析**：S4 阶段的 AI 输出中需包含 `kbd_entry_id` 字段，从 S2/S3 检索的 KBD 候选列表中选择一条，或 null（当 AI 判断是新问题时）。需同步确认 S4 的 System Prompt 或输出结构是否已包含此字段（如无需补充）。

**完成标准**：S4 确认后 `conversation.resolved_kbd_entry_id` 正确写入，`kbd_entry.hit_count +1`
