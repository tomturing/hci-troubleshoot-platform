## 任务清单

### T1: 修复 runCommand() case_id 与 Bridge session key 不对齐

- **描述**：`CaseCreateDialog.vue` 的 `runCommand()` 函数中，发送 `ssh_input` 消息时将 `case_id` 由 `createdCaseId.value || 'temp'` 改为 `'ssh-create-temp'`
- **文件变更**：`frontend/customer/src/components/CaseCreateDialog.vue`（约第 198 行）
- **验收标准**：SSH 采集流程中，集群信息、告警列表、任务状态三条命令均能在 30s 内返回结果
- **依赖**：无
- **状态**：已完成 ✅

### T2: 将采集超时时间从 10s 改为 30s

- **描述**：`COLLECT_COMMANDS` 数组三条命令的 `timeout` 从 `10000` 改为 `30000`
- **文件变更**：`frontend/customer/src/components/CaseCreateDialog.vue`（约第 125-129 行）
- **验收标准**：命令能在 30s 内返回时不误判超时
- **依赖**：无
- **状态**：已完成 ✅

### T3: 采集失败时不入库错误占位符

- **描述**：`runCollectionAndUpsert()` 中，采集失败时保持 `collectedData[name]` 为 `null`，upsert 前加 `null` 检查，跳过失败项
- **文件变更**：`frontend/customer/src/components/CaseCreateDialog.vue`（约第 260-295 行）
- **验收标准**：采集失败后数据库中对应 env_type 无 `parse_error` 字段
- **依赖**：T1
- **状态**：已完成 ✅

### T4: 确认工单后同步写入 chatStore.currentCase

- **描述**：`runSshAndCreateCase()` 调用 `caseApi.confirm()` 成功后，立即将 `confirmed.data` 写入 `chatStore.currentCase`
- **文件变更**：`frontend/customer/src/components/CaseCreateDialog.vue`（约第 394 行）
- **验收标准**：SSH 流程关闭弹框后，顶部立即显示工单 ID badge，无需刷新
- **依赖**：无
- **状态**：已完成 ✅

### T5: createConversation() 异常向上传播

- **描述**：`chat.ts` 的 `createConversation()` catch 块在打印系统消息后，追加 `throw e`，确保异常传播到调用方
- **文件变更**：`frontend/customer/src/stores/chat.ts`（`createConversation` 函数）
- **验收标准**：对话创建失败时，系统显示错误提示，且 AI 不再发出空请求
- **依赖**：无
- **状态**：已完成 ✅

### T6: resumePendingCase 补调 collectEnvironmentData

- **描述**：`resumePendingCase()` 恢复工单并清除 pending 后，添加 `collectEnvironmentData(caseId).catch(() => {})`
- **文件变更**：`frontend/customer/src/stores/chat.ts`
- **验收标准**：刷新页面后恢复的工单，若已有环境数据，环境摘要卡片能正常显示
- **依赖**：无
- **状态**：已完成 ✅

### T7: confirmCreateCase 补调 collectEnvironmentData

- **描述**：`confirmCreateCase()` 确认工单后，添加 `collectEnvironmentData(res.data.case_id).catch(() => {})`（fire-and-forget）
- **文件变更**：`frontend/customer/src/stores/chat.ts`
- **验收标准**：纯文字创建工单时，若数据库已有历史环境数据，环境摘要能正常显示
- **依赖**：无
- **状态**：已完成 ✅

### T8: switchToCase 清空旧数据并补调 collectEnvironmentData

- **描述**：`switchToCase()` 设置 `currentCase` 后，先置空 `environmentContext.value = null`，再调用 `collectEnvironmentData(caseItem.case_id).catch(() => {})`
- **文件变更**：`frontend/customer/src/stores/chat.ts`
- **验收标准**：切换历史工单时，旧工单环境数据不残留，新工单的环境数据正确加载
- **依赖**：无
- **状态**：已完成 ✅

## 测试计划

### 人工测试场景

**测试 T1+T2：SSH 采集成功**
1. 打开创建工单弹框，填写工单信息，进入 SSH 认证
2. 完成 SSH 认证，点击"开始采集"
3. 观察日志，三条采集命令均应在 30s 内显示"✅ 集群信息采集完成"等字样
4. 打开数据库查看 `environment_data` 表，应有 `cluster`、`alert`、`task` 三条成功记录（无 `parse_error` 字段）

**完成标准**：三条命令全部成功，无超时报错，数据库记录正常。

**测试 T4：顶部工单 badge**
1. 通过 SSH 流程创建并确认工单
2. 弹框关闭后，立即观察页面顶部
3. 应立即显示工单 ID badge（如 `Q2026042497781`），无需刷新

**完成标准**：弹框关闭瞬间顶部出现工单 badge。

**测试 T5：AI 助手正常响应（纯文字流程）**
1. 在聊天框输入问题，点击发送（触发纯文字工单创建流程）
2. 工单确认后，等待 AI 响应
3. AI 应在合理时间内开始打字输出

**完成标准**：AI 有响应输出，不出现永久"正在连接..."状态。

**测试 T6-T8：环境数据显示**
1. 创建一个带环境数据的工单
2. 刷新页面 → 等待工单恢复 → 观察"环境摘要"卡片（测试 T6）
3. 点击历史工单列表，切换到该工单 → 观察环境摘要（测试 T8）
4. 纯文字创建一个新工单，工单关联同一主机 → 观察环境摘要（测试 T7）

**完成标准**：以上三种场景均显示环境摘要卡片，而非"未采集环境数据"提示。
