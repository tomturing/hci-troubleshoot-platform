# SSH 集成体验修复任务

## 背景

v2.5.0 合并后，SSH 集成存在以下问题：
1. **数据落库失败**：acli 输出含 ANSI 转义码导致 JSON 解析失败；`submitCollectedData` 只处理纯数组，`{entities:[]}` 格式被静默丢弃；错误用 `console.warn` 不可见
2. **交互割裂**：创建工单弹框和 SSH 终端按钮入口完全不同，体验不一致
3. **重复插入**：相同 `case_id + env_type` 数据每次采集都新增一条，无 upsert 机制

## 任务清单

### T-BE：后端 Environment Upsert 端点

**描述**：当前后端只有 `POST /api/environments/` 创建接口，无更新能力。需增加幂等 PUT 端点，使前端可以「有则更新，无则创建」。

**设计决策（业界最佳实践）**：
- REST 幂等语义：`PUT /api/environments/case/{case_id}/type/{env_type}` 
- 含义：「以此数据替换该 case+type 的环境记录」，首次调用创建，后续调用更新
- 实现：Repository 层先查询再写入（check-then-write），避免增加数据库 UNIQUE 约束

**文件变更**：
- `backend/case-service/app/repositories/environment_repo.py`：新增 `upsert_by_case_and_type()`
- `backend/case-service/app/services/environment_service.py`：新增 `upsert_environment()`
- `backend/case-service/app/routes/environments.py`：新增 `PUT /case/{case_id}/type/{env_type}` 路由
- `backend/api-gateway/app/routes/environments.py`：新增 PUT 代理路由

**验收标准**：
- [ ] `PUT /api/environments/case/{caseId}/type/cluster` 首次调用 → 创建记录，返回 201
- [ ] 相同端点再次调用 → 更新已有记录（env_data 更新，environment_id 不变），返回 200
- [ ] 无效 env_type → 返回 400
- [ ] 无效 case_id → 返回 404

**依赖**：无
**状态**：✅ 已完成

### T1：数据采集落库 Bug 修复（store 层）

**描述**：修复 `frontend/customer/src/stores/chat.ts` 中 SSH 采集数据不落库的多个 Bug，并改用 upsert。

**具体 Bug 及修复**：

| Bug | 根因 | 修复方案 |
|-----|------|---------|
| JSON 解析失败 | acli 输出包含 ANSI 控制码 | 新增 `stripAnsi()` 函数，解析前清洗 |
| 告警/任务数据丢弃 | 只接受纯数组，`{alerts:[]}` 格式被忽略 | 重写 `submitCollectedData`，兼容包装对象和纯数组 |

| 全量日志缺失 | 仅关键节点有日志 | 所有操作添加 `console.log` |
| cluster 解析截断 | `split(':')` 多冒号截断 | 改用 `indexOf(':')` 取第一个冒号 |
| 重复插入 | 无 upsert 机制 | 改用 `environmentApi.upsert()` |

**文件变更**：
- `frontend/customer/src/stores/chat.ts`：
  - 新增 `stripAnsi()` 函数
  - 重写 `parseJsonOutput()` - 支持贪婪提取
  - 重写 `submitCollectedData()` - 全量日志 + upsert + 兼容多格式
  - 修复 `parseClusterOutput()` - 先 stripAnsi + indexOf 解析

**验收标准**：
- [ ] SSH 采集后，数据库中有 cluster/alert/task 三条记录
- [ ] 第二次采集（相同 case_id）→ 更新已有记录，不新增

### T2：SSH 流程面板组件（新建）

**描述**：创建 `SshFlowPanel.vue`，提取 SSH 连接 → acli 检测 → 采集的核心流程，供两个入口复用。

**设计**：
```
SshFlowPanel.vue
├── Props
│   ├── mode: 'create-case' | 'terminal-only'
│   └── caseId: string | null
├── 内部流程
│   ├── 挂载时：检测 Bridge（checkBridgeRunning）
│   ├── 自动填充 localStorage 中上次的 SSH 配置（不含密码）
│   ├── 阶段步骤（el-steps）：连接 → 检测 acli → 采集 → 完成
│   ├── 折叠日志面板（默认折叠）
│   ├── 错误恢复：显示错误原因 + 检查项提示 + 重试按钮
│   └── 成功后：disconnect 临时 socket → connectSSH（建全局终端连接）
├── Emits
│   ├── success: SSH 连接+采集成功
│   └── cancel: 用户取消
└── 特性
    ├── 'create-case' 模式：SSH 成功后调用 completeCaseCreationFlow
    └── 'terminal-only' 模式：仅建立全局 SSH 连接，不创建对话
```

**重要设计决策**：
- SshFlowPanel 维护自己的临时 WebSocket 用于命令执行（marker 协议）
- 采集完成后 disconnect 临时 socket，再调 `chatStore.connectSSH()` 建全局终端 socket
- 无 SSH 创建按钮**仅在 create-case 模式**显示
- 日志面板**默认折叠**，用户可手动展开

**文件变更**：
- 新建 `frontend/customer/src/components/SshFlowPanel.vue`

**验收标准**：
- [ ] Bridge 未运行时：显示警告 + 下载按钮 + 重新检测按钮，"连接"按钮禁用
- [ ] Bridge 运行时：显示 SSH 表单（自动填充上次配置），"连接"按钮可用
- [ ] SSH 连接成功后：自动运行 acli 检测
- [ ] acli 可用：自动运行三条采集命令并 upsert 到数据库
- [ ] acli 不可用：提示跳过采集，仍建立终端连接
- [ ] SSH 失败：显示错误信息、检查项列表、重试按钮
- [ ] 日志面板默认折叠，点击展开

**依赖**：T1, T-SDK
**状态**：✅ 已完成

## 测试计划

### 人工测试步骤

#### 场景 A：Bridge 未运行，无工单
1. 点击 [SSH终端] → 打开 CaseCreateDialog（基本信息表单）
2. 填写标题、描述
3. 点击"连接并创建工单" → 工单创建成功 → 打开 SshConnectDialog
4. Bridge 提示未运行，显示下载按钮

#### 场景 B：Bridge 已运行，无工单，SSH 成功
1. 点击 [SSH终端] → 打开 CaseCreateDialog
2. 填写标题、描述 → 点击"连接并创建工单"
3. 工单创建 → 打开 SshConnectDialog（SSH 表单自动填充）
4. 填写密码 → 点击"连接"
5. 观察：步骤条推进（连接 → 检测 acli → 采集 → 完成）
6. 验证：数据库有 cluster/alert/task 三条记录
7. 验证：终端自动打开，可正常交互

#### 场景 C：有工单，SSH 未连接，点击 [SSH终端]
1. 已有活跃工单（无 SSH 连接）
2. 点击 [SSH终端] → 打开 SshConnectDialog（terminal-only 模式）
3. SSH 连接成功 → acli 采集 → 数据 upsert（数据库更新，不新增）
4. 验证：终端打开，数据库记录 updated_at 更新

#### 场景 D：有工单，SSH 已连接，点击 [SSH终端]
1. 已有活跃工单且 SSH 已连接
2. 点击 [SSH终端] → 直接展开终端侧边栏（不显示任何弹框）

#### 场景 E：无 SSH 创建工单
1. 点击 [SSH终端] → CaseCreateDialog
2. 点击"无 SSH 创建"（悬浮提示出现）
3. 确认 → 工单创建并开始对话（无 SSH 数据）

#### 场景 F：SSH 连接失败后关闭弹框（回退测试）
1. create-case 模式 SSH 连接失败
2. 关闭 SshConnectDialog
3. 验证：对话自动启动（工单已创建，仅无 SSH 数据）

#### 场景 G：第二次采集 upsert 验证
1. 有工单，已有环境数据
2. 再次连接 SSH 并采集
3. 验证：`environment` 表记录条数不变，数据被更新（environment_id 相同，env_data 更新）
