### T3：修复 S0→S1 阶段切换（三处错误）

- **描述**：
  本任务修复三处彼此独立但协同造成"S0 之后永远不切换"的错误：

  **3-a**：更新 `STAGE_TRIGGERS[("S0","S1")]`
  在现有触发词数组后追加 3 条新模式，匹配候选被确认后 AI 的推进语句。

  **3-b**：解耦 `_update_conversation_category` 写入
  将该调用从 `if new_stage:` 块内移出，改为独立判断 `if current_stage == "S0" and category_info:`，保证只要 AI 输出了分类信息就写入，不依赖阶段是否同时触发转换。

  **3-c**：接入候选确认链路
  在 `send_message_stream_only()` 的步骤 2（读取当前阶段）之后、步骤 3（构建 Prompt）之前，新增 S0 候选预处理逻辑：
    - 解析用户输入是否为 ①②③ 序号（`parse_candidate_selection`）
    - 若是序号，从上一条 AI 消息中提取候选列表（新增 `_extract_s0_candidates` 方法）
    - 用户确认有效分类 → 直接写库 + 强制推进 S1，不调用 AI，yield 确认语句并 return
    - 用户选 ③ → 检查轮次阈值，超限触发 `handle_s0_failure()`，否则递增轮次继续让 AI 重跑
    - 解析失败（非序号，自然语言补充）→ 正常走 AI，让 AI 基于补充信息重新给候选

  **辅助方法**（均在 `ConversationService` 中新增）：
  - `_extract_s0_candidates(conversation_id) -> list[dict]`：从上一条 assistant 消息中用正则提取 ① ② 对应的 `{code, name}`
  - `_get_s0_candidate_rounds(conversation_id) -> int`：读取 `conversation.metadata_["s0_candidate_rounds"]`，默认 0
  - `_increment_s0_candidate_rounds(conversation_id) -> None`：原子 +1 写回

- **文件变更**：
  - `backend/conversation-service/app/services/conversation_manager.py`（3-a：STAGE_TRIGGERS）
  - `backend/conversation-service/app/services/conversation_service.py`（3-b：解耦写入；3-c：候选预处理及辅助方法）

- **验收标准**：
  - [ ] 用户回复 "①" 后，`conversation.diagnostic_stage` 从 `S0` 变为 `S1`
  - [ ] `conversation.category_id` 非 NULL
  - [ ] 用户两轮均选 "③" 后，`conversation.diagnostic_stage` 变为 `S0_FAILED`，`case.status` 变为 `in_progress`
  - [ ] 单元测试 `tests/unit/test_s0_candidate_mode.py` 通过

- **依赖**：T2（T2 修改 Prompt，确保 AI 不再输出直接标记，T3 才能稳定工作）
- **状态**：待开始
