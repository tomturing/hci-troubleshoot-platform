# Task 32：prompt_audit 写入逻辑落地与联调（P2）

```

Step 6：质量评分联调确认

# Task 33：prompt_audit 数据质量看板与告警（P3）

```
你是一名负责 hci-troubleshoot-platform 数据可观测性与质量监控的后端 / 平台工程 agent。

【仓库】
git clone https://github.com/tomturing/hci-troubleshoot-platform.git
cd hci-troubleshoot-platform

【背景】
在 Task 32 完成后，prompt_audit 将成为评分体系中 ai_quality 维度的唯一真实数据源。
为防止未来改动导致 prompt_audit 再次出现大量 NULL / 写入失败，需要一个轻量的数据质量看板与告警机制。

【任务目标】
1. 为 prompt_audit 关键字段设计并实现最小可用的数据质量监控指标
2. 在 Grafana 中提供一个简单面板，用于观察：
   - 每日 prompt_audit 写入条数
   - has_sop / kb_chunks_count / kb_top_score 非空占比
   - messages 采样占比
3. 为以下异常场景提供告警能力（可先以日志/邮件/Slack Webhook 的形式呈现）：
   - 连续 N 分钟 prompt_audit 写入为 0
   - has_sop / kb_chunks_count 非空占比 < 80%
   - messages 非空占比 > 30%（采样率被意外改高）

【实现建议】

Step 1：数据质量统计 SQL / 视图
- 在 docs/02_数据库设计.md 或独立 SQL 文件中补充数据质量统计查询示例，例如：
  - 近 7 天按日汇总：
      SELECT
          date_trunc('day', captured_at) AS day,
          COUNT(*) AS total,
          AVG((has_sop IS NOT NULL)::int) AS has_sop_not_null_ratio,
          AVG((kb_chunks_count IS NOT NULL)::int) AS kb_count_not_null_ratio,
          AVG((messages IS NOT NULL)::int) AS messages_sample_ratio
      FROM prompt_audit
      WHERE captured_at >= NOW() - INTERVAL '7 days'
      GROUP BY 1
      ORDER BY 1;

Step 2：Prometheus 导出
- 复用现有 metrics 方案，在 case-service 或 conversation-service 增加一个周期性任务（如 APScheduler / 后台协程）：
  - 每 60 秒执行一次统计 SQL，将结果写入 Prometheus Gauge：
      - hci_prompt_audit_daily_total
      - hci_prompt_audit_has_sop_not_null_ratio
      - hci_prompt_audit_kb_count_not_null_ratio
      - hci_prompt_audit_messages_sample_ratio
- 确保：
  - 未安装 prometheus_client 时，统计任务自动降级为仅日志输出
  - 统计 SQL 出错不会影响主业务流程

Step 3：Grafana 面板与告警规则
- 在现有 Grafana Dashboard 中新增一个「Prompt Audit 数据质量」分组：
  - 折线图：hci_prompt_audit_daily_total
  - 折线图：*_ratio 指标（0-1 或 0-100）
- 为以下规则配置告警（可按环境调整阈值）：
  - has_sop_not_null_ratio < 0.8 持续 10 分钟
  - kb_count_not_null_ratio < 0.8 持续 10 分钟
  - messages_sample_ratio > 0.3 持续 10 分钟

【约束】
- 不修改 prompt_audit 表结构
- 不在主请求链路中增加阻塞性统计逻辑，所有统计应在后台异步执行
- 指标命名需与现有 hci_* 指标前缀保持一致

【验收标准】
- [ ] 在 Prometheus 中能查询到 hci_prompt_audit_* 指标，且随对话量变化
- [ ] Grafana 看板「Prompt Audit 数据质量」渲染正常，能看出最近 7 天内的写入量与比例
- [ ] 人为停用 conversation-service 一段时间后，告警规则能检测到写入为 0 的异常
- [ ] 将 messages 采样率临时调高到 50% 时，messages_sample_ratio 指标明显上升，并触发预期告警
```

# Task 34：Customer UI 聊天消息渲染改造（Markdown 友好呈现 + 结构化输出）（P1）

```
你是一名负责 hci-troubleshoot-platform 前端 customer 应用（custom-ui）体验改造的 agent。

【仓库】
git clone https://github.com/tomturing/hci-troubleshoot-platform.git
cd hci-troubleshoot-platform/frontend

【背景】
当前 Customer UI 的聊天消息呈现存在明显体验问题（见需求截图）：
1) AI 输出经常“整段打包”展示，缺少分段与结构化排版，用户难以阅读
2) AI 原输出为 Markdown，但 UI 未做友好渲染（代码块/列表/标题/引用均未正确呈现）

这些问题会导致排障流程的可读性显著下降，命令与风险提示也难以被准确识别。

【任务目标】
1. 将 AI 消息内容按 Markdown 规范进行渲染，至少支持：标题、列表、引用、粗体、链接、代码块（含复制按钮）
2. 解决“整体打包输出”导致的杂乱问题：对长输出进行视觉分段（段落间距、分隔线、可折叠区域等）
3. 为后续命令卡片（Task 35）预留结构化容器：能够在 Markdown 渲染结果中识别并挂载「命令块」组件

【涉及服务 / 文件范围】
- frontend/customer/src/**（聊天消息渲染相关组件与样式）
- frontend/shared/**（如存在共享 Markdown 渲染工具可复用）

【详细实现步骤】

Step 1：定位当前消息渲染实现
- 在 customer 应用中找到聊天消息展示组件（如 ChatWindow / MessageItem / Markdown 相关组件）
- 确认目前是否是简单的纯文本渲染（v-html / innerText / pre 标签等）

Step 2：引入 Markdown 渲染能力（不得破坏安全性）
- 选择一种稳定的 Markdown 渲染方案（优先：现有依赖；否则引入轻量渲染库）
- 要求：
  - 支持 fenced code block（```）
  - 支持语法高亮（可选，但推荐）
  - 必须做 XSS 安全处理（禁止任意 HTML 注入；或白名单 sanitize）

Step 3：改造 AI 消息排版与分段
- 对长文本增加更清晰的排版层级：
  - 段落间距、列表缩进、引用样式
  - 代码块独立卡片化展示
  - 可选：对“假设 / 风险提示 / 命令清单 / 预期输出”这类小节自动加视觉强调（仅样式，不改动内容）

Step 4：为命令块挂载预留扩展点
- 在 Markdown 渲染阶段识别“命令块”的统一格式（与 Task 35 对齐）：
  - 例如：代码块语言为 bash/sh/shell 的 fenced block 作为候选命令块
  - 或：约定三段式结构：命令、命令说明、风险提示（仅识别，不强制）
- 在渲染结果中将命令块替换为 `<CommandBlock />` 组件（组件先占位即可）

Step 5：测试与验收
- 用截图中的典型 AI 输出（含标题、列表、代码块）做本地验收
- 重点检查：
  - 代码块换行、缩进正确
  - 中文标点与空格不被异常折叠
  - 滚动与复制体验不退化

【约束】
- 不允许将 AI 输出当作 HTML 直接渲染导致 XSS 风险
- 不引入体积过大的 Markdown 渲染依赖（如确需新增依赖，解释体积与收益）
- 代码注释使用中文
- 前端包管理使用 pnpm

【验收标准】
- [ ] 同一条 AI 消息包含标题/列表/引用/代码块时，均能正确渲染
- [ ] 代码块具备复制按钮（复制结果与原始命令一致）
- [ ] 长输出阅读体验明显提升（段落层级清晰，不再“糊成一坨”）
- [ ] 无明显 XSS 风险（不得支持任意 HTML 注入执行）

# Task 35：命令卡片化与“一键发送到终端”交互（P1）

```
你是一名负责 hci-troubleshoot-platform 前端 customer 应用（custom-ui）命令交互体验的 agent。

【仓库】
git clone https://github.com/tomturing/hci-troubleshoot-platform.git
cd hci-troubleshoot-platform/frontend

【背景】
当前 AI 建议命令以纯文本形式展示，用户需要手动复制/粘贴，且容易误操作。

- 每条命令以独立“命令块/卡片”呈现
- 卡片上有“执行/发送到终端”按钮
- 点击后将命令发送到终端输入框，用户只需敲回车即可执行（避免自动执行带来的风险）

【任务目标】
1. 将 AI 输出中的命令统一渲染为命令卡片（CommandBlock）
2. CommandBlock 提供按钮：一键将命令“发送到终端输入区”（仅填充，不自动执行）
3. CommandBlock 支持：
   - 复制命令
   - 展示命令说明（可折叠）
   - 展示风险提示（明显视觉提示）

【涉及服务 / 文件范围】
- frontend/customer/src/**（消息渲染、命令块组件、终端输入区组件）
- frontend/shared/**（若复用类型或工具）

【详细实现步骤】

Step 1：定义命令识别规则（与 Task 34 对齐）
- 明确“哪些文本算命令块”，优先从 Markdown fenced code block（bash/sh/shell）识别
- 对于多命令场景，需逐条拆分为多个命令块（不合并成一块）

Step 2：实现 CommandBlock 组件
- UI 要求：
  - 命令显示为等宽字体、可水平滚动、可复制
  - 提供主按钮：「发送到终端」
  - 可选按钮：「复制」
  - 可展开区域：命令说明/预期输出/风险提示

Step 3：实现“发送到终端”的数据通路
- 在 customer UI 中实现一个“终端输入区”（可能在侧边栏/底部抽屉/弹窗中，具体与 Task 36 对齐）
- 点击“发送到终端”时：
  - 将命令字符串写入终端输入框（光标置尾）
  - 不触发实际执行
  - 若终端未打开，则自动打开终端面板并聚焦输入框

Step 4：适配 AI 输出中的“风险提示/只读提示”
- 若 AI 文本中包含“只读查询/无风险/风险提示”等关键字，映射到 CommandBlock 的风险标签（仅前端标注，不改变原文）

Step 5：联调与验收
- 手动验证 3 种典型命令：
  - 单行命令
  - 多行命令（含 \ 续行）
  - 带变量占位符的命令（如 ${vmid}），需要保持原样

【约束】
- 点击按钮不得自动执行命令（必须要求用户敲回车）
- 不允许静默更改命令内容（包括去掉换行/去掉反斜杠等）
- 代码注释使用中文
- 前端包管理使用 pnpm

【验收标准】
- [ ] 命令在 UI 中以卡片形式展示，结构清晰
- [ ] 点击「发送到终端」后，命令出现在终端输入框中，用户敲回车即可执行
- [ ] 多条命令能拆分为多个卡片，逐条发送
- [ ] 命令复制内容与显示内容一致（包含换行与空格）

# Task 36：侧边栏 SSH 登录与交互终端页面（P1）

```
你是一名负责 hci-troubleshoot-platform 前端 customer 应用（custom-ui）侧边栏终端能力的 agent。

【仓库】
git clone https://github.com/tomturing/hci-troubleshoot-platform.git
cd hci-troubleshoot-platform/frontend

【背景】
需求新增一个侧边栏 SSH 登录页面，用于：
1) AI 命令的交互（接收 Task 35 的“发送到终端”命令）
2) 用户登录 HCI 后台，自行执行/补充命令并将输出反馈给 AI

当前系统的聊天与后端微服务通过 WebSocket / SSE 进行消息流转，但前端尚无“终端交互面板”作为命令执行承载。

【任务目标】
1. 新增侧边栏页面/抽屉：包含 SSH 登录表单（主机、端口、用户名、认证方式等）与终端交互区
2. 终端交互区具备最小能力：
   - 输入命令（支持粘贴、支持接收 Task 35 的一键填充）
   - 展示输出（按行、可复制、可清空）
3. 与后端打通一条“SSH 会话/执行”通路（若后端未实现，先以 mock / 预留接口对齐为准）

【涉及服务 / 文件范围】
- frontend/customer/src/**（侧边栏 UI、状态管理、终端组件）
- backend/**（如需新增 SSH 相关 API，则必须拆为单独 Task，不在本 Task 内实现）

【详细实现步骤】

Step 1：梳理交互与安全边界
- 明确：浏览器端不应直连 SSH（安全与网络限制），应通过后端代理（WebSocket）
- 本任务前端侧先完成 UI + 状态机 + 接口契约；后端能力若缺失，需在文档中明确依赖 Task（见 Step 4）

Step 2：实现侧边栏 UI 与状态管理
- 新增入口按钮（如工具栏/右侧悬浮按钮）打开侧边栏
- 侧边栏包含：
  - SSH 登录表单（保存为 session 内状态）
  - 连接状态（未连接/连接中/已连接/断开/错误）
  - 终端输出区（滚动、复制、清空）
  - 命令输入框（聚焦、发送）

Step 3：与命令卡片联动（对齐 Task 35）
- 提供一个统一的 store/action：`sendCommandToTerminalInput(command: string)`
- CommandBlock 点击后调用该 action，将命令填入侧边栏的输入框并自动展开侧边栏

Step 4：定义前后端接口契约（如后端未实现则追加后端 Task）
- 建议接口（示例）：
  - POST /api/terminal/sessions    创建会话（返回 session_id）
  - WS  /ws/terminal/{session_id}  发送命令/接收输出（流式）
  - POST /api/terminal/sessions/{id}/close  关闭会话
- 若仓库中尚无类似 API，需要在本文件新增一个后端 Task（Task 37）实现代理与鉴权

Step 5：验收
- UI 层验收：
  - 侧边栏可打开/关闭，登录表单交互正常
  - “发送到终端”可把命令填到输入框
- 若后端已实现：
  - 输入命令后能看到输出流式追加
  - 断开/重连流程正常

【约束】
- 前端不得直接实现 SSH 连接（必须走后端代理）
- 不在前端保存长期明文密码（若需记忆，必须使用短期 session + 明确清除）
- 代码注释使用中文
- 前端包管理使用 pnpm

【验收标准】
- [ ] 侧边栏 SSH 登录页面可用（表单 + 状态 + 终端区）
- [ ] Task 35 的命令卡片能一键填充到终端输入框
- [ ] 终端输出区支持复制与清空
- [ ] 明确记录后端依赖（若后端未实现则必须新增 Task 37）
```

# Task 37：SSH 代理与终端交互后端能力（API + WS）（P1）

```
你是一名负责 hci-troubleshoot-platform 后端（建议 api-gateway 或 conversation-service）终端代理能力的 agent。

【仓库】
git clone https://github.com/tomturing/hci-troubleshoot-platform.git
cd hci-troubleshoot-platform

【背景】
Customer UI 需要一个“侧边栏 SSH 终端”用于执行 AI 建议命令并将输出反馈给用户/AI。
浏览器端无法安全直连 SSH，因此必须由后端提供代理能力：
- 建立 SSH 会话
- 转发命令
- 以 WebSocket 流式返回 stdout/stderr
- 做鉴权与审计（trace_id）

【任务目标】
1. 提供创建/关闭终端会话的 HTTP API
2. 提供终端交互的 WebSocket 端点（发送命令 → 流式输出）
3. 记录必要审计信息（trace_id、client_id、case_id（如可获得）、连接目标脱敏）

【涉及服务 / 文件范围】
- backend/api-gateway/**（若作为统一 WS 入口）
- backend/conversation-service/**（若复用现有 WS/SSE 会话管理）
- backend/shared/**（若需新增通用工具类/审计结构）

【详细实现步骤】

Step 1：选择落点与路由方式
- 若 api-gateway 已是 WebSocket 统一入口，优先落在 api-gateway
- 若 conversation-service 更贴近 case/conversation 上下文，则可落在 conversation-service，并由 gateway 代理
- 输出一张最小数据流：
  UI → gateway(ws) → terminal-proxy → ssh server → output stream → UI

Step 2：实现会话管理（最小可用）
- POST /api/terminal/sessions
  - 入参：host, port, username, auth（password 或 keyRef）
  - 出参：session_id
- POST /api/terminal/sessions/{id}/close
  - 关闭 SSH 连接与后台资源
- 会话存储：
  - 仅保存在内存或 Redis（推荐 Redis，避免服务重启丢失）
  - 设置 idle timeout 自动回收

Step 3：实现 WebSocket 交互协议
- WS /ws/terminal/{session_id}
- 客户端消息：
  {"type":"stdin","data":"ls -la\n"}
- 服务端消息：
  {"type":"stdout","data":"..."}
  {"type":"stderr","data":"..."}
  {"type":"status","state":"connected|disconnected|error","message":"..."}
- 注意：
  - 必须区分 stdout/stderr
  - 输出需流式追加（不要整段缓存后一次性发）

Step 4：安全与审计
- 鉴权：
  - 复用现有 gateway token / session 机制
  - 防止任意外部 host 被滥用（建议加 allowlist / 网段限制 / 管理员开关）
- 审计：
  - 所有 API/WS 消息日志携带 trace_id
  - 连接目标 host 做脱敏或哈希存储

Step 5：测试与验收
- 单元测试：会话创建/关闭，WS 协议序列化
- 集成测试：连接一个测试 SSH 容器（或 mock server），发送命令并收到输出

【约束】
- 不允许将 SSH 凭据长期落库为明文
- 不允许浏览器端直连 SSH
- 代码注释使用中文

【验收标准】
- [ ] 能创建 session_id，并通过 WS 发送命令获得 stdout/stderr
- [ ] 连接断开/超时会回收资源，UI 能收到状态事件
- [ ] 审计日志包含 trace_id 且不泄露敏感凭据
- [ ] 与 Task 36 前端联调通过（命令一键填充 → 回车执行 → 输出可见）
```
