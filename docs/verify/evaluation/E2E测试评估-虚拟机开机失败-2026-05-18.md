## 二、AI 输出质量问题清单（基于 SOP branch-T 对比）

> **前提说明**：以下问题在"系统应已命中SOP"的前提下成立。
> 实际测试中 SOP 完全未被加载，AI 所有建议均为自由推理，偏差是必然结果。

### P0 - 严重（直接影响操作安全/正确性）

#### ISS-01：修复命令错误
- **AI 给出**: `kill -9 <PID>` 或 `kill <PID>`
- **SOP branch-T 要求**: `acli system kill <PID>`
- **风险**: Linux 通用 kill 命令在 HCI 平台可能无法正常终止 KVM 进程，且不经过 HCI 平台的进程管理层，可能导致状态不一致。

#### ISS-02：引入非 SOP 命令（virsh）
- **AI 额外提供**: `virsh list --all`、`virsh destroy <domain>` 等
- **SOP branch-T**: 无此命令，HCI 平台不使用 virsh 直接管理虚拟机
- **风险**: virsh 在 HCI 环境中可能无效或产生平台层面的状态不一致

#### ISS-03：遗漏高危操作授权警告
- **AI 表述**: 仅标注"请谨慎操作"
- **SOP branch-T 要求**: "执行前必须获得明确授权（高危）"
- **风险**: 未按规范执行授权流程，违反运维安全基线

### P1 - 重要（影响诊断准确性/流程规范性）

#### ISS-04：跳过 SOP check-1 验证步骤
- **AI 行为**: 未引导执行 `acli task get -v <vmid> -t <date> -k '虚拟机镜像忙' -s 'failed'`
- **SOP branch-T check-1**: 必须先查询任务失败记录，确认存在"虚拟机镜像忙"报错，才能进入 check-2
- **影响**: 跳过了分支确认逻辑，可能走错诊断路径

#### ISS-05：工作流进度条未随对话阶段推进
- **现象**: 全程停在第2步"信息确认"，未随 S2→S3→S4→S5 推进
- **影响**: 用户无法直观感知诊断进展，界面可观测性缺失

#### ISS-06：阶段跳跃，缺少用户交互确认
- **现象**: S2（问题确认）/S3（原因分析）/S5（解决方案）在单条消息中连续呈现
- **设计意图**: 每个阶段应等待用户确认，ReAct 执行器应逐阶段推进
- **影响**: 用户无法在中间节点介入纠正诊断路径

### P2 - 一般（影响诊断精度/完整性）

#### ISS-07：进程查找命令不精确
- **AI 给出**: `ps aux | grep qemu | grep <vmname>`（用名称）
- **SOP branch-T**: `ps aux | grep qemu | grep <vmid>`（用 VMID）
- **影响**: 虚拟机名称可能重复或含特殊字符，VMID 更精确

#### ISS-08：全程未引导采集环境数据
- **现象**: 界面始终显示"⚠️ 未采集环境数据"警告
- **影响**: AI 无实际环境信息辅助，全凭文字描述推断，诊断精度下降

#### ISS-09：缺少 solution-1.1 后验证步骤
- **AI 行为**: 给出 `kill` 命令后直接进入虚拟机重启步骤
- **SOP branch-T solution-1.1**: 执行进程终止后，应再次执行 `ps aux | grep qemu | grep <vmid>` 确认进程已消失
- **影响**: 无验证可能导致"进程未终止但已尝试重启"的隐患

#### 推导依据：基于第一性原理的 Token 预算分析

**第一步：确定模型有效上下文区间**

业界研究（Liu et al. 2023 "Lost in the Middle"；Hsieh et al. 2024 "RULER"）的核心结论：

| 输入 Token 规模 | LLM 推理能力表现 |
|---|---|
| < 4K | 可能缺乏足够上下文，诊断分支覆盖不完整 |
| **4K – 32K** | **最优区间**：指令遵循精度最高，SOP 决策树执行最准，检索准确率峰值 |
| 32K – 64K | 良好：GLM-4/5 在此区间仍保持有效利用，略有衰减 |
| > 64K | 衰减加速："Lost in the middle" 效应显著，相关内容被稀释 |

重要条件：SOP 内容位于 **system message 开头**（B1 段 = 系统提示第4段、紧靠前3个静态段），而非对话历史中间，因此不受 "Lost in the Middle" 问题影响。

**第二步：实测 Token 预算（GLM-5 128K context window）**

```
GLM-5 总 context：              128,000 tokens
减：最大输出 reserve：           -  4,096 tokens  （max_tokens 典型值）
减：A1+A2+A3+D1 静态段：        -    533 tokens  （~800 chars ÷ 1.5）
减：20 条历史消息窗口：          -  1,728 tokens  （avg 130 chars × 20 ÷ 1.5）
────────────────────────────────────────────────
可用于知识注入：                = 121,643 tokens
对应最大字符数：                ≈ 182,464 chars  （× 1.5 chars/token）

当前 SOP 全文：                   18,843 chars   =  12,562 tokens（9.8% 可用预算）
"最优区间" 上限对应字符数：        32,000 tokens × 1.5 = 48,000 chars
```

**第三步：确定 KB_CONTEXT_MAX_CHARS = 40,000 的依据**

```
目标：将全部知识注入后，总输入 token 保持在 4K–32K 最优区间内

总输入 = A1/A2/A3/D1(533) + SOP(≤26,667) + History(1,728) = ≤28,928 tokens
                                                              ← 在 32K 最优区间内 ✅

SOP 安全上限字符 = 26,667 tokens × 1.5 = 40,000 chars
                                         ↑ 即 KB_CONTEXT_MAX_CHARS = 40,000

当前 SOP(18,843) / 上限(40,000) = 47%  ← 留有 2.1x 增长余量
```

**第四步：SOP 类型的特殊要求（为何不能截断）**

SOP 是**决策树型程序性知识**，与一般文档截然不同：

- 普通 RAG 文档截断后仍能提取部分事实 → 尚可接受  
- SOP 截断后，未被加载的分支节点对 AI **完全不可见** → 诊断路径必然走错  
- "内部异常"分支（位置 2,190）在 2,000 字符截断时消失，AI 无从得知该分支的存在  
- **结论：对 SOP 类型知识，任何低于文档全长的上限设置都存在系统性错误风险**

**KB_CONTEXT_MAX_CHARS = 40,000 的完整理由**：

| 维度 | 数值 | 结论 |
|------|------|------|
| 当前 SOP 全文 | 18,843 chars | 完整覆盖，有余量 |
| 2.1x 增长余量 | 40,000 chars | 防止文档扩展后再次截断 |
| 注入后总输入 | ~28,928 tokens | 在 4K-32K 最优推理区间内 |
| GLM-5 可用预算占比 | 21.9%（28,928/128,000）| 远低于安全阈值 |

## 六、audit_log 完整性分析（第一性原理）

### 6.1 audit_log 记录了什么

`_write_prompt_audit` 实际存储的字段（`audit_log.payload`）：

| 字段 | 来源 | 说明 |
|------|------|------|
| `case_id` | 入参 | 工单 ID |
| `assistant_type` | 入参 | 助手类型（openclaw/glm 等）|
| `message_count` | 入参 | 历史消息总条数 |
| `has_sop` | `audit_meta` | 本次是否命中 SOP |
| `kb_chunks_count` | `audit_meta` | KBD 命中条数 |
| `kb_top_score` | `audit_meta` | KBD 最高分 |
| `messages` | `history_messages` | **完整 LLM 消息列表**（含 system_prompt + 最近20条历史）|
| `context_breakdown` | `audit_meta` | 各 Segment 字符/token 统计 |
| `total_chars` | `audit_meta` | system_prompt 总字符数 |
| `total_token_est` | `audit_meta` | 预估 token 数 |

**结论**：`audit_log.payload.messages` 忠实记录了发给 LLM 的完整消息列表，`messages[0]` 就是包含 SOP（截断前）内容的 system prompt。`ai_client.chat_completion_stream` 直接序列化 `messages` 不作修改，audit 与 LLM 实际收到的内容一致。

验证 SQL：
```sql
SELECT
  length(payload->'messages'->0->>'content') AS system_prompt_len,
  position('内部异常' IN (payload->'messages'->0->>'content')) AS pos_neibuyichang
FROM audit_log
WHERE payload->>'has_sop' = 'true'
ORDER BY started_at DESC LIMIT 1;
-- 修复 BUG-02 前：system_prompt_len=2803, pos=0（截断，"内部异常"缺失）
-- 修复 BUG-02 后：system_prompt_len≈19700, pos>0（完整，"内部异常"存在）
```

### 6.2 audit_log 缺失的字段（设计缺口）

`knowledge_retriever.retrieve()` 返回的 `audit_meta` 中有以下字段**未被写入** `audit_log.payload`：

| 缺失字段 | `audit_meta` 中有 | `payload` 中有 | 影响 |
|---------|---------|---------|------|
| `fallback_level` | ✅ | ❌ | 无法直接查询走了哪条轨道（SOP/KBD/机制推理），只能从 `context_breakdown` 的 B* code 间接推断 |
| `sop_document_id` | ✅ | ❌ | 无法从 audit_log 直接关联 `sop_document` 表，必须跨查 `conversation.sop_document_id` |
| `category_id` | ✅ | ❌ | 意图识别结果（如 "虚拟机-003"）不可查 |
| `category_score` | ✅ | ❌ | 意图识别置信度不可查 |
| `needs_review` | ✅ | ❌ | 是否需要人工审核的标志不可查 |
| `turn_index` | 表字段存在 | NULL | 无法关联到第几轮对话 |
| `system_prompt_id` | 表字段存在 | NULL | 无法关联 system_prompt 模板版本 |

### 6.3 潜在的浅拷贝风险

```python
# conversation_service.py
_sample_payload = history_messages           # ← 引用，非 deep copy
asyncio.create_task(_write_prompt_audit(..., sample_payload=_sample_payload))
# ... 若后续代码修改 history_messages，audit 记录将与实际发送内容不一致
brain_router.process(messages=history_messages)  # 当前不修改列表，暂无实际问题
```

当前 `HTPBrainAdapter` 和 `ai_client.chat_completion_stream` 均不修改 `messages` 列表，**实际无问题**。但这是隐性风险——未来若新增 adapter 在发送前修改消息，audit 将失真。

**建议修复**（IMP-05）：
```python
_sample_payload = [dict(m) for m in history_messages]  # 快照而非引用
```
