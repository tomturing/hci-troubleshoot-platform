## 一、环境管理

平台支持两种部署模式，**不可同时运行**（会产生端口与数据卷冲突），切换环境前请务必先停止当前环境。

### 1. 环境说明与访问方式

| 对比项 | Docker Compose 模式 | K3s 模式 (Helm) |
|---|---|---|
| **定位** | 本地开发、代码级调试、快速验证 | UAT测试、生产环境模拟、高可用验证 |
| **运行时** | Docker 本地容器 | K3s 集群 (containerd / docker runtime) |
| **入口网关** | 8000/3001/3002/3000 各服独立端口 | Traefik Ingress (统一 80/443 端口) |
| **外部 AI** | 容器直连外部网络 | 通过 K3s CoreDNS 解析外部网络 |

**服务访问对照表：**

| 访问目标 | Docker Compose 访问地址 | K3s 访问地址 (需替换为 WSL2 IP) |
|---|---|---|
| 客户端 UI | `http://localhost:3001` | `http://<WSL2_IP>.nip.io/` |
| 管理端 UI | `http://localhost:3002` | `http://admin.<WSL2_IP>.nip.io/` |
| API 网关 | `http://localhost:8000/api` | `http://api.<WSL2_IP>.nip.io/api/` |
| Grafana 监控 | `http://localhost:3000` | `http://grafana.<WSL2_IP>.nip.io/` |

*(注：部署脚本 `scripts/k3s-deploy.sh` 已自动化将系统入口绑定到全局 `*.<WSL2_IP>.nip.io` 子域名体系中，利用 Ingress 提供标准的七层负载均衡能力。获取 WSL2_IP: `hostname -I \| awk '{print $1}'`。测试期间请确保关闭本机的网络代理工具以防 Fake-IP 劫持域名)*

### 2. 一键启停指南

#### 2.1 Docker Compose 模式

启动平台与监控栈 (在项目根目录 `/mnt/d/hci-troubleshoot-platform` 下执行)：

```bash
# 后台启动业务容器
docker compose -f deploy/docker/docker-compose.yml up -d

# 后台启动可观测性栈 (Grafana, Loki, Tempo 等)
docker compose -f deploy/observability/docker-compose-obs.yml up -d
```

停止平台：

```bash
docker compose -f deploy/docker/docker-compose.yml down
docker compose -f deploy/observability/docker-compose-obs.yml down
```

#### 2.2 K3s 模式

若之前运行过 Docker Compose，请确保已经 `down`。

**启动 K3s 集群：**

```bash
sudo systemctl start k3s
# (可选) 若遇到 kubeconfig 权限报错，执行：
# sudo chmod 644 /etc/rancher/k3s/k3s.yaml
```

**部署/更新平台服务（自动构建缺失镜像）：**

```bash
./scripts/k3s-deploy.sh
```

**停止 K3s 集群：**

```bash
# 停止 K3s，停止所有相关容器
sudo systemctl stop k3s
```

### 3. 服务健康检查

无论哪种模式，都可以使用统一健康状态验证：

#### 3.1 命令行一键验证 (自动化检查)

如果运行在 **K3s 模式下**，直接执行提供的验证脚本即可完成全链路健康检查：

```bash
./scripts/k3s-verify.sh
```

该脚本将自动验证：K3s 核心组件、持久化存储 PVC、Helm 部署状态、所有 Pod `Running` 状态、以及 API 服务及 Web 页面的 HTTP 200 状态。

#### 3.2 手动检查命令

**Docker Compose：**
```bash
docker ps --format "table {{.Names}}\t{{.Status}}\t{{.Ports}}"
curl -sf http://localhost:8000/api/health
```

**K3s：**
```bash
kubectl get pods -n hci-troubleshoot
kubectl get pods -n hci-observability
```

### 4. 环境清理

当需要重置数据库和清除所有日志数据时，可执行清理操作。**警告：该操作将永久丢失所有历史工单、对话和监控数据。**

**4.1 Docker Compose 深度清理：**
```bash
docker compose -f deploy/docker/docker-compose.yml down -v
docker compose -f deploy/observability/docker-compose-obs.yml down -v
```

**4.2 K3s 深度清理：**
```bash
# 1. 卸载 Helm 发布
helm uninstall hci-platform -n hci-troubleshoot
helm uninstall hci-observability -n hci-observability

# 2. 删除持久化数据卷声明 (PVC)
kubectl delete pvc -l app.kubernetes.io/instance=hci-platform -n hci-troubleshoot
kubectl delete pvc -l release=hci-observability -n hci-observability
```

## 二、测试用例

测试执行需确保服务处于健康状态。对于特定的集成测试，请优先在 Docker Compose 模式下调测。

### 1. 单元测试 (Unit Tests)

测试平台核心微服务的业务逻辑和算法，不依赖外部数据库等组件。

**用例 1.1：Conversation Service 逻辑测试**
- **步骤：**
  ```bash
  cd backend/conversation-service
  uv run pytest tests/
  ```
- **预期结果：**
  100% Pass。测试应成功模拟 OpenClaw API 响应并验证内部代理与消息拦截逻辑。

**用例 1.2：Case Service 数据处理逻辑**
- **步骤：**
  ```bash
  cd backend/case-service
  uv run pytest tests/
  ```
- **预期结果：**
  涉及工单状态机流转的代码覆盖率正常且全部 Pass。

---

### 2. 集成测试 (Integration Tests)

测试微服务间的相互通信（如 Gateway -> Service -> Database / AI）。

**用例 2.1：工单微服务数据库连通性**
- **步骤：** 对 case-service 直接执行 CURL `GET http://localhost:8001/health`。
- **验证点：** 返回体中需包含 `"db_status": "connected"` 且状态码为 200。

**用例 2.2：SSE 对话流测试**
- **步骤：**
  1. 创建一个工单（`POST /api/cases`）并获取 `case_id`。
  2. 对该工单发送对话消息（`POST /api/conversations/{session_id}/message`），观察 SSE 流式响应。
- **验证点：** 接收到 `Content-Type: text/event-stream` 流式输出，最终收到 `[DONE]` 事件。

---

### 3. Web 端测试 (Web UI Tests)

测试界面组件功能、兼容性及流式响应表现。

**用例 3.1：客服端消息流式呈现**
- **步骤：**
  1. 打开 Customer UI，输入测试消息：“网络出现抖动故障怎么排查？”
  2. 观察 AI 气泡的渲染过程。
- **验证点：**
  - 不可出现长时间的空白后再整体弹出（必须是打字机流式效果 SSE）。
  - 支持渲染 Markdown 格式（如代码块需正确高亮）。
  - （已修复验证）非安全环境下“复制”按钮应当正常工作，不再因为原生 API 限制而失效。
  - （已修复验证）UI AI 回复气泡不得再弹出 `Error: internal error` 报错字样。

**用例 3.2：管理端监控面板集成**
- **步骤：** 打开 Admin UI，切换至“系统监控”页面。
- **验证点：**
  - Grafana iframe 正确嵌入，且能动态适配原生 Kubernetes Ingress (`grafana.*.nip.io`) 和 Docker Compose (`localhost:3000`) 的差异化访问地址。
  - 无需手动刷新，能正常展示系统资源占用面板。

---

### 4. API 接口测试 (API Tests)

针对 API Gateway 的主要路由和权限校验。

**用例 4.1：创建排障工单**
- **步骤：** 发送纯 API 请求创建新工单。
  ```bash
  curl -s -X POST "http://<入口地址>/api/cases/" \
    -H "Content-Type: application/json" \
    -H "X-Client-ID: test-api-001" \
    -d '{"client_id": "test-api-001", "title": "API 测试", "description": "连通性测试"}'
  ```
- **验证点：** 应当响应 201 状态，返回合法的 `case_id` 和初始化状态。

**用例 4.2：流式问答回包结构验证 (SSE)**
- **步骤：** 通过 curl 验证 `POST /api/conversations/<id>/message`。
  ```bash
  curl -s -N -X POST "http://<入口地址>/api/conversations/<id>/message" \
    -H "Content-Type: application/json" \
    -H "X-Client-ID: test-api-001" \
    -d '{"content":"hi","case_id":"<id>","role":"user"}'
  ```
- **验证点：** 头部必须是 `Content-Type: text/event-stream`，并且响应体中包含以 `data: ` 开头的可解析 JSON 块。最终以 `data: [DONE]` 结尾。

---

### 5. 可观测性测试 (Observability Tests)

保证在微服务架构下可以定位慢请求和异常堆栈。

**用例 5.1：分布式调用追踪 (OpenTelemetry + Tempo)**
- **步骤：**
  1. 通过上述业务测试触发 3~4 次对话交互。
  2. 在本地 Grafana 面板中进入 Explore（探索），选择 Tempo 数据源。
  3. 输入 `service.name="api-gateway"` 运行查询。
- **验证点：**
  - 应当检索出 Trace 列表。
  - 点击单条 Trace 可看到完整的 Span 调用链（如 `Gateway` -> `Conversation Service` -> `HTTP Request to OpenClaw`），没有链路断点。

**用例 5.2：集中式日志检索 (Promtail + Loki)**
- **步骤：**
  1. 在 Grafana Explore 选择 Loki 数据源。
  2. 输入查询日志语句（例如 `{app="conversation-service"}`）。
- **验证点：**
  - 是否包含应用启动记录和 `HTTP Request` 日志。
  - 确认在错误情况下的堆栈日志（如 400 Bad Request 等）不会发生截断。

---

### 6. 基础设施测试 (Infrastructure Tests)

测试平台对各种硬件与隔离环境的适配性。

**用例 6.1：OpenClaw AI 宿主目录权限**
- **步骤：** 进入新部署好的该 Pod （`kubectl exec -it deployment/openclaw -n hci-troubleshoot -- sh`），向 `/home/bot/.openclaw/workspace/` 下写入临时文件。
- **验证点：** （已修复验证）不能报 `EACCES: permission denied` 或 `Missing config`，非 Root 容器必须能正常拉起会话缓冲能力。

**用例 6.2：Ingress 路由转发正常性**
- **步骤：** 确保 K3s 处于 Running 后，直接通过 `http://<WSL2_IP>/` 访问（非 port-forward）。
- **验证点：** Customer UI 前端静态资源必须加载成功大纲，前端内请求的 `/api/` 相对路径不能出现 404，表明 Ingress `Prefix` 设置且 Traefik 反向代理功能正常。
