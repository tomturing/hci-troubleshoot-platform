# Task 01：Prompt 手术——重写对话核心约束（P0）

```
你是一名负责 hci-troubleshoot-platform conversation-service 的 agent。

【仓库】
git clone https://github.com/tomturing/hci-troubleshoot-platform.git
cd hci-troubleshoot-platform

【背景】
当前系统最核心的 AI 排障逻辑在：
  backend/conversation-service/app/services/conversation_service.py

该文件中存在一个致命约束 _SYSTEM_BASE，内容大意为：
  - "只能根据下方参考资料回答，不得使用训练知识"
  - "如果知识库无内容，必须直接拒绝回答"

这导致系统在知识库为空、或未命中 RAG 时，对用户的任何 HCI 故障描述都只能返回
"知识库暂无相关内容"——无法发挥 GLM 的 HCI 领域知识。这是系统最根本的功能缺陷。

HCI 排障的专家工作方式是：
  1. 以客户描述为基准，提取关键实体（虚拟机/集群/存储/网络）
  2. 同时调用告警日志和操作日志工具，拿到系统侧证据
  3. 提出 1-3 个确认问题，定位到最精确的故障分类
  4. 即使没有相关 SOP，也能用机制知识推理，明确标注"基于领域知识推理"

【知识架构背景（必读）】
本系统采用「双轨知识架构」：
  - SOP 轨道：症状/错误码匹配 SOP 手册 → 注入「SOP排障流程」→ AI 走程序性排障步骤
  - KB 轨道：SOP 未命中 → 语义检索历史案例 → 注入「历史案例参考」→ AI 用 root_cause 生成假设
  - 双轨均未命中：AI 基于训练知识推理，标注【机制推理】

两类知识本质不同，Prompt 中必须给出不同的使用指令：
  - SOP 内容：AI 应「按步骤执行」（程序性知识）
  - KB 案例内容：AI 应「提取假设，参考而非照搬」（陈述性知识）

详细设计依据见：docs/architecture/08_HCI平台效果差距分析与重构方案.md
  § ⚡ 知识源深度对比分析（真实数据验证，2026-03-23）

【任务目标】
1. 重写 _SYSTEM_BASE，改为 5 段式专家 Prompt 结构
2. 修改知识库 fallback 逻辑：从「直接拒绝」→「三级 fallback（SOP/案例/机制推理）」
3. 在 ConversationSession 模型中预留 diagnostic_stage 字段（为 Phase 2 状态机预留）
4. 确认改动后 GLM 能对 HCI 故障做有意义的回答（不再直接拒绝）

Step 2：重写 _SYSTEM_BASE 为 5 段式结构

将 _SYSTEM_BASE 替换为以下 5 段式结构（在 conversation_service.py 顶部定义
或提取为单独的 prompt_builder.py 文件）：

```python
# Segment 1: 专家身份定义
_SEGMENT_IDENTITY = """你是深信服超融合基础设施（HCI）智能排障专家助手。
你拥有完整的 HCI 平台工作原理知识：虚拟机生命周期、分布式存储、vxlan网络、
IPMI硬件管理、acli诊断工具集的完整用法。
你的目标是协助现场工程师快速定位和解决 HCI 平台故障。"""

# Segment 2: 诊断方法论（随诊断阶段 diagnostic_stage 动态变化）
_SEGMENT_METHODOLOGY = """【工作方法论】
当前诊断阶段：{stage_desc}

标准诊断流程：
S0 意图识别：从客户描述提取关键实体（虚拟机名/集群/时间点），
  同时查看告警日志和操作日志，确认客户真实问题
S1 故障定位：向客户提出 1-3 个精准确认问题，定位到最小故障分类
S2 假设生成：列出 2-3 个最可能的根因假设，按概率排序
S3 验证执行：逐一执行诊断命令，收集系统状态证据
S4 根因确认：根据证据确定根因
S5 方案输出：提供明确可执行的修复步骤
S6 验证闭环：确认问题已解决，记录知识"""

# Segment 3: 推理规范（说明如何使用各类知识）
# ⚠️ 注意：此段不得硬编码特定故障类型的领域知识（如"虚拟机开机失败的4大根因"），
# 那样做会造成 Prompt 对其他故障类型完全失效。
# 此段的职责是说明「如何推理」，不是「推理的结论是什么」。
_SEGMENT_REASONING_MODE = """【知识使用规范】
根据当前可用的参考资料，你的工作模式如下：

当有「SOP排障流程」时：
  - 这是针对此类故障的权威操作手册，按其步骤顺序执行
  - 在决策节点（有判断条件的步骤），先执行命令获取证据再做判断
  - 严格区分「临时修复步骤」和「永久解决方案」

当有「历史案例参考」时：
  - 这是历史上发生过的相似故障记录，用于辅助假设形成
  - 重点关注「根因」字段：它揭示了深层原因，是生成假设的核心依据
  - 不要直接照搬解决方案，当前环境版本/配置可能不同，需要先确认

当两者均无时（机制推理模式）：
  - 基于你对 HCI 平台工作原理的训练知识进行推理
  - 所有推断结论必须明确标注【机制推理】
  - 增加一条追问：收集更多信息以便触发知识库匹配"""

# Segment 4: 动态参考资料（双轨知识注入，具体内容由代码填充）
# SOP 命中时使用：
_SEGMENT_SOP_REFERENCE = """【SOP 排障流程 | 来源：{sop_source}】
{sop_content}

请严格按照上述排障流程执行，在每个判断节点收集证据后再做决策。"""

# KB 案例命中时使用：
_SEGMENT_CASE_REFERENCE = """【历史案例参考 | {case_count} 条相似案例】
{case_content}

上述案例供参考，重点关注「根因」字段中揭示的深层原因，用于形成假设。
当前环境可能与案例版本不同，执行修复方案前请先确认版本适用性。"""

# 双轨均未命中时使用：
_SEGMENT_NO_REFERENCE = """【机制推理模式】
当前知识库中暂未找到与此故障高度匹配的 SOP 或历史案例。
请基于 HCI 平台架构机制知识进行推理：
  - 所有推断必须标注【机制推理】以提示用户这不是经过验证的排障步骤
  - 在回复末尾追加：「如您能提供更具体的报错信息（如错误码、任务ID），
    我可以尝试匹配更精确的排障流程」"""

# Segment 5: 当前任务上下文
_SEGMENT_CONTEXT_TEMPLATE = """【当前工单上下文】
工单ID：{case_id}
客户描述：{case_description}
已知信息：{known_facts}
当前诊断阶段：{stage}"""
```

修改 _build_system_prompt 方法，按上述 5 段拼接。

Step 3：修改知识库注入逻辑为三级 fallback

⚠️ 关键：SOP 内容和 KB 案例内容在 Prompt 中必须使用不同的标签和使用指令，
因为它们是本质不同的两类知识（程序性 vs 陈述性）。

找到类似这样的代码：
```python
if not knowledge_chunks:
    system_prompt += "\n\n【重要：当前知识库暂无相关内容】\n禁止推测，只能告知用户知识库无内容。"
```

替换为以下三级 fallback 逻辑：
```python
# 三级知识注入 fallback：SOP > KB案例 > 机制推理
# knowledge_chunks 中通过 chunk_type 字段区分来源（kb-service 返回）
sop_chunks = [c for c in knowledge_chunks if c.get('chunk_type') == 'sop']
case_chunks = [c for c in knowledge_chunks if c.get('chunk_type') == 'case']

if sop_chunks:
    # 优先级1：SOP 命中 → 程序性执行路径
    sop_content = "\n\n".join(
        f"[步骤路径: {c.get('source_path', '/')}]\n{c['content']}"
        for c in sop_chunks
    )
    system_prompt += _SEGMENT_SOP_REFERENCE.format(
        sop_source=sop_chunks[0].get('source', 'SOP手册'),
        sop_content=sop_content
    )
elif case_chunks:
    # 优先级2：KB 案例命中 → 假设生成路径
    case_content = "\n\n---\n".join(
        f"[案例 #{c.get('case_id', '?')} | 版本: {c.get('applicable_version', '未知')}]\n{c['content']}"
        for c in case_chunks
    )
    system_prompt += _SEGMENT_CASE_REFERENCE.format(
        case_count=len(case_chunks),
        case_content=case_content
    )
else:
    # 优先级3：双轨均未命中 → 机制推理兜底
    system_prompt += _SEGMENT_NO_REFERENCE
```

⚠️ 实现说明：
- 如果当前 kb-service 返回的 chunks 没有 chunk_type 字段，临时处理方案是：
  检查 chunk 的 source 字段，含 'sop' 关键字的视为 sop_chunks，其余视为 case_chunks
- 完整的 chunk_type 区分由 Task 04（知识原子迁移）负责实现，此处做兼容处理

Step 4：在 ConversationSession 预留 diagnostic_stage

⚠️ 关键约束：此字段**只能加在 Pydantic Response Schema 中**，禁止加入 SQLAlchemy ORM 的 Column() 定义。
原因：DB 迁移由 Task 07 负责，贸然加 Column() 会导致 ORM 与数据库表结构不一致，
服务启动时抛出 `column "diagnostic_stage" does not exist` 错误。

正确做法（Pydantic Schema，不影响 DB）：
```python
# backend/conversation-service/app/schemas/conversation.py
class ConversationSessionResponse(BaseModel):
    ...（现有字段）...
    diagnostic_stage: str = "S0"    # 诊断阶段占位符，Task 07 迁移后会有真实 DB 值
```

禁止写法（会导致 ORM/DB 不一致，严禁）：
```python
# ❌ 不要在 SQLAlchemy 模型中加这行
diagnostic_stage = Column(String(8), default="S0")  # 禁止！
```

```python
#!/usr/bin/env python3
# scripts/dev/fix_sop_format.py
"""修复 SOP 文件中的 docx 导入 artifact：命令文本重复、多余换行"""
import re
import sys
from pathlib import Path

def fix_sop_file(path: Path) -> int:
    """返回修复的行数"""
    content = path.read_text(encoding='utf-8')
    original = content

    # 修复1：去除 acli 命令重复（"acli xxx\nacli xxx" → "acli xxx"）
    # 匹配模式：某段 acli 命令后紧跟同一命令
    lines = content.split('\n')
    fixed_lines = []
    prev_line = None
    for line in lines:
        stripped = line.strip()
        # 跳过与前一行完全相同的 acli 命令行（排除代码块注释行）
        if stripped and stripped == (prev_line or '').strip() and stripped.startswith('acli'):
            continue
        fixed_lines.append(line)
        prev_line = line

    content = '\n'.join(fixed_lines)

    # 修复2：去除行内重复（同一行出现两次相同的 acli 命令）
    # 如 "acli task get -v ${vmid}acli task get -v ${vmid}"
    def deduplicate_inline(m):
        half = len(m.group(0)) // 2
        first_half = m.group(0)[:half]
        if m.group(0) == first_half * 2:
            return first_half
        return m.group(0)

    # 匹配 30-200 字符的命令重复
    content = re.sub(r'(acli [^\n]{15,100})\1', r'\1', content)

    if content != original:
        path.write_text(content, encoding='utf-8')
        return content.count('\n') - original.count('\n')  # 减少的行数
    return 0

if __name__ == '__main__':
    sop_dir = Path('data-pipeline/sop_skills')
    files = [sop_dir / 'vm_power_failure.md'] + \
            list((sop_dir / 'vm_power_failure' / 'chapters').glob('*.md'))
    for f in files:
        n = fix_sop_file(f)
        print(f'修复 {f.name}: {n} 处改动')
```
