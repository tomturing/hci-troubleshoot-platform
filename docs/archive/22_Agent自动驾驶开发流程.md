# 22_Agent自动驾驶开发流程

> 适用场景：项目采用 Vibe Coding，开发、测试、发布、回滚全流程由 AI Agent 主导执行，人类主要负责策略与审批。

---

## 1. 目标与边界

### 1.1 目标

1. 实现端到端自治交付闭环：需求 -> 编码 -> 审查 -> 测试 -> 发布 -> 巡检 -> 回滚。
2. 将人工参与降低到最少：只保留需求确认、生产审批、异常仲裁。
3. 让每一步都有结构化输出，可被追踪、重放和持续优化。

### 1.2 边界

1. Agent 不得越权修改未授权目录。
2. Agent 不得绕过生产审批直接发布。
3. Agent 不得跳过质量门禁直接生成生产候选。

---

## 2. Agent 体系与职责

## 2.1 Agent 角色

1. Orchestrator Agent（编排）
- 需求解析、任务拆解、依赖管理、执行调度、状态汇总。

2. Coder Agent（编码）
- 在限定文件范围实现功能与测试。

3. Reviewer Agent（审查）
- 只审查不改码，输出结构化缺陷与风险结论。

4. QA Agent（质量门禁）
- 统一执行 lint、test、helm 校验、制品一致性检查。

5. Release Agent（发布）
- 构建镜像、更新环境声明、触发 GitOps 同步。

6. SRE Agent（运行保障）
- 发布后巡检、指标观察、异常定位、触发回滚。

7. Rollback Agent（回滚）
- 依据稳定版本基线执行自动回退并验证恢复。

## 2.2 人类角色

1. Product Owner：确认需求与验收口径。
2. Release Approver：审批生产发布令牌。
3. Incident Commander：重大异常仲裁。

---

## 3. Agent 状态机

```text
INTAKE
  -> PLAN
  -> EXECUTE_CODING
  -> REVIEW
  -> QA_GATE
  -> RELEASE_CANDIDATE
  -> STAGING_DEPLOY
  -> PROD_APPROVAL_WAIT
  -> PROD_DEPLOY
  -> POST_DEPLOY_OBSERVE
  -> DONE

任何阶段失败 -> AUTO_REPAIR 或 ROLLBACK_DECISION
连续失败超阈值 -> HUMAN_ESCALATION
```

### 3.1 状态转换规则

1. REVIEW 未通过：返回 EXECUTE_CODING。
2. QA_GATE 未通过：返回 EXECUTE_CODING 并附失败报告。
3. STAGING 冒烟失败：进入 AUTO_REPAIR，超过阈值则人工仲裁。
4. PROD 无审批令牌：停留在 PROD_APPROVAL_WAIT，不可自动越过。

---

## 4. 端到端流程

## Phase A：需求接入与任务编排

### 任务

1. Orchestrator Agent 解析需求并补全非功能约束：
- 可观测性
- 回滚路径
- 安全要求
2. 自动输出依赖图与子任务清单。
3. 自动创建任务、工作区和执行队列。

### 交付物

1. 任务依赖图。
2. 每个子任务的文件边界与验收标准。
3. Agent 执行计划（并行度、优先级、预计耗时）。

### 验收标准

1. 子任务覆盖 100% 需求范围。
2. 每个子任务都有可执行验收标准。

---

## Phase B：并行编码与交叉审查

### 任务

1. 多个 Coder Agent 并行编码。
2. Reviewer Agent 异构交叉审查（编码与审查必须是不同 Agent 类型）。
3. 自动修复循环：审查问题自动回传并修复。

### 交付物

1. 结构化审查报告（风险等级、阻断项、建议项）。
2. 修复记录与差异摘要。

### 验收标准

1. 无阻断项后方可进入 QA_GATE。
2. 所有修改均在授权目录内。

---

## Phase C：质量门禁自动化

### 任务

1. QA Agent 执行统一门禁：
- 后端 lint/test
- 前端 lint/test/build
- Helm lint/template
- 发布前一致性检查
2. 输出标准化报告：
- pass_rate
- failed_checks
- retry_suggestion

### 交付物

1. 质量门禁报告（JSON + Markdown）。
2. 失败修复建议。

### 验收标准

1. 核心门禁全部通过才可进入发布候选。
2. 门禁失败自动回流修复，不需人工介入。

---

## Phase D：Agent 驱动发布

### 任务

1. Release Agent 生成不可变镜像标签。
2. 自动更新环境声明仓库（dev/staging 自动，prod 等审批）。
3. 触发 Argo CD 同步部署。

### 交付物

1. 发布候选记录。
2. 环境声明变更 PR。
3. 部署执行日志与结果摘要。

### 验收标准

1. dev/staging 发布全自动完成。
2. prod 必须等待审批令牌后执行。

---

## Phase E：发布后巡检与自动回滚

### 任务

1. SRE Agent 执行固定 30 分钟观察：
- 健康检查
- 错误率
- 延迟
- 资源水位
2. 触发阈值策略：
- 超阈值自动触发 Rollback Agent
- 同时生成故障摘要并通知人工

### 交付物

1. 观察报告。
2. 回滚报告（若触发）。
3. 复盘摘要（机器可读 + 人类可读）。

### 验收标准

1. 故障可在 10 分钟内自动回退。
2. 发布结果可完整追溯到任务与变更。

---

## 5. 规范体系

### 5.1 Agent 输入规范

1. 必须包含：任务目标、文件边界、验收标准、禁止项。
2. 不允许模糊指令（如“顺便优化”）。

### 5.2 Agent 输出规范

1. 统一结构字段：
- task_id
- phase
- changed_files
- risk_level
- qa_result
- rollback_point
2. 输出必须给出“下一步动作”建议。

### 5.3 权限规范

1. 编码 Agent：只读全仓 + 仅写授权目录。
2. 发布 Agent：只写环境仓库、不可写业务仓库。
3. 回滚 Agent：仅允许回退到已认证稳定版本。

### 5.4 安全规范

1. 所有凭据由 Secret 管理，不进入 Agent 输出日志。
2. 生产发布必须有人工审批令牌。
3. 连续失败超过阈值自动熔断，切换半自动模式。

---

## 6. 7 天压缩实施计划（Agent 版）

## Day 1

1. 定义 Agent 角色、状态机、任务模板。
2. 验收：可自动创建任务并进入执行状态。

## Day 2

1. 打通 Coder + Reviewer 交叉审查循环。
2. 验收：完成至少 1 个真实需求自动编码与审查闭环。

## Day 3

1. QA Agent 接管质量门禁。
2. 验收：可自动产出标准化门禁报告。

## Day 4

1. Release Agent 接管 dev 发布。
2. 验收：dev 环境无人值守发布成功。

## Day 5

1. 接管 staging，加入冒烟与回流修复。
2. 验收：staging 连续 2 次发布成功。

## Day 6

1. 接入 prod 审批令牌与回滚代理。
2. 验收：完成一次生产演练发布和一次回滚演练。

## Day 7

1. 文档化、培训、阈值优化、知识沉淀。
2. 验收：团队可按手册独立运行 Agent 自动驾驶流程。

---

## 7. 对持续开发迭代的影响

1. 第 1-2 天吞吐下降约 10%-15%。
2. 第 3-4 天下降约 5%-10%。
3. 第 5 天后恢复至 95%-100%。

### 降影响策略

1. 平台改造代理池与业务开发代理池分离。
2. 保留旧链路作为发布兜底。
3. 仅冻结编排规则与环境策略，不冻结业务功能开发。

---

## 8. 风险与防护

1. 风险：Agent 越权改动。
- 防护：文件边界白名单 + 审查阻断。

2. 风险：Agent 无限修复循环。
- 防护：重试次数阈值 + 人工仲裁出口。

3. 风险：发布代理误触生产。
- 防护：审批令牌 + 环境隔离权限。

4. 风险：异常检测不及时。
- 防护：固定巡检指标 + 自动回滚触发阈值。

---

## 9. 完成定义（Definition of Done）

1. 需求到发布全过程可由 Agent 自动完成（prod 除审批外）。
2. 每个阶段都有结构化记录与审计证据。
3. 故障可自动回滚并形成复盘素材。
4. 团队可持续迭代，不依赖特定个人手工发布。
