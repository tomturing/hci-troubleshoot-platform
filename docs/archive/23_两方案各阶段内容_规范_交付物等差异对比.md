# 23_两方案各阶段内容_规范_交付物等差异对比

> 对比对象：
> - 方案一：GitOps 标准化开发流程（文档 21）
> - 方案二：Agent 自动驾驶开发流程（文档 22）

---

## 1. 总体定位差异

| 维度 | 方案一：GitOps 标准化 | 方案二：Agent 自动驾驶 |
|---|---|---|
| 主体模式 | 人主导 + 自动化辅助 | Agent 主导 + 人类审批兜底 |
| 核心价值 | 可追溯、可审计、降低发布失误 | 高并行、低人工、闭环自治 |
| 组织要求 | 标准 DevOps 团队 | 具备 Agent 编排与治理能力 |
| 适配度 | 中短期落地最稳妥 | 中长期效率与规模优势更大 |

---

## 2. 阶段对比（流程内容）

## 2.1 需求与规划阶段

| 项目 | 方案一 | 方案二 |
|---|---|---|
| 需求拆解 | 人工主导拆分、评审确认 | Orchestrator 自动拆分并给出依赖图 |
| 验收标准 | 人工定义为主 | Agent 自动补全非功能验收 |
| 风险识别 | Reviewer 人工识别 | Agent 预警 + 人工确认 |

## 2.2 开发与审查阶段

| 项目 | 方案一 | 方案二 |
|---|---|---|
| 编码执行 | 人工开发为主 | 多 Coder Agent 并行 |
| 代码审查 | 人工或半自动 | 异构 Reviewer Agent 交叉审查 |
| 修复闭环 | 人工回改 | 自动回流修复，超阈值再人工介入 |

## 2.3 质量门禁阶段

| 项目 | 方案一 | 方案二 |
|---|---|---|
| 门禁执行 | CI 固定流水线 | QA Agent 调度流水线并输出结构化报告 |
| 失败处理 | 人工查看日志修复 | Agent 先自修复，失败再升级 |
| 报告格式 | 平台原生日志为主 | 统一 JSON/Markdown 双格式 |

## 2.4 发布与部署阶段

| 项目 | 方案一 | 方案二 |
|---|---|---|
| 发布触发 | 人工触发或半自动触发 | Release Agent 自动触发 |
| 环境声明更新 | CI 或人工提交 | Release Agent 自动提交 |
| 生产控制 | 审批后人工执行 | 审批令牌后 Agent 执行 |

## 2.5 运行保障与回滚阶段

| 项目 | 方案一 | 方案二 |
|---|---|---|
| 观察方式 | 人工看板 + 脚本巡检 | SRE Agent 自动巡检 |
| 异常处理 | 人工判断后回滚 | 阈值触发自动回滚，人工仲裁异常场景 |
| 复盘产出 | 人工整理 | Agent 自动生成复盘初稿 |

---

## 3. 规范对比

## 3.1 权限与安全规范

| 项目 | 方案一 | 方案二 |
|---|---|---|
| 权限控制 | 以人和 CI 账号为主 | 以 Agent 角色最小权限为主 |
| 生产审批 | 必须人工 | 必须人工（审批令牌） |
| 凭据管理 | Secret + 平台策略 | Secret + Agent 输出脱敏策略 |

## 3.2 变更治理规范

| 项目 | 方案一 | 方案二 |
|---|---|---|
| 变更记录 | PR + 发布记录 | PR + Agent 执行轨迹 + 发布记录 |
| 追溯粒度 | 到提交与流水线级别 | 到任务、Agent、状态转换级别 |
| 合规审计 | 满足常规审计 | 更适合高频审计与自动审计 |

## 3.3 质量规范

| 项目 | 方案一 | 方案二 |
|---|---|---|
| 质量门禁 | 静态定义 | 静态门禁 + 动态修复策略 |
| 缺陷收敛 | 依赖人工排查 | Agent 自动收敛优先 |
| 失败熔断 | 人工决策 | 自动熔断 + 人工接管 |

---

## 4. 交付物对比

## 4.1 共同交付物

1. 分支与发布策略文档。
2. CI 门禁流水线。
3. 环境声明仓库。
4. Argo CD 应用清单。
5. 发布、回滚、巡检手册。

## 4.2 方案一新增交付物

1. 人工发布清单（Preflight/Postflight）。
2. 人工值守与应急切换手册。
3. 发布审计台账模板。

## 4.3 方案二新增交付物

1. Agent 角色定义文档。
2. Agent 状态机与转移规则。
3. Agent 输入/输出规范（结构化字段）。
4. Agent 权限矩阵与审批令牌策略。
5. 自动复盘模板与失败模式知识库。

---

## 5. 时间、成本与风险对比

| 维度 | 方案一 | 方案二 |
|---|---|---|
| 最短可落地时间 | 5-7 工作日 | 7 工作日（可用闭环） |
| 稳妥落地时间 | 10-15 工作日 | 10-20 工作日（含治理优化） |
| 初期组织成本 | 较低 | 中等偏高 |
| 对团队能力要求 | DevOps 基础 | DevOps + Agent 治理能力 |
| 人工依赖度 | 中等 | 低 |
| 运行期效率潜力 | 中等 | 高 |

---

## 6. 对持续迭代影响对比

| 指标 | 方案一 | 方案二 |
|---|---|---|
| 第 1-2 天吞吐影响 | 10%-20% | 10%-15% |
| 第 3-4 天吞吐影响 | 5%-10% | 5%-10% |
| 第 5 天后恢复 | 约 90%-95% | 约 95%-100% |
| 依赖关键人程度 | 中等 | 低（审批岗除外） |

---

## 7. 采用建议

### 7.1 若目标是“快速稳妥上线”

1. 优先采用方案一，先完成 GitOps 基线。
2. 并在流程中预留 Agent 接口，避免二次返工。

### 7.2 若目标是“长期高频迭代效率”

1. 直接采用方案二，建设 Agent 自动驾驶体系。
2. 先从 dev/staging 全自动开始，prod 保持审批制。

### 7.3 推荐路径（组合式）

1. 第 1 周：先完成方案一最小闭环。
2. 第 2-3 周：叠加方案二的 Agent 编排、审查、巡检与回滚能力。
3. 第 4 周：进入“Agent 主导 + 人工审批”的稳定运行态。

---

## 8. 决策检查清单

1. 是否已明确生产审批责任人？
2. 是否可接受第 1 周 10%-20% 的吞吐波动？
3. 是否具备 Agent 权限治理与审计能力？
4. 是否已规划回滚演练窗口？
5. 是否已定义异常升级路径？

若以上 5 项均为“是”，可直接进入方案二实施；若存在 2 项以上“否”，建议先执行方案一再平滑升级。

## 9. 21->22 迁移执行清单 v1

> 目标：先按文档 21 稳定运行，再平滑升级到文档 22 的 Agent 自动驾驶模式。
> 
> 迁移原则：不一次性切换，不中断业务迭代，始终保留回退开关。

## 9.1 迁移总览（4 周）

| 周次 | 迁移重点 | 发布控制方式 | 风险级别 |
|---|---|---|---|
| 第 1 周 | 21 基线固化 | 人工审批 + GitOps | 低 |
| 第 2 周 | 引入 Agent 编排与审查 | 人工审批 + GitOps | 中低 |
| 第 3 周 | Agent 接管 dev/staging 发布 | dev/staging 自动，prod 人工审批 | 中 |
| 第 4 周 | Agent 接管 prod 执行（保留人工审批） | 审批令牌 + Agent 执行 | 中 |

## 9.2 第 1 周执行清单（先稳住 21）

### 目标

1. 建立稳定、可审计、可回滚的 GitOps 基线。
2. 不引入生产自动驾驶，仅修好底座。

### 任务

1. 固化分支保护与 PR 模板（必须有风险和回滚项）。
2. 固化 CI 门禁（后端、前端、Helm、镜像构建）。
3. 固化环境仓库目录与 dev/staging/prod 配置分层。
4. 固化发布后巡检与回滚 SOP。

### 角色分工

1. Platform Owner：分支保护、CI、环境仓库治理。
2. Backend/Frontend Owner：质量门禁与构建稳定性。
3. Release Manager：发布窗口与审批规则。

### 交付物

1. 基线流程文档 v1。
2. 发布清单（Preflight/Postflight）。
3. 回滚演练记录（至少 1 次）。

### 验收标准

1. dev/staging 至少 2 次连续发布成功。
2. 生产发布可审批、可追溯、可回滚。
3. 回滚平均恢复时间达到目标值。

## 9.3 第 2 周执行清单（接入 22 的“前半段能力”）

### 目标

1. 引入 Agent 编排、编码、审查、门禁自动化。
2. 生产发布继续由 21 流程控制。

### 任务

1. 上线 Orchestrator Agent：任务拆解、依赖排序、并行调度。
2. 上线 Coder + Reviewer 交叉审查闭环。
3. 上线 QA Agent：统一门禁报告格式。
4. 建立 Agent 输出规范（任务 ID、风险等级、下一步动作）。

### 角色分工

1. AI Workflow Owner：Agent 模板、状态机、执行策略。
2. Tech Leads：审核 Agent 输出质量与边界。
3. Release Manager：把关“仅开发态自动化，不接管生产发布”。

### 交付物

1. Agent 角色说明书。
2. Agent 输入/输出规范。
3. 结构化 QA 报告模板。

### 验收标准

1. 至少 1 个真实需求可完成“自动编码 -> 自动审查 -> 自动门禁”闭环。
2. Agent 越权改动为 0。
3. 门禁失败自动回流修复可用。

## 9.4 第 3 周执行清单（接入 22 的“发布自动化能力”）

### 目标

1. Release Agent 接管 dev/staging 发布执行。
2. prod 仍保持人工审批和人工发令。

### 任务

1. Release Agent 自动更新环境声明仓库。
2. 自动触发 Argo CD 同步 dev/staging。
3. SRE Agent 执行发布后巡检与阈值告警。
4. 建立异常自动降级：失败超阈值时切回第 1 周发布链路。

### 角色分工

1. Platform Owner：Release Agent 与 GitOps 集成。
2. SRE Owner：观察指标、告警阈值、降级规则。
3. Release Manager：审批策略与异常处置。

### 交付物

1. Release Agent 执行日志模板。
2. 自动巡检报告模板。
3. 发布失败降级开关说明。

### 验收标准

1. dev/staging 连续 3 次无人值守发布成功。
2. 异常时可自动降级到 21 链路。
3. 发布后 30 分钟观察报告自动生成。

## 9.5 第 4 周执行清单（prod 执行面迁移）

### 目标

1. 进入“人工审批 + Agent 执行”的生产模式。
2. 保持合规边界与回滚能力不下降。

### 任务

1. 引入审批令牌机制，未审批不得执行生产发布。
2. Release Agent 执行生产发布流程。
3. Rollback Agent 接管标准化回滚并自动验证。
4. 完成一次生产演练发布 + 一次演练回滚。

### 角色分工

1. Release Approver：审批令牌签发。
2. Release Agent：执行发布与状态回传。
3. Incident Commander：重大异常最终裁决。

### 交付物

1. 生产审批记录与执行轨迹。
2. 自动回滚报告。
3. 复盘文档（含行动项）。

### 验收标准

1. 生产发布全链路可追溯。
2. 生产回滚可在目标时间内完成。
3. 无未审批生产发布事件。

## 9.6 阶段切换闸门（Go/No-Go）

每周切换前必须满足以下指标：

1. 质量门禁通过率 >= 95%。
2. 发布成功率 >= 98%（对应环境）。
3. 回滚演练成功率 = 100%。
4. Agent 越权改动 = 0。
5. 无阻断级未修复问题。

若任一指标不达标：

1. 暂停进入下一周阶段。
2. 维持当前阶段并完成缺口修复。

## 9.7 持续迭代保障清单

1. 平台改造与业务开发使用独立任务池。
2. 每天固定平台改造窗口（建议 60-90 分钟）。
3. 仅冻结发布规则，不冻结业务代码开发。
4. 任何生产异常优先切回 21 稳定链路。
5. 每周复盘一次迁移指标，动态调整节奏。

## 9.8 里程碑完成定义

1. M1（第 1 周末）：21 基线稳定可运行。
2. M2（第 2 周末）：22 前半段能力接入成功。
3. M3（第 3 周末）：dev/staging 自动驾驶发布稳定。
4. M4（第 4 周末）：prod 进入“人工审批 + Agent 执行”稳定态。
