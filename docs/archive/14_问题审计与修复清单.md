# HCI 智能排障平台 - 问题审计与修复清单

## 文档信息
- **版本**: 1.0
- **创建日期**: 2026-03-07
- **范围**: 架构设计、数据库设计、可观测性设计、K8s 设计
- **目标**: 基于当前代码与文档差异，给出可落地修复清单（含优先级、改动点、验收标准、回归项）

---

## 1. 结论摘要

当前项目存在多处**设计与实现不一致**的问题，其中包含会直接导致运行异常的 P0 级缺陷。建议按以下顺序推进：

1. 先修复对话主链路（Conversation + KB 接口契约）  
2. 再修复 Case→KB 写入链路和调度指标链路  
3. 最后处理数据库一致性、K8s 模板缺口与文档对齐

---

## 2. 修复优先级总览

| 优先级 | 问题 | 影响面 | 状态 |
|---|---|---|---|
| P0 | ConversationService 构造参数与成员不一致，且存在未定义常量 | 对话核心链路不可用/启动失败风险 | 待修复 |
| P0 | KBClient 与 kb-service 接口契约不一致 | RAG 注入静默失效 | 待修复 |
| P0 | Case 关闭推送 KB payload 不符合 ingest 入参约束 | 知识沉淀写入失败 | 待修复 |

## 3. 分项修复清单

### 3.1 P0-1 对话主链路依赖注入与实现不一致

**现象**
- `ConversationService` 初始化参数与调用处不一致
- 服务内使用了未定义成员 `self.kb_client`
- 服务内使用了未定义常量 `_SYSTEM_BASE`

**改动点**
- 文件：`backend/conversation-service/app/services/conversation_service.py`
  - 调整 `__init__` 参数，显式接收并保存 `kb_client`
  - 补全 `_SYSTEM_BASE` 常量（或改为配置化 Prompt）
- 文件：`backend/conversation-service/app/routes/conversations.py`
  - 与新构造参数保持一致

**完成标准（唯一）**
- 启动 `conversation-service` 后，`POST /api/conversations/{id}/message` 可正常进入流式响应，不再出现属性缺失或未定义异常。

**回归验证**
- 单测：新增/修复 `ConversationService` 初始化与 `send_message_stream_only` 相关测试
- 人工：创建工单→创建对话→发送消息，确认返回 SSE 正常结束 `[DONE]`

---

### 3.2 P0-2 KB 客户端与服务接口契约对齐

**现象**
- `KBClient` 未发送内部鉴权头
- `search` 读取字段 `results`，但服务返回为 `chunks`
- `sop_match` 读取字段 `node`，但服务返回为扁平结构（`matched + 节点字段`）

**改动点**
- 文件：`backend/conversation-service/app/services/kb_client.py`
  - 所有请求透传 `Authorization: Bearer <INTERNAL_API_TOKEN>`
  - `search()` 解析 `chunks`
  - `sop_match()` 兼容 `matched=true` 的扁平结构返回
- 文件：`backend/conversation-service/app/services/conversation_service.py`
  - 对 SOP 返回结构做兼容读取（标题、内容字段）

**完成标准（唯一）**
- 在 KB 服务可用时，SOP 命中与 RAG chunks 能稳定注入对话上下文，日志中出现 `kb_sop_hit` 或 `kb_search_hit` 且命中数 > 0。

**回归验证**
- 单测：KBClient 响应解析测试（search/sop）
- 人工：构造可命中 SOP 的查询，验证系统提示中出现 SOP 注入片段

---

### 3.3 P0-3 Case 关闭知识沉淀入库失败

**现象**
- `kb_pusher` 的 payload 使用 `content` 字段，但 ingest 要求 `content_md`
- `source_type` 使用 `case_summary`，不在 ingest 校验允许值（`kb|sop|realtime`）

**改动点**
- 文件：`backend/case-service/app/services/kb_pusher.py`
  - 字段名改为 `content_md`
  - `source_type` 改为合法值（建议 `realtime`）
  - 附加 `yaml_meta` 传递 `case_id` 等结构化信息
- 文件：`backend/kb-service/app/routes/ingest.py`（可选）
  - 如需保留 `case_summary`，同步扩展 pattern 与文档

**完成标准（唯一）**
- 关闭工单后，KB ingest 接口返回 2xx，且在 `kb_document` 中可查到对应记录。

**回归验证**
- 单测：`kb_pusher` payload 构造测试
- 人工：关闭工单后检查 kb-service 日志含 `ingest_completed`

---

### 3.4 P1-1 调度指标接口字段错配

**现象**
- `/pool-metrics` 使用 `idle_count/active_count/active_allocations`
- 实际 `SchedulerService.get_status()` 返回 `idle/active/allocated_cases`

**改动点**
- 文件：`backend/scheduler-service/app/routes/scheduler_routes.py`
  - 统一字段映射到真实返回结构
  - 若要保留旧字段，对外做兼容别名

**完成标准（唯一）**
- `/api/scheduler/pool-metrics` 返回的空闲/活跃/分配总数与 `/api/scheduler/status` 一致。

**回归验证**
- 单测：状态对象→metrics 响应转换测试
- 人工：分配/释放 pod 前后对比两个接口值一致

---

### 3.5 P1-2 工单号生成策略偏离设计

**现象**
- 代码采用随机 5 位序号，不是文档设计的当日递增

**改动点**
- 文件：`backend/case-service/app/services/case_service.py`
  - 移除随机 ID 生成
  - 统一使用数据库函数 `generate_case_id()` 或新增序列表实现
- 文件：`backend/case-service/app/repositories/case_repo.py`
  - 增加数据库侧取号方法，确保并发安全

**完成标准（唯一）**
- 同一天连续创建 1000 个工单时，`case_id` 唯一且按时间单调递增。

**回归验证**
- 单测：并发创建工单 ID 唯一性测试
- 人工：查询最近工单，确认序号连续

---

### 3.6 P1-3 Helm 缺失 kb-service 模板

**现象**
- `values.yaml` 已有 `kbService`，但 templates 下无 `kb-service` Deployment/Service

**改动点**
- 新增目录：
  - `deploy/helm/hci-platform/templates/kb-service/deployment.yaml`
  - `deploy/helm/hci-platform/templates/kb-service/service.yaml`
- 补充 `SOP_SKILLS_DIR` 挂载、探针、资源、ConfigMap/Secret 注入

**完成标准（唯一）**
- `helm template` 输出包含 `kb-service` 的 Deployment 和 Service，且能通过 `kubectl get svc` 解析到 `kb-service:8004`。

**回归验证**
- 人工：部署后执行 `curl http://kb-service:8004/health` 返回 200

### 3.7 P2-1 代码质量与文档一致性收敛

**现象**

**改动点**

- 文件：`docs/05_K8s设计.md`

- 文件：`docs/04_可观测性设计.md`

**完成标准（唯一）**
- 文档与代码主线一致，不再出现同一主题的互相矛盾描述。
