**版本**: v1.0  
**日期**: 2026-03-23  
**定位**: 参考终态文档，厘清"目标是什么"，避免方向性偏差  
**核心问题**: Claude 使用的是方案 A/B/C 哪一种？都不是——请看本文。  
**适读对象**: 架构师、产品负责人、所有参与本项目的工程师

## 一、Claude 用的是哪种方案？

**直接结论：以上三种方案（A/B/C）都不是 Claude 的工作方式。**

方案 A/B/C 本质上都是在解决同一个工程问题：**如何让资源有限的 GLM-4 表现得像知识丰富的 Claude**。它们是**对 Claude 能力的工程性逼近方案**，不是 Claude 本身的架构。

要理解 Claude 实际上是如何工作的，需要从底层原理出发。

## 二、Claude 的实际工作机制

### 2.1 三个基础支柱

```
┌────────────────────────────────────────────────────────────────┐
│                     Claude 的能力来源                          │
│                                                               │
│  ┌──────────────────────┐  ┌──────────────────────────────┐  │
│  │  参数化知识           │  │  超大 Context Window          │  │
│  │  (Parametric Memory) │  │  (Working Memory / 200K token)│  │
│  │                      │  │                              │  │
│  │  训练进权重的知识：    │  │  运行时注入的信息：           │  │
│  │  • 编程语言语法       │  │  • 当前文件内容              │  │
│  │  • 框架 API          │  │  • 工具执行结果              │  │
│  │  • 领域机制理解       │  │  • 历史对话                  │  │
│  │  • 推理模式          │  │  • 用户提供的文档             │  │
│  │  • 数十亿文档的摘要   │  │  → 所有工具输出直接进上下文   │  │
│  └──────────┬───────────┘  └────────────┬─────────────────┘  │
│             │                           │                     │
│             └──────────────┬────────────┘                     │
│                            ▼                                  │
│  ┌─────────────────────────────────────────────────────────┐  │
│  │             工具调用（Tool Use）                         │  │
│  │        Claude 主动决定调用什么，获取什么信息             │  │
│  │                                                         │  │
│  │  read_file()  grep_search()  run_in_terminal()         │  │
│  │  semantic_search()  file_search()  create_file()        │  │
│  └─────────────────────────────────────────────────────────┘  │
└────────────────────────────────────────────────────────────────┘
```

### 2.2 Claude 没有 RAG

这是理解 Claude 的最重要认知：**Claude 自己不做 RAG**。

- Claude 没有"检索数据库"
- Claude 没有"向量索引"
- Claude 的知识库就是它的参数权重（1750亿+参数中压缩的信息）

当需要特定信息时，Claude 通过**工具调用**主动获取，而不是被动接受检索注入。

```
传统 RAG 系统:
  平台检索 → 注入 prompt → LLM 生成

Claude Agent:
  LLM 决定需要什么 → 调用工具获取 → 工具结果进 context → 继续推理
                                                            ↑ 循环
```

### 2.3 Claude 的 ReAct 循环

ReAct = Reasoning + Acting，是 Claude 在 Agent 模式下的核心工作模式：

```
Input: 用户请求
│
└──→ ┌─────────────────────────────────────────────┐
     │  Thought（思考，不可见）                     │
     │  "我需要什么信息？当前已知什么？              │
     │   下一步最优行动是什么？"                    │
     └─────────────────────────────────────────────┘
              │
              ▼
     ┌─────────────────────────────────────────────┐
     │  Action（行动）                              │
     │  tool_call: read_file("src/service.py")     │
     └─────────────────────────────────────────────┘
              │
              ▼
     ┌─────────────────────────────────────────────┐
     │  Observation（观察）                         │
     │  工具返回结果，进入 context window           │
     └─────────────────────────────────────────────┘
              │
              ▼
     ┌─────────────────────────────────────────────┐
     │  Thought（再思考）                           │
     │  "这个文件揭示了什么？下一步是什么？"         │
     └─────────────────────────────────────────────┘
              │
     继续循环，直到完成任务
```

**每次循环的核心是：LLM 主动决定下一步做什么。** 没有外部路由，没有策略模型，纯 LLM 推理驱动。

### 2.4 Claude 的"隐式记忆架构"

Claude 没有显式的 MemGPT 式内存管理，但通过以下机制实现了等效甚至更强的效果：

```
Core Memory 的等效物:
  → 训练时压缩进权重的海量知识（不占 context window）
  → 系统卡（CLAUDE.md / AGENTS.md）注入的规则和私有信息

Episodic Memory 的等效物:
  → 工具调用结果直接进 context window（200K tokens 容量）
  → 不需要"召回"——相关信息直接被工具采集到

Working Memory 的等效物:
  → 整个 context window 本身就是工作记忆
  → Claude 在 context window 内直接看到所有历史推理步骤
```

**核心差异**：Claude 的"记忆"不需要主动管理，因为 context window 足够大（200K），且工具调用能按需获取新信息。这是参数规模带来的架构优势，是功能对等而实现完全不同。

## 三、Claude 为何如此强大——本质原因

```
Claude 的核心竞争力不是某种特定的架构设计，
而是三个相互增强的要素的综合效果：

1. 参数规模（Scale）
   1750亿+ 参数中压缩着大量人类书写的知识、推理模式和价值观。
   这是"出厂自带"的认知能力，不需要额外知识工程。

2. 大上下文窗口（Large Context）
   200K tokens ≈ 一本 400 页书的内容
   允许在单次对话中持有极大量的工作信息，无需记忆管理

3. 工具调用（Tool Use）
   主动、按需获取信息，而不是被动接受
   完全由 LLM 推理主导，工具是它的"手"，不是它的"脑"
```

**这是一个"大力出奇迹"的方案**：足够大的模型 + 足够大的上下文 + 工具 = 无需复杂知识工程就能表现优秀。

## 四、本项目与 Claude 的本质差距

### 4.1 差距地图

| 维度 | Claude 的方式 | 本项目（GLM-4）的困境 |
|------|--------------|---------------------|
| **参数化知识** | 1750亿+参数，大量 HCI 相关知识已在权重中 | GLM-4 对 Sangfor HCI 的私有命令（acli/SCP）几乎无感知 |
| **上下文容量** | 200K tokens，基本不需要记忆管理 | GLM-4 当前约 128K tokens，实际稳定约 32K |
| **工具集成** | Anthropic 精心设计的 tool use 机制，JSON 输出稳定 | GLM-4 function calling 需要 JSON 修复兜底（T09 的 `_safe_parse_json`）|
| **推理深度** | 可自然推理 15+ 步不丢失上下文 | 长推理链路中 GLM-4 容易发生"遗忘"和"幻觉" |
| **知识覆盖** | 大量技术文档在训练集中 | HCI 特有私有知识（acli 命令、Sangfor 错误码）不在训练集 |

### 4.2 差距的根本性质

这不是工程差距（实现多几个功能），而是**规模和数据差距**：

```
Claude 能力 = f(规模, 训练数据, 对齐技术)
我们不能改变的: GLM-4 的规模和训练数据

我们能做的:
  1. 通过工具调用弥补"缺乏私有知识"
     → SCP Adapter, acli 工具，知识库检索工具
  
  2. 通过知识工程弥补"缺乏领域训练"
     → DocxExtractor, EpisodicMemory, CausalGraph
  
  3. 通过 Prompt 设计让 GLM-4 知道"该如何工作"
     → 5段式 Prompt, 诊断方法论, 机制知识注入
  
  4. 通过系统架构弥补"上下文容量不足"
     → 记忆管理（方案A）, 知识压缩（情节记忆 vs 原始案例）
```

**这就是方案 A/B/C 存在的根本原因：它们都是在用工程方法弥补 GLM-4 与 Claude 之间的规模差距。**

## 五、Claude 式工作流在 HCI 排障中的理想形态

### 5.1 如果 Claude 直接做 HCI 排障（理想终态参考）

```
用户: "3台VM起不来，存储扩容后"
│
│  Claude 内部（不可见的思考）:
│    "存储扩容后VM无法启动，第一反应是存储相关。
│     需要先看VM的错误信息，再看存储状态。
│     这个系统有 acli 工具，可以直接查。"
├── tool_call: acli vm.get_task_error(scope="all_failed")
│     → 结果: "磁盘路径不可达: /sf/data/volume_3"
│
│  Claude 思考:
│    "路径写的是 volume_3，这是扩容新加的卷。
│     路径不到达意味着挂载可能还没完成，或者路径发生了变化。
│     先查存储挂载状态。"
├── tool_call: acli storage.list_volumes()
│     → 结果: "volume_3: STATUS=REMOUNTING"
│
│  Claude 思考:
│    "确认了，volume_3 还在重新挂载中。这是正常的扩容后行为。
│     VM 在检测到磁盘不可达时直接失败，而不是等待。
│     需要查一下这个挂载还需要多久，并告知用户等待策略。"
├── tool_call: acli storage.remount_check()
│     → 结果: "volume_3: progress=73%, estimated_time=90s"
│
│  Claude 思考:
│    "挂载 90 秒内完成，这是最好的消息。
│     告诉用户等待 + 如何监控 + 恢复后的验证步骤。"
│
└── 回答用户:
    "找到原因了。存储扩容时新卷（volume_3）正在重新挂载。
     VM 因为检测到磁盘临时不可达而启动失败，这是正常的扩容后现象。
     
     ✅ 好消息：挂载已完成 73%，约 90 秒后完成。
     
     建议：等待 2 分钟后，在控制台重新启动这 3 台 VM。
     监控命令（无风险）：acli vm.list --status=error 观察状态变化。
     
     如果 10 分钟后挂载仍未完成，再执行：
     acli storage.force_remount volume_3（此操作需要我帮你确认风险）"
```

这个过程中，Claude：
- ✅ **没有任何 RAG 检索**（不需要找"相似案例"）
- ✅ **没有任何策略模型路由**（LLM 直接推理）
- ✅ **工具调用 3 次就收敛**（主动采集的信息恰好够）
- ✅ **回答有根因、有时间线、有操作步骤、有风险说明**

### 5.2 为什么我们暂时做不到这一点

```
Claude 能如此简洁高效，因为：
  • acli 工具是它自主选择调用的（不是平台预设的）
  • 它知道"volume_3 存储不可达" + "STATUS=REMOUNTING" 的机制关系
    （因为训练数据中有类似的 Linux/存储相关知识，可以迁移推理）
  • 200K context 允许不加压缩地保留所有工具输出

GLM-4 的限制：
  • acli 命令的语义 GLM-4 不熟悉（需要 Segment 3 机制知识注入）
  • REMOUNTING 状态到底意味着什么，GLM-4 可能推理错误（需要情节记忆支持）
  • 如果推理链条更长（5+ 步工具调用），context 中的早期信息可能被"淡化"
```

## 六、本项目的技术演进路线图（修订版）

这个路线图的每一步都清楚地标明：**我们在弥补 GLM-4 vs Claude 的哪一个差距**。

```
Phase 0-2（已规划）: "让 GLM-4 知道该如何工作"
──────────────────────────────────────────────
弥补的差距: GLM-4 不知道 HCI 排障的方法论
方法: Prompt 重写（5段式）+ 诊断状态机（S0-S6）
目标得分: 10 → 60

Phase 3（方案 A: 认知记忆）: "让 GLM-4 有 HCI 私有知识"
──────────────────────────────────────────────────────
弥补的差距: GLM-4 的参数中没有 acli/SCP 的私有语义
方法: 情节记忆（7000 案例结构化）+ LLM 主动 recall
目标得分: 60 → 75

Phase 4（方案 B: 因果图）: "让 GLM-4 走最优诊断路径"
──────────────────────────────────────────────────────
弥补的差距: GLM-4 在标准场景中可能走弯路（Claude 不会）
方法: 因果图遍历，算法辅助工具选择
目标得分: 75 → 85

Phase 5（方案 C: 策略蒸馏）: "卸载 GLM-4 的认知负担"
──────────────────────────────────────────────────────
弥补的差距: GLM-4 处理标准场景时的速度和成本劣势
方法: 小模型处理 70% 标准场景，GLM-4 专注复杂场景
目标得分: 85 → 92

Phase X（长期）: "逐渐接近 Claude 式体验"
──────────────────────────────────────────────────────
手段: 更强的基础模型（或开放 Claude/GPT-4 接入）
      + 方案 A/B 积累的私有知识体系
      + 完整工具集（SCP + acli + 知识库）
目标得分: 92 → 95+
```

**关键认知**：本项目的知识工程（方案A/B/C）不是"超越 Claude 的技术创新"，而是"用工程方法弥补 GLM-4 规模不足"。越早认清这一点，越不会在架构选择上走弯路。

## 七、关键设计原则（从 Claude 架构提炼）

以下原则是 Claude 工作方式的核心，**本项目所有架构设计必须向这些原则靠拢**：

### 原则 1：LLM 主权（LLM Sovereignty）

```
❌ 错误设计: 平台决定注入什么知识 → LLM 被动生成
✅ 正确设计: LLM 决定需要什么工具/知识 → 主动获取
```

衡量标准：当用户的问题来临时，是"平台预先决定了 LLM 看到什么"，还是"LLM 自己决定去看什么"？

### 原则 2：工具即信息渠道（Tools as Information Channels）

```
❌ 错误设计: 所有知识都预先注入 Prompt，LLM 从注入内容生成答案
✅ 正确设计: 工具是信息获取的唯一渠道，知识和诊断工具平等对待
```

衡量标准：检索知识库和查询 VM 状态，对 LLM 来说应该是同等地位的工具调用。

### 原则 3：最小注入（Minimal Injection）

```
❌ 错误设计: 每次消息都注入大量 RAG chunks 和历史案例
✅ 正确设计: 只注入永久不变的机制知识（Core Memory），其余按需 recall
```

衡量标准：context window 中预注入的内容是否随每条用户消息增长？如果是，就违反了此原则。

### 原则 4：推理先于检索（Reason Before Retrieve）

```
❌ 错误设计: 用户说"VM起不来" → 立刻检索相关案例 → ...
✅ 正确设计: 用户说"VM起不来" → 思考"缺少什么信息" → 问问题或调工具
```

衡量标准：S0 阶段（症状采集）是否触发任何检索？如果是，就违反了此原则。

### 原则 5：行动前说明（Explain Before Act）

```
Claude 的透明性原则: 任何工具调用都要说明"为什么调这个，看什么"
本项目对应: 建议执行任何命令前，告诉用户此命令的用途和预期输出
```

## 八、一个清醒的认知校准

在用方案 A/B/C 构建知识工程系统时，需要时刻牢记：

```
1. 方案 A/B/C 是手段，不是目的。
   目的是让用户获得类似 Claude 的 HCI 排障体验。

2. HCI 私有知识（acli 命令语义、错误码含义）是刚性需求。
   无论基础模型多强，这些专有知识都需要某种形式的注入。
   区别只是：方案A注入为情节记忆，方案B注入为因果图，Claude注入为参数权重。

3. 工具调用（Tool Use）是不可妥协的核心。
   没有工具调用，无论知识库多完善，系统仍然是"过去案例的搜索引擎"。
   有了工具调用，系统才有可能成为"主动诊断的专家系统"。

4. 当基础模型更强时（GLM-5、DeepSeek V3 等），
   知识工程（方案A/B/C）的积累仍然有价值：
   情节记忆库、因果图、诊断策略模型——
   这些资产换一个更强的基础模型，效果会更好，而不是浪费。
```
