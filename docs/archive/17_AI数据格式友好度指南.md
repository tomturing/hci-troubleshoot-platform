# AI 数据格式友好度指南

**版本**: v1.0  
**日期**: 2026-03-23  
**定位**: 决策依据文档，用于数据格式选型、知识库设计、API 设计、工具 I/O 规范  
**适读对象**: 所有工程师，在做涉及数据格式的技术决策前必读

---

## 一、核心结论（先给结论，后面是详细推导）

```
放入 Context Window 让 LLM 推理    → Markdown 第一
工具调用 I/O（结构化数据交换）      → JSON 第一
知识库存储（供检索的内容）          → Markdown 文档 + JSON 元数据
知识结构定义（类型约束、接口声明）   → TypeScript Types / Pydantic Schema
表达知识关系（因果图、知识图谱）    → 知识图谱（Cypher / 自定义结构）
配置文件（规则、参数、映射）        → YAML
纯表格数据（统计、日志）           → CSV / Parquet
```

## 二、格式友好度完整排序（1 = 最友好）

### 2.1 以"LLM 能否准确理解和推理"为标准

| 排名 | 格式 | 友好度评分 | 核心理由 |
|------|------|-----------|---------|
| **1** | **Markdown** | ⭐⭐⭐⭐⭐ | 结构+自然语言+密度三者最优平衡，LLM 训练语料大量是 Markdown |
| **2** | **纯文本散文** | ⭐⭐⭐⭐½ | 最接近训练分布；擅长表达推理、解释、上下文；缺点是缺结构提示 |
| **3** | **JSON（有 Schema）** | ⭐⭐⭐⭐ | 工具 I/O 的事实标准；明确的 key-value 无歧义；嵌套过深时可读性下降 |
| **4** | **YAML** | ⭐⭐⭐½ | 比 JSON 更接近自然语言；适合配置和规则；缩进报错是隐患 |
| **5** | **TypeScript Types / Pydantic Schema** | ⭐⭐⭐ | 代码推理场景最优；直接表达约束；仅适合结构定义，不适合内容 |
| **6** | **XML** | ⭐⭐½ | 自描述性强（标签即语义）；但过于冗长，向量化后信噪比低 |
| **7** | **CSV** | ⭐⭐ | 适合纯表格；严重损失层次结构；每行没有 key 需要依赖 header 行 |
| **8** | **知识图谱（Cypher / Turtle / RDF）** | ⭐⭐ | 表达关系最优；但 LLM 对 RDF 语法不熟悉；检索时需"翻译"损耗 |
| **9** | **SQL Schema（DDL）** | ⭐½ | LLM 能读懂，但需要额外心理翻译"表结构→现实语义"，认知负担高 |
| **10** | **Parquet / 二进制** | ⭐ | 完全不可读；AI 看二进制等于无信息 |

### 2.2 评分维度说明

每种格式从五个维度评分（各 1 分，满分 5 分）：

| 维度 | 说明 |
|------|------|
| **训练分布相似度** | 这种格式在 LLM 训练语料中的占比和质量 |
| **语义密度** | 单位 token 携带的语义信息量 |
| **结构提示性** | 格式本身是否给 LLM 提供处理提示（如标题、层级、标注）|
| **解析稳定性** | LLM 在处理这种格式时的错误率（幻觉、格式混淆）|
| **人机双读性** | 人类和 AI 同时能高效理解（文档传播、审计、维护）|

## 三、各格式详细分析

### 3.1 Markdown（排名第 1）

**为什么 Markdown 最友好？**

```
维度分析:
  训练分布相似度: ████████ 5/5  （GitHub、技术文档、论文、教程…都大量使用 Markdown）
  语义密度:       ████████ 5/5  （结构符号不占太多 token，内容比例高）
  结构提示性:     ████████ 5/5  （# 表层级，**加粗**表强调，代码块明确语义块边界）
  解析稳定性:     ███████- 4/5  （偶尔 LLM 会多写/少写 # 级别，但总体稳定）
  人机双读性:     ████████ 5/5  （人类可读，LLM 处理效率高）
```

**适用场景**：
- ✅ 知识库文档（SOP、机制知识、案例描述）
- ✅ System Prompt 的各段内容
- ✅ 任何放进 Context Window 让 LLM 推理的内容
- ✅ 情节记忆（EpisodicMemory）的 textual 描述字段
- ✅ 生成给用户的回答

**注意事项**：
- 代码块应明确写语言类型（````python` 而非单独 ` ``` `）
- 嵌套列表不超过 3 层（LLM 超过 3 层后结构感知下降）
- 表格对 LLM 友好，但中文长文本表格会增加 token 消耗

**反例（不要这样做）**：
```markdown
# 故障案例
内容：用户反映虚拟机开不了机，原因是CPU不足，解决方法是关几台VM释放资源，
版本是6.7，时间是2025年1月，相关命令是acli system top查看CPU，
然后acli vm.off关闭目标VM，完成后acli vm.on重新开机...
```

**正确做法（结构化 Markdown）**：
```markdown
## 案例：虚拟机开机失败 — CPU 资源不足

**故障现象**: 虚拟机无法启动  
**根因**: CPU overcommit，宿主机可用 CPU < VM 配置需求  
**适用版本**: HCI 6.5+

### 诊断步骤
1. 查看 CPU 占用: `acli system top`（观察 %CPU 列）
2. 确认 VM 配置: `acli vm.get --detail`

### 解决方案
- 临时: `acli vm.off [低优先级VM]` 释放资源
- 根本: 扩容宿主机或重新规划 VM 资源
```

### 3.2 JSON（排名第 3）

**适用场景**：
- ✅ 工具调用的参数（tool_call arguments）
- ✅ 工具调用的返回值（tool_call results）
- ✅ 知识原子的元数据字段（PostgreSQL JSONB 列中的内容）
- ✅ API 请求/响应 Body
- ⚠️ **不适合**放进 Context Window 让 LLM"阅读理解"

**最优 JSON 设计原则**：

```json
// ❌ 不友好：嵌套过深，key 过于技术化
{
  "d": {"t": "S2", "h": [{"i": 1, "t": "cpu", "c": 0.7}]}
}

// ✅ 友好：扁平化，key 见名知意
{
  "diagnostic_stage": "S2",
  "hypotheses": [
    {
      "id": 1,
      "description": "CPU资源不足导致VM无法获得调度",
      "confidence": 0.70,
      "status": "pending"
    }
  ]
}
```

**关键原则**：
1. key 名称使用完整语义词（`diagnostic_stage` 而非 `d_s`）
2. 嵌套不超过 3 层（超过 3 层 LLM 出错率上升）
3. 数组元素保持同质性（同类型对象）
4. 布尔值优于 0/1（`"is_verified": true` 而非 `"verified": 1`）
5. 时间用 ISO 8601（`"created_at": "2026-03-23T10:00:00Z"`）

### 3.3 YAML（排名第 4）

**适用场景**：
- ✅ 配置文件（规则映射、参数设定）
- ✅ 情节记忆的规范定义文档（category_baseline.yaml 风格）
- ✅ 工具的 Function Schema 声明
- ⚠️ 生产数据交换慎用（缩进错误难以调试）

**最优实践**：
```yaml
# ✅ 适合 YAML：结构清晰，接近自然语言
knowledge_atom:
  id: "ka-vm003-cpu-step-01"
  type: diagnostic_step
  
  trigger:
    stage: S2
    keywords:
      - "CPU不足"
      - "此主机剩余可配置CPU不足"
  
  steps:
    - seq: 1
      description: "查看主机 CPU 占用"
      command: "acli system top"
      risk_level: 1
```

### 3.4 TypeScript Types / Pydantic Schema（排名第 5）

**特别价值**：在代码推理场景，这两种格式是**最友好的**。LLM 处理 TypeScript 类型和 Python Pydantic 模型时，理解精度比 JSON Schema 高约 30%。

```typescript
// TypeScript — LLM 理解约束的最优格式
interface DiagnosticState {
  stage: 'S0' | 'S1' | 'S2' | 'S3' | 'S4' | 'S5' | 'S6';
  hypotheses: Array<{
    id: number;
    text: string;
    confidence: number; // 0.0 to 1.0
    status: 'pending' | 'confirmed' | 'excluded';
  }>;
  confirmedFacts: Record<string, string>;
}
```

```python
# Pydantic — Python 工程场景最优
class DiagnosticState(BaseModel):
    stage: Literal['S0', 'S1', 'S2', 'S3', 'S4', 'S5', 'S6']
    hypotheses: list[Hypothesis]
    confirmed_facts: dict[str, str] = {}
    
    class Config:
        # schema 约束直接影响 LLM 的理解精度
        json_schema_extra = {"example": {...}}
```

**适用场景**：
- ✅ LLM 生成代码的上下文（让 LLM 了解类型约束）
- ✅ Tool Call Schema 定义
- ✅ 内部数据模型文档化
- ❌ 不适合作为知识内容的传递格式

## 四、本项目的格式选型决策表

这是一个可直接引用的决策表，任何涉及数据格式选型的场景都应先查阅。

| 使用场景 | 推荐格式 | 理由 | 禁止格式 |
|---------|---------|------|---------|
| **System Prompt 的机制知识（Segment 3）** | Markdown | 结构清晰，高密度，LLM 推理效率最高 | XML、纯文本大段 |
| **情节记忆（EpisodicMemory）的文本描述** | Markdown | 保留结构，方便 LLM 召回时理解上下文 | JSON 纯结构 |
| **情节记忆的元数据（symptom_signature 等）** | JSON（JSONB）| 结构化字段，便于过滤和查询 | YAML（运行时用） |
| **工具调用参数（tool_call arguments）** | JSON | 工具调用的事实标准，解析稳定 | YAML、Markdown |
| **工具调用返回值（SCP API / acli 输出）** | JSON + 关键字段 Markdown 化 | JSON 保证结构，Markdown 提升 LLM 解读精度 | XML |
| **SOP 排障手册（DocxExtractor 输出）** | Markdown | 保留章节层级，直接可用于知识检索 | CSV、纯文本 |
| **acli 命令文档** | Markdown + 代码块 | 命令在代码块中，说明在文本中，LLM 识别率最高 | 纯文本混排 |
| **因果图节点/边（数据库存储）** | JSON（JSONB）+ 图数据结构 | JSONB 是 PostgreSQL 的最优选择 | XML、CSV |
| **API 接口规范（工程师读）** | YAML（OpenAPI）| OpenAPI 规范的事实标准格式 | JSON（冗长） |
| **配置文件（keywords_map、category_baseline）** | YAML | 人类可读，AI 可解析，最优平衡 | CSV、XML |
| **训练数据集（蒸馏数据集）** | JSONL（每行一个 JSON）| 大批量机器处理的标准格式 | CSV（丢失嵌套）|
| **Prompt 中注入的参考案例** | Markdown（精简）| 单条案例 < 300 token，结构化 Markdown 效率最高 | 原始 JSON |
| **诊断状态（存入数据库）** | JSON（JSONB）| 结构化，便于查询和更新 | 文本序列化 |
| **对话历史（存入数据库）** | JSONL | 每条消息一个 JSON 对象，标准对话格式 | 纯文本拼接 |

## 五、LLM 解析格式的实际失败率（实测参考）

基于 GPT-4 / Claude 的公开研究和社区实践，不同格式下 LLM 的**结构解析失败率**（无法正确提取信息）：

| 格式 | 结构解析失败率 | 主要失败模式 |
|------|-------------|------------|
| Markdown | ~2-5% | 偶尔误读层级 |
| JSON（简单）| ~3-6% | 遗漏字段，类型错误 |
| JSON（复杂嵌套）| ~15-25% | 深层嵌套时结构混淆 |
| YAML | ~8-12% | 缩进敏感，容易引入错误 |
| XML | ~18-30% | 标签解析混乱，遗漏属性 |
| CSV | ~20-35% | 列对齐错误，多值字段处理失败 |
| 知识图谱（RDF/Cypher）| ~35-50% | 语法不熟悉，关系方向混淆 |

**来源说明**：这些数据来自 LangChain、LlamaIndex 社区的 benchmark 报告（2024），以及 Anthropic/Google 的 Function Calling 性能测试报告，仅供参考量级，具体数值随模型版本和 Prompt 设计变化。

## 七、快速 Cheatsheet

```
你要做的事 → 用这个格式
─────────────────────────────────────────────
写 Prompt 内容      → Markdown
定义数据结构        → Pydantic / TypeScript
工具调用参数/返回   → JSON
存数据库           → JSONB（PostgreSQL）
写配置文件         → YAML
写知识文档         → Markdown
大批量机器处理      → JSONL
表格数据导出       → CSV
─────────────────────────────────────────────
永远不要用 XML      → 冗长，高失败率
放进 Prompt 的 JSON → 必须扁平化（≤3层）
```
