# 24_发布检查清单与回滚SOP

> 适用范围：方案 21（GitOps 标准化开发流程）
> 
> 目标：将发布前检查、发布后验证、异常回滚流程标准化，确保发布可控、可追溯、可恢复。

---

## 1. 发布前检查（Preflight）

## 1.1 变更准备

1. 变更已通过 PR 审核并合并。
2. CI 全部通过（后端/前端/Helm/构建）。
3. 已确认目标镜像标签存在且不可变。
4. 已确认影响范围与风险等级。

## 1.2 配置与依赖检查

1. 环境声明仓库（或 values override）已更新到目标 tag。
2. Secret/ConfigMap 变更已审核，未暴露敏感信息。
3. 数据库变更有迁移脚本与回退说明。
4. 关键依赖服务状态正常（DB、Redis、Ingress、观测栈）。

## 1.3 发布审批

1. 发布窗口已确认。
2. 发布责任人与观察责任人已明确。
3. 生产发布已获得审批（必须）。

---

## 2. 发布执行（Runbook）

## 2.1 GitOps 路径（推荐）

1. 合并环境声明变更 PR（更新 image.tag）。
2. 触发 Argo CD 同步（或等待自动同步）。
3. 观察 rollout 过程，确认所有目标工作负载升级成功。

## 2.2 脚本兜底路径（仅应急）

1. 使用脚本执行发布：`bash scripts/k3s-release.sh --services all`。
2. 发布后必须补齐环境声明，避免配置漂移。

---

## 3. 发布后验证（Postflight）

## 3.1 健康检查

1. 核心服务 Pod 全部 Ready。
2. 关键接口健康检查通过。
3. 前端关键页面可访问，核心交互正常。

## 3.2 指标观察（建议 30 分钟窗口）

1. 错误率无异常升高。
2. P95 延迟无明显退化。
3. CPU/内存无持续异常飙升。
4. 关键业务链路日志包含 trace_id。

## 3.3 验证命令（示例）

```bash
# 命名空间与工作负载状态
k3s kubectl -n hci get pods
k3s kubectl -n hci get deploy

# 最近发布历史
helm -n hci history hci-platform

# 项目内验证脚本
bash scripts/k3s-verify.sh
```

---

## 4. 回滚触发条件

满足任一条件立即回滚：

1. 核心接口持续不可用（超过 5 分钟）。
2. 错误率超过基线阈值且持续上升。
3. 用户核心路径不可用（创建工单、会话、回复）。
4. 发布后出现不可接受的数据一致性风险。

---

## 5. 回滚SOP

## 5.1 GitOps 回滚（首选）

1. 回退环境声明仓库到上一个稳定 commit。
2. 触发 Argo CD 同步到回退版本。
3. 执行发布后验证，确认服务恢复。

## 5.2 Helm 回滚（应急）

```bash
# 查看修订历史
helm -n hci history hci-platform

# 回滚到上一个稳定 revision
helm -n hci rollback hci-platform <REVISION>

# 验证状态
k3s kubectl -n hci get pods
bash scripts/k3s-verify.sh
```

## 5.3 回滚后动作

1. 标记本次发布失败并冻结继续发布。
2. 输出故障摘要：影响范围、根因、恢复耗时。
3. 创建修复任务并安排复盘。

---

## 6. 发布记录模板

```text

发布责任人：

镜像标签：

验证结果：通过/失败
是否回滚：否/是（回滚到 revision X）
恢复耗时（分钟）：
备注：
```

---

## 7. 例行演练要求

1. 每周至少执行 1 次 staging 回滚演练。
2. 每月至少执行 1 次生产级别桌面推演。
3. 每次演练必须产出记录并更新 SOP。
