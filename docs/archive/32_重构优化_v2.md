# HCI 智能排障平台 - 重构优化记录

更新时间: 2026-03-01

---

## 1. 背景

在完成 v2.0 多助手架构代码落地及 K3s 部署后，对全量代码进行了系统性深度审查。围绕 10 个维度与业界最佳实践对照，识别出 21 项改进点，按紧急程度分为 4 个阶段（Phase）实施。

本文档记录审查发现、改进方案、已完成的 Phase 1 + Phase 2 变更，以及后续 Phase 3 / Phase 4 计划。

---

## 2. 代码审查 — 10 维度对照

### 2.1 数据一致性 (Data Consistency)

| # | 发现 | 严重度 | 状态 |
|---|------|--------|------|
| D-1 | `conversation_repo.add_message()` 既手动 `conversation.message_count += 1` 又有 DB 触发器 `tg_update_message_count`，导致双重递增 | 🔴 Bug | ✅ 已修复 |
| D-2 | `add_message()` 显式 `await session.commit()` 与 `get_session()` 上下文管理器的自动 commit 冲突，可能导致部分提交 | 🔴 Bug | ✅ 已修复 |

**修复方案**: 移除手动 `message_count += 1` 和显式 `session.commit()`，改用 `session.flush()` + `session.refresh(message)` 获取 DB 触发器生成的值。

### 2.2 安全性 (Security)

| # | 发现 | 严重度 | 状态 |
|---|------|--------|------|
| S-1 | API Gateway CORS 配置 `allow_origins=["*"]` + `allow_credentials=True`，RFC 6454 不兼容 | 🔴 Bug | ✅ 已修复 |

**修复方案**: 新增 `ALLOWED_ORIGINS` 环境变量，`config.py` 提供 `cors_origins` 属性返回 `list[str]`。CORS 中间件改用 `settings.cors_origins`。

### 2.3 可靠性 (Reliability)

| # | 发现 | 严重度 | 状态 |
|---|------|--------|------|
| R-1 | 关闭工单后未释放 Scheduler Pod 资源 | 🟡 资源泄漏 | ✅ 已修复 |
| R-2 | `routes/assistants.py` 通用 `except Exception` 吞掉 HTTPException，返回 500 而非原始状态码 | 🟡 Bug | ✅ 已修复 |

**修复方案**:
- R-1: `api-gateway/routes/cases.py` 关闭工单成功后调用 `POST /api/scheduler/pods/release` 释放 Pod。
- R-2: 在 `except Exception` 前添加 `except HTTPException: raise`。

### 2.4 状态持久化 (State Persistence)

| # | 发现 | 严重度 | 状态 |
|---|------|--------|------|
| P-1 | Scheduler 的 `allocations` 使用内存 Dict，服务重启后全部丢失 | 🟡 架构缺陷 | ✅ 已修复 |

**修复方案**: 迁移至 Redis Hash (`scheduler:allocations`)，通过 `_get_allocation` / `_set_allocation` / `_del_allocation` / `_get_all_allocations` 封装读写。服务重启后状态自动恢复。

### 2.5 启动健壮性 (Startup Robustness)

| # | 发现 | 严重度 | 状态 |
|---|------|--------|------|
| B-1 | `pod_pool.initialize()` 是空 `TODO` stub，重启后无法感知已运行的 K8s Pod | 🟡 架构缺陷 | ✅ 已修复 |
| B-2 | `asyncio.create_task()` 无异常回调，后台任务静默失败 | 🟡 隐患 | ✅ 已修复 |

**修复方案**:
- B-1: `PodPool.initialize()` 实现 K8s label selector 扫描，按 Pod 状态分类进 idle/active 队列；新增 `PodPoolManager.initialize_all()`。
- B-2: 引入 `_safe_create_task()` + `_task_done_callback()`，异常自动记录日志。

### 2.6 依赖注入 (Dependency Injection)

| # | 发现 | 严重度 | 状态 |
|---|------|--------|------|
| DI-1 | 所有 4 个服务使用全局变量做依赖注入 (`database_manager = None; def set_dep()`) | 🟡 架构问题 | ✅ 已修复 |
| DI-2 | 15+ 文件使用 `sys.path.insert(0, ...)` hack 解决导入路径 | 🟡 工程问题 | ✅ 已修复 |

**修复方案**:
- DI-1: 全部迁移至 `app.state` 模式（`request.app.state.xxx`），消除全局可变状态。
- DI-2: 在 `docker-compose.yml` 所有后端服务添加 `PYTHONPATH: /app`；创建 5 个 `conftest.py` 文件处理测试导入。移除了 14 处 `sys.path.insert` 调用。

### 2.7 错误处理 (Error Handling)

| # | 发现 | 严重度 | 状态 |
|---|------|--------|------|
| E-1 | `conversation_service.send_message_stream_only()` 的 SSE 错误仅打日志，不回传客户端 | 🔵 改进项 | ⬜ Phase 3 |
| E-2 | `k8s_client.create_pod()` 无重试机制 | 🔵 改进项 | ⬜ Phase 3 |

### 2.8 可观测性 (Observability)

| # | 发现 | 严重度 | 状态 |
|---|------|--------|------|
| O-1 | AI 调用相关的关键指标（首字延迟、TTFT、Token 使用量）未暴露 | 🔵 改进项 | ⬜ Phase 3 |
| O-2 | `pod_pool` 无 gauge 类型指标 (pool_size / idle_count / active_count) | 🔵 改进项 | ⬜ Phase 3 |

### 2.9 配置管理 (Configuration)

| # | 发现 | 严重度 | 状态 |
|---|------|--------|------|
| C-1 | Scheduler 缺少 `REDIS_URL` 配置项 | 🟡 缺失 | ✅ 已修复 |

### 2.10 测试覆盖 (Test Coverage)

| # | 发现 | 严重度 | 状态 |
|---|------|--------|------|

## 4. Phase 1 — 线上 Bug 修复（已完成）

### 4.1 修复双重提交 + 双重递增 (D-1, D-2)

**文件**: `backend/conversation-service/app/repositories/conversation_repo.py`

**变更**:
- 移除 `await self.session.commit()`，依赖 `get_session()` 上下文管理器统一 commit
- 移除 `conversation.message_count += 1`，依赖 DB 触发器 `tg_update_message_count`
- 改用 `await self.session.flush()` + `await self.session.refresh(message)` 获取 DB 生成值

### 4.2 修复 CORS 安全配置 (S-1)

**文件**: `backend/api-gateway/app/config.py`, `backend/api-gateway/app/main.py`

**变更**:
- `config.py` 新增 `ALLOWED_ORIGINS` 环境变量（默认 `http://localhost:3001,http://localhost:3002`）
- `config.py` 新增 `cors_origins` 属性，将逗号分隔字符串解析为 `list[str]`
- `main.py` CORS 中间件改用 `settings.cors_origins`
- `docker-compose.yml` 为 api-gateway 注入 `ALLOWED_ORIGINS`

### 4.3 修复工单关闭后 Pod 资源泄漏 (R-1)

**文件**: `backend/api-gateway/app/routes/cases.py`

**变更**:
- 工单关闭成功后，调用 `POST {SCHEDULER_SERVICE_URL}/api/scheduler/pods/release`
- 释放调用使用 try/except 包裹，Pod 释放失败不影响工单关闭结果（记录警告日志）

### 4.4 修复 HTTPException 被吞 (R-2)

**文件**: `backend/api-gateway/app/routes/assistants.py`

**变更**:
- 在通用 `except Exception` 之前添加 `except HTTPException: raise`
- 确保 HTTPException 携带的状态码正确传播到客户端

### 4.5 删除过时文件

**文件**: `backend/conversation-service/app/services/openclaw_client.py`

**说明**: 该文件为 v1.0 硬编码 OpenClaw 客户端，已被 `ai_client.py` (Protocol + Factory 模式) 完全替代。v2.0 代码落地时遗留未清理。

## 5. Phase 2 — 架构加固（已完成）

### 5.1 Scheduler 状态迁移至 Redis (P-1)

**文件**: `backend/scheduler-service/app/services/scheduler_service.py`, `backend/scheduler-service/app/config.py`

**变更**:
- 新增 `REDIS_URL` 配置项
- `SchedulerService` 构造函数新增 `redis_manager` 参数
- 使用 Redis Hash `scheduler:allocations` 存储 `case_id → {pod_name, assistant_type}` 映射
- 封装 `_get_allocation()`, `_set_allocation()`, `_del_allocation()`, `_get_all_allocations()` 方法
- `get_allocation_info()` 和 `get_status()` 改为 async 方法

### 5.2 实现 pod_pool.initialize() (B-1)

**文件**: `backend/scheduler-service/app/services/pod_pool.py`

**变更**:
- `PodPool.initialize()` 实现 K8s label selector 扫描（`app={assistant_type}-assistant`）
- 按 Pod 状态分类：Running → idle/active；Failed/Unknown → 自动清理
- 新增 `PodPoolManager.initialize_all()` 方法
- 引入 `_safe_create_task()` + `_task_done_callback()` 替代裸 `asyncio.create_task()`

### 5.3 消除全局变量 DI (DI-1)

**文件**: 4 个服务的 `main.py`

**变更**:
| 服务 | 旧模式 | 新模式 |
|------|--------|--------|
| api-gateway | `redis_manager = None; session_manager = None` | `app.state.redis_manager`, `app.state.session_manager` |
| case-service | `database_manager = None` | `app.state.database_manager` |
| conversation-service | `database_manager = None; ai_registry = None` | `app.state.database_manager`, `app.state.ai_registry`, `app.state.scheduler_client` |
| scheduler-service | 全局变量 | `app.state.redis_manager`, `app.state.scheduler_service` |

### 5.4 消除 sys.path.insert (DI-2)

**文件**: 14 个文件（7 源文件 + 7 测试文件）

**变更**:
- `docker-compose.yml`: 所有后端服务添加 `PYTHONPATH: /app` 环境变量
- 移除所有 `sys.path.insert(0, ...)` 调用
- 创建 5 个 `conftest.py` 文件处理测试环境的 Python 路径

### 5.5 修复 Scheduler 服务启动流程

**文件**: `backend/scheduler-service/app/main.py`

**变更**:
- 初始化 `RedisManager`，lifespan 中 connect/close
- 后台初始化任务包裹在 `_safe_start()` 中，异常自动记录
- Shutdown 时取消未完成的初始化任务
- `k8s_client.list_pods()` 返回标准化 dict（而非原始 K8s Pod 对象）

### 5.6 更新过时测试

**文件**: 3 个测试文件

| 测试文件 | 变更 |
|---------|------|
| `tests/unit/test_case_service.py` | 完全重写，匹配 v2.0 API 签名 `create_case(case_create)`, `confirm_case(case_id)`, `close_case(case_id)` |
| `backend/scheduler-service/tests/unit/test_scheduler_service.py` | 完全重写，Mock Redis (hget/hset/hdel/hgetall)，测试 allocate/release/status |
| `backend/conversation-service/tests/integration/test_conversation_integration.py` | Mock 路径从 `openclaw_client.OpenClawClient` 更新为 `ai_client.OpenClawAssistant` |
