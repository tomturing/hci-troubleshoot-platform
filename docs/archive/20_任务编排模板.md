# 18_任务编排模板

> 本文件作为通用「任务编排」模板，供后续在 docs/18_xx、19_xx、20_xx 等文件中复制使用。  
> 优先级说明：P1 = 生产环境完整性；P2 = 功能完善；P3 = 低优先级优化  
> 创建日期: 2026-03-13

---

# Task N：任务标题（Px）

```
你是一名负责 hci-troubleshoot-platform <子系统/模块> 的 agent。

【仓库】
git clone https://github.com/tomturing/hci-troubleshoot-platform.git
cd hci-troubleshoot-platform

【背景】
- 简要描述当前业务/技术背景
- 说明已有实现状态（哪些在 docs / DB / 代码里已经有，哪些是缺口）
- 指明需要解决的核心问题或风险

【任务目标】
用 3–5 点列清楚本任务完成后**可验证**的目标，例如：
1. 在 <模块> 中补齐 <功能/逻辑>，与现有设计文档对齐
2. 完成 <服务 A> 与 <服务 B> 的联调，打通完整链路
3. 提供端到端验证步骤，确保无回归

【涉及服务 / 文件范围】
- backend/<xxx-service>/
- frontend/<xxx>/
- database/<xxx>.sql
- docs/<相关设计文档>.md
（仅列允许修改/阅读的范围，便于 Agent 约束变更面）

【详细实现步骤】

Step 1：对齐设计文档与当前实现
- 阅读：
  - docs/<架构或设计>.md 的相关章节
  - docs/<评分/任务编排等>.md 中对应章节
- 使用 grep / rg 查找关键符号，确认当前实现状态
- 梳理「已实现 vs 仅在文档中」的差异点，必要时在 PR 描述中简要列出

Step 2：核心逻辑实现
- 描述本任务最核心的一两处代码改动：
  - 需要新增/修改的类、函数或接口
  - 输入/输出、边界条件、错误处理要求
- 给出关键算法/状态机/数据流的伪代码或要点

Step 3：数据库 / 配置 / 兼容性检查（如适用）
- 若涉及 DB：
  - 说明是否需要新建 migration 文件（以及命名规范）
  - 明确禁止直接改 init_schema.sql 的场景
- 若涉及配置（env / Helm / 前端 env）：
  - 指明新增的配置项及默认值策略
- 说明对现有调用方的兼容性要求（向后兼容 / 允许破坏性变更）

Step 4：测试与验证
- 单元测试：
  - 新增测试文件路径
  - 关键用例列表（正常、边界、异常）
- 集成 / E2E 验证步骤：
  - curl / 前端操作步骤
  - 需要在日志 / DB / Grafana 中观察到的结果

【约束】
- 不修改与本任务无关的模块 / 服务
- 代码注释必须使用中文
- Python 使用 uv / 前端使用 pnpm，不引入无必要的新依赖
- 遵守 CLAUDE.md / AGENTS.md 中的项目规范

【验收标准】
- [ ] 单元测试通过（列出具体命令，如 uv run pytest backend/xxx/tests -q）
- [ ] 对应接口 / 功能在本地或 dev 环境按步骤验证通过
- [ ] 相关日志 / 指标 / DB 状态符合设计预期
- [ ] 无新增 linter 错误（make lint 或对应命令通过）
```

---

> 后续新增任务时，只需复制上面的 Task 模板块，将 Task 编号、标题、优先级、背景、目标与文件范围替换即可。  
> 同一文件内可以连续定义多个 Task：Task N，Task N+1，Task N+2 …… 每个 Task 独立闭环、可单独派发给 agent。
