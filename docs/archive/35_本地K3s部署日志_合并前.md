## 问题详情

### DEPLOY-001：global.domain 空字符串触发 Helm chart 安全校验

**发现时间**: 2026-03-24，helm lint 阶段  
**错误日志**:
```
engine.go:227: [INFO] Fail: ❌ 检测到非开发域名，但 secrets.postgresPassword 仍为默认弱密码 'dev_password_123'
```

**根因**:  
`deploy/helm/hci-platform/templates/secret.yaml` 中有以下校验逻辑：
```go
{{- if and (ne .Values.global.domain "hci.local") (eq .Values.secrets.postgresPassword "dev_password_123") -}}
  {{- fail "❌ 检测到非开发域名，但 secrets.postgresPassword 仍为默认弱密码..." -}}
{{- end -}}
```
`.local/values-prod.override.yaml` 中 `domain: ""` 不等于 `"hci.local"`，同时密码为默认弱密码，触发 `fail`。

`helm lint` 将 `fail` 输出为 `[INFO] Fail:` 且不计入 chart 失败数，但 `helm install/upgrade` 实际执行时 `fail` 会导致部署中断。

**修复**:  
将 `.local/values-prod.override.yaml` 中的 `global.domain` 从 `""` 改为 `"hci.local"`，旁路安全校验（本地开发不强制要求强密码）。

**影响范围**: 首次本地部署时必须注意；脚本 `k3s-deploy.sh --env dev` 在 override 不存在时会通过 `--set global.domain=<IP>.nip.io` 自动注入域名，走 nip.io 路径需同时设置非默认密码。

### DEPLOY-002：publicUrl 使用裸 IP，Ingress host 验证失败

**发现时间**: 2026-03-24，第 1 次 helm install 时  
**错误日志**:
```
Error: 2 errors occurred:
  * Ingress.networking.k8s.io "hci-ingress" is invalid: spec.rules[0].host: 
    Invalid value: "172.26.101.255": must be a DNS name, not an IP address
  * Ingress.networking.k8s.io "grafana-ingress" is invalid: spec.rules[0].host: 
    Invalid value: "172.26.101.255": must be a DNS name, not an IP address
```

**根因**:  
`ingress.yaml` 模板从 `publicUrl` 提取 hostname 写入 `spec.rules[].host`：
```go
{{- $host = .Values.global.publicUrl | trimPrefix "https://" | trimPrefix "http://" | regexFind "^[^/:]+" }}
```
当 `publicUrl: "http://172.26.101.255"` 时，提取结果为 `172.26.101.255`（裸 IP），Kubernetes Ingress 规范要求 `host` 字段必须是 DNS 名称。

**修复**:  
将 `publicUrl` 改为 `"http://172.26.101.255.nip.io"`。  
[nip.io](https://nip.io) 是公共 DNS 服务，将 `172.26.101.255.nip.io` 自动解析回 `172.26.101.255`，满足 DNS 名称格式要求。

**访问方式**: 部署后通过 `http://172.26.101.255.nip.io/` 访问（需要 DNS 可达 nip.io）。

**替代方案**: 若环境无法访问 nip.io，可将 `publicUrl` 设为 `""` 使 Ingress 规则不带 host 限制（匹配任意主机），再通过裸 IP 访问。

### DEPLOY-003：hci-openclaw 镜像未构建，openclaw/learningclaw Pod 失败

**发现时间**: 2026-03-24，第 2 次 helm install 后 Pod 调度时  
**错误现象**:
```
openclaw-*         0/1  ErrImageNeverPull
learningclaw-0     0/1  Init:ErrImageNeverPull
productionclaw-*   0/1  Init:ErrImagePull
```

**根因**:  
`hci-openclaw` 镜像未在本地构建，需要独立的 `openclaw` 仓库（路径 `../openclaw`）。`k3s-build.sh` 在检测到该目录不存在时会跳过该镜像构建并提示警告，但 Helm chart 默认启用了三个依赖该镜像的工作负载：
- `openclaw` Deployment（主服务）
- `learningclaw` StatefulSet（使用 hci-openclaw:latest）
- `productionclaw` StatefulSet（使用 hci-openclaw:2026.03.24-1533-ea69e26）

`productionclaw-pool-*` Pod 的 `imagePullPolicy` 不是 `Never`，因此内核尝试网络拉取（`ErrImagePull`），而非静默失败。

**修复**:  
在 `.local/values-prod.override.yaml` 中添加：
```yaml
openclaw:
  enabled: false
learningclaw:
  enabled: false
productionclaw:
  enabled: false
config:
  assistantRegistryJson: "{}"  # 防止 scheduler 继续创建热备 Pod
```

**范围影响**:  
- AI 对话功能不可用（需要 openclaw 作为 AI backend）
- conversation-service 健康检查返回 `degraded` 而非 `healthy`
- 核心工单流程（创建/查询/关闭）不受影响

### DEPLOY-004：生产 claw pool 孤立 Pod 无法被 Helm 清理

**发现时间**: 2026-03-24，第 3 次 helm upgrade 后  
**错误现象**:
```
productionclaw-pool-bb0bb207  0/1  Init:ImagePullBackOff
productionclaw-pool-e2bd78bb  0/1  Init:ImagePullBackOff
```

**根因**:  
第 2 次 Helm 部署成功后，`scheduler-service` 根据 `assistantRegistryJson` 配置动态创建了 `productionclaw-pool-*` Pod（这些 Pod 不是 Helm 管理的资源）。第 3 次 Helm upgrade 禁用 `productionclaw` StatefulSet 后，这些孤立 Pod 继续存在，仍然因为镜像拉取失败而报错。

**修复**:
```bash
sudo -n k3s kubectl delete pod productionclaw-pool-bb0bb207 productionclaw-pool-e2bd78bb -n hci-troubleshoot
```

**后续防范**:  
- 首次部署前在 override 文件中设置 `config.assistantRegistryJson: "{}"` 可避免 scheduler 创建孤立 pod
- 如需部署 openclaw，必须先构建 `hci-openclaw` 镜像并导入 k3s，再更新 `assistantRegistryJson`

### DEPLOY-006 详情：kb-service FastAPI 204 断言错误

**发现时间**: 2026-03-24，Pod 启动日志  
**错误日志**:
```
AssertionError: Status code 204 must not have a response body
File "/app/app/routes/review.py", line 286
@router.delete("/{atom_id}", status_code=204, response_class=Response)
```

**根因**:  
FastAPI 0.109.0 在注册路由时会调用 `is_body_allowed_for_status_code(status_code)` 断言。当 `status_code=204` 时，此断言要求路由不能有任何响应体定义。即使指定 `response_class=Response` 也不能绕过该断言（`Response` 作为 `response_class` 不影响路由注册时的检查）。

**正确修复方式**:  
不在装饰器中指定 `status_code=204`，而是在函数体内直接返回 `Response(status_code=204)`：

```python
# 修复前（错误）
@router.delete("/{atom_id}", status_code=204, response_class=Response)
async def delete_atom(request: Request, atom_id: str) -> None:
    ...

# 修复后（正确）
@router.delete("/{atom_id}")
async def delete_atom(request: Request, atom_id: str) -> Response:
    ...
    return Response(status_code=204)
```

**文件**: `backend/kb-service/app/routes/review.py`

### DEPLOY-008 详情：dataLayer.manage=false 导致 postgres/redis 未部署

**发现时间**: 2026-03-24，健康检查阶段  
**错误现象**:
```json
{"status":"degraded","service":"case-service","dependencies":{"database":"unavailable"}}
```

**根因**:  
`values.yaml` 中 `dataLayer.manage: false`，导致 postgres/redis StatefulSet 的 Helm 模板被条件跳过：
```yaml
{{- if and .Values.dataLayer.manage .Values.postgres.enabled }}
```
`values-dev.yaml` 也未覆盖此值，因此 postgres/redis 未被 Helm 部署。pgvector 和 redis 镜像已存在于 k3s containerd（之前 master 分支部署时已拉取），无需再次导入。

**修复**:  
在 `.local/values-prod.override.yaml` 中添加：
```yaml
dataLayer:
  manage: true
```
然后执行 `helm upgrade`。同时提前创建了 postgres 备份目录：
```bash
mkdir -p /home/node/hci-backups/postgres
```

**注意事项**:  
- postgres/redis 镜像使用默认 `imagePullPolicy`（`IfNotPresent`），不受 `global.imagePullPolicy: Never` 影响
- 存储使用 `local-path`（k3s 内置 StorageClass），PVC 自动创建

> **背景**：按照 GitOps 双仓规范重新部署，使用独立的 `hci-platform-env` 环境仓库管理各环境差异化配置，实现 app repo 与 config repo 分离。

### 双仓模式概述

| 仓库 | 路径 | 职责 |
|------|------|------|
| App Repo | `/mnt/d/aihci/hci-troubleshoot-platform` | 业务代码、Helm chart、部署脚本 |
| Env Repo | `/mnt/d/aihci/hci-platform-env` | 各环境 values 差异文件（含密钥） |

**env repo 结构**:
```
hci-platform-env/
  environments/
    dev/values.yaml     ← 开发环境配置（含 ghcr token、postgres 密码）
    staging/values.yaml
    prod/values.yaml
```

### 双仓模式 Helm values 叠加顺序

```
1. deploy/helm/hci-platform/values.yaml      ← chart 默认值
2. hci-platform-env/environments/dev/values.yaml  ← env repo 差异值
3. .local/values-dualrepo-local.yaml          ← 本地覆盖（不进 git，含 publicUrl 等）
```

### 新建文件

| 文件 | 说明 |
|------|------|
| `scripts/ops/k3s-deploy-dualrepo.sh` | 双仓模式本地 K3s 部署脚本（自动检测 postgres 管理模式、处理 `assistantRegistryJson` Helm 解析问题） |

### DUAL-003 详情：Postgres 密码不匹配（PVC 持久化问题）

**发现时间**: 2026-03-25，页面首次使用"创建工单"功能时  
**错误链**:
```
前端: 创建工单失败: Request failed with status code 500
case-service 日志: asyncpg.exceptions.InvalidPasswordError: password authentication failed for user "hci_admin"
```
**根因详解**:
- postgres-0 容器于 Revision 1（2026-03-24）初始化，当时密码为 `dev_password_123`（旧版 override 默认值）
- PostgreSQL 密码写入 PVC 中的 `pg_authid` 系统表，随数据持久化
- 双仓 env repo `dev/values.yaml` 中 `secrets.postgresPassword: "dev_postgres_2026"`，K8s Secret `hci-secrets` 中该字段为新密码
- **Helm upgrade 只更新 K8s Secret，不会执行 `ALTER USER`**，导致应用连接串密码 ≠ 数据库实际密码

**修复命令**:
```bash
# 将数据库密码同步为 K8s Secret 中的值
sudo -n k3s kubectl exec -n hci-troubleshoot postgres-0 -- \
  psql -U hci_admin -d hci_troubleshoot -c "ALTER USER hci_admin PASSWORD 'dev_postgres_2026';"

# 重启所有 Deployment 重置连接池
sudo -n k3s kubectl rollout restart deployment -n hci-troubleshoot
```
**参考文档**: 生产运维指南 E-03 节（内容与此完全吻合）

**教训**: 本地重新部署或切换 env 时，若 postgres PVC 已存在且密码变更，**必须手动 `ALTER USER` 同步密码**，或先删除 PVC 重建（会丢失数据）。

### DUAL-005 详情：Clash TUN 模式劫持 nip.io DNS

**发现时间**: 2026-03-25，部署成功后首次浏览器访问  
**诊断过程**:
1. WSL 内部测试正常：`http://172.26.101.255:80/` → 200，`http://172.26.101.255:32436/` → 200
2. DNS 解析异常：`socket.gethostbyname('172.26.101.255.nip.io')` → `198.18.0.175`（Fake-IP）
3. iptables 规则正常：klipper-lb 已配置 `DNAT --to 10.43.x.x:80`，端口 80 通过 iptables 转发

**根因**: Clash Verge 的**虚拟网卡（TUN）模式**接管所有 DNS 查询，Fake-IP 模式将 `nip.io` 解析为内置代理 IP `198.18.0.x`，浏览器连接到此假 IP 无响应。

**修复方案对比**:
| 方案 | 操作 | 持久性 | 推荐 |
|------|------|--------|------|
| Clash 前置规则 | 订阅配置添加 `prepend-rules: DOMAIN-SUFFIX,nip.io,DIRECT` | 持久 | ✅ 推荐 |
| Windows hosts | `172.26.101.255  172.26.101.255.nip.io` + `ipconfig /flushdns` | 持久 | ✅ 备选 |
| 关闭 TUN 模式 | Clash 切换至"系统代理"模式 | 临时 | 仅验证用 |
