# 文档管理规范

> 本规范定义 `docs/` 目录下所有文档的分类原则、目录结构、命名规则、生命周期管理和核心管理规则。
>
> **所有 docs/ 下的文档作者和审查者必须在提交 PR 前阅读此文档。**

---

## 一、文档分类体系（Divio 象限）

文档按**用途**分为四种类型，不能混用：

```
              学习导向                    工作导向
          ┌──────────────────┬──────────────────┐
 实践性   │   教程（Tutorial）│  操作指南（Guide）│
          │  带我从零做一遍  │  我要解决一个问题 │
          ├──────────────────┼──────────────────┤
 理论性   │ 解释（Explanation）│ 参考资料（Ref）  │
          │  告诉我为什么   │  精确查规格/接口  │
          └──────────────────┴──────────────────┘
```

| 类型 | 放哪里 | 典型内容 |
|------|-------|---------|
| 操作指南（Guide） | `guides/` | 本地开发、发布流程、测试步骤 |
| 参考资料（Reference） | `reference/` | K8s 规格、API 接口表、评分机制 |
| 架构说明（Explanation） | `architecture/` | 系统设计、数据模型、AI层架构 |
| 架构决策记录（ADR） | `adr/` | 每个重要架构/技术选型决策 |
| 需求说明 | `requirements/` | 产品需求规格 |

## 三、文档命名规则

### 3.1 语言

- **文件名**：中文优先，除非是专有技术名称（如 `SSH终端交互.md` 而非 `ssh-terminal.md`）
- **目录名**：全小写英文（`architecture/`、`guides/`），便于 CLI 操作
- **ADR 文件名**：`NNN-简短中文描述.md`（三位数字前缀，如 `001-选择K3s作为生产编排.md`）

### 3.2 禁止的命名方式

- ❌ 数字前缀流水号（`21_发布流程.md`）：无法表达文档类型，无法快速定位
- ❌ 英文文件名（`release-guide.md`）：团队使用中文，中文更直接
- ❌ 带版本号的文件名（`架构设计v2.md`）：版本信息放 frontmatter 或 git history

> **豆免说明**：`architecture/` 目录下的文件使用 `NN_标题.md` 格式（如 `01_系统架构.md`），这是历史遗留且已大量被引用，保持现状。新增架构文档继续遵守该规则。

## 四、文档 Frontmatter 规范

每个文档**必须**在文件头部包含以下 frontmatter：

```markdown
---
status: active
category: guide
audience: developer
last_updated: 2026-03-19
owner: team
---
```

| 字段 | 必填 | 取值说明 |
|------|------|---------|
| `status` | ✅ | `draft`（草稿）/ `active`（当前有效）/ `superseded`（被取代）/ `archived`（已归档） |
| `category` | ✅ | `architecture` / `adr` / `guide` / `reference` / `requirements` |
| `audience` | ✅ | `developer` / `operator` / `architect` / `all` |
| `last_updated` | ✅ | `YYYY-MM-DD` 格式 |
| `owner` | ✅ | 负责人（个人或 `team`） |
| `supersedes` | 条件 | 若本文档取代了旧文档，列出被取代的文件名 |
| `superseded_by` | 条件 | 若本文档已被取代，列出新文档路径 |

## 五、文档生命周期

```
draft（草稿）→ active（当前有效）→ superseded（被取代）→ archived（归档）
```

| 状态 | 含义 | 处理方式 |
|------|------|---------|
| `draft` | 编写中，未正式生效 | 不对外引用 |
| `active` | 当前有效标准 | 正常使用 |
| `superseded` | 被新文档取代 | 更新 `superseded_by` 字段，移至 archive/ |
| `archived` | 仅历史追溯 | 放 archive/，frontmatter 标注归档原因 |

## 六、核心管理规则

### 规则一：文档与代码同步（最重要）

> **凡涉及 `backend/`、`frontend/`、`deploy/`、`database/` 的 PR，必须同步更新对应文档。**

CI 中的 `docs-governance` job 会检查此规则。文档不更新 → PR 无法合入。

**对应关系参考**：

| 代码改动 | 需更新文档 |
|---------|----------|
| API 接口变更 | `architecture/接口设计.md` |
| 数据库 schema 变更 | `architecture/数据库设计.md` |
| 部署配置变更 | `reference/K8s部署规格.md` |
| 发布流程变更 | `guides/发布手册.md` |
| 重要架构决策 | `adr/` 新增 ADR |

### 规则二：一个主题只有一个 active 文档

> **同一主题禁止同时存在两个 `status: active` 的文档。**

- 旧文档被新文档取代时：旧文档 `status` 改为 `superseded`，并填写 `superseded_by`，移至 `archive/`
- 禁止直接删除文档（保留历史可追溯）

### 规则三：内部工作记录不进 docs/

> **进展日志、任务台账、周报模板、看板模板属于团队内部工作记录，不是项目文档。**

这类内容放 `.vk/` 或项目 Wiki，不放 `docs/`。

| 不属于 docs/ 的内容 | 应放哪里 |
|--------------------|---------|
| 项目进展记录 | `.vk/` 或 GitHub Issues |
| 任务台账模板 | `.vk/prompts/` |
| 周期性看板模板 | `.vk/` |
| 问题修复流水账 | GitHub Issues / PR 描述 |
| 迁移过程记录 | `archive/`（归档后） |

### 规则四：ADR 不可随意删除

> **已被采纳的架构决策（`status: accepted`）不可删除，只能标记为 `deprecated` 并说明原因。**

ADR 的价值在于：6 个月后新加入的团队成员能看懂"为什么当初这样决定"。

### 规则五：归档只进不出

> **`archive/` 目录中的文件不允许重新激活使用，需要重写为新文档并放到对应目录。**

### 规则六：避坑指南的边界与写入流程

> **`docs/pitfalls/` 是项目避坑指南的唯一权威来源，所有 AI Agent 工具均从此目录自动加载。**

**内容边界：**

| 内容类型 | 存放位置 |
|---------|--------|
| 可复现的技术坑点（有症状/根因/修复三要素） | `docs/pitfalls/` |
| 操作流程（部署/发布/测试步骤） | `docs/guides/` |
| 技术选型依据 | `docs/adr/` |
| 运维 Runbook（健康检查/一键恢复脚本） | `docs/guides/` 或 `scripts/` |

**写坑流程（必须遵守）：**

1. 在 `docs/pitfalls/_index.md` 的注册表中分配新编号（PIT-XXX）
2. 在对应分类文件中写入条目（症状 / 根因 / 修复 / 预防）
3. 以上两步必须在同一 commit/PR 中提交，CI 门禁会检查

**编号规则：**
- PIT 编号全局唯一，顺序递增，由 `_index.md` 统一分配
- 已废弃的编号保留占位注释，禁止复用
- 条目删除后编号不得被新内容重用

## 七、文档 PR 审查清单

提交包含文档变更的 PR 时，reviewer 需检查：

- [ ] frontmatter 字段完整（status/category/audience/last_updated/owner）
- [ ] 文件名为中文（除专有技术名称和目录名外）
- [ ] 放在正确的目录（按 Divio 分类）
- [ ] 若取代旧文档：旧文档已更新 `superseded_by`，已移至 `archive/`
- [ ] 若是架构决策：使用 `adr/` 下的 ADR 格式
- [ ] 内容与代码实现一致，无过时信息
