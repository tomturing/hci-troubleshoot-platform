> **写文档前、提 PR 前，必须先读本文件。**

## 一、文档用户

本文档系统服务于四类用户，所有文档的设计决策都从用户需求出发。

| 用户 | 典型触发场景 | 核心问题 |
|------|------------|---------|
| **AI Agent**（Copilot / Claude） | 每次对话开始，无历史上下文 | 系统现在是什么状态？我有哪些操作约束？ |
| **新成员工程师** | 第一天 / 接手任务 | 怎么把环境跑起来？为什么系统是这样设计的？ |
| **现有工程师** | 改功能 / 发版 / 出故障 | 这块的规范是什么？这个坑踩过吗？ |
| **未来的自己** | 离开数月后回来 | 当初为什么这样决策？现在系统实际是什么状态？ |

**最重要、最通用的需求：冷启动**

以上四类用户共享一个最高优先场景：**在没有任何上下文的情况下，快速重建完整认知，独立开始贡献**。AI Agent 的每次对话、新成员的第一天、自己休息数月后回来，都面对同样的冷启动问题。

`docs/README.md` 是冷启动的唯一入口。

---

## 二、设计原则：事件循环驱动

文档系统从实际工作循环中生长，不套外部框架。

```
需求 → 方案 → 任务 → 代码 → 部署 → 验证 → 新需求（循环往复）
```

每个循环产生**两类**文档，职责不同、生命周期不同：

```
┌──────────────────────────────────────────────────────┐
│  事件文档（Event Docs）                               │
│  需求 / 方案 / 任务 / 部署 / 验证                      │
│  · 带日期前缀，记录"当时发生了什么、为什么这样决策"      │
│  · 方案文档必须包含"为什么不选其他方案"                 │
│  · 事件完成后冻结，永久保留，不再修改                   │
│  · 核心用户：未来的自己（追溯历史决策）                 │
└──────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────┐
│  现行全量文档（Current Docs）                         │
│  需求说明 / 各设计主干 / 部署规格 / 测试指南 / ...     │
│  · 始终反映当前真实状态，以代码为准，非设计目标         │
│  · 每次相关事件完成后蒸馏更新，追加变更记录             │
│  · 每个主题只有一份，永远是当前权威                    │
│  · 核心用户：现有工程师 + AI Agent                    │
└──────────────────────────────────────────────────────┘
```

**文档腐化的根源**：事件完成了，但现行全量文档没有同步更新。PR 门禁的唯一目的是强制关闭这个缺口。

---

## 三、文档保鲜期分类

| 类型 | 说明 | 典型文档 | 腐化代价 |
|------|------|---------|---------|
| **实时型** | 代码即文档，工具自动生成，禁止手工分离 | 目录结构 / Alembic Migration / OpenAPI Spec / Helm Values / 单元测试 | 立即报错，无法忽视 |
| **事件驱动型** | 工作循环中持续产出，是腐化高危区 | 事件文档 + 现行全量文档 | 静默腐化，最危险 |
| **累积型** | 只增不改，漏记才危险 | `verify/pitfalls/`（避坑指南） | 漏记时才危险 |

## 五、README.md：冷启动入口规范

README.md 是所有用户的冷启动入口，结构固定：

**第一屏（30 秒读完，每个工作循环完成后必须更新）**

1. 系统现在是什么——当前部署拓扑，已实现功能清单（以代码为准）
2. 当前处于哪个阶段——进行中 / 已完成里程碑（一行概括）
3. 冷启动阅读路径——按顺序读这几个文件，即可独立开始贡献

**第二屏（按需查阅，渐进披露）**

- 文档导航：指向 `solution/` 各主干现行全量文档（主干引用分支，不展开细节）
- 操作入口：指向 `deploy/` 和 `verify/`

## 六、solution/ 与 task/ 一一对应原则

`solution/` 描述**做什么以及为什么（WHAT + WHY）**，`task/` 描述**怎么执行（HOW）**。两个目录结构完全镜像。

### 主干 vs 分支的定义

**主干**：系统级汇总文档，维护跨模块的全局视图，其他主题都与它有关联。
**分支**：模块/组件级专项文档，聚焦单一职责域，主干只引用它，不内联其细节。

| 判断标准 | 主干 | 分支 |
|---------|------|------|
| 覆盖范围 | 系统全局（多个模块都涉及） | 单一模块/组件 |
| 典型文档 | 系统架构、数据库设计、接口设计、可观测性设计 | AI助手、知识库、客户端 |
| 新增规则 | 由架构师决策，轻易不新增 | 新增模块时随之新增对应分支目录 |

### solution/ 与 task/ 镜像关系

| solution/ | task/ | 说明 |
|----------|-------|------|
| 主干 `系统架构.md` | 主干 `系统架构.md` | task 侧描述该主题的执行任务清单与完成状态 |
| 分支 `AI助手/方案.md` | 分支 `agent/AI助手任务.md` | 分支粒度一致，一方新增则另一方同步新增 |
| `events/YYYY-MM-DD-X.md` | `events/YYYY-MM-DD-X.md` | 同一工作循环，方案与任务事件同日期同主题 |

**强制规则**：
- 新建 `solution/{子模块}/` 时，必须同步新建 `task/{子模块}/`
- 提交方案事件文档时，必须同步提交对应任务事件文档
- solution 现行全量更新时，同步评估 task 侧是否需要更新

## 七、现行全量文档规范

### 7.1 变更历史节（必须）

每份现行全量文档必须包含变更历史节，作为事件文档与现行文档的连接点：

```markdown
## 变更历史

| 日期 | 版本 | 变更内容 | 关联事件文档 |
|------|------|---------|------------|
| 2026-04-05 | v2.1 | 新增 X 模块 | [2026-04-05-X方案](events/2026-04-05-X方案.md) |
| 2026-03-19 | v1.0 | 初版 | — |
```

### 7.2 主干与分支原则

- **主干文档**：只用链接引用分支，不内联细节，保持全局视图精炼
- **分支文档**：维护子模块完整内容，可引用主干，禁止逆向内联到主干

### 7.3 Frontmatter

```markdown
---
status: active
category: solution      # requirement / solution / task / deploy / verify / meta
audience: developer     # agent / developer / operator / all
last_updated: 2026-04-05
owner: team
---
```

现行全量文档没有"被取代"状态——内容直接更新，历史通过变更历史节追溯。

## 八、事件文档规范

### 8.1 命名与存放

| 类型 | 存放目录 | 命名格式 |
|------|---------|---------|
| 需求事件 | `requirement/events/` | `YYYY-MM-DD-{中文主题}.md` |
| 方案事件 | `solution/events/` | `YYYY-MM-DD-{中文主题}.md` |
| 任务事件 | `task/events/` | `YYYY-MM-DD-{中文主题}.md` |
| 部署事件 | `deploy/events/` | `YYYY-MM-DD-{中文主题}.md` |
| 验证事件 | `verify/events/` | `YYYY-MM-DD-{中文主题}.md` |

### 8.2 方案事件文档必须包含的章节

```
## 背景与需求
## 方案（WHAT）
## 决策依据（WHY：为什么选此方案，为什么不选其他方案）
## 影响范围（哪些现行全量文档需要更新）
## 验收标准
```

"决策依据"章节是方案文档的核心价值，承担原 `adr/` 的职责，不再需要独立维护。

### 8.3 事件完成后

PR 合并时需满足：
1. 相关现行全量文档的变更历史节已追加（PR 门禁检查）
2. 事件文档本身**不修改、不删除**，永久保留

## 九、PR 门禁规则

### 工作循环关闭条件

> **循环关闭 = PR 合并 + 所有受影响的现行全量文档已更新 + README.md 第一屏已更新**

| 本次改动类型 | 必须更新的现行全量文档 |
|------------|-------------------|
| REST API 接口新增/变更/删除 | `solution/接口设计.md` |
| 数据库表结构变更 | `solution/数据库设计.md` |
| 新增/下线微服务 | `solution/系统架构.md` + `README.md` 第一屏 |
| 部署配置变更（Helm / 构建脚本） | `deploy/部署指南.md` |
| 发布流程变化（CI / ArgoCD 策略） | `deploy/发布指南.md` |
| 本地开发环境依赖变更 | `deploy/部署指南.md` 第一部分 |
| AI 助手层架构变更 | `solution/AI助手/AI助手设计.md` |
| 可观测性配置变更 | `solution/可观测性设计.md` |
| 踩到新的可复现技术坑 | `verify/pitfalls/` 对应文件 + `_index.md` |
| 任何工作循环完成 | `README.md` 第一屏 |

### CI Agent 检查（演进目标）

静态 CI 只能检查"有没有改文件"，无法判断"改了正确的文件且内容充分"。目标机制：CI Agent 读取 PR diff，语义判断本次改动类型，验证现行全量文档更新是否覆盖变更，不充分则阻断合并。

## 十、pitfalls/ 写入规则

避坑指南分两处，按坑点发生阶段归属，所有 AI Agent 自动路由加载：

| 坑点类型 | 归属目录 | 典型内容 |
|---------|---------|--------|
| 部署、发布、基础设施类 | `deploy/pitfalls/` | K8s/Helm、Shell 脚本、网络、Grafana |
| 代码验证、功能测试、服务调试类 | `verify/pitfalls/` | 调试原则、OpenClaw、前端、Python、状态机 |

**写入流程（两步必须在同一 PR 提交）：**

1. 在对应目录的 `_index.md` 注册表中分配新编号
2. 在对应分类文件中写入条目（**症状 / 根因 / 修复 / 预防**四要素）

**编号规则：** 全局唯一，顺序递增，废弃后保留占位注释，禁止复用。两个 pitfalls 目录编号空间独立互不干扰。

**内容边界：**

| 内容类型 | 应放位置 |
|---------|---------|
| 部署/基础设施可复现坑 | `deploy/pitfalls/` |
| 代码/服务验证可复现坑 | `verify/pitfalls/` |
| 操作步骤流程 | `deploy/` 现行全量文档 |
| 方案 / 决策依据 | `solution/events/` |

## 十一、文件命名规则

| 目录 | 命名格式 | 示例 |
|------|---------|------|
| 现行全量文档（主干/分支） | 中文，无编号，无版本号 | `系统架构.md` |
| 各阶段 `events/` | `YYYY-MM-DD-{中文主题}.md` | `2026-04-05-AI助手重构方案.md` |

**禁止：** 带版本号的文件名（`架构设计v2.md`）；现行全量文档名加数字前缀。
