# 21_GitOps标准化开发流程

> 适用场景：项目采用 GitOps 发布模型，但研发与发布仍以“人主导 + 自动化工具辅助”为主。
> 
> 项目背景：
> - 开发环境：WSL Ubuntu 24.04（多台）
> - 云上环境：Ubuntu 24.04（2 台，16C32G4T）
> - 代码仓库：GitHub Private（hci-troubleshoot-platform）
> - 编排方案：K3s + Helm Chart（hci-platform）

---

## 1. 目标与原则

### 1.1 目标

1. 实现开发、测试、发布、回滚全流程标准化。
2. 让“代码状态”和“集群状态”可追踪、可审计、可回放。
3. 降低人工发布误操作风险，提升发布稳定性与恢复效率。

### 1.2 核心原则

1. 声明式部署：环境状态以 Git 为唯一事实来源。
2. 不可变发布：镜像标签不可覆盖，只能新增。
3. 分层配置：base + env overlay（dev/staging/prod）。
4. 先验证再晋级：dev -> staging -> prod。
5. 回滚优先：任何发布步骤都必须提供可执行回滚路径。

---

## 2. 架构与仓库模型

### 2.1 双仓模型

1. 应用仓库（当前仓库）
- 存放业务代码、测试、Dockerfile、Helm Chart 模板。
- CI 负责质量检查、构建镜像、输出发布元数据。

2. 环境仓库（建议新建：hci-platform-env）
- 存放环境声明文件（各环境 values、发布版本记录、变更审计）。
- CD（Argo CD）监听环境仓库并同步集群。

### 2.2 目录建议（环境仓库）

```text
environments/
  dev/
    values.yaml
  staging/
    values.yaml
  prod/
    values.yaml
argo-apps/
  app-dev.yaml
  app-staging.yaml
  app-prod.yaml
```

---

## 3. 全流程阶段与实施步骤

## Phase 0：基线准备（Day 1）

### 任务

1. 确认分支策略：main/develop/feature/release/hotfix。
2. 配置分支保护：main/release 禁止直推，必须 PR。
3. 统一版本规范：
- 语义版本：vX.Y.Z（里程碑）
- 镜像标签：YYYY.MM.DD-HHMM-<git短sha>（日常发布）
4. 固化 PR 模板字段：变更范围、风险、测试证据、回滚方案。
5. 确认现有脚本作为兜底发布链路（如 scripts/k3s-release.sh）。

### 交付物

1. 分支保护规则配置截图或说明。
2. PR 模板与发布模板。
3. 版本命名规范文档。

### 验收标准

1. main/release 分支无法绕过 CI 直接合并。
2. 任意 PR 都必须填写风险与回滚信息。

---

## Phase 1：CI 标准化（Day 2）

### 任务

1. 在现有 CI 基础上增强：
- 后端 lint/test
- 前端 lint/test/build
- Helm lint/template
- Docker 镜像构建与推送
2. 输出发布元数据（release-metadata.json）：
- commit_sha
- image_tags
- build_time
- changed_services
3. 设置失败阻断策略：
- 任一核心门禁失败，禁止进入发布候选。

### 交付物

1. CI 工作流文件。
2. 镜像清单与元数据产物。

### 验收标准

1. PR 可看到完整门禁结果。
2. main 合并后可稳定产出镜像与元数据。

---

## Phase 2：环境仓库与 GitOps 接入（Day 3-4）

### 任务

1. 初始化环境仓库与目录结构。
2. 抽离环境差异配置：
- dev：快速验证、资源低配
- staging：接近生产、用于冒烟与回归
- prod：稳定优先、变更受控
3. 部署 Argo CD 并创建 3 个 Application：
- dev/staging 自动同步
- prod 手动同步（需审批）
4. 应用仓库 CI 自动发起环境仓库 PR（更新 image.tag）。

### 交付物

1. 环境仓库目录与初始配置。
2. Argo CD Application 清单。
3. 自动更新环境声明的流水线。

### 验收标准

1. 代码合并后，dev 可自动部署成功。
2. staging 可通过人工确认后晋级。
3. prod 必须审批后才可执行同步。

---

## Phase 3：发布治理与回滚（Day 5-6）

### 任务

1. 建立发布策略：
- 默认 Rolling Update
- 高风险变更先 staging 观察
2. 建立发布后巡检：
- 接口健康检查
- 错误率、延迟、资源监控
- 核心链路冒烟
3. 建立统一回滚 SOP：
- 回退环境仓库 commit
- 触发 Argo 同步回滚
- 执行验证脚本
4. 保留紧急兜底：极端情况可临时切回脚本发布。

### 交付物

1. 发布清单（Preflight/Postflight）。
2. 回滚清单与执行脚本说明。
3. 发布记录模板。

### 验收标准

1. 30 分钟观察窗口内可自动完成巡检。
2. 任意失败版本可在 10 分钟内回滚到上一稳定版本。

---

## Phase 4：稳定化与审计（Day 7）

### 任务

1. 建立发布审计链路：谁、何时、发了什么、结果如何。
2. 固化运行手册：发布、回滚、故障处理、值班。
3. 输出周度运营指标：
- 部署频率
- 变更失败率
- 平均恢复时间

### 交付物

1. 运行手册 v1。
2. 发布审计台账模板。
3. 周报模板。

### 验收标准

1. 任意一次发布可追溯完整链路。
2. 发生异常时有明确责任人、升级路径和恢复记录。

---

## 4. 规范清单

### 4.1 开发规范

1. 每个变更必须可测试、可回滚。
2. 禁止在生产环境直接手工改配置。
3. 变更必须关联任务编号与验收标准。

### 4.2 发布规范

1. 生产发布必须走审批流程。
2. 发布后必须执行固定巡检步骤。
3. 无回滚方案的变更不得进入生产。

### 4.3 安全规范

1. 密钥只允许通过 Secret 管理。
2. 禁止在仓库明文提交凭据。
3. 重要依赖和镜像建议引入漏洞扫描门禁。

---

## 5. 角色与职责（RACI）

1. 开发负责人：代码质量与测试完整性（R）。
2. Reviewer：变更风险评估与审查（R）。
3. Release Manager：发布审批与窗口控制（A）。
4. SRE/平台：发布稳定性与回滚链路（R）。
5. 产品/业务：需求优先级与验收确认（C/I）。

---

## 6. 时间压缩建议

### 6.1 最短可行周期

1. 5-7 个工作日：仅落地最小闭环（dev/staging 自动，prod 审批）。
2. 10-15 个工作日：补齐稳定性、审计、演练与治理。

### 6.2 对持续迭代的影响

1. 第 1-2 天吞吐下降约 10%-20%。
2. 第 3-4 天下降约 5%-10%。
3. 第 5 天后基本恢复。

### 6.3 降影响策略

1. 双轨并行：旧发布链路保留，新链路先接管 dev。
2. 小窗口改造：每天固定 60-90 分钟平台改造。
3. 最小冻结：只冻结发布配置，不冻结业务开发。

---

## 7. 风险与应对

1. 风险：环境声明与集群漂移。
- 应对：以 Argo 状态为准，禁止生产手工改动。

2. 风险：标签错发。
- 应对：只允许 CI 自动写入 image.tag。

3. 风险：发布后问题发现滞后。
- 应对：固定 30 分钟观察窗口 + 自动巡检。

4. 风险：回滚流程不熟练。
- 应对：每周至少 1 次演练。

---

## 8. 完成定义（Definition of Done）

1. dev/staging 发布可无人值守自动完成。
2. prod 发布可审批、可审计、可回滚。
3. 发布失败可在 10 分钟内恢复。
4. 团队成员可按文档独立执行发布流程。
