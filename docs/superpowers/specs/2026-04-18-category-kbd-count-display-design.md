# 分类基线页面 KBD/SOP 已发布数量展示功能设计

> 创建日期：2026-04-18
> 状态：设计评审（第二轮）

---

## 1. 需求概述

在分类基线管理页面（CategoryManageView）增加已发布 KBD/SOP 数量的展示功能：

1. **分类项右侧标签**：显示 `[SOP:N][KBD:N]`，所有层级节点都显示
2. **大类汇总显示**：L1 域节点显示汇总的 SOP/KBD 数量（统计其下所有子分类）
3. **左侧统计栏**：新增「已发布KBD」总数统计
4. **右侧详情面板**：
   - 标题行：`分类详情` | 启用/禁用开关 | `[保存修改]` 按钮
   - 基本信息表格改成 4 列 × 2 行布局
   - 增加已发布 SOP 和 KBD 列表（SOP在上，KBD在下）
   - 每行条目开头显示命中次数标签 `[命中:N]`，右侧有「详情」按钮
5. **详情弹窗**：复用现有 KbdReviewView 的详情弹窗逻辑
6. **业务逻辑修复**：意图识别时过滤禁用分类（`is_active=True`）

---

## 2. 数据关联逻辑

| 关联字段 | 说明 |
|----------|------|
| KBD.category_id | 人工确认的分类编码（如 `虚拟机-003`） |
| SOP.category_id | SOP 文档关联的分类编码 |

**统计条件**：
- KBD：`status='published' AND category_id=分类code`
- SOP：`status='published' AND category_id=分类code`

## 3. 后端设计

### 3.1 API 改动

**接口**：`GET /api/kb/categories?grouped=true`

**返回数据结构变更**：

```typescript
interface KbCategory {
  id: number
  code: string                  // 业务键
  name: string                  // 分类名称
  domain: string                // 一级技术域
  level: number                 // 层级 1-4
  parent_id: number | null
  path_labels: string[]         // 完整路径
  hit_count: number             // S0 命中次数
  is_active: boolean            // 启用状态
  // 新增字段
  published_kbd_count: number   // 已发布 KBD 数量
  published_sop_count: number   // 已发布 SOP 数量
}
```

**注意**：删除 `has_sop` 字段，由 `published_sop_count > 0` 判断。

### 3.2 数据库查询改动

**文件**：`backend/kb-service/app/repositories/category_repo.py`

**改动方法**：`get_all()` 和 `get_all_active()`

**SQL 实现思路**：

```sql
SELECT
  c.id, c.code, c.name, c.domain, c.level, c.parent_id, c.path_labels,
  c.hit_count, c.is_active, c.keywords, c.source, c.version,
  -- 新增：已发布 KBD 数量（仅统计人工确认分类）
  (SELECT COUNT(*) FROM kbd_entry
   WHERE status = 'published' AND category_id = c.code) AS published_kbd_count,
  -- 新增：已发布 SOP 数量
  (SELECT COUNT(*) FROM sop_document
   WHERE status = 'published' AND category_id = c.code) AS published_sop_count
FROM kb_category c
ORDER BY c.level, c.domain, c.code
```

### 3.3 大类汇总统计逻辑

**L1 域节点汇总**：前端根据 `domain` 字段聚合计算，统计同域下所有子分类的 SOP/KBD 数量之和。

```typescript
// 前端计算域汇总（只统计 is_active=true 的子分类）
const domainStats = computed(() => {
  const stats: Record<string, { sop: number; kbd: number }> = {}
  for (const cat of allCategories) {
    if (!cat.is_active) continue  // 禁用分类不参与汇总
    if (!stats[cat.domain]) {
      stats[cat.domain] = { sop: 0, kbd: 0 }
    }
    stats[cat.domain].sop += cat.published_sop_count || 0
    stats[cat.domain].kbd += cat.published_kbd_count || 0
  }
  return stats
})
```

### 3.4 业务逻辑修复：意图识别过滤禁用分类

**文件**：`backend/kb-service/app/routes/classify.py`

**改动函数**：`fetch_categories_for_classify`

**改动内容**：增加 `is_active = TRUE` 过滤条件

```sql
-- 改动前
SELECT code, name, domain, path_labels
FROM kb_category
WHERE code IS NOT NULL
ORDER BY domain, code

-- 改动后
SELECT code, name, domain, path_labels
FROM kb_category
WHERE code IS NOT NULL AND is_active = TRUE
ORDER BY domain, code
```

**影响**：禁用的分类将不会参与 S0 意图识别，真正实现"下线"效果。

### 3.5 新增详情查询接口

**KBD 详情接口**（已有）：`GET /api/admin/kbd/pending?status=published&category_id={code}`

**需要新增**：支持单个 KBD 详情查询的接口

```
GET /api/admin/kbd/{kbd_id}
```

返回完整 KBD 详情（含 content_md）。

**SOP 详情接口**（已有）：`GET /api/admin/sop/{document_id}`

## 8. 注意事项

- **性能**：分类数量约 230 条，子查询统计需确保查询效率（建议添加索引）
- **缓存**：后端可考虑缓存统计数据，避免每次请求都执行子查询
- **前端复用**：详情弹窗的 Markdown 渲染逻辑应提取为共享工具，避免代码重复
- **命中次数数据**：KBD 的命中次数来自 `kbd_entry.hit_count`，SOP 的命中次数需要新增字段或使用其他统计方式
- **禁用分类**：禁用后不再参与意图识别，已关联的 KBD/SOP 不受影响
