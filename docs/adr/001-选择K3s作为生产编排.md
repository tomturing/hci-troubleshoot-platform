# ADR-001：选择 K3s 而非标准 K8s 作为生产编排工具

## 状态

**已采纳**（2025-10）

---

## 背景

项目生产环境为 2 台 Ubuntu 24.04 云主机（各 16C / 32G / 4T），需要选择容器编排方案。同时，开发环境为多台 WSL Ubuntu 24.04，需要本地也能验证部署流程。

---

## 决策问题

> 在资源受限的 2 节点环境中，选择哪个 Kubernetes 发行版？

---

## 评估选项

| 选项 | 说明 | 优势 | 劣势 |
|------|------|------|------|
| **标准 K8s（kubeadm）** | 官方发行版 | 100% 兼容 API，生态最广 | 控制面资源消耗高（~1-2GB），安装复杂，不支持 ARM 轻量化 |
| **K3s**（✅ 已采纳）| Rancher 轻量版 | 单二进制，安装 2 分钟，内存占用 ~500MB，完全兼容 K8s API | 与上游 K8s 有版本差，部分高级功能需额外配置 |
| **Docker Swarm** | Docker 原生集群 | 运维最简单 | 生态萎缩，Helm 不原生支持，不可迁移到标准 K8s |
| **裸 Docker Compose** | 单机进程编排 | 开发环境完美 | 无自愈、无多节点，不适合生产 |

---

## 决策

**采用 K3s**，版本与上游 K8s 保持一个小版本滞后（稳定优先）。

**核心理由**：
1. 2 节点规模下，标准 K8s 控制面占用资源超过业务本身，不合理
2. K3s 完全兼容 Helm Chart，现有 `hci-platform` Chart 无需修改
3. WSL 本地环境可用 k3d 起本地集群，与生产流程完全一致
4. Rancher Labs 活跃维护，边缘/小规模场景业界标准

---

## 后果

**正面**：
- 安装部署极简，运维团队学习成本低
- 节省约 1.5GB 内存用于业务 Pod
- Helm 生态完全可用

**负面/注意事项**：
- K3s 默认使用 containerd，不是 Docker daemon，镜像导入用 `k3s ctr images import`，不用 `docker load`
- 默认使用 Traefik 作为 Ingress Controller，需注意与 Nginx Ingress 配置差异
- 节点扩展到 5+ 个时应重新评估是否升级到标准 K8s
