# ADR-004：发布工作流方案选型（方案21 vs 方案22）

## 状态

**已采纳**（2026-03-19），当前执行方案21，方案22为未来演进目标

---

## 背景

项目在设计发布工作流时，评估了两种路径的发展方向：
- **方案21**："人主导 + 自动化工具辅助" 的 GitOps 标准化流程
- **方案22**：AI Agent 自动驾驶开发流程（Agentic）

---

## 决策问题

> 当前阶段优先落地哪种发布工作流？两种方案如何衔接？

---

## 方案对比

### 方案21：GitOps 标准化（人主导）

**核心特征**：
- 人工做所有决策：PR review、发布审批、环境晋级确认
- 工具自动化执行：CI 构建、ArgoCD 同步、监控告警
- 标准成熟：与业界 GitOps 实践完全对齐

**适用阶段**：团队规范尚未稳固，产品功能仍在迭代，需要人工控制变更风险

**主要交付物**：
- 分支保护 + PR 模板
- GitHub Actions CI/CD
- ArgoCD + 环境仓库
- 发布检查清单 + 回滚 SOP
- Preflight/Postflight 验证

### 方案22：Agent 自动驾驶（AI 主导）

**核心特征**：
- AI Agent 负责：代码审查、测试、发布决策、回滚判断
- 人工只做：策略制定、异常升级处理
- 依赖前提：方案21的所有标准化流程已经稳定运行

**适用阶段**：方案21完全成熟后的下一阶段，需要高度成熟的自动化基础设施

**主要交付物**：
- Agent 编排框架（VK / 多 Agent 工作流）
- 自动化质量决策引擎
- 无人值守发布流水线

---

## 决策

**当前阶段执行方案21，方案22作为方案21稳定后的演进目标。**

**核心理由**：
1. 方案22依赖方案21作为基础设施，顺序不可颠倒
2. 当前团队规模（1-2人），人工审批成本可接受，不需要 AI 自动决策
3. 方案21落地后的稳定运行数据（部署频率、失败率、恢复时间）是方案22的输入
4. 过早引入方案22的复杂性会增加调试难度，风险收益不合算

---

## 迁移触发条件

当以下条件**全部满足**时，可开始方案22探索：

- [ ] 方案21各环节（CI/ArgoCD/监控告警）连续稳定运行 ≥ 4 周
- [ ] 部署成功率 ≥ 95%
- [ ] 平均回滚耗时 ≤ 10 分钟
- [ ] 发布检查清单已能无脑执行（无需思考）
