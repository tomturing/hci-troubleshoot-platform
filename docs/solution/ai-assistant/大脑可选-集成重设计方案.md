## 一、理解用户模型：第一性原理确认

用户提出了一个精准的系统切分方式：

```
骨架和手脚（不可替换，公共基础设施）
  ├── acli 工具执行（SSH → HCI节点命令）
  ├── SCP 工具调用（REST API → 告警/任务/虚机数据）
  ├── 工单生命周期管理（case-service）
  ├── 消息持久化（PostgreSQL）
  ├── 高危操作风险确认（ConfirmService + Redis）
  ├── 审计追踪（audit_log）
  └── 实时告警/环境上下文（EnvironmentClient）

大脑（可热切换，A/B 测试对象）
  ├── 选项 A：htp 大脑（S0→S6 + kb-service + ReactExecutor）
  └── 选项 B：ops-agent 大脑（intake→routing→...→closed + SOP JSONL）
```

**这个切分的第一性推论是对的**：acli 工具、工单管理、消息持久化都是"执行基础设施"，与推理逻辑无关，不需要为不同大脑各实现一套。两个大脑的竞争点只是：**谁的诊断推理更优秀**。

要做到这一点，需要先彻底理解两个大脑的内部机制，才能设计出正确的接口边界。

## 二、两个"大脑"的深度解剖

### 2.1 htp 大脑的核心机制

**状态机的控制权在代码，不在 LLM**

```python
# conversation_service.py — 每次消息处理的主流程
async def send_message_stream_only(...):
    # 1. 从 PostgreSQL 读取当前阶段（代码控制）
    current_stage = conv.diagnostic_stage  # "S0" / "S1" / ... / "S6"
    
    # 2. 代码拦截 S0 用户选择（代码控制，不经过 LLM）
    if current_stage == "S0":
        _selection = parse_candidate_selection(content)  # 检测用户输入 ①②③
        if _selection is not None:
            _update_diagnostic_stage(new_stage="S1")  # 代码直接推进阶段
    
    # 3. 实时数据预取（在 LLM 调用前）
    if current_stage == "S0" and environment_client:
        env_context = await environment_client.get_context_info(case_id)
        # 告警日志、失败任务 → 注入 system prompt
    
    # 4. 基于当前阶段构建 5 段式 system prompt（代码决定内容）
    system_prompt = build_system_prompt(stage=current_stage, sop_content=..., kb_content=...)
    
    # 5. 调用 LLM（LLM 只是生成回复，不控制状态）
    async for chunk in ai_client.chat_completion_stream(messages):
        yield chunk
    
    # 6. 从 LLM 输出中后验检测阶段转换（代码 + 正则）
    new_stage = detect_stage_transition_with_category(full_reply)
    if new_stage:
        _update_diagnostic_stage(new_stage)  # 代码写库
```

**关键特征**：
- **代码是状态机的掌舵人**：阶段存 PostgreSQL，转换逻辑在 Python 代码里
- **LLM 是单轮回复生成器**：每次 HTTP 请求 = 一次 LLM 调用 = 一轮对话
- **阶段信息通过 system prompt 注入 LLM**：告诉 LLM "你现在在 S3 阶段，方法论是..."
- **阶段转换后验检测**：靠正则从 LLM 输出中提取"分类确认"等信号来决定推进

还有一个 **ReactExecutor 模式**（Phase 4）是真正的 ReAct：GLMClient + ToolRouter 在一次请求内运行 15 步 ReAct 循环，使用 acli/SCP 工具。但目前这是独立于 `send_message_stream_only` 的另一条执行路径，没有与 S0-S6 状态机集成。

### 2.2 ops-agent 大脑的核心机制

**状态机的控制权在 LLM，通过 prompt 强制执行**

```python
# ops_agent/agent/ops_agent.py — 单次任务入口
class OpsAgent(BaseAgent):
    def _get_initial_messages(self) -> list[LLMMessage]:
        system_prompt = load_agent_prompt_template("ops_agent_prompt").render()
        # system prompt 包含完整的状态机定义：
        # current_stage: intake/routing/validation/solution/confirmation/reroute/escalation/closed
        # 何时调用 ops_state_update、何时调用 query_sop_candidates...
        return [
            LLMMessage(role="system", content=system_prompt),
            LLMMessage(role="user", content=f"用户原始问题：{task}"),
        ]
```

```
# ops_agent_prompt.jinja2 — 状态机定义（LLM 的行为约束）
current_stage: intake / routing / validation / solution / confirmation / reroute / escalation / closed

触发 ops_state_update 写入的时机（缺一不可）：
- 进入新阶段（intake → routing、routing → validation...）
- 拿到新强信号 / 用户明确否定某信号
- 同一 step 进入第 2 次尝试前
- 出现新的高风险点，或风险解除
- 准备 case_intake 或 task_done 前
```

**关键特征**：
- **LLM 是状态机的掌舵人**：通过 `ops_state_update` 工具调用显式写入状态，但能否调用取决于 LLM
- **一次 `agent.run()` = 一个完整案例**：从 intake 到 closed，不是"一轮对话"
- **对话历史靠 messages 数组传入**：但 `AgentExecution` 对象存于内存，进程重启即丢失
- **500 步 ReAct 循环**：每步可以是工具调用（query_sop、get_info_from_user）或推理输出

---

### 2.3 最关键的语义差异

这是两个大脑最根本的不同，决定了集成接口必须在正确的层次上切开：

| 概念 | htp 大脑 | ops-agent 大脑 |
|------|----------|---------------|
| **"一次会话"的粒度** | 多轮 HTTP 请求 + 持久状态 | 单次 `agent.run()`（内含多步 ReAct） |
| **"一次 LLM 调用"的语义** | 一轮对话的回复生成 | ReAct 循环中的一步推理 |
| **状态管理者** | Python 代码 + PostgreSQL | LLM + ops_state_update 工具 |
| **工具执行者** | htp 的 ToolExecutor（acli/SCP） | ops-agent 内部（get_info_from_user → 等用户） |
| **知识来源** | kb-service（外部 REST API） | 本地 SOP JSONL（随进程） |

## 三、两个大脑的优缺点深度对比

### 3.1 htp 大脑

#### 优势

**① 状态可持久、服务可重启**  
诊断阶段存 PostgreSQL，对话历史存 PostgreSQL，Redis 存确认状态。服务重启不丢失，支持断点续接。这在生产环境中是刚需。

**② 实时数据感知**  
S0 阶段 `EnvironmentClient` 自动预取告警日志、失败任务并注入 system prompt。LLM 在回答用户前就已经看到了实时的平台状态——不需要用户手动粘贴告警信息。

**③ 知识可运营**  
kb-service 支持知识热更新、A/B 测试（`is_active` 字段）、命中统计（`hits` 路由）。SOP 命中率、KBD 使用频率都可以在 Grafana 上观测，形成"知识→使用→优化"的运营闭环。

**④ 代码级状态机更可靠**  
S0→S1 的转换是代码触发的，不依赖 LLM 是否"愿意"推进。S0 阶段用户选择 ①②③ 时，代码直接拦截并推进阶段，LLM 不参与这个决策。

**⑤ 平台工具天然集成（Phase 4）**  
ReactExecutor 可以直接调用 `acli_service_restart`、`get_active_alerts` 等工具，在一次对话内完成从诊断到执行的闭环。

#### 缺陷

**① 阶段转换后验检测脆弱**  
`detect_stage_transition_with_category()` 靠正则从 LLM 输出中提取阶段信号，LLM 输出格式变化就会失效。如果 LLM 回复了"我来定位故障"但没有输出特定格式的分类标记，阶段就卡住了。

**② SOP 内容全量注入**  
通过三轨路由拿到 SOP 或 KBD 内容后，整章节注入 system prompt。长 SOP 会大量占用 token，且低相关内容会引入噪声。没有 ops-agent 的"按当前步骤定位章节"的精准注入。

**③ 单步无 ReAct（主路径）**  
`send_message_stream_only` 是主路径，每次 HTTP 请求只调用一次 LLM。复杂的"先查告警 → 再查磁盘 → 再决策"逻辑必须用户多轮手动触发，而不是 LLM 自主完成。ReactExecutor 虽然有 ReAct，但与 S0-S6 状态机是割裂的。

**④ 步数上限 MAX_STEPS=15 偏紧**  
15 步对于"执行诊断命令 → 观察结果 → 执行下一个诊断命令"这样的验证循环可能不够，会触发"⚠️ 诊断步骤已达上限"。

---

### 3.2 ops-agent 大脑

#### 优势

**① SOP 驱动的强制推理，不可跳过**  
prompt 中的 `<routing_decision>` 和 `<state_management>` 规则明确约束了 LLM 的行为：何时必须调用 `query_sop_candidates`、何时必须先调用 `get_sop_discriminators` 区分 branch、何时才能进入 solution。LLM 无法绕过这些规则直接给出答案。

**② `SOPQuerySubAgent` 精准章节导航**  
不是把整个 SOP 文档塞进上下文，而是 SOPQuerySubAgent 作为独立 ReAct 子智能体，在 SOP 内部做 PageIndex 导航，找到与当前诊断步骤最相关的章节（check-N 或 solution-N）后精准注入。这是 htp 目前没有的能力。

**③ `present_sop_step_instruction` 强制可解释性**  
每个诊断步骤都通过 `present_sop_step_instruction` 展示：`operation_goal`（目的）、`execution_guidance`（来自 SOP 的具体命令，不由模型编造）、`expected_result`（预期结果）、`risk_notice`（风险提示）。输出高度结构化、可追溯。

**④ 500 步深度诊断支持复杂案例**  
复杂故障（如存储 RAID 重建过程中的 VM 迁移失败）可能需要几十步的验证循环，ops-agent 的 500 步上限可以支撑完整的诊断流程。

**⑤ 双 Agent 协作（Tool-as-SubAgent）**  
主 Agent 专注推理和用户交互，SOPQuerySubAgent 专注知识检索。职责分明，主 Agent 不需要知道 SOP 的内部结构。

#### 缺陷

**① 单次任务语义，不是多轮对话语义**  
`agent.run()` 被设计为"完整处理一个案例"，不是"处理对话的某一轮"。这导致集成时出现根本的语义错位：每次 HTTP 请求都触发新的 `agent.run()`，每次都从 intake 重新开始。

**② AgentExecution 无持久化，状态在内存中**  
进程重启、服务容器替换，诊断状态全部丢失。在生产 K8s 环境中 Pod 随时可能重启，这是不可接受的。

**③ 无实时数据感知**  
`get_info_from_user` 等待用户手动输入告警数据。用户成为数据搬运工。htp 已经可以直接查 SCP API 和 acli，但 ops-agent 无法使用这些能力。

**④ 假流式输出，TTFT 极差**  
当前 OpenAI-compatible 实现是先跑完整个 ReAct 链路，再伪造 SSE 流。500 步的 ReAct 链路可能需要数分钟，用户看到任何输出前都要等待全部完成。这在 Web UI 场景是致命的体验缺陷。

**⑤ 知识孤岛**  
SOP JSONL 文件与 htp kb-service 是两套独立的知识库，内容可能不一致，维护成本双倍。

---

### 3.3 对比总结

```
维度                    htp 大脑              ops-agent 大脑
─────────────────────────────────────────────────────────────

```

## 四、当前集成方案的完整诊断

### 4.1 层次错位：接口插在了错误的位置

当前集成在 `AIAssistantRegistry` 层做分支，而 `AIAssistantRegistry` 在 `conversation_service` 的内部，是被 `send_message_stream_only` 调用的：

```
send_message_stream_only()
  │
  ├── 构建 htp system prompt（S0-S6 方法论 + SOP + KB）→ ops-agent 会丢弃
  ├── 读取历史消息（最近 20 条）
  ├── ai_registry.get_client("ops-agent")  ← 分支在这里
  │     └── OpsAgentAssistant.chat_completion_stream(messages)
  │           └── ops-agent /v1/chat/completions
  │                 └── 从 messages 取最后一条 user 消息
  │                 └── agent.run(user_query) ← 新的完整任务，无视历史
  └── detect_stage_transition_with_category()  ← 从 ops-agent 输出检测 S0-S6 阶段（无意义）
```

ops-agent 绕过了 htp 大脑的所有能力（知识检索、阶段管理），但 htp 仍然在浪费资源执行这些逻辑，然后 ops-agent 全部丢弃。

### 4.2 两套状态机并行运行，相互不知

htp 认为对话处于 S3（验证执行），发来携带 S3 方法论的 system prompt。
ops-agent 认为这是新任务，自己处于 intake 阶段，重新开始问问题。
用户每轮对话都在重新开始，诊断上下文不会积累。

### 4.3 "手脚"被闲置：htp 的 acli 能力对 ops-agent 不可见

ops-agent 靠 `get_info_from_user` 问"告警状态是什么？"——但 htp 已经有 `get_active_alerts` 可以直接查。没有机制把 htp 的"手脚"能力透出给 ops-agent 使用。

## 五、重新设计：正确的架构

### 5.1 设计原则（第一性原理推导）

**原则一：大脑的接口边界应在"会话管理"之上，不在"LLM 客户端"之下**

htp 的 `send_message_stream_only` 是"会话管理 + 大脑执行"的混合体，应该拆开：
- **会话管理层**（永远执行）：保存消息、读取历史、审计日志、重复检测
- **大脑执行层**（可切换）：接收问题 + 历史 + 上下文，输出流式回复

**原则二：ops-agent 大脑需要会话连续性，不能每次从头开始**

ops-agent 被设计为"单次任务"，但集成后需要支持"多轮对话"。解决路径：在 ops-agent 侧维护 `session_id → AgentExecution` 的内存 session pool，并将状态序列化到 Redis 以支持跨进程恢复。

**原则三：htp 的实时数据能力应该作为"上下文注入"传给 ops-agent**

在调用 ops-agent 前，htp 先预取告警/环境数据，通过 messages 的首个 system message 注入。这样 ops-agent 无需修改工具，就能感知实时环境数据。

**原则四：ops-agent 的工具请求应该 bridge 到 htp 的执行能力**

ops-agent 调用 `get_info_from_user`（问用户"虚机状态是什么？"）时，htp 应该能自动执行 `acli_vm_list` 拿到结果，而不是等用户手动输入。这是"手脚共用"的核心价值。

```python
class DiagnosticBrain(Protocol):

class EscalationEvent:
    reason: str                   # 升级原因
```

### 5.4 htp 大脑实现（改造最小）

htp 大脑基本复用现有 `conversation_service` 的逻辑，只是包装为 `DiagnosticBrain` 接口：

```python
class HTPBrain:
    """htp 原生大脑：S0-S6 状态机 + kb-service + ReactExecutor"""
    
    def __init__(
        self,
        ai_client: AIAssistantClient,   # OpenClaw / GLMClient
        kb_client: KBClient | None,
        tool_router: ToolRouter | None,  # Phase 4 ReactExecutor
        confirm_service: ConfirmService | None,
    ): ...
    
    async def process(self, session_id, case_id, user_message, history, env_context, tool_bridge):
        # 1. 从 session_id 查询当前诊断阶段（PostgreSQL）
        # 2. 构建 S0-S6 system prompt + 知识注入
        # 3. 注入 env_context（告警/任务）
        # 4. 调用 LLM（或 ReactExecutor）
        # 5. 检测阶段转换，写库
        # yield TextChunkEvent / StateUpdateEvent / EscalationEvent
```

对现有代码的改动：在 `send_message_stream_only` 外包一层 `HTPBrain`，逻辑基本不变。

### 5.5 ops-agent 大脑实现（关键改造）

ops-agent 大脑需要三个关键改造：

#### 改造 A：会话持久化（解决无状态问题）

```python
# ops-agent 侧新增：会话池
class AgentSessionPool:
    """维护 session_id → AgentExecution 的映射，Redis 持久化"""
    
    async def get_or_create(self, session_id: str) -> ContinuableAgent:
        """从 Redis 恢复会话，或创建新会话"""
        state = await redis.get(f"ops_agent:session:{session_id}")
        if state:
            return ContinuableAgent.resume(state)
        return ContinuableAgent.new()
    
    async def save(self, session_id: str, agent: ContinuableAgent):
        """将 AgentExecution 状态序列化到 Redis（TTL 4h）"""
        await redis.setex(
            f"ops_agent:session:{session_id}",
            14400,  # 4小时过期
            agent.serialize()
        )
```

核心实现：`ContinuableAgent` 支持序列化/反序列化 messages 历史和 `current_stage`，新轮次时恢复 messages 继续 ReAct，而不是重新开始。

#### 改造 B：环境上下文接收（解决实时数据盲区）

```python
# ops-agent 的 /v1/chat/completions 扩展
class ChatCompletionRequest(BaseModel):
    messages: list[dict]
    session_id: str | None = None              # 新增：会话 ID
    hci_context: HCIContext | None = None      # 新增：htp 预取的实时数据
    stream: bool = True

class HCIContext(BaseModel):
    active_alerts: list[dict] = []             # get_active_alerts 结果
    failed_tasks: list[dict] = []              # get_failed_tasks 结果
    env_info: dict | None = None              # 集群信息

# 在 agent 初始化时，将 HCIContext 注入为第一条 system message
def _inject_hci_context(messages, hci_context):
    if not hci_context:
        return messages
    context_msg = {
        "role": "system",
        "content": f"""【实时平台数据（htp 预取，{datetime.now().isoformat()}）】
当前活跃告警（{len(hci_context.active_alerts)} 条）：
{json.dumps(hci_context.active_alerts, ensure_ascii=False, indent=2)}

最近失败任务（{len(hci_context.failed_tasks)} 条）：
{json.dumps(hci_context.failed_tasks, ensure_ascii=False, indent=2)}

注意：以上为实时数据，无需向用户询问这些已有的信息。"""
    }
    return [context_msg, *messages]
```

#### 改造 C：工具调用 Bridge（解决手脚闲置问题）

这是最有价值但最复杂的改造。当 ops-agent 调用 `get_info_from_user` 且问题是"执行命令并告诉我结果"时，htp 可以自动代替用户执行：

```python
# ops-agent 侧：结构化的工具调用事件
class ToolBridgeRequest(BaseModel):
    """ops-agent 发出的结构化工具请求（通过 SSE 发出）"""
    request_id: str
    tool_type: str           # "acli_command" | "scp_query" | "ask_user"
    command: str             # 如果是 acli，要执行的命令
    question: str | None     # 如果是 ask_user，问题文本
    
# htp 侧：ToolBridge 接收请求并执行
class OpsAgentToolBridge:
    def __init__(self, tool_router: ToolRouter, confirm_service: ConfirmService):
        self.tool_router = tool_router
        self.confirm_service = confirm_service
    
    async def handle(self, request: ToolBridgeRequest, session_id: str) -> str:
        if request.tool_type == "acli_command":
            # 识别出 acli 命令 → 调用 tool_router.execute
            # 高风险命令先走 confirm_service
            return await self.tool_router.execute(
                f"acli_{request.command.split()[0]}", 
                {"command": request.command}
            )
        elif request.tool_type == "ask_user":
            # 真的需要用户输入 → 走 SSE 推送等待
            return await self.confirm_service.wait_for_user_input(
                session_id, request.question
            )
```

> **注意**：改造 C 是渐进式的，可以分两期实现：
> - **一期**：只做环境上下文注入（改造 B），ops-agent 看到实时数据但仍然通过 `get_info_from_user` 引导用户执行命令
> - **二期**：实现工具 Bridge，高危命令仍然需要用户确认，只读命令自动执行

### 5.6 真实流式输出改造（解决 TTFT 问题）

ops-agent 的假流式问题根源在于 `agent.run()` 是一次性完成的。改造方案：

```python
# ops-agent 侧：真流式 generator
async def stream_run(self, session_id: str, message: str, context: HCIContext | None):
    """逐步 yield 推理过程，而不是等待完成后分块"""
    session = await self.session_pool.get_or_create(session_id)
    
    async for event in session.step_generator(message, context):
        if isinstance(event, LLMTextDelta):
            yield SSEChunk(content=event.delta)    # LLM 生成时实时输出
        elif isinstance(event, ToolCallStarted):
            yield SSEChunk(event="tool_start", data=event.tool_name)
        elif isinstance(event, ToolCallResult):
            # 工具结果不直接 yield 给用户，加入上下文继续推理
            session.add_tool_result(event)
        elif isinstance(event, AgentFinished):
            await self.session_pool.save(session_id, session)
            break
```

这需要 ops-agent 的 `base_agent.py` 的 ReAct 循环支持"yield 中间步骤"而不是只在最后 yield 最终结果。这与 htp 的 `ReactExecutor.run()` 已有的异步生成器模式完全一致。

## 八、给两个大脑的具体改进建议

### 给 htp 大脑的改进建议

1. **ReactExecutor 与 S0-S6 集成**：目前 `send_message_react_stream` 和 `send_message_stream_only` 是两条割裂的路径。应该让 ReactExecutor 在 S3（验证执行）阶段自动激活，S0/S1/S2 仍走单次 LLM，S3 激活 ReAct + acli 工具执行，S4 之后回到单次 LLM。

2. **阶段转换前验机制**：放弃后验正则检测，改为 LLM 通过结构化工具调用 `stage_advance` 来显式推进阶段（类似 ops-agent 的 `ops_state_update`），代码验证后执行转换。

3. **SOP 章节精准注入**：借鉴 ops-agent 的 SOPQuerySubAgent 思路，在 KB 三轨路由拿到 SOP 后，不是整章注入，而是根据当前诊断阶段（S2/S3）只注入"假设生成"或"验证步骤"相关的章节。

4. **MAX_STEPS 动态调整**：根据 S0 识别的故障复杂度（复杂 = 存储/集群级别，简单 = 单 VM 操作）动态设置 ReactExecutor 的 MAX_STEPS（简单 10 步，复杂 30 步）。

### 给 ops-agent 大脑的改进建议

1. **真流式输出**（高优先级）：base_agent.py 的 ReAct 循环应该在每步 LLM 推理时实时 yield token，而不是等待整个链路完成。这是在 Web UI 场景中使用 ops-agent 的前提条件。

2. **会话持久化**（高优先级）：AgentExecution 的 messages 历史和 `current_stage` 应该能序列化到 Redis，支持跨轮次恢复。这样才能在 htp 多轮对话环境中正常使用。

3. **接受外部上下文注入**：增加 `hci_context` 参数，接受调用方预取的实时数据，并自动注入为 system message。这样不需要修改工具，就能解决实时数据盲区。

4. **`present_sop_step_instruction` 的执行 bridge**：当指导内容是"执行 acli 命令"时，结构化输出一个 `ToolBridgeRequest`，让 htp 代为执行，而不是等用户手动输入结果。

## 九、总结

### 对用户模型的确认

用户提出的"大脑可选 + 骨架手脚共用"的架构切分是完全正确的。这不仅是工程上合理的，也是第一性原理的必然推导：真正需要 A/B 测试的是推理策略，而不是基础设施。

### 当前方案的核心问题

当前集成在错误的层次（LLM 客户端层）做了分支，导致：htp 的状态管理逻辑和 ops-agent 的状态管理逻辑同时运行，两者互相不知道对方的存在。实时数据被 htp 预取后注入 system prompt，但被 ops-agent 丢弃。两套状态机并行干扰，对话上下文每轮重置。

### 重设计的核心价值

重设计后的架构将实现：
- **ops-agent 大脑**看到实时的 HCI 告警数据（htp 预取注入）
- **ops-agent 大脑**保持跨轮次的诊断上下文（Redis session 持久化）
- **ops-agent 大脑**可以通过 ToolBridge 使用 htp 的 acli/SCP 执行能力（选做）
- **htp 大脑**保持现有所有能力，完全不受影响
- **前端**提供 A/B 切换，通过真实故障场景对比两个大脑的效果

这样才能真正回答那个核心问题：**SOP 驱动的 ops-agent 推理方式，与 S0-S6 + kb-service 的 htp 方式，在实际运维场景中哪个更实用？**

## 十、跨仓库代码管理方案（补充）

> 本节针对"两个独立仓库如何安全地共享接口边界"问题给出具体方案。  
> 详细实现规范见 [方案文档 §四](../solution/ai-assistant/events/2026-04-28-ops-agent大脑可选集成重设计方案.md) 和 [brain-http-api.yaml](../contracts/brain-http-api.yaml)。

### 10.1 架构模式：六边形架构（Ports & Adapters）

两仓库之间使用**六边形架构**解耦：

```
htp 仓库内部
┌─────────────────────────────────────────────────────┐
│  ConversationService                                │
│       │ 只依赖                                      │
│       ▼                                             │
│  BrainPort（Python Protocol 接口）                  │
│       ▲                    ▲                        │
│  HTPBrainAdapter       OpsAgentBrainAdapter         │
│  （htp 原有逻辑封装）  （HTTP 调用 ops-agent）      │
│                              │                      │
└──────────────────────────────┼──────────────────────┘
                               │ HTTP（跨仓库边界）
                        ops-agent 仓库
                    ┌──────────▼──────────┐
                    │  /v1/chat/completions│
                    │  实现 brain-http-api │
                    │  不知道 htp 内部    │
                    └─────────────────────┘
```

**核心原则**：依赖方向从外到内。`ConversationService` → `BrainPort`（接口）← `Adapters`（实现）。htp 和 ops-agent 只通过 HTTP API 接触，通过 OpenAPI 契约管理接口变更。

### 10.2 文件位置与所有权

| 文件 | 仓库 | 所有权 |
|------|------|-------|
| `docs/contracts/brain-http-api.yaml` | htp（权威来源）| 两仓库联合 review |
| `app/core/brain_port.py` | htp | htp team |
| `app/adapters/htp_brain_adapter.py` | htp | htp team（搬家，不新增逻辑）|
| `app/adapters/ops_agent_brain_adapter.py` | htp | htp team（反腐层）|
| `app/adapters/brain_router.py` | htp | htp team |
| `server/openai_compat.py` | ops-agent | ops-agent team（实现契约）|
| `server/session_store.py` | ops-agent | ops-agent team（内部实现）|

### 10.3 变更治理流程

```
向后兼容变更（新增可选字段）：
  ops-agent PR 先合并（实现新字段）
  → htp PR 再合并（开始使用新字段）
  → 无缝，用户无感知

破坏性变更（修改/删除已有字段）：
  两个 PR 必须在同一时间窗口合并
  → ops-agent PR 描述中必须引用 htp PR 编号

任何变更（无论兼容性）：
  必须先修改 brain-http-api.yaml 并合并
  → 契约文件是接口的唯一权威来源
```

### 10.4 为什么不用共享 Python 包

| 方案 | 适用场景 | 此项目 |
|------|---------|-------|
| HTTP + OpenAPI 契约 | 两端独立演进，未来可能废弃一侧 | ✅ 最适合 |
| 共享 PyPI 包 | 多消费方，强类型，高频变更 | ❌ 过度工程；ops-agent 是实验项目 |
| Monorepo | 强耦合，频繁跨仓库变更 | ❌ 两项目定位不同，不应绑定 |

## 十一、SOP ReAct：pydantic-ai vs 自研 ReactExecutor 深度对比，以及 A/B/C 三向测试方案

> 分析日期：2026-05-19  
> 背景：在"有限重构"决策下，讨论两个衍生问题：  
> 1. SOP 生产（`sop_template.py`）使用了 Pydantic 数据模型，消费端用 pydantic-ai 是否"更原生"？  
> 2. 是否可以直接新增 pydantic-ai 方案 C，形成 A/B/C 三向对比，而不是替换现有 ReactExecutor？  
> 分析方法：精读 pydantic-ai 源码（`pydantic_ai_slim/pydantic_ai/_agent_graph.py`、`tools.py`），比对 htp `ReactExecutor` 及 `sop_template.py` 数据结构

---

### 11.0 事实澄清：pydantic vs pydantic-ai

这是分析的前提，必须先澄清：

| | 当前 htp 代码库 |
|---|---|
| `sop_template.py`（SOPNode、DiagnosisDetail 等） | 使用 `pydantic.BaseModel`（数据校验库）|
| `sop_parser.py`（docx → SOPNode 树） | 纯 Python + 正则，**不使用任何 AI 框架** |
| 整个 htp / kb-service 代码库 | **从未引入 pydantic-ai**（代码全局搜索确认）|

**pydantic**（`pydantic.BaseModel`）和 **pydantic-ai**（`pydantic_ai.Agent`）是**两个独立的库**：

```
pydantic           → 数据模型校验库（数据类型、字段验证）
                     用于：SOPNode 数据结构定义、API Schema 校验

pydantic-ai        → AI Agent 框架（ReAct 执行引擎、LLM 调用、工具管理）
                     用于：构建 Agent 执行循环、工具注册、流式输出
```

SOP 生产（`sop_template.py`）使用了 pydantic（数据模型），**不是** pydantic-ai。

**但是**，这个区分不妨碍后面的论点——SOPNode 是 Pydantic 数据模型，而 pydantic-ai 天然支持 Pydantic 模型作为工具参数类型和返回类型，这个集成优势是真实存在的。

---

### 11.1 pydantic-ai 核心架构（代码精读）

pydantic-ai 不是"更好的 while 循环"，而是把 ReAct 整体建模为一个**类型安全的 Graph**（来自 `pydantic_graph`），由三个核心节点组成：

```
pydantic-ai 执行图：
  UserPromptNode
     │  处理用户输入、构建 system prompt
     ▼
  ModelRequestNode ──────────────────────────────────────────────────┐
     │  发送 LLM 请求（支持 request + request_stream 两种模式）        │
     │  内置 OTel instrumentation、capability middleware              │
     ▼                                                               │
  CallToolsNode                                                      │
     │  并发执行工具调用                                               │
     │  工具调用可以是 DeferredToolRequests（暂停等待人工审批）         │
     │  工具返回 ToolRetryError 时 LLM 自动感知并重试                  │
     └──────────────────────────────────────────────────────────────►┘
                       （下一个 ModelRequestNode）
```

**关键状态结构（`GraphAgentState`）：**
```python
@dataclasses.dataclass(kw_only=True)
class GraphAgentState:
    message_history: list[ModelMessage]      # 完整消息历史，可序列化
    usage: RunUsage                           # token 用量统计
    output_retries_used: int                 # 输出格式重试次数
    run_step: int                            # 当前步数
    run_id: str                              # 唯一运行 ID
    conversation_id: str                     # 对话 ID（跨轮次持久化）
```

**Human-in-Loop（`DeferredToolRequests`）：**
```python
# 工具调用可以被标记为"需要审批"
class DeferredToolRequests:
    """当 agent 返回这个类型时，运行暂停，执行结果以 DeferredToolResults 传入下次 run()"""
    calls: dict[str, ToolCallPart]     # 待审批的工具调用
    approvals: dict[str, Any]          # 审批决定

# 使用示例
result = await agent.run("诊断集群")
if isinstance(result, DeferredToolRequests):
    # 序列化到 Redis，等待 HTTP 回调
    redis.set(f"deferred:{session_id}", result.model_dump_json())
    # 用户审批后恢复
    results = DeferredToolResults(approvals={"tool_call_id": ToolApproved()})
    final = await agent.run(deferred_tool_results=results, message_history=...)
```

### 11.2 当前 ReactExecutor 对比（代码层面）

当前 `ReactExecutor`（`backend/conversation-service/app/core/react_executor.py`）：

```python
class ReactExecutor:
    async def run(self, state: AgentState, system_prompt: str) -> AsyncGenerator[str, None]:
        messages = [{"role": "system", "content": system_prompt}, *state.messages]
        
        while state.step_count < MAX_STEPS:   # MAX_STEPS = 15（硬编码常量）
            response = await self.glm.chat(messages=messages, tools=tools)
            
            if not response.tool_calls:
                yield response.content          # 裸文本字符串
                return
            
            for tool_call in response.tool_calls:
                result = await self._execute_tool_call(tool_call, state)
                # _execute_tool_call 内含完整风险控制：
                # risk_level=3 → block（直接拒绝）
                # risk_level=2 → confirm_service.request_confirm()（Redis + SSE 等待）
                # risk_level=1 → notify（SSE 通知）
                # risk_level=0 → auto（直接执行）
            messages.append(assistant_msg + tool_results)
```

### 11.3 逐维度对比表

| 对比维度 | 当前 `ReactExecutor`（自研） | pydantic-ai |
|---------|---------------------------|-------------|
| **架构范式** | 单一 `while` 循环，平铺迭代 | `pydantic_graph` 节点图（3节点），每步可观测 |
| **工具注册** | 静态 `TOOL_REGISTRY` 字典，手写 JSON Schema | `@agent.tool` 装饰器，自动从 Python 函数签名 + docstring 生成 Schema |
| **参数验证** | 手动 `dict` 解析，无类型约束，错误不 retry | Pydantic 全自动验证，参数错误自动触发 `ModelRetry` |
| **Human-in-Loop** | 自写 `Redis + SSE` + `confirm_service`（约 200 行） | 内置 `DeferredToolRequests / DeferredToolResults`，HTTP 层原生暂停/恢复 |
| **流式输出** | `yield str`（裸文本块） | `AgentStream`，结构化事件流（`TextPart`、`ToolCallPart` 等） |
| **状态持久化** | `AgentState` 纯内存，无序列化 | `GraphAgentState` 有完整 dataclass schema，可 JSON 序列化恢复 |
| **错误重试** | 工具报错直接返回错误字符串 | 工具 raise `ModelRetry`，LLM 自动感知并重试，有次数限制 |
| **步数限制** | `MAX_STEPS=15`（硬编码，项目内所有 ReAct 共用） | `UsageLimits(max_steps=N)` 可在每次 run 时独立配置 |
| **可观测性** | 手动加 OTEL span（缺失 tool-level trace） | 内置 Logfire / OTEL，每个节点、每次工具调用自动 trace |
| **多模型支持** | `GLMClient` 单一，仅支持 GLM | 原生支持 OpenAI / Anthropic / Gemini 等 30+ 模型；任意 OpenAI-compatible 端点通过 `base_url` 复用 |
| **输出类型验证** | 纯文本字符串，无结构验证 | 输出可以是 Pydantic 模型，自动验证并在不符合时重试 |
| **依赖注入** | 构造函数传参（`glm_client`、`tool_executor` 等） | 泛型 `DepsT`，类型安全依赖注入，运行时通过 `RunContext` 访问 |

### 11.4 SOPNode 与 pydantic-ai 的天然集成（核心论点）

用户提出"SOP 生产用了 pydantic（数据模型），消费端用 pydantic-ai 更原生"，这个论点的本质是：

**SOP 数据结构是 Pydantic 模型，pydantic-ai 工具系统天然支持 Pydantic 类型，无需手写 JSON Schema。**

具体体现如下：

```python
# 当前方式（自研 ReactExecutor）：手写 JSON Schema
TOOL_REGISTRY["navigate_sop_node"] = ToolDefinition(
    name="navigate_sop_node",
    description="导航到 SOP 决策树中的指定节点",
    parameters={
        "type": "object",
        "properties": {
            "node_id": {"type": "string", "description": "节点 ID"},
        },
        "required": ["node_id"],
    },
    risk_level=0,
    policy="auto",
    category="sop_navigation",
)

# pydantic-ai 方式：直接用 SOPNode 作为工具返回类型，Schema 自动生成
@agent.tool
async def navigate_sop_node(ctx: RunContext[SOPDeps], node_id: str) -> SOPNode:
    """导航到 SOP 决策树中的指定节点。

    Args:
        node_id: 目标节点 ID（如 "n-1-2-3"），来自父节点的 children 列表。
    """
    return await ctx.deps.sop_service.get_node(node_id)
    # SOPNode 是 pydantic.BaseModel，pydantic-ai 自动序列化为 JSON 返回给 LLM
    # LLM 看到的是结构化的 prerequisites / diagnosis / solution 字段
```

**优势**：`SOPNode` 的 `prerequisites`、`diagnosis.page_methods`、`solution.quick_recovery` 等字段直接以结构化 JSON 形式传递给 LLM，LLM 可以精准访问每个字段，不需要从文本中提取。这比当前把 SOPNode 序列化为 markdown 字符串注入 prompt 要干净得多。

### 11.5 迁移收益与代价

**迁移收益（含 SOPNode 天然集成）：**
- **SOPNode 结构化输出**：工具返回 `SOPNode`，LLM 直接看到 prerequisites / diagnosis / solution，无需从文本中提取字段
- 消除手写 JSON Schema（`TOOL_REGISTRY` 手工维护 → `@agent.tool` 自动生成）
- `DeferredToolRequests` 替代手写的 `confirm_service` + Redis + SSE（约 200 行可消除）
- 内置 OTEL / Logfire，tool-level trace 零配置
- 未来切换 LLM 只改 `model=` 参数

**迁移代价（三件独立工程任务）：**

1. **GLMClient / OpenClaw 适配**：如果是 OpenAI-compatible 端点，直接 `OpenAIModel(base_url="...")`，几乎零成本。否则写 custom model（约 100 行）。**这是关键先决条件，需要先验证。**

2. **confirm_service 重构**：`risk_level=2` 的确认逻辑用 `DeferredToolRequests` 实现，SSE 事件格式需同步更新前端。逻辑等价，但是前端变更不可忽视。

3. **TOOL_REGISTRY risk_level 元数据迁移**：在工具函数内部根据业务规则决定是否 raise `DeferredToolRequests`（高危）或 notify-only（低危）。业务逻辑，不是框架问题，但需要逐工具 review。

### 11.6 A/B/C 三向测试方案（推荐路径）

> 用户问：为什么走"两段式路径（有限重构 → 迁移 pydantic-ai）"，能不能直接新增 pydantic-ai 作为第三个大脑 C，而不是替换 A？

这个方向**比两段式路径更合理**。原因如下：

**两段式路径的问题**：
```
有限重构（A 扩展）：       后续迁移（A 被替换）：
 ↓                         ↓
手写工具 JSON Schema     →  pydantic-ai 自动 JSON Schema
扩展 TOOL_REGISTRY      →  @agent.tool 装饰器
调整 MAX_STEPS 配置     →  UsageLimits 配置
（这部分工作大量重叠，
  有限重构的成果部分被浪费）
```

**A/B/C 三向方案**：

```
A 大脑（htp 自研 ReactExecutor）  ← 不动，保持现有能力
B 大脑（ops-agent 自由 ReAct）    ← 已接入，保持不变
C 大脑（pydantic-ai Guided ReAct）← 新增，独立实现

接口层：BrainPort Protocol 已就绪，新增 C 只需实现 PydanticAIBrainAdapter
```

**方案接口（利用现有 BrainPort）：**
```python
# 仅需新增一个 Adapter，BrainRouter 增加一条路由
class PydanticAIBrainAdapter:
    """C 大脑：pydantic-ai SOP Guided ReAct"""
    
    def __init__(self, agent: Agent[SOPDeps, str], ...): ...
    
    async def process(
        self, *, session_id, messages, env_context, stream, ...
    ) -> AsyncGenerator[BrainEvent, None]:
        # pydantic-ai agent.run_stream() → 转换为 BrainEvent 流
        async with agent.run_stream(...) as result:
            async for text in result.stream_text(delta=True):
                yield BrainTextChunk(content=text)

# BrainRouter 增加 C 的路由分支（约 5 行）
class BrainRouter:
    async def process(self, *, assistant_type, ...):
        if assistant_type == "ops-agent":   → OpsAgentBrainAdapter（不变）
        elif assistant_type == "pydantic-ai":  → PydanticAIBrainAdapter（新增）
        else:                               → HTPBrainAdapter（不变）
```

**三向对比维度：**

| 指标 | A（htp 自研）| B（ops-agent）| C（pydantic-ai）|
|------|------------|--------------|----------------|
| **SOP 遍历方式** | 手写工具集 + while 循环 | LLM 自由 ReAct | @agent.tool + Graph 节点 |
| **SOPNode 集成** | 序列化为 markdown 文本 | JSONL 文件解析 | 原生 Pydantic 结构化返回 |
| **风险管控** | risk_level 硬性管控 | 无管控 | DeferredToolRequests（等价）|
| **工具数** | 当前约 8 个，扩展后 12+ 个 | 约 15 个 | 按需定义，理论相同 |
| **多模型切换** | 绑定 GLMClient | 绑定 ops-agent LLM | 任意 OpenAI-compatible |
| **状态持久化** | 内存，无序列化 | Redis（改造后）| GraphAgentState 可序列化 |
| **维护成本** | 手写 JSON Schema | 手写 JSON Schema | 自动 Schema，低维护 |

**✅ 前提条件验证（已完成，2026-05-19）**

代码精读 `glm_client.py` 和 `ai_client.py`，验证结论：

**A/B/C 方案正式确认可行，推荐直接执行，不走"有限重构"两段式路径。**

**任务路线（已分解，见任务文档）**：

**A/B/C 方案的长期价值**：

## 十二、有限重构对大脑可热切换 / A/B 测试的影响分析

> 背景：有限重构改变了 HTPBrainAdapter 的内部执行路径（S1+ 阶段引入 SOP Guided ReAct）。  
> 核心问题：这是否破坏原有的"大脑热切换"或"A/B 测试对象"机制？

---

### 12.1 接口边界分析

有限重构的边界约束（已在决策中明确）：
- **完全不动**：`BrainPort` Protocol、`BrainRouter` 路由逻辑、`OpsAgentBrainAdapter`
- **内部变更**：`HTPBrainAdapter.process()` 在 S1+ 阶段增加 `sop_document_id` 判断，选择性激活 SOP Guided ReAct
- **接口不变**：`process(*, session_id, messages, env_context, stream, ...) → AsyncGenerator[BrainEvent]`

从 `BrainRouter` 的视角看：

```python
# brain_router.py（完全不受影响）
class BrainRouter:
    async def process(self, *, assistant_type, session_id, messages, ...) -> AsyncGenerator[BrainEvent, None]:
        if assistant_type == "ops-agent":
            async for event in self.ops_agent_adapter.process(...):  # 独立路径，完全不动
                yield event
        else:
            async for event in self.htp_adapter.process(...):        # 内部实现变了，但接口签名不变
                yield event
```

`BrainRouter` 感知不到 `htp_adapter` 内部发生了什么，它只关心 `BrainPort.process()` 接口是否遵守——这个接口完全没有变化。

---

### 12.2 切换时序（完整追踪）

```
用户操作：在前端选择大脑类型
    │
    ▼
前端发送：POST /api/conversations  body: { assistant_type: "ops-agent" | "openclaw" }
    │
    ▼
ConversationService.send_message_stream_only()
    │  读取 conversation.assistant_type
    ▼
BrainRouter.process(assistant_type=...)
    │
    ├── assistant_type == "ops-agent"  ──► OpsAgentBrainAdapter.process()
    │                                       ops-agent 整套 ReAct（不受有限重构任何影响）
    │
    └── assistant_type == "openclaw"  ──► HTPBrainAdapter.process()
                                          ├── S0：用户故障描述（不变）
                                          ├── S1：KB 路由 + SOP 定位（不变）
                                          │    └── 若找到 SOP：sop_document_id 写入 AgentState
                                          ├── S2+：
                                          │    ├── 有 sop_document_id → SOP Guided ReAct（新增）
                                          │    └── 无 sop_document_id  → 原有 LLM 推理（不变）
                                          └── S6：结案（不变）
```

**结论：热切换机制本身（BrainPort/BrainRouter 架构）不受任何影响。** 切换粒度是会话级别（每次新建对话选择大脑），切换触发点在 BrainRouter，与 HTPBrainAdapter 内部实现完全解耦。

### 12.4 A/B 测试比较基准的变化（必须显式说明）

有限重构后，A/B 测试的两侧对比：

```
重构前（当前状态）：
  htp 大脑：KB 检索 + 单次 LLM + S0-S6 代码控制（无自主工具执行）
  ops-agent 大脑：SOP 驱动 + 自由 ReAct（自主工具执行，但不走 htp 工具基础设施）
  → 对比维度："代码控制 FSM" vs "LLM 控制 FSM"

重构后（有限重构）：
  htp 大脑：KB 检索 + SOP Guided ReAct + S0-S6 外壳（自主工具执行，受 risk_level 管控）
  ops-agent 大脑：SOP 驱动 + 自由 ReAct（相对无管控）
  → 对比维度：
    - "SOP 导航精准度"：代码导航 vs LLM 导航
    - "工具执行管控"：risk_level 硬性管控 vs LLM 自主判断
    - "故障覆盖范围"：SOP 覆盖的故障（htp 更好）vs 无 SOP 的探索（ops-agent 更好）
```

**A/B 测试假设更新建议**：

| 指标 | 重构前假设 | 重构后假设 |
|------|-----------|-----------|
| **诊断准确率** | htp（规则）vs ops-agent（LLM） | htp（Guided ReAct）vs ops-agent（Free ReAct） |
| **高危操作安全性** | htp 强（代码阻断）vs ops-agent 弱 | htp 强（risk_level 管控）vs ops-agent 弱（仍然）|
| **无 SOP 场景覆盖** | 相当（都靠 LLM 自由发挥）| htp 降级为原有逻辑，ops-agent 自由探索 |
| **运行时间** | htp 快（单次 LLM）vs ops-agent 慢（多步 ReAct） | 两者都是多步 ReAct（差距缩小，取决于 SOP 复杂度）|

### 12.5 总结

**有限重构对大脑热切换/A/B 测试的影响分两层：**

1. **机制层（零影响）**：`BrainPort` 接口、`BrainRouter` 路由、前端切换 UI、`BrainEvent` 格式，这些都完全不变。任何时刻可以无缝在 htp 和 ops-agent 之间切换，热切换能力完整保留。

2. **语义层（有意变化）**：htp 大脑从"单次 LLM + 代码 FSM"变成"SOP Guided ReAct + 代码外壳"。这是重构的预期效果——htp 大脑的内在能力得到了提升。A/B 测试的比较基准随之变化，但比较本身仍然有意义（不同的 ReAct 策略，不同的管控模型）。

**需要更新的工件**：A/B 测试用例设计文档（如有），需要更新"htp 大脑行为预期"一栏，以反映 S1+ 阶段的 SOP Guided ReAct 执行模式。
