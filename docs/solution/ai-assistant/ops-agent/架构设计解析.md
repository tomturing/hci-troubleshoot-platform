# Ops Agent 架构设计文档

> **详细模块文档**：如需深入了解各模块的具体实现，请参阅 [modules/](./modules/) 目录下的模块设计文档。

## 1. 项目概述

Ops Agent 是一个面向本地 SOP（标准操作程序）知识库的智能故障排查与操作引导工具。它通过大语言模型驱动，结合结构化的 SOP 数据，为用户提供智能化的故障排查流程引导。

### 1.1 核心能力

- **SOP 检索与收敛**：基于用户问题检索相关 SOP 候选，并通过多轮对话收敛到最匹配的排障路径
- **结构化引导**：提供清晰的步骤指引、风险提示和上下文信息
- **用户交互**：支持多轮对话，动态补充缺失信息
- **轨迹记录**：完整记录执行过程，便于复盘、审计和调试
- **双模式运行**：支持 CLI 命令行和 Web 界面两种运行方式

### 1.2 技术栈

- **语言**：Python 3.12+
- **依赖管理**：uv
- **大语言模型**：支持 OpenRouter、OpenAI、Anthropic 等多种提供商
- **Web 框架**：Streamlit
- **协议支持**：ACP (Agent Client Protocol)、MCP (Model Context Protocol)

## 3. Agent 模式设计与实现

### 3.1 核心设计理念

Ops Agent 采用**主从 Agent 协作模式**：
- **OpsAgent**：负责任务 orchestration、用户交互、流程控制
- **SOPQuerySubAgent**：专注于 SOP 检索与路径收敛，子任务执行

这种设计的优势：
1. **职责分离**：主 Agent 处理流程，子 Agent 专注检索
2. **资源隔离**：子 Agent 有独立的预算、步数限制
3. **可扩展性**：可以轻松添加新的子 Agent 处理特定任务

### 3.2 BaseAgent 抽象基类

`BaseAgent` 是所有 Agent 的基类，提供通用的 Agent 运行时能力。

#### 3.2.1 核心职责

```python
class BaseAgent(ABC):
    """Agent 抽象基类"""

    # 核心属性
    - _llm_client: LLMClient           # LLM 客户端
    - _model_config: ModelConfig       # 模型配置
    - _max_steps: int                  # 最大执行步数
    - _tools: List[Tool]               # 可用工具列表
    - _tool_caller: ToolExecutor       # 工具执行器
    - _trajectory_recorder: TrajectoryRecorder  # 轨迹记录器
    - _message_summarizer: MessageSummarizer    # 消息摘要器
    - _approval_manager: ApprovalManager        # 审批管理器
```

#### 3.2.2 核心执行流程

`BaseAgent.execute_task()` 实现了 Agent 的主循环：

```
初始化
    │
    ▼
┌─────────────────────────────────┐
│   检查并压缩消息历史           │◄───────────────────┐
└─────────────────────────────────┘                    │

    ▼                                                  │
┌─────────────────────────────────┐                    │
│   调用 LLM 生成响应             │                    │
└─────────────────────────────────┘                    │
    │                                                  │
    ▼                                                  │
┌─────────────────────────────────┐                    │
│   检查任务是否完成？            │                    │
└─────────────────────────────────┘                    │
    │                                                  │
    ├─ 是 → 提取最终结果 → 结束                       │
    │                                                  │
    否                                                │
    │                                                  │

┌─────────────────────────────────┐                    │
│   执行工具调用                  │                    │
└─────────────────────────────────┘                    │
    │                                                  │
    ▼                                                  │
┌─────────────────────────────────┐                    │

└─────────────────────────────────┘                    │
    │                                                  │
    └──────────────────────────────────────────────────┘
```

#### 3.2.3 关键特性

##### 1. 智能重试机制

Agent 内置了三层重试逻辑：

| 重试场景 | 触发条件 | 处理方式 |
|---------|---------|---------|
| 空响应 | LLM 返回空内容 | 调整 temperature 重试 |
| 仅文本响应 | 有内容但无工具调用 | 调整 temperature 重试 |
| 工具执行失败 | 工具调用返回错误 | 根据配置决定是否重试 |

```python
# 重试温度调度
retry_temperatures = [0.6, 0.8, 1.0]
```

##### 2. 消息压缩与上下文管理

为避免上下文窗口溢出，`MessageSummarizer` 实现了：
- Token 使用量监控
- 自动压缩历史消息
- 保留关键信息的摘要

##### 3. 预算管理

支持工具调用预算控制：
```python
_tool_call_budget: int              # 总预算
_tool_call_budget_used: int         # 已使用
BUDGET_WARNING_THRESHOLD: int = 10  # 警告阈值
```

##### 4. 重复调用检测

检测连续两轮完全相同的工具调用，防止死循环：
```python
REPEATED_TOOL_CALL_WHITELIST = {
    "sequentialthinking",
    "ops_state_update",
    "get_info_from_user",
    # ...
}
```

##### 5. MCP 协议支持

支持通过 MCP (Model Context Protocol) 集成外部工具：
- 并行初始化多个 MCP 服务器
- 工具冲突检测与处理
- 运行时状态管理

### 3.3 OpsAgent 主 Agent

`OpsAgent` 是面向用户的主 Agent，负责端到端的排障流程。

#### 3.3.1 核心职责

1. **任务初始化**
   - 解析用户问题
   - 加载 SOP 知识库
   - 初始化工具集

2. **流程编排**
   - 信号抽取与路由决策
   - SOP 检索与收敛
   - 状态管理与更新

3. **用户交互**
   - 信息收集
   - 步骤指引
   - 结果展示

#### 3.3.2 初始化流程

```python
def __init__(self, ops_agent_config: OpsAgentConfig, ...):
    # 1. 解析 SOP 目录路径
    self.sop_catalog_path = _resolve_sop_catalog_path(...)

    # 2. 初始化 BaseAgent
    super().__init__(agent_config=ops_agent_config, ...)

def new_task(self, task: str, extra_args: dict = None, ...):
    # 1. 设置任务与工作目录
    self._task = task
    self.project_path = extra_args.get("project_path", ...)

    # 2. 初始化工具集
    self._tools = [
        tools_registry[tool_name](model_provider=...)
        for tool_name in DEFAULT_OPS_AGENT_TOOLS
    ]

    # 3. 为工具注入依赖
    for tool in self._tools:
        if hasattr(tool, "set_sop_catalog_path"):
            tool.set_sop_catalog_path(self.sop_catalog_path)
        if hasattr(tool, "set_ops_agent_config"):
            tool.set_ops_agent_config(self._ops_agent_config)
        # ...

    # 4. 构建初始消息
    self._initial_messages = self._get_initial_messages()

    # 5. 开始轨迹记录
    if self._trajectory_recorder:
        self._trajectory_recorder.start_recording(...)
```

### 3.4 SOPQuerySubAgent 子 Agent

`SOPQuerySubAgent` 是专门负责 SOP 检索的子 Agent。

#### 3.4.1 设计特点

1. **独立配置**
   - 独立的模型配置
   - 独立的步数限制
   - 独立的工具调用预算

2. **专注检索**
   - 不处理用户交互
   - 专注于 SOP 候选的比较与收敛

3. **预算控制**
   - 限制工具调用次数
   - 防止无限检索

#### 3.4.2 子 Agent 工具集

```python
DEFAULT_SOP_QUERY_SUBAGENT_TOOLS = [
    "sequentialthinking",           # 思维链
    "sub_agent_task_done",          # 子任务完成

    "sop_query_read_target",        # 读取 SOP 目标
    "sop_query_compare_candidates", # 比较 SOP 候选
    "sop_query_checkpoint",         # 检索检查点
]
```

## 5. 工具系统设计

### 5.1 工具基类

所有工具继承自 `Tool` 基类：

```python
class Tool(ABC):
    @property
    @abstractmethod
    def name(self) -> str:
        pass

    @property
    @abstractmethod
    def description(self) -> str:
        pass

    @property
    @abstractmethod
    def parameters(self) -> dict:
        pass

    @abstractmethod
    async def execute(self, arguments: dict) -> ToolResult:
        pass

    def requires_user_approval(self, arguments: dict) -> bool:
        """是否需要用户审批，默认不需要"""
        return False
```

### 5.2 工具注册机制

使用装饰器注册工具：

```python
@tools_registry.register("tool_name")
class MyTool(Tool):
    # ... 实现
```

### 5.3 核心工具详解

#### 5.3.1 sequentialthinking

思维链工具，帮助 Agent 组织思考过程。

#### 5.3.2 ops_state_update

状态管理工具，维护排障过程的状态机。

状态包含：
- 当前阶段 (intake / routing / troubleshooting / ...)
- 已确认的信号
- 未知信号
- 当前 SOP 路径
- 已执行步骤
- 等等

#### 5.3.3 query_sop_candidates

SOP 检索工具，启动子 Agent 执行检索。

这是连接主 Agent 和子 Agent 的桥梁工具。

#### 5.3.4 get_info_from_user

用户交互工具，向用户提问并收集信息。

通过 ACP 协议的 `_ops/request_input` 扩展实现。

#### 5.3.5 present_sop_step_instruction

展示排障步骤指引，提供结构化的操作指导。

## 6. ACP 协议桥接层设计

### 6.1 设计目标

ACP (Agent Client Protocol) 桥接层的目标是：
- 将 Agent 的执行过程标准化为消息流
- 支持多前端（CLI、Web、其他）接入
- 实现前后端分离

### 6.2 核心协议

#### 6.2.1 标准 ACP 消息

| 消息类型 | 用途 |
|---------|------|
| `initialize` | 初始化会话 |
| `session/new` | 创建新会话 |
| `session/prompt` | 发送提示 |
| `session/update` | 会话状态更新 |
| `session/request_permission` | 请求工具执行权限 |

#### 6.2.2 扩展协议：_ops/request_input

由于 ACP 标准没有覆盖"Agent 向用户发起业务问题确认"的场景，Ops Agent 扩展了协议：

```json
{
  "method": "_ops/request_input",
  "params": {
    "type": "info_request | step_instruction",
    "content": { ... }
  }
}
```

用于承载：
- `get_info_from_user` 的选项选择 / 自定义输入
- `present_sop_step_instruction` 的结构化步骤卡交互

### 6.3 Web 前端集成

`ops_web` 模块使用 Streamlit 构建，通过 ACP 协议与 Agent 交互：

```
ops_web (Streamlit)
    │
    │ ACP 消息
    │
    ▼
ACP 桥接层
    │
    │
    ▼
OpsAgent
```

这种设计使得：
- Web 前端不依赖 CLI 的渲染逻辑
- 可以轻松替换为其他前端实现
- 前后端可以独立演进

## 7. 轨迹记录系统

### 7.1 设计目标

轨迹记录系统用于：
- 复盘：重现完整执行过程
- 审计：检查 Agent 的决策逻辑
- 调试：定位问题所在
- 评估：用于 benchmark 和效果评估

### 7.2 记录内容

轨迹记录包含：
- 任务描述
- 模型与提供商信息
- 每一步的详细信息：
  - 思考内容
  - 工具调用
  - 工具结果
  - 时间戳
  - 持续时间
- 最终结果与总结

### 7.3 存储格式

默认存储位置：
```
.trajectories/
└── YYYY-MM-DD/
    └── {timestamp}_{agent_name}.jsonl
```

## 10. 总结

Ops Agent 的架构设计具有以下特点：

1. **分层清晰**：用户交互层、Agent 层、工具层、运行时层、数据层职责明确
2. **主从协作**：OpsAgent 与 SOPQuerySubAgent 分工协作
3. **配置灵活**：支持多模型提供商、多模型配置
4. **协议标准**：基于 ACP 协议，支持多前端接入
5. **扩展友好**：支持自定义 Agent、自定义工具、MCP 集成
6. **生产就绪**：内置重试、预算、轨迹、审批等生产级特性

这种设计使得 Ops Agent 既可以作为独立工具使用，也可以作为组件集成到更大的系统中。
