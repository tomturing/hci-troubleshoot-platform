# HCI 智能排障平台 — Agent 框架全景比较

> **文档目的**：在不受现有代码约束的前提下，评估所有主流 Agent 框架，为"再造专业HCI智能排障Claude"选择理论最优解。
> **创建日期**：2026-03-21
> **关联文档**：[HCI平台效果差距分析与重构方案.md](./HCI平台效果差距分析与重构方案.md)

## 一、问题定义：本项目对框架的核心需求

评估必须从需求出发，否则"最优解"没有意义。

### 1.1 核心约束

| 约束 | 具体说明 | 对框架的影响 |
|------|---------|------------|
| **模型**：GLM-4（通过 OpenClaw） | OpenAI 兼容协议，但细节有差异 | 框架必须支持自定义 LLM 接入 |
| **人工确认**：高风险操作必须等用户授权 | 操作 HCI 不可逆，硬性要求 | 框架必须支持执行中断/恢复 |
| **流式输出**：SSE 实时推文字 | 用户体验要求 | 框架必须支持 token 级流式 |
| **状态持久化**：对话可中断，跨请求延续 | 人工确认可能几分钟到几小时 | 框架需要外部持久化支持 |
| **审计日志**：每次工具调用必须可追溯 | 生产环境安全要求 | 框架不能黑盒吞没操作记录 |
| **领域状态机**：7 阶段诊断流程 | 特定领域逻辑不能被通用框架覆盖 | 框架必须允许业务逻辑嵌入 |

### 1.2 评分维度（5分制）

- **GLM 兼容性**：接入成本，是否有隐式格式假设
- **中断/恢复**：人工确认能力，暂停后继续执行
- **流式支持**：token 级 SSE 推送
- **可观测性**：每步可记录、可审计、可回放
- **学习曲线**：团队掌握成本
- **生产成熟度**：社区、稳定性、已知 production case
- **灵活性**：能否嵌入自定义状态机、业务逻辑

## 二、候选框架全景

### 2.1 LangGraph（LangChain 下一代）

**定位**：有状态 Agent 的生产级编排框架，基于图结构

**核心机制**：
```
StateGraph（有向图）
  ├── 节点（Node）= 任意 Python 函数
  ├── 边（Edge）= 转换条件
  ├── 检查点（Checkpointer）= 状态持久化（支持 Postgres/Redis）
  └── interrupt()  = 执行暂停原语，等外部输入后恢复
```

**杀手级功能**：
- `interrupt()` 原语：执行图在任意节点暂停，状态序列化到 DB，任意时刻恢复
- 时间旅行调试：可回放任意历史检查点，复现问题
- `astream_events()`：原生 token 级流式，可区分 LLM token / 工具调用事件
- LangSmith 集成：完整可观测（也可不用，使用 OpenTelemetry）

**代码示例（人工确认中断）**：
```python
from langgraph.types import interrupt
from langgraph.checkpoint.postgres.aio import AsyncPostgresSaver

async def confirm_node(state: AgentState):
    # 整个图在此暂停，状态持久化到 Postgres
    # 前端可以在 10 分钟后发请求来恢复
    decision = interrupt({
        "type": "confirmation",
        "tool": state["pending_tool"],
        "message": "即将执行迁移操作，是否确认？"
    })
    return {"confirmed": decision}

# 挂载 Postgres 检查点
checkpointer = AsyncPostgresSaver.from_conn_string(DATABASE_URL)
app = graph.compile(checkpointer=checkpointer)

# 恢复执行（可以是分钟甚至小时后）
await app.ainvoke(
    Command(resume={"confirmed": True}),
    config={"configurable": {"thread_id": "session-123"}}
)
```

**GLM 兼容性**：使用 `ChatOpenAI(base_url=OPENCLAW_URL)` 接入，**需要测试 parallel tool calls 行为**，GLM 有时不严格遵循 OpenAI 的 `tool_calls` 格式。

**评估**：

| 维度 | 分数 | 说明 |
|------|------|------|
| GLM 兼容性 | ⭐⭐⭐⭐ | OpenAI 兼容，需处理边缘 case |
| 中断/恢复 | ⭐⭐⭐⭐⭐ | interrupt() 是最成熟的实现 |
| 流式支持 | ⭐⭐⭐⭐⭐ | astream_events() 原生支持 |
| 可观测性 | ⭐⭐⭐⭐⭐ | LangSmith + OTel，时间旅行调试 |
| 学习曲线 | ⭐⭐⭐ | 图模型概念需要适应 |
| 生产成熟度 | ⭐⭐⭐⭐⭐ | 最多生产 case，2025 年事实标准 |
| 灵活性 | ⭐⭐⭐⭐ | 图结构可以很灵活，但有概念强制 |
| **综合** | **32/35** | |

### 2.2 smolagents（HuggingFace，2024年底）

**定位**：回归论文原始设计，极简 ReAct，代码即工具

**核心理念**：工具不是 JSON schema，而是 **Python 函数**；LLM 生成的不是 JSON 参数，而是**Python 代码片段**（Code Agent 模式）

```python
from smolagents import CodeAgent, tool

@tool
def get_active_alerts(cluster_id: str) -> list:
    """获取指定集群的活跃告警列表"""
    return scp_client.get_alerts(cluster_id)

agent = CodeAgent(tools=[get_active_alerts, ...], model=glm_model)
agent.run("查看集群 A 的当前告警并分析根因")
```

**Code Agent 模式的独特优势**：
- LLM 生成代码，可以做任意逻辑组合（循环、条件、多工具串联）
- 不受工具 schema 格式限制
- 对 GLM 的兼容性最好（不依赖 function calling 格式）

**致命弱点**：
- **Code 执行安全**：LLM 生成的代码在沙箱里运行，HCI 排障场景里生成错误代码后果严重
- 没有原生 interrupt() 机制
- 流式支持弱

**评估**：

| 维度 | 分数 | 说明 |
|------|------|------|
| GLM 兼容性 | ⭐⭐⭐⭐⭐ | 不依赖 function calling，最兼容 |
| 中断/恢复 | ⭐⭐ | 无原生支持，需自实现 |
| 流式支持 | ⭐⭐⭐ | 有 streaming 但不完善 |
| 可观测性 | ⭐⭐⭐ | 基础日志，无高级功能 |
| 学习曲线 | ⭐⭐⭐⭐⭐ | 极简，一天上手 |
| 生产成熟度 | ⭐⭐⭐ | 新框架，production case 少 |
| 灵活性 | ⭐⭐⭐⭐⭐ | 极高，代码即工具 |
| **综合** | **27/35** | |

**本项目适用性**：⚠️ 不推荐。Code Agent 模式在基础设施操作场景下安全风险过高。

### 2.3 AutoGen（Microsoft，2024 v0.4 重构）

**定位**：多 Agent 对话编排，核心场景是多个专家 Agent 协作

**核心模式**：
```
用户(Human) ←→ AssistantAgent（通用协调）
                    ↕
              NetworkAgent（网络专家）
              StorageAgent（存储专家）
              VMAgent（虚拟机专家）
```

**v0.4 重构后的新特性**：
- 事件驱动架构（非阻塞）
- Actor 模型（每个 Agent 是独立进程）
- `AutoGen Studio`：可视化设计 Agent 交互

**本项目分析**：
- 多 Agent 协作在 **知识获取阶段** 可能有价值（网络专家 + 存储专家并行诊断）
- 但我们的核心场景是**单 Agent + 工具调用**，多 Agent 增加复杂度
- Actor 模型与现有 FastAPI 异步架构集成需要额外工作

**评估**：

| 维度 | 分数 | 说明 |
|------|------|------|
| GLM 兼容性 | ⭐⭐⭐⭐ | 支持自定义 model client |
| 中断/恢复 | ⭐⭐⭐ | 有 human_input_mode，但不如 LangGraph |
| 流式支持 | ⭐⭐⭐⭐ | v0.4 支持流式 |
| 可观测性 | ⭐⭐⭐⭐ | AutoGen Studio 可视化 |
| 学习曲线 | ⭐⭐⭐ | 多 Agent 概念复杂 |
| 生产成熟度 | ⭐⭐⭐⭐ | 微软背书，v0.4 较稳定 |
| 灵活性 | ⭐⭐⭐⭐ | 多 Agent 场景下灵活 |
| **综合** | **26/35** | |

**本项目适用性**：⚠️ 有用但不是最优。多 Agent 场景是未来 Phase 4+ 的方向（多域并行诊断），当前阶段过度设计。

### 2.4 CrewAI

**核心模式**：

# Agent 有角色、目标、背景

# Task 明确分工

# Crew 编排执行
crew = Crew(agents=[network_expert, storage_expert], tasks=[task])

### 2.5 OpenAI Assistants API + Threads

**定位**：OpenAI 托管的 Agent 运行时（有状态、持久化由平台管理）

**特点**：
- 对话状态由 OpenAI 平台管理（Threads）
- 工具调用（Function Calling + Code Interpreter + File Search）
- 流式支持完整

**本项目问题**：
- 依赖 OpenAI 服务，本项目使用私有部署 GLM → **不可用**
- 数据不能上传外部服务（安全合规）

**评估**：❌ 排除，私有化部署约束。

### 2.6 自实现 ReAct 执行器（Vanilla ReAct）

**定位**：直接实现论文算法，无框架依赖

**核心实现**（完整可运行版本约 150 行）：
```python
import asyncio
from typing import AsyncGenerator
from pydantic import BaseModel

class ToolCall(BaseModel):
    name: str
    args: dict
    risk_level: int
    policy: str  # auto | notify | confirm | block

class ReactExecutor:
    def __init__(self, llm, tool_registry, max_steps=15):
        self.llm = llm
        self.tools = {t["name"]: t for t in tool_registry}
        self.max_steps = max_steps

    async def run(self, thread_id: str, user_input: str) -> AsyncGenerator:
        session = await load_session(thread_id)
        session.messages.append({"role": "user", "content": user_input})

        for step in range(self.max_steps):
            # 推理
            response = await self.llm.chat(
                session.messages,
                tools=list(self.tools.values()),
                stream=True
            )
            async for chunk in response:
                yield {"type": "token", "content": chunk}   # SSE 实时推送

            # 判断下一步
            if response.finish_reason == "stop":
                await save_session(thread_id, session)
                return

            # 工具调用
            for tool_call in response.tool_calls:
                tool_def = self.tools[tool_call.name]

                # 授权检查（框架强制，不问 LLM）
                if tool_def["policy"] == "confirm":
                    yield {"type": "confirm_request", "tool": tool_call}
                    decision = await wait_confirmation(thread_id)
                    if not decision:
                        session.messages.append(tool_result(tool_call, "用户取消操作"))
                        continue

                # 执行 + 审计
                result = await execute_tool(tool_call)
                await audit_log(thread_id, tool_call, result)    # 不可绕过

                session.messages.append(tool_result(tool_call, result))
                yield {"type": "tool_result", "tool": tool_call.name, "result": result}
```

**评估**：

| 维度 | 分数 | 说明 |
|------|------|------|
| GLM 兼容性 | ⭐⭐⭐⭐⭐ | 自己写格式处理，完全掌控 |
| 中断/恢复 | ⭐⭐⭐ | 需自实现（Redis + 等待队列） |
| 流式支持 | ⭐⭐⭐⭐⭐ | 直接对接现有 SSE，零适配 |
| 可观测性 | ⭐⭐⭐⭐⭐ | 完全透明，每步可记录 |
| 学习曲线 | ⭐⭐⭐⭐⭐ | Python，无新概念 |
| 生产成熟度 | ⭐⭐⭐ | 需自己保证质量 |
| 灵活性 | ⭐⭐⭐⭐⭐ | 完全自由 |
| **综合** | **30/35** | |

## 四、最终推荐：分阶段框架策略

### 4.1 结论

**理论最优 = LangGraph**（最成熟的人工确认 + 状态持久化方案）  
**工程最优 = 自实现 ReAct，接口按 LangGraph 概念设计**（考虑现有代码集成成本）

两者综合分相同（31分），决策因子是**迁移代价和团队熟悉度**。

### 4.2 推荐的分阶段策略

```
Phase 3 初期（MVP）：自实现 ReAct
  → 快速验证 ReAct + 工具调用的诊断效果
  → 接口设计对齐 LangGraph 概念（State, Node, Edge, interrupt）
  → 人工确认用 Redis pub/sub + SSE 实现

Phase 3 成熟期（6个月后，视情况）：评估迁移 LangGraph
  → 如果 interrupt() 自实现维护成本高 → 迁移
  → 如果跨会话状态持久化成为痛点 → 迁移
  → 如果需要时间旅行调试 → 迁移

Phase 4+（多域并行诊断）：引入 AutoGen 多 Agent 模式
  → 不同故障域由专门 Agent 并行诊断
  → LangGraph 作为顶层 Orchestrator
```

### 4.3 不推荐直接上 LangGraph 的具体原因（工程现实）

1. **SSE 适配**：现有 FastAPI SSE 通道需要适配 `astream_events()` 的事件格式（`on_llm_stream` vs 直接 token）
2. **会话模型迁移**：现有 `Conversation` 表需要适配 LangGraph `thread_id` + `Checkpoint` 模型
3. **GLM 边缘 case**：LangGraph 内部对 `tool_calls` 格式有假设，GLM 返回格式偶有差异，需要 patch
4. **调试复杂性**：图模型出错时调试路径比直接 Python 代码复杂，初期会影响迭代速度

如果从零开始（无现有代码约束）：**直接选 LangGraph**，以上问题从新项目开始时成本极低。

## 五、附录：框架选型决策树

```
开始评估框架
   ↓
是否有现有 FastAPI + SSE 代码需要保留？
  ├── 是 → 自实现 ReAct（对齐 LangGraph 概念）→ 视情况迁移 LangGraph
  └── 否 ↓
是否需要人工确认（操作不可逆）？
  ├── 否 → smolagents 或 自实现 Vanilla ReAct
  └── 是 ↓
是否需要跨会话状态持久化（中断后小时内恢复）？
  └── 是 → LangGraph + PostgresSaver  ← 直接选这个
是否是多 Agent 协作场景？
  └── 是 → LangGraph Multi-Agent 或 AutoGen
```
