# ops-agent 大脑可选：集成重设计方案

> 关联需求：[2026-04-28-ops-agent大脑可选集成重设计需求.md](../../../requirement/events/2026-04-28-ops-agent大脑可选集成重设计需求.md)  
> 分析基础：[大脑可选-集成重设计方案.md](../大脑可选-集成重设计方案.md)（含代码精读和对比分析）

## 一、方案概述（WHAT）

将 htp 的 `ConversationService.send_message_stream_only()` 内部拆分为两层：

```
【改造前】send_message_stream_only()
    └── 构建 system_prompt + 调用 AIAssistantRegistry（LLM 客户端）

【改造后】send_message_stream_only()
    ├── 会话管理层（永远执行：保存消息、读历史、预取环境数据、审计）
    └── BrainRouter.process()（委托给对应的大脑适配器）
          ├── assistant_type=openclaw/glm → HTPBrainAdapter（原有逻辑，不变）
          └── assistant_type=ops-agent  → OpsAgentBrainAdapter（新增）
```

**核心原则：htp 是"骨架手脚"的所有者，BrainRouter 是"大脑"的切换器。** 两个大脑彼此独立，互不耦合，通过 `BrainPort` Protocol 约束接口。

### 2.2 BrainPort：两仓库之间的代码内契约

```python
# htp 仓库：backend/conversation-service/app/core/brain_port.py
# 这是唯一的接口定义，ConversationService 只依赖此 Protocol

from __future__ import annotations
from typing import AsyncGenerator, Protocol, runtime_checkable
from dataclasses import dataclass

@dataclass
class BrainTextChunk:
    content: str                  # LLM 生成的文字片段（用于 SSE 输出）

@dataclass
class BrainStageUpdate:
    stage: str                    # 大脑内部阶段（仅用于前端展示，不控制 htp 状态机）
    label: str = ""               # 阶段的人类可读标签

@dataclass
class BrainEscalation:
    reason: str                   # 升级原因（无法解决、需要人工介入）

BrainEvent = BrainTextChunk | BrainStageUpdate | BrainEscalation


@runtime_checkable
class BrainPort(Protocol):
    """
    诊断大脑端口（Hexagonal Architecture Port）。
    所有大脑实现都必须符合此协议。
    ConversationService 只依赖此接口，对具体实现一无所知。
    """

    async def process(
        self,
        session_id: str,                        # htp conversation.id，跨轮次标识
        case_id: str,                           # 工单 ID
        user_message: str,                      # 用户最新消息
        history: list[dict],                    # 历史对话 [{"role":..., "content":...}]
        env_context: dict | None = None,        # htp 预取的实时环境数据
    ) -> AsyncGenerator[BrainEvent, None]:
        """
        处理一轮用户消息，以流式方式 yield BrainEvent 序列。
        实现者负责：维持多轮上下文、推理、返回结果。
        实现者不负责：消息持久化、审计日志（由 ConversationService 统一处理）。
        """
        ...
```

### 2.5 OpsAgentBrainAdapter：反腐层

```python
# htp 仓库：adapters/ops_agent_brain_adapter.py
# 反腐层（Anti-Corruption Layer）：所有 htp ↔ ops-agent 的概念翻译在这里完成
# htp 内部代码不知道 ops-agent 的任何实现细节

class OpsAgentBrainAdapter:
    """将 BrainPort 调用翻译为 ops-agent HTTP API 调用。"""

    def __init__(self, base_url: str, timeout: int = 300):
        self._base_url = base_url
        self._timeout = timeout

    async def process(
        self, session_id, case_id, user_message, history, env_context
    ) -> AsyncGenerator[BrainEvent, None]:
        payload = {
            "model": "ops-agent",
            "session_id": session_id,       # ops-agent 用于恢复会话上下文
            "messages": [{"role": "user", "content": user_message}],
            "hci_context": env_context,     # htp 预取的实时数据
            "stream": True,
        }

        try:
            async with httpx.AsyncClient(timeout=self._timeout) as client:
                async with client.stream(
                    "POST", f"{self._base_url}/v1/chat/completions", json=payload
                ) as resp:
                    async for line in resp.aiter_lines():
                        if not line.startswith("data: ") or line == "data: [DONE]":
                            continue
                        chunk = json.loads(line[6:])
                        delta = chunk["choices"][0]["delta"]
                        if content := delta.get("content"):
                            yield BrainTextChunk(content=content)
                        # ops-agent 自定义扩展字段（非标准 OpenAI 字段）
                        if stage := chunk.get("x_stage_update"):
                            yield BrainStageUpdate(stage=stage["current_stage"])
        except (httpx.ConnectError, httpx.TimeoutException) as e:
            # 服务不可达 → 降级信号（由 BrainRouter 捕获并切换到 HTPBrainAdapter）
            raise BrainUnavailableError(f"ops-agent 服务不可达: {e}") from e
```

### 2.6 BrainRouter：切换器 + 降级

```python
# htp 仓库：adapters/brain_router.py

class BrainRouter:
    """
    根据 assistant_type 路由到对应大脑。
    新增大脑：只需在 _brains 字典中增加一行。
    """

    def __init__(
        self,
        htp_brain: HTPBrainAdapter,
        ops_agent_brain: OpsAgentBrainAdapter,
    ):
        self._brains: dict[str, BrainPort] = {
            "openclaw": htp_brain,
            "glm":      htp_brain,
            "ops-agent": ops_agent_brain,
        }
        self._fallback = htp_brain  # 降级目标始终是 htp 大脑

    def get(self, assistant_type: str) -> BrainPort:
        return self._brains.get(assistant_type, self._fallback)

    async def process_with_fallback(
        self, assistant_type: str, **kwargs
    ) -> AsyncGenerator[BrainEvent, None]:
        """带自动降级的执行：ops-agent 不可达时无缝切换到 htp 大脑。"""
        brain = self.get(assistant_type)
        try:
            async for event in brain.process(**kwargs):
                yield event
        except BrainUnavailableError:
            logger.warning("ops-agent 不可达，降级到 htp 大脑", session_id=kwargs.get("session_id"))
            async for event in self._fallback.process(**kwargs):
                yield event
```

### 2.7 ConversationService 改造（最小化）

```python
# 改造后的 send_message_stream_only — 只剩会话管理骨架
async def send_message_stream_only(
    self, conversation_id: uuid.UUID, case_id: str, content: str,
    assistant_type: str | None = None
):
    # ── 会话管理层（永远执行）────────────────────────────────
    await self._save_message(conversation_id, role="user", content=content)
    history = await self._load_history(conversation_id)
    env_context = await self._prefetch_env_context(case_id)
    resolved_type = await self._resolve_assistant_type(conversation_id, assistant_type)
    trace_id = _current_trace_id()

    # ── 大脑执行层（委托，不感知内部实现）──────────────────
    full_reply_chunks: list[str] = []
    async for event in self.brain_router.process_with_fallback(
        assistant_type=resolved_type,
        session_id=str(conversation_id),
        case_id=case_id,
        user_message=content,
        history=history,
        env_context=env_context,
    ):
        if isinstance(event, BrainTextChunk):
            full_reply_chunks.append(event.content)
            yield f"data: {json.dumps({'content': event.content})}\n\n"
        elif isinstance(event, BrainStageUpdate):
            yield f"data: {json.dumps({'type': 'stage', 'stage': event.stage})}\n\n"
        elif isinstance(event, BrainEscalation):
            await self._handle_escalation(conversation_id, event.reason)

    # ── 收尾（永远执行）────────────────────────────────────
    full_reply = "".join(full_reply_chunks)
    await self._save_message(conversation_id, role="assistant", content=full_reply)
    await self._audit_log(conversation_id, resolved_type, trace_id)
```

#### 改造 C：Redis 会话持久化（Phase 2）

```python
# ops-agent 仓库：agent/session_pool.py（新增）

class AgentSessionPool:
    """
    维护 session_id → 会话状态 的映射。
    Phase 1：in-memory dict（进程重启丢失，可接受）
    Phase 2：Redis（TTL 4h，支持跨进程恢复）
    """
    def __init__(self, redis_client=None):
        self._redis = redis_client
        self._local: dict[str, dict] = {}  # 降级存储

    async def load(self, session_id: str) -> dict | None:
        if self._redis:
            raw = await self._redis.get(f"ops_agent:session:{session_id}")
            return json.loads(raw) if raw else None
        return self._local.get(session_id)

    async def save(self, session_id: str, state: dict):
        if self._redis:
            await self._redis.setex(
                f"ops_agent:session:{session_id}", 14400, json.dumps(state)
            )
        else:
            self._local[session_id] = state

## 三、决策依据（WHY）

### 3.1 方案选择

| 方案 | 描述 | 优点 | 缺点 | 评分 |
|------|------|------|------|------|
| **方案 A（选中）：Ports & Adapters** | BrainPort 接口，两个 Adapter 分别实现 | 两大脑完全解耦；新增大脑只加 Adapter；ConversationService 不感知实现细节 | 需要搬迁现有逻辑到 HTPBrainAdapter | ★★★★★ |
| 方案 B：在 ai_client.py 加分支 | 在 OpsAgentAssistant 里修复会话逻辑 | 改动最少 | 治标不治本；两套状态机仍然并行 | ★★☆☆☆ |
| 方案 C：Monorepo 合并 | htp 和 ops-agent 放到同一仓库 | 类型安全，无 HTTP 开销 | 破坏两个项目的独立性；ops-agent 是研究项目，不应与生产平台强耦合 | ★★☆☆☆ |
| 方案 D：共享 SDK 包 | 提取 brain-sdk PyPI 包 | 类型安全 | 过度工程；三个仓库的版本管理复杂度不可控 | ★★★☆☆ |

### 3.2 为什么选 Ports & Adapters（六边形架构）

**第一性原理推导**：
- `ConversationService` 需要调用"某种大脑"，但它不关心大脑内部如何工作
- 两个大脑在不同仓库、不同语义层次（多轮对话 vs 单次任务）
- 需要 A/B 切换，且将来可能加入第三个大脑
- 以上三条约束合并 → 必须有一个稳定的接口（Port），具体实现后置（Adapter）

这是 Ports & Adapters 的标准适用场景：**依赖方向指向内部，外部实现通过 Port 接入。**

### 3.3 为什么不用共享 Python 包

ops-agent 是**实验性研究项目**，htp 是**生产平台**。
- 如果用共享包，ops-agent 的每次重构都需要发版，htp 跟着升级依赖，研究迭代速度受限
- HTTP API 是最低耦合的集成方式，任何一侧只要保持接口兼容，可以完全独立演进
- 两年后如果 ops-agent 被 htp 完全内化，删除 OpsAgentBrainAdapter 即可，零残留

### 3.4 插入层次的选择

当前集成插在 `AIAssistantRegistry`（LLM 客户端层），重设计插在 `send_message_stream_only` 内部的"会话管理/大脑执行"分界线。

选择后者的原因：
- ops-agent 的"一次 agent.run()"语义 = htp 的"完整工单诊断"，不是"一次 LLM 调用"
- 插在 LLM 客户端层意味着 htp 的 system_prompt 构建、KB 检索、阶段检测全部执行然后被 ops-agent 丢弃 —— 纯浪费
- 插在分界线意味着 htp 只做公共服务（消息持久化、实时数据预取），大脑专注于推理

## 四、跨仓库代码管理方案

### 4.1 分界线与责任归属

```

  负责：实现 brain-http-api.yaml 定义的 HTTP 接口
  不知道：htp 内部的 S0-S6 状态机、PostgreSQL schema

契约文件（权威来源，存于 htp 仓库）
  hci-troubleshoot-platform/docs/contracts/brain-http-api.yaml

            current_stage:
              type: string

## 五、影响范围

### 改动范围的最终确认

| 文件 | 仓库 | 变化性质 |
|------|------|--------|
| `core/brain_port.py` | htp | 新增（约 30 行）BrainPort Protocol + BrainEvent 类型 |

| `adapters/htp_brain_adapter.py` | htp | 新增，逻辑从 conversation_service 搬家，不修改业务逻辑 |
| `adapters/ops_agent_brain_adapter.py` | htp | 新增，替换原来的 OpsAgentAssistant |
| `services/conversation_service.py` | htp | 改薄：send_message_stream_only 委托给 brain_router |
| `services/ai_client.py` | htp | **保留不变**，openclaw/GLM 仍由 HTPBrainAdapter 内部使用 |
| `docs/contracts/brain-http-api.yaml` | htp | 新增（跨仓库 HTTP 接口 OpenAPI 契约，权威来源） |

## 七、风险与缓解

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|---------|
| HTPBrainAdapter 搬家过程中引入回归 | 高 | 中 | 搬家前全量运行现有 E2E 测试，搬家后再次运行，diff 为零 |
| base_agent.py 真流式改造影响 CLI 模式 | 高 | 中 | 流式生成器封装在独立方法中，CLI 走原有 `execute_task()` 不变 |
| ops-agent 服务 OOM 或超时导致 htp 阻塞 | 高 | 低 | OpsAgentBrainAdapter 设置硬超时（300s），超时触发降级 |
| session_registry 内存无界增长 | 中 | 中 | Phase 1 加 LRU(1000) 上限；Phase 2 统一换 Redis TTL |
