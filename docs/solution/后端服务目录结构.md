## 三、各服务职责与目录差异原因

### 1. agent-service — ReAct 推理引擎

```
adapters/  ← 外部系统集成适配器（ACLI/HTP/PAI/OPS/SCP）
core/      ← 核心引擎（GLMClient/ReactExecutor/ToolRegistry）
routes/    ← API 端点
services/  ← 业务服务
```

**差异原因**：
- 这是 **AI 推理核心服务**，封装了 ReAct 循环、工具调用、LLM 对话
- `adapters/` — 适配多种外部 Agent（htp_agent、ops_agent、pai_agent 等）
- `core/` — 核心推理引擎
- **无 models/repositories** — 不直接操作数据库

---

### 2. api-gateway — 流量入口层

```
models/    ← 仅 terminal.py（WebSocket 会话模型）
routes/    ← 路由代理 + WebSocket + 健康检查
services/  ← 会话管理 + 终端代理
```

**差异原因**：
- 这是 **BFF（Backend for Frontend）**，只做路由转发和 WebSocket 管理
- **无 repositories** — 不直接访问数据库

---

### 3. case-service — 工单生命周期管理

```
models/       ← 工单、客户、环境、评估等实体
repositories/ ← 数据库访问层
routes/       ← 工单 CRUD API
services/     ← 业务逻辑 + 状态机
```

**差异原因**：
- 这是 **CRUD 为主的服务**，管理工单全生命周期
- `models/` + `repositories/` — 标准三层架构

---

### 4. conversation-service — 对话管理 + SSE 流式

```
models/       ← 会话、消息、诊断状态等实体
repositories/ ← 数据库访问层
routes/       ← 对话 API + SSE 流式
schemas/      ← API 请求/响应 DTO
services/     ← 对话管理 + AI 客户端 + 知识检索
```

**差异原因**：
- 这是 **最复杂的业务服务**，对话管理 + AI 推理编排 + SSE 流式
- `schemas/` — 对话 API 有复杂的请求/响应结构

---

### 5. eval-service — 评估统计服务

```
routes/    ← 评估 API
services/  ← 统计计算逻辑
```

**差异原因**：
- 这是 **轻量计算服务**，从已有数据计算评估指标
- **极简结构** — 功能单一

---

### 6. kb-service — 知识库管理

```
models/       ← 文档、分块、分类等实体
repositories/ ← 数据库访问层
routes/       ← 搜索、分类、导入等 API
schemas/      ← SOP 模板等 DTO
services/     ← 搜索引擎 + 向量化 + SOP 解析
utils/        ← jieba 分词 + 文本分割
config/       ← 分类基线配置
data/         ← SOP 知识库文件
```

**差异原因**：
- 这是 **数据处理密集型服务**，涉及文档解析、向量化、搜索

---

### 7. scheduler-service — Pod 热备池调度

```
routes/    ← 调度 API
services/  ← K8s 客户端 + Pod 池管理 + 调度算法
```

**差异原因**：
- 这是 **基础设施服务**，管理 K8s Pod 生命周期
- **无 models/repositories** — 状态存储在 Redis

---

## 四、差异的本质原因

| 差异维度 | 需要更多目录 | 需要更少目录 |
|---------|-------------|-------------|
| **数据持久化** | 有 PostgreSQL → `models/` + `repositories/` | 无数据库 → 只需 `services/` |
| **业务复杂度** | 复杂业务 → `schemas/` + `services/` 多文件 | 简单业务 → 极简结构 |
| **外部集成** | 多系统适配 → `adapters/` | 单一协议 → 无适配层 |
| **核心引擎** | AI 推理引擎 → `core/` | 业务逻辑 → `services/` |
| **数据处理** | NLP/向量 → `utils/` + `config/` + `data/` | 无处理 → 无工具层 |

---

## 五、设计原则

每个服务的目录结构反映其职责边界，遵循以下原则：

1. **不为未来设计** — 只创建当前需要的目录，不预留空目录
2. **职责驱动** — 目录边界与服务职责严格对应
3. **拒绝模板复制** — 不套用"标准微服务模板"，每个服务按实际需求定制
