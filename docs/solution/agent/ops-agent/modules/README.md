## 核心设计模式

### 主从 Agent 协作模式

- **OpsAgent**：负责任务 orchestration、用户交互、流程控制
- **SOPQuerySubAgent**：专注于 SOP 检索与路径收敛

这种设计的优势：
1. **职责分离**：主 Agent 处理流程，子 Agent 专注检索
2. **资源隔离**：子 Agent 有独立的预算、步数限制
3. **可扩展性**：可以轻松添加新的子 Agent 处理特定任务

### 工具驱动的工作流

Agent 通过工具调用来推进工作流：
- `ops_state_update`：维护内部状态
- `query_sop_candidates`：启动子 Agent 检索
- `get_sop_context`：读取 SOP 详细内容
- `present_sop_step_instruction`：展示操作指引
- `task_done`：输出总结并结束

### 显式状态管理

通过 `ops_state_update` 工具显式维护状态：
- 当前阶段、当前路径
- 已确认/排除/未知信号
- 候选路径、已排除路径
- 步骤尝试次数、风险标记

## 关键设计决策

### 1. 为什么选择主从 Agent 模式？

- SOP 检索是一个独立的、可封闭的子任务
- 需要独立的预算控制，防止无限检索
- 可以使用不同的模型配置

### 2. 为什么使用显式状态管理？

- LLM 在多轮对话中容易遗忘或矛盾
- 显式状态支持自检（重复信号、路径回流等）
- 便于审计和复盘

### 3. 为什么强制结构化输出？

- 确保子 Agent 输出可被父 Agent 解析
- 路径验证确保候选真实存在
- 标准化输出便于后续处理

### 4. 为什么使用预算控制？

- 防止 Agent 陷入无限检索循环
- 强制 Agent 在有限步数内产生结果
- 提升用户体验，避免长时间等待
