## 一、现状问题诊断

### 1.1 核心问题：category_baseline.yaml 是孤立的死文件

`backend/kb-service/config/category_baseline.yaml` 包含 **198 个 HCI 故障分类**，覆盖虚拟机/网络/存储/硬件/平台五个域，是整个系统的故障分类知识中枢。

**但当前状态：运行时完全未使用。**

```
category_baseline.yaml（198条）
           │
           ├── data-pipeline/etl/         ← 仅在 ETL 批处理中使用（离线）
           │
           └── 运行时 AI 交互             ← ❌ 完全未使用
                    │
                    ├── S0 意图识别       ← 没有分类列表注入
                    ├── SOP 路由          ← 关键字匹配，不用 category_id
                    └── KB 检索           ← category 过滤参数始终 None
```

### 1.2 九大差距（GAP 编号）

| GAP | 位置 | 问题描述 | 严重级别 |
|-----|------|---------|---------|
| GAP-1 | `knowledge_retriever.py` S0 prompt | S0 无 198 分类注入，LLM 靠自由联想猜测故障类型 | 🔴 严重 |
| GAP-2 | `knowledge_retriever.py` S0 阶段 | SOP 在 S0 注入（信息最模糊时），浪费 token 且干扰 | 🔴 严重 |
| GAP-3 | `data-pipeline/sops/` | SOP 覆盖率极低 | 🔴 高 |
| GAP-4 | `sop_matcher.py` | SOP 匹配 = 子字符串关键字扫描，无语义，用户措辞与 SOP 关键字 gap 是永久性的 | 🟠 中 |
| GAP-5 | `knowledge_retriever.py` | KB 检索在 S0 触发（query 最模糊时），召回质量最差 | 🔴 高 |
| GAP-6 | `kb_service.py` | KB 检索 `category` 参数始终传 `None`，分类过滤形同虚设 | 🟠 中 |
| GAP-7 | `knowledge_atoms.py` | 知识原子完整实现，调用方为零 | 🟡 低 |
| GAP-8 | `prompt_builder.py` | 完整的知识原子注入逻辑，调用方为零（死代码） | 🟡 低 |
| GAP-9 | `conversation_manager.py` | 阶段跳转用 regex 匹配 LLM 自然语言，极易失败 | 🟠 中 |

## 二、目标架构设计

### 2.1 正确的两阶段设计

```
┌─────────────────────────────────────────────────────────────────┐
│  S0：故障分类确认阶段                                            │
│                                                                 │
│  输入：用户描述 + 环境信息（env_info + alert_logs + task_logs）  │
│                                                                 │
│  System Prompt 注入：                                           │
│    • 角色定义：你是 HCI 故障分类专家                             │
│    • 198 个分类列表（从 kb_category 表读取，按 domain 分组）    │
│    • 输出格式约束：已确认故障分类：{code} {name}                 │
│    • ❌ 不注入 SOP（不知道分类就不知道用哪个 SOP）               │
│    • ❌ 不做 KB 检索（query 太模糊）                             │
│                                                                 │
│  S0 结束条件：LLM 输出含「已确认故障分类：」标记                 │
│  → 提取 category_id（对应 kb_category.code）                    │
│  → 写入 Conversation.category_id                                │
│  → 更新 kb_category.hit_count + 1                              │
└─────────────────────────────────────────────────────────────────┘
                            │ category_id 确定
                            ▼
┌─────────────────────────────────────────────────────────────────┐
│  S1+：精准诊断阶段（三轨串行路由）                                │
│                                                                 │
│  输入：category_id（已确定）                                     │
│                                                                 │
│  第1轨：SOP 优先                                                 │
│    SELECT FROM sop_document WHERE category_id = X               │
│    命中 → 加载 sop_chunk → SOP 步骤引导 → ✅ 返回                │
│    未命中 → 进入第2轨                                            │
│                                                                 │
│  第2轨：KBD 覆盖                                                 │
│    SELECT FROM kbd_entry WHERE category_id = X                  │
│    BM25(tsv) + 向量(embedding) → RRF → top5 条目                │
│    命中 → LLM RAG context → ✅ 返回                              │
│    未命中 → 进入第3轨                                            │
│                                                                 │
│  第3轨：人工兜底                                                 │
│    生成升级工单 + 标记 category_id → 转人工处理                  │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 使用远端 kb_category 表（方案B）

**设计依据**：远端 `21_知识库模块重设计.md` 方案B 已确定 `kb_category` 作为全局分类枢纽表。

**表结构（远端已实现 + 本地补充）**：

```sql
-- kb_category 表（远端已有 + 本地补充 2 个字段）
CREATE TABLE kb_category (
    id              SERIAL       PRIMARY KEY,
    parent_id       INTEGER      REFERENCES kb_category(id),  -- 树形结构
    name            VARCHAR(100) NOT NULL,                    -- 分类名称
    level           SMALLINT     NOT NULL,                    -- 1=L1, 2=L2, 3=L3, 4=L4
    keywords        TEXT[],                                    -- 触发关键字
    source          VARCHAR(20)  DEFAULT 'manual',
    version         VARCHAR(20)  DEFAULT '1.0',
    -- 远端新增字段（migration 20260401001）
    code            VARCHAR(32)  UNIQUE,                      -- 业务键，如 "虚拟机-001"
    domain          VARCHAR(50),                              -- 一级技术域
    path_labels     JSONB        DEFAULT '[]'::jsonb,         -- 完整路径数组
    embedding       vector(1536),                             -- 意图识别向量
    -- 本地补充字段（migration 20260402001）
    hit_count       INTEGER      DEFAULT 0,                   -- S0 命中次数统计
    is_active       BOOLEAN      DEFAULT TRUE,                -- 软删除标记
    created_at      TIMESTAMPTZ  DEFAULT CURRENT_TIMESTAMP
);

-- 索引
CREATE INDEX idx_kb_category_parent    ON kb_category(parent_id);
CREATE INDEX idx_kb_category_level     ON kb_category(level);
CREATE INDEX idx_kb_category_code      ON kb_category(code);
CREATE INDEX idx_kb_category_keywords  ON kb_category USING GIN (keywords);
CREATE INDEX idx_kb_category_is_active ON kb_category(is_active) WHERE is_active = TRUE;
```

**字段说明**：

| 字段 | 来源 | 用途 |
|------|------|------|
| `id` | 远端 | 整型主键，内部引用 |
| `parent_id` + `level` | 远端 | 4 层树形结构（L1:域 → L4:叶节点） |
| `name` | 远端 | 分类显示名称 |
| `code` | 远端 | **业务键**，格式 `虚拟机-001`，S0 确认输出值 |
| `domain` | 远端 | 一级技术域（虚拟机/网络/存储/硬件/平台） |
| `path_labels` | 远端 | 完整路径数组，如 `["虚拟机", "虚拟机创建"]` |
| `embedding` | 远端 | 分类节点语义向量，用于意图识别 topK 检索 |
| `keywords` | 远端 | 触发关键字数组 |
| `hit_count` | **本地新增** | S0 意图确认命中次数，分析热门/冷门分类 |
| `is_active` | **本地新增** | 软删除标记，FALSE 时不参与 S0 意图识别 |

### 2.3 SOP 路由设计（使用 sop_document.category_id FK）

**设计依据**：远端方案B 使用 `sop_document.category_id` 外键关联分类，无需冗余的 `sop_skill_id` 字段。

```
kb_category.code ←──────────────────────────────┐
    │                                            │
    ├── kbd_entry.category_id (FK)               │
    │                                            │
    └── sop_document.category_id (FK) ───────────┘
```

**S1+ 路由逻辑**：

```python
async def route_by_category(category_id: str, query: str) -> RouteResult:
    """三轨串行路由（S1+ 阶段）"""

    # 第1轨：SOP 优先
    sop = await sop_document_repo.get_by_category(category_id)
    if sop and sop.status == 'published':
        chunks = await sop_chunk_repo.search(sop.id, query)
        return RouteResult(track='sop', sop=sop, chunks=chunks)

    # 第2轨：KBD 覆盖
    kbd_entries = await kbd_entry_repo.search(category_id=category_id, query=query)
    if kbd_entries:
        return RouteResult(track='kbd', entries=kbd_entries)

    # 第3轨：人工兜底
    return RouteResult(track='human_escalation', category_id=category_id)
```

### 2.4 S0 LLM 意图确认的输出格式约束

**设计决策：路径乙 + 方式甲**

- **路径乙**：LLM 用自然语言回复用户（不用 JSON API），对话更自然
- **方式甲**：LLM 在自然语言中嵌入固定格式标记，后端 regex 提取 category_id

**LLM 输出示例：**
```
我理解您描述的情况。根据您提供的环境信息和告警日志，
结合症状特征，这应该属于虚拟机开机启动类故障。

已确认故障分类：虚拟机-003 虚拟机开机失败

接下来我将为您进行系统性诊断...
```

**后端提取 regex：**
```python
import re

CATEGORY_PATTERN = re.compile(r'已确认故障分类：([\u4e00-\u9fa5]+-\d+)')

def extract_category_id(full_reply: str) -> str | None:
    """从 LLM 完整回复中提取故障分类 ID（对应 kb_category.code）"""
    match = CATEGORY_PATTERN.search(full_reply)
    return match.group(1) if match else None
```

### 2.5 上下文信息（context_info）存储方案

用户环境信息（env_info + alert_logs + task_logs）需要在 S0 prompt 中注入，采用**双存储**避免跨服务 HTTP 调用：

```
Case.metadata["context_info"]         ← 工单归档（长期）
Conversation.metadata["context_info"] ← Prompt 构建时直接读取（高频）
```

**结构定义：**
```python
class ContextInfo(TypedDict):
    env_info: str        # 环境描述（集群版本、节点数量等）
    alert_logs: str      # 告警日志文本
    task_logs: str       # 任务日志文本
```

#### 10.1.2 CategoryService 内存缓存设计

```python
# backend/kb-service/app/services/category_service.py

import asyncio
from datetime import datetime, timezone
from typing import Optional
from backend.kb-service.app.models.kb_category import KbCategory
from backend.shared.db.session import get_async_session

class CategoryService:
    """分类业务服务：内存缓存 + TTL 刷新"""

    _cache: dict[str, KbCategory] = {}      # code → KbCategory 映射
    _cache_by_domain: dict[str, list[KbCategory]] = {}  # domain → 分类列表
    _last_refresh: datetime = None
    _ttl_seconds: int = 300  # 5 分钟 TTL
    _lock: asyncio.Lock = asyncio.Lock()

    async def get_all_active(self) -> list[KbCategory]:
        """获取全部启用的分类（按 domain 分组）"""
        await self._ensure_cache()
        return [c for c in self._cache.values() if c.is_active]

    async def get_by_code(self, code: str) -> Optional[KbCategory]:
        """按业务键查询分类"""
        await self._ensure_cache()
        return self._cache.get(code)

    async def get_by_domain(self, domain: str) -> list[KbCategory]:
        """按域获取分类列表（Prompt 分组用）"""
        await self._ensure_cache()
        return self._cache_by_domain.get(domain, [])

    async def increment_hit(self, code: str) -> None:
        """S0 确认后 hit_count +1（异步更新数据库）"""
        async with get_async_session() as session:
            category = await session.get(KbCategory, self._cache[code].id)
            if category:
                category.hit_count += 1
                await session.commit()
                # 同步更新缓存
                self._cache[code].hit_count = category.hit_count

    async def _ensure_cache(self) -> None:
        """确保缓存有效，过期则刷新"""
        async with self._lock:
            now = datetime.now(timezone.utc)
            if self._last_refresh is None or (now - self._last_refresh).total_seconds() > self._ttl_seconds:
                await self._refresh_cache()

    async def _refresh_cache(self) -> None:
        """从数据库刷新缓存"""
        async with get_async_session() as session:
            result = await session.execute(
                select(KbCategory).where(KbCategory.is_active == True).order_by(KbCategory.code)
            )
            categories = result.scalars().all()

            # 重建缓存
            self._cache = {c.code: c for c in categories}
            self._cache_by_domain = {}
            for c in categories:
                if c.domain not in self._cache_by_domain:
                    self._cache_by_domain[c.domain] = []
                self._cache_by_domain[c.domain].append(c)

            self._last_refresh = datetime.now(timezone.utc)
            logger.info(f"CategoryService 缓存刷新完成，共 {len(categories)} 条分类")

    async def invalidate_cache(self) -> None:
        """手动清除缓存（YAML 导入后触发）"""
        async with self._lock:
            self._cache.clear()
            self._cache_by_domain.clear()
            self._last_refresh = None
```

**缓存策略**：

| 参数 | 值 | 说明 |
|------|-----|------|
| TTL | 5 分钟 | 分类数据变更频率低，5 分钟刷新足够 |
| 锁 | `asyncio.Lock` | 防止并发刷新 |
| 触发时机 | 启动时 + TTL 过期 + YAML 导入后 | 保证数据一致性 |

        alert_logs = context_info.get("alert_logs", "（无告警）")

#### 10.2.3 category_id 正则提取逻辑

```python
# backend/conversation-service/app/services/conversation_manager.py

import re

# 正则模式详解
CATEGORY_PATTERN = re.compile(
    r'已确认故障分类：'      # 固定前缀标记
    r'([\u4e00-\u9fa5]+)'   # 域名（中文）：如 "虚拟机"
    r'-'
    r'(\d+)'                # 序号：如 "003"
    r'\s+'
    r'(.+?)'                # 分类名称（可选，用于日志）
)

def extract_category_id(full_reply: str) -> str | None:
    """
    从 LLM 完整回复中提取故障分类 ID（对应 kb_category.code）

    匹配示例：
      "已确认故障分类：虚拟机-003 虚拟机开机失败"
      → 返回 "虚拟机-003"

    返回 None 表示未匹配（S0 未完成）
    """
    match = CATEGORY_PATTERN.search(full_reply)
    if match:
        domain = match.group(1)   # "虚拟机"
        seq = match.group(2)      # "003"
        name = match.group(3)     # "虚拟机开机失败"（用于日志）
        code = f"{domain}-{seq}"
        logger.debug(f"提取 category_id: {code} ({name})")
        return code
    return None
```

from dataclasses import dataclass
from backend.conversation-service.app.clients.kb_client import KbClient

@dataclass
class RouteResult:
    """路由结果"""
    track: str           # "sop" / "kbd" / "human_escalation"
    sop: SopDocument | None = None
    sop_chunks: list[SopChunk] | None = None
    kbd_entries: list[KbdEntry] | None = None
    category_id: str | None = None

class KnowledgeRetriever:
    """知识检索：S1+ 三轨串行路由"""

    kb_client: KbClient

    async def route_by_category(self, category_id: str, query: str) -> RouteResult:
        """
        S1+ 三轨串行路由

        第1轨：SOP 优先（sop_document.category_id FK 查询）
        第2轨：KBD 覆盖（kbd_entry.category_id 过滤 + BM25/向量）
        第3轨：人工兜底
        """

        # ─── 第1轨：SOP 优先 ───
        sop = await self.kb_client.fetch_sop_by_category(category_id)
        if sop and sop.status == "published":
            chunks = await self.kb_client.search_sop_chunks(sop.id, query)
            logger.info(f"S1+ 路由 → SOP 轨道，category={category_id}, sop={sop.id}")
            return RouteResult(track="sop", sop=sop, sop_chunks=chunks)

        # ─── 第2轨：KBD 覆盖 ───
        kbd_entries = await self.kb_client.search_kbd_entries(category_id=category_id, query=query)
        if kbd_entries:
            logger.info(f"S1+ 路由 → KBD 轨道，category={category_id}, entries={len(kbd_entries)}")
            return RouteResult(track="kbd", kbd_entries=kbd_entries)

        # ─── 第3轨：人工兜底 ───
        logger.warning(f"S1+ 路由 → 人工轨道，category={category_id} 无 SOP/KBD 覆盖")
        return RouteResult(track="human_escalation", category_id=category_id)

    async def build_s1_prompt(self, conversation: Conversation, route_result: RouteResult) -> str:
        """构建 S1+ 阶段 System Prompt（注入 SOP/KBD 知识）"""

### 13.2 Prompt 构建 Python 代码

```python
# backend/conversation-service/app/services/prompt_builder.py

async def build_s0_prompt(self, context_info: ContextInfo) -> str:
    """构建 S0 阶段 System Prompt"""

    segments = [
        self._build_role_segment(),
        self._build_env_segment(context_info),
        await self._build_category_segment(),
        self._build_format_segment(),
    ]

    return "\n\n".join(segments)

def _build_role_segment(self) -> str:
    """Segment 1: 角色定义"""
    return """你是 HCI 云平台故障诊断专家，负责识别并确认故障分类。
你的首要任务是根据用户描述和环境信息，从给定的 198 个故障分类中选择最匹配的一个。
请务必遵循输出格式约束。"""

def _build_env_segment(self, context_info: ContextInfo) -> str:
    """Segment 2: 环境上下文"""
    return f"""## 当前环境信息
{context_info.get('env_info', '（用户未提供）')}

## 最新告警
{context_info.get('alert_logs', '（无告警）')}

## 近期任务日志
{context_info.get('task_logs', '（无任务日志）')}"""

async def _build_category_segment(self) -> str:
    """Segment 3: 198 分类列表"""
    categories = await self.kb_client.fetch_categories()

    lines = ["## 故障分类（共 198 个，请从中选择最匹配的 1 个）"]
    for domain, items in sorted(categories.items()):
        lines.append(f"\n### {domain}域（{len(items)}个）")
        for c in items:
            lines.append(f"- {c['code']} {c['name']}")

    return "\n".join(lines)

def _build_format_segment(self) -> str:
    """Segment 4: 输出格式约束"""
    return """## 输出要求
1. 先用自然语言向用户解释你的判断依据（1-2句）
2. 如需澄清，向用户提问（不超过 1 个问题）
3. 当你有足够信息确定分类时，**必须**在回复末尾输出：
   「已确认故障分类：{分类code} {分类name}」
   例：已确认故障分类：虚拟机-003 虚拟机开机失败
4. 在确认分类之前，不注入 SOP，不做诊断推理"""
```

## 十五、S0 交互模式深度审视与设计决策

> **背景**：本节记录 2026-04-05 对 S0 意图识别交互方式的第一性原理审视，以及基于此做出的设计决策。

---

### 15.1 第一性原理：S0 的最终目标是什么？

**S0 的唯一目标：得到一个 `kb_category.code`（如"虚拟机-003"）。**

这个 code 的全部用途：

| 用途 | 说明 |
|------|------|
| S1+ SOP 路由 | `sop_document WHERE category_id = X` — 结构化 SOP 引导，防 LLM 幻觉 |
| S1+ KBD 检索 | `kbd_entry WHERE category_id = X` — 精准召回，避免跨域噪声 |
| 统计热度 | `kb_category.hit_count + 1` — 运营分析热门/冷门分类 |
| 状态同步 | 触发 `case.status: created → confirmed`（两状态机第一个同步点） |

**结论：S0 本质是一个分类任务（Classification），不是一个对话任务（Conversation）。**

---

### 15.2 现行"提问式"方案的三个问题

现行 `STAGE_METHODOLOGY["S0"]` 的核心指令是：

```
3. 提出 1-3 个精准确认问题，不要一次问超过 3 个
4. 参考分类基准，尝试初步定位到技术域
```

用第一性原理逐项审视：

#### 问题一：LLM 的追问不可控

S0 已经注入了三个工具的环境快照（alerts / failed_tasks / vm_list）。LLM 看到"提出 1-3 个问题"后，可能问出：

- "请问虚拟机是什么时候无法启动的？" → 告警日志里已经有时间戳，重复收集
- "您是否尝试过重启宿主机？" → 这是 S3 才该执行的操作，S0 不应触及
- "贵公司环境有多少个节点？" → 与分类无关，对缩小候选范围毫无帮助

**现行方案无法保证追问是"锁定 category_id 所需的最少信息"。**

#### 问题二：追问本质上是低效的

面对 198 个分类候选，LLM 需要通过追问来"排除"候选。每个开放性问题：
- 用户需要思考如何回答（认知负担高）
- 回答的质量和措辞不确定（影响 LLM 下一轮判断）
- 多轮追问后 S0 仍可能无法收敛（超时风险）

**对比：如果 LLM 直接列出 top-3 候选，用户只需要"选一个"。选择的认知负担远低于回答。**

#### 问题三：收敛性无法保证

"不超过 3 个问题"是数量限制，不是质量保证。3 轮追问后 LLM 仍不确定时，可能行为：
- 输出一个低置信度的分类（硬猜）→ S1+ 路由走错轨道，浪费整个诊断流程
- 继续追问 → 用户体验差，超时

---

### 15.3 新设计：两路分支策略

**核心原则：LLM 做分类推理，用户做最终确认；追问改为候选选择。**

```
用户描述 + 工具环境快照（alerts / failed_tasks / vm_list）
                          │
              LLM 内部推理：候选列表 + 置信度
                          │
          ┌───────────────┴───────────────┐
          │                               │
    单一候选，置信度高               多个候选，置信度接近
          │                               │
    直接确认（无需提问）           给用户展示 top-3 候选
    「已确认故障分类：X」          （附带判断说明 + 置信度依据）
          │                               │
      → 跳 S1                    用户选一个 → 一轮确认 → 跳 S1
```

**置信度判断标准（Prompt 约束）**：

| 置信度 | 条件 | 行为 |
|--------|------|------|
| 高（≥80%）| 单一候选明显领先 | 直接输出「已确认故障分类：{code} {name}」 |
| 中（存在 2-3 个相近候选）| 候选之间差异不明显 | 展示 top-3，附判断说明，让用户选择 |
| 低（无法确定，与 S0 失败兜底相关）| 信息严重不足 | 见 15.4 |

---

### 15.4 设计决策一：候选确认模式（UX）

**决策：给候选让用户确认，是当前最优 UX 方案。**

**理由（用户确认）**：
> "给出选项，并备注判断说明（简单说明置信度及依据）。选择这个动作永远比回答这个动作更简单。"

**具体 LLM 输出样式（中置信度时）**：

```
根据您描述的症状（虚拟机无法启动）和环境告警（存储 IO 超时告警），
我初步判断可能属于以下几种故障之一，请确认：

① 虚拟机-003 虚拟机开机失败
   判断依据：LOG-ERROR: "VM start failed: storage volume not accessible"，
   概率估计 ~60%

② 存储-021 存储卷IO异常  
   判断依据：告警中存在 ASAN IO timeout，可能是开机失败的根因
   概率估计 ~30%

③ 以上都不是（请描述更多信息）

请回复序号 ①②③ 进行确认。
```

**后端解析**：增加对 `①②③` 序号选择的正则识别（补充现有的标记提取逻辑）。

### 15.5 设计决策二：S0 彻底失败的兜底方案

**决策：S0 无法完成意图识别时，`case.status → in_progress`，移交人工处理。**

**理由（用户确认）**：
> "S0 彻底失败意味着 AI 无法胜任相关工作，快速由人接管是保障问题高效闭环的兜底方案。"

**触发条件**（满足任一）：
- 多轮追问后（超过 N 轮，建议 N=2）仍未确认分类
- 用户明确选择"③ 以上都不是"，且再次展示候选后仍无法确认
- 超时（会话创建后超过 T 分钟仍在 S0，建议 T=10）

**兜底行为**：

```python
# S0 失败兜底流程
async def handle_s0_failure(conversation_id, case_id):
    # 1. case.status: created/confirmed → in_progress（人工处理）
    await case_service.update_status(case_id, "in_progress", 
                                     close_reason="s0_classification_failed")
    
    # 2. conversation.diagnostic_stage 标记为失败
    await update_conversation(conversation_id, 
                              diagnostic_stage="S0_FAILED",
                              metadata={"s0_failure_reason": "classification_timeout"})
    
    # 3. 推送提示给用户
    yield "抱歉，当前 AI 无法确认故障类型，已为您转接人工工程师。\n" \
          "工单编号：{case_id}，工程师将在 30 分钟内联系您。"
```

**状态机影响**：

```
S0 正常路径：  case.status: created → confirmed（分类成功）→ in_progress（诊断中）
S0 失败路径：  case.status: created → in_progress（直接跳 confirmed，人工接管）
```

**注意**：失败路径下 `case.status` 跳过 `confirmed` 直接进 `in_progress`，需要在 case-service 侧允许这个跳转（添加 `created → in_progress` 的合法跳转规则）。
