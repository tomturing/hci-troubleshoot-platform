# ops-agent 集成失效：根因分析与方案全景对比

> 触发背景：hci-troubleshoot-platform 集成 ops-agent 后，选择 Ops-Agent 助手发送消息时，系统返回  
> `[系统提示] ops-agent 暂时不可用，已自动切换到备用助手继续为您服务。`  
> 关联方案E拆解：[2026-05-08-ops-agent方案E-ACP-REST接口设计与实现.md](./2026-05-08-ops-agent方案E-ACP-REST接口设计与实现.md)

---

## 第一章：问题本质

### 1. 问题本质（两个维度，不可混淆）

**维度1（运维层，已修复）**：截图3的降级提示是 `BrainUnavailableError`，是 ops-agent-service Pod 反复 OOMKilled（Exit Code 137）导致不可达。根因：`memory.limits: 512Mi` 过低，Agent 在首次推理请求时峰值内存 >512Mi 被 K8s 强制终止。**已修复（2026-05-08）**：`hci-platform-env/environments/dev/values.yaml` 将内存 limit 调整为 2Gi，CPU limit 调整为 2000m，commit `d389052` 已推送，ArgoCD 自动同步。

**维度2（架构层，根本缺陷）**：ops-agent 被设计为"交互式长程推理引擎"，其核心诊断价值来自 `present_sop_step_instruction` 和 `get_info_from_user` 的双向闭环。当前 HTTP 模式（方案C变体）用 [auto_approve=True](vscode-file://vscode-app/c:/Users/tomtu/scoop/apps/vscode/1.114.0/e7fb5e96c0/resources/app/out/vs/code/electron-browser/workbench/workbench.html) + 无 console 绕过了所有交互，本质是把 ops-agent 降级成了"单向文本生成器"，SOP 引导的核心价值丧失。

### 2. 方案对比核心判断

| 方案             |     可行      | 交互完整 | 架构干净 | 复用现有 ACP 代码 |
| -------------- | :---------: | :--: | :--: | :---------: |
| A WebSocket    |      ✅      |  ✅   |  ✅   |     部分      |
| B 轮询持久化        | ❌ (协程不可序列化) |  —   |  —   |      —      |
| C 当前实现         |      ✅      |  ❌   |  ✅   |      ❌      |
| D 嵌入式          |      ✅      |  ✅   |  ❌   |      ✅      |
| **E ACP REST** |      ✅      |  ✅   |  ✅   |  **✅ 最大**   |

### 3. 方案E核心设计

最小改造量实现最完整功能：**保持 `acp/` 目录完全不变**，在 `server/` 新增 HTTP 包装层。`session_prompt()` 在 `threading.Thread` 中运行（与 Streamlit 相同模式），通过 `asyncio.to_thread()` 轮询同步 `drain_outbox()` 推送 SSE 事件流。conversation-service 作为 ACP 客户端，将 `_ops/request_input` 事件翻译为新的 `BrainInteractiveRequest` 类型，通过 SSE 透传给前端。

### 1.1 失效路径还原（代码级）

截图3的失效路径精确如下：

```
用户选择 Ops-Agent 助手 → conversation-service 路由到 OpsAgentBrainAdapter
  → OpsAgentBrainAdapter.process() → POST /v1/chat/completions → ops-agent-service
    → [连接失败 / 超时 / 非200] → raise BrainUnavailableError
      → BrainRouter._route_ops_agent() 捕获异常
        → yield BrainTextChunk("[系统提示] ops-agent 暂时不可用...")
          → 降级到 HTPBrainAdapter（openclaw 大脑）
```

触发 `BrainUnavailableError` 的条件是三选一（`brain_router.py`）：

| 触发条件 | 对应场景 |
|---------|---------|
| `httpx.ConnectError` | ops-agent-service Pod 未运行 / 网络不通 |
| `httpx.TimeoutException` | LLM 响应慢，180s read timeout 超时 |
| HTTP 状态码 ≠ 200 | Pod 启动失败 / 配置错误 / OOM 后崩溃 |

**运维层面最可能的根因（按概率排序）：**

1. **SCP_API_KEY secret 未创建**：`hci-secrets.SCP_API_KEY` 不存在 → Pod 启动时 `envFrom.secretKeyRef` 失败 → Pod CrashLoop → 永远未就绪
2. **SOP 数据 HostPath 未准备**：`/aihci/ops-agent/data/case_sop_data/hci/sop` 不存在 → Agent 初始化失败 → 500
3. **内存限制 OOM**：`resources.limits.memory: 512Mi` → Python + FastAPI + LLM Client 运行时超限 → OOMKill → CrashLoop
4. **API Key 未注入 ops_config.yaml**：`OPENAI_COMPATIBLE_API_KEY` 环境变量未被 ops_config.yaml 读取 → LLM 调用 401 → Agent 报错

**排查命令：**
```bash
kubectl get pod -l app.kubernetes.io/name=ops-agent-service -n hci-platform
kubectl describe pod <pod-name> -n hci-platform   # 看 Events：OOMKilled / ErrImagePull / CreateContainerError
kubectl logs <pod-name> -n hci-platform --tail=50 # 看启动错误
```

### 1.2 架构层面的深层缺陷：执行模型的阻抗失配

即使运维问题被修复，ops-agent 在 HTTP 模式下仍存在**架构性的功能缺失**。

**ops-agent 的原生心智模型：**

ops-agent 从第一天起被设计为**交互式长程推理引擎**。它有两个核心交互工具：

- `present_sop_step_instruction`：呈现 SOP 操作卡片，**等待用户执行后反馈结果** → Agent 依据反馈决定下一步
- `get_info_from_user`：向用户提问以澄清故障信息，**等待用户选择选项** → Agent 依据选择收窄诊断路径

这两个工具是 ops-agent 诊断质量的核心保障：故障排查 ≠ 信息查询，必须存在"操作→反馈→推进"的双向闭环。

**交互机制的技术实现（Streamlit 模型，可行）：**

```
┌─────────────────────── Streamlit 进程 ──────────────────────────┐
│                                                                  │
│  UI Thread (主线程)              ACPServer (Agent 执行线程)     │
│  ─────────────────               ───────────────────────────    │
│  drain_outbox()                  ...                             │
│    ← 发现 request_input 事件      bridge.request_client()        │
│  渲染交互 UI 组件                    pending.event.wait()  ←阻塞 │
│  用户点选选项                        ...                         │
│  submit_response(选项)           ...                             │
│    ──────────────────────────────► pending.event.set()           │
│                                  response = pending.response     │
│                                  Agent 继续执行下一步            │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
关键前提：Agent 和 UI 在同一进程，共享 ACPSession 内存对象
```

**HTTP + SSE 模型的根本矛盾（不可行的原因）：**

```
┌────────────────────────────────────────────────────────────────────┐
│  ops-agent-service 进程                                            │
│                                                                    │
│  FastAPI Handler                  Agent 执行                       │
│  ─────────────────                ──────────────────────────────── │
│  POST /v1/chat/completions        execute_task_streaming()         │
│  → SSE 流已建立并返回              ...                              │
│  → HTTP 响应体开始流出             [需要用户输入时]                 │
│  → 客户端开始读取 SSE              ← 无人调用 submit_response()!   │
│                                   request_client() 永久阻塞       │
│                                   或：无 console → 静默跳过        │
└────────────────────────────────────────────────────────────────────┘

问题1：HTTP 是半双工（请求期间客户端不能发新请求）
问题2：ops-agent-service 进程与 conversation-service 是独立进程
问题3：conversation-service 无法跨进程回调 submit_response()
```

**现行代码（方案C变体）的实际行为：**

`openai_compat.py` 创建的 Agent **没有 console**，`auto_approve=True`。当 Agent 调用交互工具时：

- `get_info_from_user.execute()`：`self._cli_console` 为 None → 执行路径 fallback → 工具返回空结果或占位文本 → Agent 拿到空反馈继续推进 → **诊断质量严重下降**
- `present_sop_step_instruction.execute()`：通过 `_tool_display_hook` 将卡片文本推入 SSE 流 → 用户看到 SOP 卡片 → 但 Agent 不等反馈就继续 → **操作确认闭环断裂**

**问题精确定义：**

> ops-agent 的价值在于"交互式、有据可查的 SOP 引导"，而当前 HTTP 集成模式将其降级为"单向文本生成器"。  
> 运维问题修复后，用户能看到回复，但 ops-agent 的核心能力（SOP 双向闭环）无法工作。

### 1.3 问题的完整维度地图

```
ops-agent 集成问题
├── 第一层：运维问题（当前失效的直接原因）
│   ├── Pod 启动失败（Secret/Config 问题）
│   ├── OOM Crash
│   └── 排查：kubectl describe/logs → 立即可修复
│
└── 第二层：架构问题（修复运维后暴露的设计缺陷）
    ├── 无交互工具反馈闭环
    │   ├── present_sop_step_instruction → 单向输出，无用户确认
    │   └── get_info_from_user → 问题发出，无用户回答
    └── 根本原因：执行模型阻抗失配
        ├── ops-agent 原生模型：同进程阻塞式交互
        └── HTTP 模式：跨进程单向 SSE 流
```

## 第二章：方案全景对比

基于第一章的问题定义，评估所有可行方案。评估维度：交互完整性、改造成本、架构侵入性、运维复杂度、复用现有代码。

---

### 方案 A：WebSocket 双向通道

**架构：** ops-agent-service 新增 WebSocket 端点；conversation-service 升级为 WebSocket 客户端；前端通道不变（现有 SSE）。

```
前端 ←── SSE ──── conversation-service ──── WebSocket ────► ops-agent-service
                         ↑                                        │
                    用户消息注入                               Agent 执行
                    (新增 WS 消息处理)                   request_input 事件
                                                         ────────────────►
                                                         conversation-service
                                                         ←────────────────
                                                         submit_response()
```

**优点：**
- 真正的双向实时通信
- 延迟最低（无轮询间隔）
- 成熟协议，FastAPI/HTTPX 均支持

**缺点：**
- K8s Ingress 需配置 WebSocket 支持（`nginx.ingress.kubernetes.io/proxy-read-timeout: "3600"`）
- conversation-service 需维护 WS 连接生命周期（连接断开重连、heartbeat）
- ops-agent-service 并发连接管理复杂（一个 Agent 会话 = 一个 WS 连接）
- 前端到 conversation-service 的通道不变，但 conversation-service 内部逻辑需要大幅重写（从 HTTP 客户端改为 WS 客户端）
- `ACPClientBridge.request_client()` 仍是 `threading.Event` 阻塞，与 asyncio WS 混合困难

**交互完整性：** ✅ 完整  
**改造成本：** 高（约 15 天）  
**复用 ACP 基础设施：** 部分复用（outbox/submit 逻辑可用，但传输层需重写）  
**推荐度：** ⚠️ 不推荐（复杂度高，不必要）

---

### 方案 B：轮询 + Agent 状态持久化

**架构：** Agent 执行到交互节点时"挂起"到外部存储，用户回复后"恢复"执行。

```
POST /v1/chat/completions
  → Agent 运行到 request_input
  → 将 Agent 执行上下文序列化到 Redis
  → 返回 HTTP 200 + {suspended: true, request_id: xxx}

前端轮询 GET /v1/sessions/{id}/pending
  → 发现 pending request_input 事件 → 展示交互 UI

POST /v1/sessions/{id}/responses
  → 从 Redis 反序列化 Agent 执行上下文
  → 将用户响应注入 → 继续执行
```

**优点：**
- 无长连接，对网关友好
- 天然支持分布式（Agent 可在不同实例间恢复）

**缺点（致命）：**
- **Python asyncio coroutine 不可序列化**：`execute_task_streaming()` 本质是 asyncio `Task`，内部包含局部变量、awaitable 栈帧等，无法 pickle/persist
- 要实现真正的"挂起恢复"，需要把 Agent 执行引擎改写为**显式状态机**（类似 Temporal Workflow），重写代价极高
- 即使改写为状态机，每轮 LRU 历史恢复重放的计算成本也很高
- 与 ops-agent 的 ReAct 循环设计理念冲突

**交互完整性：** ✅ 完整（如果实现）  
**改造成本：** 极高（需重写 Agent 执行引擎，约 30 天+）  
**可行性：** ❌ 在现有代码基础上基本不可行  
**推荐度：** ❌ 不推荐

---

### 方案 C：Agent 无用户交互（当前实现）

**架构：** 现行方案。Agent 以 `auto_approve=True` 运行，无 console，`execute_task_streaming()` 直接输出文本。

**当前代码行为：**
```python
# openai_compat.py
agent = Agent(
    agent_type="ops_agent",
    config=self.config,
    auto_approve_mode="all",   # 工具自动批准
    debug_llm=self.debug_llm,  # 无 console
)
async for text_chunk in agent.agent.execute_task_streaming():
    yield text_chunk
```

**工具执行情况：**

| 工具 | 无 console 时行为 | 对诊断的影响 |
|-----|-----------------|------------|
| `get_sop_context` | ✅ 正常（知识查询，无交互） | 无影响 |
| `ops_state_update` | ✅ 正常（状态写入，无交互） | 无影响 |
| `present_sop_step_instruction` | ⚠️ 通过 `_tool_display_hook` 输出卡片文本，不等反馈 | Agent 在无确认下推进，诊断路径可能错误 |
| `get_info_from_user` | ⚠️ 输出问题文本，返回空响应 | Agent 拿到空答案继续，故障分类不准确 |

**优点：**
- 已实现，零额外改造
- 对纯信息查询类问题（不需要用户交互）效果尚可

**缺点：**
- SOP 引导的核心价值（双向闭环）完全丢失
- Agent 在无用户确认的情况下推进危险操作
- 实际上是"伪装成 ops-agent 的 LLM 问答"

**交互完整性：** ❌ 无交互（核心功能缺失）  
**改造成本：** 零  
**推荐度：** ⚠️ 仅作为临时方案，解决运维问题后即需升级

---

### 方案 D：嵌入式 Agent（conversation-service 内运行）

**架构：** 将 ops-agent 作为 Python 包直接引入 conversation-service，在同进程内运行，通过内存共享对象实现交互。

```
conversation-service 进程
├── ConversationService（原有）
├── BrainRouter
│   └── OpsAgentBrainAdapter
│       └── 直接 import ops_agent 包
│           └── ACPServer（in-process）
│               └── Agent 执行（background thread）
│                   ↕ threading.Event（同进程，可行）
└── 前端交互通过 SSE 传递 request_input 事件
    → 前端响应后通过新 POST 端点 submit
      → conversation-service （同进程）→ ACPSession.submit_response()
```

**优点：**
- 交互完全可行（同进程共享内存）
- 复用现有 `ACPServer`/`ACPClientBridge`/`ACPConsole` 全部逻辑

**缺点：**
- **破坏微服务架构边界**：conversation-service 与 ops-agent 强耦合，版本需同步
- **依赖冗余**：conversation-service 需引入 ops-agent 的全部依赖（yaml 配置、SOP 数据目录、LLM client 等）
- **扩展性差**：ops-agent 是 CPU/内存密集型，混入 conversation-service 会干扰其 I/O 密集型服务的扩缩容策略
- **独立部署/维护能力丧失**：ops-agent 无法独立升级、回滚、重启

**交互完整性：** ✅ 完整  
**改造成本：** 中等（约 5-8 天）  
**推荐度：** ❌ 不推荐（牺牲架构质量换交互完整性，方向错误）

### 方案 E：ACP REST 接口（最优解）

**核心思路：** ops-agent-service 将已有 `ACPServer` 的内部接口（`session_new`、`drain_outbox`、`submit_client_response`）**升级为异步 HTTP REST 端点**，conversation-service 作为 **ACP 协议客户端**完成交互代理。

```
前端 ←── SSE ──── conversation-service
                    │
                    ├─ POST /acp/sessions         → 创建会话
                    ├─ POST /acp/sessions/{id}/prompt  → 启动 Agent（非阻塞）
                    ├─ GET  /acp/sessions/{id}/events  → SSE 事件流（文本+交互请求）
                    ├─ POST /acp/sessions/{id}/responses/{req_id}  → 提交用户响应
                    └─ GET  /acp/sessions/{id}/state   → 查询 Agent 状态
                                  ↕
                         ops-agent-service
                         （ACPServer 异步化）
                                  ↕
                         Agent 执行（background asyncio.Task）
                         ACPConsole → bridge.request_client()
                         → asyncio.Event.wait()  ← 等待 responses 接口
```

**优点：**
- ✅ 完整交互（present_sop_step_instruction + get_info_from_user 全部工作）
- ✅ 不破坏微服务边界（两服务独立）
- ✅ **大量复用现有 opts-agent ACP 代码**（`ACPServer`/`ACPClientBridge`/`ACPConsole`/`ACPSession` 均可复用）
- ✅ 无 WebSocket，HTTP-only，对 K8s Ingress 友好
- ✅ 前端无需修改（交互由 conversation-service 代理，通过现有 SSE 透传）
- ✅ 用户响应路径：前端 POST → conversation-service → ops-agent-service（简洁）

**缺点：**
- 需改造 `ACPServer` 中的线程模型：`threading.Event` → `asyncio.Event`，`queue.Queue` → `asyncio.Queue`
- ops-agent-service 需新增 5 个 REST 端点
- conversation-service `OpsAgentBrainAdapter` 需扩展为 ACP 协议客户端
- 前端需要新增 `confirm_request` 类事件处理（类似现有的人工确认机制 `ConfirmService`）

**交互完整性：** ✅ 完整  
**改造成本：** 中等（约 8-12 天）  
**推荐度：** ✅ **推荐，最优解**
