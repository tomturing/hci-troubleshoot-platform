## 2. 三大支柱详解

### 2.1 链路追踪 (Distributed Tracing)

**技术选型**: OpenTelemetry Python SDK + Grafana Tempo

**核心原理**: 
- 每个微服务在启动时调用 `init_telemetry(service_name)` 完成自动仪表化。
- OTel 的 **FastAPIInstrumentor** 会自动为每个入站 HTTP 请求创建 Span，并从请求 Header 中解析 W3C `traceparent` 实现上下文传播。
- OTel 的 **HTTPXClientInstrumentor** 会自动为每个出站的 HTTPX 代理请求注入 `traceparent` Header，确保下游服务能继承同一条 Trace。
- OTel 的 **SQLAlchemyInstrumentor** 会自动为每一次数据库查询生成子 Span，记录 SQL 语句和耗时。

**数据模型**:
```
Trace (一次完整的用户请求)
├── Span: api-gateway POST /api/cases/         [120ms, OK]
│   └── Span: httpx → case-service             [115ms, OK]
│       ├── Span: case-service 路由处理         [80ms, OK]
│       │   └── Span: sqlalchemy INSERT case    [45ms, OK]
│       └── Span: pydantic 序列化               [2ms, OK]
└── (W3C traceparent 自动注入到每一跳的 HTTP Header)
```

**传播协议**: W3C Trace Context（`traceparent` Header），取代此前自定义的 `X-Trace-ID`。

**采集流转**:
1. 微服务 OTel SDK → BatchSpanProcessor → OTLPSpanExporter
2. → Tempo gRPC 端口 (4317) 接收
3. → Tempo 本地存储索引
4. → Grafana Tempo 数据源 → 瀑布图可视化

### 2.2 日志 (Centralized Logging)

**技术选型**: JSON stdout + Promtail + Grafana Loki

**核心原理**:
- 遵循 12-Factor App 原则，所有微服务只向标准输出 (stdout) 打印 JSON 格式的结构化日志。
- **关键字段**: 每条日志自动注入 OTel 上下文中的 `trace_id` 和 `span_id`，实现与 Trace 的精确关联。
- Promtail 以只读方式挂载 Docker 守护进程的 `/var/lib/docker/containers`，实时采集所有容器日志并推送至 Loki。

**日志格式示例**:
```json
{
  "timestamp": "2026-02-25T08:30:00Z",
  "level": "ERROR",
  "service": "conversation-service",
  "event": "openclaw_call_failed",
  "message": "AI模型调用失败",
  "trace_id": "abc123def456789...",
  "span_id": "0123456789abcdef",
  "error_type": "ConnectionRefusedError",
  "error_message": "[Errno 111] Connection refused"
}
```

**Trace ↔ Log 关联钻取**:
- 在 Grafana 的 Tempo 瀑布图中，点击任意 Span → 自动跳转到 Loki 查看该时间窗内带有相同 `trace_id` 的详细日志。
- 在 Loki 日志面板中，点击一条包含 `trace_id` 的日志 → 自动跳转到 Tempo 查看完整的调用链路瀑布图。

### 2.3 指标 (Metrics) — 已实现

**技术选型**: Prometheus（通过 `prometheus_client` 库暴露）

**实现状态**:
- ✅ 所有五个微服务均已暴露 `GET /metrics`（Prometheus 文本格式）：
  `api-gateway`、`case-service`、`conversation-service`、`scheduler-service`、`kb-service`
- 监控维度：请求速率、错误率、延迟百分位（RED 方法）
- Grafana 面板：自定义业务大盘

## 3. 部署指南 (Docker Compose)

### 3.1 目录结构
```
deploy/observability/
├── config/
│   ├── promtail-config.yaml    # Promtail 采集配置
│   └── tempo-config.yaml       # Tempo OTLP 接收配置
├── grafana/
│   └── provisioning/
│       └── datasources/
│           └── datasources.yml  # Loki + Tempo 数据源自动配置
└── docker-compose-obs.yml      # 可观测性全栈编排
```

### 3.2 一键部署
```bash
# 确保业务微服务已先行启动（docker compose up -d）
cd deploy/observability
docker compose -f docker-compose-obs.yml up -d
```

### 3.3 Grafana 数据源自动配置 (Provisioning)

Grafana 启动时会自动加载 `grafana/provisioning/datasources/datasources.yml`，无需手动添加数据源。

**自动配置内容**:
- **Loki** 数据源（uid: `loki`）→ `http://loki:3100`，设为默认
- **Tempo** 数据源（uid: `tempo`）→ `http://tempo:3200`
- **Trace ↔ Log 双向钻取**：Tempo 配置 `tracesToLogsV2.datasourceUid=loki`，点击 Span 即可跳转到 Loki 查看关联日志

启动后打开 `http://localhost:3000`，使用 `admin` / `admin` 登入，数据源已就绪。

### 3.4 验证全链路打通

1. **触发测试流量**:
   ```bash
   bash scripts/docker-e2e-test.sh
   ```

2. **在 Tempo 查看瀑布图**:
   - Grafana 左侧 → Explore → 选择 **Tempo**
   - Search → Service Name 选 `api-gateway` → Run query
   - 点击任意一条 Trace → 查看完整的跨服务调用瀑布图

3. **从 Trace 跳转到 Log**:
   - 在瀑布图中点击某个 Span
   - 点击 "Logs for this span" → 自动跳转到 Loki 显示该请求的详细日志

4. **在 Loki 用 TraceID 反查**:
   - Explore → 选择 **Loki**
   - 切换到 **Code** 模式，输入:
     ```logql
     {container=~"hci-.*"} | json | trace_id="从Tempo复制的TraceID"
     ```

## 4. 代码层实现说明

### 4.1 OTel 初始化模块 (`shared/utils/otel.py`)

所有微服务共用的初始化函数，调用即自动完成以下仪表化:

| 仪表化目标 | 效果 |
|-----------|------|
| FastAPI | 为每个入站请求自动创建 Span，记录路径、方法、状态码 |
| HTTPX | 为网关的出站代理请求自动传播 traceparent Header |
| SQLAlchemy | 为每次 DB 查询自动生成子 Span，记录 SQL 和耗时 |
| Logging | 自动向 Python logging 注入 trace_id 和 span_id |

### 4.2 结构化日志 (`shared/utils/logger.py`)

`StructuredLogger._format_log()` 在生成 JSON 日志时，优先从 OTel 当前 Span 上下文中读取 `trace_id` 和 `span_id`，确保每条日志都能与 Trace 精确匹配。

### 4.3 环境变量

| 变量名 | 默认值 | 描述 |
|--------|--------|------|
| `OTEL_EXPORTER_OTLP_ENDPOINT` | `http://tempo:4317` | Tempo 的 OTLP gRPC 接收端点 |

## 5. 排障实战场景示例

### 场景：用户发起对话后返回错误

**第一步 — 秒级定位**:
打开 Grafana → Tempo → 搜索近 5 分钟内状态为 ERROR 的 Trace:
```
瀑布图立刻展示:
┌─ api-gateway  POST /conversations/xxx/message  ── 3200ms ✅
│  └─ conversation-service /message               ── 3150ms ❌ ERROR
│     ├─ sqlalchemy SELECT conversation           ──   12ms ✅
│     └─ openclaw_client.chat()                   ── 3000ms ❌ ERROR
```

**第二步 — 深入查因**:
点击红色的 `openclaw_client.chat()` Span，查看异常详情:
- Status: ERROR
- Exception: `ConnectionRefusedError`

**第三步 — 关联日志**:
点击 "Logs for this span" → 跳转到 Loki:
```json
{"level":"ERROR", "service":"conversation-service",
 "event":"openclaw_call_failed",
 "error_message":"[Errno 111] Connection refused",
 "trace_id":"abc123...", "span_id":"def456..."}
```

**结论**: AI 模型服务不可达 → 检查 OpenClaw Pod 状态。

## 6. 2026-04-16 全面修复说明

### 6.1 持久化（#3 #8）
- **Tempo** 存储路径从 `/tmp/tempo/blocks` 改为 `/var/tempo/blocks`，新增 named volume `tempo_data`
- **Loki** 新增 named volume `loki_data`，重启后日志不再丢失

### 6.2 Trace→Log 联动（#4）
`promtail-config.yaml` 添加 `pipeline_stages`，从 JSON 日志中提取 `trace_id`/`level`/`service` 作为 Loki 标签。
`filterByTraceID: true` 现在可精确过滤，点击 Span 跳转 Loki 只显示当前请求的日志。

### 6.3 X-Trace-Id 与 OTel 对齐（#5）
`TraceIDMiddleware` 改为从 `otel_trace.get_current_span()` 读取真实 32 位十六进制 Trace ID 写入 `X-Trace-Id` 响应头。用户可直接从 F12 拿到值粘贴至 Tempo 搜索框。

### 6.4 HTTP 指标中间件（#2）
`shared/utils/metrics.py` 新增 `HTTP_REQUEST_DURATION_SECONDS` (Histogram) 和 `HTTP_REQUESTS_TOTAL` (Counter)，`HTTPMetricsMiddleware` 已注入全部五个服务。`HighApiLatency` 和 `HighErrorRate` 告警现为真实有效规则。

### 6.5 Pod 池告警修复（#1）
- `alert_rules.yml` 的 `WarmPoolExhausted`/`WarmPoolLow`/`PodPoolExhausted` 从 `hci_warm_pool_size{state=...}` 改用正确指标 `hci_pod_pool_idle`/`hci_pod_pool_active`
- `scheduler-service/pod_pool.py` 在 `acquire_pod`/`release_pod` 后调用 `_update_metrics()`，真实上报指标

### 6.6 KB 检索指标（#6）
`kb-service/app/services/search_engine.py` 在 `search()` 方法末尾调用 `KB_SEARCH_DURATION_SECONDS.observe()`。

### 6.7 Tempo metrics_generator 落地（#10）
`tempo-config.yaml` 添加 `storage.remote_write` 指向 Prometheus，Prometheus 启动参数添加 `--web.enable-remote-write-receiver`，service-graph 和 span-metrics 现在真实写入 Prometheus。

### 6.8 serviceMap 数据源修复（#9）
`datasources.yml` 的 `serviceMap.datasourceUid` 从 `loki` 改为 `prometheus`，服务拓扑图可正常渲染。

### 6.9 Grafana 安全加固（#11）
关闭匿名访问，admin 密码改用环境变量 `GF_ADMIN_USER`/`GF_ADMIN_PASSWORD`，通过 `deploy/env/obs.env` 注入（示例：`obs.env.example`）。

### 6.10 Dashboard 代码化（#14）
`grafana/provisioning/dashboards/` 新增 `dashboards.yml`（provider）和 `hci-overview.json`（HCI 综合大盘：QPS/错误率/P95延迟/TTFT/Pod池/KB检索耗时）。

### 6.11 StructuredLogger.exception()（#13）
`shared/utils/logger.py` 新增 `exception()` 方法，在 `except` 块内调用时自动捕获 `traceback.format_exc()` 并写入日志 `traceback` 字段。

### 6.12 Grafana Trace-to-Logs 跳转修复 & 镜像版本固定
- `deploy/helm/hci-platform-obs/templates/configs.yaml`：`tracesToLogsV2.tags` 从 `["pod","namespace","app"]`（span 中不存在的 key）改为 `[{key: service.name, value: service_name}]`，使 Grafana 能从 span 的 resource attribute `service.name` 提取值并匹配 Loki label `service_name`，Tempo → Loki 跳转按钮由此生效。
- `deploy/helm/hci-platform-obs/values.yaml`：Grafana/Loki/Promtail 镜像从 `latest` 固定到稳定版（`grafana:11.5.2`、`loki:3.4.2`、`promtail:3.4.2`），避免拉到预发布版本（如 13.0.0-pre）导致 Loki 插件 JS 崩溃。
