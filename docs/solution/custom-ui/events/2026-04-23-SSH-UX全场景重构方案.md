## 背景与需求

PR #203 完成了 SSH 集成体验重构（SshFlowPanel/SshConnectDialog/upsert 机制），但用户验证后发现 3 个问题：

1. **问题1 — 状态不刷新**：SSH 采集完成后，页面仍显示"未采集环境数据"（`environmentSummary` 未重新拉取）
2. **问题2 — 双弹框**：CaseCreateDialog 精简时移除了 SSH 表单，导致点击"连接SSH并创建工单"会弹出第二个弹框（SshFlowPanel），体验割裂
3. **问题3 — 三态按钮缺失**：TerminalPanel 的按钮未区分「无工单无SSH」/「有工单无SSH」/「已连接」三种状态

用户在讨论中进一步明确了期望的完整交互设计（详见本文档 §方案详细设计）。

## 方案概述 (WHAT)

### 核心设计原则

1. **Bridge 检测前置**：弹框弹出前完成检测，检测结果决定弹框初始视图
2. **弹框内状态机**：CaseCreateDialog 包含多个视图（Bridge 引导 / SSH 表单 / 进度 / 错误 / 成功），所有状态在弹框1内完成，不跳出新弹框
3. **三态按钮**：TerminalPanel 根据业务状态显示不同按钮和行为
4. **失败始终兜底**：所有错误视图都提供"重试"和"无SSH直接创建"两个选项

### 两个触发入口（同等优先级）

```
触发入口 A：
  用户点击顶部 [SSH终端] → 右侧终端抽屉打开
  → 用户在抽屉内点击 [连接SSH并创建工单]（状态A时）
  → 前置 Bridge 检测（弹框弹出前）

触发入口 B：
  用户直接在底部输入框输入问题并发送
  → 检测到无当前工单
  → 前置 Bridge 检测（弹框弹出前）
  → 弹出 CaseCreateDialog

两个入口都经过同一个前置 Bridge 检测逻辑，检测结果决定弹框初始视图。
```

## 详细设计

### 前置 Bridge 检测（弹框弹出前）

```
用户触发动作（任意入口）
       │
  短暂 loading（~500ms）
       │
  WebSocket 连接 ws://localhost:9999（3s 超时）
       │
  ┌────┴────┐
未运行      运行正常
  │             │
弹框显示      弹框显示
视图A         视图B
```

**实现说明**：
- 检测复用 `chatStore.checkBridgeStatus()` 现有逻辑
- 检测期间按钮显示 loading 状态，不可点击
- 检测完成后立即弹框，不需要用户额外操作

### 业务主流程与逻辑约束

```
主流程（SSH认证成功）：
  输入SSH信息 → 认证成功 → 自动采集环境信息 → 创建工单
  → upsert 环境数据入库（使用工单号作为 case_id）
  → 弹框关闭 → 进入AI对话 → 刷新 environmentSummary

失败流程（SSH认证失败）：
  输入SSH信息 → 认证失败 → [重试：回到表单重走认证 | 无SSH创建：跳过直接创建]
  → 进入AI对话

失败流程（采集失败，SSH已连通）：
  认证成功 → 采集失败 → [重试采集：保留SSH连接只重走采集 | 无SSH创建：直接创建]
  → 进入AI对话
```

**强制约束**：
- SSH 认证成功后**必须执行采集**，两步强绑定，不允许认证成功但跳过采集
- 采集成功才创建工单（保证 upsert 时 `case_id` 已存在）
- 弹框关闭后（无论成功还是无SSH创建）必须调用 `fetchEnvironments(caseId)` 刷新前端 `environmentSummary` 显示

### TerminalPanel 三态

```
状态A：无工单 + 无SSH
  [🖥 连接SSH并创建工单]
  → 触发前置 Bridge 检测
  → 弹出 CaseCreateDialog（根据检测结果显示视图A或视图B）

状态B：有工单 + 无SSH
  [🖥 连接SSH]
  → 触发前置 Bridge 检测
  → 弹出 SshConnectDialog（同样根据检测结果显示对应视图，无工单创建表单）

状态C：已SSH连接
  → 直接显示 xterm 终端（无按钮，直接可操作）
  root@192.168.1.100:~# █
```

**三态判断逻辑**（`checkAndOpenTerminal()` 实现）：

```typescript
function checkAndOpenTerminal() {
  if (!currentCaseId.value) {
    // 状态A：无工单
    // 1. 前置 Bridge 检测
    // 2. 打开 CaseCreateDialog，根据检测结果设置初始视图
  } else if (sshConnectionState.value !== 'connected') {
    // 状态B：有工单但无SSH
    // 1. 前置 Bridge 检测
    // 2. 打开 SshConnectDialog
  }
  // 状态C：已连接，TerminalPanel 已显示终端，无需操作
}
```

## 数据流（完整主流程）

```
用户点击 [连接SSH并创建工单]
  │
  ├─ checkBridgeStatus()
  │       │ 失败 → viewState = 'bridge-guide'（视图A）
  │       │ 成功 → viewState = 'form'（视图B）
  │
  用户填写 SSH 信息 → 点击 [连接SSH并创建工单]
  │
  ├─ viewState = 'progress'
  ├─ step 1: connectSSH(host, port, username, password)
  │       │ 失败 → viewState = 'error-auth'（视图C）
  │       │ 成功 →
  ├─ step 2: checkAcli() + collectEnvironment()（强绑定）
  │       │ 失败 → viewState = 'error-collect'（视图D）
  │       │ 成功 →
  ├─ step 3: createCase(title, description, assistantType)
  │       │ 成功 → environmentApi.upsert(caseId, envType, envData)
  │       │ 成功 → viewState = 'success'（视图F）
  │       │ 2秒后 → closeDialog() → navigateToChat(caseId)
  │                → fetchEnvironments(caseId)（刷新状态）
```

## 决策依据 (WHY)

### 方案对比

| 方案 | 说明 | 优点 | 缺点 |
|------|------|------|------|
| **方案A（选中）：弹框前检测，弹框内状态机** | Bridge 检测在弹框弹出前完成，弹框内视图根据检测结果初始化 | 用户体验流畅，无二次弹框；错误信息集中展示；Bridge 引导和 SSH 表单共用弹框1空间 | 弹框逻辑稍复杂（多视图切换） |
| 方案B：弹框内检测，弹框弹出后再检测 | 弹框弹出后才检测 Bridge | 实现简单 | 用户打开弹框后才看到 Bridge 未运行，体验割裂 |
| 方案C：保持双弹框（现有 PR #203 实现）| 弹框1（表单）→ 弹框2（SshFlowPanel） | 组件职责清晰 | 弹框套弹框体验差，用户反馈问题2 |

**选择方案A的理由**：
1. 用户明确要求"连接过程保留在弹框1内，不跳新弹框"
2. Bridge 检测前置让弹框展示的内容是确定的，无内容跳变
3. 视图状态机是成熟模式，Vue 3 `<component :is>` 或 `v-if` 实现都很直接

### 为什么仅支持 Windows 下载 Bridge？

当前 terminal_bridge 工具仅有 Windows 构建版本（`build_windows.bat`），macOS/Linux 版本尚未开发。提供不存在的下载链接会误导用户，因此仅展示 Windows 下载。

### 为什么步骤条是 3 步而不是 4 步？

Bridge 检测在弹框弹出前完成，不属于弹框内的交互流程，因此弹框内步骤条只显示用户可感知的 3 个操作步骤：SSH 认证 / 采集环境 / 创建工单。
