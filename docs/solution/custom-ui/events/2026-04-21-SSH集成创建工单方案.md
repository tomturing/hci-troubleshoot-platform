## 背景与需求

参见需求文档 [SSH集成创建工单需求.md](../../requirement/events/2026-04-21-SSH集成创建工单需求.md)

核心问题：PR181 实现了 Environment API 后端，但前端缺少 UI 入口触发采集。

---

## 方案概述 (WHAT)

将 SSH 连接流程集成到创建工单对话框中，实现：

1. **弹框弹出时 Bridge 检测**：自动检测 terminal_bridge.exe 是否运行
2. **SSH 信息输入**：主机、端口、用户名、密码（自动填充上次成功的信息）
3. **一键连接创建**：连接 SSH → 检查 acli → 采集环境 → 终端打开 → 展示结果
4. **无 SSH 创建**：灰色按钮，悬浮告警，跳过 SSH 流程

## 决策依据 (WHY)

### 方案选择

| 方案 | 优点 | 缺点 | 评分 |
|------|------|------|------|
| **A：集成到创建工单弹框**（选中） | 流程连贯，一次操作完成所有步骤；用户体验最佳 | 弹框复杂度增加 | ★★★★☆ |
| B：保持独立 SSH 终端面板 | 改动小，现有组件不变 | 流程割裂，用户需要多次点击 | ★★☆☆☆ |
| C：工单创建后弹 SSH 配置框 | 分步操作，逻辑清晰 | 两次弹框，体验不连贯 | ★★★☆☆ |

### 为什么选择方案 A？

1. **第一性原理**：用户的核心需求是「快速开始排障」，一次操作完成所有前置步骤是最高效的
2. **业界最佳实践**：VS Code Copilot 连接工作区时，一次性完成所有上下文采集，不需要用户多次确认
3. **降低操作成本**：当前用户需要：点击新建 → 填写信息 → 点击终端按钮 → 填写 SSH → 连接 → 手动触发采集。新方案减少到：点击新建 → 填写信息 → 一键连接创建

### 为什么不选方案 B？

方案 B（保持独立）无法解决「流程割裂」的痛点，用户仍需多次点击，且缺少自动触发采集的逻辑入口。

### 权衡与妥协

1. **仅支持密码认证**：密钥认证 UI 复杂度高，暂不支持，后续迭代添加
2. **密码存储在 localStorage**：生产环境需加密方案，当前 MVP 优先实现功能

### upsert 设计（业界最佳实践）

**接口**：`PUT /api/environments/case/{case_id}/type/{env_type}`

**RESTful 语义**：
- 幂等（Idempotent）：多次调用结果相同
- 首次调用 → 创建记录（HTTP 201）
- 后续调用 → 更新 env_data（HTTP 200）

**后端实现**（check-then-write）：
```python
async def upsert_by_case_and_type(case_id, env_type, env_data, collected_at):
    existing = await get_by_case_and_type(case_id, env_type)
    if existing:
        existing.env_data = env_data
        existing.collected_at = collected_at
        await session.flush()
        return existing
    else:
        return await create(Environment(...))
```

**前端调用**：
```typescript
// environmentApi.upsert 替代原 environmentApi.create
await environmentApi.upsert(caseId, 'cluster', clusterData)
await environmentApi.upsert(caseId, 'alert', alertData)
await environmentApi.upsert(caseId, 'task', taskData)
```

### SshFlowPanel 状态机

```
idle
  ↓ (mount) checkBridge
checking-bridge
  ├── bridge not running → bridge-not-running (显示下载提示，连接按钮禁用)
  └── bridge running → ready (显示 SSH 表单)

ready
  ↓ (用户点击"连接")
connecting  (步骤1)
  ├── ssh_connected → acli-check (步骤2)
  └── 超时/错误 → error

acli-check  (步骤2)
  ├── acli 可用 → collecting (步骤3)
  └── acli 不可用 → acli-not-found → done（跳过采集）

collecting  (步骤3)
  ├── 三条命令执行完成 → upsert → done (步骤4)
  └── 错误 → error（仍继续，记录警告）

done
  ↓ disconnect 临时 socket → connectSSH(全局) → emit 'success'

error
  ↓ 显示错误信息 + 检查项 + "重试"按钮
  → 点击重试 → 回到 ready
```

### SshFlowPanel marker 命令协议

marker 协议用于从持续输出的 SSH WebSocket 中精确提取每条命令的结果：

```
发送：command; status=$?; printf '\n__HCI_DONE_{caseId}_{name}_{ts}__:%s\n' "$status"
接收：累积输出直到看到 marker，然后解析 exitCode
```

每条命令有独立的 timeout（`cluster: 30s, alert: 60s, task: 60s`）。
