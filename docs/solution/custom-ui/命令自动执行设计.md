## 1. 需求背景与原始动机

### 1.1 当前瓶颈

现有的命令执行链路有**两个手动授权点**：

1. CommandBlock 上的「发送到终端」按钮（填充输入框）
2. 终端内用户手动按回车（确认执行）

在实际排障场景中，AI 给出多条诊断命令时，工程师需要反复执行"看命令 → 点发送 → 按回车 → 看输出 → 切回对话 → 贴结果"的循环。这个流程的摩擦成本在连续多步诊断时尤为明显。

### 1.2 需求本质

> **降低人机协作的摩擦，让 AI 能"动手"收集诊断信息，工程师专注在判断和决策层面。**

这与 VS Code Copilot Agent 模式的演进方向一致：AI 提出工具调用（命令） → 自动执行 → 结果回填上下文 → AI 继续推理。

### 1.3 与 Copilot/Claude Code 的本质差异

HCI 场景的特殊性在于：**命令执行的对端是客户的生产环境**，而非本地沙盒。

| 工具 | 执行环境 | 最坏情况 |
|------|---------|---------|
| VS Code Copilot | 本地文件系统 / 本地终端 | 本地文件损坏 |
| Claude Code | 本地 shell / 沙盒 | 本地代码库损坏 |
| **HCI 客户端** | **客户 HCI 生产集群（via SSH）** | **客户业务中断** |

这一差异是所有设计决策（风险矩阵、串行执行、审计日志）的出发点。

## 2. 第一性原理分析

### 2.1 谁在获益，谁在承担风险？

| 角色 | 获益 | 承担风险 |
|------|------|---------|
| 工程师 | 省去重复操作，专注分析 | 命令被自动执行，可能触发非预期副作用 |
| 客户 | 排障效率提升，故障恢复更快 | 生产环境被自动操控（最坏情况）|
| HCI 平台 | 差异化能力 | 如有生产事故，可能对产品信任度造成影响 |

**结论**：风险不对等——工程师省时，客户承担生产风险。因此必须提供足够细粒度的控制权，让工程师能精确划定"哪类命令可以自动执行"。

### 2.2 命令本质上分几类？

从运维角度，SSH 对端的命令只有两种本质：
- **只读（read）**：查询状态，不改变任何系统状态，可安全重复执行
- **写入（write）**：改变系统状态，必须幂等或不可逆，一旦执行无法撤销

基于这一分类，HCI 的 `riskLevel` 定义合理：
- `none` / `readonly`：纯只读，完全安全
- `caution`：有副作用但通常低风险（如 `--dry-run`、状态重置）
- `danger`：不可逆写操作（迁移、重启、扩容、删除）

### 2.3 设计约束的推导

从以上分析直接推导出三条硬约束：

1. **`danger` 级命令永远需要手动确认**，无论开关处于什么模式
2. **自动执行开关的默认值必须是 `off`**，需要工程师主动开启
3. **自动执行开关仅在 SSH 已连接时可见**，不连接时无意义也不应展示

## 3. 设计决策记录

### 3.1 Q1：执行结果如何处理？

**决策：B — 自动作为用户消息注入对话，触发 AI 继续分析**

**理由**：
- 自动执行的核心价值在于"闭环"：命令 → 执行 → 结果回 AI → AI 继续推理
- 如果结果只显示在终端而不进入对话上下文，AI 看不到输出，自动执行的诊断效率提升近乎为零
- 与 Copilot Agent `tool_result` 机制完全一致，是行业验证的做法

**注入格式**（追加为 `role: user` 消息）：
```
[命令执行结果] #<序号>
$ <执行的命令>
退出码: <exit_code>

<stdout/stderr 输出>
（超过 200 行时截断，完整输出见终端历史）
```

### 3.2 Q2：多命令执行顺序——串行还是并行？

**决策：严格串行，等待结果后再触发 AI，AI 决定是否继续**

**业界最佳实践梳理**：

| 工具 | 执行策略 | 原理 |
|------|---------|------|
| Claude Code | 严格串行 | `tool_use` / `tool_result` 严格交替，每轮只执行一个工具调用 |
| Copilot Agent | 严格串行 | 单个 `run_in_terminal` 完成后才进入下一个工具调用循环 |
| GitHub Actions | 显式并行声明（`needs`）| 依赖关系需要人工显式标注，默认独立 job 并行 |
| Cursor Composer | 支持批量执行 | 一次性 apply 多个文件修改，但每批等待确认 |

**结论**：Claude Code 和 Copilot Agent 都选择严格串行，核心原因：

1. **因果依赖**：诊断命令通常有依赖链。例如：`acli task list` → 发现 task_id=4412 → `acli task get 4412` → 发现 vm_name=vm-01 → `acli vm get vm-01`。后一条命令的参数依赖前一条的输出，并行根本无法实现这类动态参数链。

2. **资源竞争**：SSH 连接是单路复用通道，并行命令的输出流无法可靠区分归属，marker 协议在并行场景下容易产生竞态。

3. **认知负载**：串行产生清晰的因果链，工程师和 AI 都能追踪"为什么要执行这条命令"。

**执行模型**：

```
AI 回复（含命令块 ①②③）
    │
    ├──→ 命令① 自动执行 → 结果追加对话
    │         AI 收到结果后决定：继续执行 or 改变方向 or 给出结论
    │         AI 可能在结果消息后输出命令②（或者不输出）
    │
    └──→（不是一次性批量执行所有命令，而是 AI 主导节奏）
```

**一次 AI 回复包含多条命令块的处理规则**：

- 只自动执行**第一条**命令块，结果注入对话后等待 AI 的下一轮回复
- 第二条及以后的命令块降级为手动模式（显示正常的"发送到终端"按钮）
- 原因：多命令块通常是 AI 给出"备选方案"，而非严格的执行序列；让 AI 看到第一条结果后自行决定是否需要后续命令

### 3.3 Q3：`caution` 级命令在 `safe-only` 模式下如何处理？

**决策：A — 拦截，需要手动确认（更保守）**

**理由**：
- HCI 对端是客户生产环境，`caution` 级操作（如 `acli storage snapshot delete --force`）即使"通常安全"，在边界条件下仍可能造成数据丢失
- 保守优先：工程师明确授权比系统假设"通常安全"更健壮
- 未来可以放宽（`safe-only` 改为允许 caution），但无法回收已经执行的生产操作

## 4. 核心设计

### 4.1 自动执行模式状态机

```
autoExecuteMode: 'off' | 'safe-only' | 'aggressive'
```

| 模式 | 描述 | 适用场景 |
|------|------|---------|
| `off` | 关闭自动执行，回到手动模式 | 默认，高度谨慎操作时 |
| `safe-only` | 仅 `none`/`readonly` 级命令自动执行 | **推荐日常使用** |
| `aggressive` | `none`/`readonly`/`caution` 自动执行，`danger` 拦截 | 排障紧急阶段，工程师已充分了解风险 |

**硬规则**（无论任何模式）：`danger` 级命令一律弹出强制确认对话框，不可绕过。

### 4.2 命令执行决策矩阵

| riskLevel | `off` | `safe-only` | `aggressive` |
|-----------|-------|-------------|--------------|
| `none` / `readonly` | 手动 | **自动执行** | **自动执行** |
| `caution` | 手动 | 拦截（手动确认） | **自动执行** |
| `danger` | 手动 | 强制确认弹框 | 强制确认弹框 |

### 4.3 完整执行流程

```
AI 回复中出现命令块（第一条）
    │
    ├─ [autoExecuteMode === 'off'] ──→ 正常 UI 展示，等工程师手动操作
    │
    └─ [autoExecuteMode 非 off]
           │
           ├─ 前提条件检查
           │    ├─ SSH 已连接？  否 ──→ 降级为手动，Tooltip 提示"请先连接 SSH"
           │    ├─ 串行锁空闲？  否 ──→ 排队等待（最多等 30s，超时降级为手动）
           │    └─ 工单有效？    否 ──→ 停止，提示"工单已关闭"
           │
           ├─ 风险级别检查（见 4.2 矩阵）
           │    ├─ 通过 ──→ 进入自动执行链
           │    └─ 拦截 ──→ 显示确认对话框，等工程师授权
           │
           ├─ 🔒 获取串行锁（isExecutingCommand = true）
           │
           ├─ 构建 marker：buildBridgeMarker(caseId, cmd, index)
           ├─ 发送命令：sshWebSocket.send(buildBridgeCommandPayload(cmd, marker))
           ├─ CommandBlock UI 切换为「执行中」状态（loading + spinner）
           │
           ├─ 等待 marker 出现（parseBridgeCommandResult 轮询 sshOutputBuffer）
           │    └─ 超时（30s）──→ 提示超时，释放串行锁，降级为手动
           │
           ├─ 命令完成
           │    ├─ CommandBlock UI → ✓ exit=0 / ✗ exit=N
           │    ├─ 输出截断（> 200 行 → 截断 + 注释）
           │    ├─ 追加 role:user 消息（格式见 3.1）
           │    └─ 调用 streamAIResponse 触发 AI 继续分析
           │
           └─ 🔓 释放串行锁（isExecutingCommand = false）
```

### 4.4 开关 UI 位置

开关放在 **TerminalPanel.vue 头部状态栏**，理由：
1. 语义关联：自动执行依赖 SSH 连接，与终端面板语义一致
2. 可见性：仅当终端面板打开时可见，避免工程师不看终端时误触
3. 状态关联：开关旁显示 SSH 连接状态，提供完整上下文

```
┌─ 终端 ─────────────────────────────────────────────────────────┐
│  root@192.168.1.100  ● 已连接    自动执行 [关闭 ▼]             │
│                                                                │
│  (xterm.js 终端内容)                                           │
└────────────────────────────────────────────────────────────────┘
```

开关下拉选项：
- `关闭`（默认，灰色）
- `安全模式`（仅 none/readonly 自动执行，绿色）
- `激进模式`（none/readonly/caution 自动执行，橙色 + ⚠️ 警告标注）

### 4.5 CommandBlock 状态变化

```
当前状态：
  [复制] [发送到终端]

新增状态（SSH 已连接 + autoExecuteMode 非 off）：
  等待执行：[▶ 将自动执行]（灰色倒计时，最多 3s 延迟，工程师可点 × 中断）
  执行中：  [⏳ 执行中...]（loading spinner，不可点击）
  成功：    [✓ exit=0]（绿色）
  失败：    [✗ exit=N]（红色，N 为退出码）
```

**3 秒延迟**的设计意图：在真正发出命令前给工程师 3 秒视觉感知时间，即使开了自动执行，工程师仍能在倒计时结束前点 × 中断单条命令。这是 Claude Code 的"软取消窗口"思路。

### 4.6 状态管理新增字段（chat.ts 变更）

```typescript
// 自动执行模式（持久化到 localStorage，key: 'hci_auto_execute_mode'）
const autoExecuteMode = ref<'off' | 'safe-only' | 'aggressive'>('off')

// 串行执行锁
const isExecutingCommand = ref(false)

// 当前正在执行的命令信息（供 UI 展示）
const executingCommand = ref<{
  command: string
  startedAt: Date
  blockId: string   // 对应 ContentSegment.id
} | null>(null)

// 自动执行计数（防止 agent loop，当前会话内计数）
const autoExecCount = ref(0)
const AUTO_EXEC_MAX = 10  // 单次 AI 会话内最多自动执行 10 条命令

// 页面可见性（blur 时暂停自动执行）
const pageVisible = ref(true)
```

## 5. 潜在问题详述与解决方案

### PIT-A：CommandBlock 缺少 AI 意图的 riskLevel

**问题描述**

当前 `CommandBlock` 的 `riskLevel` prop 由渲染层（`MessageBubble.vue`）静态推断传入，但 `MessageBubble` 解析 markdown 时并没有实现任何风险分析逻辑——实际上 `riskLevel` 一直是默认值 `none`，风险标签只是 UI 摆设。

在自动执行功能里，`riskLevel` 直接决定是否触发自动执行，一旦 `danger` 级命令被误判为 `none`，将在 `aggressive` 模式下自动执行不可逆操作。

**风险等级**：高（误判 danger 命令 → 自动执行 → 生产事故）

**解决方案**：分两层防御

**方案一（前端静态白名单，短期）**：在 `MessageBubble` 解析命令块时，根据命令前缀做静态风险分析：

```typescript
// utils/commandRisk.ts
const READONLY_PREFIXES = [
  'acli platform info', 'acli alert list', 'acli task list',
  'acli vm list', 'acli vm get', 'acli node list', 'acli node get',
  'acli storage list', 'acli network', 'acli --formatter json',
  'cat ', 'ls ', 'ps ', 'df ', 'free ', 'uname ', 'hostname',
]

const DANGER_PREFIXES = [
  'acli vm migrate', 'acli vm delete', 'acli vm reboot',
  'acli node reboot', 'acli node evacuate', 'acli storage delete',
  'rm ', 'dd ', 'mkfs', 'fdisk',
]

export function inferRiskLevel(command: string): RiskLevel {
  const cmd = command.trim().toLowerCase()
  if (DANGER_PREFIXES.some(p => cmd.startsWith(p))) return 'danger'
  if (READONLY_PREFIXES.some(p => cmd.startsWith(p))) return 'readonly'
  return 'caution'  // 未知命令默认 caution，宁可多拦截
}
```

**方案二（后端 AI 元数据标注，长期）**：AI 在输出命令块时附带结构化注释：

````markdown
<!-- hci:risk:readonly hci:cmd:acli-task-list -->
```bash
acli --formatter json task list
```
````

前端解析 `<!-- hci:risk:* -->` comment 节点，优先使用 AI 标注的风险级别，回退到前端静态推断。

**实施路径**：先实现方案一（2 天），方案二在 AI 提示词迭代时同步推进。

### PIT-B：Agent Loop（AI 无限循环执行命令）

**问题描述**

自动执行 + 结果回填 AI 的组合形成了一个正反馈循环：

```
AI 输出命令① → 自动执行 → 结果注入对话 → AI 输出命令② → 自动执行 → ...
```

在以下场景中，循环可能不会自然终止：

1. **命令执行失败**，AI 不断换不同参数重试（例如 `Permission denied` → AI 尝试 `sudo` → 再次失败 → ...）
2. **AI 收到输出后判断"信息不够"**，继续输出新命令
3. **极端情况**：AI 对某类指令产生重复输出，前端持续执行

理论上 AI 会在诊断完成后给出结论（终止条件），但没有硬性约束时，错误路径可能触发无界循环。

**风险等级**：中（正常使用概率低，但一旦触发可能产生大量 SSH 命令噪音；`danger` 命令有强制拦截，不会导致生产事故）

**解决方案**：三层保护

**层一：单次会话计数上限**

```typescript
const AUTO_EXEC_MAX = 10  // 可配置，建议 5~15

function checkAutoExecBudget(): boolean {
  if (autoExecCount.value >= AUTO_EXEC_MAX) {
    // 超限：关闭自动执行，提示工程师
    autoExecuteMode.value = 'off'
    addSystemMessage(
      `[自动执行] 本次会话已累计自动执行 ${AUTO_EXEC_MAX} 条命令，已自动关闭。` +
      `如需继续，请手动重新开启。`
    )
    return false
  }
  return true
}
```

**层二：连续失败熔断**

```typescript
const consecutiveFailures = ref(0)
const FAILURE_CIRCUIT_BREAKER = 3  // 连续 3 次失败 → 熔断

// 每次自动执行完成后
if (result.exitCode !== 0) {
  consecutiveFailures.value++
  if (consecutiveFailures.value >= FAILURE_CIRCUIT_BREAKER) {
    autoExecuteMode.value = 'off'
    addSystemMessage('[自动执行] 连续 3 次命令失败，已自动关闭。请检查命令后手动继续。')
  }
} else {
  consecutiveFailures.value = 0
}
```

**层三：工程师随时强制中断**

- 提供全局「停止自动执行」按钮（TerminalPanel 头部，只在 `isExecutingCommand === true` 时显示）
- 按 `Ctrl+C` 在终端面板中中断当前正在执行的命令（现有机制）

### PIT-C：Marker 竞争（两个 WebSocket 消费者冲突）

**问题描述**

当前 `sshOutputBuffer` 由全局 `sshWebSocket` 的 `onmessage` 持续写入。但在以下场景中，`CaseCreateDialog.vue` 的采集流程（`sshCreationSocket`）和全局 SSH 连接（`sshWebSocket`）可能同时存在：

- 实际检查代码后发现：两者使用**不同的 WebSocket 实例**（`sshCreationSocket` vs `sshWebSocket`），Bridge 侧需要区分 `case_id` 路由
- 但 `sshCommandConsumer` 全局变量使得 `sshOutputBuffer` 在 `'terminal'` 和 `'collection'` 状态间切换，如果自动执行（`'terminal'` 消费者）和采集（`'collection'` 消费者）同时触发，输出流会发生串扰

**具体竞态场景**：

```
工程师手动触发"刷新环境数据"
    + 同时 AI 输出命令块（safe-only 模式下触发自动执行）
    │
两个消费者都在 sshOutputBuffer 中等待各自的 marker
    → 先完成的消费者匹配到 marker 并截取输出
    → 后完成的消费者的 buffer 已被清空或包含污染数据
```

**风险等级**：中（采集数据错乱，但不会直接影响生产环境）

**解决方案**：采用**命令队列（Command Queue）+ 单消费者**模型

```typescript
type CommandRequest = {
  id: string
  command: string
  marker: string
  consumer: 'auto-exec' | 'collection'
  resolve: (result: BridgeCommandResult) => void
  reject: (err: Error) => void
  timeoutMs: number
}

const commandQueue = ref<CommandRequest[]>([])
const activeCommand = ref<CommandRequest | null>(null)

// 统一的命令调度器，确保任意时刻只有一个命令的 marker 处于待匹配状态
function enqueueCommand(req: Omit<CommandRequest, 'id'>): Promise<BridgeCommandResult> {
  return new Promise((resolve, reject) => {
    commandQueue.value.push({ ...req, id: nanoid(), resolve, reject })
    drainQueue()
  })
}

function drainQueue() {
  if (activeCommand.value || commandQueue.value.length === 0) return
  const next = commandQueue.value.shift()!
  activeCommand.value = next
  // 发送命令 + 开始匹配 marker
  executeOneCommand(next).finally(() => {
    activeCommand.value = null
    drainQueue()  // 继续处理队列
  })
}
```

采集流程和自动执行都通过 `enqueueCommand` 发送命令，队列保证严格串行，杜绝 marker 竞争。

### PIT-D：页面不可见时的安全盲区

**问题描述**

浏览器标签页切换到后台或最小化时，自动执行仍在继续：

- 工程师正在操作其他窗口，命令已在客户环境执行完成
- 工程师回来后看到 N 条命令的执行记录，但整个过程无人监督
- 在 `aggressive` 模式下，`caution` 级命令可能在无人监督时改变了系统状态

**风险等级**：中低（大多数情况只是信息丢失，但极端场景有状态变更风险）

**解决方案**：Page Visibility API 感知 + 暂停队列

```typescript
// 监听页面可见性变化
document.addEventListener('visibilitychange', () => {
  pageVisible.value = document.visibilityState === 'visible'
  if (!pageVisible.value && isExecutingCommand.value) {
    // 不中断正在执行的命令（中断 SSH 流有副作用）
    // 但暂停队列，不派发新命令
    devLog('AUTO-EXEC', '页面不可见，暂停命令队列')
  }
  if (pageVisible.value) {
    devLog('AUTO-EXEC', '页面恢复可见，继续命令队列')
    drainQueue()
  }
})

// drainQueue 中添加可见性检查
function drainQueue() {
  if (!pageVisible.value) return  // 页面不可见时不派发新命令
  if (activeCommand.value || commandQueue.value.length === 0) return
  // ...
}
```

**附加**：在 TerminalPanel 头部显示"后台暂停"状态提示，工程师切回时一目了然队列状态。

## 7. 安全设计摘要

| 安全层 | 措施 |
|--------|------|
| 默认安全 | 全局默认 `off`，需工程师主动开启 |
| 命令分级 | `danger` 无论任何模式强制手动确认 |
| 软取消窗口 | 自动执行前 3 秒倒计时，可中断 |
| 循环防护 | 单会话 10 条上限 + 连续 3 次失败熔断 |
| 并发防护 | 命令队列单消费者，杜绝 marker 竞争 |
| 可见性 | 页面后台时暂停队列派发 |
| 审计 | 执行的命令全量写入 `message` 表（Phase 3） |
