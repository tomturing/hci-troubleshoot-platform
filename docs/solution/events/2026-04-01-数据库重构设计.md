## 一、审核摘要

### 1.1 核心原则

本文档严格遵循 `21_知识库模块重设计.md`（方案B）的架构决策：

1. **知识库模块只有 4 张表**：`kb_category`、`kbd_entry`、`sop_document`、`sop_chunk`
2. **不存在独立的"数据管道"模块**：知识生产流水线由 `data-pipeline/kbd/` 脚本实现，中间产物使用文件存储，不入库
3. **kb-service 是唯一有权写入知识库表的服务**：外部脚本只做数据采集，最终必须调用 kb-service API

### 1.2 当前数据库表总览

| 模块 | 当前表 | 目标态 |
|------|--------|--------|
| **用户工单** | `user`, `case`, `environment`, `session` | 保留 3 张，废弃 `session` |
| **对话** | `conversation`, `message`, `assistant_evaluation`, `prompt_audit`, `tool_audit_log` | 合并为 3 张（`audit_log` 替代两张审计表） |
| **知识库** | `kb_category`, `kb_document`, `kb_chunk`, `kb_sop_node`, `kb_synonym`, `kbd_entry` | **只保留 4 张**：`kb_category`, `kbd_entry`, `sop_document`, `sop_chunk` |
| **历史遗留（待处置）** | `raw_cases`, `knowledge_atoms`, `error_code_index` | 废弃或合并 |

### 1.3 目标表总数：**11 张**

```
模块一：用户工单（case-service）— 3 张
  ├── user
  ├── case
  └── environment

模块二：对话（conversation-service）— 3 张
  ├── conversation
  ├── message
  └── audit_log（合并自 prompt_audit + tool_audit_log）

模块三：知识库（kb-service）— 4 张
  ├── kb_category
  ├── kbd_entry
  ├── sop_document（已实现）
  └── sop_chunk（已实现）

模块四：AI 助手评估（归属修正）— 1 张
  └── assistant_evaluation（从对话模块迁入工单模块）
```

### 1.4 废弃/合并清单

| 表名 | 处置方式 | 原因 |
|------|----------|------|
| `session` | 直接删除 | 从未写入 PG，Redis 是 Source of Truth |

| `tool_audit_log` | 合并入 `audit_log` | 同上 |
| `kb_document` | 废弃 | 被 `kbd_entry` 替代（方案B：KBD 整条存储，不分块） |

## 二、模块一：用户工单（case-service）

### T01 `user` — 用户表

| 项 | 内容 |
|----|------|
| **评估结论** | ✅ 保留（不变） |
| **当前作用** | 存储临时用户与认证用户身份信息，以 `client_id`（前端 localStorage UUID）作为业务幂等键 |
| **设计质量** | 优秀：`client_id UNIQUE`、`user_type` 预留认证升级、触发器维护 `updated_at`、三个精准索引 |
| **改进建议** | 无。认证用户 password_hash/refresh_token 等字段待实现认证系统时通过 migration 添加 |

### T02 `case` — 工单表

| 项 | 内容 |
|----|------|
| **评估结论** | ✅ 保留（不变） |
| **当前作用** | 排障工单核心实体，承载从问题提交到关闭的全生命周期，含状态机（6 态）、来源信息、分类、AI 助手类型 |
| **设计质量** | 优秀：`case_id` 业务可读格式（`Q20260331NNNNN`）、`close_reason` CHECK 约束、冗余 `client_id` 加速查询 |
| **改进建议** | `resolved_at`/`in_progress` 状态当前未使用，未来若引入人工处理升级流程可启用 |

### T03 `environment` — 环境信息表

| 项 | 内容 |
|----|------|
| **评估结论** | ⚠️ 保留，明确角色定位 |
| **当前作用** | 存储前端采集的 HCI 现场环境数据，JSONB 全量存储 |
| **设计问题** | 需求文档标注为"未来功能"，MVP 阶段几乎无写入 |
| **改进建议** | 若单次采集数据量 < 10KB 且无多次快照需求，数据直接写入 `case.metadata`；若需多时点快照才写 `environment` 表 |

### T04 `session` — WebSocket 会话表

| 项 | 内容 |
|----|------|
| **评估结论** | ❌ 废弃（直接删除） |
| **当前作用** | 原设计意图：记录用户+工单+AI POD 绑定关系。从未实际写入 PG |
| **设计问题** | **孤儿表**：连接状态由 Redis 承接，POD 绑定由 `conversation.pod_id` 承接 |
| **处置方案** | `DROP TABLE IF EXISTS session` |

### T08 `audit_log` — AI 行为统一审计表（合并自 prompt_audit + tool_audit_log）

| 项 | 内容 |
|----|------|
| **评估结论** | 🆕 新建（合并两张旧表） |
| **当前作用** | 统一 AI 行为审计：prompt 类型记录 System Instructions 构建，tool_call 类型记录工具调用 |
| **设计规范** | 见下方 DDL |

**`audit_log` 表设计**：

```sql
CREATE TABLE audit_log (
    id           VARCHAR(36)  PRIMARY KEY,           -- UUID
    audit_type   VARCHAR(20)  NOT NULL,               -- 'prompt' | 'tool_call'
    conversation_id  UUID     NOT NULL REFERENCES conversation(conversation_id),
    turn_index   SMALLINT,                            -- prompt 类型：第几轮对话
    tool_name    VARCHAR(100),                        -- tool_call 类型：工具名称
    risk_level   SMALLINT,                            -- 1=只读 2=写 3=高危
    policy       VARCHAR(20),                         -- auto|notify|confirm|block
    authorized_by VARCHAR(100),                       -- 高危操作确认用户
    payload      JSONB        NOT NULL DEFAULT '{}',  -- 类型专属字段
    error        TEXT,
    duration_ms  INTEGER,
    started_at   TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
    completed_at TIMESTAMPTZ,
    trace_id     VARCHAR(64)
);

CREATE INDEX idx_audit_log_conversation ON audit_log(conversation_id, started_at DESC);
CREATE INDEX idx_audit_log_type ON audit_log(audit_type, started_at DESC);
CREATE INDEX idx_audit_log_risk_level ON audit_log(risk_level) WHERE risk_level >= 2;
```

## 四、模块三：知识库（kb-service）— 核心设计

> **依据**：`21_知识库模块重设计.md` 方案B（双轨分离）
> **核心原则**：KBD 和 SOP 有本质形态差异，用独立表保持 schema 语义清晰

### 4.1 知识库 4 张表全景

| # | 表名 | 主键类型 | 核心作用 | 状态 |
|---|------|----------|----------|------|
| 1 | `kb_category` | SERIAL | 分类树（198 节点），全局分类枢纽，意图识别锚点 | ✅ DDL 已落地 |
| 2 | `kbd_entry` | BIGSERIAL | KBD 知识条目（~600 字/条），整条 embedding，无分块 | ✅ DDL 已落地 |
| 3 | `sop_document` | SERIAL | SOP 文档（~20,000 字/个），结构化排障手册 | 🔴 待实现 |
| 4 | `sop_chunk` | SERIAL | SOP 分块检索，按章节拆分 + embedding | 🔴 待实现 |

### 4.2 三轨串行路由架构

```
用户提问 → 意图识别（kb_category.embedding）→ category_id
    │
    ▼
┌─────────────────────────────────────────────────────┐
│ 第1轨：SOP 优先                                      │
│  sop_document WHERE category_id=X AND status='published' │
│  有结果 → 加载 sop_chunk → SOP 步骤引导 → ✅ 返回    │
│  无结果 → 进入第2轨                                  │
└─────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────┐
│ 第2轨：KBD 覆盖                                      │
│  kbd_entry WHERE category_id=X AND status='published' │
│  BM25(tsv) + 向量(embedding) → RRF → top5 条目      │
│  有结果 → LLM RAG context → ✅ 返回                  │
│  无结果 → 进入第3轨                                  │
└─────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────┐
│ 第3轨：人工兜底                                      │
│  生成升级工单 + 标记 category_id → 转人工处理        │
└─────────────────────────────────────────────────────┘
```

**关键设计点**：
- `sop_document` 和 `kbd_entry` **串行独立查询，无需合并结果**
- KBD 整条 embedding（无分块），对 ~600 字短文档天然适配
- SOP 需分块（`sop_chunk`），单篇 ~20,000 字

### 4.3 T09 `kb_category` — 分类基础数据

| 项 | 内容 |
|----|------|
| **评估结论** | ✅ 保留，DDL 已改造 |

### 4.4 T10 `kbd_entry` — KBD 知识条目

| 项 | 内容 |
|----|------|
| **评估结论** | ✅ 保留，已落地 |
| **当前作用** | KBD 知识条目全生命周期：生产 → 审核 → 消费 → 归档 |
| **核心特点** | **整条 embedding，不分块**（对 ~600 字短文档最优） |
| **状态机** | `draft` → `published` → `archived`<br>`draft` → `rejected` |

**完整 DDL**（已写入 migration 20260401001）：

```sql
CREATE TABLE kbd_entry (
    id                 BIGSERIAL    PRIMARY KEY,
    support_id         VARCHAR(20)  UNIQUE NOT NULL,  -- 深信服案例ID
    support_url        TEXT,
    title              TEXT         NOT NULL,
    content_md         TEXT,                         -- 全段结构化 Markdown
    metadata           JSONB        NOT NULL DEFAULT '{}',
    -- 分类（双轨：AI 建议 + 人工确认）
    category_id        VARCHAR(32)  REFERENCES kb_category(code),
    ai_category_id     VARCHAR(32),                  -- AI 分类建议
    ai_category_conf   FLOAT,                        -- 置信度
    ai_category_reason TEXT,                         -- AI 分类理由
    -- 检索字段（published 时生成）
    embedding          vector(1536),                 -- 全文语义向量
    tsv                tsvector,                     -- BM25 全文检索
    -- 状态机
    status             VARCHAR(20)  NOT NULL DEFAULT 'draft',
    reviewer_id        INTEGER,
    reviewed_at        TIMESTAMPTZ,
    review_note        TEXT,
    published_at       TIMESTAMPTZ,
    archived_at        TIMESTAMPTZ,
    created_at         TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
    updated_at         TIMESTAMPTZ  NOT NULL DEFAULT NOW()
);
```

| **评估结论** | ❌ 废弃 |

```

                                     │
```

## 九、v6.2 深度重设计（2026-04-04）

> **背景**：v2.1（11 张表）是基于"已有什么"做的整理清单，未经第一性原理审查。
> 本章节是第一次从"为什么需要这张表、为什么需要这个字段"出发的全面设计。

### 9.1 设计方法论

#### 独立表的第一性原理判定

一个数据是否应独立成表，必须满足以下条件中的 **至少两条**：

| 条件 | 描述 |
|------|------|
| **1. 1:N 子实体** | 父实体对应 N 条该类记录（不是 1:1） |
| **2. 独立生命周期** | 有自己的 created_at/updated_at，可独立创建/更新/删除 |
| **3. 独立查询需求** | 需要按字段过滤、聚合、排序，或需要独立原子更新 |

#### JSONB blob 反模式的判别

以下场景不适合 JSONB blob 存储，必须独立成表：

- **原子更新问题**：需要修改数组/对象中的某一项时，必须读-改-写整个 JSONB（无原子操作，有并发竞态）
- **独立时间戳**：需要知道每条记录何时生成、何时状态变更
- **独立查询**：需要按字段过滤（如"查询所有 status=rejected 的假设"）
- **规模问题**：预期条数 > 5，JSONB 解析显著增加复杂度

### 9.2 从 11 张表到 17 张表的决策矩阵

| 新增表 | 第一性原理证明 | 核心问题 | 决策 |
|--------|--------------|----------|------|
| **`customer`** | (1)✅ 1个客户 N 个工单 (2)✅ 独立更新客户信息 (3)✅ 按客户聚合工单统计 | `case.customer_id` 指向谁？ | 新增 `customer` 表，与 `user` 完全独立（user=登录身份，customer=HCI产品采购方） |
| **`diagnostic_item`** | (1)✅ 1个对话 N 条诊断结论 (2)✅ 每条有独立生命周期和状态 (3)✅ Prompt构建时按会话+类型查询 | `conversation.hypothesis JSONB` 并发竞态，无法独立审计时间戳，无法按条件过滤 | 废弃 JSONB blob，新增 `diagnostic_item` 子实体表（与 `message` 完全同构的设计模式） |
| **`tool_result`** | (1)✅ 1个对话 N 次工具调用 (2)✅ 每次工具调用有独立开始/结束时间 (3)✅ 按工具名、风险等级、步骤号独立查询 | `audit_log` 混合 prompt 和 tool_call，语义不清；BUG-03：缺少 `step_no` 字段 | 从 `audit_log` 中拆分，专用表存储工具执行记录，修复 step_no 缺失 |
| **`system_prompt`** | (1)✅ 1个系统有 N 个模板（多阶段+多版本） (2)✅ 模板有独立版本历史和激活状态 (3)✅ `audit_log` 记录每次使用的模板 ID | Prompt 模板硬编码在代码中，无法版本管理，无法 A/B 测试，无法快速回滚 | 新增 `system_prompt` 表，实现 Prompt 版本化管理 |
| **`tool_definition`** | (1)✅ 工具定义是全局共享的（N 个会话共用） (2)✅ 工具定义独立于对话存在，可独立启停 (3)✅ Prompt 构建时按 category 动态过滤注入 | AI 不知道如何调用工具；工具列表硬编码在代码中，新增工具需改代码 | 新增 `tool_definition` 表，Prompt 构建时动态查询注入，新增工具只需 INSERT |

### 9.3 关键字段设计决策

#### `conversation.hypothesis` → 废弃（BUG-06 根因）

```
发现路径：
  DiagnosticSession.hypotheses (Pydantic, 内存) 
    → session_state["hypothesis"] (dict, 调用时传入)
    → prompt_builder.py (读取 state["hypothesis"])
    → 从未写入 conversation.hypothesis DB 列

BUG-06 后果：Pod 重启 → 假设全丢 → AI 从头开始
修复方案：在 S2 生成假设后，INSERT 到 diagnostic_item 表
结论：conversation.hypothesis JSONB 列 DROP，不可修并非字段不用而是设计根本错误
```

#### `conversation.react_state` → 废弃（正确决策，不是 BUG）

```
react_state 是 ReAct 循环的"思考草稿"（正在进行的推理过程）
属性：
  - 写频率极高（每一步都变）
  - 是过程，不是结论（没有历史价值）
  - 不需要持久化（Pod 重启时 S3/S4 已有 diagnostic_item 可恢复状态）
结论：DROP 此列，AgentState 继续在内存中存活，这是正确设计
```

#### `conversation.pending_confirm` → 保留

```
pending_confirm 是高危工具调用的"断线重连恢复锚点"
属性：风险等级>=2 等待用户确认期间有值，确认/拒绝/超时后清空
断线重连时：前端重连 → 后端读此字段 → 恢复待确认状态给用户
结论：保留，此字段有明确的持久化价值（跨连接恢复）
```

#### `audit_log` 字段精简（工具字段剥离）

```
v2.1 的 audit_log 同时记录 prompt 构建 和 工具调用（audit_type 区分）
问题：
  - tool_call 记录有 tool_name/risk_level/policy/authorized_by 4个专属字段
  - prompt 记录有 turn_index/system_prompt_id 2个专属字段
  - 互相"污染"，空字段过多，语义不清
v6.2 的分离：
  - audit_log = 仅 Prompt 构建审计（tool_call 记录迁移到 tool_result）
  - 保留字段：id, audit_type, conversation_id, turn_index, system_prompt_id, payload, error, duration_ms, started_at, completed_at, trace_id
  - 删除字段：tool_name, risk_level, policy, authorized_by
```

### 9.4 目标态架构（v6.2 — 17 张表）

```
模块一：用户工单（case-service）— 5 张
  ├── user              — 平台用户身份（临时用户 / 认证用户）
  ├── customer          — HCI 产品客户（公司级，非登录用户）      ★ 新增
  ├── case              — 工单全生命周期 + customer_id 外键        ★ 新增字段
  ├── environment       — 环境快照（JSONB, 前端采集）
  └── assistant_evaluation — 工单级 AI 服务质量双轨评分

模块二：对话（conversation-service）— 7 张
  ├── conversation      — 对话会话容器（DROP hypothesis/react_state） ★ 字段变更
  ├── message           — 对话消息历史（LLM context window 来源）
  ├── diagnostic_item   — 诊断结论子表：假设/验证步骤/根因/方案    ★ 新增（解BUG-06）
  ├── tool_result       — 工具执行记录（从 audit_log 拆分，补 step_no）★ 新增（解BUG-03）
  ├── audit_log         — Prompt 构建审计（精简后仅 prompt 类型）   ★ 字段变更
  ├── system_prompt     — System Instructions 模板库（版本管理）    ★ 新增
  └── tool_definition   — 工具知识库（Prompt 构建时动态注入）       ★ 新增

模块三：知识库（kb-service）— 4 张
  ├── kb_category       — 分类树（235 节点，含父子关系）
  ├── kbd_entry         — KBD 知识条目（整条 embedding，600字/条）
  ├── sop_document      — SOP 排障手册（完整 Markdown）
  └── sop_chunk         — SOP 分块检索（BM25 + 向量 RRF）

模块四：迁移管理（1 张）
  └── schema_migrations — dbmate 版本管理
```

### 9.5 表间关系图

```
user(1) ──────────────────────────────┐
  └──(N) case(N) ──(1) customer       │
           └──(N) environment         │
           └──(1) assistant_evaluation│
           └──(N) conversation        │
                    └──(N) message    │
                    └──(N) diagnostic_item
                    └──(N) tool_result
                    └──(N) audit_log
                               └──(N→1) system_prompt
tool_result(N→1) tool_definition
```

**级联规则：**
- `user → case → conversation → message`：ON DELETE CASCADE（删用户清所有数据）
- `customer → case`：ON DELETE SET NULL（删客户保留历史工单）
- `system_prompt → audit_log`：ON DELETE SET NULL（删模板保留审计记录）
- `conversation → diagnostic_item/tool_result/audit_log`：ON DELETE CASCADE

#### 步骤 3：拆分 audit_log 数据（数据迁移）
```sql
-- 将 audit_log 中 audit_type='tool_call' 的记录迁移到 tool_result

DELETE FROM audit_log WHERE audit_type = 'tool_call';
-- 然后删除 tool_call 专属字段
ALTER TABLE audit_log DROP COLUMN IF EXISTS tool_name, DROP COLUMN IF EXISTS risk_level, ...;
```
