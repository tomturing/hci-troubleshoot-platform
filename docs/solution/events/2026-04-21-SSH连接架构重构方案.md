# SSH 连接架构重构方案

## 背景与需求
参见 [需求文档](../../requirement/events/2026-04-21-SSH连接架构重构需求.md)

核心问题：
1. SSH 连接分散在 TerminalPanel.vue 和 chat.ts 各自管理
2. 环境数据采集时机错误（只在创建工单时，而非 SSH 连接成功后）
3. Bridge 检测弹框无刷新按钮
4. 连接过程卡住，缺少日志

## 方案概述 (WHAT)

**将 SSH WebSocket 连接从组件私有改为全局统一管理。**

```
架构变更：

之前：
┌─────────────────────┐     ┌─────────────────────┐
│ TerminalPanel.vue   │     │ CaseCreateDialog    │
│ ├─ bridgeSocket     │     │ (通过 chat.ts)      │
│ └─ connectSsh()     │     │ ├─ sshCreationSocket│
│ └─ 独立状态管理     │     │ └─ connectSSH...()  │
└─────────────────────┘     └─────────────────────┘
        ↓                           ↓
    WebSocket A                WebSocket B
        ↓                           ↓
   terminal_bridge.exe        terminal_bridge.exe

之后：
                    ┌─────────────────────────────┐
                    │ chatStore（全局唯一）        │
                    │ ├─ sshWebSocket             │
                    │ ├─ sshConnectionState       │
                    │ ├─ sshConfig                │
                    │ ├─ connectSSH()             │
                    │ ├─ disconnectSSH()          │
                    │ ├─ sendCommand()            │
                    │ └─ refreshEnvironmentData() │
                    └─────────────────────────────┘
                              ↓
                         WebSocket（唯一）
                              ↓
                      terminal_bridge.exe
                              ↓
                    ┌─────────────────────────────┐
                    │ 命令输出分发                │
                    │ ├─ 用户命令 → TerminalPanel │
                    │ ├─ 采集命令 → Environment   │
                    └─────────────────────────────┘
```

## 详细设计

### 1. chatStore 新增全局 SSH 状态

```typescript
// === 全局 SSH 连接状态 ===
const sshWebSocket = ref<WebSocket | null>(null)
const sshConnectionState = ref<'disconnected' | 'connecting' | 'connected' | 'error'>('disconnected')
const sshConfig = ref<{
  host: string
  port: number
  username: string
  authType: 'password' | 'key'
} | null>(null)
const sshOutputBuffer = ref<string>('')  // 终端输出缓冲
const sshCollectionBuffer = ref<Record<string, string>>({})  // 采集输出缓冲
const sshCommandConsumer = ref<'terminal' | 'collection' | null>(null)  // 当前命令消费者

// SSH 连接方法
function connectSSH(config: SSHConnectConfig): Promise<void>
function disconnectSSH(): void
function sendCommand(cmd: string, consumer: 'terminal' | 'collection'): void
function refreshEnvironmentData(): Promise<void>  // 采集环境数据
```

### 2. 消息分发机制

```typescript
// WebSocket 消息处理
function handleSshMessage(msg: TerminalWsMessage) {
  if (msg.type === 'ssh_output' && msg.output) {
    if (sshCommandConsumer.value === 'terminal') {
      // 发送到 TerminalPanel 展示
      sshOutputBuffer.value += msg.output
      emitTerminalOutput(msg.output)  // 通过 ref/event 通知 TerminalPanel
    } else if (sshCommandConsumer.value === 'collection') {
      // 发送到采集缓冲
      handleCollectionOutput(msg.output)
    }
  }
}
```

### 3. 智能采集触发

```typescript
// SSH 连接成功后
function onSshConnected() {
  sshConnectionState.value = 'connected'
  
  // 检查环境数据状态
  const lastCollected = environmentContext.value?.collected_at
  const isExpired = !lastCollected || (Date.now() - new Date(lastCollected).getTime() > 5 * 60 * 1000)
  
  if (!environmentContext.value || isExpired) {
    // 自动采集
    refreshEnvironmentData()
  }
}
```

## 决策依据 (WHY)

### 方案选择

| 方案 | 优点 | 缺点 | 评分 |
|------|------|------|------|
| **方案A：全局统一管理** | 单一连接、状态共享、采集时机正确 | 改动较大、需处理输出分发 | ★★★★☆ |
| 方案B：保持分散 + 采集同步 | 改动较小 | 资源浪费、认证重复、状态不一致 | ★★☆☆☆ |
| 方案C：仅在 TerminalPanel 添加采集 | 改动最小 | 创建工单流程不统一、架构不合理 | ★☆☆☆☆ |

### 为什么选择方案A？

**第一性原理分析**：

1. **SSH 连接是能力，不是动作**
   - 能力应该随时可用，而不是绑定某个时机
   - 连接成功后，采集作为后台能力自动执行

2. **WebSocket 连接是有限资源**
   - 每个 WebSocket 对应一个 SSH 会话
   - 同一目标主机不应重复建立会话
   - 统一管理减少资源浪费和认证开销

3. **状态一致性**
   - 用户在任何地方看到的 SSH 状态应该一致
   - 「已连接」或「未连接」是全局事实，不是组件局部状态

### 为什么不选方案B和C？

- **方案B**：保持分散会导致状态不一致，用户在 TerminalPanel 连接后，CaseCreateDialog 不知道已连接
- **方案C**：只在某个组件加采集，另一个组件的行为不一致，架构不合理

## 影响范围

### 受影响的文件
- `frontend/customer/src/stores/chat.ts` - 添加全局 SSH 状态管理，移除 sshCreationSocket
- `frontend/customer/src/components/TerminalPanel.vue` - 移除独立 WebSocket，改用 chatStore
- `frontend/customer/src/components/CaseCreateDialog.vue` - 移除独立 WebSocket，改用 chatStore，添加 Bridge 刷新按钮
- `frontend/customer/src/components/EnvironmentSummary.vue` - 刷新按钮绑定 SSH 状态

### API 兼容性
- 无 API 变更，仅前端重构

## 实施计划

### T1: chatStore 全局 SSH 状态
- 添加 sshWebSocket、sshConnectionState、sshConfig
- 实现 connectSSH、disconnectSSH、sendCommand

### T2: TerminalPanel 改用全局连接
- 移除 bridgeSocket
- 使用 chatStore.sshWebSocket 和状态
- 通过 event/ref 接收输出

### T3: CaseCreateDialog 改用全局连接
- 移除 sshCreationSocket
- 使用 chatStore.connectSSH
- 添加 Bridge 刷新按钮
- 优化连接状态 UX

### T4: 智能采集逻辑
- SSH 连接成功后自动检查环境数据
- 实现 refreshEnvironmentData 方法

### T5: 连接日志增强
- 各阶段添加 console.log
- 确保卡住问题可排查

## 风险与缓解

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|---------|
| 输出分发混乱 | 高 | 中 | 使用 consumer 标记明确命令归属 |
| 连接断开后组件状态不一致 | 中 | 低 | 监听 sshConnectionState 变化更新各组件 |
| 采集命令执行时间过长 | 中 | 中 | 设置超时，失败不影响用户操作 |

## 测试策略

1. **单元测试**：connectSSH/disconnectSSH 状态流转
2. **集成测试**：TerminalPanel 和 CaseCreateDialog 共用连接
3. **人工测试**：
   - 创建工单时 SSH 连接 → 自动采集
   - 打开终端面板 SSH 连接 → 自动采集（环境数据过期时）
   - Bridge 未运行 → 点击刷新检测

## 验收标准

- [ ] TerminalPanel 和 CaseCreateDialog 共用 chatStore 的 WebSocket
- [ ] SSH 连接成功后自动采集（无数据或过期时）
- [ ] Bridge 弹框有「重新检测」按钮
- [ ] 连接不卡住，有明确反馈
- [ ] 连接状态显示分层（蓝色主 + 灰色辅助）
