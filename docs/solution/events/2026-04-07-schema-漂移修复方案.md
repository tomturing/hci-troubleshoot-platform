# DB Schema 漂移修复方案

## 背景与需求

详见 [2026-04-07-schema-漂移修复需求](../../requirement/events/2026-04-07-schema-漂移修复需求.md)。

核心问题：dev 环境 `schema_migrations` 记录了 8 次迁移已执行，但实际 DDL 未生效，导致 10 张表 + 15 个列缺失。

---

## 1. 根因分析

### 1.1 漂移发现过程

```
kubectl exec postgres-0 -n hci-dev -- psql -U hci_admin -d hci_troubleshoot -c "\dt+"
```

对比 `数据库设计.md` v6.2 中定义的 17 张业务表，实际仅存在部分早期表。

### 1.2 漂移清单

| 缺失类型 | 数量 | 详情 |
|----------|------|------|
| 缺失表 | 10 | customer, diagnostic_item, tool_result, system_prompt, tool_definition, kbd_entry, sop_document, sop_chunk, raw_cases, knowledge_atoms |
| 缺失列 | 15 | case.customer_id, case.close_reason, case.closed_at, conversation.repeat_question_count, conversation.hypothesis, conversation.diagnostic_stage, conversation.stage_updated_at, assistant_evaluation 多个评分列, error_code_index 表, 等 |

---

## 2. 修复方案

### 2.1 核心原则

1. **幂等迁移**：所有 DDL 使用 `CREATE TABLE IF NOT EXISTS` / `ALTER TABLE ... ADD COLUMN IF NOT EXISTS`
2. **单文件修复**：将所有缺失 DDL 合并为一个修复迁移 `20260407001_schema_repair.sql`
3. **双系统同步**：dbmate 迁移 + Alembic 0005 版本同步
4. **先修后启**：手动执行修复 → 验证 → 再启用 `dbMigrate.enabled: true`

### 2.2 迁移文件设计

**`database/migrations/20260407001_schema_repair.sql`**（447 行）

分 10 个 Section 覆盖所有修复：

| Section | 涉及对象 | DDL 类型 |
|---------|---------|---------|
| §1 | customer 表 | CREATE TABLE IF NOT EXISTS |
| §2 | case 表新增列 | ADD COLUMN IF NOT EXISTS × 3 |
| §3 | conversation 表新增列 | ADD COLUMN IF NOT EXISTS × 4 |
| §4 | assistant_evaluation 表新增列 | ADD COLUMN IF NOT EXISTS × 多列 |
| §5 | kbd_entry 表 | CREATE TABLE IF NOT EXISTS |
| §6 | sop_document、sop_chunk 表 | CREATE TABLE IF NOT EXISTS × 2 |
| §7 | raw_cases、knowledge_atoms 表 | CREATE TABLE IF NOT EXISTS × 2 |
| §8 | error_code_index 表 | CREATE TABLE IF NOT EXISTS |
| §9 | tool_result 表 | CREATE TABLE IF NOT EXISTS |
| §10 | diagnostic_item 表 | CREATE TABLE IF NOT EXISTS |

每个 Section 包含：表/列创建 + 索引创建 + 外键约束。

### 2.3 Alembic 同步

**`backend/shared/migrations/versions/0005_schema_repair.py`**

- `revision = '0005'`，`down_revision = '0004'`
- 使用 `op.execute()` 执行与 dbmate 相同的幂等 DDL
- 本地开发者执行 `alembic upgrade head` 即可对齐

### 2.4 自动化脚本

**`scripts/ops/db-repair-env.sh`**

一键修复任意 namespace 的 schema：
```bash
./scripts/ops/db-repair-env.sh <namespace>
# 示例: ./scripts/ops/db-repair-env.sh hci-staging
```

流程：诊断 → 复制 SQL → 执行修复 → 注册版本 → 验证表数量

### 2.5 多环境推进策略

| Phase | 环境 | 状态 | 操作方式 |
|-------|------|------|---------|
| 1 | dev | ✅ 已完成 | 手动 kubectl exec |
| 2 | staging | ⬜ 待执行 | db-repair-env.sh + 启用 dbMigrate |
| 3 | prod | ⬜ 待执行 | db-repair-env.sh + pg_dump 备份 + 审批 |

详细分环境操作步骤见 [2026-04-07-schema-漂移修复部署](../../deploy/events/2026-04-07-schema-漂移修复部署.md)。

## 3. 长期防护建议

### 3.1 CI Schema 一致性检查

在 CI 中新增 `schema-consistency` job，每次构建时：
1. 启动临时 PostgreSQL 容器
2. 从 ConfigMap 模板提取 SQL，执行全量迁移
3. 对比 ORM metadata 与实际 DB schema

### 3.2 双迁移系统对齐矩阵

| dbmate 版本 | Alembic 版本 | 内容 |
|------------|-------------|------|
