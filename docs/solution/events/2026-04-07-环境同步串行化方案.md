# 环境同步串行化方案

## 背景与需求

详见 [2026-04-07-环境同步并发竞争修复需求](../../requirement/events/2026-04-07-环境同步并发竞争修复需求.md)。

核心问题：CI 同时触发 dev 和 staging 环境同步，两个 workflow 并发推送同一 main 分支，导致后推送者被拒绝。

## 方案（WHAT）

### 方案概述

将环境同步从**并发触发**改为**串行执行**，并确保 **dev 环境优先**。

### 实现方案

修改 `.github/workflows/ci.yml` 中的 `auto-deploy-non-prod` job，将并发循环改为串行执行：

```yaml
- name: 触发环境仓库同步（dev 优先，串行执行）
  env:
    GH_TOKEN: ${{ secrets.ENV_REPO_PAT }}
  run: |
    # 串行执行环境同步，dev 优先
    # 每个环境同步完成后等待，确保 Git 推送完成，避免并发竞争
    environments=("dev" "staging")

    for env in "${environments[@]}"; do
      echo "========== 触发 ${env} 环境同步 =========="
      gh workflow run env-repo-sync.yml \
        -R tomturing/hci-platform-env \
        -f env_repo_name=tomturing/hci-platform-env \
        -f target_env=${env} \
        -f image_tag="${{ needs.prepare.outputs.image_tag }}"

      # 等待 workflow 启动并完成 Git 推送（约 30 秒）
      echo "等待 ${env} 环境同步完成..."
      sleep 30

      # 输出最近一次同步状态
      echo "检查 ${env} 环境同步状态..."
      gh run list -R tomturing/hci-platform-env \
        --workflow env-repo-sync.yml \
        --limit 1 \
        --json conclusion,displayTitle
    done

    echo "✅ dev/staging 环境同步已触发（串行执行完成）"
```

### 关键设计点

1. **dev 优先**：`environments=("dev" "staging")` 数组顺序决定执行顺序
2. **串行执行**：`sleep 30` 确保 Git 推送完成后再触发下一个环境
3. **状态可观测**：每个环境同步后打印状态，便于排查问题
4. **容错设计**：即使 staging 失败，dev 已完成同步

## 决策依据（WHY）

### 为什么选此方案

| 评估维度 | 方案A：并发触发（现状） | 方案B：串行执行（本方案） | 方案C：独立分支 |
|---------|---------------------|----------------------|--------------|
| 实现复杂度 | 低（已实现） | 低（仅修改脚本） | 高（需改造 env-repo-sync） |
| dev 优先保证 | ❌ 无法保证 | ✅ 明确保证 | ✅ 可保证 |
| 并发竞争风险 | ❌ 高风险 | ✅ 无风险 | ✅ 无风险 |
| 失败隔离 | ❌ staging 阻塞 dev | ✅ dev 先完成，不受 staging 影响 | ✅ 完全隔离 |
| 维护成本 | 低 | 低 | 中 |

**选择方案B的理由**：
1. 实现简单，仅修改一个 job 的脚本逻辑
2. 彻底解决并发竞争问题
3. 满足"dev 优先"和"staging 不阻塞 dev"两个核心需求

### 为什么不选其他方案

**方案C：独立分支**：
- 需要改造 `env-repo-sync.yml`，为每个环境使用独立分支（如 `env/dev`、`env/staging`）
- 增加维护复杂度，需要处理分支同步问题
- 对于当前规模（2 个非生产环境）过度设计

**方案A：保持并发但加重试**：
- 重试逻辑复杂，难以确定重试时机
- 治标不治本，仍有失败可能
- 日志混乱，难以排查问题
