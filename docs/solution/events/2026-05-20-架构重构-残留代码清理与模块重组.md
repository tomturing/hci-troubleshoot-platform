# 架构重构：残留代码清理 + 可观测性模块重组 + 文档目录对齐

**日期**：2026-05-20  
**背景**：hci-troubleshoot-platform 随 PR #298（C 大脑 pydantic-ai）、#300（kbd 迁移）、#301（Markdown SOP）陆续合并后，进行一次全面的结构审计，发现三类结构性问题需要系统修复。

---

## 一、背景与问题发现

### 问题触发

PR #300 合并后扫描仓库目录结构，发现以下不一致：

1. `adapters/cli_openai_adapter/` 位于根目录，但不在 `backend/` 内
2. `backend/shared/utils/` 混放了可观测性工具与通用工具
3. `docs/solution/agent/` 与 `docs/task/agent/` 的目录名使用了 `ai-assistant`，与业界 2025-2026 年 agent 标准术语不一致

### 第一性原理分析

**目录结构应反映三种边界的统一维度**：

| 边界维度 | 含义 | 混乱后果 |
|---------|------|---------|
| 部署边界 | 独立容器、独立扩缩容 | 无法单独部署 |
| 职责边界 | 单一变化轴 | SRP 违反 |
| 层级边界 | 应用代码 vs 基础设施 | 运维/研发边界模糊 |

## 二、`adapters/cli_openai_adapter/` — 决策：删除

### 事实陈述

- 最后提交：2026-03-03（2.5 个月无更新）
- 未被 `deploy/helm/` 引用（不部署到集群）
- 未被 `.github/workflows/` 引用（不参与 CI 构建）
- `scripts/dev/start_nanoclaw_adapter.sh` 自注释："NanoClaw 当前没有稳定 one-shot CLI"
- 有 4 个 Dockerfile，无一进入生产

### 根本原因

**命名哲学污染了部署拓扑**：该目录用了设计模式语言（`Adapter` 模式），而 `backend/` 用部署单元语言（`Service`），两者正交。

该代码是 2026-03 月评估 CLI AI 工具（nanobot/nanoclaw/picoclaw）时的过渡产物。自项目引入 B 大脑（ops-agent HTTP）和 C 大脑（pydantic-ai 原生）后，"CLI 工具包装为 OpenAI API"路线已被放弃。

### 决策

**删除 `adapters/cli_openai_adapter/` 及整个 `adapters/` 目录**。  
同步更新 `scripts/dev/` 中的 3 个启动脚本（标注为已废弃）。

## 四、`backend/shared/utils/` 可观测性文件 — 决策：移入子包

### 当前状态

`backend/shared/utils/` 混放了两类职责不同的文件：

| 文件 | 职责类型 |
|-----|---------|
| `exceptions.py` | 通用工具 |
| `exception_handlers.py` | 通用工具 |
| `internal_http.py` | 通用工具 |
| `logger.py` | **可观测性** |
| `metrics.py` | **可观测性** |
| `otel.py` | **可观测性** |
| `trace_id.py` | **可观测性** |

### 第一性原理分类

这 4 个文件是**共享库（被所有服务 import 的公共函数）**，不是独立可部署服务。

| 代码本质 | 正确容器 |
|---------|---------|
| 被其他服务 import 的公共函数/类 | `backend/shared/` 子包 |
| 独立启动、有 Dockerfile、暴露 HTTP 端口 | `backend/<service-name>/` |

### 为什么不放 `backend/observability/` 顶级目录

顶级 `backend/<name>/` 暗示独立可部署服务（与 `case-service`、`kb-service` 同级），但这 4 个文件没有 HTTP 入口，不能独立部署。放在顶级目录会制造歧义。

### 决策

在 `backend/shared/` 内新建子包 `backend/shared/observability/`，迁入 4 个文件：

```
backend/shared/
├── observability/          # 新建子包（可观测性共享库）
│   ├── __init__.py
│   ├── logger.py
│   ├── metrics.py
│   ├── otel.py
│   └── trace_id.py
└── utils/                  # 保留真正的通用工具
    ├── exception_handlers.py
    ├── exceptions.py
    └── internal_http.py
```

Import 路径从 `from shared.utils.metrics import ...` 变为 `from shared.observability.metrics import ...`。

## 五、服务命名 — 决策：未来新服务命名为 `agent-service`

### 背景

当前 3 个大脑 Adapter（A/B/C）+ BrainPort + ReactExecutor 嵌入在 `conversation-service` 内部，违反单一职责原则（对话管理 vs AI 推理引擎是两个独立变化轴）。

本次重构**不执行服务拆分**（中等风险，需等 C 大脑稳定后单独 PR），仅确定命名决策供后续实施参考。

### 四个候选名分析

| 名称 | 人类可读性 | AI 友好性 | 歧义风险 | 业界范式 |
|-----|-----------|---------|---------|---------|
| `brain-service` | 项目内部行话 | 需读项目文档 | 低（但晦涩）| ❌ 无对应概念 |
| `ai-assistant-service` | "assistant" 被特定含义占用 | 误导为 OpenAI Assistants API | ⚠️ 高 | ❌ 错误范式 |
| `ai-agent-service` | "ai-" 在 AI 原生平台冗余 | 双重冗余 | 低 | 部分 |
| **`agent-service`** | 简洁直观，2026 年标准词汇 | 无需额外上下文 | **最低** | **✅ 完全符合** |

### 决策

**命名为 `agent-service`**，理由：

1. **业界已收敛**：Amazon Bedrock Agents、Vertex AI Agents、LangGraph、Semantic Kernel 均用 `agent` 指代自主推理执行单元
2. **仓库内一致性**：`ops-agent`（B 大脑外部仓库）已用 `agent` 命名，BrainRouter 路由的目标也叫 Agent
3. **无冗余前缀**：平台内所有服务均无 `ai-` 前缀（`case-service`、`kb-service` 等）
4. **API/容器命名友好**：镜像名 `hci-agent-service:v1.x`、Helm release `agent-service`、K8s Service `agent-service` 全链路一致
5. **防止 OpenAI 语义干扰**：`assistant` 已强绑定到 OpenAI Assistants API（线程/文件/工具托管模式）

### 未来服务拆分方向

```
backend/
├── conversation-service/     # 对话管理（消息持久化、SSE 推送、会话 CRUD）
├── agent-service/            # AI 推理引擎（BrainPort + BrainRouter + A/C 大脑 + ReactExecutor）
│   └── ...（从 conversation-service 抽取，独立 PR）
└── ...
```
