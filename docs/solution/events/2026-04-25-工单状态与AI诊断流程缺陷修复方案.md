# 工单状态与 AI 诊断流程缺陷修复方案

## 背景与需求

修复 4 个被识别出的设计实现偏差，详见需求文档：
[docs/requirement/events/2026-04-25-工单状态与AI诊断流程缺陷修复需求.md](../../../requirement/events/2026-04-25-工单状态与AI诊断流程缺陷修复需求.md)

---

## 方案概述

本次修复围绕两大主线：

1. **AI 诊断流程贯通**（最高优先级）：修复 S0→S1 阶段切换的三处错误（触发词、写库、死代码），让诊断流程真正跑起来
2. **基础设施正确性**：修复 `TerminalService` 数据库注入缺失、前端 confirm 时序错误、S0 Prompt 强制用户确认

## 详细设计

---

### T1：修复工单状态流转（`confirmed` 不再由前端抢占）

#### 方案选择

| 方案 | 描述 | 优点 | 缺点 | 评分 |
|------|------|------|------|------|
| **方案A（选中）** | 前端删除立即 confirm 调用；conversation-service 在 `_update_conversation_category` 中调用 case-service confirm 接口 | 符合设计文档，状态语义准确 | 需要在 conversation-service 新增 HTTP 调用 | ★★★★☆ |
| 方案B | 保留前端立即 confirm，但修改消息文案隐藏时序问题 | 改动最小 | 治标不治本，状态语义依然错误，SP-1 永远不会被真正触发 | ★★☆☆☆ |
| 方案C | 前端轮询工单状态，等 confirmed 再继续 | 状态驱动 | 引入轮询复杂度；S0 结束时没有机制通知 case-service 更新状态 | ★★★☆☆ |

**选择方案A**：conversation-service 是唯一知道 S0 何时完成的服务，由它负责推进状态符合设计意图（SP-1 同步点）。

#### 实现细节

**前端变更**（[frontend/customer/src/stores/chat.ts](../../../frontend/customer/src/stores/chat.ts)）：
- 删除 `confirmCreateCase()` 中的 `caseApi.confirm(res.data.case_id)` 调用及相关消息
- 工单创建后直接进入对话，UI 显示"工单 Q... 已创建，AI 正在识别故障类型…"
- 通过已有 SSE 连接接收 `case_status_updated` 事件（如该事件尚未实现，则在 S0 完成后由 AI 回复内容隐式体现，前端不主动 confirm）

**conversation-service 变更**（[backend/conversation-service/app/services/conversation_service.py](../../../backend/conversation-service/app/services/conversation_service.py)）：
- `_update_conversation_category()` 成功更新 `category_id` 后，调用 case-service 的 `PUT /api/cases/{case_id}/confirm`
- 使用 `httpx.AsyncClient` 异步非阻塞调用，失败时只记录 warning 不影响主流程

```python
# _update_conversation_category() 末尾新增：
async with httpx.AsyncClient(timeout=5.0) as client:
    resp = await client.put(
        f"{settings.CASE_SERVICE_URL}/api/cases/{case_id}/confirm"
    )
    if resp.status_code not in (200, 404):
        logger.warning(
            event="case_confirm_failed",
            message=f"案例 confirm 接口调用失败: {resp.status_code}",
            case_id=case_id,
        )
```

### T2：S0 Prompt 强制候选确认（删除高置信度直接确认分支）

#### 方案选择

| 方案 | 描述 | 评分 |
|------|------|------|
| **方案A（选中）** | 修改 `_segment_s0_output_format()` 和 `_segment_s0_methodology()`，删除情况一（高置信度直接确认），统一走候选确认模式 | ★★★★★ |
| 方案B | 保留两路分支，在 conversation-service 接到 AI 高置信度回复后程序性注入选项 | 过于复杂，且需要二次调用 AI | ★★☆☆☆ |

**选择方案A**：最简单直接，Prompt 是唯一需要修改的地方。

#### 新输出格式规范

```
==== 统一格式（高/中置信度均使用）====
根据 [简要判断依据]，您的问题可能属于以下故障之一，请确认：

① {分类code-1} {分类名-1}
   判断依据：[引用告警/日志原文或描述关键词]，概率估计 ~XX%

② {分类code-2} {分类名-2}（可选，有明确次选项时提供）
   判断依据：[引用具体证据]，概率估计 ~XX%

③ 以上都不是（请补充症状描述）

请回复 ① ② 或 ③ 完成确认。

==== 约束 ====
- 候选最多 2 个（+固定的③）；若只有 1 个候选则只列 ① 和 ③
- 每个候选必须附判断依据，引用实际日志或告警内容
- 禁止输出"已确认故障分类：…"直接标记（只有后端在用户确认后才可写入）
- 不得输出任何开放性问题（"请问..." / "您是否..." / "能描述一下..."）
```

**同步修改方法论段落**：删除"高置信度 ≥80% 直接确认"的描述，改为"无论置信度，一律展示候选"。

### T3：修复 S0→S1 阶段切换（三处错误一次性修复）

#### 错误 3-a 修复：更新 STAGE_TRIGGERS

在 `STAGE_TRIGGERS[("S0","S1")]` 追加匹配用户候选选择完成后的触发词：

```python
("S0", "S1"): [
    # 旧触发词（保留，防止兼容性问题）
    r"确认(?:您?的|一下)?(?:故障|问题|情况).*(?:是|为)",
    r"(?:故障|问题)(?:已|初步)?(?:定位|确认)",
    r"根据.*(?:回答|描述).*(?:判断|确定).*(?:故障|问题)",
    # 新增：匹配 AI 在候选被用户确认后输出的阶段推进语句
    r"好的，确认故障分类为",
    r"已收到您的确认",
    r"进入.*(?:故障定位|S1)",
    r"开始.*(?:故障定位|定位分析)",
],
```

#### 错误 3-b 修复：解耦 category_info 写入

将 `_update_conversation_category` 从 `if new_stage:` 块内移出：

```python
# 修复前（错误）
if new_stage:
    asyncio.create_task(_update_diagnostic_stage(...))
    if current_stage == "S0" and category_info:
        asyncio.create_task(_update_conversation_category(...))  # 被 new_stage 门控

# 修复后（正确）
if new_stage:
    asyncio.create_task(_update_diagnostic_stage(...))

# S0 分类结果独立写入，不依赖阶段转换是否发生
if current_stage == "S0" and category_info:
    asyncio.create_task(_update_conversation_category(...))
```

#### 错误 3-c 修复：接入候选确认链路

在 `send_message_stream_only()` 保存用户消息后（步骤 1 完成后），当 `current_stage == "S0"` 时，**先进行候选确认预处理**，再决定是否发送给 AI：

```python
# S0 候选确认预处理（在读取 diagnostic_stage 之后，调用 AI 之前）
if current_stage == "S0":
    selection = self._conversation_manager.parse_candidate_selection(content)
    if selection is not None:
        # 用户输入了序号，需要从上一条 AI 消息中提取候选列表
        candidates = await self._extract_s0_candidates(conversation_id)
        if candidates:
            chosen = self._conversation_manager.resolve_candidate_category(selection, candidates)
            if chosen:
                # 用户确认了分类，直接写库并强制推进阶段
                asyncio.create_task(
                    self._update_conversation_category(conversation_id, case_id, chosen)
                )
                asyncio.create_task(
                    self._update_diagnostic_stage(conversation_id, "S1", "S0")
                )
                # 构造 AI 回复，告知进入 S1
                confirmation_reply = (
                    f"好的，确认故障分类为【{chosen['code']} {chosen['name']}】。\n"
                    f"开始故障定位分析，请稍候…"
                )
                yield confirmation_reply
                return  # 不再调用 AI，本轮直接结束
            else:
                # 用户选 ③（以上都不是）
                # 检查兜底阈值
                s0_rounds = await self._get_s0_candidate_rounds(conversation_id)
                if ConversationManager.should_trigger_s0_failure(s0_rounds):
                    failure_msg = await self.handle_s0_failure(conversation_id, case_id)
                    yield failure_msg
                    return
                # 否则继续让 AI 重新提问（round +1）
                asyncio.create_task(self._increment_s0_candidate_rounds(conversation_id))
```

辅助方法 `_extract_s0_candidates()` 从上一条 AI 消息中用正则提取 ①② 列表；`_get_s0_candidate_rounds()` 和 `_increment_s0_candidate_rounds()` 读写 `conversation.metadata_` 中的 `s0_candidate_rounds` 字段。

### T4：修复 TerminalService 数据库注入缺失

**根因**：`api-gateway/app/main.py` 初始化 `TerminalService` 时缺少 `db_manager` 参数。

**修复**：初始化顺序调整，`DatabaseManager` 在 `TerminalService` 之前实例化并注入。

```python
# 修复前
terminal_service = TerminalService(redis_manager)

# 修复后
from shared.database.postgres import DatabaseManager
db_manager = DatabaseManager(settings.DATABASE_URL)
await db_manager.connect()
terminal_service = TerminalService(redis_manager, db_manager=db_manager)
```

同时确认 `api-gateway/app/config.py` 中已有 `DATABASE_URL` 字段（或根据实际字段名适配）。

## 风险与缓解

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|---------|
| T3-c 新增 `_extract_s0_candidates()` 正则不够健壮 | S0 候选无法解析，流程退化到当前行为（死循环） | 中 | 加兜底：解析失败时继续走 AI，不 crash |
| T1 conversation-service confirm 调用网络超时 | `confirmed` 状态未更新 | 低 | timeout=5s，失败只记 warning，不阻塞 |
| T4 `api-gateway` 新增 DB 连接影响启动时间 | 启动慢 | 低 | DB 连接池复用现有配置，仅新增 `connect()` 调用 |
