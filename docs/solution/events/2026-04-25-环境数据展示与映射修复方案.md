# 环境数据展示与映射修复方案

## 背景与需求

参见：[2026-04-25-环境数据展示与映射修复需求.md](../requirement/events/2026-04-25-环境数据展示与映射修复需求.md)

SSH 采集后的环境数据经过以下链路处理才能展示/注入给 Agent：

```
acli 命令输出
  → 前端 parseClusterOutput / parseJsonOutput    （前端 chat.ts）
  → 入库 env_data（JSONB）
  → 后端 _extract_alert_logs / _extract_task_logs （case-service）
  → EnvironmentContextResponse
  → PromptBuilder._segment_s0_context_info
  → Agent S0 Prompt
```

链路上有 3 处独立 bug，导致全链路数据错误。

---

## 方案概述 (WHAT)

**定点修复三处字段映射/解析逻辑，不改架构、不改数据库结构、不改接口。**

| 编号 | 位置 | 问题 | 修复 |
|------|------|------|------|
| B1 | 前端 `chat.ts` `parseClusterOutput` | `acli platform info get` 输出为 `key=value` + `[section]` ini 格式，但代码按 `:` 分割；同时命令回显行混入输出 | 重写为 ini 风格解析器，先剥离 DONE marker 和命令回显行 |
| B2 | 后端 `environment_service.py` `_extract_alert_logs` | 取 `level`/`trigger_time`/`content`/`source` 四个不存在的字段；实际字段为 `urgent_type`/`start`/`description`/`target` | 修正字段名 + 映射 urgent_type 为标准 level |
| B3 | 后端 `environment_service.py` `_extract_task_logs` | 取数字字段 `status`（值 3）和不存在的 `start_time`/`name`/`error_msg`；实际有效字段为 `process`/`start`/`type`/`description` | 修正字段名，以 `process` 替代数字 `status` |

## 详细设计

### B1：前端 cluster 解析器重写

#### 输入格式（实际 acli 输出）

```
[cluster]
ip=
name=
mask=

[ZTA]
zta_enable=0
6.10.0_R2
build 2024-09-11 03:01:31
Linux host-fefcfe7d4542 4.18.0-6.10.0_R2 #1 SMP Wed Sep 4 02:13:52 CST 2024 x86_64 GNU/Linux
[cluster]
mcastaddr=239.208.228.124
authkey=389f8da8833c417084934db5d42bb0d2
2025-09-01 17:17:30    update  | [ ] ---> [6.10.0_R2-001 build 2024-09-11 03:01:31]
Sangfor:aSV/host-fefcfe7d4542(HOST-OS) /sf #
```

Bridge 实际下发命令（含 DONE marker）：
```
acli platform info get; status=$?; printf '\n__HCI_DONE_Q2026042594462_cluster_1777077256922_1777077256922__:%s\n' "$status"
```

所以原始输出还会包含 marker 行（`__HCI_DONE_...__:0`）。

#### 解析策略

1. 剥离 DONE marker 行（`__HCI_DONE_` 开头的行）
2. 剥离命令提示符行（以 `Sangfor:` 或 `#` 结尾的行）
3. 剥离 Shell 提示符/日志行（如 `2025-09-01 17:...` 的 update 历史行）
4. 按 `[section]` 分节收集 `key=value` 行，忽略无法解析的行
5. 额外提取特殊行（版本号行如 `6.10.0_R2`、内核行如 `Linux host-...`）
6. 将所有有效 kv 展平为一级字典存入 DB

#### 提取后 env_data 结构（存入 DB 的内容）

```json
{
  "ip": "",
  "name": "",
  "mask": "",
  "zta_enable": "0",
  "mcastaddr": "239.208.228.124",
  "authkey": "389f8da8833c417084934db5d42bb0d2",
  "hci_version": "6.10.0_R2",
  "kernel": "Linux host-fefcfe7d4542 4.18.0-6.10.0_R2 #1 SMP ...",
  "build_time": "build 2024-09-11 03:01:31"
}
```

#### `_extract_env_info` 同步更新（后端）

后端提取集群信息时映射字段：

```python
{
  "hci_version":   env_data.get("hci_version", "未知"),    # 直接来自前端解析
  "cluster_name":  env_data.get("name", "未知"),           # section [cluster] name=
  "host_count":    env_data.get("host_count", "未知"),     # 当前无法从 platform info get 中采集，保留未知
  "storage_type":  env_data.get("storage_type", "未知"),   # 同上
  "network_config":env_data.get("mcastaddr", "未知"),      # 用组播地址体现网络配置
}
```

### B2：后端 alert 字段映射修复

#### 实际入库的 alert 条目字段

```json
{
  "urgent_type": "紧急",
  "description": "主机（10.25.41.3）虚拟网络控制面服务状态异常：...",
  "start": "2026-04-25 08:25:50",
  "target": "本地控制器",
  "type": "虚拟网络服务状态异常告警",
  "count": 571,
  "unhandle_count": 571
}
```

#### 映射规则

| 输出字段 | 来源字段 | 转换规则 |
|---------|---------|---------|
| `level` | `urgent_type` | `"紧急"` → `"CRITICAL"`，`"普通"` → `"WARNING"`，其他 → `"INFO"` |
| `time` | `start` | 直接取值 |
| `content` | `description` | 截断至 200 字符，避免 prompt 过长 |
| `source` | `target` | 直接取值 |

---

### B3：后端 task 字段映射修复

#### 实际入库的 task 条目字段

```json
{
  "status": 3,
  "process": "失败",
  "type": "系统备份",
  "start": "2026-04-25 00:59:46",
  "description": "执行系统备份失败，系统备份失败：备份存储可用空间小于10GB...",
  "target": "10.25.41.3"
}
```

#### 映射规则

| 输出字段 | 来源字段 | 转换规则 |
|---------|---------|---------|
| `status` | `process` | 直接取字符串（"失败"/"完成"/"执行中"），而非数字 `status` |
| `time` | `start` | 直接取值 |
| `name` | `type` | 直接取值（任务类型名称） |
| `error` | `description` | 截断至 300 字符 |

## 决策依据 (WHY)

### 为什么不改架构（方案 A vs B vs C）

| 方案 | 描述 | 优点 | 缺点 | 结论 |
|------|------|------|------|------|
| **A（选中）**：修复字段映射，保留标准化层 | 在已有映射函数中修正字段名 | 最小改动；架构正确，实现错误；prompt 模板不受 acli 输出格式变化影响 | 无 | ✅ 采用 |
| B：直接把原始 JSON 传给 Agent | 跳过提取层，整体 env_data 注入 prompt | 信息完整 | 原始 30+ 字段混入 prompt：`bcancel`/`sysloged`/`reserved3`/`pid` 等噪声字段消耗 token 并分散注意力；每条 task 原始 ~1500 tokens vs 标准化 ~80 tokens | ❌ 否决 |
| C：前后端各加新字段/表 | 新增 `normalized_env_data` 列 | 原始数据保留 | 数据库 migration、接口变更、工作量 5 倍以上；对现有问题是过度工程 | ❌ 否决 |

### 为什么 parseClusterOutput 需要完全重写而非打补丁

原代码：
```typescript
for (const line of lines) {
  if (line.includes(':')) {
    // 按第一个冒号分割
  }
}
```

`acli platform info get` 完全不使用 `:` 作为 kv 分隔符（使用 `=`），含冒号的行全是时间戳、内核信息等非 kv 行，补丁无法修复根本误判，必须重写。
