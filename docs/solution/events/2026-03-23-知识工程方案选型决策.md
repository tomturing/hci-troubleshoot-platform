## 一、背景与问题

在 HCI 智能排障平台 v2.0 规划阶段，需要确定知识注入与管理的核心架构。当时面临的挑战：

1. **7000+ 在网知识案例**如何有效利用？
2. **SOP 排障手册**如何与 AI 对话流程结合？
3. **跨 Case 经验积累**如何实现？

---

## 二、备选方案对比

| 维度 | 方案 A：认知记忆架构 | 方案 B：因果图推理 | 方案 C：策略蒸馏 |
|------|-------------------|------------------|----------------|
| **核心理念** | LLM 主动管理记忆 | 知识图谱 + 因果推理 | 从成功案例提炼策略 |
| **学术依据** | MemGPT (Stanford 2023) | 因果推理 + 知识图谱 | 模型蒸馏 |
| **实现复杂度** | ⭐⭐⭐ 高 | ⭐⭐⭐⭐ 最高 | ⭐⭐ 中 |
| **Token 效率** | ✅ LLM 自主分页 | ⚠️ 需构建图谱 | ✅ 压缩后注入 |
| **当前可落地** | ⚠️ 需验证 openclaw 能力 | ❌ 需大量知识工程 | ⚠️ 需标注数据 |

## 三、最终决策：双轨知识架构

**决策**：不采用 A/B/C 任一方案，而是采用更务实的"双轨知识架构"。

### 双轨架构核心

```
用户描述故障（S0 意图识别 → category_id 确认）
        │
        ▼ category_id 已知
[第1轨：SOP 直查] sop_document WHERE category_id = X
        │
        ├─ 命中 → 加载 sop_chunk → SOP 步骤引导 → 返回
        │
        └─ 未命中
                │
                ▼
        [第2轨：KBD 检索] kbd_entry WHERE category_id = X
                │
                ├─ 命中 → 注入「历史案例参考」
                │
                └─ 未命中 → 升级人工
```

### 决策依据

1. **精准优于召回**：SOP 优先确保关键场景正确，而非追求大而全
2. **工程可行性**：双轨架构与现有 KB Service 设计一致，可快速落地
3. **渐进演进**：先跑通双轨，后续可向方案 A/B/C 演进

---

## 四、影响范围

- **知识库设计**：采用 `sop_document` + `kbd_entry` 双表结构
- **AI 层设计**：S0 专职分类，S1+ 按分类路由
- **评分机制**：双轨来源分别统计命中率
