# dbmate 迁移机制全面修复方案

## 背景

详见需求文档：[2026-04-08-dbmate迁移机制全面修复需求.md](../requirement/events/2026-04-08-dbmate迁移机制全面修复需求.md)

## 方案概述

分三个层次修复，所有变更在同一 PR 完成：

| 层次 | 内容 | 文件 |
|------|------|------|
| 第一层（紧急） | 修复 sop 表缺失 + 修正 Alembic 废弃表代码 | migrations/ + 0005.py |
| 第二层（流程） | CI 重复 version 检测 + Makefile 快捷命令 + README 规范 | ci/ + Makefile + README |
| 第三层（架构） | 彻底废弃 Alembic 迁移链，锁定 dbmate 单一权威 | helm/ + values.yaml |

## 第一层：紧急修复

### 1.1 修复 sop_document / sop_chunk 缺失

**根因**：两个文件 `20260402001_sop_tables.sql` 和 `20260402001_kb_category_sop_hit_tracking.sql` 共享 version `20260402001`，dbmate 按字典序只执行 `kb_category` 那个，`sop_tables` 永远被跳过。

**操作**：
1. `git rm database/migrations/20260402001_sop_tables.sql`
2. 创建 `database/migrations/20260408001_sop_tables_fix_version.sql`，内容与原文件完全一致，但所有索引和触发器改为幂等写法（`IF NOT EXISTS` + `DO $$ BEGIN/END $$`）
3. 更新 `database/seeds/00_baseline.sql`：添加 `20260407001/002/003` 的历史记录，修正注释

### 1.2 修正 Alembic 0005_schema_repair.py

**问题**：§5-§8 创建了 7 张废弃表（`kb_document`, `kb_chunk`, `kb_sop_node`, `kb_synonym`, `raw_cases`, `knowledge_atoms`, `prompt_audit`），这些表已由 dbmate 路径负责清理。

**操作**：删除 `§5-§8` 代码块，保留 `§1-§4`（customer/case/conversation/assistant_evaluation 补列）和原 §9-§10 重新编号为 §5-§6（tool_result/diagnostic_item）。添加废弃说明注释。

## 第二层：流程加固

### 2.1 CI 重复 version 检测

在 `scripts/ci/check-db-migrations-sync.sh` 的同步检查之前，增加 version 唯一性检查：

```bash
DUPLICATE_VERSIONS=$(ls database/migrations/*.sql \
    | xargs -I{} basename {} \
    | grep -oP '^\d+' \
    | sort | uniq -d)

if [ -n "$DUPLICATE_VERSIONS" ]; then
    # 输出重复文件列表，以非零退出
    exit 1
fi
```

CI `check-db-migrations-sync` job 覆盖此检查，PR 时自动运行。

### 2.2 Makefile 快捷命令

```makefile
db-sync:   # 调用 sync-db-migrations.sh，同步 ConfigMap
db-check:  # 调用 check-db-migrations-sync.sh，本地验证
```

### 2.3 database/README.md 规范更新

- 命名规范：明确禁止日期和序号之间加下划线，并解释原因
- 幂等性规范：列出每种 DDL 类型的强制写法，含错误/正确示例
- 工作流：新增 `make db-sync` 步骤，强调迁移文件和 ConfigMap 在同一 commit
- Alembic 废弃声明：置于文档开头显眼位置

## 第三层：架构层弃用 Alembic

### 3.1 hci-platform-data Alembic K8s Job

| 文件 | 变更 |
|------|------|
| `deploy/helm/hci-platform-data/templates/hooks/db-migrate.yaml` | 容器从 `alembic` 改为 `busybox noop`，Job 名加 `-alembic-deprecated` 后缀，注释说明已废弃 |
| `deploy/helm/hci-platform-data/values.yaml` | `dbMigrate.enabled: false` 注释改为"已废弃，禁止改为 true" |

### 3.2 dbmate 成为唯一权威

- `deploy/helm/hci-platform/templates/hooks/db-migrate-job.yaml`：保持不变，继续作为 ArgoCD PreSync Hook
- 所有未来 schema 变更通过 `database/migrations/` 下的 dbmate 文件进行

## 风险评估

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| 存量 K8s 环境 sop_document 已存在（手动创建过） | 中 | 低 | 新迁移使用 `IF NOT EXISTS`，幂等无副作用 |
| 存量环境 schema_migrations 已有 20260408001 | 极低 | 无 | `ON CONFLICT DO NOTHING` |
| Alembic 0005 在某个环境已执行过废弃表代码 | 低 | 中 | 废弃表已由 dbmate 路径的 20260402003/20260407002/003 清理 |
