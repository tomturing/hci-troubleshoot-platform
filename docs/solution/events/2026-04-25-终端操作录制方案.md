## 背景与需求

详见 [需求事件文档](../../requirement/events/2026-04-25-终端操作录制功能.md)

核心需求：将终端操作（输入命令 + 输出回显）录制并存储到数据库，支持 custom-ui 和 admin-ui 双端回放。

---

## 方案概述 (WHAT)

### 整体架构

采用 **浏览器层录制 + 后端存储** 方案：

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                              终端操作录制数据流                                    │
└─────────────────────────────────────────────────────────────────────────────────┘

【录制阶段】
用户输入 → TerminalPanel.vue (executeCommand)
         → ①拦截输入 → POST /api/terminal/operations (input)
         
SSH输出 → WebSocket (ssh_output)
         → TerminalPanel.vue (writeTerminal)
         → ②拦截输出 → POST /api/terminal/operations (output)

【存储】
Backend API Gateway → PostgreSQL (terminal_operation 表)

【回放阶段】
custom-ui/admin-ui → GET /api/terminal/operations?case_id=xxx
                   → TerminalReplay.vue 渲染回放
```

### 呈现位置

| UI | 用户角色 | 功能侧重 | 权限 |
|----|---------|---------|------|
| **custom-ui** | 工程师 | 排障复盘、引用到对话、生成报告 | 仅查看自己参与的工单 |
| **admin-ui** | 管理员 | 全工单审计、搜索过滤、统计报表 | 查看所有工单 |

## 详细设计

### 架构变更

不涉及微服务新增，仅在现有 api-gateway 和前端模块增加功能。

### 数据模型

新增 `terminal_operation` 表：

```sql
CREATE TABLE terminal_operation (
  id SERIAL PRIMARY KEY,
  case_id VARCHAR(32) NOT NULL,              -- 工单ID
  conversation_id UUID,                       -- 会话ID（可选关联）
  session_id VARCHAR(64),                     -- SSH session ID
  seq_number INT NOT NULL,                    -- 操作序号（同一工单内递增）
  direction VARCHAR(10) NOT NULL,             -- 'input' / 'output'
  command TEXT,                               -- 输入命令（仅 input 类型）
  content TEXT NOT NULL,                      -- 原始内容（含 ANSI 码）
  content_clean TEXT,                         -- 剥离 ANSI 的纯文本（搜索用）
  exit_code INT,                              -- 命令退出码（仅 output 类型）
  diagnostic_stage VARCHAR(10),               -- 诊断阶段 S0-S6（可选）
  created_at TIMESTAMP NOT NULL DEFAULT NOW(),
  
  -- 索引
  INDEX idx_case_seq (case_id, seq_number),
  INDEX idx_case_time (case_id, created_at),
  INDEX idx_content_clean (content_clean)     -- 全文搜索索引
);
```

### 前端实现

#### 录制模块（custom-ui）

**采集点**：`TerminalPanel.vue`

```typescript
// executeCommand() 第230-243行
function executeCommand() {
  const command = localInput.value.trim()
  if (!command || !isConnected.value) return
  
  // ← 新增：记录输入
  recordTerminalOperation({
    direction: 'input',
    command: command,
    content: command,
    seq_number: getNextSeqNumber()
  })
  
  chatStore.sendSSHCommand(command, 'terminal')
  localInput.value = ''
}

// writeTerminal() 第179-188行
function writeTerminal(text: string) {
  if (!text) return
  
  // ← 新增：记录输出（debounced）
  debouncedRecordOutput(text)
  
  fullOutputText.value += text
  if (xterm) {
    xterm.write(normalizeTerminalText(text))
  }
}
```

**批量上传策略**：
```typescript
// 每 500ms 或累计 5KB 输出时上传
const debouncedRecordOutput = debounce(async (text: string) => {
  if (currentCommandSeq) {
    await recordTerminalOperation({
      direction: 'output',
      content: outputBuffer,
      content_clean: stripAnsi(outputBuffer),
      seq_number: currentCommandSeq + 1,
      parent_seq: currentCommandSeq  // 关联输入
    })
    outputBuffer = ''
  }
}, 500)
```

### 关键技术点

#### 1. 序号维护

前端维护 `seq_number`：
- 使用 sessionStorage 缓存当前工单的序号
- 每次操作后递增
- 页面刷新时从后端获取最新序号

#### 2. ANSI 处理

```typescript
// 保留原始 ANSI 码用于回放渲染
// 剔除 ANSI 用于搜索索引
function stripAnsi(text: string): string {
  return text.replace(/\x1b(?:[@-Z\\-_]|\[[0-?]*[ -/]*[@-~])/g, '')
}
```

#### 3. 大输出处理

```typescript
const MAX_OUTPUT_SIZE = 64 * 1024  // 64KB

function truncateOutput(content: string): string {
  if (content.length > MAX_OUTPUT_SIZE) {
    return content.slice(0, MAX_OUTPUT_SIZE) + '\n... [输出过长已截断]'
  }
  return content
}
```

## 决策依据 (WHY)

### 方案选择

| 方案 | 优点 | 缺点 | 评分 |
|------|------|------|------|
| **方案A：浏览器层录制**（选中）| 无需改 Bridge.exe，改动集中前端，易于维护 | 前端逻辑增加，可能影响性能 | ★★★★☆ |
| 方案B：Bridge.exe 层录制 | 更底层，性能更好 | 需重新分发客户端工具，用户需升级 | ★★★☆☆ |
| 方案C：WebSocket 代理层录制 | 对前端透明 | 需改后端 WebSocket 转发逻辑，增加延迟 | ★★☆☆☆ |

### 为什么选择方案A？

1. **改动集中**：仅在 api-gateway 和前端增加功能，不涉及 Bridge.exe 重新分发
2. **易于维护**：前端逻辑可控，调试方便
3. **渐进实施**：可以先实现基础功能，再逐步优化性能
4. **复用现有架构**：利用已有的 WebSocket 连接和 xterm.js

### 为什么不选方案B和C？

**方案B（Bridge.exe）缺点**：
- 需用户重新下载 Bridge.exe，推广成本高
- Go/Rust 代码调试不如前端方便
- 对现有用户（已安装 Bridge）造成中断

**方案C（WebSocket代理）缺点**：
- 需在后端增加 WebSocket 代理层，架构复杂度增加
- 增加网络延迟（每条消息都需转发）
- 对实时性敏感的终端交互影响较大

### 权衡与妥协

1. **性能 vs 易用性**：选择浏览器层录制，可能略影响性能，但用户无需升级工具
2. **实时上传 vs 批量上传**：选择 debounced 批量上传，牺牲实时性换取性能
3. **ANSI 精度**：保留原始 ANSI，回放时渲染；搜索用剔除版本
