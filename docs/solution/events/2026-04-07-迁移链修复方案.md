# 迁移链修复方案

## 背景与问题

### 问题描述

数据库迁移 Job `db-migrate-1` 执行失败：

```
Error: relation "tool_audit_log" does not exist
```

### 根因分析

**迁移执行链断裂**：

```
schema_migrations 记录：
20260326003 ✓ → 创建 tool_audit_log
...
20260402001 ✓

未执行的迁移：
20260402002 ✗ → 从 tool_audit_log 迁移数据 → 失败：表不存在
    ↓
20260402003 ✗ → 删除废弃表 → 被阻塞
    ↓
20260404001 ✗ → 创建 session/system_prompt/tool_definition → 被阻塞
    ↓
20260407002 ✗ → 再次清理废弃表 → 被阻塞
```

**根本原因**：

1. `tool_audit_log` 表不存在：尽管 `20260326003` 标记为已执行，但表实际不存在（可能被删除或备份恢复丢失）

2. 迁移脚本非幂等：`20260402002_migrate_audit_data.sql` 假设源表存在，没有处理表不存在的场景

3. 连锁阻塞：`20260402002` 失败后，后续所有迁移无法执行

### 当前数据库状态

| 指标 | 值 |
|------|-----|
| 表数量 | 21 张（应为 17 张） |
| 缺失表 | session, system_prompt, tool_definition |
| 残留废弃表 | kb_document, kb_chunk, kb_sop_node, kb_synonym, knowledge_atoms, raw_cases, prompt_audit |

## 方案（WHAT）

创建幂等修复迁移 `20260407003_fix_migration_chain.sql`：

### §1 幂等迁移 prompt_audit 数据
- 检查表是否存在后再迁移
- 表不存在时跳过并记录日志

### §2 幂等迁移 tool_audit_log 数据
- 检查表是否存在后再迁移
- 表不存在时跳过（预期行为）

### §3 删除废弃表
- 7 张废弃表全部删除

### §4 创建缺失表
- session, system_prompt, tool_definition

### §5 更新 schema_migrations
- 标记阻塞的迁移为已执行

## 决策依据（WHY）

迁移链设计假设每个迁移都幂等执行，但 `20260402002` 不满足这个假设。需要：

1. 修复当前阻塞状态
2. 确保后续可安全重新执行
3. 同步数据库到正确的 v6.2 状态
