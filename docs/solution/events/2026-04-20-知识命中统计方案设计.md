## 方案概述 (WHAT)

在 `conversation` 表增加 2 个字段记录本次对话实际使用的知识资产引用（`sop_document_id`、`resolved_kbd_entry_id`），通过 case 级去重约束保证 `hit_count` 物化列的统计准确性，修复现有 `kb_category.hit_count` 的重复计数缺陷。

---

## 核心决策依据与澄清过程（WHY）

> 本节详细记录设计讨论过程，是方案最重要的部分。

### D-01：hit_count 应该是「记录下来的」还是「统计出来的」？

**结论：事件是原始真相，计数是派生物（物化缓存）**

- **事件表**（append-only 记录）是不可变的真相来源
- **`hit_count` 计数列**是展示层的物化缓存，写事件时同步 +1
- 两者并存：计数列用于快速展示，原始数据用于深度分析和校验

本方案将「命中事件」记录在 `conversation` 表的新字段中（而非独立事件表），原因见 D-03。

### D-02：hit_count 的原始数据来源是 case 还是 conversation？

**结论：技术事件来自 conversation，但去重粒度必须是 case**

- 知识检索发生在 conversation-service 的诊断推进过程中（S0→S1，S2/S3 检索）
- `case : conversation = 1 : N`（断线重连会创建新 conversation）
- 若按 conversation 计数（现有实现），断线重连会虚增 hit_count
- 因此：**原始事件在 conversation 层产生，但计数时以 case 为去重单位**

### D-03：是否需要新增 kb_usage_event 通用事件表？

**结论：不需要，最小化设计更优**

分析三类知识的关系基数：

| 知识类型 | case 与知识的基数 | 可否放入 conversation 字段 |
|---------|----------------|--------------------------|
| kb_category | 1 : 1（S0 确认唯一分类）| 数据已存在（`category_id` + `case_id`），无需新字段 |
| sop_document | 1 : 0~1（三轨路由取 results[0]，1:1）| ✅ 一个 nullable FK 字段 |
| kbd_entry | 1 : 0~1（S4 确认唯一根因叶节点）| ✅ 一个 nullable FK 字段 |

kb_category 命中数据已完整存在于 `conversation.category_id + case_id`，无需任何新字段。

sop_document 和 kbd_entry 都是「一个 case 最终对应 0 或 1 条知识」的 1:1 关系（非 M:N），用单个字段表达最精确。

若使用通用事件表（knowledge_type + knowledge_id），则：
- 失去外键约束（不能直接 FK 多张表）
- 增加 JOIN 复杂度
- 与故障树「case 对应唯一叶节点」的领域模型背道而驰

### D-04：为何 kbd_entry 不用 M:N 联结表？

**初始方案曾考虑 `conversation_kbd_usage` M:N 联结表**，后经领域模型澄清放弃：

- 故障树模型中，一个 case 最终只对应一个叶节点（唯一根因）
- S2/S3 阶段的 top-N KBD 检索是「候选」，不是「结论」
- **诊断结论**（S4 根因）才是真正的命中，语义清晰，是 1:0~1 关系
- M:N 联结表混淆了「检索候选」和「诊断结论」两个概念

### D-05：SOP hit_count 的统计边界

SOP 文档可能覆盖 L2 子树（如"开机失败"），但 `conversation.category_id` 最终确认的是 L3 叶子（如"内存不足"）。因此：

- `sop_document_id` 在 S1 阶段按 `category_id` 查 SOP（`category_id` 可能是 L2 或 L3）
- hit_count 统计的是「有多少 case 使用了该 SOP」，不要求 category 精确匹配到叶节点
- 这与业务语义一致：该 SOP 指导了多少次排障过程

### D-06：AI 自动推断 vs 人工标注 resolved_kbd_entry_id

**结论：AI 自动推断 + admin-UI 可编辑修正**

- S4 根因确认时，AI 结构化输出携带 `kbd_entry_id`（从 S2/S3 检索的候选中）
- AI 推断不可避免存在误差
- 管理员在 admin-UI 可手动修正（修正时需重新计算 hit_count：减旧值 +1，加新值 +1）

## 详细设计

### 数据模型变更

#### 变更 1：`conversation` 表增加 2 个字段

```sql
ALTER TABLE conversation
    ADD COLUMN sop_document_id integer
        REFERENCES sop_document(id) ON DELETE SET NULL,
    ADD COLUMN resolved_kbd_entry_id bigint
        REFERENCES kbd_entry(id) ON DELETE SET NULL;

COMMENT ON COLUMN conversation.sop_document_id IS
    'S1 阶段按 category_id 命中的 SOP 文档 ID（FK → sop_document.id）；
     NULL 表示该分类无对应 SOP 或尚未到达 S1。
     每个 case 的第一个到达 S1 的 conversation 写入此字段，
     重复 conversation（断线重连）不覆盖已写入的值。';

COMMENT ON COLUMN conversation.resolved_kbd_entry_id IS
    'S4 根因确认后，AI 推断该 case 对应的 KBD 叶节点 ID（FK → kbd_entry.id）；
     NULL 表示新问题未收录，或尚未到达 S4。
     admin-UI 可手动修正（修正时重新计算 hit_count）。';
```

#### 变更 2：`kbd_entry` 增加 `hit_count`

```sql
ALTER TABLE kbd_entry
    ADD COLUMN hit_count integer NOT NULL DEFAULT 0;

COMMENT ON COLUMN kbd_entry.hit_count IS
    '命中计数：有多少个唯一 case_id 的 conversation.resolved_kbd_entry_id = 此条目。
     物化列，写入 conversation.resolved_kbd_entry_id 时原子 +1（case 级去重）。
     校验 SQL：SELECT COUNT(DISTINCT case_id) FROM conversation WHERE resolved_kbd_entry_id = id;';
```

#### 变更 3：`sop_document` 增加 `hit_count`

```sql
ALTER TABLE sop_document
    ADD COLUMN hit_count integer NOT NULL DEFAULT 0;

COMMENT ON COLUMN sop_document.hit_count IS
    '命中计数：有多少个唯一 case_id 的 conversation.sop_document_id = 此文档。
     物化列，写入 conversation.sop_document_id 时原子 +1（case 级去重）。
     校验 SQL：SELECT COUNT(DISTINCT case_id) FROM conversation WHERE sop_document_id = id;';
```

#### 无变更：`kb_category.hit_count`

字段已存在，仅修正写入逻辑（case 级去重），无 Schema 变更。

### 触发链路设计

#### 链路 A：kb_category.hit_count（修复）

```
S0→S1 转换（conversation_service._update_conversation_category）
    ↓
查询：该 case_id 是否有其他 conversation 已写入相同 category_id？
    ├── 是（断线重连）→ 仅更新当前 conversation.category_id，跳过 hit +1
    └── 否（首次确认）→ 更新 category_id + POST /api/kb/categories/{code}/hit（hit +1）
```

**关键代码位置**：`conversation-service/app/services/conversation_service.py::_update_conversation_category`

#### 链路 B：sop_document.hit_count（新增）

```
链路 A 完成（category_id 已写入 conversation）后
    ↓
KnowledgeRetriever.retrieve() → track == "sop" 时提取 results[0]["id"]
    ↓
conversation_service 将 sop_document_id 写入 audit_meta
    ↓
_update_conversation_category（或新增 _update_sop_usage）：
    查询：该 case_id 是否有其他 conversation 已写入相同 sop_document_id？
    ├── 是 → 仅写字段，跳过 hit +1
    └── 否 → 写字段 + POST /api/kb/sop/{id}/hit（hit +1）
```

**新增 API**：`kb-service POST /api/kb/sop/{document_id}/hit`

#### 链路 C：kbd_entry.hit_count（新增）

```
S4 根因确认（diagnostic_item type=root_cause INSERT 时）
    ↓
AI 结构化输出解析提取 kbd_entry_id（nullable）
    ↓
conversation_service._update_resolved_kbd：
    写入 conversation.resolved_kbd_entry_id
    若非 NULL：POST /api/kb/kbd/{id}/hit（无需去重，字段本身是 case 级唯一的）
```

**新增 API**：`kb-service POST /api/kb/kbd/{kbd_id}/hit`

#### 链路 D：admin-UI 手动修正（新增）

```
管理员修改 resolved_kbd_entry_id
    ↓
PATCH /api/admin/cases/{case_id}/resolved_kbd（新增接口）
    ↓
conversation-service：
    旧值非 NULL → POST /api/kb/kbd/{旧id}/hit/decrement（hit -1）
    写入新 conversation.resolved_kbd_entry_id
    新值非 NULL → POST /api/kb/kbd/{新id}/hit（hit +1）
```

---

### 接口设计

#### 新增：`POST /api/kb/sop/{document_id}/hit`

```
请求：Bearer token（internal）
响应：{ success: bool, document_id: int, hit_count: int }
实现：UPDATE sop_document SET hit_count = hit_count + 1 WHERE id = :id RETURNING hit_count
```

#### 新增：`POST /api/kb/kbd/{kbd_id}/hit`

```
请求：Bearer token（internal）
响应：{ success: bool, kbd_id: int, hit_count: int }
实现：UPDATE kbd_entry SET hit_count = hit_count + 1 WHERE id = :id RETURNING hit_count
```

#### 新增：`POST /api/kb/kbd/{kbd_id}/hit/decrement`（供 admin 修正链路使用）

```
请求：Bearer token（internal）
响应：{ success: bool, kbd_id: int, hit_count: int }
实现：UPDATE kbd_entry SET hit_count = GREATEST(0, hit_count - 1) WHERE id = :id RETURNING hit_count
```

#### 新增：`PATCH /api/admin/cases/{case_id}/resolved_kbd`

```
请求：{ kbd_entry_id: int | null }  + Bearer token
响应：{ success: bool, case_id: str, old_kbd_id: int|null, new_kbd_id: int|null }
实现：conversation-service，更新 conversation.resolved_kbd_entry_id + 触发 hit count 调整
```

---

### admin-UI 变更

**CategoryManageView 分类详情面板**：

- `sop.hit_count`：从 `sop_document.hit_count` 读取（改为真实数据，不再是 0）
- `kbd.hit_count`：从 `kbd_entry.hit_count` 读取（改为真实数据，不再是 0）

**新增：工单详情或分类详情中的 KBD 关联编辑**：

- 展示 `resolved_kbd_entry_id`（当前 AI 推断的 KBD 条目）
- 提供编辑入口：从该分类的已发布 KBD 列表中选择正确条目
- 保存时调用 `PATCH /api/admin/cases/{case_id}/resolved_kbd`

## 方案比较

| 方案 | 描述 | 优点 | 缺点 | 评分 |
|------|------|------|------|------|
| **A（选中）** | conversation 表增 2 字段 + 物化 hit_count | 最小变更；FK 保证数据完整性；1:0~1 语义精确；可校验 | 需修改 conversation-service 核心逻辑 | ★★★★★ |
| B | 新增通用 kb_usage_event 事件表 | 灵活，支持 M:N | 无 FK 约束；与领域模型矛盾（1:1 误用 M:N）；过度设计 | ★★★☆☆ |
| C | 仅统计查询，不维护物化列 | 无 Schema 变更 | 统计查询性能差；无法快速展示；缺乏写入时机记录 | ★★☆☆☆ |
