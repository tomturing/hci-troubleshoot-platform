# 废弃表再次清理方案

## 背景与需求

PR #108 的 `20260407001_schema_repair.sql` 修复 schema 漂移时，误重建了已在 `20260402003_drop_deprecated_tables.sql` 中删除的 7 张废弃表。

**根因**：Copilot 创建迁移文件时，直接从数据库设计文档复制了所有 CREATE TABLE 语句，没有排除废弃表清单中的表。

**后续问题**：PR #113 合并后，ConfigMap 未同步更新，导致 ArgoCD 部署后迁移不会执行。

**错误重建的表**：
- `kb_document`, `kb_chunk`（已被 `kbd_entry` 替代）
- `kb_sop_node`, `kb_synonym`（已被 `sop_document/sop_chunk` 替代）
- `raw_cases`, `knowledge_atoms`（数据管道已废弃）
- `prompt_audit`（已合并入 `audit_log`）

## 方案（WHAT）

### 1. 创建清理迁移

新建 `20260407002_cleanup_deprecated_tables.sql`，删除 7 张错误重建的废弃表。

### 2. 修正 schema_repair.sql

删除 `20260407001_schema_repair.sql` 中 §5-§8 创建废弃表的代码块，更新注释说明不创建废弃表。

### 3. 同步迁移到 ConfigMap

运行 `scripts/ci/sync-db-migrations.sh` 将所有迁移同步到 Helm ConfigMap。

### 4. 添加 CI 检查

新增 `check-db-migrations-sync` CI job，防止迁移遗漏同步。

## 决策依据（WHY）

废弃表清单已在数据库设计文档 v6.2 中明确列出，不应再次创建。需要：
1. 清理已创建的废弃表
2. 修正迁移文件防止未来误重建
3. 自动化同步机制防止 ConfigMap 遗漏
