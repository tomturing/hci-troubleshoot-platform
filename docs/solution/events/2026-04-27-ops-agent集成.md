# ops-agent 集成方案

## 背景与需求
参考需求事件文档：[2026-04-27-ops-agent集成.md](../requirement/events/2026-04-27-ops-agent集成.md)

## 方案概述

### 核心架构
采用**松耦合集成方案**，通过OpenAI-compatible API适配，最小侵入现有代码。

### 设计原则
1. **最小侵入**：尽可能少修改现有代码，主要通过适配器层实现集成
2. **协议优先**：复用现有AIAssistantClient协议，不引入新协议
3. **可回滚**：提供完整的回滚方案，5分钟内完成回滚
4. **独立演进**：两个项目仍可独立开发和发布

## 详细设计

### 架构变更
```
HCI-Troubleshoot-Platform
       ↓
AIAssistantClient (现有协议)
       ↓
OpenAI兼容API端点 (NEW)
       ↓
Ops-Agent (独立部署)
```

### 受影响的模块
- **OA端：新增server模块，提供OpenAI-compatible API
- **HCI-TP端：新增OpsAgentAssistant适配器

### 关键技术点
1. 复用现有的AIAssistantClient协议
2. OA暴露OpenAI-compatible API
3. 通过适配器层实现集成

## 决策依据

### 方案比较
| 方案 | 优点 | 缺点 | 评分 |
|------|------|------|------|
| 松耦合（选中） | 风险低、改动小、可演进 | 增加一次网络调用 | ★★★★★ |
| 深度集成 | 性能更好 | 改动大、风险高 | ★★☆☆☆ |
| 完全合并 | 统一架构 | 架构冲突、不可回滚 | ★☆☆☆☆ |

### 为什么选择松耦合方案
1. **第一性原理分析**：两个项目解决不同问题，是互补关系
2. **风险控制**：可以快速回滚，不影响现有功能
3. **可维护性**：两个项目可以独立演进
4. **复用性**：复用现有协议，无需重新设计
