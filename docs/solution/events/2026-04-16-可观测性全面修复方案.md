# 可观测性全面修复方案

## 背景与需求

参见：[docs/requirement/events/2026-04-16-可观测性全面修复需求.md](../../../requirement/events/2026-04-16-可观测性全面修复需求.md)

## 方案概述

针对 14 项缺陷，分四个优先级层次修复：
- **P0**（2项配置 + 2项代码）：解决数据丢失、联动失效、告警哑火
- **P1**（4项）：链路 ID 归一、KB 指标上报、日志持久化
- **P2**（3项）：配置纠错、Grafana 安全加固
- **P3**（3项）：代码质量、Dashboard 代码化

## 详细设计

### Fix #3 — Tempo 持久化卷

**根因**：`tempo-config.yaml` 中 `storage.trace.local.path: /tmp/tempo/blocks`，`/tmp` 在容器重启后清空。

**方案**：
1. 将存储路径改为 `/var/tempo/blocks`
2. 在 `docker-compose-obs.yml` 的 Tempo service 挂载 named volume `tempo_data:/var/tempo`

### Fix #4 — Promtail JSON Pipeline

**根因**：`promtail-config.yaml` 只有 `relabel_configs`，没有 `pipeline_stages`，Loki 无法按 `trace_id`、`level` 字段索引日志。

**方案**：在 `scrape_configs` 下添加 `pipeline_stages`:
```yaml
pipeline_stages:
  - docker: {}    # 解析 Docker JSON 外层（timestamp/log/stream）
  - json:
      expressions:
        level: level
        trace_id: trace_id
        service: service
  - labels:
      level:
      trace_id:
      service:
```

这样 Grafana Tempo 的 `tracesToLogsV2.filterByTraceID: true` 就能精确过滤。

### Fix #8 — Loki 持久化卷

**根因**：Loki service 没有 volume，重启后日志丢失。

**方案**：添加 `loki_data:/loki` volume 挂载（Loki 默认数据目录 `/loki`）。

### Fix #9 — serviceMap 数据源

**根因**：`datasources.yml` 中 `serviceMap.datasourceUid: loki`，Tempo 的 Service Graph 指标写入 Prometheus，应该 `datasourceUid: prometheus`。

### Fix #10 — Tempo metrics_generator remote_write

**根因**：`tempo-config.yaml` 启用了 metrics_generator 但没有 `storage.remote_write`，生成的 span-metrics 无处落地。

**方案**：添加：
```yaml
storage:
  remote_write:
    - url: http://prometheus:9090/api/v1/write
      send_exemplars: true
```
同时在 `prometheus.yml` 里开启 remote_write 接收（`--web.enable-remote-write-receiver`）。

### Fix #1 — Pod 池告警指标名

**根因**：
- `alert_rules.yml` 使用 `hci_warm_pool_size{state="idle"}`
- 代码中实际注册的是 `hci_pod_pool_idle`（无 `state` label）
- 且 scheduler-service 从未 import `POD_POOL_IDLE/ACTIVE` 指标

**方案**：
1. 修改 `alert_rules.yml` 使用正确的指标名 `hci_pod_pool_idle` 和 `hci_pod_pool_active`
2. 在 `scheduler-service/app/services/pod_pool.py` 中 import 并在空闲/活跃池变化时 `.set()` 指标
3. 在 `prometheus.yml` 添加 `kb-service` 的 scrape job（文档中声称已实现但配置缺失）

### Fix #2 — HTTP 请求指标

**根因**：`HighApiLatency` 和 `HighErrorRate` 告警依赖 `http_request_duration_seconds_bucket` 和 `http_requests_total`，这两个指标从未被注册。

**方案**：在 `shared/utils/metrics.py` 中添加标准 HTTP 指标，在各服务 `main.py` 添加请求计时中间件（Starlette middleware）。

> **为何不用 `prometheus-fastapi-instrumentator`**：避免引入新依赖；用标准 `prometheus_client` + Starlette `BaseHTTPMiddleware` 即可，与现有架构一致。

### Fix #5 — X-Trace-Id 与 OTel 对齐

**根因**：`TraceIDMiddleware` 用 `uuid.uuid4()` 生成 trace_id，与 OTel Span Context 无关。

**方案**：在 `call_next(request)` 之后，从 OTel 当前 Span 上下文读取真实 trace_id，回写到响应头。若 OTel Span 无效（如健康检查），则降级为 `uuid.uuid4()`。

### Fix #6 — KB 检索指标上报

**根因**：`KB_SEARCH_DURATION_SECONDS` 已定义但 kb-service 未 import 使用。

**方案**：在 kb-service 的向量检索核心路径（`search_similar_chunks`）添加 `with KB_SEARCH_DURATION_SECONDS.time()` 计时。

### Fix #7 — node-exporter / kube-state-metrics

**方案**：在 `prometheus.yml` 添加注释形式的 scrape 配置，说明需要先部署对应 exporter。当前环境若未部署则不影响其他功能。

### Fix #11 — Grafana 安全加固

**方案**：
- `admin` 密码改为 `GF_SECURITY_ADMIN_PASSWORD` 环境变量（docker-compose 读取 `.env.local`）
- 关闭 `GF_AUTH_ANONYMOUS_ENABLED` 或改为只读角色，并加 `.env.local` 到 `.gitignore`

### Fix #12 — LoggingInstrumentor 注释澄清

**方案**：在 `otel.py` 中补充注释说明 `LoggingInstrumentor` 只影响三方库日志，业务日志走 `StructuredLogger` 自定义路径。

### Fix #13 — StructuredLogger.exception() 方法

**方案**：添加 `exception()` 方法，调用 `traceback.format_exc()` 将完整堆栈写入 `traceback` 字段。

### Fix #14 — Grafana Dashboard 代码化

**方案**：在 `grafana/provisioning/dashboards/` 下添加：
- `dashboards.yml`（provider 配置）
- `hci-overview.json`（系统综合大盘：AI TTFT、错误率、Pod 池状态）
