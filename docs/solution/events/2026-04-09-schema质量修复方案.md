# 方案：desired_schema.sql 质量修复（Batch 1 + 2）

> 关联需求：[2026-04-09-schema质量修复需求](../requirement/events/2026-04-09-schema质量修复需求.md)
> 关联 PR：#126

## 背景与需求

基于两份 Schema 审核报告（`docs/solution/desired-schema-review.md` 与 `docs/solution/database-schema-review.md`）识别出 11 个真实存在的问题。本方案修复其中优先级 P0/P1 共 9 项，其余 5 项经评审驳回。

## 方案概述

直接修改 `database/desired_schema.sql`（唯一权威 Schema 定义），分两批：
- **Batch 1**（P0 缺陷）：影响数据一致性和系统稳定性的问题
- **Batch 2**（P1 重要问题）：影响性能和可维护性的问题

## 详细设计

### Batch 1 — 缺陷修复

#### D-001: generate_case_id() 并发安全

**问题**: `SELECT COUNT(*) + 1 INTO v_seq` 无锁，两个并发请求同时读到相同 COUNT 时产生重复 case_id。

**方案选择**:
| 方案 | 机制 | 优点 | 缺点 | 选择 |
|------|------|------|------|------|
| A: `pg_advisory_xact_lock` | 事务级排他锁 | 自动释放，代码简单 | 同天请求串行 | ✅ 选中 |
| B: `SELECT ... FOR UPDATE` | 行锁（锁最大ID行） | 标准SQL | 空表无行可锁，需额外处理 | ❌ |
| C: Redis 分布式锁 | 服务层解决 | 高并发性能好 | 引入跨服务依赖，复杂度高 | ❌ |

**修复**:
```sql
PERFORM pg_advisory_xact_lock(hashtext('generate_case_id:' || v_today));
SELECT COALESCE(MAX(CAST(SUBSTRING(case_id FROM 10 FOR 5) AS INTEGER)), 0) + 1
    INTO v_seq FROM "case" WHERE case_id LIKE 'Q' || v_today || '%';
```
锁 key 按日期区分，不同日期并行无干扰，同天串行化。

#### D-002: 向量索引

**问题**: 3 个 `embedding vector(1536)` 字段无索引，语义检索是全表扫描。

**方案选择**:
| 类型 | 适用场景 | 内存 | 精度 |
|------|---------|------|------|
| IVFFlat (lists=100) | 中等数据量（<100万） | 低 | 中 |
| HNSW | 大规模数据（>100万） | 高 | 高 |

当前数据量：`kb_category` ~235条，`kbd_entry`/`sop_chunk` 初期较小，选 **IVFFlat lists=100**。

```sql
CREATE INDEX IF NOT EXISTS idx_kb_category_embedding ON kb_category
    USING ivfflat (embedding vector_cosine_ops) WITH (lists = 100);
-- kbd_entry / sop_chunk 同上
```

> ⚠️ IVFFlat 需要 >1000 条数据才能有效，入库初期全表扫描性能实际差别不大。

#### D-005: session.user_id 外键

```sql
CONSTRAINT fk_session_user_id FOREIGN KEY (user_id) REFERENCES "user" (user_id) ON DELETE CASCADE
```

> ⚠️ 迁移前置检查：`DELETE FROM session WHERE user_id NOT IN (SELECT user_id FROM "user");`

#### D-006: CHECK 约束（7 个）

所有 CHECK 均为 NULL-safe（`IS NULL OR (条件)`），兼容历史空值：

| 表 | 字段 | 约束 |
|----|------|------|
| `assistant_evaluation` | `score` | `[1,5]` |
| `assistant_evaluation` | `composite_score` | `[0,100]` |
| `conversation` | `diagnostic_stage` | `IN ('S0'-'S6')` |
| `diagnostic_item` | `probability` | `[0,1]` |
| `diagnostic_item` | `stage` | `IN ('S2'-'S5')` |
| `tool_result` | `risk_level` | `[1,3]` |
| `tool_definition` | `risk_level` | `[1,3]`（NOT NULL，无需 NULL-safe） |

### Batch 2 — 重要问题

#### D-003: 全文检索分词

```sql
-- 修改前
CREATE INDEX ... USING GIN (to_tsvector('english', content));
-- 修改后
CREATE INDEX ... USING GIN (to_tsvector('simple', content));
```

`simple` 按标点/空格分词，不做语言相关的词根还原，适合中文内容。如生产环境安装了 `zhparser` 可改为 `'chinese'`。

#### D-004: diagnostic_item 唯一约束

```sql
CONSTRAINT uq_diagnostic_item_conv_type_seq UNIQUE (conversation_id, "type", seq)
```

防止 AI 多次写入同一 `(conversation_id, type, seq)` 组合，避免 Prompt 构建时遍历到重复条目。

#### P2-2: updated_at 触发器补全

`kbd_entry` 和 `sop_document` 均有 `updated_at` 字段，字段注释也说明"由触发器维护"，但触发器实际缺失：

```sql
DROP TRIGGER IF EXISTS update_kbd_entry_updated_at ON kbd_entry;
CREATE TRIGGER update_kbd_entry_updated_at
    BEFORE UPDATE ON kbd_entry
    FOR EACH ROW EXECUTE FUNCTION update_updated_at_column();
-- sop_document 同上
```

#### P1-1: 移除冗余索引

`idx_user_client_id`：`client_id` 字段已有 `UNIQUE` 约束，PostgreSQL 自动为 UNIQUE 约束创建 B-tree 索引，手动创建的同字段索引完全冗余。

#### O-001: 移除低价值 trace_id 裸索引

保留策略：
- **保留**：`case` / `conversation` / `message`（核心业务表，可能通过 trace_id 反查）
- **保留**（已有 WHERE 条件）：`tool_result` / `audit_log`（`WHERE trace_id IS NOT NULL`）
- **移除**：`user` / `environment` / `assistant_evaluation`（非热路径，链路追踪通过 Tempo/Loki 而非 DB）

## Atlas 迁移注意事项

本 PR 只修改了 `desired_schema.sql`（声明式期望状态）。实际 Atlas 迁移文件需要在有 PostgreSQL 连接的环境中执行：

```bash
export DATABASE_URL="postgres://hci_admin:<password>@localhost:5432/hci_troubleshoot"
atlas migrate diff --env local db-schema-v6.5-fixes
# 生成后审查 database/atlas-migrations/ 下的新文件
# 确认无破坏性变更后提交
```

**迁移前必须执行的清理**（仅用于已有数据的环境）：
```sql
-- D-005 前置清理：消除孤儿 session 记录
DELETE FROM session WHERE user_id NOT IN (SELECT user_id FROM "user");

-- D-006 前置清理：修正历史脏数据（如有）
UPDATE assistant_evaluation SET score = NULL WHERE score < 1 OR score > 5;
UPDATE assistant_evaluation SET composite_score = NULL WHERE composite_score < 0 OR composite_score > 100;
UPDATE tool_result SET risk_level = NULL WHERE risk_level < 1 OR risk_level > 3;
UPDATE tool_definition SET risk_level = 1 WHERE risk_level < 1 OR risk_level > 3;
```
