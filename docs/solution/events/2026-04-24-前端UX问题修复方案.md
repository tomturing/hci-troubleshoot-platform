## 详细设计

### Bug 1 修复：SSH 命令 case_id 对齐

**问题根因**：

Terminal Bridge（`terminal_bridge/main.go`）用 `case_id` 字符串作为 map key 存储 SSH session：

```go
case "ssh_input":
    if s := b.get(msg.CaseID); s != nil {
        s.send(msg.Data)
    }
    // 找不到时静默丢弃，无任何错误输出
```

SSH 连接建立（`buildConnectMessage`）使用 `case_id: 'ssh-create-temp'`，但 `runCommand()` 发命令时使用 `case_id: createdCaseId.value || 'temp'`（如 `Q2026042497781`），key 不匹配导致命令被静默丢弃。

**修复**：`runCommand()` 改为 `case_id: 'ssh-create-temp'`，与连接时对齐。

**附带修复 A（超时）**：三条采集命令超时从 `10000ms` 改为 `30000ms`，避免在大型集群下误判超时。

**附带修复 B（错误入库）**：原来采集失败时会将 `{ parse_error: 'xxx', raw: '' }` 写入数据库，干扰 AI 推理。修复后失败项保持 `null`，跳过 upsert。

### Bug 2 修复：顶部工单 badge 即刻显示

**问题根因**：`CaseCreateDialog.vue` SSH 确认流程调用了 `caseApi.confirm()`，但只更新了组件内的 `createdCaseId.value`，没有写入 `chatStore.currentCase`。`App.vue` 用 `v-if="chatStore.currentCase"` 控制 badge，因此不显示。

**修复**：在 `runSshAndCreateCase()` 确认工单后，添加 `chatStore.currentCase = confirmed.data`。

### Bug 3 修复：AI 助手无反应

**问题根因**：`confirmCreateCase()` 调用 `createConversation()` 后，若出现异常（如 `conversationApi.create()` 失败），catch 块打印系统消息后直接 return，不重新抛出。导致 `conversationId.value` 仍为 null，`streamAIResponse()` 第一行 `if (!conversationId.value) return` 静默退出，AI 无响应。

**修复**：`createConversation()` catch 块添加 `throw e`，异常向上传播，阻止 `streamAIResponse` 执行。

### Bug 5 修复：有数据时仍显示"未采集环境数据"

**问题根因**：`EnvironmentSummary.vue` 用 `v-if="chatStore.environmentContext"` 判断是否显示数据，而 `environmentContext` 只在 SSH 采集流程结束时被填充。以下三条路径设置了 `currentCase` 但从不调用 `collectEnvironmentData()`：

| 路径 | 触发场景 |
|------|---------|
| `resumePendingCase()` | 页面刷新后恢复未关闭工单 |
| `switchToCase()` | 从历史工单列表切换 |
| `confirmCreateCase()` | 纯文字创建工单（无 SSH） |

**修复**：在三处添加 `collectEnvironmentData(caseId).catch(() => {})` fire-and-forget 调用。
`switchToCase` 额外先清空 `environmentContext.value = null`，避免展示旧工单的残留数据。

**后端说明**：`GET /environments/case/{id}/context` 始终返回 `EnvironmentContextResponse` 对象（即使无数据），不会 404，因此调用永远不会误报错误。

## 决策依据

### 方案选择

| 方案 | 优点 | 缺点 | 评分 |
|------|------|------|------|
| **纯前端修复（选中）** | 改动最小、不影响后端接口 | 无 | ★★★★★ |
| 修改 Bridge 以支持多 key | 更灵活 | 改动大，引入新风险 | ★★☆☆☆ |
| 修改后端 `/environments` 返回 404 | 触发前端错误路径 | 破坏性变更，影响其他调用方 | ★☆☆☆☆ |

### 为什么选纯前端修复？

- Bug 根因都在前端逻辑（key 不匹配、状态未同步、异常被吞、未触发数据加载），不需要修改后端协议
- 改动集中在 2 个文件，易于 review 和回滚
- fire-and-forget 模式对于环境数据加载是合适的——失败不应阻塞对话主流程
