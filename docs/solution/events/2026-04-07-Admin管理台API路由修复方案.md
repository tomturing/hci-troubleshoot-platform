# Admin 管理台 API 路由修复方案

## 背景与需求

Admin 管理台分类基线、知识审核、KBD 审核三个页面的数据加载全部失败。详见 [需求文档](../../requirement/events/2026-04-07-Admin管理台API路由修复需求.md)。

## 方案（WHAT）

### 1. 分类基线修复

**api-gateway 新增 `categories_router`**

在 `backend/api-gateway/app/routes/kb.py` 新增独立路由器，前缀 `/api/kb/categories`，代理以下端点到 kb-service：

| 方法 | 网关路径 | kb-service 路径 |
|------|---------|----------------|
| GET | `/api/kb/categories` | `/api/kb/categories` |
| GET | `/api/kb/categories/stats` | `/api/kb/categories/stats` |
| PUT | `/api/kb/categories/{code}` | `/api/kb/categories/{code}` |
| POST | `/api/kb/categories/{code}/hit` | `/api/kb/categories/{code}/hit` |
| POST | `/api/kb/categories/import` | `/api/kb/categories/import` |

YAML 导入端点使用 raw body + content-type 透传方式处理 multipart 文件上传。

**前端响应解析修复**

`CategoryManageView.vue` 的 `fetchCategories()` 修改：
- 请求参数：`?active_only=false&include_stats=true` → `?grouped=true`
- 响应解析：`data.data.domains`（不存在）→ `data.domains`（dict → DomainGroup[] 转换）

### 2. 知识审核清理

`knowledge_atoms` 表已在迁移 `20260402003` 中删除，相关前端代码应一并清理：

- 删除 `frontend/admin/src/views/KnowledgeReviewView.vue`
- 从 `router/index.ts` 移除 `/knowledge/review` 路由
- 后续菜单 order 顺延（KB 审核 5→4，用户管理 6→5，系统监控 7→6）

### 3. KBD 审核实现

**kb-service 新增端点**

在 `backend/kb-service/app/routes/admin.py` 的 `kbd_router` 上新增：

| 方法 | 路径 | 功能 |
|------|------|------|
| GET | `/api/admin/kbd/pending` | 分页查询（支持 status + category_id 过滤） |
| PATCH | `/api/admin/kbd/{kbd_id}/reject` | 拒绝条目（状态 draft → rejected） |

**api-gateway 新增 `kbd_router`**

前缀 `/api/v1/kbd`，代理以下端点到 kb-service：

| 方法 | 网关路径 | kb-service 路径 | 备注 |
|------|---------|----------------|------|
| GET | `/api/v1/kbd/pending` | `/api/admin/kbd/pending` | 列表查询 |
| PATCH | `/api/v1/kbd/{id}/approve` | `POST /api/admin/kbd/{id}/approve` | 方法转换 PATCH→POST |
| PATCH | `/api/v1/kbd/{id}/reject` | `PATCH /api/admin/kbd/{id}/reject` | 拒绝 |

**前端修复**

`KbdReviewView.vue` 的 `fetchPending()` 补充 `authHeader`（之前缺失鉴权头）。

## 决策依据（WHY）

### 为什么 api-gateway 使用独立 router 而非改 `/api/v1/kb` 前缀？

分类基线前端直接请求 `/api/kb/categories`（无 v1 前缀），KBD 前端请求 `/api/v1/kbd/pending`（无 kb 中间层）。改虽然可以统一到 `/api/v1/kb` 前缀，但需要同时改动多个前端组件。风险最低的方案是在网关侧按前端实际请求路径建立独立 router。

### 为什么不保留 KnowledgeReviewView 组件？

`knowledge_atoms` 表已明确废弃（`20260402003` 迁移删除），后端无 API 实现计划，保留前端组件只会带来维护负担和用户困惑。

### 为什么 KBD 列表端点用原生 SQL 而非 ORM？

与 `admin.py` 中已有的 `approve_kbd_entry` 保持一致风格（均使用 `text()` 原生 SQL），避免在同一文件混用两种查询模式。列表查询使用 `LEFT(content_md, 300)` 截断大文本字段，ORM 反而更难表达。

## 验收标准

1. 访问 `/admin/category` — 分类基线数据正常加载，按域分组显示
2. 侧边栏无"知识审核"菜单项
3. 访问 `/admin/knowledge/kbd-review` — KBD 条目列表正常加载（draft 状态）
4. KBD 审核通过/拒绝操作正常工作
