# SOP Agent 节点选择与聚类分析

> 本文档详细分析 `sop-agent` 项目中节点选择与聚类的实现逻辑。
>
> 源代码位置：`/mnt/d/aihci/sop-agent/scripts/select_sop_nodes.py`

---

## 一、核心目标

### 1.1 要解决的问题

分类树是多层结构，案例最终被归类到叶子节点。**但 SOP 不一定要在叶子节点生成**：

```
分类树层级示例：
云计算-排障                              (depth=2)
├─ 虚拟机                                (depth=3)
│  ├─ 虚拟机生命周期                      (depth=4)
│  │  ├─ 虚拟机管理                       (depth=5)
│  │  │  ├─ 虚拟机开机失败                 (depth=6, 叶子)
│  │  │  ├─ 虚拟机关机失败                 (depth=6, 叶子)
│  │  │  └─ ...
│  │  └─ ...
│  └─ ...
└─ ...
```

**核心问题**：在分类树的哪个层级产出 SOP 最合适？

| 场景 | 问题 | 决策 |
|------|------|------|
| 叶子节点案例数只有 5 个 | 太少，不值得单独生成 SOP | 向上合并到父节点 |
| 叶子节点案例数有 200 个 | 太多，单个 SOP 太复杂 | 内部聚类，拆分多个 SOP |
| 中间节点有 50 个案例，子节点各 25 个 | 可以在当前节点生成，也可以在子节点生成 | 根据熵值判断 |
| 分类质量差（校验率高） | 分类可能不准确 | 需要内部聚类重新梳理 |

### 1.2 与 SOP 聚合的区别

| 方面 | 节点选择与聚类 | SOP聚合 |
|------|---------------|---------|
| **核心问题** | 在分类树的**哪个层级**产出 SOP？ | **如何生成** SOP 内容？ |
| **决策维度** | 分类树结构上的位置选择 | 案例内容的语义合并 |
| **输入** | 分类结果（案例 → 叶子节点映射） | 结构化事实卡（case_facts.jsonl） |
| **输出** | 节点列表 + 案例归属表 | 标准化 SOP 流程（node_sops.jsonl） |
| **处理粒度** | 节点级别 | 案例内容级别 |

### 1.3 输出示例

```
输入：5000 个案例，分布在 200 个叶子节点

处理：节点选择与聚类决策

输出：
├─ selected_nodes.csv: 选中 45 个 SOP 聚合节点
├─ case_assignments.csv: 每个案例归属到哪个节点
└─ summary.json: 统计信息
    ├─ stop_here: 30 个节点
    ├─ cluster_inside: 15 个节点
    ├─ split: 100 个节点（继续下钻）
    └─ skip: 55 个节点（案例不足）
```

### 1.4 节点与案例的对应关系（1:N）

节点与案例是 **一对多** 关系：一个节点对应多个案例。

| 数据结构 | 数量 | 说明 |
|---------|------|------|
| selected_nodes.csv | N 个节点 | 选中的 SOP 聚合节点 |
| case_assignments.csv | M 个案例 | 每个案例归属到唯一节点 |
| case_facts.jsonl | M 行 | 每个案例一行事实卡 |
| node_sops.jsonl | N 行 | 每个节点一行 SOP |

**关系链**：

```
节点 : 案例 = 1 : N  （一个节点对应多个案例）
案例 : 事实卡 = 1 : 1 （一个案例一张事实卡）
节点 : SOP = 1 : 1   （一个节点一个 SOP）
```

**数据示例**：

```
selected_nodes.csv:
├─ 节点A: 虚拟机开机失败 (50 个案例) ←── case-1, case-2, ..., case-50
├─ 节点B: 虚拟机迁移失败 (35 个案例) ←── case-51, case-52, ..., case-85
└─ 节点C: 存储扩容失败 (20 个案例)  ←── case-86, ..., case-105

case_assignments.csv:
├─ case-1  → 节点A
├─ case-2  → 节点A
├─ ...
├─ case-50 → 节点A
├─ case-51 → 节点B
└─ ...

后续：SOP聚合时，节点A 的 50 个事实卡合并为 1 个 SOP
```

## 二、概述

项目采用 **基于规则的节点指标决策系统**，而非传统机器学习聚类算法（如 K-Means、DBSCAN）。

**核心思路**：
- 将分类结果视为"带噪声的路径证据"
- 通过多维度指标计算节点状态
- 基于规则决策是否需要在节点内部进行语义聚类

---

## 二、决策类型定义

系统定义了四种节点决策：

| 决策类型 | 含义 | 说明 |
|---------|------|------|
| `split` | 继续下钻 | 当前节点不产出 SOP，继续向子节点探索 |
| `stop_here` | 直接聚合 | 当前节点直接作为 SOP 聚合节点 |
| `cluster_inside` | **内部聚类** | 不沿分类树下钻，需在节点内部聚类后再生成 SOP |
| `skip` | 跳过 | 不单独产出 SOP，由其他节点承接 |

---

## 三、聚类触发条件

聚类决策 `cluster_inside` 在以下场景触发（`decide_node` 函数，第525-591行）：

### 场景1：叶子节点案例过多

```python
if metrics.tree_is_leaf:
    if metrics.raw_case_count >= args.max_cases_single_sop:  # 默认180
        return "cluster_inside", "leaf_too_many_cases"
```

### 场景2：粗糙叶子节点

```python
if (
    metrics.depth <= args.cluster_leaf_max_depth      # 默认深度≤4
    and metrics.raw_case_count >= args.cluster_leaf_min_cases  # 默认≥30
):
    return "cluster_inside", "coarse_tree_leaf"
```

**含义**：分类树中深度较浅的叶子节点，说明分类不够精细，需要内部聚类。

### 场景3：叶子节点高不确定性

```python
if (
    metrics.raw_case_count >= args.cluster_leaf_min_cases  # 默认≥30
    and metrics.manual_review_rate >= args.cluster_leaf_review_threshold  # 默认≥35%
):
    return "cluster_inside", "leaf_with_high_uncertainty"
```

**含义**：人工校验率高 → 分类质量不确定 → 需要内部聚类重新梳理。

### 场景4：极端不确定性

```python
if metrics.manual_review_rate >= args.leaf_extreme_uncertainty_threshold:  # 默认≥55%
    return "cluster_inside", "leaf_extreme_uncertainty"
```

**含义**：无论案例数量，只要人工校验率极高，强制聚类。

### 场景5：无强子节点但案例过多

```python
if metrics.active_child_count == 0:  # 没有符合条件的子节点
    if metrics.raw_case_count >= args.max_cases_single_sop:  # 默认180
        return "cluster_inside", "no_strong_child_but_too_many_cases"
    return "stop_here", "no_strong_child"
```

## 四、关键指标计算

### 4.1 稳健支持度（`robust_support`）

```python
metrics.robust_support += review_weight * depth_penalty(
    depth,
    args.deep_node_safe_depth,  # 默认4
    args.deep_node_decay,        # 默认0.78
)
```

**计算逻辑**：
- 每个案例向上游节点累积支持度
- 深度惩罚：`depth > safe_depth` 时，每层衰减 `decay` 倍
- 人工校验惩罚：`review_weight = 0.5`（默认）

**公式**：
```
depth_penalty = 1.0                          if depth ≤ 4
depth_penalty = decay^(depth - safe_depth)   if depth > 4

robust_support = Σ(review_weight × depth_penalty)
```

### 4.2 子节点熵值（`child_entropy`）

```python
def entropy_from_weights(weights: list[float]) -> float:
    total = sum(weights)
    entropy = 0.0
    for weight in weights:
        prob = weight / total
        entropy -= prob * math.log(prob, 2)  # 以2为底
    return entropy
```

**含义**：
- 熵值高 → 子节点权重分散 → 存在多个"势均力敌"的子节点
- 熵值低 → 权重集中 → 有主导子节点

### 4.3 最大子节点比例（`max_child_ratio`）

```python
metrics.max_child_ratio = max(strong_child_weights) / total_weight
```

**含义**：最大子节点占比。用于判断是否存在"主导子节点"。

## 六、决策流程图

```
节点决策流程
│
├─ 案例数=0 ──────────────────────► skip (无案例支持)
│
├─ 案例数<min 或 支持度<min ──────► skip (低于阈值)
│
├─ 是根节点 ─────────────────────► split (必须下钻)
│
├─ 是叶子节点
│   ├─ 案例数≥180 ───────────────► cluster_inside (案例过多)
│   ├─ 深度≤4 且 案例数≥30 ──────► cluster_inside (粗糙叶子)
│   ├─ 案例数≥30 且 校验率≥35% ──► cluster_inside (高不确定性)
│   ├─ 校验率≥55% ───────────────► cluster_inside (极端不确定)
│   └─ 否则 ─────────────────────► stop_here (具体叶子)
│
├─ 无强子节点
│   ├─ 案例数≥180 ───────────────► cluster_inside (无子节点但过多)
│   └─ 否则 ─────────────────────► stop_here (无法下钻)
│
├─ 深度<5 ───────────────────────► split (太浅)
│
├─ 案例数>180 且 子节点≥2 ───────► split (需要拆分)
│
├─ 子节点≥3 且 熵值≥1.15 且 无主导子节点 ─► split (子节点过于分散)
│
├─ 深度>6 ───────────────────────► stop_here (已足够深)
│
├─ 校验率≥38% ───────────────────► stop_here (不确定性高，停在祖先)
│
└─ 默认 ─────────────────────────► stop_here (在目标窗口内)
```

## 十一、常见问题解答

### Q1：四种节点决策的区别，如何判断该走哪个决策点？

| 决策 | 触发场景 | 后续行为 | 是否产出 SOP |
|------|---------|---------|-------------|
| `split` | 有符合条件的子节点可继续下钻 | 递归处理子节点 | 否，交给子节点 |
| `stop_here` | 当前节点适合直接聚合 | 标记为 SOP 节点 | 是，直接聚合 |
| `cluster_inside` | 不沿分类树下钻，但需内部聚类 | 标记为 SOP 节点，但需二次聚类 | 是，需内部聚类 |
| `skip` | 案例不足或支持度低 | 跳过该节点 | 否，由其他节点承接 |

**判断逻辑**（按优先级从高到低）：

1. **无案例或低于阈值** → `skip`
2. **是目标根节点** → `split`（必须下钻）
3. **是叶子节点** → 根据案例数/校验率决定 `cluster_inside` 或 `stop_here`
4. **无强子节点** → `cluster_inside` 或 `stop_here`
5. **深度/案例数/熵值条件** → `split` 或 `stop_here`

---

### Q2：粗糙叶子节点是指的什么？

**定义**：分类树中深度较浅（≤4）但已经是叶子节点的节点。

```python
if (
    metrics.depth <= args.cluster_leaf_max_depth  # 默认≤4
    and metrics.raw_case_count >= args.cluster_leaf_min_cases  # 默认≥30
):
    return "cluster_inside", "coarse_tree_leaf"
```

**为什么叫"粗糙"**：
- 正常叶子节点应该在较深层（如深度5-6）
- 深度≤4就到达叶子，说明**分类树不够精细**
- 这类节点需要**内部聚类**来进一步细分

**示例**：
```
云计算-排障-ECS-实例异常  (深度4，叶子节点)
├─ 案例数：50
└─ 问题：分类不够细，需要内部聚类拆分成多个 SOP
```

---

### Q3：关键指标计算是怎么设计出来的，为什么要设计这些指标？

| 指标 | 设计原因 |
|------|---------|
| `raw_case_count` | 基础指标，判断节点是否有足够案例支撑 SOP |
| `robust_support` | 加权支持度，考虑深度衰减和人工校验惩罚，反映案例的"可信度" |
| `manual_review_rate` | 校验率，高值表示分类质量不确定，需要谨慎处理 |
| `step_post_rate` | 步骤贴比例，用于后续 SOP 质量评估 |
| `child_entropy` | 子节点熵值，判断子节点是否"势均力敌"，熵值高说明需要 split |
| `max_child_ratio` | 最大子节点占比，判断是否存在主导子节点，高值可安全 stop |

**设计理念**：
- **容错性**：分类结果可能不准确，通过支持度衰减和校验惩罚来降低噪声影响
- **多维度**：单一指标无法全面评估，需要案例数、质量、结构综合判断
- **可解释性**：每个指标都有明确的业务含义

### Q4：决策流程图中的"是根节点"是什么意思如何判断的？

**定义**：目标分析范围的起始节点，由 `--target-root` 参数指定。

```python
# 判断逻辑（第603行）
is_root = (node_path == target_root)
```

**示例**：
```bash
# HCI 场景
--target-root "云计算-排障"
# 或
--scene hci  # 自动设置为 "云计算-排障"
```

**为什么根节点必须 split**：
- 根节点是分析起点，不应该产出 SOP
- SOP 应该在更具体的子节点产出，粒度更细

### Q5：校验率是指的什么？

**定义**：需要人工校验的案例占总案例的比例。

```python
metrics.manual_review_rate = (
    metrics.manual_review_true_count / metrics.raw_case_count
)
```

**数据来源**：Excel 文件中的 `分类-需人工校验` 列（值为"是"的案例）。

**业务含义**：
- 校验率高 → 分类质量不确定 → 分类可能有问题
- 处理策略：
  - 校验率 ≥ 35%：触发 `cluster_inside`，内部聚类重新梳理
  - 校验率 ≥ 55%：强制 `cluster_inside`，无论案例数量
  - 校验率 ≥ 38%（非叶子）：倾向于 `stop_here`，停在祖先节点避免错误传递

---

### Q6：无强子节点指的是什么？

**定义**：当前节点没有符合"强子节点"条件的子节点。

**强子节点条件**（第501-504行）：
```python
if (
    child.raw_case_count >= args.min_cases      # 默认≥20
    and child.robust_support >= args.min_robust_support  # 默认≥15
):
    strong_children.append(child_path)
```

**判断逻辑**：
```python
if metrics.active_child_count == 0:  # 无强子节点
    if metrics.raw_case_count >= args.max_cases_single_sop:
        return "cluster_inside", "no_strong_child_but_too_many_cases"
    return "stop_here", "no_strong_child"
```

**示例场景**：
```
云计算-排障-ECS-实例异常 (案例数：100)
├─ 子节点1: 启动失败 (案例数：5) → 不符合强子节点
├─ 子节点2: 运行卡顿 (案例数：8) → 不符合强子节点
└─ 结果：无强子节点，触发 cluster_inside
```

### Q7：split 跟 cluster_inside 的区别是什么？

| 方面 | split | cluster_inside |
|------|-------|----------------|
| **下钻方向** | 沿**分类树**继续下钻 | 在节点**内部**聚类 |
| **适用场景** | 有符合条件的子节点 | 无子节点或子节点都不符合条件 |
| **产出方式** | SOP 在子节点产出 | SOP 在当前节点内部产出（多个） |
| **分类树利用** | 充分利用分类树结构 | 分类树已无法提供有效分类 |

**示例对比**：

```
【split 场景】
云计算-排障-ECS (split)
├─ 云计算-排障-ECS-启动失败 → 产出 SOP
└─ 云计算-排障-ECS-运行卡顿 → 产出 SOP

【cluster_inside 场景】
云计算-排障-ECS-实例异常 (cluster_inside，案例数：200)
└─ 内部聚类后产出多个 SOP（不依赖分类树）
    ├─ SOP1: CPU 占用高
    ├─ SOP2: 内存溢出
    └─ SOP3: 网络超时
```

### Q8：深度是包括前面做分类的层数还是单独指聚类后的层数？

**定义**：节点在**分类树**中的层级深度，从根节点开始计算。

```python
# 第456-457行
depth = node_path.count("-") + 1
```

**计算方式**：根据路径中 `-` 的数量 + 1

**示例**：
```
云计算                    → depth=1
云计算-排障               → depth=2
云计算-排障-ECS           → depth=3
云计算-排障-ECS-启动失败  → depth=4
```

**重要澄清**：
- 深度是**分类阶段**的结果，不是聚类后的层数
- 聚类阶段不会产生新的深度层级
- `cluster_inside` 后的内部聚类结果是多个 SOP，不是新的树层级
