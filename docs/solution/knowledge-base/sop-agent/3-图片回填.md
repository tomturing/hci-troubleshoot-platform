# SOP Agent 图片回填分析

> 本文档详细分析 `sop-agent` 项目中图片回填阶段的实现设计。
>
> 源代码位置：`/mnt/d/aihci/sop-agent/scripts/add_image.py`

---

## 一、概述

图片回填阶段的目标是将图片链接**精确补回**到重写后的纯文本中，生成图文兼备的最终案例文本。

**核心约束**：只补图、不改字。

**输入输出**：

| 项目 | 内容 |
|------|------|
| **输入列** | `合并-clean`（含图片链接）+ `合并-text-rewrite`（纯文本重写） |
| **输出列** | `合并-rewrite`（图文兼备的最终版本） |

**命令**：

```bash
python scripts/add_image.py -i "data/SPT源数据-HCI.xlsx"
```

---

## 二、核心约束

add_image.py 的核心约束是**只补图、不改字**：

```
1. 保持"合并-text-rewrite"的所有非图片文本内容不变
2. 仅补入"合并-clean"中已有的 Markdown 图片链接
3. 图片顺序必须与"合并-clean"完全一致
4. 图片应插入到与"合并-clean"语义对应的位置
```

提示词中定义了严格的校验规则：

| 规则 | 说明 |
|------|------|
| 文本不可改 | 不得改写任何标题、正文、列表、代码块、表格 |
| 图片只能回填 | 每张图片必须且只能出现一次，不得修改 alt/URL |
| 位置选择 | 以 `合并-clean` 为定位参考，在语义对应位置插入 |

## 四、输出校验

这是 add_image.py 与其他阶段最大的区别——有严格的**双重校验**：

```python
# add_image.py 第136-151行
def validate_merged_content(clean_content, text_rewrite_content, merged_content):
    # 校验1：图片列表必须完全一致
    expected_images = extract_image_markdowns(clean_content)
    actual_images = extract_image_markdowns(merged_content)
    if actual_images != expected_images:
        return False, "图片列表不一致"

    # 校验2：去掉图片后，文字内容必须与 text-rewrite 完全一致
    expected_text = normalize_text_ignoring_images(text_rewrite_content)
    actual_text = normalize_text_ignoring_images(merged_content)
    if actual_text != expected_text:
        return False, "检测到非图片文本发生变化"

    return True, ""
```

**校验失败处理**：重新请求 LLM，最多重试 5 次（MAX_RETRIES）。

**校验流程**：

```
LLM 输出 <MERGED_CONTENT>
     │
     ▼
提取 Markdown 图片列表
     │
     ▼ 图片列表一致？──否──→ 重新请求 LLM
     │
     是
     ▼
去掉图片后比较文字内容
     │
     ▼ 文字内容一致？──否──→ 重新请求 LLM
     │
     是
     ▼
写入单元格
```

## 五、数据行分类处理

```
扫描数据行
  ├── already_done（已有合并-rewrite）   → 跳过
  ├── empty_clean（源列为空）            → 跳过
  ├── empty_text_rewrite（重写列为空）    → 跳过
  ├── pass_through（无图片 或 跳过标记）   → 直接复制 text-rewrite 内容
  └── need_add_image（需要补图）         → 调用 LLM
```

**跳过标记识别**：识别 rewrite 阶段写入的 `[跳过: 含不可访问的图片URL]`，遇此标记则直接复制 text-rewrite 内容。

## 六、模型配置

add_image.py 使用全局模型配置，默认 `deepseek-reasoner`：

```python
# add_image.py 第40-50行
API_BASE_URL = get_env("SOP_AGENT_API_BASE_URL", "https://openrouter.ai/api/v1")
MODEL_NAME = get_env("SOP_AGENT_MODEL_NAME", "deepseek-reasoner")
MAX_CONCURRENCY = 500    # 纯文本任务，并发可更高
REQUEST_TIMEOUT = 1200    # 超时较长（任务较重）
SAVE_INTERVAL = 500       # 写盘频率更高
```

| 配置项 | 说明 |
|--------|------|
| MAX_CONCURRENCY = 500 | 纯文本任务无视觉开销，并发可设置更高 |
| REQUEST_TIMEOUT = 1200s | 单次请求超时较长 |
| SAVE_INTERVAL = 500 | 每 500 条写盘一次，避免频繁保存拖慢整体性能 |

## 七、容错与可靠性

| 机制 | 说明 |
|------|------|
| 断点续跑 | 检测已有输出列，跳过已完成行 |
| 指数退避重试 | 2s 基础延迟，最多 5 次重试 |
| 双重校验 | 图片列表 + 文字内容一致性校验 |
| 校验失败重试 | 模型输出不合规时重新请求，最多重试 5 次 |
| 跳过标记识别 | 识别 rewrite 阶段的 `[跳过:...]` 标记 |
| 原子写盘 | `.tmp` + `os.replace()`，防中断损坏 |

## 八、与其他阶段的关系

```
clean.py → 合并-clean 列 ──┬── rewrite.py ──→ 合并-text-rewrite 列
                          │                        │
                          └─────── add_image.py ←┘
                                          │
                                          ▼
                                    合并-rewrite 列
                                          │
                                          ▼
                                   classify.py（后续阶段）
```

- `合并-rewrite` 是后续事实提取（`extract_case_facts.py`）的**默认输入列**
- 执行顺序：`clean.py → rewrite.py → add_image.py`
- `add_image.py` 依赖前两者的输出作为输入
