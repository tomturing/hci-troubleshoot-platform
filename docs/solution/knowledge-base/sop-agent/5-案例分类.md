# SOP Agent 分类逻辑分析

> 本文档详细分析 `sop-agent` 项目中案例自动分类的实现设计。
>
> 源代码位置：`/mnt/d/aihci/sop-agent/scripts/classify.py`

---

## 一、概述

分类阶段的目标是将每个案例准确归类到预定义分类树的**叶子节点**。

**核心思路**：
- 使用 LLM（大语言模型）进行智能分类
- 分类树作为约束条件，确保分类结果的合法性
- 多次独立判断验证一致性，提高分类质量
- 完整的校验机制，自动检测和修复常见问题

## 二、分类树结构

### 2.1 来源与构建方式

**分类树是预定义的配置文件（由CTI统一维护）。**

```python
# classify.py 第146-149行
def load_and_clean_tree(yaml_path: str) -> dict:
    """加载 YAML 分类树。"""
    with open(yaml_path, "r", encoding="utf-8") as f:
        return yaml.safe_load(f)
```

**文件位置**：

```python
# classify.py 第135-138行
SCENE_TREES = {
    "hci": "output-hci/云计算_tree.yaml",
    "af": "output-af/AF_tree.yaml",
}
```

| 场景 | 分类树文件 |
|------|-----------|
| HCI | `output-hci/云计算_tree.yaml` |
| AF | `output-af/AF_tree.yaml` |

**设计特点**：

| 方面 | 说明 |
|------|------|
| **构建方式** | 人工设计，基于业务知识 |
| **格式** | YAML 层次结构 |
| **修改方式** | 直接编辑 YAML 文件 |
| **运行时行为** | 只读加载，不修改 |

**为什么是预定义而非自动生成**：

| 原因 | 说明 |
|------|------|
| **业务知识** | 分类树需要反映产品的功能模块和故障类型，需要领域专家设计 |
| **稳定性** | 预定义的分类树保证分类结果的一致性和可解释性 |
| **可控性** | 人工设计可以控制分类粒度，避免过粗或过细 |
| **维护成本** | 自动生成的分类树可能不稳定，增加后续维护成本 |

### 2.2 定义格式

分类树采用 **YAML 格式** 定义，支持任意深度的层次结构。

```yaml
name: "云计算"
children:
  - name: "排障"
    children:
      - name: "虚拟机"
        children:
          - name: "虚拟机生命周期"
            children:
              - name: "虚拟机管理"
                children:
                  - name: "虚拟机创建"
                    children:
                      - name: "虚拟机创建失败"  # 叶子节点
```

### 2.3 深度计算

```python
# 第456-457行
depth = node_path.count("-") + 1
```

**示例**：
```
云计算                              → depth=1
云计算-排障                         → depth=2
云计算-排障-虚拟机                   → depth=3
云计算-排障-虚拟机-虚拟机生命周期     → depth=4
```

### 2.4 叶子节点

**定义**：没有 `children` 属性或 `children` 为空的节点。

**标记方式**：在提示词中用 `*` 标记，表明只能将案例归类到叶子节点。

## 三、提示词设计

### 3.1 提示词模板结构

提示词采用 **Jinja2 模板**，动态注入分类树和参数。

```
# 角色
你是 {{ classification_root_name }} 产品的技术支持案例分类专家...

# 分类树
{{ classification_tree }}

# 输出格式
{{ output_json_example }}
```

### 3.2 核心分类规则

提示词中定义了详细的分类规则，确保分类准确性：

| 规则 | 说明 |
|------|------|
| **按核心问题分类** | 区分核心问题、触发操作、提及组件，避免被表象误导 |
| **关注最终影响对象** | 问题链涉及多层时，以最终定位和解决的那一层为准 |
| **区分表面与根因** | 以案例最终定位的根因为准，而非表面告警 |
| **必须匹配叶子节点** | 分类结果只能是叶子节点，不能停在中间节点 |
| **层级递进判断** | 先判断大类，再逐层缩小到具体叶子节点 |

### 3.3 搜索策略

提示词要求 LLM 采用 **"按层广度优先、逐层剪枝"** 的搜索策略：

1. 先查看第 1 层候选节点，比较匹配度，选出最可能的 1 个
2. 仅在已选中的父节点下，继续查看下一层候选子节点
3. 每一层先横向比较，再向下深入
4. 直到最终命中叶子节点

### 3.4 易混淆分类辨析

提示词中预定义了常见易混淆场景的辨析规则：

| 混淆场景 | 辨析方法 |
|---------|---------|
| `排障` vs `协助配置与变更` | 是否有明确的故障现象 |
| `虚拟机` vs `平台` | 聚焦单台虚拟机还是集群/主机层 |
| `存储` vs `虚拟机磁盘问题` | 问题发生在存储设施层还是虚拟机使用层 |
| `网络` vs `虚拟机网络异常` | 问题发生在主机网口层还是虚拟机内部网络 |

## 四、分类流程

### 4.1 整体流程图

```
加载分类树 (YAML)
      ↓
渲染提示词模板 (Jinja2)
      ↓
扫描待分类行 (断点续跑)
      ↓
构造用户消息 (案例内容)
      ↓
调用 LLM API (异步并发)
      ↓
解析 JSON 响应
      ↓
校验分类结果
      ↓
一致性判断 (多次独立)
      ↓
写入 Excel (定期保存)
```

### 4.2 输入数据

| 列名 | 用途 |
|------|------|
| `标题` | 案例标题（必需） |
| `合并-text-rewrite` | 案例重写后的内容（必需） |
| `主模块` | 原系统粗分类标签（参考） |
| `是否仅作为步骤贴` | 区分排障与配置案例 |

### 4.3 输出数据

| 列名 | 说明 |
|------|------|
| `分类-一级` ~ `分类-十级` | 各级分类名称 |
| `分类-全路径` | 完整分类路径（从根到叶子） |
| `分类-依据` | LLM 的分类推理过程 |
| `分类-需人工校验` | 是否需要人工复核 |
| `一致性校验` | 独立判断次数 |

## 五、一致性校验机制

### 5.1 设计目的

通过**多次独立判断**验证分类结果的可靠性，只有当所有判断结果一致时才通过。

### 5.2 实现方式

```python
# 第1047-1101行
for run_idx in range(1, consensus_runs + 1):
    response = await call_api(...)
    result = extract_json_result(response or "")
    
    # 校验通过后检查一致性
    current_path = result.get("全路径", "")
    if round_paths and current_path != round_paths[0]:
        # 结果不一致，重新请求整轮
        attempt_failed = True
        break
    
    round_results.append(result)
    round_paths.append(current_path)
```

### 5.3 参数配置

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `CONSENSUS_RUNS` | 3 | 同一案例独立判断次数 |
| `MAX_RETRIES` | 8 | 单条最大重试次数 |

### 5.4 不一致处理

当多次判断结果不一致时：
1. 记录不一致的路径信息
2. 计入一次模型错误
3. 按 `MAX_RETRIES` 进行重试
4. 超过重试次数后标记为失败

## 十、常见问题解答

### Q1：为什么要将案例归类到叶子节点，而不是中间节点？

**原因**：
- 叶子节点代表最具体的分类，信息量最大
- 中间节点可能有多个子节点，无法精确定位问题
- SOP 生成需要细粒度的案例聚合

### Q2：一致性校验为什么要多次独立判断？

**原因**：
- LLM 输出具有随机性（temperature > 0）
- 多次独立判断可以过滤偶然错误
- 只有一致的结果才更可靠

### Q3：为什么需要"需人工校验"标记？

**触发条件**：
- 案例涉及多个分类领域，主要归属难以判断
- 案例描述模糊，关键信息不足
- 分类判断与"主模块"标签严重矛盾
- 现有分类树中没有精确的叶子节点
- 对分类结果信心不高

**后续处理**：这些案例在聚类阶段会触发 `cluster_inside` 决策。

### Q4：分类树是如何构建的？

分类树是**人工设计**的，不是自动生成的。设计原则：
- 覆盖产品的主要功能模块和故障类型
- 层级结构反映问题的逻辑分类
- 叶子节点粒度适中，既不过粗也不过细

### Q5：主模块字段的作用是什么？

主模块是原系统中的粗分类标签，**仅供参考**：
- 帮助 LLM 快速定位大类
- 不能直接作为分类依据
- 如果判断与主模块严重矛盾，需要人工校验

### Q6：为什么分类依据要按"逐层判断"格式输出？

**原因**：
- 便于人工复核分类逻辑
- 便于调试和优化提示词
- 提供可解释的分类过程

### Q7：路径修复机制什么时候会触发？

**触发场景**：
- 模型输出的路径缺少根节点前缀
- 模型输出的路径使用了不同的分隔符
- 模型输出的路径只包含部分层级

**修复方式**：
- 添加根节点前缀
- 统一分隔符为 `-`
- 通过后缀查找完整路径
