# SOP Agent 数据清洗分析

> 本文档详细分析 `sop-agent` 项目中数据清洗阶段的实现设计。
>
> 源代码位置：`/mnt/d/aihci/sop-agent/scripts/clean.py`

---

## 一、概述

数据清洗阶段的目标是将原始案例文本转换为**规范 Markdown**，消除格式噪声、统一章节结构，为后续重写和事实提取提供高质量输入。

**核心思路**：
- 使用 LLM（大语言模型）执行文本清洗，而非正则替换
- 通过严格的提示词约束，确保"只清洗不改写"
- 支持按场景（HCI / AF）选择不同的清洗提示词
- 异步并发 + 断点续跑，适合大规模数据处理

## 二、输入输出

| 项目 | 内容 |
|------|------|
| **输入文件** | `data/SPT源数据-HCI.xlsx` 或 `data/SPT源数据-AF.xlsx` |
| **输入列** | `合并`（原始案例文本，含 HTML、论坛格式噪声） |
| **输出列** | `合并-clean`（新增列，清洗后的规范 Markdown） |
| **输入格式** | 原始文本，可能包含 HTML 标签、论坛标记、错位章节等 |
| **输出格式** | 规范 Markdown，统一章节结构 |

**命令**：

```bash
python scripts/clean.py -i "data/SPT源数据-HCI.xlsx" --scene hci
python scripts/clean.py -i "data/SPT源数据-AF.xlsx" --scene af
```

## 三、清洗规则体系

清洗规则由提示词文件定义，代码本身只负责调度和写入。

### 3.1 提示词路径

```python
# clean.py 第76-80行
DEFAULT_PROMPT_DIR = Path(__file__).resolve().parent.parent / "prompt" / "clean_prompt"
SCENE_PROMPTS = {
    "hci": DEFAULT_PROMPT_DIR / "clean_prompt_hci.md",
    "af": DEFAULT_PROMPT_DIR / "clean_prompt_af.md",
}
```

**优先级**：`--prompt` 参数 > `--scene` 参数 > 报错退出。

### 3.2 红线规则（不可违反）

1. **禁止改写与新增**：仅允许文本润色（语句通顺、用词规范、标点修正），严禁引入原文中不存在的事实内容
2. **空白章节保持原样**：若原文某章节为空，保留标题写"无"，不得补写
3. **错位内容允许归位**：若某段内容明显写错了位置，允许移动到正确章节
4. **不得删除实质内容**：任何有意义的原始信息必须保留

### 3.3 章节结构统一

HCI 场景的标准化章节（按固定顺序排列）：

```
## 问题描述
## 告警信息
## 有效排查步骤
## 根因
## 解决方案
## 操作影响范围
## 是否是临时解决方案
## 建议与总结
```

**章节合并规则**：

| 原始章节 | 目标章节 |
|---------|---------|
| 处理过程 | 有效排查步骤 |
| 排查内容 | 有效排查步骤（有实质）/ 丢弃（为空） |
| 排查进展 | 有效排查步骤 |
| 临时解决方案 + 永久解决方案 | 解决方案（用三级标题区分） |
| 版本信息 | 问题描述末尾 |
| 问题现象 | 问题描述 |
| 高危操作 | 解决方案对应步骤处标记 |

**空章节与缺失章节**：
- 原文存在但内容为空 → 保留标题，写"无"
- 原文完全不存在 → 不新增
- 非标准章节 → 归入对应标准章节，无法归类放入"有效排查步骤"末尾

### 3.4 格式转换

| HTML | 转换为 |
|------|--------|
| `<br>` `<br />` | 删除，靠空行换行 |
| `<img src="URL" ...>` | `![截图](URL)`，去除所有其他属性 |
| `<a href="URL">文字</a>` | `[文字](URL)` |
| `<table><tr><td>` | Markdown 表格 |
| `<font>` `<div>` 等样式标签 | 删除标签，保留内部文本 |

**需删除的噪声**：
- 图片尺寸说明文字（如 `47645b6d7fe9a62cb.png (36.43 KB)`）
- `[color=...]`、`class="zoom"`、`onclick="..."` 等论坛样式属性
- 论坛遗留的"复制代码"文字

### 3.5 文本润色规则

- 第三人称陈述句，避免口语化
- 避免"我们""你""客户反馈说"等主观表述
- 中文标点，代码内保持英文标点
- 修正错别字和语法问题
- 阿拉伯数字与中文之间加空格

## 四、技术实现

### 4.1 模型配置

```python
# clean.py 第40-51行
API_BASE_URL = get_env("SOP_AGENT_API_BASE_URL", "https://openrouter.ai/api/v1")
API_KEY = get_env("SOP_AGENT_API_KEY", "")
MODEL_NAME = get_env("SOP_AGENT_MODEL_NAME", "deepseek-reasoner")
TEMPERATURE = 0.8
MAX_CONCURRENCY = 100
MAX_RETRIES = 5
RETRY_BASE_DELAY = 2
REQUEST_TIMEOUT = 360
SAVE_INTERVAL = 20
```

### 4.2 请求格式

**用户消息**（Jinja2 模板）：

```python
# clean.py 第65-68行
USER_MSG_TEMPLATE = jinja2.Template(
    "<RAW_CONTENT>\n{{ content }}\n</RAW_CONTENT>",
    undefined=jinja2.StrictUndefined,
)
```

**模型响应提取**：

```python
# clean.py 第71-74行
EXTRACT_RE = re.compile(
    r"<CLEANED_CONTENT>\s*(.*?)\s*</CLEANED_CONTENT>",
    re.DOTALL,
)
```

模型输出必须包含 `<CLEANED_CONTENT>...</CLEANED_CONTENT>` 标签，脚本从中提取清洗结果。

### 4.3 处理流程

```
打开 xlsx → 定位"合并"列 → 创建/定位"合并-clean"列
     │
     ▼
扫描数据行 → 分类为：待处理 / 已完成(跳过) / 空行
     │
     ▼
待处理行入队 → Worker 并发调用 LLM API
     │
     ▼
每行：渲染提示词 → 调用 API → 提取 <CLEANED_CONTENT> → 写入单元格
     │
     ▼
按 SAVE_INTERVAL 定期写盘 → 最终写盘 → 汇总统计
```

### 4.4 断点续跑

```python
# clean.py 第170-190行
def scan_rows(ws, col_merge: int, col_clean: int):
```

**机制**：重新运行时自动检测已有 `合并-clean` 列，跳过已完成的行。

### 4.5 容错机制

1. **指数退避重试**：单条最多重试 5 次，延迟 2s → 4s → 8s → 16s → 32s
2. **标签解析重试**：模型响应不含 `<CLEANED_CONTENT>` 标签时，重新请求
3. **原子写盘**：先写 `.tmp` 文件再 `os.replace()` 替换，防止中断导致文件损坏
4. **失败行记录**：记录失败行号，重新运行时自动重试

### 4.6 并发模型

```
asyncio.Queue (待处理行)
     │
     ├── Worker 1 → process_row() → call_api()
     ├── Worker 2 → process_row() → call_api()
     ├── ...
     └── Worker N → process_row() → call_api()

信号量控制并发上限 = MAX_CONCURRENCY = 100
保存锁 (save_lock) 保护 Excel 写入
保存计数器 (save_counter) 控制写盘频率
```

## 五、统计输出

运行完成后输出：

```
=======================================================

失败行号: [42, 187, 531]
提示: 重新运行脚本即可自动重试失败行
文件已保存: data/SPT源数据-HCI.xlsx
```

| 统计项 | 含义 |
|--------|------|
| ✓成功 | LLM 返回有效 `<CLEANED_CONTENT>` 并成功写入 |
| ✗失败 | 重试 5 次后仍失败（API 异常或超时） |
| ⚠解析失败 | 模型连续 5 次未输出 `<CLEANED_CONTENT>` 标签 |

## 六、与其他阶段的关系

```
合并 列 ──clean.py──→ 合并-clean 列 ──rewrite.py──→ 合并-text-rewrite 列
                                                │
                                                └──add_image.py──→ 合并-rewrite 列
                                                                    │
                                                                    ▼
                                                              classify.py
```

- `clean.py` 是全流程的**第一个处理步骤**
- 输出 `合并-clean` 列是 `rewrite.py` 的直接输入
- 清洗质量直接影响后续所有阶段的效果
