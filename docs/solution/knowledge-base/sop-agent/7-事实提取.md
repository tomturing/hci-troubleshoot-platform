## 一、概述

事实提取阶段的目标是从每个案例中提取结构化的事实信息，为后续 SOP 聚合提供标准化的输入。

**核心思路**：
- 使用 LLM 从案例内容中提取结构化事实
- 本地校验确保步骤图结构正确
- 支持断点续跑，避免重复处理
- 异步并发处理，提高效率

### 7.3 权重计算

```python
# 第585-600行
def build_ancestor_weights(leaf_path, review_flag, penalty, safe_depth, decay):
    """计算祖先节点的证据权重（用于后续聚合）"""
    weights = {}
    parts = split_path(leaf_path)
    for i in range(len(parts)):
        ancestor_path = "-".join(parts[:i+1])
        depth = i + 1
        weight = depth_penalty(depth, safe_depth, decay)
        if review_flag:
            weight *= penalty  # 人工校验案例权重降低
        weights[ancestor_path] = weight
    return weights
```

## 十二、常见问题解答

### Q1：fact_card 和 SOP 的区别是什么？

| 方面 | fact_card | SOP |
|------|-----------|-----|
| 层级 | 单案例 | 节点级（多案例聚合） |
| 来源 | 单个案例提取 | 多个 fact_card 合并 |
| 用途 | 证据层，用于聚合 | 运行层，用于执行 |

---

### Q2：为什么需要步骤图校验？

**原因**：
- 确保诊断流程逻辑正确
- 避免死循环导致执行卡住
- 确保所有分支都有明确的退出点
- 验证命令语法正确性

---

### Q3：ancestor_evidence_weights 是什么？

**定义**：案例在分类树各层级的证据权重。

**用途**：
- 在 SOP 聚合时加权计算
- 深层节点权重更高（更精确）
- 人工校验案例权重降低（可信度低）

**计算公式**：
```
weight = depth_penalty × review_penalty
depth_penalty = 1.0                    if depth ≤ safe_depth
depth_penalty = decay^(depth-safe)     if depth > safe_depth
review_penalty = 0.5                   if review_flag
review_penalty = 1.0                   otherwise
```

---

### Q4：decision_tree_visual 的作用是什么？

**作用**：
- 将步骤图渲染为 ASCII 树形图
- 便于人工复核和理解诊断流程
- 作为输出的一部分，提供可读性

---

### Q5：为什么要区分 explicit 和 inferred？

**原因**：
- `explicit`：案例中明确提及，可信度高
- `inferred`：根据上下文推断，需要验证
- 在 SOP 聚合时，优先使用 explicit 信息
