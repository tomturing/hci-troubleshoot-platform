## 1. 背景与核心问题

### 1.1 问题陈述

传统评价体系的局限：**用户不评分 ≠ 没有质量数据**。

在实际运营中：
- 工程师使用工具的主要目的是解决问题，不是填写评价
- 用户主动评分率通常在 20–40%（行业参考值）
- 仅依赖显式星级评分，70% 以上的工单质量将是盲区

### 1.2 我们的优势

HCI 排障平台在每次 AI 对话中自动产生大量**隐式质量信号**，这些信号无需用户任何操作即可 100% 覆盖采集：

| 信号 | 采集方式 | 覆盖率 | 质量含义 |
|------|---------|--------|---------|
| 关闭原因 | `close_reason` 字段，系统自动识别 | **100%** | 用户主动解决 vs 放弃 |
| 会话时长 | `closed_at - created_at` | **100%** | 解决效率 |
| 消息轮数 | `conversation.message_count` | **100%** | 交互复杂度 |
| **用户重复提问次数** | conversation-service 统计相邻用户消息相似度 | **100%** | AI 回答是否有效（反复提问 = 未解决）|
| KB 命中率 | `prompt_audit.kb_chunks_count` 元数据字段 | **100%** | AI 是否用到了领域知识 |
| SOP 匹配 | `prompt_audit.has_sop` 元数据字段 | **100%** | 是否有标准化处置依据 |
| 完整 Prompt payload | `prompt_audit.messages` JSONB 大字段 | ~10–20%（采样）| 完整上下文用于深度 review 和 fine-tuning |
| 用户星级评分 | 工单关闭后弹出评分卡 | ~20–40%（自愿）| 用户主观满意度 |

> **注意**：`KB 命中率` 和 `SOP 匹配` 是 `prompt_audit` 表的**元数据字段**（小字段，~200 bytes/条），每次 AI 请求都会写入，覆盖率 100%。只有 `messages[]` 完整 payload 大字段（5000–20000 chars）才按 10–20% 采样存储。

### 1.3 设计目标

> 构建一套**双轨制质量评价体系**：被动隐式信号（100% 覆盖）+ 主动显式评分（非阻塞），将所有可利用的评价维度综合为一个 `composite_score`（0–100），用于持续优化 AI 质量。

## 2. 双轨制评价设计

### 2.1 主动评分轨道（Active Track）

**触发时机**：工单关闭成功后，前端弹出非阻塞评分卡

```
工单关闭成功 ──→ 评分卡弹出
                  ┌─────────────────────────────┐
                  │  本次排障对您有帮助吗？        │
                  │  ☆ ☆ ☆ ☆ ☆               │
                  │                             │
                  │  (可选) 补充说明...           │
                  │                             │
                  │         [跳过]   [提交]      │
                  └─────────────────────────────┘
                               │
              3 秒无操作后自动消失（不阻塞任何后续操作）
```

**关键原则**：
- 评分卡是**可选的**，不阻塞工程师完成后的下一步操作
- 自动 3 秒倒计时消失（防止评分卡成为"弹窗打扰"）
- 只有一次机会触发（不在案例详情页重复出现）
- 数据提交后在 `assistant_evaluation` 表写入/更新 `score` 和 `feedback`

### 2.2 被动信号轨道（Passive Track）

**触发时机**：case-service 的 close case 逻辑完成时，自动采集

被动信号是评价体系的**基础盘**，无论用户是否评分都必须采集。

#### close_reason 设计

`close_reason` 是最重要的被动信号，代表了工单关闭的"语义"：

| 值 | 语义 | 触发条件 | 质量信号强度 |
|----|------|---------|------------|
| `user_command` | 用户主动输入 `/close` 结束 | 工程师在对话框输入 `/close` | **正向强信号**（主动解决问题）|
| `timeout` | 超时自动关闭 | scheduler-service 检测到 N 分钟无新消息 | **中性**（可能忙，也可能放弃）|
| `abandon` | 用户关闭页面/断开连接 | WebSocket 断开 + case 仍 open 状态 | **负向信号**（大概率问题未解决）|
| `admin_close` | 管理员后台强制关闭 | 管理后台手动操作 | **中性**（人工判断的特殊情况）|

> **注意**：`timeout` 需要进一步结合 `session_duration` 判断。如果会话持续 60 分钟后超时，AI 大概率帮用户解决了问题，工程师只是忘记关闭。

## 3. 综合质量评分引擎（Composite Score）

### 3.1 评分维度与权重

| 维度 | 子指标 | 基础权重 | 无用户评分时 |
|------|--------|---------|------------|
| **用户满意度** | 星级评分 1–5 → 归一化 | 30% | 降至 0%，其余维度等比提升 |
| **关闭意图** | `close_reason` 映射分 | 20% | 等比提升 |
| **解决效率** | 会话时长 + 消息轮数 | 20% | 等比提升 |
| **用户重复提问** | 同一工单中重复相似问题的轮数 | 15% | 等比提升（始终可采集，100% 覆盖）|
| **AI 能力质量** | KB 命中率 + SOP 匹配（元数据字段，100% 覆盖）| 15% | 等比提升 |

> 权重在无用户评分时自动归一化（各维度原始权重等比放大至总和 100%）。

### 3.2 各维度评分细则

#### 维度 1：用户满意度分（0–100）

```
星级 1 → 用户满意度分 = 0
星级 2 → 用户满意度分 = 25
星级 3 → 用户满意度分 = 50
星级 4 → 用户满意度分 = 75
星级 5 → 用户满意度分 = 100
无评分  → 不参与计算，权重归零
```

#### 维度 2：关闭意图分（0–100）

```python
CLOSE_REASON_SCORE = {
    "user_command": 90,   # 主动解决：很高分
    "timeout":      50,   # 超时：中性基础分
    "admin_close":  60,   # 人工干预：略高于中性
    "abandon":      10,   # 放弃：较低分
    None:           50,   # 未知：中性
}
```

`timeout` 时，结合 `session_duration_sec` 做修正：

```python
def timeout_score(session_duration_sec: int) -> int:
    if session_duration_sec > 3600:   # > 1 小时才超时 → 很可能已解决
        return 70
    if session_duration_sec > 1800:   # 30–60 分钟
        return 55
    return 40                          # < 30 分钟就超时 → 可能放弃
```

#### 维度 3：解决效率分（0–100）

会话时长分：

```
< 5 min   → 100  (快速解决)
5–15 min  → 80
15–30 min → 60
30–60 min → 40
> 60 min  → 20   (长时间未解决)
```

消息轮数分：

```
≤ 3 轮    → 100  (一问即答)
4–8 轮    → 80
9–15 轮   → 60
16–25 轮  → 40
> 25 轮   → 20   (反复沟通)
```

效率维度分 = `(时长分 + 轮数分) / 2`

#### 维度 4：用户重复提问分（0–100）

**数据来源**：conversation-service 在每次 `save_user_message()` 时统计，与前 N 条用户消息做关键词重叠比对，发现相似即 `repeat_count += 1`，写入 `conversation.repeat_question_count`（需加字段）。

```python
def repeat_question_score(repeat_count: int) -> int:
    """重复提问次数越多，说明 AI 回答越无效"""
    if repeat_count == 0:
        return 100   # 无重复，AI 一次回答即解决
    if repeat_count == 1:
        return 70    # 轻微重复，可接受
    if repeat_count == 2:
        return 45    # 明显重复，AI 回答有问题
    if repeat_count == 3:
        return 20    # 严重重复，AI 基本无效
    return 0         # ≥4 次重复，完全失败
```

> **实现说明**：关键词重叠检测（Jaccard 相似度 > 0.5 判定为重复）无需额外模型，conversation-service 内直接实现，100% 覆盖无采样损耗。

#### 维度 5：AI 能力质量分（0–100）

`prompt_audit` 元数据字段（`has_sop`、`kb_chunks_count`、`kb_top_score`）**100% 写入**，始终参与计算：

```python
def ai_quality_score(has_sop: bool, kb_chunks_count: int, kb_top_score: float) -> int:
    score = 0
    # SOP 命中 +40 基础分
    if has_sop:
        score += 40
    # KB 命中质量（相关度加权）
    if kb_chunks_count > 0:
        score += min(60, int(kb_top_score * 60))  # 最高 60 分
    return min(100, score)
```

### 3.3 综合评分算法实现

```python
from dataclasses import dataclass
from typing import Optional

@dataclass
class QualitySignals:
    case_id: str
    close_reason: Optional[str]           # "user_command" | "timeout" | "abandon" | "admin_close" | None
    session_duration_sec: Optional[int]
    message_count: Optional[int]
    repeat_question_count: int            # 相似重复提问次数，conversation-service 统计，始终有值
    user_rating: Optional[int]            # 1–5，用户未评分则 None
    has_sop: Optional[bool]               # 来自 prompt_audit 元数据，100% 覆盖
    kb_chunks_count: Optional[int]        # 来自 prompt_audit 元数据，100% 覆盖
    kb_top_score: Optional[float]         # 来自 prompt_audit 元数据，100% 覆盖


@dataclass
class QualityScore:
    composite_score: int                  # 0–100 最终综合分
    rating_included: bool                 # 是否含用户评分维度
    breakdown: dict                       # 各维度详细分解


# 各维度基础权重（归一化前）
BASE_WEIGHTS = {
    "user_rating":    0.30,   # 用户满意度（无评分时降为 0）
    "close_intent":   0.20,   # 关闭意图
    "efficiency":     0.20,   # 解决效率
    "repeat_penalty": 0.15,   # 用户重复提问（负向信号）
    "ai_quality":     0.15,   # AI 能力质量
}


def compute_quality_score(s: QualitySignals) -> QualityScore:
    raw_weights = dict(BASE_WEIGHTS)
    dim_scores = {}

    # 维度 2：关闭意图（始终参与）
    close_score = CLOSE_REASON_SCORE.get(s.close_reason, 50)
    if s.close_reason == "timeout" and s.session_duration_sec:
        close_score = timeout_score(s.session_duration_sec)
    dim_scores["close_intent"] = close_score

    # 维度 3：解决效率（始终参与）
    duration_s = _duration_score(s.session_duration_sec)
    msg_s = _message_count_score(s.message_count)
    dim_scores["efficiency"] = (duration_s + msg_s) / 2

    # 维度 4：用户重复提问（始终参与，100% 覆盖）
    dim_scores["repeat_penalty"] = repeat_question_score(s.repeat_question_count)

    # 维度 5：AI 能力质量（prompt_audit 元数据，100% 覆盖；极少情况下为 None 则跳过）
    if s.has_sop is not None and s.kb_chunks_count is not None:
        dim_scores["ai_quality"] = ai_quality_score(s.has_sop, s.kb_chunks_count, s.kb_top_score or 0.0)
    else:
        raw_weights.pop("ai_quality")  # 无数据时权重转移到其他维度

    # 维度 1：用户满意度（仅有评分时参与）
    if s.user_rating is not None:
        dim_scores["user_rating"] = (s.user_rating - 1) / 4 * 100
    else:
        raw_weights.pop("user_rating")  # 无评分时权重归零，其余维度等比放大

    # 归一化加权平均（确保权重之和 = 1.0）
    active_dims = list(dim_scores.keys())
    total_weight = sum(raw_weights[k] for k in active_dims)
    composite = sum(dim_scores[k] * raw_weights[k] for k in active_dims) / total_weight

    return QualityScore(
        composite_score=round(composite),
        rating_included=s.user_rating is not None,
        breakdown={k: round(dim_scores[k]) for k in active_dims},
    )
```

### 3.4 评分解读与分级

| 综合分范围 | 等级 | 含义 | 处理建议 |
|-----------|------|------|---------|
| 80–100 | 🟢 优秀 | 快速解决，用户满意 | 提炼为优质案例 |
| 60–79 | 🔵 良好 | 正常解决，无明显问题 | 定期抽查 |
| 40–59 | 🟡 待改进 | 解决过慢或用户中性 | 分析原因，优化 KB/SOP |
| 0–39 | 🔴 差 | 放弃/评分低/长时未解决 | **必须 review + 触发 payload 补采** |

## 4. 是否需要新增服务？

### 4.1 方案对比

| 方案 | 实现位置 | 优点 | 缺点 | 推荐场景 |
|------|---------|------|------|---------|
| **A：case-service 内置模块** | `case-service/app/services/quality_score.py` | 零新服务；直接访问同库数据；实现最快 | case-service 职责略重 | **当前推荐** |
| B：独立 quality-service | 新 Docker 服务 + 新端口 | 完全解耦，可独立扩展，可接入不同数据源 | 需要新服务、Helm chart、K8s 部署配置 | 日均工单 > 500 时 |
| C：event-driven（消息队列）| case-service 发 `case.closed` 事件 | 彻底解耦，异步不阻塞 close 流程 | 需要引入 Kafka/RabbitMQ | 未来中台化 |

### 4.2 推荐方案：case-service 内置 `QualityScoreService`

**理由**：
1. 当前所有数据（case、conversation、assistant_evaluation、prompt_audit）在同一个 PostgreSQL 实例，跨表查询直接 JOIN，无需跨服务 HTTP 调用
2. 质量评分的触发点就是 case close，自然在 case-service 生命周期内处理
3. 代码变化量最小，风险最低
4. 接口设计保持扩展性：`QualityScoreService` 是独立类，未来可以整体迁移为独立服务

**预留扩展接口**：
- `QualityScoreService` 只通过 `session` 参数接收 DB 连接，不直接依赖 case-service 的其他组件
- 评分逻辑全部在 `quality_score.py` 中，没有任何 FastAPI/路由依赖，未来直接复制到新服务即可

### 4.3 QualityScoreService 设计

```python
# backend/case-service/app/services/quality_score.py

class QualityScoreService:
    """综合质量评分服务
    
    触发时机：case close（case-service 的 close_case() 调用后）
    职责：
      1. 采集所有可用的质量信号
      2. 计算 composite_score
      3. UPSERT assistant_evaluation 记录
      4. 上报 Prometheus 指标
    """

    def __init__(self, session_factory: Callable):
        self._session_factory = session_factory

    async def calculate_and_save(
        self,
        case_id: str,
        close_reason: str,
        trace_id: str,
    ) -> int:
        """在 case close 时调用，返回 composite_score (0-100)"""
        async with self._session_factory() as session:
            # 1. 读取 case 基础信息
            case = await self._get_case(session, case_id)
            session_duration = int((case.closed_at - case.created_at).total_seconds())

            # 2. 读取 conversation 消息数 + 重复提问数
            conv = await self._get_conversation(session, case_id)
            message_count = conv.message_count if conv else 0
            repeat_question_count = conv.repeat_question_count if conv else 0  # 新增字段

            # 3. 读取最近一条 prompt_audit 元数据（如有）
            audit = await self._get_latest_prompt_audit(session, case_id)

            # 4. 读取已有的 assistant_evaluation（用户可能已评分）
            existing_eval = await self._get_evaluation(session, case_id)
            user_rating = existing_eval.score if existing_eval else None

            # 5. 组装信号
            signals = QualitySignals(
                case_id=case_id,
                close_reason=close_reason,
                session_duration_sec=session_duration,
                message_count=message_count,
                repeat_question_count=repeat_question_count,
                user_rating=user_rating,
                has_sop=audit.has_sop if audit else None,
                kb_chunks_count=audit.kb_chunks_count if audit else None,
                kb_top_score=audit.kb_top_score if audit else None,
            )

            # 6. 计算
            quality = compute_quality_score(signals)

            # 7. UPSERT assistant_evaluation
            await self._upsert_evaluation(session, case_id, close_reason, session_duration, message_count, quality)
            await session.commit()

            # 8. Prometheus 指标
            QUALITY_SCORE_HISTOGRAM.observe(quality.composite_score)
            CLOSE_REASON_COUNTER.labels(reason=close_reason).inc()

            logger.info(
                event="quality_score_computed",
                case_id=case_id,
                composite_score=quality.composite_score,
                close_reason=close_reason,
                rating_included=quality.rating_included,
                trace_id=trace_id,
            )
            return quality.composite_score
```

## 5. 数据库改动

### 5.1 `case` 表新增 `close_reason`

```sql
-- 迁移脚本：migrate_evaluation_v1.sql
ALTER TABLE "case"
    ADD COLUMN IF NOT EXISTS close_reason VARCHAR(20)
        CHECK (close_reason IN ('user_command', 'timeout', 'abandon', 'admin_close'));

COMMENT ON COLUMN "case".close_reason IS '工单关闭原因：user_command/timeout/abandon/admin_close';
```

### 5.2 `conversation` 表新增 `repeat_question_count`

```sql
ALTER TABLE conversation
    ADD COLUMN IF NOT EXISTS repeat_question_count INT NOT NULL DEFAULT 0;

COMMENT ON COLUMN conversation.repeat_question_count IS '会话中用户重复提问次数，由 conversation-service 实时统计（Jaccard 相似度 > 0.5 判定为重复）';
```

> **统计时机**：conversation-service 在 `save_user_message()` 时，取当前消息与前 5 条用户消息做关键词 Jaccard 相似度，若最高相似度 > 0.5 则 `UPDATE conversation SET repeat_question_count = repeat_question_count + 1`。

### 5.3 `assistant_evaluation` 表增强

```sql
ALTER TABLE assistant_evaluation
    ADD COLUMN IF NOT EXISTS close_reason           VARCHAR(20),
    ADD COLUMN IF NOT EXISTS session_duration_sec   INT,
    ADD COLUMN IF NOT EXISTS repeat_question_count  INT,
    ADD COLUMN IF NOT EXISTS composite_score        SMALLINT,    -- 0–100 综合质量分
    ADD COLUMN IF NOT EXISTS score_breakdown        JSONB;       -- 各维度详细分解

-- 索引：按综合分查询差 case
CREATE INDEX IF NOT EXISTS idx_eval_composite_score
    ON assistant_evaluation(composite_score)
    WHERE composite_score IS NOT NULL;

-- 索引：按关闭原因分析
CREATE INDEX IF NOT EXISTS idx_eval_close_reason
    ON assistant_evaluation(close_reason)
    WHERE close_reason IS NOT NULL;

COMMENT ON COLUMN assistant_evaluation.composite_score IS '综合质量分 0-100，由 QualityScoreService 计算';
COMMENT ON COLUMN assistant_evaluation.score_breakdown IS '各维度分解：{"close_intent": 90, "efficiency": 70, "user_rating": 80, "ai_quality": 65}';
```

### 5.4 `prompt_audit` 表（新增）

```sql
CREATE TABLE IF NOT EXISTS prompt_audit (
    audit_id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    case_id             VARCHAR(20) REFERENCES "case"(case_id),
    conversation_id     UUID REFERENCES conversation(conversation_id),
    assistant_type      VARCHAR(50),
    model               VARCHAR(100),
    -- 元数据（始终记录，100% 覆盖）
    message_count       INT,
    has_sop             BOOLEAN,
    kb_chunks_count     INT,
    kb_top_score        FLOAT,
    system_prompt_chars INT,
    -- 完整 payload（按策略采样，~10-20%）
    messages            JSONB,           -- 完整 messages[] 数组
    payload_ref         VARCHAR(200),    -- MinIO 文件路径（方案 C 使用）
    -- 关联回路
    user_rating         SMALLINT,        -- 用户评分后回填
    created_at          TIMESTAMPTZ DEFAULT now(),
    trace_id            VARCHAR(64)
);

CREATE INDEX idx_prompt_audit_case     ON prompt_audit(case_id);
CREATE INDEX idx_prompt_audit_has_sop  ON prompt_audit(has_sop) WHERE has_sop = false;
CREATE INDEX idx_prompt_audit_rating   ON prompt_audit(user_rating) WHERE user_rating IS NOT NULL;

COMMENT ON TABLE prompt_audit IS 'AI 层入口 Prompt 审计镜像，每轮对话采集一条元数据记录，按策略采样存储完整 payload';
```

## 6. API 设计

### 6.1 用户评分提交（主动评分轨道）

**接口**：
```
POST /conversations/{conversation_id}/evaluate
Authorization: Bearer <user_token>
Content-Type: application/json

{
  "score": 4,              // 1–5 星
  "feedback": "回答很准确，但第一步指导有点模糊"   // 可选
}
```

**响应**：
```json
{
  "evaluation_id": "uuid",
  "score": 4,
  "composite_score": 82,      // 更新后的综合分（含用户评分）
  "message": "感谢您的评价！"
}
```

**触发副作用**：
1. 写入 / 更新 `assistant_evaluation.score` 和 `feedback`
2. 重新计算 `composite_score`（加入用户评分维度）
3. 如评分 ≤ 3，触发 `prompt_audit` 完整 payload 补充采集

**错误码**：
- `404`：conversation 不存在或不属于当前用户
- `409`：已评分过（幂等设计，实际返回已有评分记录，不报错）

## 7. 前端实现

### 7.1 评分卡交互流程

```
handleCloseCase() 执行流程：
  1. 调用 caseApi.close(case_id, { close_reason: "user_command" })
  2. case 关闭成功 → 在对话框追加系统消息："工单已关闭"
  3. 触发 showRatingCard = true（评分卡弹出）
```

## 10. 关键设计决策记录

### 决策 1：为什么 composite_score 不强依赖用户评分？

**问题**：用户评分是最直接的质量指标，为什么不以它为主？  
**决策**：composite_score 在无用户评分时自动降级为三维模型（关闭意图 + 效率 + AI质量），保证 100% 覆盖。  
**理由**：评分覆盖率 ~20–40%，如果强依赖，60–80% 的 case 将没有质量分，失去整体质量监控能力。

### 决策 2：close_reason 存在哪张表？

**决策**：同时写入 `case.close_reason` 和 `assistant_evaluation.close_reason`。  
**理由**：`case` 表存的是工单业务状态，`assistant_evaluation` 存的是 AI 质量评估数据，两张表职责不同但都需要 `close_reason`，冗余存储避免跨表 JOIN。

### 决策 3：综合分权重是否固定？

**当前**：权重硬编码（`user_rating: 35%`，`close_intent: 25%`，`efficiency: 20%`，`ai_quality: 20%`）。  
**未来演进**：权重可以通过配置文件（`values.yaml`）或后台管理页面动态调整，支持 A/B 实验不同权重方案。

### 决策 4：评分卡 3 秒自动消失的 timeout 是否合适？

**当前**：3 秒。  
**依据**：参考 Toast 通知 UX 规范，3–5 秒是"用户能看到但不打扰操作"的合理范围。如果测试中发现提交率过低，可调至 5–8 秒。
