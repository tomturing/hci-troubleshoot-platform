# KBD 截图识别与语义化设计

## 变更历史

| 日期 | 版本 | 变更内容 |
|------|------|---------|
| 2026-04-17 | v1.0 | 初版：确立 Vision LLM 单次调用 + 上下文注入的图片语义化方案（v3 desc.txt 格式）；废弃 OCR 路线与双 LLM 调用路线 |

---

## 一、问题定义

### 1.1 目标

KBD（知识库文档）原本是给人阅读的故障排查案例，文档中夹杂大量截图。现在要把这些文档输入给**没有视觉能力的 AI**（RAG 检索），因此需要把截图"语义化"——翻译成 AI 可理解的文字。

所以截图处理的最终目标不是"把图里的字提出来"，而是：

> **结合图片在文档中的上下文，用自然语言描述这张截图在排查过程中展示了什么、说明了什么问题**。

### 1.2 服务对象与需求差异

| 消费方 | 需要什么 | 对应字段 |
|--------|---------|---------|
| AI（RAG 召回） | 图片的语义解释，可被向量检索 | `DESCRIPTION` |
| 人工审核员 | 图片的原始文字内容（审核文档中无图片） | `FULL_TEXT` |
| 前端展示 | 截图类型标签、卡片颜色 | `TYPE`、`BACKGROUND` |

三类需求互相独立，需要同时满足。

## 二、方案选型

### 2.1 OCR vs Vision LLM

| 维度 | OCR（PaddleOCR/EasyOCR） | Vision LLM |
|------|--------------------------|------------|
| 能回答的问题 | "图里有哪些字符" | "这张图在说什么" |
| 精度场景 | 等宽字体、高对比度的终端/日志 | 任意类型截图 |
| 是否理解语义 | ❌ | ✅ |
| 是否需要上下文 | ❌ | ✅（设计核心） |
| 适合本需求 | ❌（只是工具，不是答案） | ✅ |

**结论：本场景的核心需求是"语义理解"，OCR 是文字提取工具而非语义理解工具，全程使用 Vision LLM。**

> OCR 方案（ocr.py）已废弃，不再引入。

### 2.2 旧方案的问题（已废弃）

旧方案 `image_proc.py` 的实际运行路径：

```
图片 → Vision LLM（照录文字，当 OCR 用）→ qwen3-max（分析语义，无图片无上下文）
```

两个问题：
1. **把 Vision LLM 当 OCR 用**：让它照录文字，扔掉了理解语义的能力
2. **分析 LLM 无上下文**：第二次调用只收到 OCR 文字，没有文档背景，无法理解图片的"意义"

## 三、新方案：Vision LLM 单次调用 + 上下文注入

### 3.1 整体流水线

```
raw.json（HTML 文档）
  ↓
[Step 0] 文档解析与图文关系提取（纯 Python，无 LLM）
  - 用 BeautifulSoup 提取图片 URL，去重保序（与 fetcher 逻辑一致）
  - 记录每个 URL 在 HTML 中首次出现的字符位置（pos）
  - 按上下文提取规则，为每张图片生成 context_text

[Step 1] 对每张图片并发处理（asyncio.Semaphore，并发数=VISION_CONCURRENCY）
  ├── Pillow 背景色采样 → BACKGROUND（本地，无网络无 LLM）
  └── Vision LLM 单次调用（图片 + context_text + Prompt v3）
        → FULL_TEXT（图片文字原文照录）
        → DESCRIPTION（结合上下文的语义解释）

[Step 2] 规则引擎 → TYPE（本地，基于 FULL_TEXT + BACKGROUND，无 LLM）

[Step 3] FULL_TEXT 截断 → visibleContent（基于 TYPE 和行数阈值，无 LLM）

[Step 4] 组装 desc.txt v3 写入文件系统（幂等：已存在则跳过）
```

**LLM 调用次数从 2 次/图 降为 1 次/图。**

## 四、上下文提取规则

### 4.1 规则

```python
MIN_CONTEXT_CHARS = 80    # 低于此字数视为上下文不足，向上扩展
SHORT_WINDOW = 300        # 优先取图片前 300 字净文字
LONG_WINDOW = 800         # 不足时扩展到图片前 800 字净文字

def extract_context(html: str, img_pos: int) -> str:
    """
    提取图片上文，优先短窗口，不足则向上扩展。
    strip_html()：去除 HTML 标签、&nbsp;、多余空白。
    """
    short = strip_html(html[max(0, img_pos - SHORT_WINDOW):img_pos]).strip()
    if len(short) >= MIN_CONTEXT_CHARS:
        return short[-SHORT_WINDOW:]    # 取最近的 300 字
    long = strip_html(html[max(0, img_pos - LONG_WINDOW):img_pos]).strip()
    return long[-LONG_WINDOW:]          # 不足时取最近的 800 字
```

### 4.2 案例 34977 实测效果

| 图片 | img_pos | 短窗口字数 | 实际取法 | 上下文内容摘要 |
|------|---------|-----------|---------|--------------|
| img_0 | 624 | 25 字 | 扩展至 800 字 | "虚拟机已开启CBT，执行热迁移后快照触发HA" |
| img_1 | 1971 | 397 字 | 使用短窗口 | "查看问题虚拟机的qemu日志，存在如下报错" |
| img_2 | 3767 | 514 字 | 使用短窗口 | 包含排查步骤和 grep 命令 |

## 五、Prompt 设计（v3）

```python
_VISION_PROMPT_V3 = """\
你是HCI超融合平台故障排查文档助手。

这张截图出现在一篇故障排查案例文档中，截图前的文档内容如下：

【文档上下文】
{context}
【上下文结束】

请完成以下两个任务，严格按格式输出，禁止输出其他任何内容：

FULL_TEXT:
[将截图中所有可见文字原文照录，每行一个"- "条目]
[日志/终端截图：每条日志行单独一条，保留时间戳、级别、完整错误信息]
[任务列表截图：每行保留状态（失败/完成/进行中）、任务名、时间，每个字段单独一条]
[若截图中完全没有文字：- （无文字）]

DESCRIPTION:
[结合上方文档上下文，用2-4句技术语言描述：
  ① 这张截图展示了什么内容（是什么）
  ② 它与上下文中描述的故障现象有何关联（说明什么）
  ③ 截图揭示了什么问题、状态或结论（得出什么）
 不要复述上下文原文；用截图信息来解释和印证上下文；输出为连续段落，不要用列表]
"""
```

### 5.1 Prompt 设计原则

- **上下文先于图片**：让 LLM 先建立文档背景，再理解图片，符合人类专家的阅读顺序
- **两任务分离**：FULL_TEXT 是照录（客观），DESCRIPTION 是解读（语义），两者不互相污染
- **TYPE 不由 LLM 决定**：TYPE 由规则引擎本地判断，减少 LLM 输出复杂度
- **禁止列表格式的 DESCRIPTION**：DESCRIPTION 是给 RAG 用的自然语言段落，列表格式在向量召回中效果更差

## 六、TYPE 规则引擎

```python
import re

def classify_type(background: str, full_text: list[str]) -> str:
    """
    基于背景色 + 文字内容，用正则规则本地判断截图类型。
    覆盖率目标：≥ 85%，剩余"其他截图"供人工审核纠正。
    """
    text = " ".join(full_text)

    if background == "黑色":
        # 终端：含 Shell 提示符或典型命令行特征
        if re.search(r'\$\s|#\s|sudo |grep |chmod |cat |ls |\-rn |sfvt_', text):
            return "终端截图"
        # 日志：含时间戳格式 + 日志级别关键词
        if re.search(r'\d{4}-\d{2}-\d{2}[\sT]\d{2}:\d{2}:\d{2}', text) or \
           re.search(r'\b(ERROR|WARN|error|warn|INFO|DEBUG|FATAL)\b', text):
            return "日志截图"
        # 黑色背景兜底
        return "终端截图"

    if background == "白色":
        # 告警：含告警级别词
        if re.search(r'紧急|严重|告警|未处理|已触发', text):
            return "告警截图"
        # 任务：含任务状态词（优先于告警检查）
        if re.search(r'失败|完成|进行中|操作人|HA恢复|修复.*快照|新建.*快照', text):
            return "任务截图"

    return "其他截图"
```

> **规则不覆盖时**：返回"其他截图"，不 fallback 到 LLM。人工审核时可在 KBD 管理页手动纠正 TYPE。

## 七、FULL_TEXT 截断规则

### 7.1 设计原则

FULL_TEXT 在 desc.txt 中**全量存储**（不截断），截断只在**前端展示层**按阈值动态计算。

原因：
- 全量文字是人工审核的依据，不能在存储侧丢弃
- 不同场景下展示行数需求可能变化，截断阈值应为配置项
- 后端截断会导致信息不可恢复

### 7.2 截断方向规则（前端实现）

```
MAX_VISIBLE_LINES = 10      ← 当前配置值，可调整

告警截图、任务截图  → 取前 N 行（最新/最重要内容在最前面）
终端截图、日志截图  → 取后 N 行（最新命令/最近日志在末尾）
其他截图           → 取前 N 行
```

**当前代码状态**：前端 `parseScreenshotBlockV2` 已实现此逻辑（`KbdReviewView.vue` L469-L516）。后端 `image_proc.py` 无截断逻辑，符合"全量存储、前端截断"的设计原则，**无需修改**。

## 八、新 desc.txt 格式（v3）

```
BACKGROUND: 白色
TYPE: 任务截图
FULL_TEXT:
- 失败
- HA恢复虚拟机
- 2025-10-14 02:00:58
- 2025-10-14 02:01:12
- 失败
- 修复虚拟机快照链
- （... 全量照录，不截断 ...）
DESCRIPTION:
该截图为任务列表，展示了2025-10-14凌晨虚拟机触发HA后，系统自动执行"新建虚拟机快照"、"HA恢复虚拟机"、"修复虚拟机快照链"等恢复任务持续失败的情况。结合上文"虚拟机开启CBT后热迁移导致HA"的故障背景，说明底层快照链已损坏，系统自动恢复机制无法生效，需人工介入清理快照链。
```
