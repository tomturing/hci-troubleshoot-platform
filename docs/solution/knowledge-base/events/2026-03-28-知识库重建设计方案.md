## 一、现状诊断：比预想更严重

### 1.1 sop_skills 目录问题矩阵

| 文件/目录 | 状态 | 根本原因 |
|-----------|------|----------|
| `vm_no_power.md` | ❌ **完全错误** | AI 生成，使用虚构的 `hcicli` 命令（非 Sangfor CLI） |
| `network_unreachable.md` | ❌ **完全错误** | 同上，另加 `journalctl`、`libvirtd` 等非 HCI 工具 |
| `node_degraded.md` | ❌ **完全错误** | 同上，基于通用 Linux KVM 栈，非 Sangfor HCI |
| `storage_offline.md` | ❌ **完全错误** | 同上 |
| `snapshot_failed.md` | ❌ **完全错误** | 同上 |
| `vm_power_failure.md` | ⚠️ **内容真实但格式损坏** | docx 导入 artifact：命令文本重复、换行丢失 |
| `vm_power_failure/chapters/*.md` | ⚠️ **内容真实但格式损坏** | 同上，章节拆分逻辑正确但文本质量差 |

**结论：当前知识库覆盖 198 个分类中的 1 个（虚拟机开机失败），且该 1 个覆盖的格式也不达标。其余 5 个 SOP 应立即从知识库索引中移除，避免误导排障。**

### 1.2 权威参考（docx）的知识密度

`虚拟机开关机失败排障手册.docx` 一份文档即覆盖：
- **1 个 H1 主题**，**3 个 H2 场景分支**（有开机失败任务 / 无任务 / 任务完成）
- **31 个 H3 错误类型**（CPU不足、内存不足、序列号过期、存储不可访问…等）
- **91 个 H4 具体根因**，**134 个 H5 诊断/解决步骤**
- 总计 ~1004 段内容，全部为真实 acli 命令和操作步骤

这才是知识库应该达到的密度和质量基准。类比推算：198 个分类如果都有同等密度的覆盖，知识库体量将是目前 sop_skills 的 **200 倍以上**。

## 二、挑战深度分析：不止两个

用户识别了两个挑战（分类准确性 + 图片转文字）。基于数据实际观察，完整挑战有六个：

### 挑战 1（用户已识别）：准确分类

7000 篇案例 → 198 个叶节点分类。挑战点：
- **长尾分布**：VM 失败约占 40%，存储约 25%，其余稀疏；稀疏类别样本不足，分类器偏向多数类
- **多标签问题**：VM 迁移失败同时属于「虚拟机」和「存储」，单标签分类天花板低
- **描述模糊**：客户描述"网络有问题"需要额外上下文才能定位到 23 个网络子分类之一

### 挑战 2（用户已识别）：图片转文字

案例中图片类型不统一，处理复杂度差异极大：

| 图片类型 | 占比估计 | 处理方案 | 难度 |
|---------|---------|---------|------|
| 错误弹窗截图（错误码+描述） | ~50% | OCR + 模板匹配 | 低 |
| CLI 输出截图 | ~20% | OCR + 结构化解析 | 中 |
| 管理控制台状态截图 | ~20% | Vision 模型 + 结构描述 | 中 |
| 网络拓扑/架构图 | ~5% | 专用 Vision 提取 | 高 |
| 无关截图（操作流程演示）| ~5% | 过滤 | 低 |

**核心问题**：错误码是排障 AI 的关键锚点，但大量错误码被截图关押，完全不可检索。

### 挑战 3（新增）：知识时效性

HCI 平台版本（5.x / 6.x / 6.7.x / 6.11.x）行为差异显著：
- `vm_power_failure.md` 中明确有"只在 6.8.0 以后版本生效"、"升级到 6.11.1 解决"等版本标记
- **旧案例中的临时方案在新版本中可能不再需要，甚至有害**（如手动重启某服务在新版本会触发 HA）
- 没有版本元数据的知识是危险的

### 挑战 4（新增）：知识互斥与冲突

7000 篇案例来自不同版本、不同工程师：
- 同一问题可能有 3 种不同解决方案，且都宣称"已解决"
- 部分案例"解决"实际是"绕过"，客户满意但根因未修复
- SOP 是权威来源，但 SOP 本身也在更新

**缺乏冲突检测 = 知识库的幻觉来源**

### 挑战 5（新增）：知识类型不统一

现有 SOP 将 5 种本质不同的知识类型混写在一起：

```
1. 诊断假设（Hypothesis）："如果任务描述含'CPU不足'，进入此分支"
2. 可执行步骤（Executable Step）："运行 acli system top，查看 %CPU 列"
3. 判断条件（Decision Gate）："若 VmallocUsed > 10G → 上升研发"
4. 修复动作（Fix Action）："关闭非关键虚拟机，释放资源"
5. 背景知识（Background）："percpu 异常通常由 IPMI 驱动 msghandler 导致"
```

AI 排障 Agent 对这 5 种类型的使用方式完全不同：诊断假设用于决策路由，可执行步骤用于 tool_call，背景知识用于向用户解释。混存 = 降低了所有类型的检索精度。

### 挑战 6（新增）：RAG 非友好切块

树状 SOP 结构直接切块后，每个 chunk 丢失了上下文：

```
❌ 不好的 chunk（孤立节点）:
"检查 /sf/data/local 分区使用率：acli system df -h /sf/data/local
 清理分区后界面重试开机"

✼ 好的 chunk（含自描述上下文）:
"[虚拟机开机失败 > 有开机失败任务 > 主机异常-exporter服务未运行]
 现象：prometheus服务未运行导致开机失败
 判断：acli system df -h /sf/data/local 查看 /sf/data/local 分区
 条件：分区使用率满（>/95%）
 解决：清理分区 → 界面重试开机"
```

## 三、核心设计：知识原子模型

### 3.1 原则：用「知识原子」统一三类知识源

| 知识源 | 优势 | 弱势 |
|--------|------|------|
| **docx 排障手册** | 精确、可操作、有 acli 命令、有版本信息 | 仅覆盖部分场景，需专家持续维护 |
| **7000 案例库** | 真实客户语言、覆盖边缘场景、有时间维度 | 质量参差、常无根因、图片依赖 |
| **acli/SCP 工具目录** | 可执行、有确定输入输出 | 无诊断上下文 |

这三类知识**分开处理、统一表示、融合存储**。统一表示的基础单元是**知识原子（Knowledge Atom）**。

### 3.2 知识原子数据模型

```yaml
# 示例：一个知识原子
id: "ka-vm003-cpu-step-01"
type: "diagnostic_step"          # hypothesis | diagnostic_step | decision_gate
                                 # fix_action | background | escalation

# ── 分类锚点 ──────────────────────────────────────────────────────
category_id: "虚拟机-003"         # → category_baseline.yaml 叶节点
category_path: ["虚拟机", "虚拟机开机失败"]
knowledge_domain: "sop"          # sop | case | inferred

# ── 触发条件（何时检索此知识原子）────────────────────────────────
trigger:
  stage: "S2"                    # 诊断阶段（S0 意图识别 → S6 方案确认）
  task_error_keywords:           # 任务失败关键词匹配
    - "CPU不足"
    - "此主机剩余可配置CPU不足"
  symptom_vectors: null          # RAG 向量检索（fallback）

# ── 诊断步骤（type=diagnostic_step 时填写）───────────────────────
steps:
  - seq: 1
    description: "查看主机 CPU 占用情况"
    tool: "acli"
    command: "acli system top"
    risk_level: 1
    expected_output: "观察 %CPU 列，正常单 VM 不超过配置 vCPU 核数 × 100%"
    abnormal_condition: "某 KVM 进程 %CPU > 核数×100%+20%"
    abnormal_next: "ka-vm003-cpu-step-02"

  - seq: 2
    description: "定位高 CPU 进程对应的虚拟机"
    tool: "acli"
    command: "acli system ps axuf | grep ${PID}"
    risk_level: 1
    expected_output: "含虚拟机名称和完整启动参数"

# ── 解决方案（type=fix_action 时填写）───────────────────────────
solution: null

# ── 元数据（质量、来源、生命周期）───────────────────────────────
metadata:
  source_type: "docx"
  source_ref: "虚拟机开关机失败排障手册.docx § CPU不足 > 判断方法"
  applicable_versions:
    min: "5.0"
    max: null                    # null = 当前最新
  confidence: 0.95
  verified: true
  created_at: "2026-03-22"
  # 生产反馈（运行时更新）
  usage_count: 0
  feedback_positive: 0
  feedback_negative: 0
```

> **核心价值**：知识原子让每个可检索内容都有：①类型标签（AI 知道怎么用）, ②触发条件（精确检索），③元数据（质量可追溯），④版本范围（避免错用旧方案）

## 五、案例库深度处理策略

### 5.1 分类策略（用户挑战 1 的深化）

**推荐：三阶分类 + 主动学习**

```
阶段一（粗分，自动，98%+ 准确率）
  输入：案例标题 + 问题描述前 200 字
  输出：5 大域分类

阶段二（精分，LLM，目标 80%+ 准确率）
  输入：粗分结果 + 完整案例文本（含 OCR 图片）
  prompt：参考 category_baseline.yaml 198 个分类
  输出：top-3 叶节点 + 置信度

阶段三（主动学习循环）
  置信度 < 0.7 → 人工确认 → 写回训练集
  知识库被 AI Agent 引用后 → 记录准确性反馈 → 持续微调分类器
```

**多标签处理**：一个案例允许最多 3 个分类标签，权重分别为 primary/secondary/tertiary

### 5.2 图片转文字策略（用户挑战 2 的深化）

**四步处理流水线：**

```python
def process_case_image(img, context: str) -> ImageExtraction:
    # Step 1: 图片类型识别
    img_type = classify_image_type(img)
    # → error_dialog | cli_output | ui_screenshot | topology | irrelevant

    # Step 2: 按类型选择提取策略
    if img_type == "error_dialog":
        # 模板化提取：错误码、错误描述、组件
        return extract_error_dialog(img)  # 精度最高

    elif img_type == "cli_output":
        # OCR + 命令识别 + 输出结构化
        return extract_cli_with_ocr(img)

    elif img_type == "ui_screenshot":
        # Vision 模型 + 结构化描述
        return describe_ui_state(img, context)

    elif img_type == "topology":
        # 提取节点/IP/连线关系（有限支持）
        return extract_topology(img)

    else:  # irrelevant
        return None

# Step 3: 验证提取结果（错误码与已知错误目录对比）
# Step 4: 将结构化文本回写到案例，作为可检索字段
```

**关键洞察**：错误码是排障 AI 的最强锚点。`系统发现：0x010032F5` 这类错误码在截图中大量存在，必须被 OCR 提取并建立错误码索引，才能支持精确匹配。

### 5.3 案例质量评分模型

```
质量分（0~100）= 
  完整性（30分）: 有问题描述 + 有解决方案/根因
  + 可执行性（25分）: 解决步骤可复现，有具体命令
  + 图片质量（20分）: 图片数量 × 提取成功率
  + 一致性（15分）: 与 SOP 不冲突
  + 时效性（10分）: 如带版本标记则验证版本；越新分越高

质量分 < 40 → 排除（约 30-40% 案例）
质量分 40-60 → 保留，降权
质量分 > 60 → 正常入库
```

## 六、存储层设计

### 6.1 PostgreSQL 表设计

```sql
-- 知识原子表（核心）
CREATE TABLE knowledge_atoms (
    id               VARCHAR(64) PRIMARY KEY,
    type             VARCHAR(32) NOT NULL,   -- 知识类型
    category_id      VARCHAR(32),            -- → category_baseline
    category_path    JSONB,
    knowledge_domain VARCHAR(16),            -- sop|case|inferred
    trigger          JSONB,                  -- 触发条件
    content          JSONB,                  -- steps/solution/background
    metadata         JSONB,                  -- 版本、来源、置信度
    embedding        VECTOR(1536),           -- 语义向量
    created_at       TIMESTAMPTZ DEFAULT NOW(),
    updated_at       TIMESTAMPTZ DEFAULT NOW()
);

-- 错误码索引表（精确匹配的关键）
CREATE TABLE error_code_index (
    error_code       VARCHAR(32) PRIMARY KEY,
    description      TEXT,
    category_ids     JSONB,                  -- 可能对应的分类
    knowledge_atom_ids JSONB,               -- 关联的知识原子
    source           VARCHAR(16)            -- docx|case|manual
);

-- 原始案例表（保留原始，便于重处理）
CREATE TABLE raw_cases (
    case_id          VARCHAR(64) PRIMARY KEY,
    source_url       TEXT,
    title            TEXT,
    content_text     TEXT,                  -- 文字部分
    images           JSONB,                 -- 图片引用+提取结果
    classification   JSONB,                 -- 分类结果 + 置信度
    quality_score    FLOAT,
    processed_at     TIMESTAMPTZ,
    trace_id         VARCHAR(36)
);
```

### 6.2 检索策略（双路并行）

```
查询 = f(diagnostic_stage, symptom, category, tool_output)
        ↓
路由一：精确匹配
  task_error_keywords → error_code_index → knowledge_atoms
  （速度快，准确率高，但覆盖率有限）
        ↓（fallback）
路由二：语义检索
  RAG（BM25 + pgvector + RRF 融合，已在 kb-service 实现）
  （覆盖率高，但精度略低）
        ↓
结果融合 + 版本过滤（排除不适用版本的知识原子）
        ↓
置信度加权排序 → Top-5 知识原子
```

### Phase 1（短期）：docx → 知识原子 + 工具目录

**工期：2 周**

> **为什么先做一个分类再扩展**：docx 手册的来源只有"虚拟机开关机"，其他 197 个分类暂无同等质量的文档。先把这一个分类做深做对，建立 Pipeline 基础，再规模化。

**工期：3 周**

```
4. 评估分类准确率和图片提取成功率
5. 建立人工审核工作流（UI 或 Notion table）

```

### Phase 3（长期）：规模化 + 持续运营

**工期：持续**

```
1. 规模化处理 7000 案例
2. 从 category_baseline.yaml 的 198 个分类出发，识别哪些分类的案例密度最高
3. 优先为高密度分类寻找/撰写对应的权威 docx
4. 建立"知识反馈循环"：Agent 调用知识 → 用户反馈 → 自动更新质量分
5. 建立错误码目录（从所有源中提取）
```

## 八、关键决策 & 风险

### 8.1 关键决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 知识源优先级 | docx > 案例 > AI 生成 | docx 经专家校验；案例质量参差 |
| 切块粒度 | 知识原子（而非段落） | 保留类型标签和触发条件 |
| 分类体系 | category_baseline.yaml（锁定） | 不自动生成新分类，避免碎片化 |
| 冲突处理 | docx 优先，案例用于补充边缘场景 | 避免频率高 ≠ 正确的错觉 |
| 版本策略 | 强制要求 min_version 字段 | 无版本 = 知识不可信 |

### 8.2 风险项

| 风险 | 影响 | 缓解方案 |
|------|------|---------|
| docx 仅覆盖 1/198 分类 | 知识覆盖率极低 | 优先用长尾案例补充，人工撰写高频分类 docx |
| 图片提取失败率 > 30% | 案例质量分低 → 大量排除 | 降低图片对质量分的权重，保留文字质量高的案例 |
| 7000 案例中 ~40% 无明确根因 | 无法生成可操作知识原子 | 仅提取「症状」和「临时解决方案」，不强求根因 |
| 单人维护 Pipeline 瓶颈 | 长期无法规模化 | 设计为全自动 + 抽样人工审核，而非全人工 |

## 附：不建议做什么

1. **不建议自动生成 SOP**：AI 生成的 SOP（如现有 vm_no_power.md）在面对真实 HCI 环境时会产生错误指导。应以 docx 为基准，AI 只做格式转换和结构提取。

2. **不建议全量案例无筛选入库**：7000 案例不经过质量过滤直接入库，会严重稀释知识库精度，导致 AI 给出低质解决方案。

3. **不建议跳过版本标记**：HCI 跨版本变更大，无版本信息的知识不应直接呈现给用户。

4. **不建议独立维护 sop_skills/chapters/ 目录**：这种目录结构不利于 Pipeline 处理和版本管理，应统一进入 `knowledge_atoms` 表，文件系统仅作为导入源不作为最终存储。
