# SOP 解析策略与校验模式设计

> 对应需求：[2026-05-19-SOP多叉决策树解析策略设计需求.md](../../../requirement/events/2026-05-19-SOP多叉决策树解析策略设计需求.md)
> 全量设计文档：[SOP多叉决策树设计.md](../SOP多叉决策树设计.md)

---

## 一、背景与讨论还原

### 1.1 原有设计的两处错误假设

**错误假设 1：标题层级 → 节点角色存在固定映射**

原文档写道"H4=叶节点，H5=判断方法/解决方案"。但这个假设在以下场景失效：
- 只有 1 个案例的 SOP：H1 直接到 H2，H2 就是叶节点
- 深层嵌套 SOP：H6 才是叶节点
- 作者跳级：H1→H2→H4（没有 H3）

**错误假设 2：Pydantic 层严格校验（model_validator 立即崩溃）**

原设计中，任何叶节点缺失 `diagnosis` 或 `solution` 就让整棵树构建失败。
这意味着残缺文档无法入库、无法进行差异审计、无法展示具体错误位置。

### 1.2 解析策略讨论

用户提出了"找头尾"策略：

> "找头和尾。即首先找到多叉决策树的根和叶子；至于中间如何'生长'可根据格式层级差异、
> 文字内容来决定，对于查错则可以通过语义对比、结构对比等在多章节找共性而找出'异类'"

这是一个**叶优先、边界锚定**的解析策略，核心洞见是：

- SOP 唯一确定的两个语义锚点是：**根**（场景名称）和**叶**（拥有 diagnosis+solution 的节点）
- 中间节点没有独立的语义定义，只是"根到叶路径上的分类节点"
- 叶的身份由**内容证明**（关键词匹配），不由**格式假设**（H 数字）证明

## 二、核心设计决策

### D1：叶节点 = 拥有「判断+解决」配对段落的节点

**原则**：不用标题层级，用内容语义识别叶节点。

```
叶节点判断（解析器层）：
  该节点的子段落中存在「diagnosis 关键词」段落 AND「solution 关键词」段落 → 叶节点
  否则 → 路由节点（等待子节点来证明）
```

**为什么这样做**：叶节点是 SOP 的最终目的地，拥有具体的操作指引。这是 SOP 中
唯一具有稳定语义定义的节点类型——无论文档深度如何，这个特征始终成立。

**为什么不用 H 层级**：层级是格式元数据，不是语义。SOP 的作者不会也不应该
因为"这是叶节点"而被迫使用特定的 H 数字。

### D2：关键词等效表（任一匹配）

| 字段 | 等效关键词 |
|------|-----------|
| `diagnosis` | 判断方法、判断依据、排查方法、排查步骤、识别方法、确认方法、诊断方法 |
| `solution`  | 解决方案、解决方法、处理方法、处理步骤、修复方法、修复步骤、解决步骤 |

关键词表由解析器（`sop_parser.py`）维护，模型层（`sop_template.py`）只记录契约，
不重复关键词表。

### D3：source_heading 溯源字段

`DiagnosisDetail` 和 `SolutionDetail` 各新增 `source_heading: str | None`：

```
docx 原文："处理方法" → source_heading="处理方法"，模型字段名统一是 solution
```

**三层设计原则**：
```
解析器层  → 关键词同义词表（知道"处理方法"=solution）
模型层    → source_heading 保留原文（不丢失原始信息）
展示层    → 展示 source_heading 差异报告，引导作者修正
```

`source_heading` 不参与任何业务逻辑，仅用于审计。

### D4：宽松校验模式（Pydantic 层不抛异常）

**移除 `SOPNode.model_validator`**，改为：

| 层 | 职责 | 行为 |
|----|------|------|
| Pydantic 模型层 | 字段类型校验 | 残缺节点可以构建 |
| 解析器层 (`sop_parser.py`) | 结构完整性检查 | 收集 `ValidationIssue` |
| 校验结果层 (`SOPValidationResult`) | 决策 | `error` 阻断，`warning` 放行 |
| 存储层 (`sop_tree.validation_issues`) | 持久化 | 事后查询审计 |

**为什么宽松而非严格**：

| 场景 | 严格模式 | 宽松模式 |
|------|---------|---------|
| 叶节点缺 solution | 整棵树构建失败，用户看不到是哪个节点 | 树正常构建，错误定位精确到节点路径 |
| 批量导入 50 份文档，3 份有缺陷 | 3 份全部阻断 | 3 份带 warning 入库，47 份正常 |
| 差异审计（source_heading 统计） | 无法执行（树没构建成功） | 可以对已入库的 warning 文档批量审计 |

### D5：validation_issues 的三层价值

```
Layer 1：即时反馈
  上传接口响应 → 用户看到"第 3 个叶节点缺少解决方案"
  生命周期：HTTP 响应结束即丢弃

Layer 2：持久化审计
  sop_tree.validation_issues (JSONB) → 事后查询
  查询示例：SELECT * FROM sop_tree WHERE validation_status = 'warnings'

Layer 3：异类检测原材料
  SELECT source_heading, COUNT(*) FROM … GROUP BY source_heading
  → "处理方法" 出现 5 次，其余 45 份用"解决方案" → 5 份是异类
  → 触发批量话术归一流程
```

## 三、异类检测设计

对同级兄弟节点做结构一致性扫描：

| 检测类型 | 方法 | 输出 |
|---------|------|------|
| 话术不一致 | `source_heading` 频率分布，低频 = 异类 | `warning`，附原文 |
| 缺少 diagnosis 或 solution | 配对完整性检查 | `error` |
| 路由节点意外带内容 | 既有 children 又有 diagnosis | `warning` |
| 同级结构差异过大 | 兄弟节点字段集合差量比较 | `warning` |

**未识别段落的处理**：关键词不匹配的段落归入 `validation_issues`（`warning`），
不阻断入库，前端提示人工确认。

## 五、方案选择与决策依据

### 为什么叶优先而非层级驱动

| 维度 | 层级驱动（原方案）| 叶优先（新方案）|
|------|-----------------|----------------|
| 正确性 | 依赖 H 数字与角色的固定映射，格式不规范即失败 | 锚定语义特征，对任意深度有效 |
| 健壮性 | SOP 层级变化 → 解析逻辑需修改 | SOP 层级变化 → 解析逻辑不变 |
| 容错性 | 无法处理跳级、层级标错的文档 | 跳级/层级错误 → warning 但不失败 |

### 为什么宽松而非严格

见 D4 表格（§二·D4）。核心原因：差异审计需要树存在，树存在需要允许残缺节点入库。
