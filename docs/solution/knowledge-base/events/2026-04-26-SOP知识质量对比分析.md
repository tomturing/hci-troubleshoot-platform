## 四、核心结论

### ✅ 哪份质量更高？

**JSONL版本（15_node_sops.jsonl）质量显著更高**，主要优势：
1. **结构化程度提升84%**：从MD文档 → JSON决策树
2. **Agent友好度提升127%**：每步有明确标记、路径、证据
3. **知识溯源可信度提升150%**：每条知识标注来源案例和置信度
4. **排障效率提升**：决策树可在3-5步定位问题（vs 传统文档需阅读全文）

### ✅ 哪份对Agent最友好？

**JSONL版本绝对优势**：
- **直接可执行**：无需MD解析，JSON结构Agent可直接读取
- **状态可追踪**：每个step_id有明确的if_true_next/if_false_next
- **并发可支持**：多个检查步骤可分配给不同Agent并行执行
- **证据可验证**：每步有expected_result + matched_signals双重验证
- **风险可控**：is_high_risk_operation布尔标记让Agent规避危险操作

## 五、关键问题发现

### 数据库版本的主要缺陷

1. **未完成内容**：`sub_内部异常.md`标记 `#todo`，Agent无法执行
2. **重复冗余**：同一命令在同一文件重复5-10次
3. **缺少状态机**：无法判断何时进入下一章节
4. **缺少证据链**：无法追溯到原始案例

### JSONL版本的技术挑战

1. **存储成本**：79.5KB单JSONL，数据库需支持JSONB存储
2. **解析开销**：需JSONL解析器（vs MD直接阅读）
3. **版本管理**：JSON字段级版本控制需额外工具
4. **生成依赖**：依赖LLM pipeline（glm-4.7-fp8），离线生成不可控

## 六、技术难度对比

| 难度类型 | 数据库版本 | JSONL版本 | Agent适配建议 |
|---------|----------|-----------|------------|
| **解析难度** | 低（MD文本） | 中（JSON解析） | JSONL虽需解析但结构化程度高 |
| **状态管理** | 高（无状态机） | 低（明确路径） | JSONL自带状态转换 |
| **并发执行** | 高（无法并行） | 低（step_id分配） | JSONL适合多Agent协作 |
| **错误处理** | 高（缺少标记） | 低（exit_reason） | JSONL有明确的escalate路径 |
| **知识溯源** | 高（无证据链） | 低（case_ids） | JSONL可追溯到原始案例 |

**综合技术难度**：
- 数据库版本：**对人类友好（易阅读）**，但对Agent难度高
- JSONL版本：**对Agent友好（易执行）**，但需技术投入（解析+存储）

## 七、实施建议

### 短期策略（优先落地）

1. 将JSONL版本导入数据库（`KBDocument.yaml_meta`字段）
2. Agent优先使用JSONL决策树
3. 保留数据库版本作为"参考手册"（人工审核用）

### 镇期架构

1. 数据库新增 `KBSopDecisionTree` 表，存储JSONL决策树
2. 实现三路检索：
   - 精确路由：关键字 → step_id（决策树）
   - 语义检索：向量 → KBChunk（兜底）
   - BM25检索：tsv → KBDocument（传统）
3. 建立pipeline：工单案例 → LLM聚合 → JSONL → 入库

### Agent工作流优化

```
用户报错 → Agent提取关键字 → 匹配JSONL决策树入口
→ 并行执行entry检查（step_id: entry-1/2/3）
→ 匹配matched_signals → 跳转到branch
→ 串行执行checks + solution_steps
→ 验证expected_result → 结束或escalate
```

## 八、最终推荐

**优先使用JSONL版本作为Agent主要知识源**，数据库版本作为人工审核参考。

建议实现混合架构：Agent快速决策使用JSONL决策树，复杂场景fallback到数据库文档人工复核。
