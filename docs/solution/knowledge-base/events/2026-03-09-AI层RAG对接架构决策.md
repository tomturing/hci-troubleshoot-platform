# AI 层与 RAG 层对接架构决策（2026-03-09）

> **决策结果**：采用三阶段演进路线，当前处于阶段一（外挂 RAG）。
>
> **原始文档**：[15_AI层RAG对接架构决策.md](../../archive/15_AI层RAG对接架构决策.md)

---

## 一、背景与问题

在分析 ProductionClaw 如何使用 SOP 知识库时，发现核心冲突：

| openclaw 真实能力 | 当前架构问题 |
|-----------------|------------|
| Pi agent runtime（完整 tool-calling 循环） | 被当成被动 LLM 接口 |
| `TOOLS.md` 定义 KB 调用规范 | 形同虚设，无法执行 |
| 多 agent 路由能力 | 未利用 |

**核心决策问题**：ProductionClaw 应该主动拉取知识，还是被动接收注入？

---

## 二、三阶段演进决策

### 阶段一：外挂 RAG（当前阶段）

```
用户消息
  → conversation-service（RAG 编排者）
    → POST /api/kb/search → 注入 system prompt
    → POST /v1/chat/completions
    → ProductionClaw（被动接收）
```

**优点**：
- 最快让系统可用
- LLM 对照组基准对比最干净

**缺点**：
- openclaw 能力浪费
- 不支持多跳检索

### 阶段二：Agentic RAG（中期目标）

```
用户消息
  → conversation-service（消息路由器）
  → ProductionClaw（自主 tool-calling）
    → 自主 POST /api/kb/search
    → 多跳检索
```

**优点**：
- 完整利用 openclaw 的 agent 能力
- 支持多跳检索
- 与 LearningClaw 形成学习闭环

### 阶段三：混合架构（长期目标）

- SOP 本地化（零延迟）
- session.memory 跨轮持久（一 Pod 一 Case）
- MCP Server 统一接口

---

## 三、决策依据

| 考量维度 | 阶段一 | 阶段二 | 阶段三 |
|---------|-------|-------|-------|
| **实现复杂度** | 低 | 中 | 高 |
| **当前可落地** | ✅ 是 | ⚠️ 需验证 | ❌ 需架构重构 |
| **LLM 对比公平性** | ✅ 最佳 | ⚠️ 分叉 | ⚠️ 分叉加剧 |
| **能力利用度** | 30% | 80% | 100% |

---

## 四、影响范围

- **conversation-service**：职责从"RAG 编排者"逐步降级为"消息路由器"
- **ProductionClaw**：从被动接收逐步升级为自主检索
- **LLM 对照组**：阶段一最适合公平对比，后续分叉
