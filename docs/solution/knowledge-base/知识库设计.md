## 一、概述

### 1.1 系统目标

为 HCI 智能排障平台构建知识库检索增强生成（RAG）系统，实现：

1. **~7000 KBD 案例知识条目**的结构化存储与智能检索
2. **~10 篇 SOP 排障手册**的精确路由与步骤引导
3. **对话场景下的知识注入**，让 AI 助手基于真实案例和 SOP 生成精准答案

### 1.2 核心概念

| 概念 | 说明 |
|------|------|
| **KBD（Knowledge Base Document）** | 历史故障案例，陈述性知识（What-happened），从 `root_cause` 提取假设 |
| **SOP（Standard Operating Procedure）** | 排障手册，程序性知识（How-to），按决策树遍历执行 |
| **category_id** | 分类编码（如 `虚拟机-003`），S0 阶段确认，用于精确路由 |

### 1.3 双轨知识架构

```
用户描述故障（S0 意图识别 → category_id 确认）
        │
        ▼ category_id 已知
[第1轨：SOP 直查] sop_document WHERE category_id = X
        │
        ├─ 命中 ──→ 加载 sop_chunk → SOP 步骤引导 → ✅ 返回
        │
        └─ 未命中
                │
                ▼
        [第2轨：KBD 检索] kbd_entry WHERE category_id = X
                │           BM25(tsv) + 向量(embedding) → RRF → top5
                │
                ├─ 命中 ──→ 注入「历史案例参考」
                │
                └─ 未命中 → 人工兜底
```

## 二、数据存储设计

### 2.1 存储分层

| 数据层 | Source of Truth | 说明 |
|--------|----------------|------|
| kbd_entry | DB | KBD 知识条目，整条 embedding，无分块 |
| sop_document | DB | SOP 文档元信息 |
| sop_chunk | DB | SOP 分块，向量 + 全文索引 |
| kb_category | DB | 分类树 |
| sop_skills/ | 文件系统 | SOP 源文件（keywords_map.json + chapters/） |

### 2.2 数据库 Schema

#### kbd_entry — KBD 知识条目

```sql
CREATE TABLE kbd_entry (
    id                  BIGSERIAL PRIMARY KEY,
    support_id          VARCHAR(20) UNIQUE NOT NULL,    -- 深信服案例ID（幂等键）
    support_url         TEXT,                           -- 原始案例 URL
    title               TEXT NOT NULL,                  -- 案例标题
    content_md          TEXT,                           -- 结构化 Markdown
    metadata            JSONB DEFAULT '{}',             -- 补充元数据

    -- 分类字段
    category_id         VARCHAR(32),                    -- 人工确认分类
    ai_category_id      VARCHAR(32),                    -- AI 分类建议
    ai_category_conf    FLOAT,                          -- 分类置信度
    ai_category_reason  TEXT,                           -- 分类理由

    -- 检索字段（published 时生成）
    embedding           vector(1536),                   -- 语义向量
    tsv                 tsvector,                       -- BM25 全文索引

    -- 状态机
    status              VARCHAR(20) NOT NULL DEFAULT 'draft',  -- draft/published/archived/rejected
    reviewer_id         INTEGER,
    reviewed_at         TIMESTAMP WITH TIME ZONE,
    review_note         TEXT,
    published_at        TIMESTAMP WITH TIME ZONE,

    created_at          TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    updated_at          TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

CREATE INDEX idx_kbd_entry_status ON kbd_entry(status);
CREATE INDEX idx_kbd_entry_category ON kbd_entry(category_id);
CREATE INDEX idx_kbd_entry_embedding ON kbd_entry USING ivfflat (embedding vector_cosine_ops) WITH (lists = 100);
CREATE INDEX idx_kbd_entry_tsv ON kbd_entry USING GIN (tsv);
```

#### sop_document — SOP 文档

```sql
CREATE TABLE sop_document (
    id              SERIAL PRIMARY KEY,
    source_id       VARCHAR(100) UNIQUE,            -- 幂等键，如 sop-vm-start-failure
    category_id     VARCHAR(32),                    -- 关联 KB 分类
    title           VARCHAR(500) NOT NULL,          -- SOP 标题
    content_md      TEXT NOT NULL,                  -- 完整 SOP Markdown
    docx_hash       VARCHAR(64),                    -- 源文件哈希（幂等去重）
    status          VARCHAR(20) DEFAULT 'draft',    -- draft/published/archived
    reviewer_id     INTEGER,
    reviewed_at     TIMESTAMP WITH TIME ZONE,
    published_at    TIMESTAMP WITH TIME ZONE,
    created_at      TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    updated_at      TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

CREATE INDEX idx_sop_document_category ON sop_document(category_id);
```

#### sop_chunk — SOP 分块检索

```sql
CREATE TABLE sop_chunk (
    id              SERIAL PRIMARY KEY,
    document_id     INTEGER REFERENCES sop_document(id) ON DELETE CASCADE,
    chunk_index     SMALLINT NOT NULL,              -- 分块序号（0-based）
    chapter_title   VARCHAR(200),                   -- 章节标题
    content         TEXT NOT NULL,                  -- 分块内容
    embedding       vector(1536),                   -- 语义向量（审核时生成）
    tsv             tsvector,                       -- 全文检索向量
    created_at      TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

CREATE INDEX idx_sop_chunk_document ON sop_chunk(document_id);
CREATE INDEX idx_sop_chunk_embedding ON sop_chunk USING ivfflat (embedding vector_cosine_ops) WITH (lists = 100);
CREATE INDEX idx_sop_chunk_tsv ON sop_chunk USING GIN (tsv);
```

#### kb_category — 分类树

```sql
CREATE TABLE kb_category (
    id              SERIAL PRIMARY KEY,
    parent_id       INTEGER REFERENCES kb_category(id),
    code            VARCHAR(32) UNIQUE NOT NULL,    -- 分类编码，如 "虚拟机-003"
    name            VARCHAR(100) NOT NULL,          -- 分类名称
    level           SMALLINT NOT NULL,              -- 1=L1, 2=L2, 3=L3, 4=L4
    keywords        TEXT[],                         -- 关键字
    created_at      TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

CREATE INDEX idx_kb_category_parent ON kb_category(parent_id);
CREATE INDEX idx_kb_category_code ON kb_category(code);
```

## 三、SOP 代入方法

### 3.1 SOP 存储规范

| 存储单元 | 说明 |
|---------|------|
| `sop_document` | SOP 文档元信息 + 完整内容 |
| `sop_chunk` | 按章节拆分，支持向量 + 全文检索 |
| `sop_skills/` | 文件系统源文件（keywords_map.json + chapters/） |

### 3.2 SOP 路由机制

**两种路由方式：**

#### 方式一：category_id 直查（O(1)，S1+ 推荐）

```python
# S0 阶段已确认 category_id，S1+ 直接查 SOP
async def get_sop_by_category(category_id: str) -> SopDocument | None:
    async with session() as s:
        result = await s.execute(
            select(SopDocument).where(
                SopDocument.category_id == category_id,
                SopDocument.status == "published"
            )
        )
        return result.scalar_one_or_none()
```

#### 方式二：关键字精确匹配（S0 阶段）

```python
# 从 sop_skills/keywords_map.json 加载关键字索引
matcher = SopMatcher(sop_skills_dir="/app/sop_skills")
await matcher.load()

# 匹配用户问题中的关键字
result = matcher.match(user_query)
# 返回: SopMatchResult(skill_id, node_name, matched_keyword, content, file_path)
```

### 3.3 SOP 注入标签

```
【SOP 排障流程 | 来源: {sop_document.title}】
{章节内容}
```

## 四、KBD 代入方法

### 4.1 KBD 存储规范

- **存储单元**：`kbd_entry`（整条 embedding，无分块）
- **检索字段**：`embedding`（向量）+ `tsv`（BM25），RRF 融合
- **必须字段**：`title`、`content_md`、`category_id`、`embedding`、`tsv`

### 4.2 KBD 检索流程

```
POST /api/kb/search
{
  "query": "虚拟机开机失败",
  "top_n": 5,
  "category_l1": "虚拟机",        # 可选，分类过滤
  "category_l2": "开机失败"       # 可选
}
```

**检索流程：**

1. **BM25 检索**：jieba 分词 → tsquery → tsvector 全文匹配 → top-20
2. **向量检索**：embed(query) → pgvector 余弦相似度 → top-20
3. **RRF 融合**：score = Σ 1/(k + rank_i)，k=60
4. **返回 top-N**

### 4.3 KBD 注入标签

```
【历史案例参考 | 案例 #{support_id}】
{案例内容}
```

## 五、知识注入机制

### 5.1 四层 Prompt 组装

Conversation Service 在每轮对话时组装 messages[]：

```python
messages = [
    # Tier 1: 系统角色
    {"role": "system", "content": AGENTS_MD_CONTENT},

    # Tier 2: SOP 知识（可选，命中时注入）
    {"role": "system", "content": f"【SOP 排障流程】\n{sop_content}"},

    # Tier 3: KBD 知识（可选，检索结果）
    {"role": "system", "content": f"【历史案例参考】\n{kbd_chunks}"},

    # Tier 4: 工单上下文
    {"role": "system", "content": f"工单ID: {case_id}\n工程师: {engineer}"},

    # 历史对话（最近 20 条）
    {"role": "user", "content": "..."},
    {"role": "assistant", "content": "..."},

    # 当前消息
    {"role": "user", "content": current_message}
]
```

### 5.2 知识注入时序

```
S0 阶段（意图识别）:
  1. 用户描述故障
  2. 调用 POST /api/kb/classify → 确认 category_id
  3. 调用 POST /api/kb/sop/match → SOP 关键字匹配
  4. 不触发 KBD 检索（S0 专职分类）

S1+ 阶段（诊断执行）:
  1. 用 category_id 直查 SOP → 命中则注入 Tier 2
  2. 用 category_id 过滤 + query 检索 KBD → top-5 注入 Tier 3
  3. 组装完整 messages[] → 发给 LLM
```

### 7.1 KBD 导入方法

KBD 导入是完整的 ETL 流水线：抓取深信服案例 → 图片 OCR → AI 分类 → 写入 DB。

#### classify stage — LLM 集成设计说明

##### 原始设计（v4.4 之前）

| 配置项 | 原始值 | 问题 |
|--------|--------|------|
| `ZAI_BASE_URL` | 硬编码 `http://openclaw:{port}`（Helm template） | 无法切换 LLM 后端 |
| `ZAI_API_KEY` | 始终读 `ZAI_API_KEY` 环境变量 | openclaw proxy 需要 `OPENCLAW_GATEWAY_TOKEN` |
| `LLM_MODEL` | `os.environ.get("ZAI_LLM_MODEL", "gpt-4o-mini")` | 默认值 `gpt-4o-mini` 不被任何后端支持 |
| `base_url` | 直接传递环境变量值 | openai SDK 要求 URL 含 `/v1`，openclaw 无此后缀 |
| `path_labels` 解析 | `json.loads(row.path_labels)` （无类型检查） | JSONB 字段已被 asyncpg 解析为 `list`，再次 `json.loads` 抛 `TypeError` |
| Prompt 模板 | 示例 JSON 中直接写 `{` `}` | Python `.format()` 把 `{}` 当占位符，抛 `KeyError` |
| `openai` 依赖 | 未在 `requirements.txt` 声明 | 容器启动时 `ModuleNotFoundError` |

**实际效果（LLM 调用链）**：
```
脚本 → kb-service /api/kb/classify
  → http://openclaw:18789/chat/completions  ← 404（缺 /v1）
  → http://openclaw:18789/v1/chat/completions （修 URL 后）
  → OPENCLAW_GATEWAY_TOKEN + model=openclaw  ← "API rate limit reached"
     （openclaw 代理内部使用 glm-5，费用高/配额受限）
```

##### 优化后设计（v4.5）

**核心改动一：classify.py 自动适配逻辑**

```python
# 根据 ZAI_BASE_URL 是否含 "openclaw" 自动切换参数
_raw_base_url = os.environ.get("ZAI_BASE_URL", "http://host.docker.internal:18790")
_is_openclaw_proxy = "openclaw" in _raw_base_url

ZAI_BASE_URL = (          # 补齐 /v1 后缀（openai SDK 要求）
    _raw_base_url.rstrip("/") + "/v1"
    if _is_openclaw_proxy and not _raw_base_url.rstrip("/").endswith("/v1")
    else _raw_base_url
)
ZAI_API_KEY = (           # 代理用 OPENCLAW_GATEWAY_TOKEN，直连用 ZAI_API_KEY
    os.environ.get("OPENCLAW_GATEWAY_TOKEN", "")
    if _is_openclaw_proxy
    else os.environ.get("ZAI_API_KEY", "")
)
LLM_MODEL = os.environ.get("ZAI_LLM_MODEL") or (
    "openclaw" if _is_openclaw_proxy          # 代理要求 model="openclaw"
    else os.environ.get("OPENCLAW_DEFAULT_MODEL", "glm-4-flash")
)
```

**核心改动二：Helm template 支持 LLM 后端可配置**

```yaml

  value: {{ .Values.kbService.llmBaseUrl | default (printf "http://openclaw:%v" .Values.openclaw.port) | quote }}
- name: ZAI_LLM_MODEL
  value: {{ .Values.kbService.llmModel | default "" | quote }}

**核心改动三：Dev 环境切换为 ZhipuAI 直连**

#### Vision Stage 截图识别架构（v4.7/v4.8）

> v4.7 完成了前端多维度分类（bgColor + keywords）和 Vision prompt 格式约束。

##### v4.7 — Vision prompt 格式与前端分类规则

Vision（glm-4v-flash/qwen3.5-plus）输出格式约束：

```
0. **截图背景颜色**：[只写：黑色 / 白色 / 其他]
1. **截图界面类型**：[一行描述]
2. **截图中所有可见的文字内容**：
- [原文照录每行]
3. **截图中的报错信息或异常状态**：
- [精确引用；无报错写：无]
4. **对故障排查有价值的技术细节**：
- [关键细节]
```

关键约束：`禁止使用 JSON / Markdown 标题（#） / 表格`，并提供完整示例迫使模型遵守格式。

**前端多维度分类规则**（`detectScreenshotType()`）：

| 优先级 | 类型 | 判断条件 |
|--------|------|---------|
| 1 | 告警截图 🔔 | isWhite && fullText 含「紧急\|普通\|历史告警数\|未处理」 |
| 2 | 任务截图 📋 | isWhite && fullText 含操作人/对象类型/行为/开始时间/结束时间 **或** 完成/失败/进行中 |
| 3 | 终端截图 💻 | isBlack && fullText 含 Shell 特征（$# sudo chmod /var/ /etc/ 0x hex）|
| 4 | 日志截图 📄 | isBlack && typeName 含「日志」 **或** fullText 含时间戳格式（`\d{4}-\d{2}-\d{2} \d{2}:\d{2}:\d{2}`）|
| 5 | 其他截图 🖼️ | 以上均不满足 |

**isBlack/isWhite** 来自字段 0（bgColorText）中的「黑」/「白」关键词（Pillow 独立采样，非 LLM 推断）。

**JSON 格式容错处理**：glm-4v-flash/qwen3.5-plus 偶发返回 JSON 代码块格式，`image_proc.py` 在写 `desc.txt` 前调用 `_normalize_vision_output()` 自动归一化为编号字段格式。

##### v4.8 — 多后端本地 OCR + Vision LLM 兜底

**问题背景**：Vision LLM API 偶发超时（90s），超时时整张图片无法识别，无兜底机制。PaddleOCR 在 WSL2 开发环境 `import paddle` 阶段因 OneDNN 内核兼容问题挂死，无法在 dev 使用。

**解决方案**：引入 EasyOCR 作为 dev 主路径（纯 PyTorch，无 OneDNN 依赖，WSL2 完全兼容），Vision LLM 降为次级兜底。

**OCR 后端决策矩阵**：

| 环境（hci.env.role） | 主路径 | 兜底 |
|---------------------|--------|------|
| dev | EasyOCR（gpu=False） | Vision LLM（qwen3.5-plus） |
| staging / prod | PaddleOCR | Vision LLM（qwen3.5-plus） |
| kubectl 失败 / 其他 | — | Vision LLM 仅（vision_only） |

**环境自动检测**：
```python
# config.py _detect_env_role()
kubectl get ns argocd -o jsonpath='{.metadata.labels.hci\.env\.role}'
# 返回 dev/staging/prod；kubectl 失败 → 降级 "dev"（本地安全默认）
```

**image_proc.py 执行流程**：
```python
async def _process_image(img_path) -> str:
    backend = settings.ocr_backend_effective   # easyocr / paddleocr / vision_only
    if backend in ("paddleocr", "easyocr"):
        extractor = extract_text if backend == "paddleocr" else extract_text_easyocr
        full_text = await asyncio.to_thread(extractor, img_path)
    if not full_text:                          # 本地 OCR 空/失败 → Vision LLM 兜底
        full_text = await _vision_ocr_fallback(...)  # 指数退避重试
    background = await asyncio.to_thread(_detect_background, img_path)  # Pillow
    result = await analyze_screenshot(background, full_text, ...)        # qwen3-max
    return _format_desc_v2(background, result, full_text)
```

**重试机制**（Vision LLM 兜底 + 分析 LLM 均适用）：
- 单次超时：`LLM_TIMEOUT=90s`
- 最大重试次数：`LLM_MAX_RETRIES=3`
- 退避等待：`LLM_RETRY_BACKOFF=2.0` → 实际 2s / 4s / 8s

**新增 config.py 字段**：

| 字段 | 默认值 | 说明 |
|------|--------|------|
| `KBD_ENV` | `""` | 手动覆盖环境名（空=自动检测） |
| `OCR_BACKEND` | `"auto"` | auto / paddleocr / easyocr / vision_only |
| `LLM_TIMEOUT` | `90.0` | 单次 LLM 请求超时（秒） |
| `LLM_MAX_RETRIES` | `3` | 最大重试次数 |
| `LLM_RETRY_BACKOFF` | `2.0` | 指数退避基数（秒） |
| `ocr_backend_effective` | 属性 | 根据 KBD_ENV/OCR_BACKEND 推导实际后端名称 |

### 7.2 SOP 导入方法

SOP 来源是 `.docx` 格式的排障手册，脚本用 `python-docx` 解析章节结构并分块入库。

#### 前置操作（K3s 环境）

```bash
# 1. 端口转发 kb-service 到本地（导入脚本直连 kb-service，非 api-gateway）
kubectl port-forward svc/kb-service -n hci-dev 8004:8004 --address 127.0.0.1 &

# 2. 确认 INTERNAL_API_TOKEN（dev 环境已写入 scripts/kbd/.env，无需修改）
#    如需手动验证：
kubectl exec -n hci-dev deploy/kb-service -- env | grep INTERNAL_API_TOKEN
```

**`scripts/kbd/.env` 默认配置（无需修改）：**
```
KB_SERVICE_URL=http://localhost:8004
INTERNAL_API_TOKEN=dev-internalapi-api-token-2026
```

#### 执行步骤

```bash
# 在项目根目录下执行
cd /path/to/hci-troubleshoot-platform

# 单文件导入（指定对应分类 code）
uv run python -m scripts.kbd.import_sop \
  --file /path/to/vm-startup-failure.docx \
  --category-id "虚拟机-003"

# 批量导入一个目录下的所有 .docx
uv run python -m scripts.kbd.import_sop \
  --dir /path/to/sop_docs/ \
  --category-id "虚拟机-003"
```

> **注意**：`--category-id` 是分类基准中的 `id` 字段，例如 `虚拟机-003`。
> 导入前请先确认分类树已导入（见 7.3）。

#### 导入流程

```
.docx 文件（二进制）
    │
    ├─ compute_docx_hash()：SHA256(原始字节) → docx_hash
    │    └─ 同一文件重复调用 → hash 相同 → 幂等跳过入库
    │    └─ 修改文件内容后保存 → hash 变化 → 新建记录（旧记录保留）
    │
    ├─ parse_docx_to_markdown()：python-docx 读取 Heading 1/2/3
    │    ├─ 提取标题（第一个 Heading 1 或文件名）
    │    ├─ 生成完整 Markdown（full_markdown）
    │    └─ 按章节拆分成 chunks
    │
    └─ POST /api/sop/ingest → sop_document（draft）+ sop_chunk
```

#### category_id 行为说明

| 场景 | 行为 |
|------|------|
| 首次导入 | 新建 `sop_document`，写入 `category_id` |
| 相同文件名 + 相同内容（hash 相同） | 幂等跳过，返回已有 `document_id`，日志 `event=sop_ingest_duplicate_source` |
| **相同文件名 + 内容已修改（hash 不同）** | **upsert：更新内容、重建 chunks、状态重置为 draft（需重新审核）**，日志 `event=sop_ingest_updated` |
| 相同文件名，不同 `category_id` | 更新 `category_id`（仅在内容相同时；若内容也不同，则走 upsert 逻辑一并处理） |
| 不同文件名（不同 `source_id`） | 新建记录，与已有记录并存 |

**导入后**：SOP 默认 `draft` 状态，需在 Admin 界面「SOP管理」中审核发布，发布后：
1. 自动触发 embedding 生成（每个 chunk 生成向量写入 `sop_chunk.embedding`）
2. 进入可被 AI 检索的状态

## 九、分类树结构

### 9.1 数据来源

分类基准文件：`backend/kb-service/config/category_baseline.yaml`

- **来源**：云计算-结构图-AI-全.mm
- **总计**：198 条叶节点（L2-L4）
- **L1 域节点**：5 个（虚拟机/网络/存储/硬件/平台）

### 9.2 分类编码规范

```
格式：<DOMAIN>-<SEQ>

示例：
  虚拟机-001  → 虚拟机创建失败
  虚拟机-003  → 虚拟机开机失败
  网络-015    → 分布式防火墙策略不生效
  存储-008    → 存储卷扩容失败
```

### 9.3 层级结构示例

```yaml
# L2 叶节点（直接属于 L1）
- id: 虚拟机-003
  domain: 虚拟机
  label: 虚拟机开机失败
  path:
    - 虚拟机
    - 虚拟机开机失败

# L3 叶节点（中间有分组）
- id: 虚拟机-001
  domain: 虚拟机
  label: 虚拟机创建失败
  path:
    - 虚拟机
    - 虚拟机创建
    - 虚拟机创建失败

# L4 叶节点（多层分组）
- id: 硬盘-001
  domain: 硬件
  label: 单盘故障排查
  path:
    - 硬件
    - 硬盘
    - 硬盘排障
    - 单盘故障排查
```

### 9.4 五大技术域

| L1 域 | 说明 | 典型分类 |
|-------|------|---------|
| 虚拟机 | 虚拟机生命周期管理 | 创建/删除/开机/关机/迁移/克隆 |
| 网络 | 物理网络 + 虚拟网络 | 网口/VLAN/分布式防火墙/负载均衡 |
| 存储 | 虚拟存储 + 外部存储 | 创建/扩容/快照/备份/恢复 |
| 硬件 | 物理硬件故障 | 风扇/电源/CPU/内存/硬盘 |
| 平台 | 集群管理/系统运维 | 授权/升级/监控/日志 |
